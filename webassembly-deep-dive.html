WEBASSEMBLY

Your browser learned a second instruction set. Not JavaScript. Not native. A portable binary that runs at near-native speed, sandboxed by design — and it's rewriting what the web means_

01 — the escape from javascript

JavaScript was never designed to run a physics engine. It was built in ten days in 1995 to make buttons change color, and for two decades every attempt to make it fast has been an increasingly heroic act of engineering against its own nature. JIT compilers guess what types your variables are, speculate on which branches you’ll take, and when they guess wrong they throw away the compiled code and start over. By 2010, browsers could run JavaScript 100x faster than they could in 2005. That still wasn’t fast enough to render Photoshop.

The escape started with a compiler that shouldn’t have worked. In 2010, Alon Zakai at Mozilla built Emscripten — a tool that took LLVM bitcode from C and C++ programs and compiled it into JavaScript. Not a special JavaScript. Regular JavaScript. The kind your browser already knew how to run. It was slow and the output was enormous, but it proved an idea that mattered: you could use the web as a compilation target.

The asm.js Hack

By 2013, Zakai and Luke Wagner at Mozilla had refined the idea into something that actually performed. They called it asm.js — a strict subset of JavaScript where every variable was explicitly typed using a hack so elegant it was almost offensive. You told the engine a variable was a 32-bit integer by bitwise-ORing it with zero:

asm.jsfunction add(a, b) {
  a = a|0;            // ← type annotation: a is i32
  b = b|0;            // ← type annotation: b is i32
  return (a + b)|0;  // ← return type: i32
}

The |0 is a no-op at runtime — bitwise OR with zero doesn’t change the value. But it signals to the engine: this is an integer, don’t guess, don’t speculate, compile it straight to machine code. Firefox’s SpiderMonkey engine recognized the "use asm" pragma at the top of these modules and switched from its JIT pipeline to ahead-of-time compilation. No warmup. No deoptimization. No re-JIT.

It worked. In August 2013, Epic demonstrated Unreal Engine 4 running at 60 fps in a Firefox tab — compiled from C++ through Emscripten into asm.js. Mozilla’s benchmarks showed asm.js running within 1.5x of native C. On some tests, it beat Clang.

The Dead End Next Door

Google had been chasing the same problem from a different direction. Their answer was Native Client (NaCl), launched in 2011 — a sandbox for running native x86 code inside Chrome. It was fast, genuinely fast, but it only worked in Chrome and only on x86. Portable NaCl (PNaCl) tried to fix the architecture lock-in by using LLVM bitcode as the distribution format, but no other browser vendor adopted it. By 2019, Google deprecated the entire project. NaCl proved that the performance was possible. It also proved that a single-vendor solution was a dead end.

The Convergence

asm.js had the right idea — a compilation target that works everywhere — but it was fighting against the very thing that made it portable. It was still JavaScript. Still UTF-8 text that had to be parsed character by character. A Unity game compiled to asm.js shipped 19 MB of JavaScript source code that took 20–40 seconds to parse on a mobile device. The performance was there once the code ran, but getting to first instruction was brutal.

In June 2015, something unprecedented happened. Mozilla, Google, Microsoft, and Apple — companies that agree on almost nothing — formed the WebAssembly Community Group at the W3C and began designing a binary instruction format together. Not JavaScript. Not native code. Something new: a portable, size-efficient bytecode with a formal semantics that any browser could compile to native machine code in a single pass.

Twenty months later, in March 2017, WebAssembly shipped. Firefox 52 on March 7. Chrome 57 on March 10. Safari and Edge followed within months. By November 2017, all four major browsers supported it. It was the fastest cross-vendor consensus on a new web capability in the platform’s history.

The real function body is five bytes: 20 00 20 01 6A — get local 0, get local 1, add. No ambiguity. No AST. Each byte maps to exactly one instruction. The engine reads them left to right and emits native code as it goes.

Figma was one of the first to make the switch. Their C++ design tool had been compiled to asm.js; moving to WebAssembly cut load time by more than 3x. The bottleneck had never been execution speed — it was the parse tax. Wasm binaries decode more than 20x faster than equivalent JavaScript parses, because there is nothing to parse. The binary format is the instruction stream.

The Parse Tax

asm.js shipped as JavaScript text — megabytes of a = a|0 that browsers parsed character by character before compiling a single instruction. A Unity game took 19 MB of asm.js source. Wasm’s binary format for the same game: 6.3 MB — 3x smaller — and it validated in a single linear pass. No lexer, no parser, no AST. Firefox 58 could stream-compile Wasm at 30–60 MB/sec on desktop — faster than the network could deliver it.

Four Promises

WebAssembly’s design was guided by four goals that became its identity:

Fast

Near-native speed through typed instructions and AOT compilation. No warmup, no speculative optimization, no deopt. The compiler knows every type before it emits a single instruction.

Safe

Sandboxed linear memory, validated before execution. A module cannot read the browser’s memory. It gets a flat byte array and every access is bounds-checked. Validation proves safety before a single instruction runs.

Portable

Architecture-independent bytecode. The same .wasm file runs on x86 laptops, ARM phones, and RISC-V boards. Compile once, run anywhere the web reaches.

Compact

Binary encoding with LEB128 variable-length integers and single-byte opcodes. 20–30% smaller than asm.js uncompressed, 10–20% smaller after gzip. But the real win is decode speed: an order of magnitude faster than parsing text.

In December 2019, the W3C made it official: WebAssembly became a W3C Recommendation, the fourth language of the web alongside HTML, CSS, and JavaScript. Today it runs in 97% of browsers worldwide.

But knowing what Wasm promises is different from knowing how it works. The binary encodes instructions for a machine that doesn’t exist on any real chip — a stack machine with zero registers, structured control flow, and a type system that makes safety provable. That machine is where we go next.

02 — the stack machine

Every processor you have ever used thinks in registers. x86 has eax, ebx, ecx — named slots that instructions read from and write to. ARM has r0 through r15. RISC-V has 32 of them. WebAssembly has zero. It uses a stack: every instruction pops its operands from the top, computes, and pushes the result back. There are no names. There are no destinations. Position is identity.

This sounds primitive. It is also the reason WebAssembly can be validated in a single pass, compiled as fast as it downloads, and proven safe before a single instruction executes.

How It Thinks

Consider the expression (3 + 7) * 2. On a register machine, you would write something like mov eax, 3; add eax, 7; imul eax, 2 — explicitly naming which register holds each intermediate value. On the Wasm stack machine, you simply push operands and let each operator consume what it needs:

Stack trace: (3 + 7) × 2 
  Instruction        Stack (top →)      Effect
  ───────────        ──────────────      ──────
  i32.const 3        [ 3 ]               push 3
  i32.const 7        [ 3, 7 ]            push 7
  i32.add            [ 10 ]              pop 3 and 7, push 10
  i32.const 2        [ 10, 2 ]           push 2
  i32.mul            [ 20 ]              pop 10 and 2, push 20

Five instructions. No register names. No mov to shuttle values between slots. The i32.add instruction does not say “add eax and ebx.” It says “pop two values, add them, push the result.” The compiler that emits this code never has to solve a register allocation problem, and the engine that compiles it to native code can map stack positions to real registers however it likes.

Registers vs. Stack

The natural question: if real CPUs use registers, why does Wasm use a stack? Three reasons.

Compactness. A register instruction needs to encode source and destination registers in every opcode: add eax, ebx encodes three things (operation, source, destination). A stack instruction encodes one: i32.add. The operands are implicit — they are wherever the top of the stack is. This is why Wasm bytecode is denser than register-based bytecodes like LLVM IR or Dalvik — fewer bits per instruction, smaller binaries over the wire.

Validation. On a register machine, the validator has to track which registers hold which types at every point in the program — a data-flow analysis that can be expensive. On a stack machine, the validator maintains a single type stack. At i32.add, it pops two types: if both are i32, the result is i32. If not, the module is rejected. One pass, linear time, no fixed-point iteration.

No overhead in practice. The browser’s compiler maps stack positions to real CPU registers during compilation anyway. V8’s Liftoff compiler does this in a single pass — it maintains a virtual stack that records which register (or spill slot) holds each Wasm stack value. The stack encoding costs nothing at runtime; it only exists in the binary format.

x86 register machine

mov  eax, 3       ; eax = 3
mov  ebx, 7       ; ebx = 7
add  eax, ebx     ; eax = eax + ebx = 10
mov  ecx, 2       ; ecx = 2
imul eax, ecx     ; eax = eax * ecx = 20

Every instruction names its operands. 5 instructions, 3 registers, each encoding source + destination.

Wasm stack machine

i32.const 3      ;; push 3
i32.const 7      ;; push 7
i32.add           ;; pop 2, push sum
i32.const 2      ;; push 2
i32.mul           ;; pop 2, push product

No operand names. Position on the stack IS the identity. 5 instructions, 0 registers, each encoding only the operation.

Structured Control Flow

Native code can jump anywhere. A goto in C can target any label. A jmp in x86 can land on any address. This flexibility is what makes return-oriented programming (ROP) attacks possible — attackers chain together fragments of existing code by hijacking jump targets.

Wasm cannot jump anywhere. It has no goto. Instead, control flow is structured — built from nested blocks that the validator can reason about statically:

Wasm has exactly four control flow constructs. No goto. No arbitrary jumps.

ConstructBehavior
block...endA labeled scope. br jumps forward to the end.
loop...endA labeled scope. br jumps backward to the top.
if...else...endPops an i32. Nonzero → then branch, zero → else branch.
br N / br_if NBranch to the Nth enclosing block. Not an address — a nesting depth.

Every branch target is a structurally enclosing scope. The validator proves, before execution, that control never reaches an unexpected instruction.

Cint find_first_even(int* arr, int len) {
  for (int i = 0; i < len; i++) {
    if (arr[i] % 2 == 0)
      return arr[i];    // early exit
  }
  return -1;
}
WAT(func $find_first_even (param $arr i32) (param $len i32) (result i32)
  (local $i i32)
  (block $exit                   ;; br 1 jumps here ↓
    (loop $loop                  ;; br 0 jumps here ↑
      local.get $i
      local.get $len
      i32.ge_s
      br_if $exit              ;; i >= len? break

      local.get $arr
      local.get $i
      i32.const 4
      i32.mul
      i32.add
      i32.load                 ;; arr[i]
      i32.const 2
      i32.rem_u
      i32.eqz
      (if (then
        ;; found even — return it
        local.get $arr
        local.get $i
        i32.const 4
        i32.mul
        i32.add
        i32.load
        return
      ))

      local.get $i
      i32.const 1
      i32.add
      local.set $i
      br $loop                 ;; continue
    )
  )
  i32.const -1                  ;; not found
)

The key insight: br 1 does not mean “jump to address 0x4A2.” It means “jump to the end of the 1st enclosing block.” The target is a nesting depth, not an address. You cannot construct a branch that escapes its enclosing scope, which means you cannot construct an ROP gadget chain. The control flow graph is a tree of nested scopes, never a spaghetti of arbitrary jumps.

Why No Goto?

Native code’s arbitrary jumps enable ROP attacks — attackers chain together fragments of existing code by hijacking return addresses. Hardware mitigations like Intel CET and ARM BTI add 5–15% overhead to enforce control flow integrity. Wasm gets the same property for free. Structured control flow means the validator can prove, in a single linear pass, that every branch lands on a valid target. The security is baked into the instruction set, not bolted on afterward.

The Validation Pass

Every Wasm module is validated before a single instruction executes. The validator walks through the bytecode once, maintaining a type stack — a parallel stack that tracks not values but their types. At each instruction, it checks: does this operator expect what is currently on top?

Type checking in one pass 
  Instruction        Type Stack       Check
  ───────────        ──────────       ─────
  i32.const 3        [i32]            push i32 — ok
  i32.const 7        [i32, i32]       push i32 — ok
  i32.add            [i32]            expect [i32, i32], got [i32, i32] — ok
  i32.const 2        [i32, i32]       push i32 — ok
  i32.mul            [i32]            expect [i32, i32], got [i32, i32] — ok

  — but what if someone injects a bad instruction? —

  i32.const 3        [i32]            push i32 — ok
  f64.const 2.0      [i32, f64]       push f64 — ok
  i32.add            [ERROR]          expect [i32, i32], got [i32, f64] — REJECT

The module is rejected before it ever runs. Not at the first bad instruction — before any instruction. The entire module either passes validation or is discarded as malformed. This is a security property that native code does not have and JavaScript cannot provide. A Wasm module that passes validation will never execute an undefined operation, never access an illegal type, never misalign the stack. The type system is the sandbox.

We know how the machine thinks — it pushes, pops, and branches through nested scopes. But what does the file that carries these instructions actually look like? The next chapter cracks open a .wasm binary byte by byte.

03 — the binary format

A .wasm file is not source code. It is not an archive. It is a sequence of typed sections laid out in a fixed order, each one answering a single question about the module it describes. The format was designed with one obsession: a browser should be able to start compiling the first function while the last function is still downloading. No backtracking, no symbol resolution phase, no second pass. The binary is the compilation input.

Every .wasm file begins the same way — four magic bytes followed by a version number:

The first 8 bytes 
  00 61 73 6D   01 00 00 00
  ─┬──────────   ─┬──────────
   │              │
   \0asm          version 1

  The magic number spells "asm" in ASCII,
  preceded by a null byte. Every valid
  .wasm file starts with these 8 bytes.

After the header, sections appear in numerical order. Each section starts with a one-byte ID, followed by its length in bytes (encoded as a LEB128 integer), then its payload. The engine reads the ID, reads the length, and either processes the section or skips it entirely — jumping ahead by exactly length bytes to the next section.

The Twelve Sections

A Wasm module can contain up to twelve standard section types, plus any number of custom sections. They must appear in this order:

IDSectionWhat it answers
1TypeWhat function signatures exist? (i32, i32) → i32
2ImportWhat does the module need from the host?
3FunctionWhich function uses which type signature?
4TableWhat indirect call tables exist?
5MemoryHow much linear memory? (initial size, max)
6GlobalWhat mutable/immutable globals exist?
7ExportWhat does the module expose to the host?
8StartWhich function runs automatically on load?
9ElementWhat goes into the indirect call tables?
10CodeThe actual function bodies — the bytecode.
11DataWhat goes into linear memory at startup?
12DataCountHow many data segments? (for validation)

The ordering rule is strict: section 1 before 2, 2 before 3, and so on. Custom sections (ID 0) can appear anywhere between standard sections — they carry names, debug info, or metadata that engines may ignore. This fixed ordering is what makes streaming compilation possible. By the time the engine reaches the Code section, it already knows every type signature, every import, and every memory declaration. It has everything it needs to compile function bodies in parallel.

The Type section defines the signature once; the Function section says “function 0 uses type 0”; the Export section maps the name "add" to function index 0. No redundancy, no ambiguity.

LEB128: The Variable-Length Trick

Every integer in a .wasm file — section lengths, type indices, local counts, branch depths — is encoded as LEB128 (Little-Endian Base 128). The idea is simple: use 7 bits of each byte for data, and the high bit as a continuation flag. Small numbers use fewer bytes:

LEB128 encoding 
  Value     Binary                    LEB128 bytes     Fixed i32
  ─────     ──────                    ────────────     ─────────
  3         00000011                   03               03 00 00 00
  127       1111111                   7F               7F 00 00 00
  128       10000000                  80 01            80 00 00 00
  624485    0100110 0001000 1100101  E5 8E 26         E5 87 09 00

  The high bit of each byte: 1 = more bytes follow, 0 = this is the last byte.
  Data bits are read 7 at a time, least significant group first.

The number 3 costs one byte instead of four. The number 128 costs two bytes instead of four. In a format where most integers are small — type indices, local counts, branch depths are almost always under 128 — this adds up. LEB128 is the same encoding that DWARF debug info uses, and it is a significant part of why Wasm binaries are 10–20% smaller than equivalent native executables.

Streaming Compilation

The section-based format is not just organized — it is organized for speed. Because every section header declares its length upfront, the engine can make decisions before it has seen all the data:

Compilation overlaps with download 
  Network       ─── bytes arriving ──────────────────────────────────▶
                ┌────────┬──────┬──────────────────────────────────┐
                │ header │ type │          code section             │
                │ + meta │ func │  fn 0  │  fn 1  │  fn 2  │ ...  │
                └────────┴──────┴────┬───┴────┬───┴────┬───┴──────┘
                                     │        │        │
  Compile                             ▼        ▼        ▼
  threads                          ┌──────┐ ┌──────┐ ┌──────┐
                                   │ fn 0 │ │ fn 1 │ │ fn 2 │
                                   │ →x86 │ │ →x86 │ │ →x86 │
                                   └──────┘ └──────┘ └──────┘

  By the time fn 2 arrives, fn 0 is already compiled.
  Firefox achieves 30–60 MB/sec compile throughput on desktop —
  faster than most network connections can deliver the bytes.

Each function body in the Code section declares its own length, so multiple compilation threads can split the work without waiting for each other. Firefox 58 introduced this streaming compilation in January 2018, achieving 30–60 MB/sec on desktop and roughly 8 MB/sec on mobile. That means for most internet connections, compilation finishes before the download does — the module is ready to execute the instant the last byte arrives.

Compile faster than you download

On a 20 Mbps connection, you download at ~2.5 MB/sec. Firefox compiles Wasm at 30–60 MB/sec. The compiler is literally waiting for the network. Compilation is no longer on the critical path — download speed is. This is why the binary format’s compactness matters twice: smaller files download faster and compile faster.

The binary format turns out to be a masterclass in cooperative design — every decision (fixed section order, length-prefixed payloads, LEB128 integers, independent function bodies) serves the same goal: get from first byte to first instruction as fast as physics allows. But the binary only tells the engine what code exists. Where that code stores its data — every pointer, every string, every heap allocation — lives in a single flat byte array called linear memory. That is the next layer down.

04 — linear memory

Wasm programs do not have a heap allocator, a virtual address space, or memory protection in the OS sense. They have a flat byte array. One contiguous block of bytes, starting at offset zero, addressable with 32-bit indices. Every pointer in a compiled C program — every malloc result, every stack variable, every string literal — is just an integer index into this array. The i32.load instruction reads four bytes at a given offset. i32.store writes them. That is the entire memory model.

This simplicity is both the performance story and the security story. The engine does not need to emulate page tables or translation lookaside buffers. A load is an array access. And because the array has a known size, the engine can prove that every access is in bounds — in some cases without executing a single check instruction.

Pages and Growing

Linear memory is declared in units of pages, each exactly 65,536 bytes (64 KB). A module declares an initial size and optionally a maximum:

WAT(memory 1 10)  ;; initial: 1 page (64 KB), max: 10 pages (640 KB)

At runtime, a module can request more memory via memory.grow:

WATi32.const 3        ;; request 3 additional pages
memory.grow         ;; returns old size in pages (or -1 on failure)

The new pages are zero-initialized. The existing data is untouched. The base address does not change — the array simply gets longer. If the request would exceed the declared maximum or the engine’s resource limits, memory.grow returns -1 and the memory stays unchanged.

The Guard Page Trick

Every memory access in Wasm must be bounds-checked — an out-of-bounds load or store must trap, not silently read garbage. The naive approach is to insert a comparison before every access: if (offset + size > memory.length) trap(). That is a branch on every load and store, and it is expensive.

Every major engine uses a better approach: virtual memory guard pages. The engine asks the operating system to reserve a large virtual address range — typically 4 GB for a 32-bit Wasm memory plus some extra. The actual linear memory occupies the first N pages (backed by physical memory). Everything beyond that is left unmapped:

Virtual memory layout 
  0                         N pages                        4 GB
  ├─────────────────────────┼────────────────────────────────┤
  │    real memory (RW)      │    unmapped guard region        │
  │    linear memory data    │    any access = hardware trap   │
  ├─────────────────────────┼────────────────────────────────┤
  ▲                         ▲                                ▲
  base pointer              current size             max virtual reservation

  Out-of-bounds access hits unmapped memory → CPU raises fault
  Runtime catches the fault → Wasm trap
  Cost in the common case: zero instructions

An out-of-bounds access hits the unmapped region, the CPU raises a hardware fault (SIGSEGV on Linux, access violation on Windows), and the Wasm runtime’s signal handler catches it and converts it to a Wasm trap. In the common case — the access is in bounds — there is no check at all. No branch, no comparison, no overhead. The hardware does bounds checking for free.

V8, SpiderMonkey, and Wasmtime all use this technique. It is why Wasm memory access is as fast as native pointer dereferencing, despite being sandboxed.

Memory Layout

When Emscripten compiles a C program to Wasm, it lays out the linear memory like a simplified version of a native process:

Emscripten linear memory layout 
  Offset 0                                               memory size
  ├──────────┬───────────────┬──────────────┬─────────────────┤
  │  null    │  static data   │   heap →      │    ← stack     │
  │  guard   │  (globals,     │   (malloc,     │    (locals,     │
  │  page    │   strings)     │    grows up)   │     grows down)  │
  ├──────────┴───────────────┴──────────────┴─────────────────┤

  ■ Offset 0 is intentionally wasted (null pointer guard)
  ■ Static data: string literals, global variables
  ■ Heap: malloc/free, grows upward via memory.grow
  ■ Stack: function locals, grows downward from top

Offset 0 is left as a null pointer guard — accessing address 0 is always a trap. Static data (strings, global constants) sits just above that. The heap grows upward via memory.grow. The shadow stack lives at the top of memory and grows downward — this is not the Wasm operand stack from Chapter 2, but the C-style call stack that holds local variables too large for Wasm locals.

Every int* in the C source becomes an i32 in the Wasm output. Dereferencing that pointer becomes i32.load with the pointer value as the offset. Pointer arithmetic becomes i32.add. The compiler has flattened the entire abstraction of pointers and addresses into indices into a flat array. There is nothing else.

Why 64 KB Pages?

Wasm pages are exactly 65,536 bytes. This is not arbitrary. Windows’ VirtualAlloc allocates in 64 KB granularity. ARM architectures optionally support 64 KB pages. And x86 uses 4 KB pages, so 64 KB is always an exact multiple. The 64 KB page size is the lowest common denominator that lets every OS map Wasm memory without waste or misalignment — one memory.grow call maps cleanly to one or more OS-level page operations on every platform.

The Security Boundary

A Wasm module cannot read or write outside its linear memory. Period. There is no system call to map new address ranges. There is no way to dereference a pointer into the browser’s own memory. A buffer overflow in a Wasm module corrupts the module’s own data — it cannot escape the sandbox.

This is fundamentally different from native code, where a buffer overflow can overwrite return addresses on the stack (enabling ROP attacks), corrupt adjacent heap allocations, or read sensitive data from other processes. In Wasm, the call stack (return addresses) is separate from linear memory and completely inaccessible to the program. The shadow stack in linear memory holds copies of local variables, but the actual return flow is managed by the engine and cannot be touched.

For 32-bit Wasm (the current standard), linear memory is capped at 4 GB — the maximum addressable by i32 indices. The Memory64 proposal extends this to i64 addressing for applications that need more. But the principle stays the same: one flat array, one boundary, no escape.

We have the execution model (stack machine), the file format (binary sections), and the data store (linear memory). What remains is the type system that ties them together — four numeric types, a handful of reference types, and a set of rules that make every operation provably correct before it runs.

05 — types and instructions

Wasm has four numeric types. Not four hundred. Not forty. Four: i32, i64, f32, f64. No strings. No structs. No booleans — a boolean is just an i32 where 0 means false and anything else means true. This minimalism is not a limitation. It is the reason every operation is type-checkable in a single linear pass, every instruction has exactly one defined behavior on every input, and the entire machine is provably safe before it runs.

The Type Universe

Integers

i6464 bitsAny 64-bit register (RAX, X0)

No signed/unsigned distinction in the type — the same i32 can be treated as signed or unsigned depending on the instruction. i32.div_s divides signed; i32.div_u divides unsigned. The bits are the same.

Floats

TypeSizeMaps to

Fully IEEE 754 compliant. NaN propagation rules are specified exactly. f64.sqrt of a negative number returns NaN, always — no undefined behavior, no platform variation.

Beyond the four numerics, Wasm also has reference types: funcref (a reference to a function, used for indirect calls) and externref (an opaque handle to a host object, like a JavaScript DOM node). The SIMD proposal adds v128 — a 128-bit vector that holds four i32s, two i64s, four f32s, or two f64s simultaneously. And the GC proposal adds struct and array types for garbage-collected languages. But the four numeric types are the foundation. Everything else is built on top.

No Undefined Behavior

In C, signed integer overflow is undefined behavior. The compiler can assume it never happens, and if it does, anything goes — nasal demons, time travel, whatever the optimizer felt like. In Wasm, every operation has exactly one defined result for every input:

The type system and deterministic semantics together mean every module either validates and runs correctly, or is rejected before a single instruction executes. There is no third state.

The type system is small on purpose. Four numeric types, a handful of reference types, deterministic semantics for every operation. This is what makes single-pass validation possible, what makes the sandbox provable, and what makes compilation to native code straightforward — every Wasm instruction maps cleanly to one or two native instructions. Now it is time to stop reading about the machine and start writing for it.

Part II
The Craft
Writing WebAssembly by hand, one instruction at a time

06 — your first module by hand

Open a text editor. Not a compiler. Not a build system. A text editor. You are going to write a WebAssembly module from scratch in WAT — the human-readable text format that maps one-to-one to the binary. Every parenthesis you type has a corresponding byte in the .wasm output. By the end of this chapter you will have a working module running in the browser, and you will have typed every instruction yourself.

The Skeleton

Every WAT file starts with (module ...). Inside the module, you declare types, functions, memory, exports. Here is the smallest useful module — a function that doubles a number:

WAT(module
  (func $double (param $x i32) (result i32)
    local.get $x
    i32.const 2
    i32.mul)
  (export "double" (func $double))
)

Save this as double.wat. That is a complete, valid WebAssembly module. The func declaration defines a function named $double that takes one i32 parameter and returns one i32. The body pushes the parameter, pushes the constant 2, and multiplies. The export line makes the function visible to the host under the name "double".

Assemble and Run

Convert the text to binary with wat2wasm (from the WABT toolkit), then load it in a browser:

Shellwat2wasm double.wat -o double.wasm
JavaScriptconst { instance } = await WebAssembly.instantiateStreaming(
  fetch('double.wasm')
);
console.log(instance.exports.double(21)); // 42

That is the entire workflow: write WAT, assemble to .wasm, load in JavaScript, call the export. No Emscripten. No Rust. No build system. You wrote raw bytecode instructions in a text file and the browser compiled them to native machine code.

A Complete Module

The double function is trivial. Here is a module that shows off more of the WAT vocabulary — memory, globals, data segments, and multiple exports:

WAT(module
  ;; Declare one page of linear memory (64 KB) and export it
  (memory (export "memory") 1)

  ;; Write a greeting into memory at offset 0
  (data (i32.const 0) "Hello, Wasm!\00")

  ;; A mutable global: counts how many times greet was called
  (global $count (mut i32) (i32.const 0))

  ;; Return the pointer and length of the greeting
  (func (export "greet_ptr") (result i32)
    i32.const 0)           ;; pointer to "Hello, Wasm!"

  (func (export "greet_len") (result i32)
    i32.const 12)          ;; length of "Hello, Wasm!"

  ;; Increment the call counter and return the new count
  (func (export "tick") (result i32)
    global.get $count
    i32.const 1
    i32.add
    global.set $count       ;; count++
    global.get $count)       ;; return new count
)

Notice the patterns. The data segment writes bytes into linear memory at instantiation — you provide an offset expression and a string literal, and the runtime copies the bytes before any function runs. The global is mutable (mut i32) so it persists across calls. The export can be inlined into the func declaration — no need for a separate export statement.

The Full Loop

JavaScriptconst { instance } = await WebAssembly.instantiateStreaming(fetch('greet.wasm'));
const { memory, greet_ptr, greet_len, tick } = instance.exports;

// Read the greeting from Wasm memory
const bytes = new Uint8Array(memory.buffer, greet_ptr(), greet_len());
const greeting = new TextDecoder().decode(bytes);
console.log(greeting);   // "Hello, Wasm!"
console.log(tick());     // 1
console.log(tick());     // 2

You have just written a module with memory, data initialization, mutable state, and multiple exports — in plain text. The string lives in linear memory at offset 0. JavaScript reads it by creating a Uint8Array view over the shared memory.buffer and decoding the bytes as UTF-8. This is how every string crosses the Wasm/JS boundary: bytes in memory, a pointer, and a length.

WAT Is Not an Abstraction

WAT is not a “high-level” representation of Wasm. It is the binary in text form. wat2wasm converts text to bytes; wasm2wat converts bytes back to text. The round-trip is lossless. Every instruction you write in WAT produces exactly one opcode. There is no optimizer between you and the machine.

You can write a module and run it. But the real craft is in control flow — how you express loops, conditionals, and early exits in a machine that has no goto, no for, and no while. That is where the tricks live.

07 — control flow tricks

Wasm has three control flow constructs: block, loop, and if. That is all. No for. No while. No switch. No goto. Every structured loop, every conditional branch, every multi-way dispatch must be built from these three primitives and the br family of branch instructions. This sounds limiting. It is actually a design constraint that produces clean, verifiable code — once you learn the patterns.

Pattern 1: If / Else

The simplest. if pops an i32 from the stack. Nonzero executes the then branch; zero executes the else branch. Both branches must leave the stack in the same state.

WAT;; abs(x): return x >= 0 ? x : -x
(func $abs (param $x i32) (result i32)
  (if (result i32) (i32.ge_s (local.get $x) (i32.const 0))
    (then (local.get $x))
    (else (i32.sub (i32.const 0) (local.get $x)))
  )
)

The (result i32) on the if means both branches must push exactly one i32. This is how Wasm replaces the ternary operator — the if itself produces a value.

Pattern 2: While Loop

A while loop is a block containing a loop. The block provides the exit target (forward jump). The loop provides the continue target (backward jump). br_if to the block exits; br to the loop repeats:

WAT;; while (i < n) { total += arr[i]; i++; }
(block $break
  (loop $continue
    ;; test: if i >= n, exit
    local.get $i
    local.get $n
    i32.ge_s
    br_if $break           ;; ← forward jump OUT of block

    ;; body: total += arr[i]
    local.get $total
    local.get $arr
    local.get $i
    i32.const 4
    i32.mul
    i32.add
    i32.load
    i32.add
    local.set $total

    ;; i++
    local.get $i
    i32.const 1
    i32.add
    local.set $i

    br $continue           ;; ← backward jump to TOP of loop
  )
)
The Block/Loop Trick

br to a block jumps forward (past the end). br to a loop jumps backward (to the top). Same instruction, opposite direction, depending on which construct it targets. This one rule is the key to every loop pattern in Wasm. The block is the exit door; the loop is the re-entry door.

Pattern 3: Do-While

Even simpler — just a loop with a conditional branch at the bottom:

WAT;; do { ... } while (i < n)
(loop $top
  ;; body here...

  local.get $i
  local.get $n
  i32.lt_s
  br_if $top              ;; if i < n, jump back to top
)

Pattern 4: Early Return / Break

Nested blocks give you labeled break targets at any depth:

WAT;; find first even number in array, return it (or -1)
(func $find_even (param $arr i32) (param $len i32) (result i32)
  (local $i i32) (local $val i32)
  (block $done
    (loop $next
      local.get $i
      local.get $len
      i32.ge_s
      br_if $done

      ;; val = arr[i]
      local.get $arr
      local.get $i
      i32.const 4
      i32.mul
      i32.add
      i32.load
      local.set $val

      ;; if val is even, return it
      local.get $val
      i32.const 2
      i32.rem_u
      i32.eqz
      (if (then
        local.get $val
        return))             ;; ← early return with value

      local.get $i
      i32.const 1
      i32.add
      local.set $i
      br $next
    )
  )
  i32.const -1)            ;; not found

Pattern 5: Switch via br_table

C’s switch statement compiles to br_table — a jump table that branches based on an integer index:

WAT;; switch(op) { case 0: add; case 1: sub; case 2: mul; default: 0 }
(block $default
  (block $case2
    (block $case1
      (block $case0
        local.get $op
        br_table $case0 $case1 $case2 $default)
      ;; case 0: add
      local.get $a  local.get $b  i32.add
      return)
    ;; case 1: sub
    local.get $a  local.get $b  i32.sub
    return)
  ;; case 2: mul
  local.get $a  local.get $b  i32.mul
  return)
;; default
i32.const 0)

br_table pops an i32 index from the stack and jumps to the Nth label in the list. If the index is out of range, it jumps to the last label (the default). The nesting is inside-out — case 0 is the innermost block, the default is the outermost. This feels backward until you realize that br targets are counted by nesting depth: case 0 = depth 0, case 1 = depth 1, and so on.

Five patterns. That is the complete vocabulary for all control flow in WebAssembly. Every for, while, switch, break, and continue you have ever written maps to some combination of block, loop, if, and br. The next chapter adds the other dimension of indirection: functions that call functions through tables.

08 — functions and tables

In C, a function pointer is just an address. You can cast it, store it in an array, call it through a void pointer with the wrong arguments, and crash. In Wasm, there are no function pointers in memory. Functions live in a separate, type-checked table — and calling through that table validates the signature at runtime. This is how Wasm implements virtual dispatch, callbacks, and plugin systems without sacrificing the sandbox.

Direct Calls

A regular function call is straightforward. Push the arguments, call the function by index:

WAT(func $square (param $x i32) (result i32)
  local.get $x
  local.get $x
  i32.mul)

(func $sum_of_squares (param $a i32) (param $b i32) (result i32)
  (i32.add
    (call $square (local.get $a))
    (call $square (local.get $b))))

Indirect Calls and Tables

The interesting part is call_indirect — calling a function by index through a table. This is how C function pointers, C++ vtables, and Rust trait objects work in Wasm:

WAT(module
  (type $op (func (param i32 i32) (result i32)))

  (func $add (type $op) local.get 0 local.get 1 i32.add)
  (func $sub (type $op) local.get 0 local.get 1 i32.sub)
  (func $mul (type $op) local.get 0 local.get 1 i32.mul)

  ;; Table: 3 function slots
  (table 3 funcref)
  (elem (i32.const 0) $add $sub $mul)

  ;; Dispatch: call function at table[op_index](a, b)
  (func (export "calc") (param $a i32) (param $b i32) (param $op i32) (result i32)
    local.get $a
    local.get $b
    local.get $op
    call_indirect (type $op))      ;; type-checked at runtime!
)
JavaScriptconst { instance } = await WebAssembly.instantiateStreaming(fetch('calc.wasm'));
instance.exports.calc(10, 3, 0);  // add(10, 3) = 13
instance.exports.calc(10, 3, 1);  // sub(10, 3) = 7
instance.exports.calc(10, 3, 2);  // mul(10, 3) = 30
instance.exports.calc(10, 3, 9);  // TRAP! index 9 out of bounds

call_indirect does two checks at runtime: is the index within the table bounds? Does the function at that index match the declared type signature? If either check fails, the module traps. This is the safety net that replaces C’s unchecked function pointer — you get dynamic dispatch without the ability to call a function with the wrong argument types.

call_indirect Order Gotcha

The table index must be the last value on the stack — on top of the function arguments. In the flat form: local.get $a, local.get $b, local.get $op, call_indirect. In S-expression form: (call_indirect (type $op) (local.get $a) (local.get $b) (local.get $op)). Swap the order and the table index becomes a function argument. This is the most common call_indirect bug.

Direct calls, indirect calls, and tables cover all the ways code can invoke code in Wasm. But the most common thing code needs to work with is not other code — it is data. And the most painful data type in Wasm is the one that does not exist: the string.

09 — strings in the machine

Wasm has no string type. No char. No String. No &str. A “string” in WebAssembly is a sequence of bytes sitting in linear memory, addressed by an i32 pointer, with a length that you track yourself. Every operation you take for granted in a high-level language — concatenation, comparison, searching, slicing — must be built from i32.load8_u, i32.store8, and loop patterns. This chapter builds those primitives by hand.

Two Conventions

There are two ways to know where a string ends:

Null-Terminated (C style)

H e l l o \0
48 65 6C 6C 6F 00

Length = scan until you hit 0x00. Simple but O(n) to find the length, and the string cannot contain null bytes.

Length-Prefixed (Wasm/Rust style)

ptr=100, len=5
H  e  l  l  o
48 65 6C 6C 6F

Length is known upfront. O(1) length check. Can contain any byte, including 0x00. This is what Rust, Go, and most Wasm tooling use.

For our handwritten Wasm, we will use pointer + length. Two i32 values: where the bytes start, and how many there are.

strlen: Measuring a Null-Terminated String

WAT;; strlen(ptr) → length (scans for \0)
(func $strlen (param $ptr i32) (result i32)
  (local $start i32)
  local.get $ptr
  local.set $start
  (block $done
    (loop $next
      local.get $ptr
      i32.load8_u             ;; load one byte
      i32.eqz
      br_if $done             ;; if byte == 0, done
      local.get $ptr
      i32.const 1
      i32.add
      local.set $ptr           ;; ptr++
      br $next))
  local.get $ptr
  local.get $start
  i32.sub)                    ;; return ptr - start

One byte at a time: i32.load8_u reads a single unsigned byte from memory. The loop increments the pointer until it finds 0x00. The length is the difference between the final pointer and the start.

memcpy: Copying Bytes

WAT;; memcpy(dst, src, len) — copy len bytes from src to dst
(func $memcpy (param $dst i32) (param $src i32) (param $len i32)
  (local $i i32)
  (block $done
    (loop $next
      local.get $i
      local.get $len
      i32.ge_u
      br_if $done

      ;; dst[i] = src[i]
      local.get $dst
      local.get $i
      i32.add                  ;; dst + i
      local.get $src
      local.get $i
      i32.add                  ;; src + i
      i32.load8_u              ;; load byte from src
      i32.store8               ;; store byte to dst

      local.get $i
      i32.const 1
      i32.add
      local.set $i
      br $next)))

strcmp: Comparing Strings

WAT;; strcmp(a_ptr, a_len, b_ptr, b_len) → 1 if equal, 0 if not
(func $strcmp (param $a i32) (param $a_len i32)
              (param $b i32) (param $b_len i32) (result i32)
  (local $i i32)
  ;; different lengths → not equal
  local.get $a_len
  local.get $b_len
  i32.ne
  (if (then (return (i32.const 0))))
  ;; compare byte by byte
  (block $done
    (loop $next
      local.get $i
      local.get $a_len
      i32.ge_u
      br_if $done
      ;; if a[i] != b[i], return 0
      (i32.ne
        (i32.load8_u (i32.add (local.get $a) (local.get $i)))
        (i32.load8_u (i32.add (local.get $b) (local.get $i))))
      (if (then (return (i32.const 0))))
      local.get $i
      i32.const 1
      i32.add
      local.set $i
      br $next))
  i32.const 1)              ;; all bytes matched

The JS Bridge for Strings

To pass a string from JavaScript into these functions, you write the bytes into linear memory and pass the pointer and length:

JavaScriptfunction writeString(instance, str, offset) {
  const mem = new Uint8Array(instance.exports.memory.buffer);
  const encoded = new TextEncoder().encode(str);
  mem.set(encoded, offset);
  return { ptr: offset, len: encoded.length };
}

function readString(instance, ptr, len) {
  const mem = new Uint8Array(instance.exports.memory.buffer, ptr, len);
  return new TextDecoder().decode(mem);
}

These two functions — writeString and readString — are the entire bridge between JavaScript’s string world and Wasm’s byte world. TextEncoder converts a JavaScript string to UTF-8 bytes. TextDecoder converts bytes back. Everything in between is pointer arithmetic and byte manipulation.

The String Tax

Every string that crosses the JS/Wasm boundary is copied. JavaScript strings are UTF-16 internally; Wasm works with raw bytes (usually UTF-8). The TextEncoder/TextDecoder calls do the conversion and the copy. For short strings this is negligible. For megabytes of text, batch your crossings — pass a pointer to the entire buffer, not individual strings.

We have strlen, memcpy, strcmp, and a JS bridge for reading and writing strings. These are the primitives. The next chapter shows what compilers do with them — and how to read the output to spot what they get wrong.

10 — reading compiler output

You know how to write Wasm. Now learn to read what your compiler writes. The single most useful tool in the WebAssembly ecosystem is wasm2wat — the disassembler that converts any .wasm binary back into the text format you have been writing by hand. Every optimization insight, every bloat diagnosis, every “why is this function slow?” investigation starts with wasm2wat.

The Disassembly Loop

Shell# 1. Compile your code
emcc -O3 hot_loop.c -o hot_loop.wasm

# 2. Disassemble to readable WAT
wasm2wat hot_loop.wasm -f -o hot_loop.wat   # -f folds expressions

# 3. Read the WAT. Spot the bloat.

# 4. Run Binaryen's optimizer
wasm-opt -O3 hot_loop.wasm -o hot_loop_opt.wasm

# 5. Compare
wasm2wat hot_loop_opt.wasm -f -o hot_loop_opt.wat
diff hot_loop.wat hot_loop_opt.wat

What to Look For

When you open a disassembled .wat file, three patterns signal compiler waste:

Unnecessary locals. LLVM converts every SSA register to a Wasm local. A function with 5 variables might declare 12 locals — the extras are temporaries that could have stayed on the stack. Look for (local $tmpN i32) declarations that appear only in a local.set immediately followed by a local.get. Those are round-trips through a local that the stack could have handled directly.

Redundant local.set/get pairs. The pattern local.set $x → local.get $x (with nothing in between) means the compiler stored a value and immediately loaded it back. A human would leave the value on the stack. wasm-opt’s SimplifyLocals pass catches many of these, but not all.

Multiply where shift works. i32.mul (i32.const 4) can always be replaced with i32.shl (i32.const 2). Compilers usually catch this, but when the multiply is embedded in a complex address calculation, they sometimes miss it.

wasm-opt: The Second Compiler

Binaryen’s wasm-opt is a Wasm-to-Wasm optimizer. It runs after your compiler and applies passes that LLVM does not understand because they are Wasm-specific: dead local elimination, block merging, stack expression folding, and local coalescing. Running wasm-opt -O3 on a compiler-produced binary typically shaves off 10–20%.

PassWhat it does
CoalesceLocalsReuses locals with non-overlapping lifetimes. Fewer locals = smaller binary.
SimplifyLocalsReplaces local.set+local.get pairs with local.tee or stack expressions.
VacuumRemoves dead code, unreachable blocks, unused locals.
RemoveUnusedBrsEliminates branches that fall through to the same place.
MergeBlocksCollapses nested blocks that serve no structural purpose.
The Optimization Budget

Don’t hand-optimize everything. The compiler is right 95% of the time. The remaining 5% is in your hot loop — the inner loop that runs a million times per frame. Use wasm2wat to read that specific function. If the compiler introduced unnecessary locals or missed a stack folding opportunity, fix that one function. Leave the rest alone. Maintainability costs more than the microseconds you would save.

Five chapters of craft: writing modules, control flow patterns, function tables, string handling, and reading compiler output. You now know enough to write Wasm, read Wasm, and debug Wasm. Time to put it all together and build something real.

11 — the scanner

We are going to build a template engine in pure handwritten WAT. It takes a template like "Hello, {{name}}! You have {{count}} messages." and a set of key-value substitutions, and produces the rendered output. No Rust. No Emscripten. Just WAT, a text editor, and wat2wasm. The Plan

The template engine has three phases:

Template engine architecture 
  Input: "Hello, {{name}}!"  +  { name: "World" }
         │
         ▼
  1. Scan — walk the template byte by byte
         find {{ and }} delimiters
         emit literal chunks and variable keys
         │
         ▼
  2. Substitute — for each variable key,
         look up the value in the data
         copy the value bytes to the output buffer
         │
         ▼
  3. Output — return pointer + length
         to the rendered string in linear memory

  Output: "Hello, World!"

Memory Layout

We need to lay out our linear memory carefully. The template, the variable data, and the output buffer all live in the same flat array:

Linear memory map 
  Offset     Region            Size       Purpose
  ───────    ───────           ────       ───────
  0x0000     template input    4 KB       the template string
  0x1000     variable keys     2 KB       packed key strings (ptr+len pairs)
  0x1800     variable values   2 KB       packed value strings
  0x2000     output buffer     8 KB       the rendered result
  0x4000     bump pointer      →         allocation frontier

  All offsets are fixed. Simple and predictable.

The Scanner Function

The scanner walks the template byte by byte. When it sees {{ (two consecutive 0x7B bytes), it switches from “copy literal” mode to “read key” mode. When it sees }} (two 0x7D bytes), it looks up the key and copies the corresponding value to the output:

WAT(module
  (memory (export "memory") 1)  ;; 64 KB

  ;; Globals for output buffer tracking
  (global $out_ptr (mut i32) (i32.const 0x2000))  ;; output write head

  ;; emit_byte: write one byte to the output buffer
  (func $emit_byte (param $b i32)
    global.get $out_ptr
    local.get $b
    i32.store8
    global.get $out_ptr
    i32.const 1
    i32.add
    global.set $out_ptr)

  ;; emit_bytes: copy len bytes from src to output
  (func $emit_bytes (param $src i32) (param $len i32)
    (local $i i32)
    (block $done
      (loop $next
        local.get $i
        local.get $len
        i32.ge_u
        br_if $done
        (call $emit_byte
          (i32.load8_u (i32.add (local.get $src) (local.get $i))))
        local.get $i
        i32.const 1
        i32.add
        local.set $i
        br $next)))

  ;; render: scan template, substitute variables, return (ptr, len)
  (func (export "render") (param $tpl_ptr i32) (param $tpl_len i32)
                           (result i32 i32)  ;; returns (ptr, len)
    (local $i i32) (local $ch i32) (local $next_ch i32)
    (local $key_start i32)
    ;; reset output pointer
    i32.const 0x2000
    global.set $out_ptr

    (block $end
      (loop $scan
        ;; bounds check
        local.get $i
        local.get $tpl_len
        i32.ge_u
        br_if $end

        ;; read current byte
        local.get $tpl_ptr
        local.get $i
        i32.add
        i32.load8_u
        local.set $ch

        ;; check for {{ (0x7B 0x7B)
        local.get $ch
        i32.const 0x7B  ;; '{'
        i32.eq
        (if (then
          ;; peek next byte
          local.get $tpl_ptr
          local.get $i
          i32.add
          i32.const 1
          i32.add
          i32.load8_u
          local.set $next_ch
          local.get $next_ch
          i32.const 0x7B
          i32.eq
          (if (then
            ;; found {{ — skip past it, record key start
            local.get $i
            i32.const 2
            i32.add
            local.set $i
            local.get $tpl_ptr
            local.get $i
            i32.add
            local.set $key_start

            ;; scan for }} (0x7D 0x7D)
            (block $found_close
              (loop $scan_key
                local.get $tpl_ptr
                local.get $i
                i32.add
                i32.load8_u
                i32.const 0x7D  ;; '}'
                i32.eq
                (if (then
                  local.get $tpl_ptr
                  local.get $i
                  i32.add
                  i32.const 1
                  i32.add
                  i32.load8_u
                  i32.const 0x7D
                  i32.eq
                  br_if $found_close))
                local.get $i
                i32.const 1
                i32.add
                local.set $i
                br $scan_key))

            ;; key is from $key_start to tpl_ptr+$i
            ;; call imported lookup function
            local.get $key_start
            local.get $tpl_ptr
            local.get $i
            i32.add
            local.get $key_start
            i32.sub                ;; key length
            call $lookup           ;; → (value_ptr, value_len)
            call $emit_bytes

            ;; skip past }}
            local.get $i
            i32.const 2
            i32.add
            local.set $i
            br $scan
          ))
        ))

        ;; regular byte — emit it
        (call $emit_byte (local.get $ch))
        local.get $i
        i32.const 1
        i32.add
        local.set $i
        br $scan))

    ;; return (output_ptr, output_len)
    i32.const 0x2000
    global.get $out_ptr
    i32.const 0x2000
    i32.sub)

  ;; lookup: imported from JS — finds a variable value by key
  (import "env" "lookup"
    (func $lookup (param i32 i32) (result i32 i32)))
)

The scanner is the heart of the engine. It reads one byte at a time. Literal bytes go straight to the output via $emit_byte. When it finds {{, it scans ahead for }}, extracts the key, calls the imported $lookup function (provided by JavaScript), and emits the returned value bytes. The entire thing is a state machine built from block, loop, and br_if — the same patterns from Chapter 7.

Notice the $lookup import. The Wasm module does not store the variable data — it asks JavaScript to find the value for a given key. This keeps the memory layout simple and lets JavaScript handle the hash map. The module does what it is good at: scanning bytes fast.

12 — wiring it up

The scanner is pure Wasm. The variable lookup is pure JavaScript. Now we connect them. The JavaScript side has three jobs: write the template into Wasm memory, provide the lookup import function, and read the rendered output back. Here is the complete host code:

JavaScriptconst encoder = new TextEncoder();
const decoder = new TextDecoder();

// The template and variables
const template = "Hello, {{name}}! You have {{count}} new messages.";
const vars = { name: "Alice", count: "3" };

// Import object: provide the lookup function
const importObj = {
  env: {
    lookup(key_ptr, key_len) {
      // Read the key from Wasm memory
      const mem = new Uint8Array(instance.exports.memory.buffer);
      const key = decoder.decode(mem.subarray(key_ptr, key_ptr + key_len));

      // Look up the value
      const value = vars[key] || "";
      const encoded = encoder.encode(value);

      // Write the value into Wasm memory at a scratch area
      const val_ptr = 0x1800;  // variable values region
      mem.set(encoded, val_ptr);

      // Return (ptr, len) — multi-value return!
      return [val_ptr, encoded.length];
    }
  }
};

// Load and run
const { instance } = await WebAssembly.instantiateStreaming(
  fetch('template.wasm'), importObj
);

// Write template into Wasm memory
const tpl_bytes = encoder.encode(template);
const mem = new Uint8Array(instance.exports.memory.buffer);
mem.set(tpl_bytes, 0);  // template region starts at 0

// Call render
const [out_ptr, out_len] = instance.exports.render(0, tpl_bytes.length);

// Read the result
const result = decoder.decode(mem.subarray(out_ptr, out_ptr + out_len));
console.log(result);
// → "Hello, Alice! You have 3 new messages."

That is the entire application. The Wasm module scans bytes. The JavaScript host resolves variables. The rendered string comes back as bytes in shared linear memory. Every concept from the last eleven chapters is in play: stack operations, control flow with block/loop/br_if, memory loads and stores, function imports/exports, and the TextEncoder/TextDecoder string bridge.

Why This Matters

This template engine is small — maybe 200 bytes of Wasm after assembly. But the patterns are the same ones Figma uses for its rendering engine, the same ones ffmpeg.wasm uses for video processing, the same ones Cloudflare uses for edge compute. The scale changes. The architecture does not: Wasm scans and transforms bytes in linear memory. JavaScript handles I/O and high-level data structures. The bridge is TextEncoder, a pointer, and a length.

13 — ship it

The template engine works. It runs in a browser tab. But WebAssembly does not stop at the browser. The same .wasm binary can run on a server, in a CLI tool, or at the edge — without changing a single byte. This chapter takes the module we built and deploys it three ways.

Way 1: The Browser

You already did this. instantiateStreaming, an import object, and the result comes back as bytes in shared memory. The module is 200 bytes. It loads instantly. It runs in any browser that supports WebAssembly — which is 97% of browsers worldwide.

Way 2: WASI on the Command Line

The same module can run outside the browser using a WASI runtime like Wasmtime or Wasmer. WASI provides capability-based access to the filesystem, standard I/O, and the clock — but only what you explicitly grant. The module starts with zero permissions.

Shell# Run the template engine via Wasmtime
wasmtime template.wasm --invoke render

# Or via Wasmer
wasmer run template.wasm --invoke render

For a CLI version, you would add WASI imports for reading stdin and writing stdout, replacing the JavaScript bridge with WASI’s file descriptor API. The scanner logic stays identical. The byte-level code does not change — only the I/O boundary.

Way 3: Edge Compute

Cloudflare Workers, Fastly Compute, and Fermyon Spin all run Wasm at the edge. Your template engine becomes an HTTP handler: receive a request with a template and variables as JSON, render the output in Wasm, return the result. Cold start: sub-millisecond. Memory overhead: kilobytes. The same 200-byte module that ran in your browser tab now runs in 300 cities.

Where Wasm Goes Next

The Component Model and WIT (Wasm Interface Types) are turning modules into composable building blocks. A Rust image resizer, a Python ML classifier, and your handwritten WAT template engine can be wired together without sharing memory or agreeing on a calling convention — WIT handles the type translation at the boundary.

WasmGC lets garbage-collected languages (Kotlin, Dart, Java) run efficiently without shipping their own GC. WASI 0.3 adds native async/await for non-blocking I/O. And the AI ecosystem — ONNX Runtime, TensorFlow.js, MediaPipe, web-llm — has converged on Wasm as the portable floor with hardware accelerators (WebGPU, WebNN, CUDA) as the performance ceiling.

The Floor, Not the Ceiling

WebAssembly is not the fastest runtime. It never will be. The matmul belongs to the GPU. The video decode belongs to the hardware codec. The regex belongs to a hand-tuned native library. But Wasm is the only runtime that works everywhere, isolates everything, and composes with any accelerator the host provides. Write it once in WAT. Assemble it with wat2wasm. Run it in a browser, on a server, at the edge, or inside a plugin sandbox. One binary. Every platform. Zero trust required.
