WEBASSEMBLY

Your browser learned a second instruction set. Not JavaScript. Not native. A portable binary that runs at near-native speed, sandboxed by design — and it's rewriting what the web means_

01 — the escape from javascript

JavaScript was never designed to run a physics engine. It was built in ten days in 1995 to make buttons change color, and for two decades every attempt to make it fast has been an increasingly heroic act of engineering against its own nature. JIT compilers guess what types your variables are, speculate on which branches you’ll take, and when they guess wrong they throw away the compiled code and start over. By 2010, browsers could run JavaScript 100x faster than they could in 2005. That still wasn’t fast enough to render Photoshop.

The escape started with a compiler that shouldn’t have worked. In 2010, Alon Zakai at Mozilla built Emscripten — a tool that took LLVM bitcode from C and C++ programs and compiled it into JavaScript. Not a special JavaScript. Regular JavaScript. The kind your browser already knew how to run. It was slow and the output was enormous, but it proved an idea that mattered: you could use the web as a compilation target.

The asm.js Hack

By 2013, Zakai and Luke Wagner at Mozilla had refined the idea into something that actually performed. They called it asm.js — a strict subset of JavaScript where every variable was explicitly typed using a hack so elegant it was almost offensive. You told the engine a variable was a 32-bit integer by bitwise-ORing it with zero:

asm.jsfunction add(a, b) {
  a = a|0;            // ← type annotation: a is i32
  b = b|0;            // ← type annotation: b is i32
  return (a + b)|0;  // ← return type: i32
}

The |0 is a no-op at runtime — bitwise OR with zero doesn’t change the value. But it signals to the engine: this is an integer, don’t guess, don’t speculate, compile it straight to machine code. Firefox’s SpiderMonkey engine recognized the "use asm" pragma at the top of these modules and switched from its JIT pipeline to ahead-of-time compilation. No warmup. No deoptimization. No re-JIT.

It worked. In August 2013, Epic demonstrated Unreal Engine 4 running at 60 fps in a Firefox tab — compiled from C++ through Emscripten into asm.js. Mozilla’s benchmarks showed asm.js running within 1.5x of native C. On some tests, it beat Clang.

The Dead End Next Door

Google had been chasing the same problem from a different direction. Their answer was Native Client (NaCl), launched in 2011 — a sandbox for running native x86 code inside Chrome. It was fast, genuinely fast, but it only worked in Chrome and only on x86. Portable NaCl (PNaCl) tried to fix the architecture lock-in by using LLVM bitcode as the distribution format, but no other browser vendor adopted it. By 2019, Google deprecated the entire project. NaCl proved that the performance was possible. It also proved that a single-vendor solution was a dead end.

The Convergence

asm.js had the right idea — a compilation target that works everywhere — but it was fighting against the very thing that made it portable. It was still JavaScript. Still UTF-8 text that had to be parsed character by character. A Unity game compiled to asm.js shipped 19 MB of JavaScript source code that took 20–40 seconds to parse on a mobile device. The performance was there once the code ran, but getting to first instruction was brutal.

In June 2015, something unprecedented happened. Mozilla, Google, Microsoft, and Apple — companies that agree on almost nothing — formed the WebAssembly Community Group at the W3C and began designing a binary instruction format together. Not JavaScript. Not native code. Something new: a portable, size-efficient bytecode with a formal semantics that any browser could compile to native machine code in a single pass.

Twenty months later, in March 2017, WebAssembly shipped. Firefox 52 on March 7. Chrome 57 on March 10. Safari and Edge followed within months. By November 2017, all four major browsers supported it. It was the fastest cross-vendor consensus on a new web capability in the platform’s history.

The real function body is five bytes: 20 00 20 01 6A — get local 0, get local 1, add. No ambiguity. No AST. Each byte maps to exactly one instruction. The engine reads them left to right and emits native code as it goes.

Figma was one of the first to make the switch. Their C++ design tool had been compiled to asm.js; moving to WebAssembly cut load time by more than 3x. The bottleneck had never been execution speed — it was the parse tax. Wasm binaries decode more than 20x faster than equivalent JavaScript parses, because there is nothing to parse. The binary format is the instruction stream.

The Parse Tax

asm.js shipped as JavaScript text — megabytes of a = a|0 that browsers parsed character by character before compiling a single instruction. A Unity game took 19 MB of asm.js source. Wasm’s binary format for the same game: 6.3 MB — 3x smaller — and it validated in a single linear pass. No lexer, no parser, no AST. Firefox 58 could stream-compile Wasm at 30–60 MB/sec on desktop — faster than the network could deliver it.

Four Promises

WebAssembly’s design was guided by four goals that became its identity:

Fast

Near-native speed through typed instructions and AOT compilation. No warmup, no speculative optimization, no deopt. The compiler knows every type before it emits a single instruction.

Safe

Sandboxed linear memory, validated before execution. A module cannot read the browser’s memory. It gets a flat byte array and every access is bounds-checked. Validation proves safety before a single instruction runs.

Portable

Architecture-independent bytecode. The same .wasm file runs on x86 laptops, ARM phones, and RISC-V boards. Compile once, run anywhere the web reaches.

Compact

Binary encoding with LEB128 variable-length integers and single-byte opcodes. 20–30% smaller than asm.js uncompressed, 10–20% smaller after gzip. But the real win is decode speed: an order of magnitude faster than parsing text.

In December 2019, the W3C made it official: WebAssembly became a W3C Recommendation, the fourth language of the web alongside HTML, CSS, and JavaScript. Today it runs in 97% of browsers worldwide.

But knowing what Wasm promises is different from knowing how it works. The binary encodes instructions for a machine that doesn’t exist on any real chip — a stack machine with zero registers, structured control flow, and a type system that makes safety provable. That machine is where we go next.

02 — the stack machine

Every processor you have ever used thinks in registers. x86 has eax, ebx, ecx — named slots that instructions read from and write to. ARM has r0 through r15. RISC-V has 32 of them. WebAssembly has zero. It uses a stack: every instruction pops its operands from the top, computes, and pushes the result back. There are no names. There are no destinations. Position is identity.

This sounds primitive. It is also the reason WebAssembly can be validated in a single pass, compiled as fast as it downloads, and proven safe before a single instruction executes.

How It Thinks

Consider the expression (3 + 7) * 2. On a register machine, you would write something like mov eax, 3; add eax, 7; imul eax, 2 — explicitly naming which register holds each intermediate value. On the Wasm stack machine, you simply push operands and let each operator consume what it needs:

Stack trace: (3 + 7) × 2 
  Instruction        Stack (top →)      Effect
  ───────────        ──────────────      ──────
  i32.const 3        [ 3 ]               push 3
  i32.const 7        [ 3, 7 ]            push 7
  i32.add            [ 10 ]              pop 3 and 7, push 10
  i32.const 2        [ 10, 2 ]           push 2
  i32.mul            [ 20 ]              pop 10 and 2, push 20

Five instructions. No register names. No mov to shuttle values between slots. The i32.add instruction does not say “add eax and ebx.” It says “pop two values, add them, push the result.” The compiler that emits this code never has to solve a register allocation problem, and the engine that compiles it to native code can map stack positions to real registers however it likes.

Registers vs. Stack

The natural question: if real CPUs use registers, why does Wasm use a stack? Three reasons.

Compactness. A register instruction needs to encode source and destination registers in every opcode: add eax, ebx encodes three things (operation, source, destination). A stack instruction encodes one: i32.add. The operands are implicit — they are wherever the top of the stack is. This is why Wasm bytecode is denser than register-based bytecodes like LLVM IR or Dalvik — fewer bits per instruction, smaller binaries over the wire.

Validation. On a register machine, the validator has to track which registers hold which types at every point in the program — a data-flow analysis that can be expensive. On a stack machine, the validator maintains a single type stack. At i32.add, it pops two types: if both are i32, the result is i32. If not, the module is rejected. One pass, linear time, no fixed-point iteration.

No overhead in practice. The browser’s compiler maps stack positions to real CPU registers during compilation anyway. V8’s Liftoff compiler does this in a single pass — it maintains a virtual stack that records which register (or spill slot) holds each Wasm stack value. The stack encoding costs nothing at runtime; it only exists in the binary format.

x86 register machine

mov  eax, 3       ; eax = 3
mov  ebx, 7       ; ebx = 7
add  eax, ebx     ; eax = eax + ebx = 10
mov  ecx, 2       ; ecx = 2
imul eax, ecx     ; eax = eax * ecx = 20

Every instruction names its operands. 5 instructions, 3 registers, each encoding source + destination.

Wasm stack machine

i32.const 3      ;; push 3
i32.const 7      ;; push 7
i32.add           ;; pop 2, push sum
i32.const 2      ;; push 2
i32.mul           ;; pop 2, push product

No operand names. Position on the stack IS the identity. 5 instructions, 0 registers, each encoding only the operation.

Structured Control Flow

Native code can jump anywhere. A goto in C can target any label. A jmp in x86 can land on any address. This flexibility is what makes return-oriented programming (ROP) attacks possible — attackers chain together fragments of existing code by hijacking jump targets.

Wasm cannot jump anywhere. It has no goto. Instead, control flow is structured — built from nested blocks that the validator can reason about statically:

Wasm has exactly four control flow constructs. No goto. No arbitrary jumps.

ConstructBehavior
block...endA labeled scope. br jumps forward to the end.
loop...endA labeled scope. br jumps backward to the top.
if...else...endPops an i32. Nonzero → then branch, zero → else branch.
br N / br_if NBranch to the Nth enclosing block. Not an address — a nesting depth.

Every branch target is a structurally enclosing scope. The validator proves, before execution, that control never reaches an unexpected instruction.

Cint find_first_even(int* arr, int len) {
  for (int i = 0; i < len; i++) {
    if (arr[i] % 2 == 0)
      return arr[i];    // early exit
  }
  return -1;
}
WAT(func $find_first_even (param $arr i32) (param $len i32) (result i32)
  (local $i i32)
  (block $exit                   ;; br 1 jumps here ↓
    (loop $loop                  ;; br 0 jumps here ↑
      local.get $i
      local.get $len
      i32.ge_s
      br_if $exit              ;; i >= len? break

      local.get $arr
      local.get $i
      i32.const 4
      i32.mul
      i32.add
      i32.load                 ;; arr[i]
      i32.const 2
      i32.rem_u
      i32.eqz
      (if (then
        ;; found even — return it
        local.get $arr
        local.get $i
        i32.const 4
        i32.mul
        i32.add
        i32.load
        return
      ))

      local.get $i
      i32.const 1
      i32.add
      local.set $i
      br $loop                 ;; continue
    )
  )
  i32.const -1                  ;; not found
)

The key insight: br 1 does not mean “jump to address 0x4A2.” It means “jump to the end of the 1st enclosing block.” The target is a nesting depth, not an address. You cannot construct a branch that escapes its enclosing scope, which means you cannot construct an ROP gadget chain. The control flow graph is a tree of nested scopes, never a spaghetti of arbitrary jumps.

Why No Goto?

Native code’s arbitrary jumps enable ROP attacks — attackers chain together fragments of existing code by hijacking return addresses. Hardware mitigations like Intel CET and ARM BTI add 5–15% overhead to enforce control flow integrity. Wasm gets the same property for free. Structured control flow means the validator can prove, in a single linear pass, that every branch lands on a valid target. The security is baked into the instruction set, not bolted on afterward.

The Validation Pass

Every Wasm module is validated before a single instruction executes. The validator walks through the bytecode once, maintaining a type stack — a parallel stack that tracks not values but their types. At each instruction, it checks: does this operator expect what is currently on top?

Type checking in one pass 
  Instruction        Type Stack       Check
  ───────────        ──────────       ─────
  i32.const 3        [i32]            push i32 — ok
  i32.const 7        [i32, i32]       push i32 — ok
  i32.add            [i32]            expect [i32, i32], got [i32, i32] — ok
  i32.const 2        [i32, i32]       push i32 — ok
  i32.mul            [i32]            expect [i32, i32], got [i32, i32] — ok

  — but what if someone injects a bad instruction? —

  i32.const 3        [i32]            push i32 — ok
  f64.const 2.0      [i32, f64]       push f64 — ok
  i32.add            [ERROR]          expect [i32, i32], got [i32, f64] — REJECT

The module is rejected before it ever runs. Not at the first bad instruction — before any instruction. The entire module either passes validation or is discarded as malformed. This is a security property that native code does not have and JavaScript cannot provide. A Wasm module that passes validation will never execute an undefined operation, never access an illegal type, never misalign the stack. The type system is the sandbox.

We know how the machine thinks — it pushes, pops, and branches through nested scopes. But what does the file that carries these instructions actually look like? The next chapter cracks open a .wasm binary byte by byte.

03 — the binary format

A .wasm file is not source code. It is not an archive. It is a sequence of typed sections laid out in a fixed order, each one answering a single question about the module it describes. The format was designed with one obsession: a browser should be able to start compiling the first function while the last function is still downloading. No backtracking, no symbol resolution phase, no second pass. The binary is the compilation input.

Every .wasm file begins the same way — four magic bytes followed by a version number:

The first 8 bytes 
  00 61 73 6D   01 00 00 00
  ─┬──────────   ─┬──────────
   │              │
   \0asm          version 1

  The magic number spells "asm" in ASCII,
  preceded by a null byte. Every valid
  .wasm file starts with these 8 bytes.

After the header, sections appear in numerical order. Each section starts with a one-byte ID, followed by its length in bytes (encoded as a LEB128 integer), then its payload. The engine reads the ID, reads the length, and either processes the section or skips it entirely — jumping ahead by exactly length bytes to the next section.

The Twelve Sections

A Wasm module can contain up to twelve standard section types, plus any number of custom sections. They must appear in this order:

IDSectionWhat it answers
1TypeWhat function signatures exist? (i32, i32) → i32
2ImportWhat does the module need from the host?
3FunctionWhich function uses which type signature?
4TableWhat indirect call tables exist?
5MemoryHow much linear memory? (initial size, max)
6GlobalWhat mutable/immutable globals exist?
7ExportWhat does the module expose to the host?
8StartWhich function runs automatically on load?
9ElementWhat goes into the indirect call tables?
10CodeThe actual function bodies — the bytecode.
11DataWhat goes into linear memory at startup?
12DataCountHow many data segments? (for validation)

The ordering rule is strict: section 1 before 2, 2 before 3, and so on. Custom sections (ID 0) can appear anywhere between standard sections — they carry names, debug info, or metadata that engines may ignore. This fixed ordering is what makes streaming compilation possible. By the time the engine reaches the Code section, it already knows every type signature, every import, and every memory declaration. It has everything it needs to compile function bodies in parallel.

The Type section defines the signature once; the Function section says “function 0 uses type 0”; the Export section maps the name "add" to function index 0. No redundancy, no ambiguity.

LEB128: The Variable-Length Trick

Every integer in a .wasm file — section lengths, type indices, local counts, branch depths — is encoded as LEB128 (Little-Endian Base 128). The idea is simple: use 7 bits of each byte for data, and the high bit as a continuation flag. Small numbers use fewer bytes:

LEB128 encoding 
  Value     Binary                    LEB128 bytes     Fixed i32
  ─────     ──────                    ────────────     ─────────
  3         00000011                   03               03 00 00 00
  127       1111111                   7F               7F 00 00 00
  128       10000000                  80 01            80 00 00 00
  624485    0100110 0001000 1100101  E5 8E 26         E5 87 09 00

  The high bit of each byte: 1 = more bytes follow, 0 = this is the last byte.
  Data bits are read 7 at a time, least significant group first.

The number 3 costs one byte instead of four. The number 128 costs two bytes instead of four. In a format where most integers are small — type indices, local counts, branch depths are almost always under 128 — this adds up. LEB128 is the same encoding that DWARF debug info uses, and it is a significant part of why Wasm binaries are 10–20% smaller than equivalent native executables.

Streaming Compilation

The section-based format is not just organized — it is organized for speed. Because every section header declares its length upfront, the engine can make decisions before it has seen all the data:

Compilation overlaps with download 
  Network       ─── bytes arriving ──────────────────────────────────▶
                ┌────────┬──────┬──────────────────────────────────┐
                │ header │ type │          code section             │
                │ + meta │ func │  fn 0  │  fn 1  │  fn 2  │ ...  │
                └────────┴──────┴────┬───┴────┬───┴────┬───┴──────┘
                                     │        │        │
  Compile                             ▼        ▼        ▼
  threads                          ┌──────┐ ┌──────┐ ┌──────┐
                                   │ fn 0 │ │ fn 1 │ │ fn 2 │
                                   │ →x86 │ │ →x86 │ │ →x86 │
                                   └──────┘ └──────┘ └──────┘

  By the time fn 2 arrives, fn 0 is already compiled.
  Firefox achieves 30–60 MB/sec compile throughput on desktop —
  faster than most network connections can deliver the bytes.

Each function body in the Code section declares its own length, so multiple compilation threads can split the work without waiting for each other. Firefox 58 introduced this streaming compilation in January 2018, achieving 30–60 MB/sec on desktop and roughly 8 MB/sec on mobile. That means for most internet connections, compilation finishes before the download does — the module is ready to execute the instant the last byte arrives.

Compile faster than you download

On a 20 Mbps connection, you download at ~2.5 MB/sec. Firefox compiles Wasm at 30–60 MB/sec. The compiler is literally waiting for the network. Compilation is no longer on the critical path — download speed is. This is why the binary format’s compactness matters twice: smaller files download faster and compile faster.

The binary format turns out to be a masterclass in cooperative design — every decision (fixed section order, length-prefixed payloads, LEB128 integers, independent function bodies) serves the same goal: get from first byte to first instruction as fast as physics allows. But the binary only tells the engine what code exists. Where that code stores its data — every pointer, every string, every heap allocation — lives in a single flat byte array called linear memory. That is the next layer down.

04 — linear memory

Wasm programs do not have a heap allocator, a virtual address space, or memory protection in the OS sense. They have a flat byte array. One contiguous block of bytes, starting at offset zero, addressable with 32-bit indices. Every pointer in a compiled C program — every malloc result, every stack variable, every string literal — is just an integer index into this array. The i32.load instruction reads four bytes at a given offset. i32.store writes them. That is the entire memory model.

This simplicity is both the performance story and the security story. The engine does not need to emulate page tables or translation lookaside buffers. A load is an array access. And because the array has a known size, the engine can prove that every access is in bounds — in some cases without executing a single check instruction.

Pages and Growing

Linear memory is declared in units of pages, each exactly 65,536 bytes (64 KB). A module declares an initial size and optionally a maximum:

WAT(memory 1 10)  ;; initial: 1 page (64 KB), max: 10 pages (640 KB)

At runtime, a module can request more memory via memory.grow:

WATi32.const 3        ;; request 3 additional pages
memory.grow         ;; returns old size in pages (or -1 on failure)

The new pages are zero-initialized. The existing data is untouched. The base address does not change — the array simply gets longer. If the request would exceed the declared maximum or the engine’s resource limits, memory.grow returns -1 and the memory stays unchanged.

The Guard Page Trick

Every memory access in Wasm must be bounds-checked — an out-of-bounds load or store must trap, not silently read garbage. The naive approach is to insert a comparison before every access: if (offset + size > memory.length) trap(). That is a branch on every load and store, and it is expensive.

Every major engine uses a better approach: virtual memory guard pages. The engine asks the operating system to reserve a large virtual address range — typically 4 GB for a 32-bit Wasm memory plus some extra. The actual linear memory occupies the first N pages (backed by physical memory). Everything beyond that is left unmapped:

Virtual memory layout 
  0                         N pages                        4 GB
  ├─────────────────────────┼────────────────────────────────┤
  │    real memory (RW)      │    unmapped guard region        │
  │    linear memory data    │    any access = hardware trap   │
  ├─────────────────────────┼────────────────────────────────┤
  ▲                         ▲                                ▲
  base pointer              current size             max virtual reservation

  Out-of-bounds access hits unmapped memory → CPU raises fault
  Runtime catches the fault → Wasm trap
  Cost in the common case: zero instructions

An out-of-bounds access hits the unmapped region, the CPU raises a hardware fault (SIGSEGV on Linux, access violation on Windows), and the Wasm runtime’s signal handler catches it and converts it to a Wasm trap. In the common case — the access is in bounds — there is no check at all. No branch, no comparison, no overhead. The hardware does bounds checking for free.

V8, SpiderMonkey, and Wasmtime all use this technique. It is why Wasm memory access is as fast as native pointer dereferencing, despite being sandboxed.

Memory Layout

When Emscripten compiles a C program to Wasm, it lays out the linear memory like a simplified version of a native process:

Emscripten linear memory layout 
  Offset 0                                               memory size
  ├──────────┬───────────────┬──────────────┬─────────────────┤
  │  null    │  static data   │   heap →      │    ← stack     │
  │  guard   │  (globals,     │   (malloc,     │    (locals,     │
  │  page    │   strings)     │    grows up)   │     grows down)  │
  ├──────────┴───────────────┴──────────────┴─────────────────┤

  ■ Offset 0 is intentionally wasted (null pointer guard)
  ■ Static data: string literals, global variables
  ■ Heap: malloc/free, grows upward via memory.grow
  ■ Stack: function locals, grows downward from top

Offset 0 is left as a null pointer guard — accessing address 0 is always a trap. Static data (strings, global constants) sits just above that. The heap grows upward via memory.grow. The shadow stack lives at the top of memory and grows downward — this is not the Wasm operand stack from Chapter 2, but the C-style call stack that holds local variables too large for Wasm locals.

Every int* in the C source becomes an i32 in the Wasm output. Dereferencing that pointer becomes i32.load with the pointer value as the offset. Pointer arithmetic becomes i32.add. The compiler has flattened the entire abstraction of pointers and addresses into indices into a flat array. There is nothing else.

Why 64 KB Pages?

Wasm pages are exactly 65,536 bytes. This is not arbitrary. Windows’ VirtualAlloc allocates in 64 KB granularity. ARM architectures optionally support 64 KB pages. And x86 uses 4 KB pages, so 64 KB is always an exact multiple. The 64 KB page size is the lowest common denominator that lets every OS map Wasm memory without waste or misalignment — one memory.grow call maps cleanly to one or more OS-level page operations on every platform.

The Security Boundary

A Wasm module cannot read or write outside its linear memory. Period. There is no system call to map new address ranges. There is no way to dereference a pointer into the browser’s own memory. A buffer overflow in a Wasm module corrupts the module’s own data — it cannot escape the sandbox.

This is fundamentally different from native code, where a buffer overflow can overwrite return addresses on the stack (enabling ROP attacks), corrupt adjacent heap allocations, or read sensitive data from other processes. In Wasm, the call stack (return addresses) is separate from linear memory and completely inaccessible to the program. The shadow stack in linear memory holds copies of local variables, but the actual return flow is managed by the engine and cannot be touched.

For 32-bit Wasm (the current standard), linear memory is capped at 4 GB — the maximum addressable by i32 indices. The Memory64 proposal extends this to i64 addressing for applications that need more. But the principle stays the same: one flat array, one boundary, no escape.

We have the execution model (stack machine), the file format (binary sections), and the data store (linear memory). What remains is the type system that ties them together — four numeric types, a handful of reference types, and a set of rules that make every operation provably correct before it runs.

05 — types and instructions

Wasm has four numeric types. Not four hundred. Not forty. Four: i32, i64, f32, f64. No strings. No structs. No booleans — a boolean is just an i32 where 0 means false and anything else means true. This minimalism is not a limitation. It is the reason every operation is type-checkable in a single linear pass, every instruction has exactly one defined behavior on every input, and the entire machine is provably safe before it runs.

No Undefined Behavior

In C, signed integer overflow is undefined behavior. The compiler can assume it never happens, and if it does, anything goes — nasal demons, time travel, whatever the optimizer felt like. In Wasm, every operation has exactly one defined result for every input:

This is not C’s “trust the programmer.” This is “trust nobody, verify everything.” A Wasm module that passes validation will never produce an undefined result. The type system and deterministic semantics together mean every module either validates and runs correctly, or is rejected before a single instruction executes. There is no third state.

The type system is small on purpose. Four numeric types, a handful of reference types, deterministic semantics for every operation. This is what makes single-pass validation possible, what makes the sandbox provable, and what makes compilation to native code straightforward — every Wasm instruction maps cleanly to one or two native instructions. But nobody writes these instructions by hand. First you need a toolchain that turns your C, Rust, or Go into a .wasm binary. The tools that make that happen are wilder than you would expect.

06 — the toolchain

Wasm is a compilation target, and three toolchains dominate the landscape. Each makes a different trade-off between compatibility and minimalism, and the choice determines everything from your binary size to your deployment complexity. The differences start at the compiler and ripple all the way to the user’s browser.

The Three Paths

Emscripten compiles C and C++ to Wasm by routing through LLVM’s wasm32-unknown-emscripten target. It ships a full POSIX compatibility layer — a virtual filesystem, pthreads emulation via Web Workers and SharedArrayBuffer, OpenGL-to-WebGL translation, even a partial libc. This is how 30-year-old codebases like AutoCAD and Google Earth run in the browser unmodified. The cost: the runtime shim can double your binary size. A minimal “hello world” compiles to roughly 15 KB of Wasm plus a JavaScript glue file; a full 3D engine can run into megabytes.

Rust + wasm-pack takes the opposite approach. Rust’s wasm32-unknown-unknown target produces lean binaries with no runtime, no GC, and no POSIX shim. The wasm-bindgen tool generates type-safe FFI bindings between Rust and JavaScript automatically. A Rust “hello world” (just a function that adds two numbers) compiles to under 200 bytes of Wasm. Rust’s ownership model means no garbage collector overhead, and its #[no_std] mode eliminates even the standard library if you don’t need it.

Go compiles to Wasm too, but it ships its entire goroutine scheduler and garbage collector. A Go “hello world” is over 2 MB. TinyGo, a Go variant targeting embedded and Wasm, reduces this dramatically — often to 50–100 KB — but at the cost of incomplete standard library support.

The LLVM Backend

Behind both Emscripten and Rust sits LLVM’s wasm32 backend, which treats WebAssembly as just another instruction set architecture — like x86 or ARM. The same optimization passes that make native code fast (dead code elimination, loop unrolling, constant folding, inlining) run on the LLVM IR before the wasm32 code generator emits the final bytecode. The code generator maps LLVM’s virtual registers to Wasm locals and the value stack, translates phi nodes to structured control flow, and produces a valid .wasm binary.

Emscripten originally used a custom LLVM fork called “fastcomp.” In 2019, it switched to LLVM’s upstream Wasm backend, which produced 6–15% smaller binaries and linked 7x faster. Fastcomp was fully removed in Emscripten 2.0.

wasm-opt: The Second Compiler

Binaryen’s wasm-opt is a Wasm-to-Wasm optimizer that runs after your compiler. It performs passes that LLVM misses because LLVM does not understand Wasm-specific patterns: dead local elimination, block merging, stack expression folding, and code motion across block boundaries. Running wasm-opt -O3 on a Rust-produced binary typically shaves off 10–20%. Emscripten runs it automatically; if you use any other toolchain, you should run it yourself.

The WAT Round-Trip

WAT is not an intermediate language or an approximation. It is a character-for-character text rendering of the binary. wat2wasm converts text to binary; wasm2wat converts binary back to text. The round-trip is lossless. This means you can disassemble any .wasm file, read the instructions in human form, modify them in a text editor, and reassemble. The toolchain that produces Wasm and the format you read Wasm in are the same language at different zoom levels.

The toolchain gives you a .wasm binary. But the best way to understand what the tools produce — and the only way to beat them — is to write it yourself. In the next chapter, we open a text editor, type WAT by hand, and discover that the gap between human and machine is exactly one text format wide.

07 — handwriting wasm

Writing WAT directly instead of compiling from C or Rust sounds like writing x86 assembly by choice — a niche hobby for masochists. But there is a practical reason people do it: LLVM does not always produce optimal Wasm. The compiler introduces unnecessary locals, emits redundant stack manipulation, and sometimes misses patterns that a human can spot in thirty seconds. For tight inner loops where every instruction counts, handwritten WAT wins.

What the Compiler Produces

Consider a simple function that sums an array of integers. Here is the C:

Cint sum(int* arr, int len) {
    int total = 0;
    for (int i = 0; i < len; i++)
        total += arr[i];
    return total;
}

Emscripten with -O3 produces reasonable output, but it introduces extra locals and stack shuffling that a human would avoid. Here is a simplified view of what typical compiler output looks like versus what you can write by hand:

;; Typical LLVM output (simplified)
(func $sum (param $arr i32) (param $len i32) (result i32)
  (local $i i32) (local $total i32) (local $tmp i32)
  i32.const 0
  local.set $total            ;; total = 0
  i32.const 0
  local.set $i                ;; i = 0
  block $exit
    loop $loop
      local.get $i
      local.get $len
      i32.ge_s
      br_if $exit              ;; if i >= len, break
      local.get $total
      local.get $arr
      local.get $i
      i32.const 4
      i32.mul                   ;; i * 4 (byte offset)
      i32.add                   ;; arr + i*4
      i32.load                  ;; load arr[i]
      i32.add
      local.set $total          ;; total += arr[i]
      local.get $i
      i32.const 1
      i32.add
      local.set $i              ;; i++
      br $loop
    end
  end
  local.get $total)

;; 21 instructions, 3 locals (+ 2 params)
;; Hand-optimized: use pointer arithmetic, no multiply
(func $sum (param $ptr i32) (param $len i32) (result i32)
  (local $total i32) (local $end i32)
  ;; compute end pointer: ptr + len * 4
  local.get $ptr
  local.get $len
  i32.const 2
  i32.shl                     ;; len << 2 = len * 4
  i32.add
  local.set $end              ;; end = ptr + len*4
  block $exit
    loop $loop
      local.get $ptr
      local.get $end
      i32.ge_u
      br_if $exit              ;; if ptr >= end, break
      local.get $total
      local.get $ptr
      i32.load                  ;; load *ptr
      i32.add
      local.set $total          ;; total += *ptr
      local.get $ptr
      i32.const 4
      i32.add
      local.set $ptr            ;; ptr += 4
      br $loop
    end
  end
  local.get $total)

;; 18 instructions, 2 locals (+ 2 params)
;; Eliminated: i*4 multiply per iteration
;; Replaced: index multiply with pointer increment

The hand-optimized version does three things differently. First, it replaces the per-iteration i32.mul with a pointer increment — advancing by 4 bytes each loop instead of recomputing i * 4. Second, it uses i32.shl (shift left by 2) instead of i32.mul for the initial end-pointer calculation. Third, it eliminates the separate loop counter $i entirely, using the pointer itself as the loop variable. Three fewer instructions per iteration in a hot loop.

The S-Expression Syntax

WAT supports two equivalent syntaxes. The flat form (used in the examples above) lists instructions sequentially, matching how the stack machine actually executes. The S-expression form nests operations inside parentheses, making the data flow explicit:

Flat Form

local.get $a
local.get $b
i32.add
local.get $c
i32.mul

S-Expression Form

(i32.mul
  (i32.add
    (local.get $a)
    (local.get $b))
  (local.get $c))

Both produce identical bytecode. The S-expression form reads like a function call tree; the flat form reads like assembly. Most hand-writers use the flat form because it maps directly to what the engine sees. Both are valid input to wat2wasm.

The Disassembly Loop

The practical workflow for optimization is not writing WAT from scratch. It is a loop:

The optimization cycle 
  1. Compile         emcc -O3 hot_loop.c -o hot_loop.wasm
  2. Disassemble     wasm2wat hot_loop.wasm -o hot_loop.wat
  3. Inspect         Read the WAT. Find redundant locals, missed patterns.
  4. Edit            Modify the .wat file by hand.
  5. Reassemble      wat2wasm hot_loop.wat -o hot_loop_opt.wasm
  6. Benchmark       Compare hot_loop.wasm vs hot_loop_opt.wasm
  7. Repeat          Until diminishing returns.

  Tools: wabt (WebAssembly Binary Toolkit) provides wat2wasm, wasm2wat
         wasm-opt (Binaryen) for automated Wasm→Wasm optimization

The key insight is that you are not competing with the compiler on the whole program. You are competing on one function — the innermost loop that runs a million times per frame. The compiler wins everywhere else. But in that one hot path, a human who understands the stack machine can sometimes shave 10–20% off the instruction count by eliminating the overhead that LLVM’s register-to-stack lowering introduces.

When to Hand-Write

Hand-writing WAT makes sense in exactly two situations: SIMD inner loops where you need specific instruction sequences that the autovectorizer misses, and performance-critical paths where profiling shows the compiler introduced measurable overhead. In every other case, trust the compiler and spend your time on the algorithm. The maintainability cost of handwritten bytecode is real — code that nobody else can read is code that nobody else can fix.

Whether you write Wasm by hand or let a compiler do it, the binary eventually needs to reach JavaScript. The bridge between the two worlds — how functions are imported and exported, how strings cross the boundary, and what it costs to cross — is where the next chapter picks up.

08 — the javascript bridge

WebAssembly does not run alone. In the browser, it runs inside a JavaScript host that loads the module, provides imports, and calls exports. The bridge between the two is clean but constrained: you can pass numbers back and forth instantly, but you cannot pass a string, an object, or a closure. Everything that is not a number must be serialized into linear memory, passed as a pointer, and deserialized on the other side.

Loading a Module

The modern way to load Wasm is WebAssembly.instantiateStreaming, which compiles the module while it downloads — the streaming compilation from Chapter 3 in action:

JavaScriptconst { instance } = await WebAssembly.instantiateStreaming(
  fetch('module.wasm'),
  {
    env: {
      log: (x) => console.log('Wasm says:', x),  // import: JS function Wasm can call
    }
  }
);

const result = instance.exports.add(3, 7);  // export: Wasm function JS can call
console.log(result);  // 10

The second argument is the import object — a JavaScript object whose shape must match the module’s import declarations exactly. If the module declares (import "env" "log" (func (param i32))), the import object must have env.log as a function that accepts one argument. Mismatches cause a LinkError at instantiation.

Passing Strings

This is where the bridge gets uncomfortable. Wasm has no string type. A “string” in Wasm is a sequence of bytes in linear memory, addressed by a pointer (i32 offset) and a length. To pass a string from JavaScript to Wasm, you must:

JavaScript// 1. Get access to Wasm's linear memory
const memory = instance.exports.memory;
const buffer = new Uint8Array(memory.buffer);

// 2. Encode the string as UTF-8
const encoder = new TextEncoder();
const encoded = encoder.encode("hello, wasm");

// 3. Allocate space in Wasm memory (call a Wasm-side allocator)
const ptr = instance.exports.alloc(encoded.length);

// 4. Copy bytes into linear memory
buffer.set(encoded, ptr);

// 5. Call the Wasm function with pointer + length
instance.exports.process_string(ptr, encoded.length);

// 6. Free the allocation when done
instance.exports.dealloc(ptr, encoded.length);

Coming back is the same process in reverse: the Wasm function writes bytes into linear memory, returns a pointer and length, and JavaScript reads those bytes with TextDecoder. It is manual, verbose, and exactly how C programs pass strings to each other across process boundaries. Tools like wasm-bindgen (Rust) and Emscripten’s ccall/cwrap automate this, but underneath, the bytes are always being copied into and out of the flat array.

The Cost of Crossing

A JavaScript-to-Wasm function call is fast — typically under 10 nanoseconds in V8 and SpiderMonkey, comparable to a native function call through a vtable. For bulk computation, the boundary cost is negligible: call once, process a million items, return the result.

The problem is frequency. If your architecture requires a JS↔Wasm call for every pixel, every particle, or every DOM update, the overhead adds up. A million calls at 10 ns each is 10 milliseconds — a meaningful chunk of a 16 ms frame budget. The rule of thumb: keep hot loops entirely in Wasm, batch data in linear memory, and cross the bridge as infrequently as possible.

The Bridge Tax

The JS↔Wasm boundary costs almost nothing per call. But “almost nothing” times a million is something. Design your API surface to minimize crossings: pass a pointer to an array, not individual elements. Return a struct offset, not individual fields. Think of the bridge like a network call — batch your requests.

We know how to write Wasm, how to connect it to JavaScript, and how to pass data across the boundary. But what happens after instantiateStreaming returns? The browser has a .wasm binary and needs to turn it into machine code that runs on a real CPU. The next chapter opens the hood on V8, SpiderMonkey, and the compilation pipelines that make Wasm fast.

09 — how browsers actually run it

A .wasm binary is not executable. No CPU on earth understands Wasm opcodes natively — they must be compiled to real machine code first. The question is when and how well. Too slow and the user stares at a blank screen. Too aggressive and the compiler spends more time optimizing than the program spends running. Every major browser has converged on the same answer: compile twice.

V8: Liftoff and TurboFan

Chrome’s V8 engine uses a two-tier compilation strategy. The first tier, Liftoff, is a baseline compiler that generates native code in a single forward pass over the Wasm bytecode. It maintains a virtual stack that maps each Wasm stack position to a CPU register or spill slot, and emits machine instructions as it reads each opcode. No intermediate representation. No optimization passes. The goal is speed: Liftoff compiles Wasm roughly 10x faster than the optimizing tier, getting code running within milliseconds of the module arriving.

The second tier, TurboFan, is V8’s optimizing compiler. It builds a graph-based intermediate representation (a “sea of nodes”), runs sophisticated optimization passes — instruction selection, register allocation, dead code elimination, loop-invariant code motion — and produces code that runs 2–4x faster than Liftoff output. TurboFan compiles in background threads while Liftoff code is already running. When an optimized version of a function is ready, V8 swaps it in transparently.

SpiderMonkey: Baseline and Cranelift

Firefox’s SpiderMonkey engine uses a similar two-tier approach. The baseline compiler (sometimes called Rabaldr) is a fast single-pass compiler comparable to Liftoff. The optimizing tier was historically Ion, but SpiderMonkey has been transitioning to Cranelift — the same compiler backend used by the Wasmtime runtime outside the browser.

Cranelift uses an SSA-based intermediate representation and a novel optimization technique called e-graphs (equality saturation). Instead of applying optimization rules one at a time and hoping for the best order, e-graphs represent all equivalent forms of an expression simultaneously and extract the optimal version at the end. This produces better code than traditional pass-based optimizers for certain patterns, particularly constant folding and algebraic simplification.

Wasmtime and Server-Side Compilation

Outside the browser, runtimes like Wasmtime, Wasmer, and WasmEdge compile Wasm for server-side execution. Wasmtime uses Cranelift exclusively. Because server workloads tolerate higher startup latency than browsers, Wasmtime can afford to run Cranelift’s full optimization pipeline upfront — no need for a baseline tier. The trade-off is different: startup time measured in tens of milliseconds is acceptable when the module will run for hours.

Wasmer offers a choice: the Singlepass compiler (like Liftoff — fast baseline, decent code) or the LLVM backend (the same LLVM used by Emscripten, producing the highest-quality code at the cost of the slowest compilation). For short-lived serverless functions, Singlepass wins. For long-running compute, LLVM wins.

20x Faster Than JavaScript

Figma measured that their Wasm module decoded and compiled more than 20x faster than the equivalent JavaScript could be parsed. This is not a runtime benchmark — it is a startup benchmark. JavaScript must be lexed into tokens, parsed into an AST, and then compiled. Wasm skips all of that: the binary format is pre-structured, pre-typed, and can be validated and compiled in a single linear pass. The format is the parse tree.

Lazy and Eager Compilation

Not every function in a module needs to be compiled immediately. V8 uses a combination of eager and lazy compilation strategies. Small modules are compiled eagerly — every function gets Liftoff-compiled before execution starts. Large modules use lazy compilation: functions are compiled on their first call, amortizing the startup cost. The validation pass still runs eagerly on the entire module (safety first), but code generation can be deferred.

Firefox’s streaming compilation takes a different approach: it compiles functions in parallel as their bytes arrive over the network, spreading the work across multiple CPU cores. On a 4-core machine, four functions can be compiled simultaneously. This is why Firefox can compile Wasm at 30–60 MB/sec — the throughput scales with core count.

All of this machinery exists to solve one problem: turn a portable bytecode into native instructions as fast as possible without sacrificing the quality of the generated code. But there is one area where the compilation pipeline matters more than anywhere else — SIMD. When you pack four integers into a single 128-bit register and operate on all four simultaneously, the quality of the compiler’s instruction selection determines whether you get a 2x speedup or an 8x speedup.

10 — simd: the speed multiplier

Wasm’s core instruction set processes one value at a time. i32.add adds two integers. f64.mul multiplies two doubles. For most code, this is fine. But for image processing, audio DSP, physics simulations, and machine learning inference — workloads where you apply the same operation to thousands of values — doing them one by one leaves 75% of the CPU’s computational width on the table.

SIMD (Single Instruction, Multiple Data) changes the equation. The Wasm SIMD proposal adds a v128 type — a 128-bit register that holds four i32s, four f32s, two i64s, or two f64s simultaneously. One instruction operates on all lanes at once:

The Wasm SIMD spec adds over 200 instructions covering integer and floating-point arithmetic, comparisons, shuffles, conversions, and lane extraction/insertion. On the hardware side, these map to SSE/AVX on x86 and NEON on ARM — the same vector units that native code uses.

Writing SIMD Code

You can write SIMD in three ways. Autovectorization: write a normal C loop and let LLVM figure out the SIMD — works for simple patterns but misses complex ones. Intrinsics: use wasm_simd128.h to call SIMD operations directly from C — full control, portable across Wasm targets. Hand-written WAT: write the SIMD instructions directly in WAT for maximum control.

C with SIMD intrinsics#include <wasm_simd128.h>

void vec_add(float* a, float* b, float* out, int n) {
  for (int i = 0; i < n; i += 4) {
    v128_t va = wasm_v128_load(&a[i]);
    v128_t vb = wasm_v128_load(&b[i]);
    v128_t vc = wasm_f32x4_add(va, vb);
    wasm_v128_store(&out[i], vc);
  }
}

Each iteration processes four floats at once. The wasm_v128_load loads 128 bits (4 floats) from memory into a v128 register. wasm_f32x4_add adds all four lanes in parallel. wasm_v128_store writes the result back. A loop that would have taken N iterations now takes N/4.

Relaxed SIMD

Standard Wasm SIMD guarantees deterministic results across all hardware. f32x4.add produces exactly the same bits on x86 and ARM. But this determinism comes at a cost: some operations that hardware can do cheaply (fused multiply-add, reciprocal approximation) have no deterministic equivalent, because different architectures produce slightly different results.

Relaxed SIMD, finalized in 2024, adds a set of instructions that trade determinism for speed. f32x4.relaxed_madd performs a fused multiply-add that may or may not fuse depending on the hardware. i8x16.relaxed_swizzle uses hardware-native shuffle behavior, which differs between x86 (out-of-range lanes zero) and ARM (out-of-range lanes wrap). The results are implementation-defined but always correct within a reasonable range.

This matters for ML inference, where fused multiply-add is a critical operation and the minor non-determinism is acceptable. It also matters for codecs, where hardware-specific shuffle patterns can avoid extra mask instructions.

SIMD Lane Diagram

A v128 is not a “bigger number.” It is four numbers traveling together. When you see i32x4.add, read it as: “four independent i32 additions happening in the same cycle, in the same register.” The “x4” is not a loop count — it is the hardware width. SIMD does not make individual operations faster; it makes the bus wider.

SIMD multiplies the throughput of data-parallel code. But Wasm’s ambitions extend far beyond the browser’s compute pipeline. The next chapter follows Wasm outside the browser entirely — into operating system interfaces, language-agnostic composition, and the idea that a single module should run anywhere, with exactly the capabilities you grant it.

11 — wasi & the component model

In the browser, a Wasm module talks to the outside world through JavaScript imports. But what if there is no browser? What if the module is running on a server, in a CLI tool, or inside an embedded device? It needs a system interface — a way to read files, open sockets, get the current time. That interface is WASI: the WebAssembly System Interface.

WASI is not POSIX. It does not start with “everything is permitted” and try to restrict from there. It starts with nothing. A WASI module has zero capabilities by default — no filesystem access, no network, no environment variables, no clock. The host explicitly grants capabilities at instantiation: “you may read files under /data, you may open TCP connections to port 443, you may query the monotonic clock.” The module receives handles to these resources, not raw file descriptors. This is capability-based security — the same model that Capsicum and CloudABI pioneered, now baked into a bytecode format.

WASI 0.2 defined a set of worlds — named collections of imports and exports that describe what a module can do. The wasi:cli/command world is a command-line program that reads stdin, writes stdout, and accesses the filesystem. The wasi:http/proxy world is an HTTP handler that receives requests and produces responses. A module targets a world, and the runtime provides exactly those capabilities.

The Component Model

WASI 0.2 is built on something deeper: the Component Model, which solves a problem Wasm was never designed for — composing modules written in different languages.

A core Wasm module speaks only in numbers. Its imports and exports are functions with numeric signatures. If a Rust module wants to pass a string to a Go module, someone has to agree on the memory layout, the encoding, the ownership rules. The Component Model introduces WIT (Wasm Interface Types) — a high-level interface description language that defines types like strings, lists, records, variants, and enums:

WITpackage myapp:processor;

interface transform {
  record image {
    width: u32,
    height: u32,
    pixels: list<u8>,
  }

  resize: func(img: image, new-width: u32, new-height: u32) -> image;
}

world image-processor {
  export transform;
}

The Canonical ABI defines exactly how these high-level types are lowered to Wasm’s core types (i32, i64, linear memory) and how they are lifted back. A component is a Wasm module wrapped with WIT metadata that describes its interface in terms the runtime understands. Two components can be composed — the output of one wired to the input of another — regardless of what language each was written in. A Rust image resizer can be composed with a Python ML classifier, with the Component Model handling the type translation at the boundary.

WasmGC

Languages with garbage collectors (Java, Kotlin, Dart, Go) have traditionally compiled to Wasm by shipping their entire GC implementation inside the linear memory — a GC running inside a sandbox, unaware of the host’s GC. This is wasteful. WasmGC adds struct and array heap types managed by the host’s garbage collector, letting these languages use the browser’s GC directly.

The impact is concrete: Google ported the Google Sheets calculation engine from JavaScript to WasmGC and measured a 2x performance improvement on large spreadsheets. Kotlin/Wasm and Dart/Wasm both target WasmGC, producing smaller binaries and faster execution than the linear-memory-GC approach.

Solomon Hykes, Docker Founder

“If WASM+WASI existed in 2008, we wouldn’t have needed to create Docker. That’s how important it is. Webassembly on the server is the future of computing.” The statement is provocative, but the kernel of truth is real: WASI provides isolation (like containers), portability (like Docker images), and capability-based security (better than most container runtimes) — all at the instruction set level, not the OS level.

WASI gives Wasm a system interface. The Component Model gives it composability. WasmGC gives it efficient support for managed languages. Together, they are turning WebAssembly from a browser plugin into a universal compute substrate. The next chapter shows where that substrate is already running in production.

The Fourth Language

HTML structures content. CSS styles it. JavaScript makes it interactive. WebAssembly makes it fast. Since December 2019, Wasm has been a W3C Recommendation — an official web standard on equal footing with the other three. It runs in 97% of browsers, on every major operating system, and increasingly on edge networks and embedded devices. The fourth language of the web is also becoming the first language of the post-container cloud.
