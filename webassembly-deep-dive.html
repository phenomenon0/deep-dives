01 — your first module by hand

WebAssembly is a binary instruction format that runs in every major browser at near-native speed. It is not JavaScript. It is not native code. It is a portable bytecode — a compact, typed, sandboxed instruction set that browsers compile to machine code in a single pass. The same .wasm file runs on x86 laptops, ARM phones, and server-side runtimes without recompilation. As of 2026, 97% of browsers support it, and it is a W3C Recommendation — the fourth language of the web alongside HTML, CSS, and JavaScript.

Most developers encounter Wasm through compilers — Emscripten, Rust’s wasm-pack, or TinyGo. They never see the instructions. This article goes the other way. You are going to write WebAssembly by hand in WAT, the human-readable text format. You will learn the instruction set by using it, build a working template engine, and then see what happens when real teams — Figma, Adobe, Google — ship Wasm at scale.

Open that text editor. Not a compiler. Not a build system. A text editor. You are going to write a WebAssembly module from scratch in WAT — the human-readable text format that maps one-to-one to the binary. Every parenthesis you type has a corresponding byte in the .wasm output. By the end of this chapter you will have a working module running in the browser, and you will have typed every instruction yourself.

The Skeleton

Every WAT file starts with (module ...). Inside the module, you declare types, functions, memory, exports. Here is the smallest useful module — a function that doubles a number:

WAT(module
  (func $double (param $x i32) (result i32)
    local.get $x
    i32.const 2
    i32.mul)
  (export "double" (func $double))
)

Save this as double.wat. That is a complete, valid WebAssembly module. The func declaration defines a function named $double that takes one i32 parameter and returns one i32. The body pushes the parameter, pushes the constant 2, and multiplies. The export line makes the function visible to the host under the name "double".

Assemble and Run

Convert the text to binary with wat2wasm (from the WABT toolkit), then load it in a browser:

Shellwat2wasm double.wat -o double.wasm
JavaScriptconst { instance } = await WebAssembly.instantiateStreaming(
  fetch('double.wasm')
);
console.log(instance.exports.double(21)); // 42

That is the entire workflow: write WAT, assemble to .wasm, load in JavaScript, call the export. No Emscripten. No Rust. No build system. You wrote raw bytecode instructions in a text file and the browser compiled them to native machine code.

A Complete Module

The double function is trivial. Here is a module that shows off more of the WAT vocabulary — memory, globals, data segments, and multiple exports:

WAT(module
  ;; Declare one page of linear memory (64 KB) and export it
  (memory (export "memory") 1)

  ;; Write a greeting into memory at offset 0
  (data (i32.const 0) "Hello, Wasm!\00")

  ;; A mutable global: counts how many times greet was called
  (global $count (mut i32) (i32.const 0))

  ;; Return the pointer and length of the greeting
  (func (export "greet_ptr") (result i32)
    i32.const 0)           ;; pointer to "Hello, Wasm!"

  (func (export "greet_len") (result i32)
    i32.const 12)          ;; length of "Hello, Wasm!"

  ;; Increment the call counter and return the new count
  (func (export "tick") (result i32)
    global.get $count
    i32.const 1
    i32.add
    global.set $count       ;; count++
    global.get $count)       ;; return new count
)

Notice the patterns. The data segment writes bytes into linear memory at instantiation — you provide an offset expression and a string literal, and the runtime copies the bytes before any function runs. The global is mutable (mut i32) so it persists across calls. The export can be inlined into the func declaration — no need for a separate export statement.

The Full Loop

JavaScriptconst { instance } = await WebAssembly.instantiateStreaming(fetch('greet.wasm'));
const { memory, greet_ptr, greet_len, tick } = instance.exports;

// Read the greeting from Wasm memory
const bytes = new Uint8Array(memory.buffer, greet_ptr(), greet_len());
const greeting = new TextDecoder().decode(bytes);
console.log(greeting);   // "Hello, Wasm!"
console.log(tick());     // 1
console.log(tick());     // 2

You have just written a module with memory, data initialization, mutable state, and multiple exports — in plain text. The string lives in linear memory at offset 0. JavaScript reads it by creating a Uint8Array view over the shared memory.buffer and decoding the bytes as UTF-8. This is how every string crosses the Wasm/JS boundary: bytes in memory, a pointer, and a length.

You can write a module and run it. The next chapter maps out the full vocabulary — control flow, indirect calls, and the string convention that every bridge depends on.

02 — the instruction set

Wasm has four value types (i32, i64, f32, f64), three control flow constructs (block, loop, if), and a stack-based execution model. That is the entire machine. No for. No while. No switch. No goto. Every structured loop, every conditional branch, every multi-way dispatch is built from these primitives and the br family of branch instructions. This chapter is a vocabulary tour — enough to read any WAT you encounter, not a manual for hand-writing every pattern.

Control Flow: The While Loop

The most important pattern: a block containing a loop. The block provides the exit target (forward jump). The loop provides the continue target (backward jump). This one combination builds every loop you have ever written:

WAT;; while (i < n) { total += arr[i]; i++; }
(block $break
  (loop $continue
    local.get $i
    local.get $n
    i32.ge_s
    br_if $break           ;; if i >= n, jump OUT

    ;; body: total += arr[i]
    local.get $total
    local.get $arr
    local.get $i
    i32.const 4
    i32.mul
    i32.add
    i32.load
    i32.add
    local.set $total

    ;; i++
    local.get $i
    i32.const 1
    i32.add
    local.set $i
    br $continue           ;; jump to TOP of loop
  )
)
The Block/Loop Trick

br to a block jumps forward (past the end). br to a loop jumps backward (to the top). Same instruction, opposite direction, depending on which construct it targets. The block is the exit door; the loop is the re-entry door. Once you internalize this, you can read any WAT control flow.

The other patterns follow the same logic. if/else pops an i32 and branches on zero vs nonzero — both branches must leave the stack in the same state. A do-while is just a bare loop with br_if at the bottom. C’s switch compiles to br_table — a jump table of nested blocks indexed by an integer. They all reduce to block, loop, and br.

Indirect Calls: How Vtables Work

Wasm has no function pointers in memory. Functions live in a separate, type-checked table. Calling through that table — call_indirect — validates the signature at runtime. This is how C++ vtables, Rust trait objects, and any function-pointer-based dispatch work in Wasm:

WAT(module
  (type $op (func (param i32 i32) (result i32)))

  (func $add (type $op) local.get 0 local.get 1 i32.add)
  (func $sub (type $op) local.get 0 local.get 1 i32.sub)
  (func $mul (type $op) local.get 0 local.get 1 i32.mul)

  (table 3 funcref)
  (elem (i32.const 0) $add $sub $mul)

  (func (export "calc") (param $a i32) (param $b i32) (param $op i32) (result i32)
    local.get $a
    local.get $b
    local.get $op
    call_indirect (type $op))   ;; type-checked at runtime!
)

call_indirect pops the table index from the stack, checks that the function at that index matches the declared type, and traps if either the index is out of bounds or the signature is wrong. In C, a bad function pointer is silent corruption. In Wasm, it is a guaranteed trap. When Emscripten compiles a C++ codebase with virtual methods, every virtual call goes through this mechanism — the vtable becomes a Wasm table, and each dispatch is type-checked.

Strings: Pointer + Length

Wasm has no string type. A “string” is a sequence of bytes in linear memory, addressed by an i32 pointer, with a length you track yourself. Two conventions exist:

Null-Terminated (C style)

H e l l o \0
48 65 6C 6C 6F 00

Scan until 0x00. O(n) to find length. Cannot contain null bytes.

Length-Prefixed (Wasm/Rust style)

ptr=100, len=5
H  e  l  l  o
48 65 6C 6C 6F

Length known upfront. O(1). Can contain any byte. This is what Rust, Go, and most Wasm tooling use.

Every string that crosses the JS/Wasm boundary goes through the same dance: JavaScript writes bytes into linear memory with TextEncoder, passes a pointer and a length to the Wasm function, and reads the result back with TextDecoder. The bridge looks like this:

JavaScriptfunction writeString(instance, str, offset) {
  const mem = new Uint8Array(instance.exports.memory.buffer);
  const encoded = new TextEncoder().encode(str);
  mem.set(encoded, offset);
  return { ptr: offset, len: encoded.length };
}

function readString(instance, ptr, len) {
  const mem = new Uint8Array(instance.exports.memory.buffer, ptr, len);
  return new TextDecoder().decode(mem);
}
The String Tax

Every string crossing is a copy. JavaScript strings are UTF-16 internally; Wasm works with raw bytes (usually UTF-8). TextEncoder/TextDecoder do the conversion. For short strings, negligible. For megabytes of text, batch your crossings — pass one pointer to the entire buffer, not individual strings. This is the same tradeoff Figma and Photoshop navigate in production.

You now know the vocabulary: four types, three control flow constructs, function tables, and the pointer-and-length convention that every bridge depends on. But you will rarely write WAT by hand. Real teams compile C++ and Rust to Wasm. The power move is reading what those compilers produce — and knowing when they got it wrong.

03 — reading compiler output

You know how to write Wasm. Now learn to read what your compiler writes. The single most useful tool in the WebAssembly ecosystem is wasm2wat — the disassembler that converts any .wasm binary back into the text format you have been writing by hand. Every optimization insight, every bloat diagnosis, every “why is my module 300 KB?” investigation starts with wasm2wat.

The Disassembly Loop

Take a small Rust function, compile it, and look at what comes out:

Rust#[no_mangle]
pub extern "C" fn sum_squares(a: i32, b: i32) -> i32 {
    a * a + b * b
}
Shellcargo build --target wasm32-unknown-unknown --release
wasm2wat target/wasm32-unknown-unknown/release/demo.wasm -f

The disassembled output shows exactly the instructions the compiler chose:

WAT (disassembled)(func $sum_squares (param $a i32) (param $b i32) (result i32)
  (i32.add
    (i32.mul (local.get $a) (local.get $a))
    (i32.mul (local.get $b) (local.get $b))))

Clean. A human would write the same thing. But increase the complexity — add a loop, some conditionals, a heap allocation — and the compiler starts leaving artifacts.

What to Look For

Unnecessary locals. LLVM converts every SSA register to a Wasm local. A function with 5 variables might declare 12 locals — the extras are temporaries that could have stayed on the stack. Look for (local $tmpN i32) declarations that appear only in a local.set immediately followed by a local.get. Those are round-trips the stack could have handled directly.

Redundant local.set/get pairs. The pattern local.set $x → local.get $x (with nothing in between) means the compiler stored a value and immediately loaded it back. A human would leave the value on the stack.

Dead code from monomorphization. Rust generics create specialized copies for each concrete type. If you use Vec<u8> and Vec<u32>, you get two copies of every Vec method. Some of those copies may be unused after inlining — but the compiler does not always remove them.

wasm-opt: The Second Compiler

Binaryen’s wasm-opt is a Wasm-to-Wasm optimizer. It runs after your compiler and applies passes that LLVM does not understand because they are Wasm-specific: dead local elimination, block merging, stack expression folding, and local coalescing. Running wasm-opt -O3 on a compiler-produced binary typically shaves off 10–20%.

Shell# Always run wasm-opt as the final build step
wasm-opt -O3 output.wasm -o output.opt.wasm

# Compare
wasm2wat output.wasm -f -o before.wat
wasm2wat output.opt.wasm -f -o after.wat
diff before.wat after.wat
PassWhat it does

The Optimization Budget

Don’t hand-optimize everything. The compiler is right 95% of the time. The remaining 5% is in your hot loop. Use wasm2wat to read that specific function. If the compiler introduced unnecessary locals or missed a stack folding opportunity, fix that one function. Leave the rest alone.

04 — building a template engine

We are going to build a template engine in pure handwritten WAT. It takes a template like "Hello, {{name}}! You have {{count}} messages." and a set of key-value substitutions, and produces the rendered output. No Rust. No Emscripten. Just WAT, a text editor, and wat2wasm. The engine will run in the browser via the string bridge from Chapter 2.

The Architecture

Template engine pipeline 
  Input: "Hello, {{name}}!"  +  { name: "World" }
         │
         ▼
  1. Scan — walk the template byte by byte
         find {{ and }} delimiters
         emit literal chunks and variable keys
         │
         ▼
  2. Substitute — for each variable key,
         look up the value via JS import
         copy the value bytes to the output buffer
         │
         ▼
  3. Output — return pointer + length
         to the rendered string in linear memory

  Output: "Hello, World!"

The template, the variable data, and the output buffer all live in the same flat linear memory. The Wasm module does what it is good at: scanning bytes fast. The variable lookup is imported from JavaScript — let the host handle the hash map.

Linear memory map 
  Offset     Region            Size       Purpose
  ───────    ───────           ────       ───────
  0x0000     template input    4 KB       the template string
  0x1000     variable keys     2 KB       packed key strings
  0x1800     variable values   2 KB       packed value strings
  0x2000     output buffer     8 KB       the rendered result

The Scanner

The scanner reads one byte at a time. Literal bytes go straight to the output. When it sees {{ (two 0x7B bytes), it switches to “read key” mode. When it sees }}, it calls the imported $lookup function and emits the value. Here is the core:

WAT(module
  (memory (export "memory") 1)
  (global $out_ptr (mut i32) (i32.const 0x2000))

  ;; emit one byte to the output buffer
  (func $emit_byte (param $b i32)
    global.get $out_ptr  local.get $b  i32.store8
    global.get $out_ptr  i32.const 1  i32.add  global.set $out_ptr)

  ;; copy len bytes from src to output
  (func $emit_bytes (param $src i32) (param $len i32)
    (local $i i32)
    (block $done (loop $next
      local.get $i  local.get $len  i32.ge_u  br_if $done
      (call $emit_byte
        (i32.load8_u (i32.add (local.get $src) (local.get $i))))
      local.get $i  i32.const 1  i32.add  local.set $i
      br $next)))

  ;; the main render loop
  (func (export "render") (param $tpl_ptr i32) (param $tpl_len i32)
                           (result i32 i32)
    (local $i i32) (local $ch i32) (local $key_start i32)
    i32.const 0x2000  global.set $out_ptr  ;; reset output

    (block $end (loop $scan
      local.get $i  local.get $tpl_len  i32.ge_u  br_if $end
      (i32.load8_u (i32.add (local.get $tpl_ptr) (local.get $i)))
      local.set $ch

      ;; check for {{ (0x7B 0x7B)
      local.get $ch  i32.const 0x7B  i32.eq
      (if (then
        (i32.load8_u (i32.add (i32.add
          (local.get $tpl_ptr) (local.get $i)) (i32.const 1)))
        i32.const 0x7B  i32.eq
        (if (then
          ;; found {{ — skip 2 bytes, record key start
          local.get $i  i32.const 2  i32.add  local.set $i
          (i32.add (local.get $tpl_ptr) (local.get $i))
          local.set $key_start

          ;; scan for }} (0x7D 0x7D)
          (block $found (loop $sk
            (i32.load8_u (i32.add
              (local.get $tpl_ptr) (local.get $i)))
            i32.const 0x7D  i32.eq
            (if (then
              (i32.load8_u (i32.add (i32.add
                (local.get $tpl_ptr) (local.get $i)) (i32.const 1)))
              i32.const 0x7D  i32.eq
              br_if $found))
            local.get $i  i32.const 1  i32.add  local.set $i
            br $sk))

          ;; lookup key → emit value
          local.get $key_start
          (i32.sub (i32.add (local.get $tpl_ptr) (local.get $i))
                   (local.get $key_start))
          call $lookup
          call $emit_bytes
          local.get $i  i32.const 2  i32.add  local.set $i
          br $scan))))

      ;; regular byte — emit it
      (call $emit_byte (local.get $ch))
      local.get $i  i32.const 1  i32.add  local.set $i
      br $scan))

    i32.const 0x2000
    (i32.sub (global.get $out_ptr) (i32.const 0x2000)))

  (import "env" "lookup"
    (func $lookup (param i32 i32) (result i32 i32)))
)

The entire engine is a state machine built from block, loop, and br_if — the same patterns from Chapter 2. The $lookup import delegates variable resolution to JavaScript, keeping the module focused on what Wasm does well: scanning bytes in linear memory.

Wiring It Up

The JavaScript side writes the template into memory, provides the lookup function, and reads the rendered output back:

JavaScriptconst encoder = new TextEncoder();
const decoder = new TextDecoder();
const template = "Hello, {{name}}! You have {{count}} new messages.";
const vars = { name: "Alice", count: "3" };

const importObj = {
  env: {
    lookup(key_ptr, key_len) {
      const mem = new Uint8Array(instance.exports.memory.buffer);
      const key = decoder.decode(mem.subarray(key_ptr, key_ptr + key_len));
      const encoded = encoder.encode(vars[key] || "");
      mem.set(encoded, 0x1800);
      return [0x1800, encoded.length];
    }
  }
};

const { instance } = await WebAssembly.instantiateStreaming(
  fetch('template.wasm'), importObj
);

const tpl_bytes = encoder.encode(template);
const mem = new Uint8Array(instance.exports.memory.buffer);
mem.set(tpl_bytes, 0);

const [out_ptr, out_len] = instance.exports.render(0, tpl_bytes.length);
const result = decoder.decode(mem.subarray(out_ptr, out_ptr + out_len));
console.log(result);
// → "Hello, Alice! You have 3 new messages."

Every concept from the first four chapters is in play: module structure, control flow with block/loop/br_if, memory loads and stores, function imports, and the TextEncoder/TextDecoder bridge. The Wasm module scans bytes. JavaScript resolves variables. The rendered string comes back as bytes in shared linear memory.

05 — ship it

The template engine works. It runs in a browser tab. The module is roughly 200 bytes after assembly. Let’s ship it.

Shellwat2wasm template.wat -o template.wasm
wasm-opt -O3 template.wasm -o template.opt.wasm
ls -la template.opt.wasm   # ~180 bytes

Run wasm-opt -O3 as a habit. Even on handwritten WAT, it may catch a redundant local or merge a block. On a real application compiled from Rust or C, it is 10–20% off the binary size.

Serving It

The .wasm file needs the right MIME type: application/wasm. Without it, instantiateStreaming rejects the response. If your server does not set it, fall back to the buffer path:

JavaScript// Streaming (preferred — compiles as it downloads)
const { instance } = await WebAssembly.instantiateStreaming(
  fetch('template.wasm'), importObj
);

// Buffer fallback (if Content-Type is wrong)
const bytes = await (await fetch('template.wasm')).arrayBuffer();
const { instance } = await WebAssembly.instantiate(bytes, importObj);

For caching, Wasm modules work like any static asset. The browser also caches the compiled native code — instantiateStreaming from a cached response skips both download and compilation on repeat visits.

From Learning to Production

Hand-writing WAT was for learning — the way studying assembly helps you write better C. You now understand the machine: the stack, the memory model, the type system, the bridge. You know what a Wasm module actually is under the hood.

06 — the compilation tradeoff

C, C++, and Rust dominate production WebAssembly. Not because they have better compilers, but because they manage their own memory. This is the single most important structural fact about Wasm, and it explains why some codebases migrate to the browser in weeks while others cannot migrate at all.

Why Memory Ownership Matters

Wasm’s linear memory is a contiguous byte array — a flat slab that the module reads and writes through i32 indices. C’s malloc and free operate on a contiguous byte array. They are the same model. When Emscripten compiles a C program to Wasm, malloc just manages Wasm’s linear memory. The heap IS the linear memory. No translation needed.

Rust’s ownership model is even cleaner. The compiler tracks lifetimes at compile time, inserts drop calls statically, and never needs a runtime allocator for stack-allocated values. A Rust function that takes a &[u8] slice compiles to a pointer and a length in Wasm — the exact representation you used in Chapter 2. The language’s memory model maps one-to-one onto the target.

The GC Tax

Garbage-collected languages face a different problem. Java, Go, Python, Kotlin, and Dart all need a garbage collector — a runtime component that walks the heap, tracks object references, and reclaims unused memory. In native execution, that GC is provided by the language runtime and integrated with the OS. In Wasm, there is no GC. The language has to ship its own.

The GC problem 
  C/C++/Rust → Wasm                    Go/Java/Python → Wasm
  ──────────────────                    ────────────────────────

  ┌──── .wasm module ────┐              ┌──── .wasm module ──────────┐
  │                       │              │ ┌────────────────────────┐ │
  │  your code            │              │ │  GC runtime            │ │
  │  malloc/free          │              │ │  (mark-sweep, threads, │ │
  │  linear memory = heap │              │ │   scheduler)           │ │
  │                       │              │ └────────────────────────┘ │
  │  ~200 bytes to ~KB    │              │  your code                 │
  └───────────────────────┘              │  linear memory = managed   │
                                         │                            │
                                         │  ~2+ MB minimum            │
                                         └────────────────────────────┘

Go’s standard Wasm target ships the entire Go runtime: goroutine scheduler, garbage collector, reflection system. Result: 2.1 MB for “Hello World.” TinyGo strips this to 15–50 KB by using LLVM and a simpler GC, but it sacrifices standard library coverage — reflect, net/http, and cgo are missing or partial.

The GC also creates a performance cliff. A GC running inside Wasm linear memory cannot see the browser’s own garbage collector. If your Wasm module creates an object that holds a reference to a JavaScript object, neither GC can see the full picture. Cross-boundary reference cycles become memory leaks.

The Pipelines

How code becomes .wasm 
  Emscripten  (C/C++)
  your_code.cpp → Clang → LLVM IR → LLVM wasm backend → .wasm + .js glue
  The .js glue sets up memory, wires libc, provides a virtual filesystem.
  -O3 -s STANDALONE_WASM strips the glue for pure compute modules.

  wasm-pack  (Rust)
  your_code.rs → rustc → LLVM IR → LLVM wasm backend → .wasm + bindgen shim
  wasm-bindgen generates TypeScript-friendly bindings for strings,
  structs, and Result types. #[no_std] strips to ~200 bytes.

  TinyGo  (Go)
  your_code.go → TinyGo → LLVM IR → LLVM wasm backend → .wasm
  Simpler GC, no goroutine scheduler. 15-50 KB for small programs.
  Partial standard library. Not a drop-in replacement for go build.

All three end at the same LLVM backend. The divergence is in what they ship alongside the compiled code. C++ and Rust ship essentially nothing — their runtime is the compiled code itself. Go ships a garbage collector. Java ships a garbage collector and a class loader. Python ships an interpreter.

This is not a toolchain comparison chart. It is a structural constraint. Languages that manage their own memory map cleanly to Wasm’s model. Languages that need a GC fight the model. This is why the largest, most successful Wasm deployments in production are overwhelmingly C++ and Rust codebases.

07 — who actually ships wasm

The compilation tradeoff is not theory. Real teams with real codebases made the decision to compile to Wasm, and the results range from “we saved the company” to “we shipped the impossible.” Six case studies. Each one: what language, what toolchain, what the tradeoff was, what they gained.

Figma: The C++ Rendering Engine

Figma’s 2D rendering engine is written in C++. Not the UI — the actual canvas rendering: vector operations, boolean geometry, gradient rasterization. When Figma launched, the engine compiled to asm.js (Emscripten’s pre-Wasm output). When browser Wasm support landed in 2017, they migrated and measured a 3x improvement in load time.

The architecture is split: C++ owns the canvas, JavaScript owns the UI and network layer. Pixel data lives in Wasm linear memory. The C++ engine rasterizes frames into a buffer, and JavaScript blits the result to a <canvas> element via putImageData. For multi-user editing, Figma uses SharedArrayBuffer to share the linear memory between the main thread and Web Workers — parallel tile rendering without copying pixel buffers.

The key insight: Figma did not choose Wasm because it was new. They chose C++ because JavaScript was too slow for 2D rendering at interactive framerates, and Wasm was the path to running C++ in a browser tab. Over 150,000 daily users of the web version never install a native app.

AutoCAD: 35 Years of C++ in a Browser

AutoCAD’s codebase is over 35 years old. Millions of lines of C++. The DWG file format, the geometry kernel, the rendering pipeline — all C++, all battle-tested, all deeply intertwined. When Autodesk launched AutoCAD Web in 2018, they compiled the existing C++ engine to Wasm via Emscripten.

The alternative was a rewrite. Translating decades of computational geometry, constraint solvers, and rendering code into JavaScript — with full DWG format support — would have taken years and introduced regressions in an engineering tool where sub-millimeter precision matters. Emscripten let them compile the same codebase that runs on desktop. The browser became another platform target, not a separate product.

The tradeoff was binary size. The full AutoCAD Wasm bundle is large — tens of megabytes — but that is tens of megabytes of geometry kernel and CAD logic, not runtime overhead. The code is the same code that runs natively. The Wasm tax is near zero.

Doom 3: The Full Game

In 2023, Gabriel Cuvillier compiled the id Tech 4 engine — Doom 3’s C++ game engine — to Wasm through Emscripten. Roughly 500,000 lines of C++ compiled with zero game-logic changes. The renderer, the physics, the AI, the audio mixer — all running in a browser tab at 60fps.

This is the purest demonstration of the compilation tradeoff. id Tech 4 manages its own memory entirely — custom allocators, memory pools, manual lifetime management. It maps to Wasm linear memory with no friction. The Emscripten layer provides OpenGL-to-WebGL translation and input event forwarding. The game does not know it is running in a browser.

The takeaway is not “you can play Doom in a browser.” It is that a half-million-line C++ codebase with custom memory management, real-time rendering requirements, and decades of optimization can target Wasm without a rewrite. If your codebase manages its own memory, Wasm is a viable target.

Google Sheets: WasmGC Changes the Equation

Google Sheets took a different path. The calculation engine was JavaScript — and it was the bottleneck. Google migrated it to a Kotlin codebase compiled to WasmGC, the Wasm proposal that lets the browser’s garbage collector manage Wasm objects directly. No custom GC shipped inside the module. No 2 MB runtime floor.

The result: 2x faster recalculation on complex spreadsheets. This was one of the first major WasmGC production deployments. It demonstrates that the GC tax from Chapter 6 is not permanent — WasmGC is the escape valve. But it requires browser support (Chrome 119+, Firefox 120+, shipped late 2023) and language toolchain adoption. Kotlin/Wasm targets WasmGC today. Dart targets it for Flutter Web. Go and Python do not — yet.

Photoshop: Creative Filters at Native Speed

Adobe brought Photoshop to the web starting in 2021. The core of Photoshop — the rendering pipeline, the filter engine, the color science — is C++ compiled via Emscripten. The UI is Web Components and Lit, running in JavaScript. The boundary between them is the bridge: pixel buffers flowing from the JavaScript UI layer into the C++ processing layer.

Adobe did not ship the entire desktop Photoshop on day one. They started with core editing tools and progressively added features — layers, selection tools, generative fill. Each feature was already implemented in C++. The work was not implementing the feature in a new language; it was wiring the existing C++ implementation to the browser’s input/output surfaces.

For compute-heavy operations like Content-Aware Fill, the C++ code runs in a Web Worker with SharedArrayBuffer, keeping the UI thread responsive. The filter pipeline crosses the Wasm/JS bridge once per operation — passing an image buffer pointer in, getting a processed buffer pointer back. The per-pixel work stays entirely inside Wasm.

Lichess: Stockfish in Every Browser Tab

Lichess, the open-source chess platform, compiles Stockfish — the strongest open-source chess engine — to Wasm. When you analyze a game on lichess.org, the analysis runs on your machine, in your browser tab. No server round-trip. No GPU. Stockfish’s alpha-beta search tree runs entirely in Wasm at roughly 50 million nodes per second — about half of native speed, but more than enough for real-time analysis.

The economics matter. Lichess is a non-profit. Server-side analysis for millions of games per day would require massive compute infrastructure. By compiling Stockfish to Wasm, the analysis cost is zero — the client pays with their own CPU. Every user gets unlimited analysis. The chess engine is C++ with manual memory management (transposition tables, move lists, evaluation caches). It maps to Wasm linear memory cleanly.

Lichess also demonstrates a deployment nuance: they ship multiple Wasm builds. A SharedArrayBuffer build uses Web Workers for multi-threaded search on browsers that support it. A single-threaded fallback runs everywhere else. The same C++ engine, two compilation targets, automatic detection.

The Pattern

Six teams. Six products. All C++ or Kotlin compiled to Wasm — not rewritten in a new language. The rendering engine stays in C++. The chess engine stays in C++. The calculation engine moves to Kotlin. The browser is a deployment target, not a development environment. Wasm is the bridge between “we have millions of lines of code” and “it needs to run everywhere.”

08 — the bridge

Every product in the previous chapter faces the same engineering problem: moving data between Wasm and the host. The Wasm module owns linear memory. The JavaScript host owns the DOM, the network, and the Canvas API. The bridge between them is where performance is won or lost.

The Call Overhead Model

A JavaScript → Wasm function call costs roughly 10–15 nanoseconds in V8 and SpiderMonkey. One call is free. The cost becomes visible at scale:

ScenarioCalls/frameOverhead at 60fpsVerdict
One render call1~15 nsnegligible
Per-pixel callback1,920,000~29 mskills your frame budget
Batched image pass1~15 nsthe right approach

The rule: cross the bridge once with a lot of data, not many times with a little data. This is exactly how the case studies from Chapter 7 work.

How Figma Crosses the Bridge

Figma’s C++ rendering engine rasterizes an entire tile into a pixel buffer in linear memory. JavaScript reads the result as a Uint8Array view and draws it to a <canvas> with putImageData. One call in, one read out. The per-pixel rendering loop runs entirely inside Wasm:

JavaScript// Figma's pattern: pass a buffer pointer, not individual pixels
const bufferPtr = instance.exports.render_tile(tileX, tileY, width, height);

const pixels = new Uint8ClampedArray(
  instance.exports.memory.buffer, bufferPtr, width * height * 4
);
const imageData = new ImageData(pixels, width, height);
ctx.putImageData(imageData, tileX, tileY);

Photoshop’s filter pipeline works the same way. A Content-Aware Fill operation receives an image buffer pointer, processes the entire image inside the C++ Wasm module, and returns a pointer to the result. The bridge is crossed once per operation, not once per pixel.

SharedArrayBuffer: Zero-Copy Sharing

For multi-threaded rendering, Figma and Photoshop use SharedArrayBuffer to share Wasm linear memory between the main thread and Web Workers. No copying — the worker writes directly to the same memory the main thread reads from:

JavaScriptconst memory = new WebAssembly.Memory({
  initial: 256,     // 16 MB
  maximum: 512,     // 32 MB
  shared: true      // backed by SharedArrayBuffer
});

// Pass the same memory to both the main module and the worker module.
// Both threads see the same bytes. Atomics coordinate access.

This requires Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Embedder-Policy: require-corp HTTP headers. The security headers are non-negotiable — they exist because SharedArrayBuffer was the vector for Spectre-class timing attacks. Lichess ships two builds: a multi-threaded SharedArrayBuffer build for browsers that support it, and a single-threaded fallback for those that don’t.

Struct Layouts in Memory

When passing complex data, you define a memory layout and pack fields at known offsets. There is no automatic serialization — you are the serializer:

Struct layout: Particle (20 bytes) 
  Offset   Field       Type     Size
  ──────   ─────       ────     ────
  +0       x           f32      4 bytes
  +4       y           f32      4 bytes
  +8       vx          f32      4 bytes
  +12      vy          f32      4 bytes
  +16      life        i32      4 bytes

  1000 particles = 20,000 bytes starting at base_ptr
  particle[i].vy = f32.load(base_ptr + i * 20 + 12)

This is exactly how C structs are laid out in memory. No JSON parsing. No object allocation. The JavaScript side writes fields at offsets; the Wasm side reads them with f32.load. For Figma’s vector objects, Photoshop’s layer metadata, or a game engine’s entity list, this is the only pattern that keeps the frame budget.

Automation: wasm-bindgen

If you use Rust, wasm-bindgen automates the bridge. It generates JavaScript shims that handle string conversion, struct serialization, and Result type translation:

Rustuse wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn greet(name: &str) -> String {
    format!("Hello, {}!", name)
}

Under the hood, the generated code does the same pointer-and-length dance from Chapter 2. wasm-bindgen just writes it for you.

When to Stay in JavaScript

Not everything belongs in Wasm. DOM manipulation, network requests, and string-heavy business logic are faster in JavaScript because they avoid the bridge entirely. Wasm wins when the work is compute-bound: image processing, physics simulation, audio synthesis, compression, parsing binary formats. The bridge cost is not per-byte — it is per-crossing. If the compute-to-crossing ratio is low, JavaScript is faster.

09 — beyond the browser

The same .wasm binary that runs in a browser tab can run on a server, inside a CLI tool, or at the edge. Outside the browser, the host is not JavaScript — it is WASI, the WebAssembly System Interface. WASI gives Wasm modules access to files, network sockets, clocks, and random numbers, but only the capabilities you explicitly grant. A module starts with zero permissions. Every capability is an allowlist.

Traditional programs inherit the full permissions of the user who runs them. A Python package you pip install can read your SSH keys and exfiltrate them. WASI inverts this. A Wasm module must be given each capability at instantiation: wasmtime --dir=/data/input app.wasm grants access to one directory. No flags means no filesystem access at all. Permissions are added, not removed.

Who Is Running Wasm Outside the Browser

Cloudflare Workers runs over a million Wasm-capable workers across 300+ cities. Cold start: sub-millisecond. Memory overhead per isolate: kilobytes, not the 5+ MB of a V8 isolate. This is the economic argument — on a machine running 10,000 tenants, the difference between Wasm and a container is the difference between one server and a fleet. Cloudflare uses their own runtime to execute Wasm modules as HTTP handlers.

Fastly Compute is Wasm-native. Every request handler is a compiled Wasm module running on Wasmtime. The platform supports Rust, Go (via TinyGo), and JavaScript (compiled to Wasm through their JS SDK). Startup time is under 50 microseconds — fast enough that they spin up a fresh instance per request and tear it down after.

Shopify Functions lets merchants write custom business logic — discount rules, shipping calculations, payment validation — that runs as Wasm modules inside Shopify’s infrastructure. The security model is the point: merchant code runs in a sandbox with no filesystem access, no network access, and strict compute limits. It cannot escape to other tenants’ data.

Fermyon Spin is a full application framework built on Wasmtime. You write request handlers in Rust, Go, Python, or JavaScript, compile to Wasm, and deploy. Spin handles routing, key-value storage, and outbound HTTP. The component model lets you compose handlers written in different languages into a single application.

Docker + Wasm

Docker Desktop added Wasm support in 2022 through runwasi, a containerd shim that runs Wasm modules alongside traditional Linux containers. The interface is familiar:

Shelldocker run --runtime=io.containerd.wasmedge.v1 \
  --platform=wasi/wasm \
  my-wasm-app:latest

A Wasm “container” does not include a Linux filesystem, a kernel interface, or a package manager. It is a .wasm binary with WASI capabilities. Cold start: microseconds. Image size: kilobytes to low megabytes. The tradeoff is that your code must compile to Wasm — you cannot run arbitrary Linux binaries.

The Component Model

The Component Model and WIT (Wasm Interface Types) are turning modules into composable building blocks. A Rust image resizer, a Kotlin ML classifier, and a Go request handler can be wired together without sharing memory or agreeing on a calling convention. Each component has its own linear memory. They communicate through WIT-defined interfaces with automatic type translation.

Component composition 
  ┌─────────────────┐     ┌──────────────────┐     ┌────────────────┐
  │   Rust module    │     │  Kotlin module   │     │  Go module     │
  │  image-resize    │     │  ml-classify     │     │  http-handler  │
  └───────┬──────────┘     └───────┬──────────┘     └───────┬────────┘

A string in Rust becomes a string in Kotlin becomes a pointer-and-length in Go — the Component Model handles the conversion. This is the path to a universal plugin system: compose modules written in any language, running in any host, with isolation between them. Fermyon Spin, Wasmtime, and WAMR all support components today.

10 — what wasm changes

WebAssembly shipped as an MVP in 2017 with four value types and no threads. Eight years later, the spec has grown proposals like rings on a tree — each one extending the machine in a specific direction. Three matter most.

WasmGC: The GC Tax Goes Away

Before WasmGC, garbage-collected languages had to ship their own GC inside the Wasm module — megabytes of runtime code managing memory inside linear memory, blind to the browser’s own collector. WasmGC (shipped in Chrome 119, Firefox 120, late 2023) adds struct and array types that the host engine manages directly. The module declares types; the engine allocates, traces, and collects them.

WasmGC vs. linear memory GC 
  Before WasmGC                          With WasmGC
  ──────────────                          ──────────

  ┌──── Wasm module ────┐                ┌──── Wasm module ────┐
  │ ┌──────────────────┐ │                │                      │
  │ │  custom GC code  │ │                │  (no GC code)        │
  │ │  (mark & sweep)  │ │                │  struct/array types  │
  │ └──────────────────┘ │                │  managed by host     │
  │ ┌──────────────────┐ │                └──────────┬───────────┘
  │ │  linear memory   │ │                           │
  │ │  (manual heap)   │ │                  ┌────────▼────────┐
  │ └──────────────────┘ │                  │  Browser’s GC   │
  └──────────────────────┘                  │  V8 / SM / JSC  │
                                            └─────────────────┘
  Duplicated work,                        Shared GC,
  large binaries,                         smaller binaries,
  can’t collect cycles                    full cycle collection
  with JS objects                         across JS & Wasm

Google Sheets proved the model works at scale: Kotlin compiled to WasmGC, 2x faster recalculation, no shipped GC runtime. Kotlin/Wasm and Dart target WasmGC today. Go and Python do not yet — but when they do, the 2.1 MB floor for Go’s standard Wasm target becomes a rounding error.

SIMD and Memory64

The fixed-width SIMD proposal (128-bit v128) shipped in 2023. Relaxed SIMD goes further: it trades cross-platform determinism for speed, letting engines use platform-specific instructions like x86 vfmadd and ARM fmla. For audio processing, image filters, and ML inference, the 10–30% speedup is worth the non-deterministic low bits. Stockfish’s Wasm build uses SIMD to accelerate position evaluation.

Memory64 extends addressing from i32 (4 GB maximum) to i64, enabling modules that work with large datasets: genomics pipelines, video editors, scientific simulations. The API is identical; only the pointer width changes. Wasmtime and V8 have experimental support.

The AI Stack

The ML ecosystem has quietly converged on Wasm as its portable floor. ONNX Runtime compiles to Wasm for browser inference. TensorFlow.js uses Wasm as its CPU backend. MediaPipe runs pose detection through Wasm. web-llm loads quantized LLMs in the browser using Wasm for tokenization and WebGPU for matrix math.

The pattern is consistent: Wasm handles orchestration (tokenization, pre/post-processing, scheduling) while hardware accelerators handle the heavy compute. Wasm is the glue runtime — portable, sandboxed, and fast enough for everything that is not a matrix multiply.

What You Can Build Now

You started this article writing bytecode in a text editor. You built a template engine from raw WAT instructions. If you have a C++ library you want to run in the browser — Emscripten. If you are writing new compute-heavy code — Rust with wasm-pack. If you need to run untrusted code safely — WASI with capability-based permissions. If you need GC language support — WasmGC with Kotlin or Dart.

The instruction set is small. The reach is not. Now build something.
