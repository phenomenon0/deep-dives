Making things look real. From a single pixel lookup to full PBR material systems — the art and science of surface appearance._

// 01 — From Flat to Real

Before textures, every surface in a video game was a single color. Picture Wolfenstein 3D in 1992: you are staring down a corridor of flat-shaded walls, each face a uniform shade of grey or brown. The raycaster that powered the game cast one ray per screen column, found the nearest wall, and drew a vertical stripe of a single hue. Floors and ceilings were solid colors. It was fast, it ran on a 386, and it looked like a fever dream rendered in construction paper. But id Software knew that flat color was a dead end. The walls needed grain, the doors needed rivets, the world needed detail that geometry alone could never provide.

Texture mapping — the idea of gluing a 2D image onto a 3D surface — was not new in academia. Edwin Catmull described it in his 1974 PhD thesis. But making it run at 30+ frames per second on consumer hardware? That took two decades of engineering heroics, each generation solving the problem slightly better than the last.

1992: The Flat Era

Wolfenstein 3D used a raycaster, not a true 3D engine. Walls were vertical strips of texture data, sampled column by column. There was no perspective correction because the walls were always perpendicular to the view plane. Floors and ceilings? Just solid fills. The technique was brilliant in its constraints — by limiting the world to a grid of identical-height walls, Carmack could render the scene with nothing more than a handful of multiplies per column.

GLSL
// Recreating the Wolfenstein look:
// vertical wall strips, no perspective on floors
precision mediump float;
uniform vec2 u_resolution;

void main() {
    vec2 uv = gl_FragCoord.xy / u_resolution;
    vec3 col = vec3(0.4, 0.35, 0.3);
    // Wall strips — one color per vertical slice
    float stripe = step(0.5, fract(uv.x * 8.0));
    col = mix(col, col * 0.7, stripe);
    // Floor: solid color below midline
    if (uv.y < 0.3) col = vec3(0.25, 0.25, 0.22);
    gl_FragColor = vec4(col, 1.0);
}

1993: Affine Texture Mapping

Doom changed everything — and broke everything. Unlike Wolfenstein, Doom had angled walls, variable heights, and textured floors and ceilings. But the texture mapping was affine: a simple linear interpolation of texture coordinates across each polygon. This is fast (just add a constant per pixel) but mathematically wrong. Affine mapping ignores the fact that in perspective projection, equal steps in screen space do not correspond to equal steps in texture space. The result? Textures warp and swim as you move, especially on large floor surfaces viewed at oblique angles.

GLSL
// Affine vs. correct: the Doom warp
// Affine interpolation skips the perspective divide
precision mediump float;
uniform vec2 u_resolution;
uniform float u_time;

void main() {
    vec2 uv = gl_FragCoord.xy / u_resolution;
    // Simulate affine warp on a floor plane
    float depth = max(0.01, uv.y);
    vec2 texUV_correct = vec2(uv.x / depth, 1.0 / depth);
    vec2 texUV_affine  = vec2(uv.x * 4.0, (1.0 - uv.y) * 4.0);
    // Use affine — notice the wobble
    vec2 tc = texUV_affine;
    float check = mod(floor(tc.x) + floor(tc.y), 2.0);
    vec3 col = mix(vec3(0.3), vec3(0.6), check);
    gl_FragColor = vec4(col, 1.0);
}

You could see the affine warp every time you looked at a floor in Doom. The checkerboard pattern would bend and shift as you moved. Players in 1993 did not care — the game was running on a 486 and it looked incredible compared to everything else. But the warp was there, and fixing it would require dividing by depth at every single pixel.

1996: Perspective-Correct Texturing

Quake was the watershed. John Carmack implemented full perspective-correct texture mapping and, just as importantly, mipmaps — precomputed half-resolution copies of every texture that the renderer could swap in for distant surfaces. The combination eliminated the swimming artifacts of affine mapping and the shimmering moire patterns that plagued un-mipmapped textures. It cost more per pixel (a divide per span, not per pixel, thanks to Carmack’s clever subdivision), but it looked right.

GLSL
// Perspective-correct texture mapping
// The key insight: interpolate u/w, v/w, and 1/w,
// then divide at each pixel
precision mediump float;
uniform vec2 u_resolution;

void main() {
    vec2 uv = gl_FragCoord.xy / u_resolution;
    // Simulated floor with perspective depth
    float depth = max(0.05, uv.y);
    // Perspective-correct: divide by depth
    vec2 tc = vec2((uv.x - 0.5) / depth, 1.0 / depth);
    tc *= 2.0;
    float check = mod(floor(tc.x) + floor(tc.y), 2.0);
    vec3 col = mix(vec3(0.15, 0.12, 0.1), vec3(0.5, 0.45, 0.4), check);
    // Fog with distance for that Quake atmosphere
    float fog = exp(-0.3 / depth);
    col = mix(vec3(0.05), col, fog);
    gl_FragColor = vec4(col, 1.0);
}

1998: Lightmaps — Baked Illumination

Unreal Engine 1 popularized lightmaps: a second texture layer containing pre-baked lighting information. Instead of computing light at runtime (expensive) or baking it into the diffuse texture (inflexible), you store the illumination in a separate, lower-resolution texture and multiply it with the surface color at render time. This gave Unreal its characteristic moody, colored lighting — blue moonlight through a window, warm torch glow on stone walls — all for the cost of a single extra texture lookup.

GLSL
// Lightmap sampling: multiply diffuse by baked light
precision mediump float;
uniform sampler2D u_diffuse;    // base color texture
uniform sampler2D u_lightmap;   // baked illumination
varying vec2 v_uv;
varying vec2 v_lightUV;        // separate UV set for lightmap

void main() {
    vec4 diffuse  = texture2D(u_diffuse, v_uv);
    vec4 lighting = texture2D(u_lightmap, v_lightUV);
    // Multiply: diffuse color * baked light intensity
    gl_FragColor = vec4(diffuse.rgb * lighting.rgb * 2.0, 1.0);
}

The * 2.0 is not a mistake. Lightmaps are typically stored with a midpoint of 0.5 representing “normal” lighting, so multiplying by 2 maps that back to 1.0. Values above 0.5 in the lightmap create over-bright highlights; values below create shadows. Simple, elegant, and still used today in engines like Unity and Source 2.

2010s: The PBR Revolution

The final leap came when Brent Burley at Disney published the “Disney Principled BRDF” in 2012, and engines like Unreal 4 and Frostbite adopted metallic-roughness workflows. Instead of one texture, you now have a material system: albedo (base color), metallic, roughness, normal, ambient occlusion — each texture feeding a physically-motivated shading equation. A single material might consume five textures. But the result is surfaces that look correct under any lighting condition, not just the one the artist designed for.

GLSL
// The core of PBR: Cook-Torrance BRDF
// Sample multiple texture maps for one surface
vec3 albedo    = texture2D(u_albedo, uv).rgb;
float metallic = texture2D(u_metalRough, uv).b;
float rough    = texture2D(u_metalRough, uv).g;
vec3 N         = normalize(texture2D(u_normal, uv).rgb * 2.0 - 1.0);
float ao       = texture2D(u_ao, uv).r;

// F0: reflectance at normal incidence
vec3 F0 = mix(vec3(0.04), albedo, metallic);

// GGX distribution, Schlick-GGX geometry, Fresnel
float D = DistributionGGX(N, H, rough);
float G = GeometrySmith(N, V, L, rough);
vec3  F = FresnelSchlick(max(dot(H, V), 0.0), F0);

The texture mapping revolution

From flat-shaded walls to multi-layered PBR materials in 25 years. Every one of those transitions — affine to perspective, single texture to lightmaps, lightmaps to PBR — came from a single insight: you can store information in images and look it up per-pixel. Texture mapping is not just “putting a picture on a surface.” It is the fundamental mechanism by which GPUs trade memory bandwidth for computational complexity. Instead of calculating something, you pre-calculate it, store it in a texture, and look it up. This idea underpins normal maps, shadow maps, environment maps, lookup tables, noise functions, and virtually every modern rendering technique.

// 02 — Your First Texture in WebGL

Forget the history for a moment. You want to put an image on a triangle. In WebGL, that means creating a texture object, uploading pixel data to the GPU, configuring how the GPU samples it, and then reading it in your fragment shader. It is a five-step dance, and every step matters. Miss one — forget to set the wrap mode, neglect to flip the Y axis, use the wrong internal format — and you will get a black rectangle and an hour of debugging.

Let us walk through it piece by piece.

Step 1: Create and Bind

Everything in WebGL starts with object creation. A texture is an opaque handle. You create it, bind it to a texture unit, and all subsequent texture operations apply to that binding.

JavaScript
const texture = gl.createTexture();
gl.activeTexture(gl.TEXTURE0);        // select texture unit 0
gl.bindTexture(gl.TEXTURE_2D, texture); // bind to it

WebGL guarantees at least 8 texture units (TEXTURE0 through TEXTURE7). Desktop GPUs typically support 16 or 32. Each unit can have one texture bound at a time, and your shader references them by unit number.

Step 2: Upload Image Data

The workhorse function is gl.texImage2D. It takes pixel data — from an Image, a canvas, a video element, or a raw ArrayBuffer — and uploads it to GPU memory.

JavaScript
const img = new Image();
img.src = 'brick_diffuse.png';
img.onload = () => {
  gl.bindTexture(gl.TEXTURE_2D, texture);
  gl.texImage2D(
    gl.TEXTURE_2D,     // target
    0,                   // mip level (0 = base)
    gl.RGBA,             // internal format
    gl.RGBA,             // source format
    gl.UNSIGNED_BYTE,    // data type
    img                  // source element
  );
  gl.generateMipmap(gl.TEXTURE_2D);
};
The Y-flip trap

WebGL textures have their origin at the bottom-left, but HTML images have their origin at the top-left. If your texture appears upside down, add gl.pixelStorei(gl.UNPACK_FLIP_Y_WEBGL, true) before uploading. This is the single most common WebGL texture bug.

Step 3: Set Sampling Parameters

How does the GPU decide what color to return when your shader asks for the texel at coordinate (0.37, 0.82)? That depends on the filtering and wrapping parameters you set.

JavaScript
// Filtering: how to interpolate between texels
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_MIN_FILTER, gl.LINEAR_MIPMAP_LINEAR);
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_MAG_FILTER, gl.LINEAR);

// Wrapping: what happens when UV > 1.0 or < 0.0
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_WRAP_S, gl.REPEAT);
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_WRAP_T, gl.REPEAT);

The MIN_FILTER matters when the texture is minified (far away, many texels map to one pixel). The MAG_FILTER matters when magnified (close up, one texel covers multiple pixels). We will dig into filtering in the next section — for now, LINEAR_MIPMAP_LINEAR (trilinear) and LINEAR are safe defaults for most textures.

Step 4: Sample in GLSL

In your fragment shader, you declare a sampler2D uniform and call texture2D() to look up a color.

GLSL
precision mediump float;
uniform sampler2D u_texture;
varying vec2 v_uv;

void main() {
    vec4 texColor = texture2D(u_texture, v_uv);
    gl_FragColor = texColor;
}

On the JavaScript side, you connect the sampler to a texture unit:

JavaScript
const loc = gl.getUniformLocation(program, 'u_texture');
gl.uniform1i(loc, 0);  // unit 0 = TEXTURE0

Step 5: Non-Power-of-Two Textures

In WebGL 1, non-power-of-two (NPOT) textures have restrictions: no mipmaps, no REPEAT wrapping, only CLAMP_TO_EDGE. If your 300×200 texture shows as black, this is why. WebGL 2 lifts these restrictions entirely. If you must support WebGL 1, either resize your textures to powers of two (256, 512, 1024...) or accept the limitations.

WebGL vs. WebGPU

If you are coming from (or headed toward) WebGPU, the texture workflow changes significantly. WebGPU separates the texture (the data) from the sampler (the filtering/wrapping config) and from the texture view (how you interpret the data). It is more verbose but also more explicit and less error-prone.

WebGL (GLSL)

// Sampler and texture are one object
uniform sampler2D u_tex;
vec4 c = texture2D(u_tex, uv);

WebGPU (WGSL)

// Separate texture and sampler
@group(0) @binding(0)
  var t: texture_2d<f32>;
@group(0) @binding(1)
  var s: sampler;
let c = textureSample(t, s, uv);

The WebGPU approach means you can reuse one sampler across many textures, or sample the same texture with different filtering modes without re-uploading it. This maps more closely to how modern GPUs actually work.

Placeholder texture trick

While your image loads asynchronously, the texture is incomplete and will render as black. A common pattern is to upload a tiny 1×1 pixel of your desired fallback color immediately, then overwrite it when the real image arrives. This way your scene never flashes black during load.

// 03 — Filtering & Mipmaps

When your shader calls texture2D(sampler, uv), the GPU does not simply grab the nearest texel and return it. What actually happens depends on a cascade of decisions: Is the texture being magnified or minified? What filter mode is active? Are mipmaps available? Is anisotropic filtering enabled? The answer to these questions determines whether you get a crisp, blurry, shimmery, or beautifully smooth result. Understanding filtering is the difference between textures that look professional and textures that scream “hobby project.”

Nearest-Neighbor: The Pixel Art Filter

The simplest filter. The GPU rounds the UV coordinate to the nearest texel center and returns that exact color. No blending, no interpolation. The result is hard pixel edges — perfect for pixel art, retro games, and Minecraft. Terrible for photorealistic rendering.

GLSL
// Simulating nearest-neighbor sampling
vec2 texSize = vec2(64.0);  // 64x64 texture
vec2 texel = floor(uv * texSize) / texSize;
vec4 color = texture2D(u_tex, texel + 0.5 / texSize);

When you zoom in on a nearest-filtered texture, you see the raw grid. Each texel becomes a visible square. This is exactly what you want for a game like Celeste or Stardew Valley, where soft blending would destroy the pixel art aesthetic.

Bilinear: The Smooth Default

Bilinear filtering samples the four nearest texels and blends them based on how close the UV coordinate is to each one. It is a weighted average in 2D — linear interpolation along one axis, then along the other.

Bilinear Interpolation 
  UV lands between four texels:

  A ────────── B        mix_x = fract(uv.x * width)
  │            │        mix_y = fract(uv.y * height)
  │      ●     │
  │    (u,v)   │        top    = mix(A, B, mix_x)
  │            │        bottom = mix(C, D, mix_x)
  C ────────── D        result = mix(top, bottom, mix_y)

Bilinear filtering eliminates the blocky look of nearest-neighbor, but it introduces a subtle blur. At high magnification, textures look soft rather than sharp. For minification (shrinking), bilinear is not enough — you need mipmaps.

GLSL
// Manual bilinear filter (for understanding —
// the GPU does this in hardware)
vec4 bilinear(sampler2D tex, vec2 uv, vec2 texSize) {
    vec2 pixel = uv * texSize - 0.5;
    vec2 f = fract(pixel);
    vec2 base = (floor(pixel) + 0.5) / texSize;
    vec2 step = 1.0 / texSize;

    vec4 tl = texture2D(tex, base);
    vec4 tr = texture2D(tex, base + vec2(step.x, 0.0));
    vec4 bl = texture2D(tex, base + vec2(0.0, step.y));
    vec4 br = texture2D(tex, base + step);

    return mix(mix(tl, tr, f.x), mix(bl, br, f.x), f.y);
}

The Mipmap Pyramid

Consider a 1024×1024 brick texture applied to a wall that is 50 meters away. On screen, that wall might occupy only 20×20 pixels. Without mipmaps, the GPU must somehow combine thousands of texels into a handful of pixels. The result is aliasing — shimmering, moire patterns, visual noise that dances as the camera moves. Mipmaps fix this by providing pre-filtered versions of the texture at successively smaller resolutions.

Mipmap Pyramid 
  Level 0:  1024 × 1024   ← full resolution
  Level 1:   512 × 512    ← each texel = avg of 2×2 from level 0
  Level 2:   256 × 256    ← avg of 2×2 from level 1
  Level 3:   128 × 128 
  Level 4:    64 × 64  
  Level 5:    32 × 32  
  Level 6:    16 × 16  
  Level 7:     8 × 8   
  Level 8:     4 × 4   
  Level 9:     2 × 2   
  Level 10:    1 × 1      ← single average color

  Total memory: ~133% of base texture
  (1 + 1/4 + 1/16 + 1/64 + ... ≈ 4/3)

The GPU selects the mip level where one texel approximately maps to one pixel on screen. It calculates this using the screen-space derivatives of the texture coordinates — dFdx(uv) and dFdy(uv) — which tell it how fast the UVs are changing across neighboring pixels. Fast change means high minification, which means use a smaller mip.

Trilinear: Smooth Mip Transitions

With bilinear filtering and mipmaps, the GPU samples from the single closest mip level. This creates visible “mip banding” — a hard line where the GPU switches from one level to the next. Trilinear filtering fixes this by sampling from two adjacent mip levels and blending between them. The result is seamless transitions from near to far.

GLSL
// Trilinear = bilinear on two mip levels + blend
// The GPU does this automatically when you set:
//   MIN_FILTER = LINEAR_MIPMAP_LINEAR

// Conceptually:
float mipLevel = computeMipLevel(dFdx(uv), dFdy(uv));
float level0 = floor(mipLevel);
float level1 = level0 + 1.0;
float blend  = fract(mipLevel);

vec4 samp0 = bilinearSample(tex, uv, level0);
vec4 samp1 = bilinearSample(tex, uv, level1);
vec4 result = mix(samp0, samp1, blend);

Anisotropic Filtering: The Quality King

Both bilinear and trilinear assume the pixel’s footprint in texture space is roughly square. But when you view a surface at a grazing angle — a road stretching to the horizon, a floor beneath your feet — the footprint is elongated. Isotropic filtering uses a square approximation and over-blurs one axis. Anisotropic filtering samples along the elongated footprint, taking multiple bilinear samples and averaging them.

GLSL
// The derivatives tell you the pixel footprint shape
vec2 dx = dFdx(uv);  // UV change per pixel horizontally
vec2 dy = dFdy(uv);  // UV change per pixel vertically

// Anisotropy ratio = max(|dx|,|dy|) / min(|dx|,|dy|)
// A floor at 80 degrees might have ratio 16:1
// The GPU takes up to N samples along the long axis
// (where N = your anisotropy setting: 2x, 4x, 8x, 16x)

In WebGL, you enable anisotropic filtering through the EXT_texture_filter_anisotropic extension:

JavaScript
const ext = gl.getExtension('EXT_texture_filter_anisotropic');
if (ext) {
  const max = gl.getParameter(ext.MAX_TEXTURE_MAX_ANISOTROPY_EXT);
  gl.texParameterf(gl.TEXTURE_2D, ext.TEXTURE_MAX_ANISOTROPY_EXT, max);
}
Performance note

Trilinear filtering is essentially free on modern GPUs — the hardware is designed for it. Anisotropic filtering at 4× or 8× is nearly free too. At 16× you might see a slight bandwidth increase on mobile. There is almost never a reason to use anything less than trilinear + 4× aniso in 2024. The visual improvement, especially on ground planes and walls at oblique angles, is dramatic.

Mip Generation: Box vs. Kaiser

The default mip generation algorithm is a box filter: average every 2×2 block to produce the next level. It works, but it is not optimal. A box filter has poor frequency response — it does not cleanly cut off high frequencies, which means aliasing can leak into lower mip levels. The Kaiser filter (a windowed sinc function) produces sharper, more accurate mips. Most serious texture pipelines (nvcompress, texconv, ISP Compressonator) use Kaiser or Lanczos filters for mip generation.

Why does the mip filter matter?

Imagine a texture of thin parallel lines. With a box filter, the mip chain progressively blurs the lines until they merge into a uniform grey at lower levels. With a Kaiser filter, the lines stay sharper longer and transition to grey more gracefully. The difference is subtle at 1:1 viewing but noticeable when the camera moves and the GPU transitions between mip levels — box-filtered mips can produce a “pop” as detail suddenly appears or vanishes.

Blender Context

In Blender’s Shader Editor, the Image Texture node has an Interpolation dropdown: Linear, Closest, Cubic, Smart. “Closest” is nearest-neighbor. “Linear” is bilinear. “Cubic” uses a higher-order filter (bicubic) that is sharper than bilinear but slower. “Smart” (Cycles/EEVEE 4.0+) uses a combination of cubic and MIP-based anti-aliasing. If your baked textures look soft in the viewport, check that you are not accidentally using “Closest” on a photographic texture.

// 04 — Texture Formats & Compression

A 4096×4096 RGBA texture at 8 bits per channel occupies 64 MB of GPU memory. A character model with five PBR texture maps at that resolution consumes 320 MB — for a single character. Multiply by a dozen characters, plus environments, plus UI elements, and you have blown past the VRAM budget of most consumer GPUs before rendering a single frame. This is why texture compression exists. It is not optional. It is the difference between a game that ships and a game that crashes on load.

GPU texture compression is fundamentally different from PNG or JPEG compression. Those formats decompress the entire image in CPU memory before use. GPU-compressed textures stay compressed in VRAM and are decompressed on-the-fly by dedicated hardware in the texture unit, one block at a time, as the shader samples them. This means random access — the GPU can decompress any 4×4 block independently without touching the rest of the image. It also means the compression is lossy and block-based, which creates characteristic artifacts.

BC1 (DXT1): The Workhorse

The oldest GPU compression format still in widespread use. BC1 divides the texture into 4×4 pixel blocks. For each block, it stores two 16-bit colors (RGB565) and a 32-bit lookup table that maps each of the 16 pixels to one of four palette entries: the two stored colors, their 2/3-1/3 blend, and their 1/3-2/3 blend. That is 64 bits for 16 pixels — a 6:1 compression ratio (8:1 if you count the missing alpha).

BC1 Block Layout (64 bits = 8 bytes per 4×4 block) 
  Bits  0-15:  Color 0  (RGB565: 5 red, 6 green, 5 blue)
  Bits 16-31:  Color 1  (RGB565)
  Bits 32-63:  Index table  (2 bits per pixel × 16 pixels)

  If Color0 > Color1:
    Index 0 → Color0
    Index 1 → Color1
    Index 2 → 2/3 × Color0 + 1/3 × Color1
    Index 3 → 1/3 × Color0 + 2/3 × Color1

  If Color0 ≤ Color1:
    Index 3 → transparent black (1-bit alpha mode)

BC1 works beautifully for diffuse textures with broad color regions. It struggles with smooth gradients (banding from only 4 colors per block) and high-frequency detail (the block boundaries become visible). Never use BC1 for normal maps — the RGB565 quantization destroys the subtle angular information.

BC3 (DXT5): BC1 + Full Alpha

BC3 takes BC1 and adds a separate alpha channel compressed using a similar two-endpoint scheme but with 3-bit indices (8 palette entries instead of 4). The result is 128 bits per block — twice the size of BC1, but with smooth alpha gradients. Use BC3 for foliage, particles, decals, or anything with alpha transparency.

BC5: Two-Channel Normal Maps

Normal maps only need two channels (X and Y) because Z can be reconstructed: Z = sqrt(1 - X*X - Y*Y). BC5 stores two independent channels, each compressed like BC3’s alpha channel (8-bit endpoints, 3-bit indices). This gives much higher precision per channel than BC1 or BC3, which is critical for normals. In the shader:

GLSL
// Reconstruct Z from BC5 normal map (RG channels only)
vec3 unpackBC5Normal(sampler2D normalMap, vec2 uv) {
    vec2 rg = texture2D(normalMap, uv).rg;
    vec2 n = rg * 2.0 - 1.0;
    float z = sqrt(max(0.0, 1.0 - n.x*n.x - n.y*n.y));
    return vec3(n, z);
}

BC7: The Modern Default

BC7 is the Swiss Army knife. It supports 1–4 channels, uses 128 bits per block (same as BC3), but achieves dramatically higher quality by partitioning each block into subsets with independent endpoints. There are 64 possible partitions per block, and the compressor picks the one that minimizes error. The result is near-lossless quality at 4:1 compression for RGB, or 3:1 for RGBA. The catch? Compression is slow — a 4K BC7 texture can take seconds to compress, compared to milliseconds for BC1.

ASTC: Flexible Blocks

Adaptive Scalable Texture Compression, developed by ARM. Instead of a fixed 4×4 block size, ASTC supports block sizes from 4×4 (highest quality, 8 bpp) up to 12×12 (aggressive compression, 0.89 bpp). This lets you dial the quality-versus-size tradeoff per texture. ASTC is mandatory on OpenGL ES 3.2 and Vulkan on mobile. It handles HDR, 3D textures, and sRGB natively. If you are targeting mobile or console, ASTC is likely your primary format.

ETC2: The Android Baseline

ETC2 is mandatory in OpenGL ES 3.0, which means every Android device supports it. Quality sits between BC1 and BC7 for RGB textures. It includes an EAC mode for 1–2 channel data (similar to BC4/BC5). If you need a single compressed format that works on every Android phone from 2013 onward, ETC2 is your only guaranteed option.

The Basis Universal trick

Shipping separate BC7, ASTC, and ETC2 versions of every texture triples your download size. Basis Universal (and its successor KTX2 with BasisLZ/UASTC supercompression) solves this by storing textures in a single intermediate format that can be transcoded on the client to whichever GPU format the device supports. Load one file, transcode in milliseconds, get native GPU compression on every platform. This is now the standard approach for web-based 3D (Three.js, Babylon.js, PlayCanvas all support it).

Blender Export Workflow

When exporting textures from Blender for real-time use, the format matters enormously. Blender itself renders with uncompressed textures, but your game engine will compress them on import. The workflow:

Blender → Game Engine Texture Pipeline

1. Bake in Blender at 2× your target resolution (e.g., bake at 4096 if targeting 2048). This gives the compressor more data to work with.

2. Export as PNG (lossless) or EXR (for HDR/linear data). Never export as JPEG — the JPEG artifacts compound with GPU compression artifacts.

3. Import into your engine. Unity and Unreal automatically compress to the appropriate GPU format based on target platform and texture type (diffuse → BC1/BC7, normal → BC5, etc.).

4. Review the compressed result. Both Unity and Unreal have preview modes that show you the compressed texture alongside the original. Always check normal maps — they suffer most from compression.

// 05 — Normal Maps: Faking Depth

You have seen them a thousand times — those eerily blue-purple images that look like a Warhol painting of the surface they represent. Normal maps are the single most cost-effective illusion in real-time graphics. Instead of modeling every brick, every pore, every rivet with actual geometry (millions of extra triangles), you encode the direction each pixel of the surface should face into a texture. The lighting engine reads this encoded direction and shades the pixel as if the surface really had that shape. The geometry stays flat. The shading lies. And the lie is so convincing that most players never question it.

So what is a normal map, exactly? It is a texture where each pixel stores a surface normal vector instead of a color. A normal vector has three components (X, Y, Z), each ranging from -1 to 1, but texture channels store values from 0 to 255. The mapping is simple: encoded = normal * 0.5 + 0.5. A neutral surface — one that faces straight out, unchanged from the base geometry — has the normal (0, 0, 1). Encoding that: X = 0.5 (128), Y = 0.5 (128), Z = 1.0 (255). In RGB, that is (128, 128, 255) — the distinctive blue-purple. Now you know why every normal map looks blue: most of the surface points approximately "straight out," so most pixels have a Z component near 1.0 and X/Y components near 0.5.

Tangent Space: Why "Straight Out" Is Relative

Normal maps are stored in tangent space, not world space. This is crucial. Tangent space is a coordinate system local to each triangle of the mesh, where Z points along the geometric normal, X points along the texture U axis (the tangent), and Y completes the orthonormal basis (the bitangent). This means the same normal map can be applied to any surface at any orientation — a brick wall facing north uses the same texture as one facing east. The transform from tangent space to world space is the TBN matrix — a 3×3 rotation matrix built from the Tangent, Bitangent, and Normal vectors of each vertex.

Tangent Space → World Space Transform 
  Tangent space normal from texture:
  normalMap = texture2D(uNormalMap, vUV).rgb * 2.0 - 1.0

  T = tangent (along UV.u)

How do you construct this matrix? You need the tangent and bitangent vectors, which are derived from the mesh's vertex positions and UV coordinates. The tangent points in the direction of increasing U in world space, and the bitangent points in the direction of increasing V. For a single triangle, you solve a linear system relating edge vectors in position space to edge vectors in UV space. In practice, you compute tangents per-triangle and average them at shared vertices, then re-orthogonalize with Gram-Schmidt to keep the basis clean. Here is the full implementation.

// Vertex shader outputs (interpolated per-fragment)
varying vec3 vTangent;
varying vec3 vBitangent;
varying vec3 vNormal;
varying vec2 vUV;

uniform sampler2D uNormalMap;

void main() {
    // Build the TBN matrix from interpolated vertex attributes
    vec3 T = normalize(vTangent);
    vec3 N = normalize(vNormal);
    // Re-orthogonalize T with respect to N (Gram-Schmidt)
    T = normalize(T - dot(T, N) * N);
    vec3 B = cross(N, T);
    mat3 TBN = mat3(T, B, N);TBN construction

    // Sample and decode the normal map
    vec3 normalMap = texture2D(uNormalMap, vUV).rgb * 2.0 - 1.0;

    // Transform from tangent space to world space
    vec3 worldNormal = normalize(TBN * normalMap);tangent → world

    // Use worldNormal for all lighting calculations...
    vec3 lightDir = normalize(uLightPos - vWorldPos);
    float diff = max(dot(worldNormal, lightDir), 0.0);
    gl_FragColor = vec4(albedo * diff, 1.0);
}

def build_tbn(tangent: np.ndarray, normal: np.ndarray) -> np.ndarray:

function buildTBN(tangent: Vec3, normal: Vec3): Mat3 {
    const normalize = (v: Vec3): Vec3 => {
        const len = Math.hypot(...v);
        return [v[0]/len, v[1]/len, v[2]/len];
    };
    const dot = (a: Vec3, b: Vec3) =>
        a[0]*b[0] + a[1]*b[1] + a[2]*b[2];

    let N = normalize(normal);
    let T = normalize(tangent);
    // Gram-Schmidt
    const d = dot(T, N);
    T = normalize([T[0]-d*N[0], T[1]-d*N[1], T[2]-d*N[2]]);
    const B: Vec3 = [bitangent via cross
        N[1]*T[2] - N[2]*T[1],
        N[2]*T[0] - N[0]*T[2],
        N[0]*T[1] - N[1]*T[0]
    ];
    // Column-major: T | B | N
    return new Float32Array([
        T[0],T[1],T[2], B[0],B[1],B[2], N[0],N[1],N[2]
    ]);
}

function applyNormalMap(tbn: Mat3, rgb: Vec3): Vec3 {
    // Decode from [0,1] to [-1,1]
    const nm: Vec3 = [rgb[0]*2-1, rgb[1]*2-1, rgb[2]*2-1];
    // mat3 * vec3 (column-major)
    const r: Vec3 = [
        tbn[0]*nm[0] + tbn[3]*nm[1] + tbn[6]*nm[2],
        tbn[1]*nm[0] + tbn[4]*nm[1] + tbn[7]*nm[2],
        tbn[2]*nm[0] + tbn[5]*nm[1] + tbn[8]*nm[2]
    ];
    const len = Math.hypot(...r);
    return [r[0]/len, r[1]/len, r[2]/len];
}

// 06 — Parallax & Displacement

Normal maps change how light bounces off a surface, but they cannot change where the surface appears to be. Look at a normal-mapped brick wall at a steep angle and the illusion shatters — the bricks should occlude each other, mortar gaps should reveal depth, but instead the surface stays relentlessly flat. Parallax mapping fixes this by shifting the texture coordinate based on the viewing angle and a height map. It is ray marching in UV space — the fragment shader walks along the view direction through a virtual height field to find where the "surface" really is, then reads the texture at that offset location. The geometry stays flat. The UV coordinates lie. And the depth illusion is startlingly convincing.

Basic Parallax Mapping

The simplest version offsets the UV by the view direction scaled by the height:

vec2 parallaxOffset(vec2 uv, vec3 viewDir, float height) {
    return uv - viewDir.xy * (height * 0.1);single-step offset
}

This works for shallow depths but breaks down at steep viewing angles — it can only offset by a single sample, so tall features overshoot and the parallax effect looks like it is sliding across the surface. The fix is to take multiple steps.

Steep Parallax Mapping

Instead of a single offset, walk along the view ray in discrete steps through the height field. At each step, compare the ray's current depth to the height map value. When the ray passes below the surface, you have found the intersection. More steps means better accuracy but more texture reads per pixel. The step count is the quality knob: 8 steps gives visible staircase artifacts, 32 steps is smooth, 64 is overkill for everything except close-up hero surfaces.

Parallax Occlusion Mapping (POM)

POM combines steep parallax with a binary search refinement step. After the steep parallax walk finds the layer where the ray crosses the height field, POM interpolates between the last two samples to find a more precise intersection point. This gives smooth, artifact-free results even at extreme angles, and it is the technique used in most modern game engines.

POM Ray March Through Height Field 
  View ray enters from top-right:

  depth=0.0  ─────●─────────────────  surface plane
                    ╲
  depth=0.25 ────────●────────────────  step 1: above height
                      ╲
  depth=0.50 ──────────●──────────────  step 2: above height
                        ╲
  depth=0.75 ─────░░░░░░●░░░──────────  step 3: BELOW height!
                       ╱  ╲
  Binary search between step 2 & 3
  → final UV = mix(uv2, uv3, 0.63)

  ░░░ = height map surface

// Parallax Occlusion Mapping
vec2 parallaxOcclusionMap(
    vec2 uv, vec3 viewDir,
    sampler2D heightMap,
    float depthScale,
    int numSteps
) {
    float layerDepth = 1.0 / float(numSteps);
    float currentDepth = 0.0;
    vec2 deltaUV = viewDir.xy * depthScale / (viewDir.z * float(numSteps));
    vec2 currentUV = uv;

    float currentHeight = texture2D(heightMap, currentUV).r;
    // Steep parallax: march until we go below surface
    for (int i = 0; i < 64; i++) {ray march loop
        if (i >= numSteps) break;
        if (currentDepth >= currentHeight) break;
        currentUV -= deltaUV;
        currentHeight = texture2D(heightMap, currentUV).r;
        currentDepth += layerDepth;
    }

    // Binary search refinement
    vec2 prevUV = currentUV + deltaUV;
    float afterH  = currentHeight - currentDepth;
    float beforeH = texture2D(heightMap, prevUV).r
                   - currentDepth + layerDepth;
    float weight = afterH / (afterH - beforeH);interpolation
    return mix(currentUV, prevUV, weight);
}

def parallax_occlusion_map(
    uv: np.ndarray,        # [2] float
    view_dir: np.ndarray,  # [3] float, tangent space
    height_map: np.ndarray,# 2D heightmap, [0..1]
    depth_scale: float = 0.1,
    num_steps: int = 32
) -> np.ndarray:
    """POM: returns displaced UV coordinate."""
    layer_depth = 1.0 / num_steps
    current_depth = 0.0
    delta_uv = view_dir[:2] * depth_scale / (view_dir[2] * num_steps)
    current_uv = uv.copy()

    def sample_h(u):
        x = int(u[0] * height_map.shape[1]) % height_map.shape[1]
        y = int(u[1] * height_map.shape[0]) % height_map.shape[0]
        return height_map[y, x]

    current_h = sample_h(current_uv)
    for _ in range(num_steps):
        if current_depth >= current_h:
            break
        current_uv -= delta_uvmarch step
        current_h = sample_h(current_uv)

    prev_uv = current_uv + delta_uv

    w = after_h / (after_h - before_h + 1e-8)
    return current_uv * (1 - w) + prev_uv * w

function parallaxOcclusionMap(
  uv: [number, number],
  viewDir: [number, number, number],
  sampleHeight: (u: number, v: number) => number,
  depthScale = 0.1,
  numSteps = 32
): [number, number] {
  const layerDepth = 1.0 / numSteps;
  let currentDepth = 0;
  const dU = viewDir[0] * depthScale / (viewDir[2] * numSteps);
  const dV = viewDir[1] * depthScale / (viewDir[2] * numSteps);
  let cu = uv[0], cv = uv[1];
  let h = sampleHeight(cu, cv);

  for (let i = 0; i < numSteps; i++) {
    if (currentDepth >= h) break;
    cu -= dU; cv -= dV;ray march
    h = sampleHeight(cu, cv);
    currentDepth += layerDepth;
  }

  // Interpolation
  const pu = cu + dU, pv = cv + dV;
  const afterH = h - currentDepth;
  const beforeH = sampleHeight(pu, pv) - currentDepth + layerDepth;
  const w = afterH / (afterH - beforeH + 1e-8);
  return [cu * (1-w) + pu * w, cv * (1-w) + pv * w];
}

Self-Shadowing

POM can also produce self-shadows. After finding the displaced UV, you march a second ray from that point toward the light source through the height field. If the ray intersects the height map before reaching the surface plane, the pixel is in shadow. This adds dramatic depth cues — bricks casting shadows into mortar gaps, cobblestones darkening on the side away from the light — all without any actual geometry.

The Blender Connection: Displacement vs. POM

Blender gives you both options. The Displacement socket on Material Output actually modifies vertex positions — your flat plane becomes genuinely bumpy geometry. This is "real" displacement and requires subdivision (Adaptive Subdivision in Cycles). POM, by contrast, is a purely fragment-shader trick: the vertices do not move. Blender's Displacement is physically accurate but expensive. POM is a beautiful hack that runs at 60 fps. Game engines almost universally choose POM; film renderers choose displacement. The sweet spot in real-time is often both: coarse displacement for silhouette-affecting features, POM for fine detail.

POM Is a Fragment Shader Ray Tracer

Parallax occlusion mapping is conceptually identical to sphere tracing / ray marching: you step along a ray, test against a distance field (here a height map), and refine the intersection. The difference is that you are marching in UV space instead of world space, and the "geometry" is encoded as a height texture rather than a mathematical SDF. If you understand POM, you already understand the core algorithm behind Shadertoy's entire ray marching ecosystem. Game developers were doing per-pixel ray tracing on consumer GPUs in 2006 — a full 12 years before Nvidia announced RTX.

// 07 — PBR: The Material System

For decades, game materials were a collection of ad-hoc hacks. Artists would tweak ambient, diffuse, and specular colors by hand, adding a "shininess" exponent and hoping the result looked plausible under the game's particular lighting setup. Change the lighting, and materials fell apart — a surface that looked like brushed metal in one scene looked like wet plastic in another. Physically Based Rendering changed everything. PBR constrains material definitions to values that obey the laws of energy conservation and Fresnel reflectance, guaranteeing that materials look correct under any lighting condition. It is not about making things look pretty. It is about making things look right.

The Cook-Torrance Reflectance Model

PBR's workhorse is the Cook-Torrance BRDF (Bidirectional Reflectance Distribution Function). The BRDF answers a single question: given a surface point, an incoming light direction, and a viewing direction, how much light gets reflected toward the viewer? Cook-Torrance splits this into a diffuse term (light that penetrates the surface, scatters internally, and re-emerges) and a specular term (light that bounces directly off microfacets on the surface). The full equation:

Cook-Torrance BRDF 
                       D(h) · F(v,h) · G(l,v,h)
  f(l,v) = k_d · ───── + ─────────────────────────
                    π        4 · (n·l) · (n·v)

  Where:
    D = Normal Distribution Function  (how many microfacets face H)
    F = Fresnel Function               (angle-dependent reflectance)
    G = Geometry Function              (self-shadowing of microfacets)
    k_d = (1 - F)(1 - metallic)        (diffuse contribution)
    h = halfway vector between l and v

Each component of the BRDF maps directly to a physically observable property of real materials. Understanding what each function does — and what happens when you change its parameters — is the key to mastering PBR. Let us take them one at a time.

Metallic: What Materials Are Made Of

The metallic parameter is a binary classifier masquerading as a slider. In reality, a material is either a metal (conductor) or a non-metal (dielectric) — there is no "half-metal" in nature. The slider exists for blending at material boundaries (rust on metal, for instance) and for artistic control. What it actually controls is F0 — the base reflectance at normal incidence:

  • Dielectric (metallic = 0): F0 is approximately 0.04 (4% reflectance) and the reflected color is white. This is why plastic, wood, stone, and skin all reflect white highlights regardless of their surface color. The diffuse term dominates.
  • Metal (metallic = 1): F0 equals the albedo color. Gold reflects gold-colored light. Copper reflects copper-colored light. There is zero diffuse contribution — metals absorb all light that penetrates the surface.
// F0 calculation — the core of the metallic workflow
vec3 F0 = mix(vec3(0.04), albedo, metallic);metals reflect their color
vec3 kD = (1.0 - F) * (1.0 - metallic);no diffuse for metals

Roughness: The Microfacet Distribution

Every real surface is microscopically rough. Even a polished mirror has imperfections at the nanometer scale. The roughness parameter controls the statistical distribution of these microfacets — their orientations relative to the macro surface normal. Low roughness means most microfacets point the same direction (mirror-like). High roughness means they are scattered in all directions (diffuse-looking). The visual effect is that roughness blurs the specular highlight: a roughness of 0 produces a pinpoint reflection, roughness of 1 produces a broad, soft sheen that looks almost matte.

Fresnel: The Angle-Dependent Glow

Hold a sheet of paper at eye level and look straight at it — you see diffuse white. Now tilt it until you are looking at an extreme grazing angle. It becomes shiny. Every material does this. At normal incidence, you see mostly the diffuse color. At grazing angles, you see increasingly strong reflections. This is the Fresnel effect, and Schlick's approximation captures it with elegant simplicity:

// Schlick's Fresnel approximation
vec3 fresnelSchlick(float cosTheta, vec3 F0) {
    return F0 + (1.0 - F0) * pow(clamp(1.0 - cosTheta, 0.0, 1.0), 5.0);the magic exponent
}

The exponent of 5 is not arbitrary — Schlick derived it as the simplest polynomial that closely matches the full Fresnel equations across a wide range of materials. At cosTheta = 1 (looking straight at the surface), the function returns F0. At cosTheta = 0 (grazing angle), it returns 1.0 — total reflection. This is why puddles on the ground look like mirrors when you look across them at a low angle, and why even matte surfaces develop a sheen at their edges.

GGX / Trowbridge-Reitz: The Normal Distribution Function

The NDF answers: "what fraction of microfacets on this surface are oriented to reflect light from L toward V?" GGX (also called Trowbridge-Reitz) is the industry standard because it produces long specular tails that match real materials better than the older Blinn-Phong or Beckmann distributions. Its specular highlights have a sharp bright core with a gradual falloff — exactly what you see on real polished surfaces:

// GGX Normal Distribution Function
float distributionGGX(vec3 N, vec3 H, float roughness) {
    float a  = roughness * roughness;Disney remap: square it
    float a2 = a * a;
    float NdotH  = max(dot(N, H), 0.0);
    float NdotH2 = NdotH * NdotH;

    float denom = NdotH2 * (a2 - 1.0) + 1.0;
    return a2 / (3.14159265 * denom * denom);
}

Note the roughness * roughness remap. This is Disney's contribution: perceptually, doubling the roughness slider should feel like doubling the roughness. The raw GGX parameter (alpha) relates quadratically to perceptual roughness, so squaring the artist's input makes the slider feel linear. Every major engine does this — Unreal, Unity, Godot, Blender's EEVEE. When you see alpha = roughness * roughness, you are looking at Disney's fingerprint.

Smith's Geometry Function: Microfacet Self-Shadowing

Not all microfacets that face the right direction actually contribute light. Some are shadowed by neighboring microfacets (light cannot reach them) or masked (the reflected light is blocked before reaching the viewer). The geometry function models both effects. Smith's method separates the calculation into two parts — shadowing (light side) and masking (view side) — and multiplies them:

// Smith's Geometry Function with Schlick-GGX
float geometrySchlickGGX(float NdotV, float roughness) {
    float r = roughness + 1.0;
    float k = (r * r) / 8.0;direct lighting k
    return NdotV / (NdotV * (1.0 - k) + k);
}

float geometrySmith(vec3 N, vec3 V, vec3 L, float roughness) {
    float NdotV = max(dot(N, V), 0.0);
    float NdotL = max(dot(N, L), 0.0);
    return geometrySchlickGGX(NdotV, roughness)
         * geometrySchlickGGX(NdotL, roughness);shadow × mask
}

The k value differs between direct and image-based lighting. For direct lights, k = (roughness + 1)² / 8. For IBL, k = roughness² / 2. Using the wrong k is a common bug that makes rough surfaces appear either too dark or too bright under environment lighting.

The material is the math.

Energy Conservation: The Non-Negotiable Constraint

This is the constraint that makes PBR "physically based": a surface cannot reflect more light than it receives. The Fresnel term F determines how much light is reflected specularly. Whatever is not reflected (1 - F) is available for diffuse scattering. For metals, 1 - metallic zeroes out the diffuse term entirely. The result is that kD * albedo/PI + specular never exceeds the incoming light energy. This is why PBR materials do not blow out to white under strong lights the way old Phong shading did — the math prevents it.

Full BRDF Implementation

// Complete Cook-Torrance PBR BRDF
const float PI = 3.14159265;

vec3 fresnelSchlick(float cosTheta, vec3 F0) {
    return F0 + (1.0 - F0) * pow(clamp(1.0 - cosTheta, 0.0, 1.0), 5.0);
}

float distributionGGX(vec3 N, vec3 H, float rough) {
    float a  = rough * rough;
    float a2 = a * a;
    float NdotH = max(dot(N, H), 0.0);
    float d = NdotH * NdotH * (a2 - 1.0) + 1.0;
    return a2 / (PI * d * d);GGX distribution
}

float geometrySmith(vec3 N, vec3 V, vec3 L, float rough) {
    float r = rough + 1.0;
    float k = (r * r) / 8.0;
    float NdotV = max(dot(N, V), 0.0);
    float NdotL = max(dot(N, L), 0.0);
    float ggx1 = NdotV / (NdotV * (1.0 - k) + k);
    float ggx2 = NdotL / (NdotL * (1.0 - k) + k);
    return ggx1 * ggx2;Smith geometry
}

// Evaluate the full BRDF for one light
vec3 evaluateBRDF(
    vec3 N, vec3 V, vec3 L,
    vec3 albedo, float metallic, float roughness
) {
    vec3 H = normalize(V + L);

    // Material properties
    vec3 F0 = mix(vec3(0.04), albedo, metallic);

    // BRDF components
    float D = distributionGGX(N, H, roughness);
    float G = geometrySmith(N, V, L, roughness);
    vec3  F = fresnelSchlick(max(dot(H, V), 0.0), F0);

    // Specular: Cook-Torrance
    float NdotL = max(dot(N, L), 0.0);
    float denom = 4.0 * max(dot(N, V), 0.0) * NdotL + 0.0001;
    vec3 specular = (D * G * F) / denom;Cook-Torrance specular

    // Diffuse: energy-conserving Lambertian
    vec3 kD = (1.0 - F) * (1.0 - metallic);

    return (kD * albedo / PI + specular) * NdotL;
}

Correctness, Not Beauty

PBR is not about making things look good — it is about making things look correct. A gold sphere under a blue sky should have blue tinting in its fresnel rim, because that is what physics demands. A rough wooden table should scatter light broadly while a polished marble floor focuses it into tight highlights. PBR guarantees this. Once your BRDF obeys energy conservation and Fresnel reflectance, materials work under any lighting — sunrise, overcast, neon, moonlight — without artist intervention. The math does not care about aesthetics. It cares about physics. And paradoxically, that constraint produces more beautiful results than any hand-tuned hack ever could.

// 08 — Environment & Reflections

A PBR material without environment lighting is like a musical instrument in a vacuum — technically correct but lifeless. Real surfaces do not exist under a single point light. They sit in a world of sky, ground, walls, and scattered bounced light. Environment maps capture this surrounding illumination as a texture and let your shader reflect, refract, and diffusely integrate it. The result is the difference between "nice sphere demo" and "this looks real." Environment mapping is where PBR stops being a theory exercise and starts producing images that fool your eyes.

Cubemaps: Six Faces of Reality

The classic approach stores the environment as six square images — one for each face of a cube (+X, -X, +Y, -Y, +Z, -Z). Given a reflection direction vector, the GPU selects the appropriate face and samples the texture. In GLSL, this is a single call: textureCube(envMap, reflectDir). The hardware handles face selection, edge filtering, and seamless interpolation across face boundaries. Cubemaps are simple, efficient, and remain the workhorse format for real-time environment mapping.

// Cubemap reflection — the simplest environment mapping
vec3 R = reflect(-viewDir, normal);
vec3 envColor = textureCube(uEnvMap, R).rgb;one line, full environment

Equirectangular Maps: The Panoramic Alternative

An equirectangular map stores the entire environment in a single 2D texture using latitude-longitude projection — the same mapping used for world maps. This format is convenient for HDR environment images (many HDRI websites distribute them this way) and easy to capture with a 360-degree camera. Converting a direction vector to equirectangular UVs is straightforward:

vec2 dirToEquirect(vec3 dir) {
    float phi   = atan(dir.z, dir.x);          // longitude
    float theta = asin(clamp(dir.y, -1.0, 1.0));  // latitude
    return vec2(
        phi / (2.0 * 3.14159) + 0.5,normalize to [0,1]
        theta / 3.14159 + 0.5
    );
}

The weakness of equirectangular maps is heavy distortion at the poles — vast amounts of texture resolution are wasted representing the tiny patch of sky directly overhead. Cubemaps distribute resolution more evenly across all directions, which is why most engines convert equirectangular HDRIs to cubemaps at load time. But equirectangular is still the standard interchange format because it is just a regular 2D image — any image editor can open it, preview it, and manipulate it.

The Prefiltered Mip Chain: Roughness as Blur

A mirror-smooth surface should reflect the environment sharply. A rough surface should reflect a blurry version. The elegant trick is to pre-convolve the environment map at different roughness levels and store each level as a mip of the cubemap. Mip 0 is the original sharp image (roughness = 0). Each subsequent mip is progressively blurred, representing higher roughness. At render time, you sample the cubemap at the mip level corresponding to the material's roughness:

float lod = roughness * float(maxMipLevel);roughness → mip level
vec3 envColor = textureCubeLod(uEnvMap, R, lod).rgb;

This is remarkably efficient: instead of sampling hundreds of directions per pixel to simulate a rough reflection (which is what a path tracer does), you sample a single pre-blurred texel. The prefiltering is done offline or at load time, so the runtime cost is identical to a sharp reflection. It is one of the cleverest tricks in real-time rendering — trading a compute-intensive integral for a single texture lookup by precomputing the answer at every roughness level.

Split-Sum Approximation: Real-Time IBL

Image-Based Lighting (IBL) is the technique of using an environment map as a light source. The mathematically correct approach integrates the BRDF against the environment over the entire hemisphere — thousands of samples per pixel. The split-sum approximation, introduced by Epic Games for Unreal Engine 4, factors this integral into two independent parts that can be precomputed:

Split-Sum Approximation 
  Full IBL integral (too expensive for real-time):
    Lo = ∫ f(l,v) · Li(l) · (n·l) dl

  Split into two precomputable parts:

  Part 1: Prefiltered Environment
    ∫ Li(l) · D(h) · (n·l) dl
    → stored as mip chain of cubemap

  Part 2: BRDF Integration LUT
    ∫ f(l,v) · (n·l) dl
    → stored as 2D texture indexed by (NdotV, roughness)
    → returns (scale, bias) for Fresnel: F0*scale + bias

  Final: prefilteredColor * (F0 * scale + bias)

The BRDF integration LUT is a small 512×512 texture that is the same for all environments — it only depends on the BRDF itself. You compute it once and ship it with your engine. Combined with the prefiltered environment cubemap, you get full IBL with a single texture lookup per component. This is what made PBR practical for real-time gaming: the split-sum turns an O(n) integration into an O(1) lookup.

The Blender Connection

In Blender's World settings, you add an Environment Texture node (typically an HDRI) and connect it to the World Output's Surface. Cycles uses this as both the background and a light source, performing the full hemisphere integration at render time with hundreds of bounced rays. For EEVEE (Blender's real-time engine), the same HDRI is prefiltered into a cubemap with a mip chain — exactly the split-sum approach described above. When you click "Bake Indirect Lighting" in EEVEE, you are generating the prefiltered cubemap and irradiance probes that make IBL work at interactive framerates. Same art, same HDRI, two completely different rendering strategies — but the visual result converges.

How IBL Bridges Offline and Real-Time

IBL is the unifying thread that connects film rendering and game rendering. In a path tracer (like Cycles, Arnold, or RenderMan), environment lighting works by shooting rays into the scene and, when they miss geometry, sampling the environment map directly. Hundreds of rays per pixel produce a noise-free result. In a real-time engine, the split-sum approximation achieves a visually similar result with two texture lookups. The quality gap has narrowed to the point where casual observers cannot tell the difference in well-lit scenes.

This convergence is accelerating. Hardware-accelerated ray tracing (RTX, RDNA 2+) lets real-time engines cast a few rays per pixel for reflections, supplementing the prefiltered cubemap with actual traced reflections where it matters most — glossy surfaces, mirrors, wet floors. The prefiltered map handles the broad, blurry ambient lighting; traced rays handle the sharp, geometry-dependent reflections. The hybrid approach gives the best of both worlds: performance where it is easy, accuracy where it matters.

The Environment IS the Light

In PBR, there is no fundamental distinction between "lights" and "environment." A point light is just a delta function in the environment map — an infinitely bright, infinitely small dot. An area light is a bright rectangular patch. The sun is a bright circle at a specific direction. IBL unifies all of these into a single mathematical framework: integrate the BRDF against the incoming radiance from all directions. The only question is how you approximate that integral at the speeds you need. The distinction between "a light" and "a bright spot in the sky texture" is an implementation detail, not a physical one.

// 09 — UV Unwrapping & Blender Craft

Every texture technique in this entire deep dive depends on one invisible foundation: UV coordinates. A UV coordinate is a pair of floating-point numbers that maps a point on a 3D surface to a point on a 2D texture image. The U axis runs horizontally, the V axis runs vertically, and together they define a 2D address space — usually in the [0, 1] range — where the GPU looks up color, normal, roughness, and every other surface property. Without UVs, a triangle has no idea which part of the texture belongs to it. The mesh is just geometry in the dark.

The challenge is that 3D surfaces are not flat. Unwrapping a sphere into a rectangle necessarily introduces distortion — the same mathematical impossibility that plagues every world map projection. The art of UV unwrapping is choosing where to cut, where to stretch, and where to waste space — all to minimize the visual damage when a 2D image is draped over a 3D shape. Get it wrong and you see blurry texels on one face and razor-sharp ones on the adjacent face, stretched text on a cylinder, or seam lines where two UV islands meet with slightly different texel densities.

Projection Types

Before manually unwrapping every face, artists choose a projection method that provides a starting point. Each method works best for specific shapes and fails catastrophically on others.

Box / Cube projection projects the texture along the six cardinal axes (±X, ±Y, ±Z). Each face of the mesh gets assigned to whichever axis its normal most closely aligns with. This works perfectly for architectural geometry — walls, floors, crates — anything fundamentally box-shaped. It fails on organic shapes where normals transition smoothly between axes, producing visible seams at the transition zones.

Cylindrical projection wraps the texture around one axis like a label on a bottle. Ideal for tree trunks, columns, arms, legs, gun barrels — anything with a dominant axis of symmetry. The poles (top and bottom) pinch into singularities where all UV lines converge, exactly like the poles of a Mercator map.

Spherical projection maps longitude and latitude onto U and V. It handles shapes that are roughly spherical — heads, eyeballs, planets — but introduces extreme distortion at the poles and a seam along the back where the texture wraps around to meet itself. Every globe you have ever seen in a game has this seam hidden behind the character's head or under the planet's polar ice cap.

Smart UV Project (Blender's automated unwrap) analyzes the mesh geometry, identifies roughly planar regions, projects them individually, and packs the results. It is the "I just need reasonable UVs and I don't want to think about it" option. For background props that nobody will inspect closely, it works surprisingly well. For hero assets, it is a starting point that needs significant manual refinement.

Seams: Where to Cut

A seam is an edge on the 3D mesh where the UV map is allowed to split. Think of it as taking scissors to a globe: you need at least one cut to flatten it, and where you cut determines where the distortion lives. Strategic seam placement is the core skill of UV unwrapping. The rules of thumb:

  • Hide seams in concave creases — the underside of an arm, the crease behind an ear, the junction between a wall and a floor. Lighting naturally darkens these areas, hiding any texture discontinuity.
  • Cut along hard edges — where normals already have a sharp break, a UV seam is invisible because there is no smooth shading to reveal it.
  • Minimize total seam length — every seam is a potential visual artifact. Fewer seams mean fewer problems, but too few means too much distortion.
  • Preserve texel density — the ratio of texture pixels to world-space area should be roughly constant across the entire mesh. A character's face should not have 4× the texel density of their shoes (unless that is an intentional artistic choice).

UV Packing and UDIM

After unwrapping, you have a collection of UV islands — disconnected 2D shapes, one for each continuous region of the mesh. Packing these islands efficiently into the [0, 1] UV square is a bin-packing optimization problem. Wasted space between islands is wasted texture resolution. Professional tools like UVPackmaster and Blender's built-in Pack Islands operator use heuristic and brute-force search to minimize wasted space, rotating and scaling islands to achieve packing densities of 85–95%.

For film and VFX work, a single 0–1 UV tile is not enough. A hero character for a feature film might need 50+ 4K textures to capture every pore and fabric thread. UDIM (Universal Dimension) extends the UV space beyond the 0–1 range into a grid of tiles, each addressed by a four-digit number starting at 1001. Tile 1001 is the standard [0,1] × [0,1] square. Tile 1002 is [1,2] × [0,1]. Tile 1011 is [0,1] × [1,2]. Each tile maps to a separate texture file on disk. The shader (or renderer) automatically picks the right file based on which UDIM tile the UV coordinate falls into. Substance Painter, Mari, and Blender all support UDIM workflows natively. Game engines typically do not — the overhead of multi-tile UV is overkill for real-time rendering, where texture budgets are measured in megabytes, not terabytes.

UVs Are the Foundation

Normal maps, PBR materials, baked lighting, displacement, parallax — every technique in this deep dive reads from a texture, and every texture read is addressed by UVs. A model with sloppy UVs will produce sloppy results from every subsequent step in the pipeline. The UV layout is not a boring prerequisite to get past before the fun starts. It is the invisible architecture that determines whether every other technique succeeds or fails. Spend the time.

// 10 — Baking: High → Low Poly

This is how game characters are actually made. An artist sculpts a 10-million-polygon masterpiece in ZBrush — every pore, wrinkle, stitch, and scratch lovingly placed by hand or brush. Then they build a second model: a clean, 30,000-polygon cage designed to deform well for animation and run at 60fps. The two models are aligned, and a process called baking transfers the sculpted detail from the high-poly surface onto textures that wrap the low-poly mesh. The player sees a 30K-poly model that looks like it has 10 million polygons of detail, because every surface nuance lives in the normal map, the AO map, and the curvature map — all products of baking.

Baking is fundamentally a ray-casting operation. For every texel on the low-poly model's UV space, the baker asks: "If I stand at this point on the low-poly surface and look outward, where do I hit the high-poly model?" Whatever it finds at the intersection point — a surface normal, an occlusion value, a curvature measurement — gets recorded into the output texture.

The Cage

The cage is an inflated version of the low-poly model that completely encloses the high-poly model. Rays originate at the cage surface and travel inward toward the low-poly surface (or vice versa, depending on the tool). The cage defines the search volume — any high-poly geometry outside the cage will not be captured.

Cage tuning is an art. Too tight, and protruding details (belt buckles, pouches, rivets) miss the cage entirely and leave holes in the bake. Too loose, and rays from one surface region leak into neighboring regions at concave junctions — the character's inner elbow picks up normals from the forearm, producing bizarre shading artifacts. Professional artists spend significant time adjusting cage offset per vertex, expanding it around complex areas while keeping it snug against flat regions.

Normal Map Baking

The workhorse bake output. For every texel, the baker records the high-poly surface normal in tangent space (relative to the low-poly surface). This captures every sculpted wrinkle, pore, groove, bolt head, and engraved logo as a per-pixel normal perturbation. At runtime, the fragment shader reads this map and uses the perturbed normal for lighting — the surface appears to have geometric detail it physically lacks.

GLSL — Applying a baked normal map
// In the fragment shader, after fetching the baked normal map
vec3 tangentNormal = texture2D(u_normalMap, v_uv).xyz * 2.0 - 1.0;

// Transform from tangent space to world space using TBN matrix
vec3 N = normalize(v_TBN * tangentNormal);

// Use N for all lighting calculations — the sculpted
// detail is now driving the shading on the low-poly mesh
float diff = max(dot(N, lightDir), 0.0);

AO Baking

Ambient occlusion baking fires hundreds of rays from each surface point into the upper hemisphere and measures what fraction escapes versus hitting nearby geometry. The result is a grayscale texture: white means fully exposed to the sky, black means deeply occluded. Crevices, undercuts, the space between fingers, the gap between a helmet and a head — all darken naturally. The baked AO captures detail from the high-poly mesh that real-time SSAO completely misses, and it costs zero GPU time at runtime because it is just a texture lookup.

Lightmap Baking

The most compute-intensive bake: full global illumination (radiosity, path tracing) computed offline and stored as a texture. Every light bounce, every color bleed from a red wall onto a white ceiling, every soft shadow under a table — all captured once and frozen into a map. The catch: lightmaps require a second, non-overlapping UV channel (UV2), because multiple mesh faces that share the same UV space in their diffuse texture must have unique lightmap UVs. Unreal's Lightmass and Unity's Progressive Lightmapper both automate UV2 generation and lightmap baking.

Blender Bake Workflow

Blender's bake system lives in Render Properties → Bake panel (Cycles renderer only). The workflow:

  1. Prepare meshes — High-poly and low-poly aligned in the same location. Low-poly must have clean UVs with no overlap.
  2. Create target image — In the Shader Editor, add an Image Texture node connected to nothing. Create a new blank image at your target resolution (2048×2048 or 4096×4096). This node must be selected (highlighted) — that is where Blender writes the bake output.
  3. Select high-poly, then Shift+select low-poly — The low-poly (active object) receives the bake; the high-poly is the source.
  4. Enable "Selected to Active" in the Bake panel — this tells Blender to ray-cast from the active mesh to the selected mesh.
  5. Set ray distance (or use cage) — Extrusion distance controls the cage inflation. Too low clips detail; too high introduces leaks.
  6. Choose bake type — Normal, Ambient Occlusion, Diffuse, Emit, etc.
  7. Click Bake — Watch the progress bar. A 4K normal bake typically takes 10–60 seconds depending on mesh complexity.
  8. Save the image — Blender does not auto-save baked images. Image → Save As. Use 16-bit PNG for normal maps to preserve precision.

The Substance Painter Pipeline

In production, Substance Painter is where baking meets texturing. Import the low-poly and high-poly, hit Bake Mesh Maps, and Substance generates normals, AO, curvature, world-space normals, position, and thickness maps simultaneously. These maps then drive smart materials: edge wear accumulates on convex edges (curvature map), dirt settles in crevices (AO map), and scratches follow surface topology (normal map). The artist paints material layers on top of this foundation, and the baked maps ensure that physically plausible detail distribution happens automatically.

Blender Bake Checklist — Quick Reference
  • Renderer set to Cycles (EEVEE does not support baking)
  • Low-poly mesh has non-overlapping UVs
  • Image Texture node selected (highlighted blue) in Shader Editor
  • Target image created at desired resolution (2K or 4K)
  • Selected to Active enabled in Bake panel
  • High-poly is selected; low-poly is active (select high first, then Shift+click low)

The Production Pipeline

Concept art → High-poly sculpt (ZBrush) → Retopology (low-poly, Maya/Blender) → UV unwrap → Bake (Substance/Blender) → Texture paint (Substance Painter) → Export (engine-specific format). This pipeline produces every character, weapon, vehicle, and prop in every modern game. The baking step is the hinge: it is the moment where millions of polygons of artistic intent collapse into a handful of textures that run at real-time speed. Understanding baking is understanding how the games industry actually works.

// 11 — Terrain, Grass & Nature

Open worlds need ground to stand on. Terrain rendering is one of the oldest problems in real-time graphics and one of the deepest — it touches vertex shaders, fragment shaders, instancing, LOD, and nearly every texture technique covered so far. A single terrain mesh might cover 16 square kilometers at centimeter-level detail, textured with a blend of grass, rock, sand, snow, and mud that transitions based on altitude, slope, moisture, and hand-painted artist overrides. The vertex shader shapes the ground. The fragment shader paints it. And thousands of instanced grass blades wave in simulated wind on top.

Heightmaps: Vertex Displacement

A heightmap is a grayscale texture where each pixel's brightness represents elevation. Black is the lowest point, white is the highest. The terrain mesh starts as a flat grid of vertices, and the vertex shader displaces each vertex upward by sampling the heightmap at that vertex's UV coordinate. A 1024×1024 heightmap on a 1024×1024 vertex grid produces over a million vertices of terrain — mountains, valleys, riverbeds — all defined by a single grayscale image.

GLSL — Heightmap vertex displacement
uniform sampler2D u_heightmap;
uniform float u_heightScale;
uniform float u_terrainSize;
uniform float texelSize;
uniform mat4 u_mvp;

attribute vec3 a_position;
varying vec3 v_normal;
varying vec2 v_uv;

void main() {
    vec2 uv = a_position.xz / u_terrainSize;
    float h = texture2D(u_heightmap, uv).r * u_heightScale;

    vec3 displaced = vec3(a_position.x, h, a_position.z);

    // Compute normal from heightmap gradient (central differences)
    float hL = texture2D(u_heightmap, uv + vec2(-texelSize, 0.0)).r;
    float hR = texture2D(u_heightmap, uv + vec2( texelSize, 0.0)).r;
    float hD = texture2D(u_heightmap, uv + vec2(0.0, -texelSize)).r;
    float hU = texture2D(u_heightmap, uv + vec2(0.0,  texelSize)).r;
    vec3 normal = normalize(vec3(hL - hR, 2.0 * texelSize, hD - hU));

    v_normal = normal;
    v_uv = uv;
    gl_Position = u_mvp * vec4(displaced, 1.0);
}

The normal computation deserves a note: we are not using the mesh's geometric normals (which would just point straight up on a flat grid). Instead, we sample the heightmap at four neighboring texels and compute the gradient — the slope of the terrain at this point. This gives us a per-vertex normal that accurately follows the heightmap's contours, producing correct lighting on hills and valleys.

Splatmaps: Multi-Layer Blending

A splatmap is an RGBA texture where each channel controls the blend weight of a terrain material layer. Red channel = grass weight, Green = rock, Blue = sand, Alpha = snow. The fragment shader samples all four material textures and blends them according to the splatmap weights.

GLSL — Splatmap terrain blending
uniform sampler2D u_splatmap;
uniform sampler2D u_grassTex, u_rockTex, u_sandTex, u_snowTex;

vec4 splat = texture2D(u_splatmap, v_uv);

// Each material is tiled at its own frequency
vec3 grass = texture2D(u_grassTex, v_uv * 32.0).rgb;
vec3 rock  = texture2D(u_rockTex,  v_uv * 24.0).rgb;
vec3 sand  = texture2D(u_sandTex,  v_uv * 40.0).rgb;
vec3 snow  = texture2D(u_snowTex,  v_uv * 20.0).rgb;

// Blend by splatmap weights (should sum to ~1.0)
vec3 terrain = grass * splat.r + rock * splat.g
             + sand  * splat.b + snow * splat.a;

The artist paints the splatmap directly in the engine's terrain editor — switching between a "grass brush" and a "rock brush" is really just painting into the R or G channel of the splatmap texture. Height-based rules can automate the process: snow above 2000m, rock on slopes steeper than 45°, grass everywhere else. The splatmap is the artist's override layer on top of those procedural rules.

For more than four layers, you add more splatmaps. Two RGBA splatmaps give you eight layers. Three give you twelve. Each additional splatmap adds one texture sample in the fragment shader — cheap, but the material texture samples add up fast. With eight layers, you are sampling eight tiling material textures plus two splatmaps per pixel. This is why terrain rendering is often the most texture-bandwidth-intensive part of a game's rendering budget.

Triplanar Projection

Standard UV mapping breaks on terrain. Steep cliff faces stretch the texture into ugly smears because the UV coordinates — derived from the XZ plane — compress vertically on near-vertical surfaces. Triplanar projection solves this by sampling the texture three times (along X, Y, and Z axes) and blending based on the surface normal:

GLSL — Triplanar mapping
vec3 triplanar(sampler2D tex, vec3 worldPos, vec3 normal) {
    // Blend weights from absolute normal components
    vec3 w = abs(normal);
    w = pow(w, vec3(4.0)); // sharpen the blend transition
    w /= w.x + w.y + w.z;  // normalize to sum to 1

    // Sample texture projected along each major axis
    vec3 xProj = texture2D(tex, worldPos.yz * 0.1).rgb;
    vec3 yProj = texture2D(tex, worldPos.xz * 0.1).rgb;
    vec3 zProj = texture2D(tex, worldPos.xy * 0.1).rgb;

    // Blend by normal-derived weights
    return xProj * w.x + yProj * w.y + zProj * w.z;
}

On a flat ground plane (normal pointing up), the Y projection dominates — standard top-down texturing. On a cliff face (normal pointing sideways), the X or Z projection dominates, and the texture reads from a side-on projection that has no vertical stretch. The transition between projections is smooth and invisible. The cost is 3× the texture reads, but terrain is typically fill-rate bound, not bandwidth bound, so the tradeoff is worth it.

Grass Instancing

Grass in modern games is not painted onto the terrain texture — it is geometry. Thousands of small quads or triangle strips, each representing a single grass blade, are placed across the terrain via instanced rendering. One draw call can render 10,000 blades because each instance shares the same blade mesh but uses a per-instance buffer for position, rotation, height, and color variation. The GPU's instancing hardware was essentially designed for this use case.

Blade placement is driven by the splatmap: only spawn grass where the grass channel is above a threshold. Density falls off with camera distance. Blades facing away from the camera can be culled. A density map texture provides additional control — the artist can paint "no grass here" zones for paths, roads, and rocky outcrops. The entire placement system runs on the CPU each frame (or is precomputed per terrain chunk), producing a per-instance buffer that the GPU consumes in a single instanced draw call.

Wind: The Vertex Shader IS the Wind

Grass animation does not use skeletal animation or physics simulation. It is pure vertex shader math. Each grass blade's vertices are displaced by a wind function — a combination of sine waves at different frequencies and amplitudes that creates the illusion of flowing wind. The displacement is strongest at the blade tip (top vertex) and zero at the root (bottom vertex, anchored to the ground). A simple sin(worldPos.x + time * speed) creates uniform waves; add a second sine at a different frequency for visual complexity; add a random gust term keyed to time for realism.

LOD for Terrain

A terrain mesh covering 16 km² at 1-meter vertex spacing would have 16 billion vertices. Obviously impossible. Terrain LOD systems render nearby terrain at full resolution and distant terrain at progressively coarser resolution. Two approaches dominate:

Clipmaps (used in the original Far Cry and many games since) maintain concentric rings of terrain mesh around the camera. The innermost ring has full vertex density. Each successive ring doubles the grid spacing, halving the vertex count. As the camera moves, the rings shift with it, and only the edges need updating. The result is constant vertex count regardless of terrain size — a 100km×100km world has the same vertex budget as a 1km×1km one.

CDLOD (Continuous Distance-Dependent Level of Detail) refines a quadtree of terrain patches. Each node in the quadtree represents a terrain tile at a specific resolution. Near the camera, tiles are subdivided into four higher-resolution children. Far from the camera, tiles are merged into coarser parents. A morphing term blends between LOD levels to prevent visible vertex popping during transitions. Both techniques exploit the same insight: the further terrain is from the camera, the fewer pixels it occupies, and the fewer vertices it needs.

Procedural Worlds

Heightmaps do not have to be painted by hand. Noise functions (Perlin, Simplex, Worley) generate infinite terrain procedurally. Erosion simulations carve realistic river valleys and sediment deposits. Splatmaps can be derived from slope and altitude automatically. The entire world — geometry, texturing, vegetation placement — can emerge from algorithms. No Man's Sky generates 18 quintillion planets this way. The texture techniques are the same; only the authorship changes from human to machine.

// 12 — Stylized & Creative Textures

Now that you know the rules, break them. Everything up to this point has been in service of photorealism — physically based rendering, accurate normals, correct ambient occlusion. But some of the most beloved visual styles in gaming explicitly reject realism. Cel shading, hand-painted textures, pixel art, and watercolor effects all require understanding the same rendering pipeline, then deliberately subverting it to achieve an artistic vision. The technical foundation is identical; only the intent changes.

Cel Shading / Toon Shading

Cel shading quantizes the smooth lighting gradient into discrete bands. Instead of a continuous diffuse value from 0.0 to 1.0, you snap it to 2–4 levels: bright, medium, shadow. The boundary between bands creates the sharp, illustrative look of hand-drawn animation. The implementation is trivially simple:

GLSL — Basic cel shading
float celShade(float NdotL) {
    // 3-band toon ramp
    if (NdotL > 0.7) return 1.0;   // lit
    if (NdotL > 0.3) return 0.55;  // mid-tone
    return 0.2;                     // shadow
}

// Apply to base color
vec3 color = baseColor * celShade(diff);

// Add rim light for contour emphasis
float rim = 1.0 - max(dot(N, viewDir), 0.0);
color += rimColor * pow(rim, 3.0) * 0.4;

// Hard specular highlight (binary, not smooth falloff)
color += step(0.95, spec) * vec3(0.5);

The rim light — bright where the surface normal faces perpendicular to the camera — acts as the "outline" that gives cel-shaded characters their cartoon contour. Combined with a geometry-based outline pass (render back faces slightly scaled outward in a solid dark color), you get the full comic-book look that Jet Set Radio, Borderlands, and Breath of the Wild made iconic.

Toon Ramps: The 1D Lookup

A more flexible approach replaces the if/else chain with a 1D texture lookup. Compute the diffuse lighting value (0.0–1.0), use it as the U coordinate to sample a 1D gradient texture, and that texture defines the tonal response. An artist can paint any ramp they want: two sharp bands for anime, a warm orange-to-purple gradient for sunset mood, a noisy gradient for sketchy linework. Changing the entire lighting character of the scene means swapping a 256×1 pixel texture.

GLSL — Toon ramp lookup
uniform sampler2D u_toonRamp; // 256x1 gradient texture

float NdotL = max(dot(N, L), 0.0);
vec3 rampColor = texture2D(u_toonRamp, vec2(NdotL, 0.5)).rgb;
vec3 finalColor = baseColor * rampColor;

The toon ramp is one of the most powerful tiny ideas in stylized rendering. The ramp texture is the artist's direct control over the relationship between light and shade. Smooth gradients produce soft, dreamy looks. Hard step functions produce crisp anime. Warm-to-cool color shifts produce painterly atmospheres. The math is a single texture lookup; the creative range is infinite.

Hand-Painted Textures

The art style of World of Warcraft, Dota 2, and Team Fortress 2 uses hand-painted diffuse textures that bake lighting information directly into the color map. Shadows, highlights, ambient occlusion, and even a suggestion of specular reflection are painted by the artist into the texture itself. The real-time shader does minimal lighting — often just a simple Lambert diffuse or even flat shading — because the texture already encodes the lighting direction the artist intended.

This approach decouples the visual style from the lighting environment entirely. A hand-painted character looks correct regardless of the scene's light direction because the "lighting" is part of the texture, not computed from the scene. The tradeoff: the character always looks like it is lit from the same direction, and dynamic time-of-day changes do not affect the painted shadows. This is an intentional artistic choice, not a limitation — it preserves the illustrator's vision regardless of engine conditions.

Hand-painting also has a practical advantage: it ages well. PBR materials depend on complex shading models that change between engine versions; a hand-painted texture rendered with gl_FragColor = texture2D(diffuse, uv); looks identical on every engine, every API, every GPU generation. World of Warcraft launched in 2004 and its character textures still look stylistically coherent today — because the style lives in the paint, not in the shader.

Pixel Art Upscaling

Pixel art in a 3D engine or at higher resolutions requires nearest-neighbor filtering to preserve the crisp, blocky pixels — bilinear filtering would blur the carefully placed pixel boundaries into mush. But nearest-neighbor alone creates jagged edges when the pixel art is rotated or scaled to non-integer multiples. Advanced techniques like rotated grid supersampling and edge-detection-based antialiasing preserve the pixel art aesthetic while reducing the worst aliasing artifacts. The fragment shader can quantize UVs before sampling:

GLSL — Pixel art quantization
uniform float u_pixelSize; // e.g., 1.0 / 32.0 for 32x32 art

vec2 pixelUV = floor(v_uv / u_pixelSize) * u_pixelSize;
vec4 color = texture2D(u_texture, pixelUV);

// Optional: add per-pixel color palette snapping
// Quantize each channel to N levels for retro feel
color.rgb = floor(color.rgb * 8.0) / 8.0;

Watercolor & Painterly Effects

Painterly rendering simulates the imperfection of physical media. Brush stroke textures are overlaid on the rendered image, color bleeds across edges using blur kernels that respect luminance boundaries, and noise textures break up the clean perfection of digital rendering. Kubelka-Munk pigment mixing models simulate how watercolors blend on wet paper — blue and yellow make green through subtractive mixing, not the additive cyan of digital RGB blending. These effects typically run as post-processing passes, but the underlying textures (brush alpha maps, paper grain textures, noise fields) are authored and sampled using exactly the same pipeline as any other texture.

The Okami approach is particularly instructive: the entire scene is rendered normally, then a post-process applies a paper texture overlay, adds ink-wash edges using Sobel edge detection on the depth buffer, and desaturates distant objects. The 3D engine does the heavy lifting of geometry and perspective; the post-process transforms the result into the artist's chosen medium. The textures that define the medium (paper grain, ink spatter, brush stroke alpha) are as important as the textures that define the objects.

// 13 — Texture Hacks & Dark Arts

Every shipping game contains texture tricks that no textbook covers. These are the dark arts — techniques born from deadlines, memory budgets, and the creative desperation of graphics programmers who need to squeeze one more visual feature out of hardware that is already at its limit. Each trick exploits some property of how the GPU handles textures to get more visual bang per byte, per sample, or per draw call.

Channel Packing

A texture has four channels: R, G, B, A. A roughness map is grayscale. A metallic map is grayscale. An AO map is grayscale. Three separate textures, each using only one channel of four — wasting 75% of the bandwidth. Channel packing combines them into one: metallic in R, roughness in G, AO in B. One texture sample, three material properties. Unreal Engine's "ORM" texture (Occlusion, Roughness, Metallic) is exactly this pattern.

GLSL — Channel packing (ORM)
// One texture, three material properties
vec4 orm = texture2D(u_ormMap, v_uv);
float ao        = orm.r;
float roughness = orm.g;
float metallic  = orm.b;

// Compare with the wasteful approach:
// float ao        = texture2D(u_aoMap,        v_uv).r;  // 3 texture samples
// float roughness = texture2D(u_roughnessMap, v_uv).r;  // 3 cache line fetches
// float metallic  = texture2D(u_metallicMap,  v_uv).r;  // 3x the VRAM

You can go further. The alpha channel of the normal map (which normally stores just 1.0) can hold a height map for parallax. The blue channel of a tangent-space normal map can be reconstructed from R and G (since the normal is unit length and Z is always positive in tangent space), freeing blue for another parameter. Channel packing is the texture artist's equivalent of struct packing in systems programming — fitting more data into the same memory footprint through careful layout.

GLSL — Reconstruct Z from two-channel normal map
vec2 xy = texture2D(u_normalMap, v_uv).rg * 2.0 - 1.0;
float z = sqrt(max(0.0, 1.0 - xy.x * xy.x - xy.y * xy.y));
vec3 N = vec3(xy, z); // Full tangent-space normal from 2 channels

Detail Textures

A 2K texture looks fine at 5 meters. At 0.5 meters, you see blurry texels. Rather than shipping a 16K texture (which would consume enormous VRAM), add a detail texture — a small, high-frequency pattern (fabric weave, concrete grain, metal scratches) that tiles at a much higher frequency and is blended in at close range:

GLSL — Detail texture blending
vec3 base   = texture2D(u_baseTex,   v_uv).rgb;
vec3 detail = texture2D(u_detailTex, v_uv * 16.0).rgb; // tiles 16x faster

// Blend detail in at close range (distance-based fade)
float detailFade = smoothstep(5.0, 1.0, v_viewDist);
vec3 color = base * mix(vec3(1.0), detail * 2.0, detailFade);

The * 2.0 maps the detail's mid-gray (0.5) to 1.0 (no change), darker values darken the base, lighter values brighten it. This is overlay blending. The detail texture adds visual complexity at close range while consuming almost no additional VRAM — a 256×256 tiling detail texture can make a 2K base texture look like 16K at arm's length.

Hex Tiling

Repeating textures tile. You see it in every game: the same stone pattern every 2 meters, the same grass patch in a grid. The human eye is extraordinarily good at detecting repetition — even randomized patterns reveal their period at sufficient distance. Hex tiling (popularized by Ben Golus's "Practical Realtime Hex-Tiling" method) breaks the repetition by dividing UV space into a hexagonal grid. Each hex cell rotates and offsets the texture by a random amount derived from the cell's position. Adjacent cells blend smoothly at their boundaries. The visual result: the same 512×512 texture appears to cover a huge area without visible repetition.

GLSL — Hex tiling core
// Hash function for per-cell randomness
float hash(vec2 p) {
    return fract(sin(dot(p, vec2(127.1, 311.7))) * 43758.5453);
}

vec3 hexTile(sampler2D tex, vec2 uv) {
    // Hex grid dimensions (1:sqrt(3) aspect ratio)
    vec2 r = vec2(1.0, 1.732);
    vec2 h = r * 0.5;

    // Two candidate hex centers (offset grids)
    vec2 a = mod(uv, r) - h;
    vec2 b = mod(uv - h, r) - h;

    // Pick the closer center
    vec2 gv = dot(a, a) < dot(b, b) ? a : b;
    vec2 id = uv - gv;

    // Random rotation per cell
    float angle = hash(id) * 6.2832;
    float c = cos(angle), s = sin(angle);
    vec2 rotUV = mat2(c, -s, s, c) * gv;

    return texture2D(tex, rotUV + 0.5).rgb;
}

Texture Bombing

A cousin of hex tiling, texture bombing scatters copies of a pattern (leaves, pebbles, cracks, rust spots) at random positions, scales, and rotations across a surface. Each "bomb" is a small texture blended onto the base using alpha. The placement is procedural — derived from a hash of the grid cell — so no per-instance data is needed. The effect looks hand-placed but runs at the cost of a few extra texture samples per pixel. The key is the splat: each cell contributes a decal-like patch whose alpha blends it onto the accumulated result.

Decal Projection

Bullet holes, blood splatter, tire marks, graffiti — any surface detail added dynamically at runtime. A decal is a texture projected onto arbitrary geometry using a projection matrix, exactly like a spotlight projects its cookie texture. The decal system renders a box volume into the G-buffer, reads the scene depth to reconstruct world position, and projects the decal texture onto that position. The decal does not need to know anything about the underlying mesh topology — it works on any geometry that intersects its projection volume.

GLSL — Deferred decal projection
// In the decal's fragment shader
vec3 worldPos = reconstructWorldPos(gl_FragCoord.xy, depthBuffer);

// Transform world position into decal's local space
vec3 localPos = (u_decalInvMatrix * vec4(worldPos, 1.0)).xyz;

// Reject pixels outside the decal box [-0.5, 0.5]^3
if (any(greaterThan(abs(localPos), vec3(0.5)))) discard;

// Use local XY as UV for the decal texture
vec2 decalUV = localPos.xy + 0.5;
vec4 decalColor = texture2D(u_decalTex, decalUV);

// Alpha blend onto G-buffer
gl_FragColor = vec4(decalColor.rgb, decalColor.a);

Deferred decals are one of the most elegant systems in modern rendering. They require no mesh modification, no UV recalculation, and no awareness of the underlying geometry. Drop a bullet-hole decal box in the scene, and it appears on any surface it overlaps — walls, floors, props, characters. The depth buffer does all the spatial reasoning.

Distance Field Textures

Valve's 2007 paper introduced signed distance field (SDF) textures for rendering crisp text and UI elements at any resolution. Instead of storing the shape as pixels (which blur when magnified), the texture stores the signed distance to the nearest edge at each texel. Positive values are inside the shape, negative are outside, and the zero-crossing is the edge. At render time, a simple smoothstep around the zero-crossing produces a perfectly crisp edge regardless of magnification. A 64×64 SDF texture can render text as crisply as a 4096×4096 bitmap — at 1/4000th the memory.

GLSL — SDF text rendering
float dist = texture2D(u_sdfTexture, v_uv).r;

// Crisp edge with screen-space antialiasing
float edgeWidth = fwidth(dist) * 0.5;
float alpha = smoothstep(0.5 - edgeWidth, 0.5 + edgeWidth, dist);

// Optional: outline at a different distance threshold
float outlineAlpha = smoothstep(0.4 - edgeWidth, 0.4 + edgeWidth, dist);
vec3 color = mix(outlineColor, fillColor, alpha);
gl_FragColor = vec4(color, outlineAlpha);

The fwidth() call is the key: it computes the screen-space rate of change of the distance field, allowing the antialiasing width to adapt automatically as the text is zoomed in or out. Multi-channel SDF (MSDF) extends this to preserve sharp corners that single-channel SDF rounds off. The technique is now standard for text rendering in games and UI frameworks — it is how most game engines render their debug text, HUD elements, and in-world signage.

Dark Knowledge

These tricks are rarely documented in official API guides or textbook chapters. They live in GDC presentations, blog posts, forum threads, and the comment blocks of shipping engine code. Channel packing saves 66% of texture memory. Hex tiling makes a 512×512 texture look like 4K. SDF textures compress text storage by 4000×. Detail textures make 2K textures look like 16K. Each trick is small, but stacked together they are the difference between a game that ships on target hardware and one that does not. The dark arts are the practical core of real-time graphics.

// 14 — WebGPU & The Future

Everything in this deep dive has been WebGL — an API designed in 2011, mapping to OpenGL ES 2.0, limited to a single thread, no compute shaders, no bindless textures, no modern GPU features. WebGPU changes everything. It is the web's first modern graphics API, mapping to Vulkan, Metal, and D3D12 under the hood. For texture work specifically, WebGPU brings compute shader texture generation, proper texture arrays, storage textures for read-write access, and WGSL — a shader language designed for the web from scratch.

WGSL Texture Sampling

WGSL replaces GLSL's texture2D(sampler, uv) with an explicit separation of texture and sampler objects. This is not just syntax — it reflects the modern GPU API model where textures and samplers are independent resources that can be combined freely. One texture, multiple samplers with different filtering modes. One sampler, shared across all textures. The binding model is explicit and auditable.

WGSL — Texture sampling
@group(0) @binding(0) var myTexture: texture_2d<f32>;
@group(0) @binding(1) var mySampler: sampler;

@fragment
fn fs_main(@location(0) uv: vec2f) -> @location(0) vec4f {
    // Explicit texture + sampler combination
    let color = textureSample(myTexture, mySampler, uv);

    // Explicit LOD sampling (required in compute shaders)
    let mipped = textureSampleLevel(myTexture, mySampler, uv, 2.0);

    // Direct texel fetch — no filtering, integer coordinates
    let texel = textureLoad(myTexture, vec2u(128, 64), 0);

    return color;
}

The textureSampleLevel function is critical for compute shaders. In a fragment shader, the GPU can automatically compute the mip level from screen-space UV derivatives. In a compute shader, there are no screen-space derivatives — you are not rasterizing triangles. You must specify the mip level explicitly. This distinction trips up every developer moving texture code from fragment to compute for the first time.

Compute Shader Texture Generation

WebGPU's compute shaders unlock procedural texture generation at GPU speed. Instead of computing Perlin noise, Voronoi cells, or FBM in JavaScript on the CPU (which takes seconds for a 2K texture), you dispatch a compute shader that generates the texture in parallel across thousands of GPU threads. A 2048×2048 procedural texture that takes 800ms on the CPU completes in under 2ms on the GPU.

WGSL — Compute shader: Voronoi texture
@group(0) @binding(0) var output: texture_storage_2d<rgba8unorm, write>;

fn hash22(p: vec2f) -> vec2f {
    var q = vec2f(dot(p, vec2f(127.1, 311.7)),
                   dot(p, vec2f(269.5, 183.3)));
    return fract(sin(q) * 43758.5453);
}

fn voronoi(uv: vec2f) -> f32 {
    let cell = floor(uv);
    let frac_ = fract(uv);
    var minDist = 1.0;
    for (var y = -1; y <= 1; y++) {
        for (var x = -1; x <= 1; x++) {
            let neighbor = vec2f(f32(x), f32(y));
            let point = hash22(cell + neighbor);
            let diff = neighbor + point - frac_;
            minDist = min(minDist, length(diff));
        }
    }
    return minDist;
}

@compute @workgroup_size(16, 16)
fn main(@builtin(global_invocation_id) id: vec3u) {
    let dims = textureDimensions(output);
    if (id.x >= dims.x || id.y >= dims.y) { return; }

    let uv = vec2f(id.xy) / vec2f(dims);
    let v = voronoi(uv * 8.0);
    let color = vec4f(v, v * 0.8, v * 0.6, 1.0);

    textureStore(output, id.xy, color);
}

This is not just faster — it enables real-time procedural textures that update every frame. Animated lava, growing rust, dynamic weather maps, terrain erosion simulations, water ripple propagation — all become feasible when texture generation runs at GPU compute speed. The texture is no longer a static asset loaded from disk; it is a living, computed surface that responds to game state in real time.

Bindless Textures

Traditional rendering requires binding textures to specific slots before each draw call. With 16–32 texture slots and a scene containing 10,000 unique materials, you need thousands of draw calls just to switch textures. Bindless textures (Vulkan descriptor indexing, D3D12 descriptor heaps, Metal argument buffers) let you upload all textures into a massive array and index them by integer in the shader. A single draw call can access every texture in the scene:

WGSL — Bindless-style texture array
@group(0) @binding(0) var textures: binding_array<texture_2d<f32>, 1024>;
@group(0) @binding(1) var s: sampler;

@fragment
fn fs_main(
    @location(0) uv: vec2f,
    @location(1) @interpolate(flat) texIndex: u32
) -> @location(0) vec4f {
    // Sample any texture by index — no rebinding needed
    return textureSample(textures[texIndex], s, uv);
}

This is the architectural shift that makes Nanite-style rendering possible: millions of tiny triangles, each potentially from a different material, rendered in a single GPU dispatch without texture binding overhead. The old model of "sort by material, batch by texture" dissolves when the shader can access any texture by index.

Neural Textures

Machine learning is beginning to replace hand-authored textures. Neural materials use small neural networks that take a UV coordinate and viewing angle as input and output the surface appearance directly, encoding complex view-dependent effects (subsurface scattering, anisotropic highlights, iridescence) that are difficult to capture in traditional texture maps. NVIDIA's neural texture compression achieves 10–100× compression ratios by training a tiny MLP (multi-layer perceptron) to reproduce a texture block, then running inference at sample time. The "texture" is no longer a grid of pixels — it is a set of neural network weights.

Neural radiance caching uses neural networks to approximate global illumination at runtime. Instead of baking lightmaps or computing expensive screen-space reflections, a neural cache is trained on the scene's lighting and queried per pixel. The texture lookup of the future may not be a memory read at all — it may be a forward pass through a 4-layer network running on tensor cores. The distinction between "texture sampling" and "neural inference" is already blurring.

Path Tracing Kills Baked Maps

Unreal Engine 5's Lumen and hardware ray tracing (RTX, RDNA2+) are making baked lightmaps obsolete for an increasing class of scenes. When global illumination can be computed at real-time speed — even at reduced resolution with temporal accumulation and denoising — the motivation to bake evaporates. Dynamic time of day, destructible environments, moving light sources, and reflective surfaces that actually reflect the scene become trivial when lighting is computed, not stored.

Normal maps and material textures remain essential — they describe the surface, not the lighting. Albedo is an intrinsic property of the material. Roughness is an intrinsic property. These need to be authored regardless of how the lighting is computed. But baked AO, baked shadows, and lightmaps are losing their monopoly. The 30-minute bake that was previously non-negotiable is increasingly replaced by a 16ms-per-frame computation that adapts dynamically to everything happening in the scene.

Nanite and Virtual Geometry

Nanite renders cinema-quality meshes (millions of polygons per object) in real time by streaming geometry the same way virtual texturing streams texels. When a surface has enough geometric detail to capture every bump, groove, and scratch at the polygon level, the need for normal maps diminishes. A 50-million-polygon statue does not need a normal map — the actual geometry captures what the normal map was faking. As GPU triangle throughput continues to scale, the boundary between "geometry detail" and "texture detail" shifts. The elaborate ecosystem of normal maps, displacement maps, and parallax tricks may eventually simplify as raw geometry takes over the job of surface shape.

But textures are not going away. Albedo is color. Roughness is how light scatters off the surface. Metallic-ness determines Fresnel behavior. These are intrinsic material properties that no amount of geometric detail can replace. A billion-polygon marble statue still needs to know it is marble, not plastic — and that information lives in a texture.

The Surface Endures

The tools will change. GLSL will yield to WGSL. Baked lightmaps will yield to real-time GI. Normal maps may yield to raw geometry. Neural networks may replace pixel grids. But the fundamental problem — making a surface look like something — is eternal. Whether the GPU reads a texel from VRAM, infers an appearance from a neural network, or shades a micro-polygon from a 50-million-triangle mesh, it is still answering the same question: "What does this point on this surface look like from this angle in this light?" Textures are not a technology. They are the answer to that question. And that question will be asked for every pixel, in every frame, of every 3D application, forever.
