NAVIGATION

Localization, mapping, and planning under noise. We are going to build the stack that lets a robot know where it is, decide where to go, and recover when the world refuses to stay still._

01 -- Frames Before Motion

A robot does not move through a world. It moves through a stack of coordinate frames that are only temporarily pretending to agree with each other. base_link is where the chassis thinks it is. odom is where dead reckoning thinks it is. map is where the long-term world model thinks it is. The first hard problem in navigation is not path planning. It is keeping those stories coherent long enough to act.

If this sounds abstract, watch what breaks when you skip it. Mount a lidar five centimeters off-center and forget the transform: walls appear to curve. Apply wheel odometry in the wrong frame: the robot drifts sideways while your logs insist it is going straight. Feed a controller target points in map frame while the controller expects body frame: the robot spins like it has lost its mind. This is why mature robotics stacks make transforms first-class.

SE(2)struct Pose2 {
  x: f64,
  y: f64,
  theta: f64,
}

fn compose(a: Pose2, b: Pose2) -> Pose2 {
  let c = a.theta.cos();
  let s = a.theta.sin();
  Pose2 {
    x: a.x + c * b.x - s * b.y,
    y: a.y + s * b.x + c * b.y,
    theta: a.theta + b.theta,
  }
}

Cross-reference: the numerical stability of repeated pose composition lives in the same world as floating-point error. Your map will inherit every tiny lie.

02 -- Odometry Is a Necessary Lie

Odometry is the robot's first draft of reality. Wheel encoders count rotations. IMUs integrate angular velocity and acceleration. Legged robots integrate contact events and joint encoders. Every one of these estimates is local, cheap, and doomed to drift. But you still need them because they are the only signals fast enough to close the control loop at tens or hundreds of hertz.

The right mental model is not "odometry is wrong, so we should replace it." The right model is "odometry is high-rate but low-trust." It gives the controller continuity between slower corrections from lidar, vision, or loop closure. Navigation stacks that feel stable treat odometry as a prediction and global localization as a correction.

The ugliest odometry bug is silent frame mismatch. The second ugliest is assuming wheel slip is a rare event. On polished concrete, carpet transitions, gravel, or ramps, slip is not an exception. It is the environment speaking.

03 -- Maps Are Probability, Not Geometry

Humans imagine maps as clean line drawings. Robots do not get that luxury. They see a world through sparse, noisy, delayed measurements. The simplest useful representation is the occupancy grid: carve the world into cells and assign each one a belief about whether it is occupied. This is crude, memory hungry, and absurdly effective.

Why not just store line segments or meshes? Because navigation needs more than surface geometry. It needs uncertainty and cost. An occupancy grid naturally extends into a costmap: inflate obstacles by the robot radius, then layer in semantic penalties like "slow near glass walls" or "avoid this pallet lane during peak traffic." The map that drives motion is not the map you render in a slide deck.

Log Odds// each cell stores log-odds rather than raw probability
L_t(cell) = L_{t-1}(cell) + inverse_sensor_model(z_t, cell) - L_0

// convert back when needed
P(occupied) = 1 - 1 / (1 + exp(L_t))

Static Map

Great for known warehouses, offices, and labs. Cheap to plan on. Bad at handling chairs, carts, and humans unless paired with a dynamic obstacle layer.

Rolling Costmap

Tracks the local world around the robot at high frequency. Short memory. Excellent for obstacle avoidance. Terrible as a long-term truth source by itself.

Cross-reference: occupancy grids turn sensor streams into mutable arrays. The memory layout questions start to rhyme with buffers the moment your map becomes large enough to thrash cache.

04 -- Sensor Fusion Means Structured Skepticism

A navigation stack wins by trusting each sensor only where it deserves trust. Encoders are great until the wheels slip. IMUs are great until bias accumulates. Lidar is great until smoke, glass, or rain. Cameras are great until the lighting changes or the hallway has no texture. Sensor fusion is not averaging. It is a disciplined argument about what to believe right now.

In practice this often means a state estimator: an extended Kalman filter, an unscented filter, or a factor graph back end. The estimator maintains a state vector and covariance, predicts forward with motion, and corrects with measurements. The mathematics matters, but the engineering lesson matters more: every measurement must declare not just a value, but confidence.

EKF Step// predict
x_hat = f(x_prev, u)
P_hat = F * P_prev * F^T + Q

// update
y = z - h(x_hat)
S = H * P_hat * H^T + R
K = P_hat * H^T * inverse(S)
x = x_hat + K * y
P = (I - K * H) * P_hat
Engineering Rule

If you cannot explain where the covariance numbers came from, you are not doing sensor fusion. You are decorating guesswork with matrices.

05 -- Localization: Where Am I, Really?

Localization takes a map and a stream of observations and asks which pose makes those observations most plausible. In a tidy corridor this seems easy. Then you deploy into a symmetric warehouse aisle and discover that half the world looks identical. Localization is not a single estimate. It is a probability distribution over plausible poses, sometimes broad, sometimes multimodal, sometimes embarrassingly wrong.

This is why particle filters remain so popular. Instead of forcing a single Gaussian belief, they track many hypotheses at once. Predict each particle forward using motion. Score it against the latest scan. Resample toward the plausible regions. When the robot gets kidnapped and moved across the building, a well-tuned particle filter can actually recover. A brittle point estimate cannot.

Particle Filter// particles carry pose + weight
for p in particles:
  p.pose = sample_motion_model(p.pose, odom_delta)
  p.weight = scan_likelihood(map, p.pose, laser_scan)

normalize(particles)
particles = resample_low_variance(particles)

06 -- SLAM Is What Happens When the Map Also Moves

Localization assumes a map already exists. Simultaneous localization and mapping throws away that comfort. The robot must estimate its pose while building the map it needs in order to estimate its pose. This circular dependency is why SLAM literature looks like a small religion. But the core idea is straightforward: accumulate relative constraints, then optimize them jointly.

Front ends extract local constraints: scan matching, visual features, inertial preintegration. Back ends solve the global consistency problem with pose graphs or bundle adjustment. The dramatic moment is loop closure. You revisit a place from twenty minutes ago, recognize it, and suddenly a long warped corridor snaps into alignment. Everything before that was provisional.

Pose Graph// nodes are robot poses, edges are relative constraints
minimize sum over edges(i, j):

// solve iteratively with Gauss-Newton / Levenberg-Marquardt

07 -- Global Planning Is Search Over Cost

Once the robot knows roughly where it is, global planning solves a simpler problem: from this pose to that pose, through which free cells should we travel? The most common answer is still some flavor of graph search over a costmap. A* works because the world is mostly boring, and a good heuristic keeps the boredom cheap.

The real engineering choice is not A* versus not A*. It is what cost you search over. Euclidean distance alone produces paths that scrape walls and thread absurdly tight gaps. Add obstacle inflation, lane preferences, turn penalties, semantic keep-out zones, and suddenly the planner begins to look less like geometry and more like policy.

A*f(n) = g(n) + h(n)

g(n): cost from start to n
h(n): heuristic estimate to goal

Typical navigation cost:
g = distance
  + obstacle_inflation_penalty
  + turn_penalty
  + terrain_penalty
  + semantic_zone_penalty

08 -- Local Planning Is About Dynamics and Panic

A global path is an intention, not a control signal. The local planner must turn that path into actual velocities while reacting to obstacles that were not in the static map. This is where robots stop looking like search algorithms and start looking like physical bodies with momentum, wheel slip, blind spots, and braking distance.

For differential drive robots, Dynamic Window Approach is still common: sample feasible linear and angular velocities, roll them forward briefly, score them for goal progress, path alignment, and obstacle clearance, then pick the best. More sophisticated systems use model predictive control, but the shape of the problem stays the same. You are optimizing a short future while the long future stays fuzzy.

DWA Style Scoringscore(v, w) =
  alpha * heading_alignment(v, w) +
  beta  * obstacle_clearance(v, w) +
  gamma * forward_velocity(v, w) -
  delta * oscillation_penalty(v, w)

choose argmax over dynamically feasible (v, w)
Common Failure

The robot freezes not because planning failed, but because every candidate trajectory looks slightly bad and the scoring function quietly prefers paralysis. Humans call this "being careful." Warehouses call it downtime.

09 -- The Controller Lives in the Fast Loop

Planning can run at 1 to 20 Hz. Control usually cannot. The low-level controller consumes a target trajectory or pose error and produces wheel torques, motor commands, or steering angles at much higher rate. This is where the stack touches hardware, and hardware is far less patient than your simulator.

A classic stack looks like this: estimator publishes pose, global planner publishes a coarse path, local planner publishes a short horizon trajectory, controller tracks that trajectory, motor firmware closes the inner torque or velocity loop. Each layer runs at its own cadence. Good systems make these rate mismatches explicit instead of pretending the whole robot ticks together.

Trajectory Tracking// controller works in robot frame
error_x = cos(theta) * dx + sin(theta) * dy
error_y = -sin(theta) * dx + cos(theta) * dy
error_theta = wrap(goal_theta - theta)

v = kx * error_x
w = ky * error_y + kth * error_theta

The scheduling problem here starts to resemble systems work in concurrency. Estimation, planning, control, and sensor IO are separate loops with different deadlines and failure costs.

10 -- Latency Turns a Good Stack Into a Bad Robot

Navigation bugs often masquerade as algorithm bugs when they are really timing bugs. The lidar scan is stamped late. The camera pipeline buffers three frames. The planner uses a costmap from 180 milliseconds ago. The controller happily tracks a path that was valid for a world that no longer exists. Everything looks mathematically reasonable in isolation. The robot still clips the corner.

This is one reason distributed robotics feels harder than desktop robotics. The moment your sensors, estimator, planner, and supervisor live on different processes or machines, you need synchronized clocks, explicit message timestamps, bounded queues, and back-pressure. Otherwise the robot becomes an accidental historian.

Timing Budget

That is why robotics middleware obsesses over timestamps. It is not bureaucracy. It is causal hygiene. The moment you move navigation over Ethernet, Wi-Fi, or a fleet network, you inherit the same queuing and transport problems you already know from networking.

11 -- Recovery Behavior Is Part of Navigation

The clean academic diagram ends at "follow the path." The real robot needs another box: "what do we do when confidence collapses?" Navigation is not complete until recovery is explicit. Clear the local costmap. Back up. Rotate to acquire features. Slow down under localization uncertainty. Ask for remote help. Enter a safe stop. These are not edge cases. They are the difference between a lab demo and a deployable system.

The deepest design choice is not which recovery behaviors you implement. It is what evidence triggers them. If localization covariance spikes, maybe you should reinitialize. If wheel slip exceeds threshold, maybe you should reduce trust in encoders. If the local planner rejects all trajectories for two seconds, maybe the problem is not blocked motion but an overinflated costmap. Recovery logic is diagnosis under pressure.

What We Built

A navigation stack with explicit transforms, dead reckoning, occupancy-grid mapping, probabilistic localization, pose-graph SLAM, global and local planners, a fast controller handoff, timestamp-aware transport, and failure recovery. Every layer exists because the layer below it cannot carry the full truth by itself.

This subsystem is the bridge between robotics and the rest of the repo. The map is a mutable buffer. The estimator is floating-point error management with better branding. The planner is graph search under deadlines. The transport is distributed systems. The acceleration path often lands on GPU. That is why navigation is such a good deep dive: it is one build, but it forces you to think in every layer at once.
