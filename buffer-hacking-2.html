// network buffers — packets in flight

Every TCP connection has four buffers: a send buffer and receive buffer on each side. When you call write() on a socket, you're copying data into the kernel's send buffer. The kernel's TCP stack decides when and how to actually transmit it. This is why write() returns instantly even though the data hasn't reached the other side yet.

TCP Buffer Architecture 

  Your App                  Kernel                    Network
  ────────                  ──────                    ───────
                     ┌─────────────────┐
  write(sock, ──────→│  Send Buffer    │────→ TCP segment ────→ ...
  data, len)         │  (128 KB default)│     with seq #
                     └─────────────────┘
                                              ACK comes back
                     ┌─────────────────┐         │
  read(sock, ◄───────│  Recv Buffer    │◄────────┘
  buf, len)          │  (128 KB default)│
                     └─────────────────┘

  Buffer states:
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │ sent │ sent │ sent │write │      │      │      │
  │ ACKd │ ACKd │ wait │ head │ free │ free │ free │
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┘
  ← can free →  ← in flight →  ← available for write →

  When send buffer is FULL → write() BLOCKS (or returns EAGAIN)
  When recv buffer is FULL → TCP advertises window=0 → sender stops
  This is TCP FLOW CONTROL — it's all buffer management

Tune the Buffers

// Default socket buffers are 128KB — fine for most things
// For high-throughput (video streaming, model weight transfer): go bigger

int bufsize = 4 * 1024 * 1024;  // 4MB buffer
setsockopt(sock, SOL_SOCKET, SO_SNDBUF, &bufsize, sizeof(bufsize));
setsockopt(sock, SOL_SOCKET, SO_RCVBUF, &bufsize, sizeof(bufsize));

// System-wide limits (needs root):
// echo 16777216 > /proc/sys/net/core/wmem_max   # 16MB send
// echo 16777216 > /proc/sys/net/core/rmem_max   # 16MB recv

// TCP_NODELAY: disable Nagle's algorithm
// Nagle buffers small writes to combine them into bigger packets
// Great for throughput, terrible for latency (games, trading)
int one = 1;
setsockopt(sock, IPPROTO_TCP, TCP_NODELAY, &one, sizeof(one));
// Now every write() becomes its own packet immediately

Insight: Protocols Are Buffer Formats

TCP, UDP, HTTP, DNS, TLS — they're all just conventions for how to fill a buffer. An IP header is 20 bytes at specific offsets, just like your shard header is 64 bytes. A DNS query is a buffer with a 12-byte header, then variable-length question sections. If you can build a shard file, you can build a network packet. Same skills: offset math, endianness, length-prefixed fields, checksums.

// disk buffers — the page cache

You think you're writing to disk. You're not. You're writing to a buffer. The Linux kernel maintains a massive in-memory cache of disk data called the page cache. When you write(), data goes to the page cache. The kernel flushes dirty pages to disk in the background, whenever it feels like it. This is why a power failure can lose your last few seconds of writes — they were in a buffer, not on disk.

The Write Path: App → Disk 

  write(fd, buf, 4096)
       │
       ▼
  ┌────────────────┐
  │  App Buffer    │  Your []byte or char buf[4096]
  └───────┬────────┘
          │ copy_from_user()
          ▼
  ┌────────────────┐
  │  Page Cache    │  Kernel memory. Marked "dirty".
  │  (RAM)         │  write() returns HERE. Not at disk.
  └───────┬────────┘
          │ pdflush / writeback thread (background, async)
          ▼
  ┌────────────────┐
  │  Block Layer   │  I/O scheduler reorders for disk efficiency
  │  Queue         │  (elevator algorithm: minimize seek distance)
  └───────┬────────┘
          │ DMA transfer
          ▼
  ┌────────────────┐
  │  NVMe / SSD    │  Actually persistent now.
  │  (Disk)        │  But SSD has its OWN buffer too...
  └────────────────┘

  Total buffers: 4 (app → page cache → block queue → SSD FTL)

Force Data to Disk

// write() is a LIE — data might still be in page cache
write(fd, tensor_data, size);  // Returns "success" but it's in RAM

// fsync() — force THIS file's dirty pages to disk
fsync(fd);  // Blocks until data is on disk. Slow (~1ms SSD, ~10ms HDD)

// fdatasync() — like fsync but skips metadata (faster)
fdatasync(fd);  // Only flushes data, not timestamps/size

// O_DIRECT — bypass page cache entirely
int fd = open("data.bin", O_WRONLY | O_DIRECT);
// Now write() goes straight to disk. But:
//   - Buffer must be 512-byte aligned (use posix_memalign)
//   - Write size must be multiple of 512
//   - No page cache = no read-ahead = terrible for small reads
// Used by databases (PostgreSQL, MySQL) that manage their own cache

// O_SYNC — every write is an implicit fsync
int fd = open("wal.log", O_WRONLY | O_SYNC);
// Every write() blocks until disk confirms. Bulletproof but slow.
// WAL (write-ahead log) files use this for crash safety.
C — disk buffer control
Page Cache Is Why mmap Is Fast

When you mmap a file, you're mapping the page cache into your address space, not the disk. If the file was recently read, the pages are already in RAM — mmap access is a memory read, not a disk read. Multiple processes mmap'ing the same file share the same physical pages. This is why your MoSH InstantLoader works: the first load pages the file into cache, subsequent loads are instant.

The readahead() Trick

// Tell the kernel: "I'm going to read this region soon"
// The kernel starts loading pages in the background
posix_fadvise(fd, offset, length, POSIX_FADV_WILLNEED);
// ...or the more direct version:
readahead(fd, offset, length);

// While the kernel is fetching from disk, you do other work
prepare_compute_graph();

// By the time you actually read, the data is already in page cache
read(fd, buf, length);  // Cache hit! No disk wait.

// Conversely, tell the kernel you're done with these pages:
posix_fadvise(fd, offset, length, POSIX_FADV_DONTNEED);
// Kernel evicts them from page cache, freeing RAM
// Essential for large model files — don't hog the page cache

// buffer overflow — the legendary exploit

The buffer overflow is the most consequential bug in computing history. It's been exploited since the Morris Worm in 1988. The idea is devastatingly simple: write more data into a buffer than it can hold, and the excess overwrites whatever comes next in memory. If "whatever comes next" is a return address on the stack, you can redirect execution to code you control.

Stack Buffer Overflow (Classic) 

  BEFORE overflow:                 AFTER overflow:
  ┌─────────────────┐             ┌─────────────────┐
  │  Return Address │ ← CPU      │  0xDEADBEEF     │ ← OVERWRITTEN
  │  (back to main) │   jumps    │  (attacker's    │    CPU jumps to
  ├─────────────────┤   here     │   shellcode)    │    attacker code!
  │  Saved EBP      │            ├─────────────────┤
  ├─────────────────┤            │  AAAAAAAAAA     │
  │                 │            │  AAAAAAAAAA     │
  │  char buf[16]   │            │  AAAAAAAAAA     │
  │                 │            │  AAAAAAAAAA     │ ← Overflow
  ├─────────────────┤            ├─────────────────┤
  │                 │            │                 │
  │  (other vars)   │            │  (other vars)   │
  └─────────────────┘            └─────────────────┘
     Stack grows ↓                  Stack grows ↓

  The write goes PAST buf[16] and smashes the return address.
  When the function returns, CPU jumps to attacker-controlled address.

// THE VULNERABLE CODE — never do this
void vulnerable(char *user_input) {
    char buf[16];
    strcpy(buf, user_input);  // No length check! Copies until \0
    // If user_input is 32 bytes, the extra 16 bytes
    // overwrite saved EBP + return address
}

// THE SAFE VERSION
void safe(char *user_input) {
    char buf[16];
    strncpy(buf, user_input, sizeof(buf) - 1);
    buf[sizeof(buf) - 1] = '\0';  // Ensure null termination
}

Why This Still Matters

Go, Rust, Python, JavaScript — they all prevent buffer overflows through bounds checking. But the libraries they call are often written in C. OpenSSL, zlib, libc, database drivers, image parsers — one unchecked memcpy in a C library and the whole process is compromised. Heartbleed (2014) was a buffer over-read in OpenSSL: the server returned up to 64KB of whatever was adjacent in memory — private keys, passwords, session tokens. All because a length field wasn't validated.

// double buffering — the flicker killer

If you draw directly to the framebuffer while the display controller is reading it, the user sees half-drawn frames — tearing, flickering, visual garbage. Double buffering solves this: draw to a back buffer, then swap it to the front in one atomic operation. The user only ever sees complete frames.

Double Buffering 

  Frame N:                          Frame N+1:
  ┌─────────────────┐              ┌─────────────────┐
  │  FRONT BUFFER   │ ← display   │  FRONT BUFFER   │ ← was back
  │  (being shown)  │   reads     │  (Frame N+1)    │   NOW shown
  ├─────────────────┤              ├─────────────────┤
  │  BACK BUFFER    │ ← you       │  BACK BUFFER    │ ← was front
  │  (being drawn)  │   write     │  (drawing N+2)  │   NOW drawing
  └─────────────────┘              └─────────────────┘
           │                                │
           └───── SWAP (pointer swap) ──────┘
                  ~0 ns (just change which buffer the display reads)

  Triple buffering: add a third buffer so the GPU never waits.
  ┌───────────┐  ┌───────────┐  ┌───────────┐
  │  DISPLAY  │  │  READY    │  │  DRAWING  │
  │  (front)  │  │  (middle) │  │  (back)   │
  └───────────┘  └───────────┘  └───────────┘
  Display shows   GPU done,      GPU currently
  this one        waiting        rendering here

// Double buffering in practice — even for terminal rendering

char front[80 * 24];  // What's on screen now
char back[80 * 24];   // What we're building

void render_frame() {
    // Clear back buffer
    memset(back, ' ', sizeof(back));

    // Draw into back buffer (no screen updates yet)
    draw_player(back, player_x, player_y);
    draw_enemies(back, enemies, enemy_count);
    draw_ui(back, health, score);

    // Diff and flush only changed characters
    for (int i = 0; i < 80*24; i++) {
        if (back[i] != front[i]) {
            int row = i / 80, col = i % 80;
            printf("\033[%d;%dH%c", row+1, col+1, back[i]);
        }
    }
    fflush(stdout);  // Send all updates at once

    // Swap
    memcpy(front, back, sizeof(back));
}

Double Buffering as a Pattern — Not Just Graphics

The terminal renderer above is one instance. But double buffering is bigger than graphics. It appears whenever you need to read old state and write new state without corrupting either.

Consider the classic demoscene fire effect. Flames lick upward from the bottom of the screen. Each frame, every pixel is computed from the pixels below it in the previous frame — average the neighbors, subtract some random cooling. But if you read and write to the same buffer, a pixel at row 10 reads from row 11 which you already overwrote this frame. The fire collapses into garbage.

Two buffers solve it:

/* The fire effect IS double buffering */

uint8_t buf_A[W * H], buf_B[W * H];

for (int y = 0; y < H - 1; y++) {
    for (int x = 0; x < W; x++) {
        /* Read neighbors from buf_A (last frame) */
        int avg = (buf_A[(y+1)*W + x-1] + buf_A[(y+1)*W + x]
                 + buf_A[(y+1)*W + x+1] + buf_A[(y+2)*W + x]) >> 2;

        /* Write cooled value to buf_B (this frame) */
        int decay = rand() % cooling;
        buf_B[y * W + x] = avg > decay ? avg - decay : 0;
    }
}

/* Swap */
uint8_t *tmp = buf_A;
buf_A = buf_B;
buf_B = tmp;
fire effect — double buffering for simulation

Conway's Game of Life uses the same trick — read neighbors from generation N, write births and deaths to generation N+1, swap. GPU swap chains are the same pattern at hardware speed. React's virtual DOM diffs two trees the same way. Read old state, write new state, swap — this is one of the most fundamental patterns in computing.

When You Need Double Buffering

Any time your computation reads from the same data structure it writes to, and the reads depend on seeing the previous state, not partially-updated state. Simulation (fire, Game of Life, physics). Rendering (framebuffer, terminal). State machines (current state → next state). Networking (incoming packet buffer while processing previous batch). If you're seeing corruption or "feedback loops" in your output, you probably need two buffers.

// DMA — buffers without the CPU

Direct Memory Access (DMA) is the ultimate buffer trick: data moves between devices and memory without the CPU touching it. The CPU sets up a transfer — "copy from this address to that address, this many bytes" — then does other work while the DMA controller handles the copy. The CPU gets interrupted when it's done.

  │     │◄────│        │                  │     │     │        │

Pinned Memory: The DMA Prerequisite

DMA needs physical memory addresses that don't change. Normal malloc memory can be paged out to swap at any time — the physical address changes, breaking the DMA. Pinned memory (cudaMallocHost, mlock) tells the OS "never page this out." The trade-off: pinned memory reduces available RAM for other processes. Pin too much and you starve the system.

Overlap: The Pipeline of Buffers

// The REAL power: double-buffer with DMA for pipeline overlap
// While GPU computes on buffer A, DMA fills buffer B

float *buf_a, *buf_b;  // Two GPU buffers
cudaMalloc(&buf_a, layer_size);
cudaMalloc(&buf_b, layer_size);

    float *current = (i % 2 == 0) ? buf_a : buf_b;
    float *next    = (i % 2 == 0) ? buf_b : buf_a;

    // Compute on current layer
    matmul_kernel<<0, compute_stream>>>(current, ...);

    // Both streams run concurrently!
    // DMA engine: copying next layer
    // GPU SMs: computing current layer
}

Trick 1: The Bounce Buffer

When you need to transform data between two different formats, use a small bounce buffer instead of converting in-place (which might corrupt data on failure).

// Convert BFloat16 → Float32 through a bounce buffer
// Read raw BF16 from mmap'd shard, convert to F32 in bounce,
// hand the bounce buffer to the compute kernel

float bounce[256];  // Small, fits in L1 cache
uint16_t *src = (uint16_t *)mapped_tensor;

for (int i = 0; i < total; i += 256) {
    int chunk = min(256, total - i);
    for (int j = 0; j < chunk; j++) {
        bounce[j] = bf16_to_f32(src[i + j]);  // Convert in L1
    }
    process_chunk(bounce, chunk);  // All data hot in cache
}

Trick 3: Buffer Gap (Text Editors)

// How real text editors store text: gap buffer
// Cursor position = gap position. Insert/delete at cursor is O(1).

// Buffer: "Hello World" with cursor at position 5
// ┌─┬─┬─┬─┬─┬───────┬─┬─┬─┬─┬─┬─┐
// │H│e│l│l│o│       │ │W│o│r│l│d│
// └─┴─┴─┴─┴─┴───────┴─┴─┴─┴─┴─┴─┘
//  ←── pre-gap ──→ GAP ←── post-gap ──→
//
// Type 'X': put X before gap, shrink gap by 1. O(1)!
// ┌─┬─┬─┬─┬─┬─┬─────┬─┬─┬─┬─┬─┬─┐
// │H│e│l│l│o│X│     │ │W│o│r│l│d│
// └─┴─┴─┴─┴─┴─┴─────┴─┴─┴─┴─┴─┴─┘
//
// Move cursor left: shift one char across the gap. O(1)!
// Delete: just widen the gap. O(1)!
//
// Used by: Emacs, Scintilla, many editors
// Alternative: piece table (VS Code), rope (Xi editor)

typedef struct {
    char *buf;
    int gap_start;  // First byte of gap
    int gap_end;    // First byte after gap
    int buf_size;
} GapBuffer;

void insert(GapBuffer *g, char c) {
    g->buf[g->gap_start++] = c;  // That's it. O(1).
}

void delete_back(GapBuffer *g) {
    g->gap_start--;  // That's it. O(1). No memmove.
}

Trick 4: XOR Swap Buffer

// Swap two buffers without a temporary — for the truly unhinged
// Works because XOR is its own inverse: a ^ b ^ b = a

void xor_swap_buffers(uint8_t *a, uint8_t *b, size_t n) {
    for (size_t i = 0; i < n; i++) {
        a[i] ^= b[i];
        b[i] ^= a[i];
        a[i] ^= b[i];
    }
}

Trick 5: Ring Buffers (Circular Queues)

Picture a conveyor belt in a factory — not a straight one, a circular one that loops back on itself. One worker places items on the belt (the producer). Another worker picks items off (the consumer). The belt keeps moving. When it reaches the end, it doesn't stop — it wraps around and the empty section comes back to the producer, ready for more.

Now picture what happens without the loop. A straight belt. The producer puts items on the left end. The consumer takes from the right. After a while, the belt is full of empty space on the right and items piling up on the left. To reuse the space, you'd have to stop the belt and shift everything back. That shift is O(n) — the more items, the slower it gets. Every normal array queue pays this tax.

The circular belt never shifts. The consumer takes an item, and that slot comes back around to the producer automatically. Two pointers — write and read — chase each other around the loop forever. Adding is O(1). Removing is O(1). The memory never grows, never shrinks, never fragments. It just circulates.

The chase

The producer runs ahead, filling slots. The consumer follows behind, emptying them. If the producer laps the consumer, the buffer is full — the belt is packed, nowhere to put new items. If the consumer catches the producer, the buffer is empty — nothing to take. They dance around the circle forever. The whole thing is two integers (write position, read position) and one array. That's it. That's the data structure used by audio engines, network stacks, kernel queues, and GPU command rings.

A ring buffer is an array with a write position and a read position that chase each other in circles. One end produces data, the other consumes it. When a position reaches the end, it wraps to the beginning. This is the fundamental data structure for streaming — audio, video, network packets, inter-thread communication.

Ring Buffer 

  Write position advances clockwise →

          ┌──┬──┬──┬──┬──┬──┬──┬──┐
          │  │██│██│██│██│  │  │  │
          └──┴──┴──┴──┴──┴──┴──┴──┘
           ↑  R              W  ↑
           │  read_pos    write_pos
           │                    │
           └── empty ───────────┘

  Available data: write_pos - read_pos
  Free space:     capacity - available

The classic problem: what happens when a write crosses the boundary? You want to write 5 bytes starting at position 30 in a 32-byte ring. Bytes 30-31 fit at the end, but bytes 32-34 need to wrap to positions 0-2. Without tricks, you need two memcpy calls:

/* Without tricks: split the write at the boundary */
size_t offset = wpos % capacity;
size_t first = capacity - offset;       /* bytes until end */
if (first > len) first = len;
memcpy(buf + offset, data, first);      /* tail end */
if (len > first)
    memcpy(buf, data + first, len - first); /* wrapped portion */
ring buffer — the boundary problem
Power-of-2 Trick

If the capacity is a power of 2, you can replace expensive modulo (%) with a bitwise AND: offset = wpos & (capacity - 1). When capacity is 2n, capacity - 1 is a bitmask of all 1s. This also means positions never need to wrap explicitly — they're monotonically increasing unsigned counters, and the bitmask extracts the correct offset automatically. This is why every serious ring buffer implementation requires power-of-2 capacity.

SPSC: Single-Producer, Single-Consumer

If exactly one thread writes and exactly one thread reads, you don't need locks. The writer only modifies write_pos; the reader only modifies read_pos. Each reads the other's position to compute available space. C11 atomic_store/atomic_load suffice — no compare-and-swap loops, no mutexes. This is how audio engines, network stacks, and inter-thread queues work at maximum throughput.

/* Lock-free SPSC ring buffer — the positions are the only shared state */
_Atomic size_t write_pos;  /* only writer touches this */
_Atomic size_t read_pos;   /* only reader touches this */

/* Writer: */
size_t avail = capacity - (atomic_load(&write_pos) - atomic_load(&read_pos));
if (avail >= len) {
    memcpy(buf + (wpos & (capacity-1)), data, len);
    atomic_store(&write_pos, wpos + len);
}

/* Reader: */
size_t have = atomic_load(&write_pos) - atomic_load(&read_pos);
if (have >= len) {
    memcpy(out, buf + (rpos & (capacity-1)), len);
    atomic_store(&read_pos, rpos + len);
}
SPSC lock-free ring — no mutex, no CAS
Where SPSC Rings Live

JACK audio uses lock-free rings between the real-time audio thread and the UI thread. io_uring (Linux kernel) uses submission and completion ring buffers shared between user space and kernel. LMAX Disruptor (Java trading systems) extends the pattern to multi-producer with CAS loops. The simple SPSC version above handles the vast majority of real-world streaming use cases.

Backpressure: When the Ring Is Full

What happens when the producer is faster than the consumer? The ring fills up. You have three choices: block (wait until space is available), drop (discard the data), or overwrite (oldest data lost). Unix pipes choose blocking — write() to a full pipe sleeps until the reader drains some. This is backpressure, and it's the simplest correct flow control:

/* Pipe-based backpressure: write() blocks when pipe is full */
/* This IS the timing mechanism for audio synthesis:
   - Synthesis thread generates samples
   - write(pipe_fd, samples, bytes) pushes to audio player
   - If player hasn't drained enough, write() blocks
   - Blocking = natural rate limiting = no explicit timer needed */

while (running) {
    generate_audio(buffer, 1024);
    write(pipe_fd, buffer, 1024 * sizeof(int16_t));  /* blocks when full */
}
backpressure via blocking write()

Now for the trick that eliminates the two-memcpy boundary problem entirely:

The mmap Double-Map: Eliminating the Boundary

// The ULTIMATE ring buffer hack: map the same physical memory TWICE
// adjacent in virtual memory. Wrap-around becomes invisible!

size_t size = 4096;  // Must be page-aligned

// Create a shared memory region
int fd = memfd_create("ring", 0);
ftruncate(fd, size);

// Map it TWICE, adjacent:
void *addr = mmap(NULL, size * 2, PROT_NONE, MAP_PRIVATE|MAP_ANON, -1, 0);
mmap(addr,        size, PROT_READ|PROT_WRITE, MAP_SHARED|MAP_FIXED, fd, 0);
mmap(addr + size, size, PROT_READ|PROT_WRITE, MAP_SHARED|MAP_FIXED, fd, 0);

// Now addr[0..4095] and addr[4096..8191] are the SAME physical memory!
//
// Virtual:  [────── Page A ──────][────── Page A (mirror) ──────]
// Physical: [────── Page A ──────]
//
// Write past the end → automatically wraps to the beginning
// memcpy across the "boundary" Just Works — no split logic!
// This is how high-performance audio engines and network stacks work
C — mmap'd ring buffer (zero-copy wrap)
Why This Is Beautiful

Normal ring buffers need special handling when a read or write wraps around the end: you split into two memcpy's. With the mmap trick, the virtual memory system is the ring. A memcpy that crosses the boundary in virtual space quietly wraps in physical space. You can write(fd, ring + offset, length) even when it spans the wrap point. The kernel handles it. No branch. No split. Zero overhead. This is the kind of trick that makes you sit back and stare at the ceiling for a while.
