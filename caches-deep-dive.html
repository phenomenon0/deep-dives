CPU CACHES

The invisible memory hierarchy that makes or breaks every cycle your code executes_

01 // The Wall

Your CPU can retire four billion operations per second. Main memory can answer one request every two hundred of those operations. The gap between those two numbers is the single most important fact in computer performance.

This was not always true. In 1980, a processor and its DRAM ran at roughly the same clock rate. A memory access cost one cycle. You asked for a byte and it was there. But transistor budgets grew faster for logic than for storage, and by the early 1990s a memory fetch took ten cycles. By 2000 it took a hundred. Today, on a modern server chip running at 4 GHz, a round trip to DDR5 costs 150–200 cycles. That is not a bottleneck. That is a wall.

The processor got ten thousand times faster. DRAM got maybe ten times faster. The result is a 1,000× mismatch between compute speed and memory speed. Without something to bridge that gap, every program would spend almost all of its time waiting for data.

That something is the cache: a small, fast memory built from SRAM transistors, placed between the CPU and main memory, holding copies of the data the processor is most likely to need next. Caches are not an optimization. They are the reason your programs run at all.

The Bet

Every cache is a bet that the future will look like the recent past. Data you just touched, you will touch again. Data near what you just touched, you will touch next. When the bet pays off, memory looks fast. When it does not, the wall reappears.

02 // The Hierarchy

One cache would be either too slow or too small. So we build a ladder.

Modern CPUs have three levels of cache, each trading size for speed. The first level is tiny and blindingly fast — a lookup completes in the same cycle the instruction needs the data. The last level is large enough to hold working sets but slow enough that a miss still hurts. Beyond it lies main memory, and beyond that, the abyss of storage.

The L1 is split into two caches: one for instructions (L1-I), one for data (L1-D). This is not arbitrary. The instruction fetch unit and the data load/store unit run in parallel stages of the pipeline. Separate caches let them access memory simultaneously without port contention. Intel's recent cores give L1-D 48 KB and L1-I 32 KB. AMD's Zen 4 gives both 32 KB.

L2 is unified — it holds both instructions and data — and private to each core. On modern chips, L2 has grown to 1–2 MB per core, making it large enough to hold the hot working set of most single-threaded workloads.

L3 is shared across all cores. It acts as a victim cache and a coherence directory, so when Core 0 needs data that Core 3 modified, the L3 mediates the transfer. Server chips allocate 2–3 MB of L3 per core; a 16-core chip might have 36 MB of shared last-level cache.

The Latency Ladder

L2 256 KB–2 MB ~4 ns 16 Grabbing a book from your shelf
L3 8–36 MB ~12 ns 48 Walking to the office library
DRAM 16–512 GB ~100 ns 400 Driving to the city archive
NVMe SSD 1–8 TB ~10 μs 40,000 Ordering a book from another state

Every level is a bet on locality. The closer to the core, the smaller and faster — and the more it depends on your access patterns being predictable. When they are, L1 delivers data in a single cycle, and the processor never sees the wall. When they are not, the entire pipeline stalls while the cache hierarchy goes hunting.

03 // Anatomy of a Lookup

A cache is not a hash table. It is something more constrained, more elegant, and more fragile. Understanding how a single lookup works is the key to understanding everything else.

The fundamental unit is the cache line: 64 bytes on every modern x86 and ARM processor. When you load a single int from memory, the cache does not fetch 4 bytes. It fetches the entire 64-byte line that contains it. This is the spatial locality bet made manifest in hardware: if you touched one byte, you will probably touch its neighbors.

Address Decomposition

Given a 64-bit virtual address (after TLB translation to a physical address), the cache splits it into three fields:

Address Decomposition — 32 KB, 8-way Set-Associative, 64B Lines 
  64-bit physical address
  ┌────────────────────────────────────────────────────┐
  │ tag  (bits 63–12)          │ set (11–6) │ offset (5–0) │
  │ 52 bits                    │ 6 bits     │ 6 bits      │
  └────────────────────────────────────────────────────┘

  offset  →  which byte within the 64-byte line       (2&sup6; = 64 bytes)
  set     →  which row (set) in the cache              (2&sup6; = 64 sets)
  tag     →  which address maps to this set             (unique ID)

The math is straightforward. A 32 KB, 8-way set-associative cache with 64-byte lines has: 

Math// Cache geometry
cache_size    = 32768 bytes   // 32 KB
line_size     = 64 bytes
associativity = 8 ways

num_sets      = cache_size / (line_size × associativity)
              = 32768 / (64 × 8)
              = 64 sets

offset_bits   = log2(64)   = 6   // byte within line
set_bits      = log2(64)   = 6   // which set
tag_bits      = 64 - 6 - 6 = 52  // everything else

The Lookup Path

When the CPU issues a load, the cache does the following in a single cycle:

1. Extract the set index (bits 11–6) and use it to select one row of the cache.
2. Read all 8 tags in that set simultaneously (parallel comparators).
3. Compare each tag against the address’s tag field.
4. If one matches and the valid bit is set: hit. Use the offset to select the requested bytes from that line.
5. If none match: miss. Forward the request to the next level and stall (or queue) until it returns.

One Set of an 8-Way Cache (64 sets × 8 ways = 32 KB) 
  Set 17:
  ┌─────┐┌──────────────────┐┌───────────────────────────────────┐
  │ V D ││ tag              ││ 64 bytes of data ...                │  Way 0
  └─────┘└──────────────────┘└───────────────────────────────────┘
  ┌─────┐┌──────────────────┐┌───────────────────────────────────┐
  │ V D ││ tag              ││ 64 bytes of data ...                │  Way 1
  └─────┘└──────────────────┘└───────────────────────────────────┘
    ...                                                             Way 2–6
  ┌─────┐┌──────────────────┐┌───────────────────────────────────┐
  │ V D ││ tag              ││ 64 bytes of data ...                │  Way 7
  └─────┘└──────────────────┘└───────────────────────────────────┘

  V = valid bit     D = dirty bit     8 parallel tag comparators per lookup

Notice the two metadata bits per line. The valid bit says whether the line holds real data (a cold cache starts with all valid bits cleared). The dirty bit marks whether the line has been written to — if it has, it must be written back to memory before eviction. These two bits, multiplied across millions of cache lines, quietly govern every memory operation your processor executes.

The 64-Byte Rule

The cache line — not the byte — is the atom of the memory system. Every access loads 64 bytes whether you need them or not. This means a bool and a struct next to it share a cache line. Change the bool, and you invalidate the struct. Understand this, and false sharing, padding, and alignment all make sense.

04 // Associativity

Direct-mapped caches are fast and fragile. Fully associative caches are flexible and impossibly expensive. Every real cache lives in the tension between them.

Associativity answers a simple question: when a new line arrives and it maps to a particular set, how many places can it go? In a direct-mapped cache, the answer is one. In an 8-way set-associative cache, the answer is eight. In a fully associative cache, the answer is anywhere — but you need a comparator for every line in the cache, and that does not scale.

The Associativity Spectrum — Same Four Accesses, Three Designs 
Direct-Mapped (1-way)                   Addresses A, B, C, D all map to set 5
┌────────────┐
│  Set 5:     │
│  [A]        │  ← load A: miss, fill
│  [B]        │  ← load B: miss, evict A
│  [C]        │  ← load C: miss, evict B
│  [A]        │  ← load A: MISS again — conflict!
└────────────┘
  Hit rate: 0%

2-Way Set-Associative
┌──────────────────┐
│  Set 5:           │
│  [A] [B]         │  ← A and B both fit
│  [C] [B]         │  ← load C: evict A (LRU)
│  [C] [A]         │  ← load A: evict B (LRU)
└──────────────────┘
  Hit rate: 0% (still thrashing with 4 addresses and 2 ways)

4-Way Set-Associative
┌────────────────────────┐
│  Set 5:                 │
│  [A] [B] [C] [D]       │  ← all four fit
│  [A] [B] [C] [D]       │  ← reload A: HIT
└────────────────────────┘
  Hit rate: higher — conflict misses eliminated

Direct-Mapped

Speed: Fastest lookup — one tag comparison, one data read.

Fragility: Any two addresses that map to the same set fight to the death. Conflict misses dominate.

Used for: TLBs in some microcontrollers, embedded L1 caches.

N-Way Set-Associative

Speed: Slightly slower — N parallel tag comparisons, mux on the result.

Resilience: N addresses can coexist in the same set. Conflict misses drop dramatically at 4–8 ways.

Used for: Every modern L1, L2, and L3 cache. 8-way is the dominant L1 choice; L3 is often 12–16 way.

The returns diminish quickly. Going from direct-mapped to 2-way cuts conflict misses by roughly half. Going from 2-way to 4-way halves them again. Beyond 8-way, the marginal benefit is measurable only in pathological access patterns, and the power and area cost of additional comparators starts to matter. This is why 8-way has been the L1 sweet spot for over a decade: it is the point where the conflict miss rate drops low enough that replacement policy, not associativity, becomes the dominant factor.

05 // Eviction

Every cache line loaded is a bet that it will be used again. Replacement policy is how the cache decides which bet to cancel.

When a new line arrives and all ways in the target set are occupied, the cache must choose a victim. The ideal policy would evict the line that will be used furthest in the future — Belady’s optimal algorithm — but that requires seeing the future. Real hardware approximates it.

LRU and Its Limits

Least Recently Used evicts the line that has not been accessed for the longest time. For small associativities (2–4 way), true LRU is implemented with a few bits of state tracking the access order. But for 8 or more ways, maintaining a precise ordering requires log2(N!) bits per set — 15 bits for 8-way — and the update logic runs on every access. That is too expensive.

Tree-Based Pseudo-LRU

Modern L1 caches use tree-PLRU: a binary tree of single-bit pointers that approximates LRU with only N−1 bits per set. For 8-way, that is 7 bits instead of 15.

Tree-PLRU for 8-Way — 7 Bits Per Set 
                     [0]                  ← root: 0=go left, 1=go right
                    /   \
                [0]       [1]
               /   \    /   \
            [1]   [0] [0]   [1]
            /\   /\  /\   /\
           W0 W1 W2 W3 W4 W5 W6 W7    ← 8 ways

  To find victim: follow pointers from root to leaf.
  Root=0 → go left. Next=0 → go left. Next=1 → go right.
  Victim = W1

  After access: flip every bit on the path from the accessed
  way back to the root. This points the tree away from
  what was just used — a rough LRU approximation.

PLRU does not track the exact access order. It can make suboptimal eviction choices, especially under scan patterns where every line gets touched once and the tree bits all point the same direction. But for typical workloads, it is within 1–2% of true LRU’s hit rate while costing half the state and running in a single cycle.

RRIP: Scan Resistance for L3

Last-level caches face a different problem: scan pollution. A one-time scan through a large array can evict frequently used data, poisoning the cache for subsequent accesses. Intel’s RRIP (Re-Reference Interval Prediction) counters this by assigning each line a re-reference prediction value. New lines enter with a near-distant prediction (not most-recently-used), so they are evicted quickly unless they are accessed again. Lines that prove useful get promoted. This simple bias protects the cache from single-pass scans while still responding to genuine working-set changes.

First Optimization

The best replacement policy is the one that never runs. If your working set fits in the cache, eviction policy is irrelevant. The first optimization is always: make the data smaller.

06 // Writes

Reading is simple: the data is either there or it is not. Writing is where caches get political.

When the CPU stores a value, the cache must decide two things: where to put it and when to tell memory. The answers define the write policy, and the choice has consequences that ripple through bus bandwidth, coherence traffic, and store buffer pressure.

Write-Through

Rule: Every store goes to the cache and immediately to the next level.

Pro: Simple. Memory is always consistent. Recovery from crashes is straightforward.

Con: Heavy bus traffic. Every store becomes a memory write, regardless of whether the line is about to be written again.

Used in: Some embedded caches. L1 on older ARM cores. Almost nowhere on modern desktop/server CPUs.

Write-Back

Rule: Stores update only the cache. The dirty bit gets set. Memory is updated only when the line is evicted.

Pro: Dramatically less bus traffic. Multiple stores to the same line collapse into one writeback.

Con: Memory is stale. Coherence protocol must track dirty state. Eviction becomes a write.

Used in: Every modern L1, L2, and L3 cache.

Write-Allocate vs No-Write-Allocate

On a write miss (the target line is not in the cache), there is a second choice: should the cache fetch the line first? Write-allocate says yes: bring the line in, then modify it in-place. This makes sense for write-back caches because subsequent reads will find the data locally. No-write-allocate says no: send the store directly to memory and do not pollute the cache with data that might not be read. This pairs naturally with write-through.

The Store Buffer

Even with write-back, a store cannot always complete instantly. If the cache is busy with a fill, or if the coherence protocol is waiting for an invalidation acknowledgment, the store must wait. Rather than stalling the pipeline, the CPU places the store in a store buffer — a small queue between the core and the L1 cache.

The Hidden Bottleneck

Modern Intel cores have a 56-entry store buffer. When it fills — because of a burst of cache misses on stores — the pipeline stalls. This is the real cost of write-side cache misses: not the miss latency itself, but the store buffer draining. Scatter writes to random addresses can stall a processor more effectively than random reads, because the store buffer saturates while the loads at least benefit from out-of-order execution and miss-status holding registers.

Write-Combining

For writes to uncacheable memory (MMIO, framebuffers, certain device-mapped regions), the CPU uses write-combining buffers. These collect multiple small stores to the same cache-line-sized region and send them as a single burst write. If you are writing 16 consecutive 4-byte values to a GPU buffer, write-combining turns 16 bus transactions into one — a 16× bandwidth improvement. But the stores must be sequential and to contiguous addresses; interleaved writes to different lines flush the WC buffer early and destroy the benefit.

07 // Locality

The cache does not know your algorithm. It knows two things: what you just touched, and what was next to it.

Temporal locality means reusing the same address. If you read a variable inside a loop, the first iteration misses and every subsequent iteration hits. Spatial locality means touching addresses near each other. When you iterate through an array, each cache line fetch gives you 64 bytes — sixteen 4-byte integers — for the price of one miss.

Loop Tiling

The canonical example of locality-aware design is loop tiling (also called loop blocking) in matrix multiplication. The naive triple loop touches matrix B in column order, striding across cache lines on every access: 

C — Naive// Naive: B accessed in column-major stride
for (int i = 0; i < N; i++)
  for (int j = 0; j < N; j++)
    for (int k = 0; k < N; k++)
      C[i][j] += A[i][k] * B[k][j];  // B[k][j] strides by N × 4 bytes
C — Tiled// Tiled: process T×T blocks that fit in L1
#define T 64  // tile size chosen so T² × sizeof(double) < L1 size
for (int ii = 0; ii < N; ii += T)
  for (int jj = 0; jj < N; jj += T)
    for (int kk = 0; kk < N; kk += T)
      for (int i = ii; i < ii+T; i++)
        for (int j = jj; j < jj+T; j++)
          for (int k = kk; k < kk+T; k++)
            C[i][j] += A[i][k] * B[k][j];

The tiled version processes small blocks that fit entirely in L1. Instead of striding across the entire matrix, B’s column accesses stay within a few cache lines. For a 1024×1024 matrix, tiling can deliver 8–10× speedup over the naive loop on the same CPU — not by reducing the number of floating-point operations, but by reducing the number of cache misses by an order of magnitude.

08 // Coherence

One core writes. Another core reads. The cache has to decide which truth to tell.

On a single-core machine, the cache is a private optimization. On a multicore chip, it becomes a distributed system. Each core has its own L1 and L2, and they can hold copies of the same memory address. If Core 0 modifies its copy, Core 1’s copy is now stale. The hardware must resolve this, on every write, without software intervention. The protocol that does this is called cache coherence.

MESI: The Four States

The MESI protocol assigns every cache line one of four states:

MESI State Machine 
  Modified     This core wrote it. No other core has a copy. Memory is stale.
               Responsible for writing back on eviction.

  Exclusive    This core is the only holder, and it matches memory.
               Can be silently promoted to M on a local write (no bus traffic).

  Shared       Multiple cores may hold copies. All match memory.
               A local write must first invalidate all other copies.

  Invalid      The line is not present (or has been invalidated).
               Any access is a miss.


  State Transitions:

  I —local read—→ E (if no other core has it) or S (if shared)
  E —local write—→ M (silent, no bus traffic)
  S —local write—→ M (broadcast invalidate to all other holders)
  M —snoop read—→ S (write back dirty data, downgrade to shared)
  M/E/S —snoop invalidate—→ I

Every transition from S → M requires a broadcast invalidation: the writing core must notify every other core that holds the line. On a bus-based system, this is a single broadcast. On a mesh interconnect (modern server chips), it goes through the L3 slice directory, which tracks which cores hold which lines. Either way, it costs time — typically 40–80 ns for a cross-core invalidation on a modern chip.

False Sharing: The Invisible Killer

Coherence operates on cache lines, not on variables. If two threads on different cores write to different variables that happen to live on the same 64-byte line, MESI sees them as writing to the same location. Each write triggers an invalidation. The line bounces between cores like a hot potato, turning what should be embarrassingly parallel into a serialized coherence storm. 

C — False Sharingstruct Counters {
    uint64_t a;  // 8 bytes — used by thread 0
    uint64_t b;  // 8 bytes — used by thread 1 (SAME CACHE LINE)
};

// Both threads hammer the same 64-byte line:
// Thread 0: counter->a++   (invalidates core 1's copy)
// Thread 1: counter->b++   (invalidates core 0's copy)
// Result: ~100M ops/sec instead of ~2B ops/sec
C — Fixedstruct Counters {
    alignas(64) uint64_t a;  // own cache line
    alignas(64) uint64_t b;  // own cache line
};

// Now each thread writes to a private line.
// No invalidations. Full parallel throughput.
The Performance Trap

False sharing is the most common cache performance bug in concurrent programs. It does not cause incorrect results. It causes correct results at 10–100× slower than necessary. The fix is always the same: pad or align so that independently-written data lives on separate cache lines. The cost is 56 bytes of wasted space per boundary. The alternative is serializing your cores.

MOESI and Beyond

AMD’s extension, MOESI, adds an Owned state: a core can hold a dirty line and share it with others without writing it back to memory first. This enables direct cache-to-cache transfer of modified data, reducing memory traffic in workloads where one producer feeds multiple consumers. Intel achieves similar effects with its MESIF protocol, adding a Forward state that designates which sharer should respond to snoop requests, preventing redundant replies on a multi-socket system.

The details differ, but the lesson is the same: cache coherence is the hidden cost of shared memory. Every shared mutable cache line triggers a distributed consensus protocol in hardware. The fastest coherence message is the one that never needs to be sent — because the data was not shared in the first place.

09 // Prefetching

The best cache miss is the one that started 200 cycles before you needed it.

Prefetching is the cache system’s attempt to predict the future. If it can start fetching a line before the CPU asks for it, the data arrives in the cache just as the load instruction fires, and the miss becomes invisible. Modern processors have multiple hardware prefetchers running in parallel, each watching for a different access pattern.

Hardware Prefetchers

Next-line prefetcher: After a miss on line N, speculatively fetch line N+1. Simple, effective for sequential scans, and essentially free. This is why iterating forward through an array is almost never bottlenecked by misses after the first few lines.

Stride prefetcher: Tracks the distance between consecutive accesses. If you access offsets 0, 128, 256, 384, it detects the stride (128 bytes = 2 cache lines) and starts prefetching two lines ahead. This handles column traversal of matrices and strided structure access.

Stream prefetcher: Monitors miss addresses for ascending or descending sequences and opens a hardware stream that fetches multiple lines ahead. Intel’s L2 streamer can track up to 32 independent streams simultaneously.

Software Prefetch

When the access pattern is predictable to the programmer but not to the hardware — because it depends on data values rather than address arithmetic — software prefetch instructions give the programmer direct control. 

C — Binary Search with Prefetchint binary_search(int *arr, int n, int target) {
    int lo = 0, hi = n - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;

        // Prefetch both children before the comparison.
        // One of them will be needed; the other is wasted
        // but the memory-level parallelism hides the cost.
        __builtin_prefetch(&arr[(lo + mid) / 2], 0, 1);
        __builtin_prefetch(&arr[(mid + 1 + hi) / 2], 0, 1);

        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) lo = mid + 1;
        else hi = mid - 1;
    }
    return -1;
}

Each iteration of binary search touches one element at an unpredictable location — the hardware stride prefetcher cannot help. But the programmer knows that the next access will be at either the left or right midpoint. Prefetching both costs one wasted fetch but converts the critical-path miss from ~100 ns to ~0 ns. For a large sorted array that does not fit in L3, this can cut binary search time by 2–3×.

When Prefetch Hurts

Software prefetch is not free. Each instruction consumes a slot in the memory pipeline and may evict useful data. If the data was already in cache (hot path), the prefetch instruction is pure overhead. If the access pattern is truly random and the working set exceeds cache capacity, prefetching just evicts other useful lines sooner. The rule of thumb: use software prefetch only when you can predict access addresses at least 100–200 cycles in advance, and only when profiling confirms that L3 misses dominate the stall time.

10 // Measuring

You cannot optimize what you have not measured. And cache behavior is invisible without the right counters.

Every modern CPU contains hardware performance counters — special registers that silently count events like cache hits, misses, branch mispredictions, and TLB flushes while your code runs. They add zero overhead and provide ground truth. The challenge is not collecting the data but knowing which counters to read and how to interpret the ratios.

perf stat: The First Tool

Diagnostic Ratios

Symptom Counter Pattern Likely Cause Fix Direction
High L1 miss, low LLC miss L1 miss >5%, LLC miss <5% Poor spatial locality; data fits in L3 but not L1 Improve layout: SoA, tiling, pack structs
High L1 miss, high LLC miss Both >5% Working set exceeds cache; capacity misses dominate Reduce footprint, batch processing, streaming
Low L1 miss, high LLC miss L1 miss <2%, LLC miss >20% Good spatial locality but large footprint; L1 hits are from prefetch but L3 cannot hold it all Tile for L3, reduce data size, compress

Going Deeper: perf record + Cachegrind

perf stat gives you the big picture. perf record with perf annotate shows you which instructions are missing — down to the assembly line. Cachegrind (part of Valgrind) simulates the cache hierarchy and reports miss counts per source line, at the cost of 20–50× slowdown. Intel VTune provides similar instruction-level profiling with native counter support and lower overhead.

For false sharing specifically, Linux provides perf c2c (cache-to-cache), which identifies cache lines that are being bounced between cores. It reports the exact data addresses and the source lines responsible. If you suspect false sharing but cannot find it by code inspection, perf c2c will show you the line.

11 // Patterns

Every data structure is a cache strategy in disguise.

The way you lay out data in memory determines how the cache sees your program. Two programs that compute the same result can differ by 10× in throughput — not because of algorithmic complexity, but because one organizes memory for the cache and the other organizes it for the programmer.

Array of Structs vs Struct of Arrays

Array of Structs (AoS)

struct Particle {
  float x, y, z;    // position
  float vx, vy, vz; // velocity
  float mass;
  float charge;
  int   id;
  int   flags;
};
// 40 bytes per particle
Particle particles[1000000];

Update position: loads 40 bytes per particle, uses 24 (x,y,z,vx,vy,vz). 60% of fetched data is wasted.

Struct of Arrays (SoA)

struct Particles {
  float *x, *y, *z;
  float *vx, *vy, *vz;
  float *mass;
  float *charge;
  int   *id;
  int   *flags;
};
// Each array is contiguous
Particles p;
p.x = malloc(1000000 * sizeof(float));

Update position: loads only x,y,z,vx,vy,vz arrays. 100% of fetched data is used.

The arithmetic is simple: if your hot loop touches 6 fields out of 10, AoS wastes 40% of every cache line fetch. SoA wastes nothing. For a million particles, that is the difference between fitting in L3 and spilling to DRAM.

Hot/Cold Splitting

When only some fields are accessed frequently, split the struct into hot and cold parts: 

Cstruct EntityHot {
    float x, y, z;       // position — accessed every frame
    float vx, vy, vz;    // velocity — accessed every frame
    uint32_t cold_idx;   // index into cold array
};  // 28 bytes — fits 2 per cache line

struct EntityCold {
    char name[64];
    uint64_t creation_time;
    uint32_t flags;
    // ... rarely-accessed metadata
};

The hot array is tight and sequential. The cold array is only touched when needed. The per-frame loop iterates over dense, cache-friendly data, and the cache lines are not diluted by metadata that is only read on spawn or despawn.

Data-Oriented Design (ECS)

Game engines formalized this into the Entity-Component-System pattern: entities are just IDs, components are stored in contiguous arrays indexed by entity, and systems iterate over arrays of matching components. A physics system that needs position and velocity iterates two tight arrays. A rendering system that needs position and mesh iterates two different tight arrays. The data layout matches the access pattern, not the object hierarchy.

This is not just a game engine trick. Database column stores work on the same principle — store each column contiguously so that analytical queries scan only the columns they need. The cache does not care whether your domain is particles, database rows, or network packets. It cares whether your memory access is sequential.

Cache-Oblivious Algorithms

A cache-oblivious algorithm achieves optimal cache behavior without knowing the cache size or line size. The classic example is the recursive matrix multiply: instead of tiling with a fixed block size, you recursively subdivide the problem until the subproblems naturally fit in whatever cache level they hit. At some recursion depth, the subproblem fits in L1. At a higher depth, it fits in L2. The algorithm adapts to the entire hierarchy without a single tuning parameter.

The same principle applies to the van Emde Boas tree layout for search trees: instead of storing a binary tree in breadth-first or depth-first order, you lay it out recursively so that the subtree rooted at any node is stored contiguously. Searching the tree touches a logarithmic number of nodes, and each subtree that fits in a cache line gives you several levels of the tree for the price of one miss.

12 // The Craft

The cache is not something you fight. It is something you design for.

Every topic in this series touches the cache. Allocators choose where data lives in memory — and adjacent allocations land on the same cache line, while scattered ones do not. Concurrency pays the coherence tax on every shared mutable line. Hash tables live or die by whether probe sequences stay within a few cache lines or scatter across thousands. GPU shared memory is an explicit, programmer-managed cache — the same principles apply, just without the hardware automation.

The abstraction layers we build — objects, containers, virtual dispatch, garbage collection — are all, at the hardware level, patterns of memory access. The compiler does not see your class hierarchy. The cache does not see your algorithm. They see addresses, and they see timing. When your addresses are sequential and your reuse is tight, the hardware rewards you with the illusion that memory is fast. When they are not, the illusion breaks, and the wall reappears.

Cache-aware programming is not a collection of micro-optimizations. It is a design discipline. It starts at data structure selection, flows through memory layout decisions, and surfaces in the measurement tools that tell you whether your bets paid off. The question is never “how do I make this cache-friendly?” applied to finished code. The question is “what memory access pattern does this design imply?” asked before the first line is written.

The Contract

The cache is the hardware’s contract with your code: touch memory predictably, and I will make it fast. Touch it randomly, and no amount of clock speed will save you. Every data structure, every algorithm, every layout decision is a clause in that contract. The programmers who understand this write code that runs 10× faster on the same hardware. Not because they know tricks, but because they design with the machine.
