GAME PHYSICS

Collision detection, contact generation, iterative solvers, and the machinery that makes an action game feel like matter instead of math._

We are going to build the part of the game that pushes back. Not a scientific simulator. Not a whitepaper. A frame-budgeted collision and constraint machine that can hold a stack of crates, slide a hero capsule up a stair, and stop a bullet from teleporting through a wall at 120 Hz.

Hack: Physics Is Applied Lying

The player does not want truth. The player wants resistance. If the world pushes back consistently, they call it “realistic.” If it jitters, tunnels, or snags their jump arc, they call it broken. Real-time physics is the art of choosing which lies are stable enough to feel honest.

1. Time Is the First Collider

Physics is not “objects moving around.” Physics is a contract about when the world updates. Get time wrong and everything downstream lies: gravity changes with frame rate, tunneling appears, joints explode, and your carefully tuned dodge roll from the character deep dive suddenly feels mushy. A physics engine starts with a clock, not a collider.

The fixed-step bargain

The simulation wants a constant dt. Rendering wants to run whenever the GPU and monitor permit. These are different clocks. Do not fuse them. Use an accumulator, step physics at a fixed rate, then interpolate for rendering.

C++
const float kDt = 1.0f / 120.0f;
float accumulator = 0.0f;
double previous = NowSeconds();

while (running) {
    double current = NowSeconds();
    float frame = min(float(current - previous), 0.25f);
    previous = current;
    accumulator += frame;

    while (accumulator >= kDt) {
        StepPhysics(world, kDt);
        accumulator -= kDt;
    }

    float alpha = accumulator / kDt;
    RenderInterpolated(world.previous, world.current, alpha);
}

Failure Mode

Most “physics bugs” reported by players are actually timestep bugs. If a crate jitters, a body tunnels, or a jump feels different at 144Hz, suspect time before you suspect collision math.

2. Body Data Wants Cache, Not Inheritance

A physics world is a hot loop over a lot of boring numbers: position, rotation, inverse mass, velocity, shape bounds, sleep flags. That means layout matters more than class elegance. If you store bodies as fat objects with virtual methods, the solver spends its life waiting on cache misses. This is the exact lesson from Buffers: contiguous wins.

3. Broad Phase: Who Might Hit Whom

The broad phase is a liar, but a useful one. Its job is to return too many pairs, quickly. If you ask it for the exact colliding bodies, it becomes expensive. If you ask it for plausible collisions, it becomes fast enough to run every frame. Physics engines live on that compromise.

Performance Rule

Optimize for pair count, not for theoretical purity. A slightly dirty broad phase with temporal coherence usually beats a perfectly rebuilt structure every frame.

Why use fat AABBs?

If you enlarge the proxy by a small velocity margin, tiny movements no longer force tree reinsertion every frame. You spend a few extra false positives to save a pile of maintenance work. That trade is almost always worth it.

4. Narrow Phase: Prove the Collision

Broad phase gives you suspects. Narrow phase gives you evidence. It answers three questions: are the shapes overlapping, along which normal, and by how much? For boxes and convex polygons, the Separating Axis Theorem is the workhorse because it is simple, branchy in understandable ways, and gives you a normal you can actually use.

SAT in one sentence

If you can find any axis on which the projected intervals do not overlap, the shapes are not colliding. If all candidate axes overlap, they are colliding; the axis with the smallest overlap becomes the collision normal.

Separating Axis Theorem 
shape A edges  -> candidate normals: n0 n1 ...
shape B edges  -> candidate normals: m0 m1 ...

project both shapes onto each axis:

axis n0: [-----A-----]
         [---B---]          overlap

axis m1: [-----A-----]
                       [---B---]   no overlap -> separated

one separating axis is enough to stop
Pseudo
bool SatOverlap(Shape a, Shape b, Manifold& out) {
    float minDepth = INF;
    Vec3 bestAxis;

    for (Vec3 axis : CandidateAxes(a, b)) {
        if (LengthSq(axis) < EPSILON) continue;
        axis = Normalize(axis);

        Interval ia = Project(a, axis);
        Interval ib = Project(b, axis);
        float depth = Overlap(ia, ib);

        if (depth <= 0.0f) return false;
        if (depth < minDepth) {
            minDepth = depth;
            bestAxis = axis;
        }
    }

    out.normal = bestAxis;
    out.penetration = minDepth;
    return true;
}
Precision Warning

The axis with the “smallest overlap” is often separated from the runner-up by floating-point dust. Epsilon policy matters. The nasty details are in Floating Point; physics is one of the places where that lying arithmetic becomes visible as jitter.

Circles, capsules, and spheres get bespoke tests because they are cheaper and stabler than a generic convex path. Real engines mix specialized shape pairs with a convex fallback. Purity loses to frame time.

5. Contacts Are Tiny Constraints

A collision is not “two things touched.” A useful collision is a contact manifold: normal, penetration, contact points, feature ids, and accumulated impulses from the last frame. One contact point gives you separation. Two points give you rotation resistance. Four badly chosen points give you solver noise. The manifold is where geometry turns into mechanics.

What a manifold stores

normal
The direction in which the solver pushes bodies apart.
penetration
How deep the overlap is right now. Useful, but not a license to teleport.
points
World-space anchors where the contact acts. Usually 1-2 in 2D, up to 4 in 3D patches.
cached impulse
Warm-start data from the previous frame so the solver converges quickly.

struct ContactPoint {
    Vec3 position;

};

struct Manifold {
    BodyId a, b;

Manifold persistence is a hidden performance feature. Reuse feature ids across frames and you can warm-start the solver. Throw manifolds away every step and stacks become jelly.

Why A Crate Stack Stands Up

Warm starting is the quiet hero of arcade physics. The solver remembers last frame’s contact impulses, starts near the old answer, and converges before the frame budget runs out. Delete that memory and every resting stack turns into soup.

6. Impulses: Velocity Changes in Disguise

Most real-time engines do not integrate forces until the exact moment of truth and solve a beautiful continuous system. They cheat. They compute impulses that change velocity right now so that the next frame looks physically plausible. This is why game physics feels solid long before it feels correct.

The normal impulse

For a single contact, you compute the relative velocity at the point, project it onto the contact normal, and apply an impulse that cancels the closing speed, plus maybe some bounce.

Math
vn = dot(vb + wb x rb - va - wa x ra, n)

effectiveMass =
    invMassA + invMassB +
    dot(cross(invIA * cross(ra, n), ra), n) +
    dot(cross(invIB * cross(rb, n), rb), n)

j = -(1 + restitution) * vn / effectiveMass
j = max(j, 0)

impulse = j * n
Pseudo
void SolveNormal(Contact& c) {
    Vec3 rv = PointVelocity(c.b, c.rb) - PointVelocity(c.a, c.ra);
    float vn = Dot(rv, c.normal);
    if (vn > 0.0f) return;

    float j = -(1.0f + c.restitution) * vn / c.normalMass;
    float old = c.accumulatedNormal;
    c.accumulatedNormal = max(old + j, 0.0f);
    j = c.accumulatedNormal - old;

    Vec3 impulse = j * c.normal;
    ApplyImpulse(c.a, -impulse, c.ra);
    ApplyImpulse(c.b,  impulse, c.rb);
}
Important Lie

The solver is not asking “what force acted over time?” It is asking “what velocity correction makes this frame stop looking wrong?” That framing is why sequential impulse solvers are fast enough for games.

Position correction

Even a good velocity solver leaves tiny penetrations. Engines usually add a bias term or a post-step positional correction to keep stacks from sinking into each other. Too little correction: objects melt. Too much: objects buzz.

7. Friction and Joints Are the Same Game

Once you understand a contact as a constraint, the rest of the engine snaps into focus. Friction is just a constraint along the tangent. A hinge joint is a constraint on relative position and relative angle. A distance joint is a constraint on separation. Physics engines stop looking like a bag of tricks and start looking like a constraint factory.

8. Islands, Sleeping, and the Cheapness of Doing Nothing

The most important optimization in physics is not SIMD, or clever trees, or fancy contact clipping. It is not solving bodies that do not matter. A crate at rest on the floor should be almost free. Engines get there with islands and sleeping: partition the active graph, solve only what is awake, and let quiet systems stay quiet.

Build islands from the contact graph

Connected bodies become one solve island 
[body 1]--contact--[body 2]--joint--[body 3]

[body 4]--contact--[body 5]

Island A: 1, 2, 3
Island B: 4, 5

If Island B sleeps, Island A keeps running.
If body 2 wakes up, all of Island A wakes with it.
Pseudo
for (BodyId seed : activeBodies) {
    if (visited[seed] || sleeping[seed]) continue;

    island.clear();
    stack.push_back(seed);

    while (!stack.empty()) {
        BodyId b = stack.back();
        stack.pop_back();
        if (visited[b]) continue;
        visited[b] = true;
        island.add(b);

        for (BodyId n : AdjacentBodies(b)) {
            if (!visited[n] && !sleeping[n]) stack.push_back(n);
        }
    }

    SolveIsland(island);
}
Parallelism Hook

Independent islands are the natural unit of concurrency. They can be dispatched to worker threads with minimal synchronization, which lines up neatly with the work-stealing and ownership patterns in Concurrency.

9. Character Controllers Cheat on Purpose

Rigid body simulation is great for boxes and bad for heroes. Players expect intent preservation: stairs should be climbable, slopes should be stable, tiny seams should not snag movement, and jump arcs should remain readable. That means the main character is often a kinematic controller with selective physics, not “just another dynamic body.”

Why avatars are special

Kinematic controller

Move by sweeps and depenetration, then resolve ground state manually. Predictable, tunable, and friendly to game feel.

Fully dynamic hero

Looks principled until the player catches a stair lip, bounces off a pebble, or falls over from tiny impulses.

Pseudo
void MoveCharacter(Controller& c, Vec3 wishDelta) {
    SweepHit hit = SweepCapsule(c.shape, c.position, wishDelta);

    if (!hit.blocked) {
        c.position += wishDelta;
        return;
    }

    c.position += wishDelta * hit.toi;
    Vec3 slide = Reject(wishDelta * (1.0f - hit.toi), hit.normal);
    c.position += ProjectSafeSlide(slide, hit.normal);
    c.grounded = hit.normal.y > c.minGroundDot;
}
Game Feel Rule

The character controller answers to the designer first and the physics engine second. That is why coyote time, step offset, snap-to-ground, and ledge forgiveness exist. The feel-side rationale lives in Game Characters; physics just supplies the collision queries.

10. Continuous Collision Detection Is Anti-Tunneling Surgery

Discrete simulation samples the world at ticks. Fast objects move between those samples. If the whole bullet jumps from one side of a wall to the other between frames, your overlap test never sees the intersection. That is tunneling. Continuous collision detection fixes it by asking not “are they overlapping now?” but “when, during this interval, do they first touch?”

Discrete miss vs continuous hit 
tick N:    bullet --->        wall
tick N+1:               ---> bullet passed through

discrete overlap: misses

swept test:
segment / capsule cast against wall
find TOI = 0.37 within the frame
advance body to impact, solve there

Where CCD actually matters

body type discrete only? ccd advice
slow crates yes not worth the cost
bullets / arrows no ray or sweep cast
fast hero dash sometimes shape cast if the controller feels wrong
ragdoll debris mostly yes use substeps before full CCD 
Pseudo
for (BodyId b : ccdBodies) {
    SweepHit hit = SweepShape(b.shape, b.position, b.velocity * dt);
    if (!hit.blocked) continue;

    float toi = Clamp(hit.toi - 0.001f, 0.0f, 1.0f);
    b.position += b.velocity * (dt * toi);
    RegisterImpactContact(b, hit);
}
Engineering Tradeoff

Full CCD everywhere is expensive and destabilizes pipelines if bolted on late. Most engines apply it surgically to fast, small, gameplay-critical bodies.

11. Determinism Is a Networking Problem Wearing a Physics Hat

If the same starting state and same inputs do not produce the same result, rollback netcode is painful, client prediction drifts, replays desync, and server reconciliation gets ugly. That is why physics and networking keep colliding organizationally: the physics engine secretly controls the quality of your multiplayer architecture.

Why exact determinism is hard

  • Floating-point order changes results.
  • Different SIMD paths round differently.
  • Hash map iteration order changes constraint order.
  • Threading can scramble broad-phase or island order.

All of this becomes visible when paired with prediction and reconciliation in Game Servers. The problem is not “physics is inaccurate.” The problem is “physics is inconsistently inaccurate.”

approach determinism quality cost where it fits
best-effort float weak cheap authoritative server, client corrections
strict float + stable ordering moderate engineering discipline rollback fighters, lockstep-ish sims
fixed-point / custom math strong high complexity RTS lockstep, hardcore determinism
Stable Order Rule

Determinism often dies from container iteration before it dies from floating point. Sort contacts, sort joints, sort islands. Chaos loves unspecified order.

12. The Full Step: What We Actually Built

At this point the engine stops being mystical. It is a pipeline: gather input, integrate velocities, update bounds, find candidate pairs, generate contacts, warm start, iterate constraints, write back transforms, put quiet bodies to sleep. The reason physics feels hard is not that any one stage is incomprehensible. It is that every stage is coupled to the next by frame budget.

void StepPhysics(World& world, float dt) {
    ApplyGravity(world, dt);
    IntegrateVelocities(world, dt);
    UpdateBroadPhase(world, dt);
    FindCandidatePairs(world);
    BuildManifolds(world);
    AssembleConstraintRows(world);
    WarmStart(world);

    for (int i = 0; i < world.iterations; ++i) {
        SolveContacts(world);
        SolveFriction(world);
        SolveJoints(world);
    }

    IntegratePositions(world, dt);
    FinalizeSleepStates(world);
}
