SHADERS

Painting with math. Every pixel on your screen exists because a tiny program ran millions of times per frame, turning coordinates into colors._

// 01 — The Rendering Pipeline

VERTEX → RASTERIZER → FRAGMENT → FRAMEBUFFER   ← the whole map

Millions of triangles, sixty times per second. That is what your GPU draws every frame of a 3D game, every rotation of a product model, every character walking across a scene. A circle is 32 triangles arranged in a fan. A human face is 10,000 triangles stitched together. And the GPU draws them all the same way: three vertices in, colored pixels out.

Why triangles? Because a triangle is the simplest flat shape: three points, always on a plane, no ambiguity. Every curved surface can be approximated with enough of them. And the GPU is a machine built to process triangles in parallel — thousands at once.

So the question becomes: how does a triangle become colored pixels on your screen? That path — from three points to millions of colored dots — is the rendering pipeline. And a shader is a program you write to control what happens at specific stages of that path.

Let's trace one triangle through the whole thing. Three corners, each with a position and a color:

One Triangle — the raw data 
  Vertex 0:  position = ( 0.0,  0.5, 0.0)   color = (1.0, 0.0, 0.0)   red
  Vertex 1:  position = (-0.5, -0.5, 0.0)   color = (0.0, 1.0, 0.0)   green
  Vertex 2:  position = ( 0.5, -0.5, 0.0)   color = (0.0, 0.0, 1.0)   blue

Three vertices. Each one carries six numbers: three for position, three for color. Eighteen floats total. That is everything the GPU needs to draw a colored triangle. Now watch what happens to it:

The Four Stages

Click each stage to see what it does to our triangle:

Runs once per vertex. Takes each vertex position and transforms it from model space (where the artist built it) to screen space (where it appears on your monitor). For our flat triangle, this means multiplying each position by a Model-View-Projection matrix — the camera setup.

The vertex shader can also deform geometry: waving flags, ocean waves, skeletal animation. It bends space. But it never adds or removes vertices — three in, three out.

  // The vertex shader runs 3 times for our triangle:
  gl_Position = uMVP * vec4(aPosition, 1.0);  ← transforms the point
  vColor = aColor;                              ← passes color downstream

Output: three transformed positions in clip space. The triangle still has no pixels. It's just three points with instructions attached.

Fills the triangle with pixels. The rasterizer takes the three transformed vertices and asks: which screen pixels are inside this triangle? For each one, it creates a fragment — a candidate pixel with interpolated attributes.

Interpolation is the key. Vertex 0 is red, vertex 1 is green, vertex 2 is blue. A pixel halfway between vertices 0 and 1 gets a color halfway between red and green: yellow. A pixel in the center gets an equal mix of all three: gray-ish. The GPU does this automatically for every attribute you pass through.

  Vertex 0 (red) ─────────── Vertex 2 (blue)
        \        interpolated       /
         \       colors fill      /
          \      every pixel    /
           \                    /
            Vertex 1 (green)

Output: hundreds or thousands of fragments, each with a smoothly blended color. Still not on screen — the fragment shader gets to decide what each one actually looks like.

Runs once per pixel. The fragment shader receives the interpolated color (and any other data the vertex shader passed along) and outputs the final color. For our simple triangle, it just passes the color through:

  fragColor = vec4(vColor, 1.0);  ← that's it for now

But this is where everything interesting happens. Textures, lighting, shadows, reflections, post-processing — the fragment shader is the painter. The vertex shader decided where; the fragment shader decides what it looks like.

At 1920×1080, this shader can run two million times per frame. At 60 FPS, 120 million executions per second. The GPU runs them all in parallel.

Writes the pixel to the screen. Before writing, the GPU checks: is this pixel closer than what's already there? That's the depth test. If something was drawn in front of our triangle, this pixel gets discarded. If not, it wins and gets written to the framebuffer — the final image buffer that the display reads.

For transparent objects, there's also blending: the new color mixes with whatever was already in the buffer. And stencil testing can mask off regions of the screen entirely.

Output: a colored pixel on screen. Multiply by every triangle in the scene, every frame, and that's real-time 3D graphics.

The Fixed-Function Past

Before 2001, GPUs had no programmable stages. You configured fog, light positions, and texture blend modes — but you could not write code. The introduction of programmable vertex and fragment shaders (GeForce 3, DirectX 8) was the biggest revolution in real-time graphics. Suddenly you had a parallel processor you could program however you wanted.

What Shaders Actually Control

The pipeline tells you how pixels get made. But what do shaders let you decide? Every shader effect you have ever seen is a different strategy for answering one question: what color should this pixel be?

// 02 — Vertex Shaders: Bending Space

VERTEX → RASTERIZER → FRAGMENT → FRAMEBUFFER

Ten thousand points, each carrying 48 bytes. A character mesh has that many vertices, and every one of them must be transformed from the artist's coordinate system to your screen, sixty times per second. That is the vertex shader's job. A vertex is a point with data attached: at minimum a position (three floats), usually also a normal (which direction the surface faces), a UV coordinate (where to sample a texture), and sometimes a color, a bone weight, or whatever else the shader needs.

What's Inside a Vertex

Build a Cube: 8 Vertices, 12 Triangles

A cube has 8 corners and 6 faces. Each face is 2 triangles. That's 12 triangles total — the minimum geometry for a box. Drag to rotate it:

The vertex shader runs once for each of those 8 vertices (or 36 if we duplicate vertices for hard normals). Its job: transform every position from model space (where the artist built the cube) to clip space (where it appears on screen). That transformation is a single matrix multiply:

The MVP Transform Chain

Think of a film shoot. The Model matrix is the actor's blocking: where they stand, which way they face, how tall they are. The View matrix is the camera operator: they pick the angle, the distance, the framing. The Projection matrix is the lens: wide-angle stretches the scene, telephoto compresses it, orthographic flattens it entirely.

The vertex shader applies all three in order. The GPU receives an actor standing in a room. It moves them to their mark (M), swings the camera around (V), and distorts the image through the lens (P). What comes out is a 2D coordinate on screen. Every point of every triangle in every game goes through exactly this chain, sixty-plus times per second.

gl_Position = P * V * M * vec4(position, 1.0) — that's the whole shoot in one line.

Model-View-Projection Transform 
  Model Matrix (M)        Object → World space
  ├── Translation         Move the object in the world
  ├── Rotation            Orient the object
  └── Scale               Resize the object

  View Matrix (V)         World → Camera space
  └── Inverse of camera   "Move the world so the camera
       transform           is at the origin looking down -Z"

  Projection Matrix (P)   Camera → Clip space
  ├── Perspective          Distant objects shrink (frustum → cube)
  └── Orthographic         No foreshortening (box → cube)

  gl_Position = P * V * M * vec4(position, 1.0)

  ┌───────────┐   M   ┌───────────┐   V   ┌───────────┐   P   ┌───────────┐
  │Model Space│ ────▶  │World Space│ ────▶  │View Space │ ────▶  │Clip Space │
  └───────────┘       └───────────┘       └───────────┘       └───────────┘

Why column-major?

OpenGL stores matrices in column-major order, while DirectX uses row-major. This is purely a convention difference — the math is identical. But it means the multiplication order looks reversed: OpenGL writes P * V * M * v (right-to-left), while DirectX/HLSL writes v * M * V * P (left-to-right). Same transforms, same result, endless confusion in tutorials that mix the two.

The column-major convention dates back to FORTRAN and the original LAPACK linear algebra library. OpenGL inherited it in 1992. DirectX chose row-major in 1995. Thirty years later, every graphics programmer has been bitten by this at least once.

After the vertex shader outputs gl_Position in clip space, the GPU performs perspective division (dividing by w) to get normalized device coordinates (NDC), then maps those to screen pixels via the viewport transform. You never write code for these steps — they are fixed-function.

A Minimal Vertex Shader

GLSL
#version 300 es
precision highp float;

// Attributes — per-vertex input from buffers
in vec3 aPosition;
in vec3 aNormal;
in vec2 aTexCoord;

// Uniforms — constant for the whole draw call
uniform mat4 uModel;
uniform mat4 uView;
uniform mat4 uProjection;

// Varyings — interpolated and passed to fragment shader
out vec3 vWorldNormal;
out vec2 vTexCoord;

void main() {
    // Transform position through the MVP chain
    gl_Position = uProjection * uView * uModel * vec4(aPosition, 1.0);

    // Transform normal to world space (use inverse-transpose for non-uniform scale)
    vWorldNormal = normalize(mat3(uModel) * aNormal);

    // Pass texture coordinates through unchanged
    vTexCoord = aTexCoord;
}

Notice the pattern: the vertex shader receives attributes (per-vertex data), reads uniforms (per-draw-call constants), and writes varyings (outputs that get interpolated across the triangle for the fragment shader). This input/output contract is the fundamental interface between your application and the GPU.

Beyond Transforms: Vertex Displacement

Vertex shaders are not limited to rigid transforms. You can displace vertices procedurally — wave animations on water, terrain deformation, skeletal animation, cloth simulation. The key insight is that you can modify aPosition before multiplying by the MVP matrix:

GLSL
// Simple wave displacement
float wave = sin(aPosition.x * 4.0 + uTime * 2.0) * 0.3;
vec3 displaced = aPosition + vec3(0.0, wave, 0.0);
gl_Position = uProjection * uView * uModel * vec4(displaced, 1.0);

The vertex shader bent space. Every triangle is where it should be. But right now they are all the same color. The rasterizer is about to hand millions of fragments to the next stage. Time to paint them.

What Each Preset Actually Does

Every preset above is a complete fragment shader in 4–7 lines. Here is what each line means:

vec2 uv = gl_FragCoord.xy / u_resolution;
// Normalize pixel coords to 0..1.  Bottom-left = (0,0), top-right = (1,1).
// This is the first line of almost every fragment shader.

gl_FragColor = vec4(uv.x, uv.y, 0.5 + 0.5 * sin(u_time), 1.0);
// Red = how far right (uv.x).  Green = how far up (uv.y).
// Blue = oscillates 0..1 with time.  Alpha = 1 (fully opaque).
//
// Bottom-left is black (0,0,blue).  Top-right is yellow-ish (1,1,blue).
// The whole thing pulses because blue breathes with sin(time).
vec2 uv = gl_FragCoord.xy / u_resolution;

vec2 g = floor(uv * 10.0 + u_time);
// Scale UV to a 10x10 grid.  floor() snaps each pixel to its grid cell.
// Adding u_time makes the grid scroll diagonally over time.

float c = mod(g.x + g.y, 2.0);
// Add the cell's column + row, mod 2.  Even cells = 0 (black), odd = 1 (white).
// This is the same trick as coloring a chessboard: (row+col) % 2.

gl_FragColor = vec4(vec3(c), 1.0);
// vec3(c) = (c, c, c) = grayscale.  Black or white, nothing in between.
vec2 uv = (gl_FragCoord.xy * 2.0 - u_resolution) / min(u_resolution.x, u_resolution.y);
// Remap coords to -1..+1, centered, aspect-ratio corrected.
// (0,0) is screen center.  This is the standard "Shadertoy UV" setup.

float d = length(uv) - 0.5;
// Signed distance from a circle of radius 0.5.
// d < 0 = inside.  d = 0 = on the edge.  d > 0 = outside.

float pulse = 0.02 * (1.0 + sin(u_time * 3.0));
// A small value that breathes with time. Controls edge softness.

vec3 col = mix(vec3(0,.8,.4), vec3(.05), smoothstep(-pulse, pulse, d));
// Inside the circle: green (0, 0.8, 0.4).  Outside: near-black (0.05).
// smoothstep makes the transition soft instead of a hard edge.

col += vec3(.7,.3,.1) * exp(-8.0 * abs(d));
// Glow! exp(-8*|d|) peaks at the edge (d=0) and fades fast.
// Adds an orange-ish (0.7, 0.3, 0.1) ring that pulses on the boundary.
vec2 uv = gl_FragCoord.xy / u_resolution;

float v  = sin(uv.x * 10.0 + u_time);         // vertical waves
       v += sin(uv.y * 10.0 + u_time);         // + horizontal waves
       v += sin((uv.x + uv.y) * 10.0 + u_time * 0.5); // + diagonal waves
       v += sin(length(uv - 0.5) * 20.0 - u_time * 2.0); // + radial waves from center
// Four overlapping sine waves at different angles and speeds.
// Their interference pattern creates the "plasma" effect.

v *= 0.25;
// Average the four waves back to roughly -1..+1 range.

gl_FragColor = vec4(
    sin(v * 3.14159),                // R: sine of the combined wave
    sin(v * 3.14159 + 2.094),        // G: same, phase-shifted by 2π/3
    sin(v * 3.14159 + 4.189),        // B: same, phase-shifted by 4π/3
    1.0
);
// Three sines 120° apart = full RGB rainbow cycle.
// As v changes smoothly across the screen, the color rotates
// through red → green → blue → red.  Classic 90s demo effect.

The Language
You know where shaders run. Now learn how to speak to them.

// 04 — GLSL: The GPU's Language

No pointers. No recursion. No dynamic allocation. The GPU stripped away half the language you know and gave you something back: vectors as primitives, matrices as first-class citizens, and the guarantee that your code runs on thousands of cores simultaneously. That is GLSL (OpenGL Shading Language). It looks like C, but it thinks like a graphics card: every type is designed for the operations shaders actually need, and every missing feature is missing because parallelism demanded the sacrifice.

Core Types

In C, you write float r, g, b; and then manually add them component by component. In GLSL, you write vec3 color = vec3(1.0, 0.0, 0.5); and the GPU operates on all three components in a single instruction. Vectors are not library types wrapped around arrays. They are hardware primitives: the GPU's registers are physically 4-wide. GLSL's type system is small but powerful:

GLSL
// Scalar types
float brightness = 0.8;
int   count      = 42;
bool  visible    = true;

// Vector types — 2, 3, or 4 components
vec2 uv       = vec2(0.5, 0.7);           // 2D coordinate
vec3 color    = vec3(1.0, 0.0, 0.5);      // RGB color
vec4 position = vec4(1.0, 2.0, 3.0, 1.0); // homogeneous coordinate

// Matrix types — column-major
mat4 transform = mat4(1.0);               // 4x4 identity matrix

// Sampler types — texture handles
uniform sampler2D uTexture;               // 2D texture

// Swizzling — reorder/repeat components
vec3 rgb = color.rgb;                      // same as color
vec3 bgr = color.bgr;                      // reversed
vec2 xx  = color.xx;                        // repeat x twice

Swizzling

Every pixel color is three numbers: Red, Green, Blue. Swizzling lets you rearrange which number goes where in a single instruction.

Write .bgr and the GPU takes the Blue value, puts it in the Red slot. Takes Green, keeps it. Takes Red, puts it in the Blue slot. The image flips from warm to cool in one operation. No loops, no temporary variables, no cost.

You can also duplicate channels: .rrr takes just the Red value and copies it into all three slots. That gives you a grayscale image derived only from the red channel. .ggg does the same from green.

Think of it like a mixing board. Each output channel has a selector: "which input channel feeds me?" Swizzling is setting those selectors. Three letters, three selectors, instant color transformation.

The three notations .xyzw, .rgba, and .stpq are aliases for the same four slots. Use .rgb when you're thinking about color. Use .xyz when you're thinking about position. Use .xy to grab just the first two (for UV coordinates). They all compile to the same thing.

GLSL
vec4 v = vec4(1.0, 2.0, 3.0, 4.0);
//              x/r   y/g   z/b   w/a

v.xy      // vec2(1.0, 2.0)       ← grab first two
v.zyx     // vec3(3.0, 2.0, 1.0)  ← reverse order
v.www     // vec3(4.0, 4.0, 4.0)  ← duplicate one channel
v.rgb     // vec3(1.0, 2.0, 3.0)  ← same as .xyz, color alias
v.rr      // vec2(1.0, 1.0)       ← repeat

// Swizzle on assignment too:
v.rb = vec2(0.5, 0.8);  // sets v.r=0.5, v.b=0.8, leaves g and a alone

// Practical: extract position from a homogeneous vec4
vec4 clipPos = uMVP * vec4(aPosition, 1.0);
vec3 ndc = clipPos.xyz / clipPos.w;  ← perspective divide via swizzle

// 05 — UV Coordinates and Color Math

Every fragment shader you have written so far started with the same line: vec2 uv = gl_FragCoord.xy / u_resolution. You divided your pixel coordinate by the screen size and got a number between 0 and 1. That number is a UV coordinate, and it is the address system for everything that follows: textures, patterns, noise, distance fields. UV coordinates are the bridge between 2D surfaces and 3D geometry. Every vertex in a mesh carries a vec2 called its UV (also called texture coordinate), which maps that vertex to a point on a flat 2D plane. The convention is that (0,0) is one corner and (1,1) is the opposite corner. The rasterizer interpolates UVs across each triangle, giving every fragment a unique position in texture space.

But UVs are not just for textures. In fullscreen-quad shaders, we normalize gl_FragCoord into a 0–1 range and treat it exactly like a UV. This normalized coordinate becomes the input to every procedural pattern, distance function, and color calculation.

Displays are nonlinear. A pixel value of 0.5 does not produce half the brightness of 1.0 — it produces roughly 22% brightness on a typical monitor (gamma ~2.2). // 06 — Procedural Patterns

A 512×512 checkerboard texture costs 1 MB of GPU memory and pixelates when you zoom in. The same checkerboard generated from floor(uv * 8.0) costs zero memory, stays razor-sharp at any zoom level, and takes three lines of GLSL. No textures needed. Every building block of shader graphics — stripes, grids, circles, polka dots — can be generated from pure math. Periodic functions like sin(), mod(), and floor() naturally create repeating patterns when applied to UV coordinates. A checkerboard is floor(uv.x) + floor(uv.y) passed through mod 2. Stripes are a sin() or step() on one axis. Circles are a distance function from a center point.

The beauty of procedural patterns is that they have infinite resolution. Zoom in and they remain perfectly crisp, because you are evaluating a mathematical function at each pixel, not sampling from a finite grid of texels. They also cost zero memory — the “texture” exists only as instructions.

// 07 — Noise: Controlled Chaos

You unlocked fract() in the last chapter and used it to tile patterns. Noise functions take the same idea and add randomness at the grid points. Instead of a clean checkerboard, you get clouds, terrain, fire, marble. fract() still drives the repetition, but a hash function scrambles the values at each cell boundary. Ken Perlin developed the first practical noise function after working on Tron (1982), frustrated by the lack of organic-looking textures in CGI. He published it at SIGGRAPH 1985, and noise has been the backbone of procedural graphics ever since.

Imagine a bedsheet stretched over a grid of poles. Each pole is a different random height. The sheet drapes smoothly between them — never sharp, never discontinuous. That's noise: random at the control points, smooth everywhere else.

Now stack five bedsheets, each one with poles twice as close together and half as tall. The first sheet makes broad, rolling hills. The second adds ridges. The third adds wrinkles. By the fifth layer, you have something that looks like terrain. That's fractal Brownian motion (fBm) — layered noise at different scales.

The key insight: noise is not random(). It's a smooth function that feels random. Same input always gives the same output. That's what makes it deterministic and tileable.

Value Noise

Assign random values to grid points, interpolate between them. Simple but blocky — the grid structure shows through in the artifacts. Cheap to compute.

Perlin & Simplex Noise

Perlin noise uses gradient vectors at each grid point instead of scalar values, eliminating the blocky look. Simplex noise (also by Perlin, 2001) uses a simplex grid (triangles in 2D, tetrahedra in 3D) for better performance in higher dimensions and no directional artifacts.

Fractal Brownian Motion (fBm)

Layer multiple octaves of noise at increasing frequency and decreasing amplitude. Each octave adds finer detail. 4–8 octaves covers most use cases. The lacunarity (frequency multiplier, usually 2.0) and gain (amplitude multiplier, usually 0.5) control the character of the result.

Perlin won an Oscar for this

In 1997, Ken Perlin received a Technical Achievement Academy Award for his noise function. The core implementation fits in a few dozen lines of C. It is one of the few algorithms in history to win an Oscar.

The Tron (1982) connection: Perlin worked on the film's CGI at MAGI and was frustrated by how synthetic everything looked. After the film, he developed noise to solve the organic-texture problem. The technique was published at SIGGRAPH 1985 and has been the foundation of procedural graphics for 40 years. Perlin noise was not in the original Tron, but Tron is why it exists.

Fun fact: the "classic" Perlin noise in most tutorials is actually the 2002 "improved" version. The 1985 original had a subtle directional bias that Perlin fixed 17 years later.

The Lattice Trick

All classic noise functions share a key insight: place random data at integer grid points, then smoothly interpolate between them. The “random” values are computed by a hash function from integer coordinates, so the same input always gives the same output — deterministic randomness. The magic is in the interpolation: smoothstep hides grid artifacts. Perlin’s further trick was using gradient vectors instead of scalar values, eliminating directional bias.

// 08 — Signed Distance Functions

Remember length() from the circle preset in chapter 3? length(uv) - 0.5 gave you the distance from a point to a circle edge. An SDF is that idea applied to every shape imaginable. A Signed Distance Function returns the shortest distance from any point to a surface: positive outside, negative inside, zero on the boundary. With length() and a few min()/max() combinations, you can sculpt entire worlds without a single triangle.

Close your eyes and hold your hand out. You can feel how far you are from the wall without touching it. That sense of distance — positive when you're away from the surface, zero when you touch it, "inside" when your hand passes through — is exactly what an SDF computes.

Now imagine asking that question at every pixel on screen: how far am I from the nearest surface? If the answer is positive, you're in empty space — paint the background. If it's near zero, you've found an edge — draw the outline. If it's negative, you're inside — fill with color.

The breakthrough: you don't need triangles. You don't need meshes. You define a shape as a distance equation, and the GPU evaluates it at every pixel. A circle is length(p) - radius. A box is a few abs() and max() calls. Boolean operations are just min() and max() on distances. You sculpt with algebra.

SDF Primitives

The two workhorses are the circle and box SDFs. A circle SDF is just the distance from the center minus the radius. A box SDF is trickier — it uses abs() to exploit symmetry and max() to handle corners.

Boolean Operations

Combine SDFs with simple min/max operations: union (min(a,b)), intersection (max(a,b)), subtraction (max(a,-b)). For organic blends, use smooth union with an exponential or polynomial falloff.

Ray Marching

To render an SDF in 3D, march a ray forward by the SDF distance at each step. If the SDF returns 0.01, you're 0.01 units from the surface — safe to step that far. This "sphere tracing" converges quickly near surfaces and leaps through empty space.

SDFs power your fonts

Valve published "Improved Alpha-Tested Magnification for Vector Textures and Special Effects" in 2007. The core idea: store fonts as SDF textures. A single 64×64 SDF texture can render a glyph at any size with sharp edges, outlines, drop shadows, and glow — all from the same texture, all in the fragment shader. This technique is now standard in game engines (Unity TextMeshPro, Unreal Slate) and replaced bitmap font atlases almost everywhere.

The same idea extends to icons, logos, and UI elements. Any 2D shape that can be described as a distance field gets infinite resolution for free.

Ray Marching

Ray marching is the GPU’s answer to “how do I render implicit surfaces?” For each pixel, cast a ray from the camera. At each step, evaluate the SDF — it tells you the safe step size. If the SDF returns 2.0, nothing is closer than 2.0 units, so jump ahead. Near the surface the steps get tiny, converging to a hit. The beauty is that you never need to triangulate anything — the math is the geometry. Inigo Quilez used this technique to build entire worlds in ShaderToy with zero mesh data.

// 09 — Lighting Models

You have blend (mix), distance (length), and repetition (fract). Everything you have built so far looks flat. The missing ingredient is angle: how the surface faces the light. That is dot(), the fourth and final tool. One dot product turns a flat circle into a sphere. Every lighting model in existence is built from this single operation.

Phong & Blinn-Phong

Hold a sheet of paper in front of a lamp. Tilt it to face the light directly: bright. Tilt it sideways: dim. Turn it away: dark. That brightness change is dot(N, L). The surface normal N says which way the paper faces. The light direction L says where the lamp is. The dot product measures how aligned they are: 1.0 facing the light head-on, 0.0 edge-on, negative facing away. One number, one operation, and suddenly flat shapes have depth.

The Phong model (1975) decomposes light into three components: ambient (constant fill light so nothing is pure black), diffuse (max(dot(N,L),0), the broad shading from the paper experiment above), and specular (the shiny highlight you see when the surface angle mirrors the light toward your eye). Blinn-Phong (1977) replaces the reflection vector with the half vector H = normalize(L+V), which is cheaper and looks better at grazing angles.

Physically-Based Rendering (PBR)

PBR models use the Cook-Torrance BRDF: a microfacet distribution (GGX), a geometry term (Smith), and a Fresnel term (Schlick). The key insight is energy conservation — a surface can’t reflect more light than it receives. PBR uses two intuitive parameters: metalness and roughness.

// Blinn-Phong lighting
vec3 blinnPhong(vec3 N, vec3 L, vec3 V,
                vec3 albedo, float shininess) {
    vec3 ambient = 0.05 * albedo;
    float diff = max(dot(N, L), 0.0);
    vec3 H = normalize(L + V);
    float spec = pow(max(dot(N, H), 0.0), shininess);
    return ambient + diff * albedo + spec * vec3(1.0);
}

Normal Precision

Normals must be unit length for lighting math to work. After interpolation across a triangle, normals drift from unit length — a normal of length 0.98 darkens the surface by 2%. Always normalize() interpolated normals in the fragment shader. In mobile GPUs with mediump (16-bit floats), normalization error compounds: a reflected specular highlight can flicker or smear. If your specular looks broken on mobile, upgrade the normal to highp before the lighting calculation.

ALU vs. Bandwidth


Modern GPUs have far more compute power than memory bandwidth. A shader that does 100 ALU operations per pixel can be faster than one that does 5 texture lookups, because the ALU operations run in parallel across thousands of threads while the texture lookups stall waiting for memory. This is why procedural techniques (noise, SDFs) often outperform texture-based approaches.



The Branch Problem


GPUs execute threads in warps (32 threads on NVIDIA, 64 on AMD) — think of a rowing crew: all 32 oars must move in sync. When threads in a warp take different branches, the GPU cannot split the crew. Instead it executes both branches and masks the results — called divergent branching. Half the crew rows while the other half holds still, then they swap. A simple if that looks innocent can double your shader cost if adjacent pixels take different paths.



// BAD: divergent branch — adjacent pixels may differ
if (texture2D(u_tex, uv).r > 0.5) {
    // expensive path A
} else {
    // expensive path B
}

// GOOD: branchless with mix()
float mask = step(0.5, texture2D(u_tex, uv).r);
vec3 result = mix(pathB, pathA, mask);

The Branch Tax


Here’s the insidious part: the cost of a divergent branch isn’t just running both paths. It also destroys texture cache coherency. When half a warp takes path A and samples texture coordinate X, while the other half takes path B and samples coordinate Y, you get two cache misses instead of one. The branch and the texture fetch compound each other. In a ray marching shader, this means a divergent early-exit condition (like “stop if distance < epsilon”) can actually be slower than just running all threads for the maximum step count. Profile before you optimize — your intuition about what’s fast on a GPU is probably wrong.
