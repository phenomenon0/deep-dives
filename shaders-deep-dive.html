SHADERS

Painting with math. Every pixel on your screen exists because a tiny program ran millions of times per frame, turning coordinates into colors._

// 01 — The Rendering Pipeline

Every frame your GPU draws begins as a river of data flowing through a fixed sequence of stages. This is the rendering pipeline — the assembly line that transforms raw vertex coordinates into the colored pixels you see on screen. Understanding it is the prerequisite to writing any shader, because a shader is simply a programmable stage within this pipeline. You are not painting pixels directly; you are writing tiny programs that the GPU executes thousands of times in parallel at specific points in this flow.

The modern pipeline has both fixed-function stages (you configure them but cannot reprogram them) and programmable stages (you supply the actual code). The two most important programmable stages are the vertex shader and the fragment shader.

The Full Stage Sequence

GPU Rendering Pipeline 
  Vertex Data              Raw positions, normals, UVs in buffers
       │
       ▼
  ┌─────────────────┐
  │  Vertex Shader   │    ← PROGRAMMABLE: transform each vertex
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │  Rasterizer      │    ← FIXED: convert triangles → fragments
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │ Fragment Shader  │    ← PROGRAMMABLE: compute color per pixel
  └────────┬────────┘
           ▼
  ┌─────────────────┐
  │  Framebuffer     │    ← OUTPUT: depth test, blend, write pixel
  └─────────────────┘

Vertex Data: Your application uploads arrays of vertex attributes to the GPU — positions (vec3), normals (vec3), texture coordinates (vec2), and whatever custom data you need. These sit in vertex buffer objects (VBOs) in GPU memory.

Vertex Shader: Runs once per vertex. Its primary job is to transform the vertex position from model space through world space and view space into clip space. It can also pass data (colors, UVs, normals) downstream to the fragment shader via varyings.

Rasterizer: This fixed-function stage takes the transformed triangles and determines which pixels on screen they cover. For each covered pixel, it generates a fragment — a candidate pixel with interpolated attributes. The interpolation is perspective-correct by default.

Fragment Shader: Runs once per fragment. It receives the interpolated data from the rasterizer and outputs a final color. This is where texturing, lighting, shadows, and every visual effect happen. If the vertex shader bends space, the fragment shader paints it.

Framebuffer Operations: After the fragment shader, the GPU performs depth testing (is this fragment in front of what is already there?), stencil testing, and blending (for transparency). Survivors get written to the framebuffer and eventually displayed.

For a deeper look at how the GPU executes all of this in parallel across thousands of cores, see the GPU compute deep dive.

The Fixed-Function Past

Before shaders existed, GPUs had a completely fixed pipeline. You could configure parameters — set a light position, enable fog, choose a texture blend mode — but you could not write custom code. OpenGL 1.x and DirectX 7 were entirely this way. The introduction of programmable shaders (GeForce 3 / DirectX 8, circa 2001) was the single biggest revolution in real-time graphics. Suddenly every developer had a parallel processor they could program however they wanted.

// 02 — Vertex Shaders: Bending Space

Vertices are the skeleton of every 3D scene. A mesh is nothing more than a cloud of points connected by edges into triangles. The vertex shader’s job is to take each of these points and transform it from its original model-space position into the final clip-space coordinate that the rasterizer needs. This single operation — multiplying a position by a chain of matrices — is the beating heart of all 3D rendering.

The transform chain involves three matrices, typically combined into a single MVP matrix:

The MVP Transform Chain

Model-View-Projection Transform 
  Model Matrix (M)        Object → World space
  ├── Translation         Move the object in the world
  ├── Rotation            Orient the object
  └── Scale               Resize the object

  View Matrix (V)         World → Camera space
  └── Inverse of camera   "Move the world so the camera
       transform           is at the origin looking down -Z"

  Projection Matrix (P)   Camera → Clip space
  ├── Perspective          Distant objects shrink (frustum → cube)
  └── Orthographic         No foreshortening (box → cube)

  gl_Position = P * V * M * vec4(position, 1.0)

  ┌───────────┐   M   ┌───────────┐   V   ┌───────────┐   P   ┌───────────┐
  │Model Space│ ────▶  │World Space│ ────▶  │View Space │ ────▶  │Clip Space │
  └───────────┘       └───────────┘       └───────────┘       └───────────┘

After the vertex shader outputs gl_Position in clip space, the GPU performs perspective division (dividing by w) to get normalized device coordinates (NDC), then maps those to screen pixels via the viewport transform. You never write code for these steps — they are fixed-function.

A Minimal Vertex Shader

GLSL
#version 300 es
precision highp float;

// Attributes — per-vertex input from buffers
in vec3 aPosition;
in vec3 aNormal;
in vec2 aTexCoord;

// Uniforms — constant for the whole draw call
uniform mat4 uModel;
uniform mat4 uView;
uniform mat4 uProjection;

// Varyings — interpolated and passed to fragment shader
out vec3 vWorldNormal;
out vec2 vTexCoord;

void main() {
    // Transform position through the MVP chain
    gl_Position = uProjection * uView * uModel * vec4(aPosition, 1.0);

    // Transform normal to world space (use inverse-transpose for non-uniform scale)
    vWorldNormal = normalize(mat3(uModel) * aNormal);

    // Pass texture coordinates through unchanged
    vTexCoord = aTexCoord;
}

Notice the pattern: the vertex shader receives attributes (per-vertex data), reads uniforms (per-draw-call constants), and writes varyings (outputs that get interpolated across the triangle for the fragment shader). This input/output contract is the fundamental interface between your application and the GPU.

Beyond Transforms: Vertex Displacement

Vertex shaders are not limited to rigid transforms. You can displace vertices procedurally — wave animations on water, terrain deformation, skeletal animation, cloth simulation. The key insight is that you can modify aPosition before multiplying by the MVP matrix:

GLSL
// Simple wave displacement
float wave = sin(aPosition.x * 4.0 + uTime * 2.0) * 0.3;
vec3 displaced = aPosition + vec3(0.0, wave, 0.0);
gl_Position = uProjection * uView * uModel * vec4(displaced, 1.0);

// 03 — Fragment Shaders: Painting Pixels

Fragment shaders are where the magic happens. While the vertex shader deals with geometry — where things are — the fragment shader deals with appearance — what things look like. It runs once for every fragment (potential pixel) that the rasterizer produces, and it outputs a single color. At 1920×1080 resolution, that means up to two million fragment shader invocations per frame, per layer of geometry. At 60 FPS, you are looking at over 120 million shader executions per second — and the GPU handles them all in parallel.

The fragment shader receives interpolated varyings from the vertex shader. If three vertices of a triangle have colors red, green, and blue, a fragment in the center gets a smooth blend of all three. This interpolation is automatic and perspective-correct.

The Fullscreen Quad Trick

Most shader art and post-processing effects do not use 3D geometry at all. They render a single quad (two triangles) that covers the entire screen, and the fragment shader does all the work using gl_FragCoord to know which pixel it is computing. Shadertoy, The Book of Shaders, and countless demoscene entries all use this technique. The vertex shader becomes trivially simple — just pass through the corner positions — and the fragment shader becomes the entire program. You are essentially writing a function f(x, y, t) → color that runs for every pixel on screen.

// 04 — GLSL: The Language of Light

GLSL (OpenGL Shading Language) is a C-like language purpose-built for GPU execution. It looks familiar if you know C, but it has first-class support for vectors, matrices, and the kinds of math operations that graphics programmers use constantly. There are no pointers, no dynamic allocation, no recursion — these constraints exist because GPU hardware is designed for massive parallelism, not general-purpose computing.

Core Types

GLSL’s type system is small but powerful. Vectors and matrices are built-in primitives, not library types:

GLSL
// Scalar types
float brightness = 0.8;
int   count      = 42;
bool  visible    = true;

// Vector types — 2, 3, or 4 components
vec2 uv       = vec2(0.5, 0.7);           // 2D coordinate
vec3 color    = vec3(1.0, 0.0, 0.5);      // RGB color
vec4 position = vec4(1.0, 2.0, 3.0, 1.0); // homogeneous coordinate

// Matrix types — column-major
mat4 transform = mat4(1.0);               // 4x4 identity matrix

// Sampler types — texture handles
uniform sampler2D uTexture;               // 2D texture

// Swizzling — reorder/repeat components
vec3 rgb = color.rgb;                      // same as color
vec3 bgr = color.bgr;                      // reversed
vec2 xx  = color.xx;                        // repeat x twice

// 05 — UV Coordinates and Color Math

UV coordinates are the bridge between 2D textures and 3D geometry. Every vertex in a mesh carries a vec2 called its UV coordinate (also called texture coordinate), which maps that vertex to a point in a 2D texture. The convention is that (0,0) is one corner and (1,1) is the opposite corner. The rasterizer interpolates UVs across each triangle, giving every fragment a unique position in texture space.

But UVs are not just for textures. In fullscreen-quad shaders, we normalize gl_FragCoord into a 0–1 range and treat it exactly like a UV. This normalized coordinate becomes the input to every procedural pattern, distance function, and color calculation.

UV Space

UV Coordinate Space 
  (0,1) ──────────────── (1,1)
    │                      │
    │    UV Space           │
    │    (0,0) = bottom-left│
    │    (1,1) = top-right  │
    │                      │
    │    Normalizing:       │
    │    uv = gl_FragCoord.xy / u_resolution;
    │                      │
  (0,0) ──────────────── (1,0)

Color Mixing and Lerp

The mix() function in GLSL performs linear interpolation (lerp) between two values. It is the single most-used function in shader programming:

GLSL
// mix(a, b, t) = a * (1.0 - t) + b * t

vec3 red   = vec3(1.0, 0.0, 0.0);
vec3 blue  = vec3(0.0, 0.0, 1.0);
vec3 blend = mix(red, blue, 0.5);  // purple: (0.5, 0.0, 0.5)

// Gradient across the screen:
vec3 gradient = mix(red, blue, uv.x); // red on left, blue on right

Gamma Correction

Displays are nonlinear. A pixel value of 0.5 does not produce half the brightness of 1.0 — it produces roughly 22% brightness on a typical monitor (gamma ~2.2). If you blend colors in sRGB space (the default), your gradients will have dark bands in the midtones. The fix is to work in linear space and convert back at the end:

GLSL
// sRGB → linear (approximate)
vec3 toLinear(vec3 srgb) {
    return pow(srgb, vec3(2.2));
}

// linear → sRGB (approximate)
vec3 toSRGB(vec3 linear) {
    return pow(linear, vec3(1.0 / 2.2));
}

import numpy as np

def lerp(a: np.ndarray, b: np.ndarray, t: float) -> np.ndarray:
    """Linear interpolation between two color arrays."""
    return a * (1.0 - t) + b * t

color_a = np.array([1.0, 0.2, 0.0], dtype=np.float32)  # orange
color_b = np.array([0.0, 0.4, 1.0], dtype=np.float32)  # blue

# Blend at t = 0.5
result = lerp(color_a, color_b, 0.5)
print(result)  # [0.5 0.3 0.5]

# Smoothstep equivalent
def smoothstep(edge0: float, edge1: float, x: float) -> float:
    t = np.clip((x - edge0) / (edge1 - edge0), 0.0, 1.0)
    return t * t * (3.0 - 2.0 * t)

// 06 — Procedural Patterns

Stripes, checkerboards, circles, grids — the building blocks of every shader effect can be generated from pure math. No textures needed. The key insight is that periodic functions like sin(), mod(), and floor() naturally create repeating patterns when applied to UV coordinates. A checkerboard is nothing more than floor(uv.x) + floor(uv.y) passed through mod 2. Stripes are a sin() or step() on one axis. Circles are a distance function from a center point.

The beauty of procedural patterns is that they have infinite resolution. Zoom in and they remain perfectly crisp, because you are evaluating a mathematical function at each pixel, not sampling from a finite grid of texels. They also cost zero memory — the “texture” exists only as instructions.

The Checkerboard: Hello World of Shaders

The classic checkerboard is the shader equivalent of “Hello World.” It combines scaling, floor(), and mod() in a pattern that every shader programmer should be able to write from memory:

GLSL
// Checkerboard pattern
vec2 scaled = uv * 8.0;                      // 8x8 grid
float check = mod(floor(scaled.x) + floor(scaled.y), 2.0);
vec3 color = vec3(check);                     // black and white

Procedural Checkerboard Across Languages

GLSL
#version 300 es
precision mediump float;
uniform vec2 u_resolution;
out vec4 fragColor;

void main() {
    vec2 uv = gl_FragCoord.xy / u_resolution;
    float scale = 8.0;
    vec2 grid = floor(uv * scale);
    float check = mod(grid.x + grid.y, 2.0);
    fragColor = vec4(vec3(check), 1.0);
}

Infinite Resolution

Procedural patterns never pixelate. A texture-based checkerboard at 256×256 pixels will show aliasing artifacts when you zoom in. A procedural checkerboard stays razor-sharp at any zoom level, because each pixel recomputes the function from scratch. The trade-off is that procedural patterns require GPU computation every frame, while textures just do a memory lookup. In practice, the GPU is so fast at math that procedural generation is often cheaper than the memory bandwidth cost of sampling a large texture.

// 07 — Noise: Controlled Chaos

Nature is messy. Clouds, terrain, fire, marble — none of these have clean geometric shapes. To simulate organic randomness in shaders, we need noise functions: deterministic functions that look random but are smooth and continuous. Ken Perlin invented the first practical one in 1983 for Tron, and noise has been the backbone of procedural graphics ever since.

Value Noise

The simplest approach: assign random values to grid points, then interpolate between them. The result is smooth, organic-looking variation.

Perlin and Simplex Noise

Perlin noise uses gradient vectors at each grid point instead of scalar values, eliminating blocky artifacts. Simplex noise (also by Perlin, 2001) uses a simplex grid for better performance.

Fractal Brownian Motion (fBm)

Layer multiple octaves of noise at different frequencies and amplitudes to create complex, natural-looking detail.

Noise Explorer

// 2D value noise
float hash(vec2 p) {
    return fract(sin(dot(p, vec2(127.1, 311.7))) * 43758.5453);
}
float valueNoise(vec2 p) {
    vec2 i = floor(p), f = fract(p);
    vec2 u = f*f*(3.0-2.0*f);
    return mix(mix(hash(i),hash(i+vec2(1,0)),u.x),
               mix(hash(i+vec2(0,1)),hash(i+vec2(1,1)),u.x),u.y);
}

    n = x*374761393 + y*668265263
    n = (n ^ (n >> 13)) * 1274126177
    return (n & 0x7fffffff) / 0x7fffffff
def value_noise(x, y):
    ix, iy = int(math.floor(x)), int(math.floor(y))
    fx, fy = x-ix, y-iy

The Lattice Trick

All classic noise functions share a key insight: place random data at integer grid points, then smoothly interpolate between them. The “random” values are computed by a hash function from integer coordinates, so the same input always gives the same output — deterministic randomness. The magic is in the interpolation: smoothstep hides grid artifacts. Perlin’s further trick was using gradient vectors instead of scalar values, eliminating directional bias.

// 08 — Signed Distance Functions

A Signed Distance Function (SDF) returns the shortest distance from any point to a surface — positive outside, negative inside, zero on the boundary. This deceptively simple idea powers everything from font rendering to real-time ray marching. With a handful of SDF primitives and a few boolean operations, you can sculpt entire worlds without a single triangle.

SDF Primitives

The two workhorses are the circle and box SDFs. A circle SDF is just the distance from the center minus the radius. A box SDF is trickier — it uses abs() to exploit symmetry and max() to handle corners.

Boolean Operations

Combine SDFs with simple min/max operations: union (min(a,b)), intersection (max(a,b)), subtraction (max(a,-b)). For organic blends, use smooth union with an exponential or polynomial falloff.

Ray Marching

To render an SDF in 3D, march a ray forward by the SDF distance at each step. If the SDF returns 0.01, you’re 0.01 units from the surface — safe to step that far. This “sphere tracing” converges quickly near surfaces and leaps through empty space.

SDF Builder

Combine two SDF shapes with boolean operations. Adjust the blend factor for smooth transitions.

Ray Marching

Ray marching is the GPU’s answer to “how do I render implicit surfaces?” For each pixel, cast a ray from the camera. At each step, evaluate the SDF — it tells you the safe step size. If the SDF returns 2.0, nothing is closer than 2.0 units, so jump ahead. Near the surface the steps get tiny, converging to a hit. The beauty is that you never need to triangulate anything — the math is the geometry. Inigo Quilez used this technique to build entire worlds in ShaderToy with zero mesh data.

// 09 — Lighting Models

Light is what makes flat shading look three-dimensional. Every lighting model boils down to the same question: given a surface point, its normal, the light direction, and the view direction, how bright should this pixel be? The answer has evolved from crude approximations to physically-based models that fool the eye.

Phong & Blinn-Phong

The Phong model (1975) decomposes light into three components: ambient (constant fill), diffuse (max(dot(N,L),0)), and specular (highlight from the reflection vector). Blinn-Phong (1977) replaces the reflection vector with the half vector H = normalize(L+V), which is cheaper and looks better at grazing angles.

Physically-Based Rendering (PBR)

PBR models use the Cook-Torrance BRDF: a microfacet distribution (GGX), a geometry term (Smith), and a Fresnel term (Schlick). The key insight is energy conservation — a surface can’t reflect more light than it receives. PBR uses two intuitive parameters: metalness and roughness.

// Blinn-Phong lighting
vec3 blinnPhong(vec3 N, vec3 L, vec3 V,
                vec3 albedo, float shininess) {
    vec3 ambient = 0.05 * albedo;
    float diff = max(dot(N, L), 0.0);
    vec3 H = normalize(L + V);
    float spec = pow(max(dot(N, H), 0.0), shininess);
    return ambient + diff * albedo + spec * vec3(1.0);
}

Normal Precision

Normals must be unit length for lighting math to work. After interpolation across a triangle, normals drift from unit length — a normal of length 0.98 darkens the surface by 2%. Always normalize() interpolated normals in the fragment shader. In mobile GPUs with mediump (16-bit floats), normalization error compounds: a reflected specular highlight can flicker or smear. If your specular looks broken on mobile, upgrade the normal to highp before the lighting calculation.

// 10 — Post-Processing Effects

Post-processing treats the entire rendered frame as a texture and applies full-screen shader passes to it. This is where bloom, blur, color grading, and CRT effects live. Bloom in Detail

Extract bright pixels (above a threshold), blur them with a Gaussian kernel, then add the blurred result back to the original image. The key optimization is separable blur: a 2D Gaussian can be split into a horizontal pass followed by a vertical pass, reducing an N×N kernel from O(N²) to O(2N) texture samples.

CRT Shader

The retro CRT look combines several effects: scanlines (darken every other row), chromatic aberration (offset R/G/B channels), barrel distortion (curve the screen), and a vignette (darken edges). Each is a few lines of GLSL:

// 11 — Shader Art and the Demoscene

The demoscene is a computer art subculture where programmers create real-time audiovisual demonstrations under extreme constraints. A 4K intro is an entire demo — music, visuals, everything — in 4,096 bytes. Many of these masterpieces are essentially a single fragment shader running a ray marcher through a procedural world, with the “music” synthesized by another shader or a tiny softsynth.

ShaderToy and the New Golden Age

Inigo Quilez’s ShaderToy democratized shader art. A single fragment shader gets four uniforms (iTime, iResolution, iMouse, iChannel0..3) and produces a color for every pixel. The constraints are liberating: no meshes, no engine, no boilerplate — just math. Some ShaderToy shaders render photorealistic scenes, fractal landscapes, or fluid simulations in under 300 lines of GLSL.

Techniques of the Masters

Demoscene coders exploit every trick: domain repetition (mod(p, period) to tile space infinitely), domain warping (feed noise into noise coordinates), orbit traps (color fractals by tracking closest approach to a geometric shape), and analytic ambient occlusion (approximate AO from the SDF gradient).

// 12 — Performance: The GPU’s Price List

GPUs are massively parallel but deeply unforgiving. A single branch can stall an entire warp. A single texture lookup at the wrong time can blow your bandwidth budget. Understanding what’s cheap and what’s expensive on a GPU is the difference between 60 fps and a slideshow.

ALU vs. Bandwidth

Modern GPUs have far more compute power than memory bandwidth. A shader that does 100 ALU operations per pixel can be faster than one that does 5 texture lookups, because the ALU operations run in parallel across thousands of threads while the texture lookups stall waiting for memory. This is why procedural techniques (noise, SDFs) often outperform texture-based approaches.

The Branch Problem

GPUs execute threads in warps (32 threads on NVIDIA, 64 on AMD). When threads in a warp take different branches, the GPU executes both branches and masks the results — called divergent branching. A simple if that looks innocent can double your shader cost if adjacent pixels take different paths.

// BAD: divergent branch — adjacent pixels may differ
if (texture2D(u_tex, uv).r > 0.5) {
    // expensive path A
} else {
    // expensive path B
}

// GOOD: branchless with mix()
float mask = step(0.5, texture2D(u_tex, uv).r);
vec3 result = mix(pathB, pathA, mask);

The Branch Tax

Here’s the insidious part: the cost of a divergent branch isn’t just running both paths. It also destroys texture cache coherency. When half a warp takes path A and samples texture coordinate X, while the other half takes path B and samples coordinate Y, you get two cache misses instead of one. The branch and the texture fetch compound each other. In a ray marching shader, this means a divergent early-exit condition (like “stop if distance < epsilon”) can actually be slower than just running all threads for the maximum step count. Profile before you optimize — your intuition about what’s fast on a GPU is probably wrong.
