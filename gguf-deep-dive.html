GGUF, DECODED

You type ollama pull qwen3:8b. Forty seconds later you’re chatting with 8 billion parameters. The runtime loaded 370 matrices, wired a computation graph, and started generating. How did it know what to do? The answer is in the file._
The Efficiency Papers, Part II-B

01 // The File

DOWNLOAD → HEADER → METADATA → TENSORS → WEIGHTS ← the whole map

You download one file. It contains the architecture, the tokenizer, the quantization parameters, and the weights. This is what makes it work.

If you have ever run a local model — typed ollama pull, downloaded a .gguf from HuggingFace, or pointed llama.cpp at a file — you have already used this format. You just never had to think about it. The model loaded, the chat worked, and the file sat there doing its job silently.

This article is about opening that file up.

In March 2023, a developer named Georgi Gerganov ported Meta’s freshly-leaked LLaMA model to run on a MacBook in pure C. The project — llama.cpp — needed a file format. The first attempt, GGML, was a prototype: hyperparameters stored as a flat list of unlabeled integers, different for every architecture. The tokenizer lived in a separate file. Adding a new model meant writing a new parser. It worked, barely, and broke constantly.

Five months later, Gerganov replaced it with GGUF — a ground-up redesign built around one idea: the file should describe itself. Architecture, tokenizer, quantization parameters, and weights, all in a single self-contained binary. No sidecar configs. No version mismatches. Download one file, run it. That philosophy turned llama.cpp into the engine behind Ollama, LM Studio, GPT4All, and most of the local AI ecosystem.

So what is actually in that file? Let’s find out.

The design decision

The file is the config. The file is the tokenizer. The file is the model. A GGUF file can be mmap’d directly — every tensor padded to 32-byte boundaries. The OS virtual memory manager becomes your model loader. No parsing, no buffer allocation, no copying.

02 // The Header

DOWNLOAD → HEADER → METADATA → TENSORS → WEIGHTS

24 bytes. Four fields. The handshake between the file and any runtime that opens it.

Every GGUF file starts the same way. Four ASCII bytes — G, G, U, F — then a version number, a tensor count, and a metadata count. That is the entire fixed-size header. Twenty-four bytes that tell the runtime: I am a GGUF file, I contain 370 tensors and 300 metadata entries. Start reading.

After these 24 bytes, everything is variable-length. There is no index, no seek table, no random access. The file is meant to be read front-to-back. To skip a metadata entry, you must decode its type and read its length — GGUF trades random access for zero-overhead sequential reads. The same tradeoff Unix pipes make.

The next thing the runtime reads is the metadata. This is where the file describes what kind of model it is — and what the runtime needs to do with it.

03 // The Metadata

DOWNLOAD → HEADER → METADATA → TENSORS → WEIGHTS

300 typed key-value pairs. The runtime reads them and knows: 36 layers, 32 attention heads, a 4:1 sharing ratio for key-value heads, SwiGLU activation, 152K vocabulary. It hasn’t touched a single weight yet.

The runtime needs to build a computation graph before it can run inference. How many layers? How many attention heads? How many of those heads share key-value pairs? What tokenizer? What RoPE frequency? All of this lives in the metadata section — typed, namespaced key-value pairs that the runtime reads sequentially. By the time it finishes the metadata, it knows exactly which kernels to load, how many buffers to allocate, and how to wire them together.

The first key it reads is general.architecture. This single string — "qwen3" — determines the namespace for every other architecture-specific key. It is load-bearing: get it wrong and the runtime cannot locate qwen3.block_count, qwen3.attention.head_count, or any of the structural parameters it needs.

Three namespaces organize the metadata. general.* identifies the model. {arch}.* (here qwen3.*) prescribes the architecture. tokenizer.* embeds the full vocabulary, merge rules, and special tokens. Together they form a complete instruction set for the runtime — no external files needed.

What the runtime builds

The metadata doesn’t just describe the model — it prescribes the computation. block_count=36 means allocate 36 layer blocks. head_count_kv=8 with head_count=32 means select the GQA attention kernel, not MHA. feed_forward_length=12288 means wire a SwiGLU FFN with 3 projections. Every kernel selection, every buffer size, every matrix dimension is determined here.

04 // The Tensors

DOWNLOAD → HEADER → METADATA → TENSORS → WEIGHTS

The runtime knows the architecture. Now it reads the actual matrices — every weight in the model, listed by name.

Let’s start simple. A transformer layer does two things. First, it looks at context — that’s attention, the mechanism that lets each token consider every other token in the sequence. Then it thinks about what it saw — that’s the feed-forward network, a wider space where the model processes and transforms the information. Both are just matrix multiplications. And every matrix is listed in the GGUF file by name, shape, and type. (You’ll notice tensors have types like Q4_K alongside their shapes — those are quantization formats. We’ll explain why different tensors get different precision in the next section.)

Both subsystems modify a shared representation called the residual stream — a vector that flows through all 36 layers, accumulating understanding. Each layer reads the stream, adds its contribution, and passes it on. Attention and FFN don’t replace the representation; they refine it. This additive pattern — called a residual connection — is why transformers can be deep without losing information.

But here is what makes this interesting: not every model does attention the same way. Not every model has the same feed-forward structure. And some models have replaced attention entirely. The tensor names in the file tell you exactly which variant you are looking at. Let’s see.

One Layer, Five Architectures

To see how expressive the GGUF naming convention is, here are five real architectures in a single diagram. Don’t worry if some are unfamiliar — the point is that the file format handles all of them with no special cases. Toggle between them and watch the circuit reshape: tensors appear and disappear, parallel lanes merge or split, entire subsystems get replaced. Every difference corresponds to a real tensor name in the GGUF file.

Five things to notice as you toggle:

Q, K, V — together or apart. GPT-2 fuses all three into a single attn_qkv tensor — one big matrix multiply. Qwen3 separates them, and the K and V matrices are smaller than Q. That shape difference is Grouped Query Attention: 8 key-value heads shared across 32 query heads. The KV cache is 4× smaller. You can read the GQA ratio from one tensor shape.

Gate or no gate. Look at the FFN section. GPT-2 has two projections: expand and compress. Qwen3 has three — the extra one is ffn_gate, a SwiGLU gate that decides which neurons activate. If you see ffn_gate in the manifest, you know the activation function. If it’s missing, it’s GELU.

Eight experts. Mixtral’s attention looks like Qwen3’s. But below the FFN norm, a ffn_gate_inp tensor routes each token to 2 of 8 expert FFNs. The file contains 8 complete sets of gate+up+down matrices — only 2 fire per token. 47 billion parameters total, 13 billion active.

No attention at all. Some models replace attention entirely with state-space models — a different mechanism that processes tokens sequentially in O(n) time, like an RNN but with better training properties. Switch to Mamba-2. The attention section is gone entirely. In its place: ssm_in, ssm_conv1d, ssm_dt, ssm_a — a recurrent state-space model that processes tokens in O(n) time instead of O(n²). No Q, K, or V tensors. No KV cache. Different tensor names, different computation, same GGUF format.

Hybrid layers. Qwen3.5 mixes both worlds. Some layers use full attention (attn_q/k/v), others use Gated Delta Networks (ssm_alpha, ssm_beta). The GGUF manifest tells the runtime which kernel to use for each layer — the tensor names are the dispatch table.

The Residual Stream

One thing the diagram doesn’t show as a tensor — because it isn’t one — is the residual stream. That green dashed line on the left side of the circuit. Attention and FFN outputs are added to the input, not piped through sequentially. Each layer refines the representation; it does not replace it. The residual stream accumulates understanding across all 36 layers — it’s the bus that carries information from the first token embedding to the final output projection.

Two global tensors bookend the entire network. At the bottom: token_embd.weight [151936, 4096] maps each vocabulary token into a 4,096-dimensional vector. At the top: output.weight [151936, 4096] projects the final representation back into vocabulary space to produce the next-token probabilities.

The Full Stack: 36 Layers

You now know what one block does. But the model has 36 of them. Is that arbitrary? Open the metadata: qwen3.block_count = 36. That number is the most consequential architectural decision in the file. It determines how many sequential refinement passes the model can make on a shared representation. GPT-2 had 12. Qwen3-8B has 36. The newest Qwen3.5-27B has 64.

These are not 36 copies of the same thing. They are 36 stages of progressive abstraction. Early layers learn syntax, token boundaries, and local relationships — the grammar of language. Middle layers build entity tracking and factual recall — “who is the subject of this sentence?” and “what do I know about them?” Late layers handle task completion, output formatting, and the final decision: which token comes next. Depth is how a model builds abstractions incrementally, each layer reading and refining the same residual stream.

When Layers Stop Being Identical

Toggle the visualization above to Qwen3.5-9B and something unexpected happens. The uniform tower breaks apart. Not all layers are the same circuit anymore.

Look at the difference. blk.0 has ssm_* tensors — a conv1d kernel, delta time projections, decay rates, alpha/beta gating. blk.3 has attn_q/k/v/output — the standard transformer attention you saw in Qwen3. The runtime reads these names and knows: layer 0 gets the Gated Delta Net kernel. Layer 3 gets the full attention kernel. The tensor names are the dispatch table.

The numbers tell the design story. Qwen3.5 uses fewer, bigger heads: 16 heads of 256 dimensions versus Qwen3’s 32 heads of 128. Same total attention dimension (4,096), completely different tradeoff — richer per-head representations at the cost of less head diversity. And partial_rotary_factor = 0.25 means only 25% of the head dimensions receive positional encoding. The model treats position as a partial signal, not a total one.

The vocabulary tells a generation story too. GPT-2: 50,257 tokens, English-centric. Qwen3: 151,936, multilingual. Qwen3.5: 248,320 — with vision tokens baked in, natively multimodal. The embedding matrix token_embd.weight grew from [50257, 768] to [248320, 4096]. That is not just a bigger dictionary; it is a fundamentally different conception of what a token is.

Three keys

block_count tells you depth. feed_forward_length tells you width. full_attention_interval tells you the attention budget. Three metadata keys, and you know the architect’s priorities.

Now you know the architecture — how many layers, which tensors, what shapes, which kernels. One question remains: how do you fit 8 billion 16-bit parameters into a 5.1 GB file?

05 // Not All Bits Are Equal

DOWNLOAD → HEADER → METADATA → TENSORS → WEIGHTS

5.1 gigabytes of quantized numbers. Not all compressed equally. The edges of the network get more precision because errors there propagate through every layer.

The runtime has read the header, parsed the metadata, and loaded the tensor manifest. It knows the architecture. Now it needs the actual numbers — 8 billion parameters, compressed from 16 GB of FP16 down to 5.1 GB of mixed-precision 4-bit blocks. But not uniformly compressed. Most tensors are Q4_K. But token_embd.weight is Q6_K. The attention output projections in certain layers are Q6_K. The layer norms are F32. Why are some tensors getting 50% more precision than others?

Because quantization is triage, not uniform compression. The edges of the network — the embedding matrix and the output head — are the most sensitive to error. A rounding mistake in the embedding propagates through all 36 layers. A rounding mistake in the output head directly corrupts the next-token probabilities. The middle layers are more resilient: each one is just a small refinement on the residual stream, and small errors in any single layer get diluted by the 35 others.

This is what the _S, _M, and _L suffixes mean. They are not bit widths — they are precision policies. Q4_K_S (small) promotes the fewest tensors to higher precision. Q4_K_M (medium) promotes roughly 20% of tensors — the embeddings, the output head, and the most sensitive FFN down-projections — to Q6_K. Q4_K_L promotes more. The per-bit-per-weight cost difference between _S and _M is about 0.3–0.5 bits, but the perplexity improvement is measurable: 0.05–0.15 points on standard benchmarks.

2B: super-scale   2B: super-minimum   12B: 8 sub-block scales+mins (quantized!)   128B: 256 weights × 4 bits

This is the trick. The 256 weights are divided into 8 sub-blocks of 32. Each sub-block gets its own scale and minimum — so weights in different ranges get different rulers. But storing 8 separate FP16 scale+minimum pairs would cost 32 bytes of overhead. K-quants compress those 8 scales into just 12 bytes by quantizing them against the super-block’s FP16 scale. The metadata that describes the compression is itself compressed.

Q4_K Super-Block: Two Levels of Quantization

Why does this matter? At 4-bit precision, per-sub-block FP16 scales would cost 1 bit per weight in overhead. By quantizing the scales, K-quants cut that to 0.19 bits per weight. That 0.8-bit saving is the gap between Q4_0’s mediocre quality and Q4_K’s near-FP16 perplexity. Double quantization is not a micro-optimization. It is what makes 4-bit viable.

AWQ takes a different path to the same insight. Instead of choosing precision by network position (edges versus middle), AWQ uses activation magnitude: run calibration data through the model, measure which weight groups produce the largest activations, and protect those groups with higher precision. Different signal, same principle — not all weights are equal.

The double quantization trick

256 weights, 8 sub-blocks, each sub-block with its own scale. But the scales are quantized against a single FP16 super-scale. Compression of the compression. At 4-bit, this saves 0.8 bits per weight of overhead — the difference between “noticeably degraded” and “barely distinguishable from FP16.”

The quantization type of each tensor — visible in the manifest alongside the name and shape — is the last piece. Architecture from metadata. Structure from tensor names. Precision from quant types. Now you have everything you need to read any model.

06 // Reading Any Model

DOWNLOAD → HEADER → METADATA → TENSORS → WEIGHTS ← the complete picture

You can now open any GGUF file and understand its architecture, design decisions, and quality tradeoffs — without running inference.

Theory is over. Every concept from the previous five sections shows up when you read a real model. Open Qwen3-8B and GPT-2 side by side. The six-year gap between 2019 and 2025 is legible in the file format: MHA → GQA, GELU → SwiGLU, learned positions → RoPE, LayerNorm → RMSNorm, bias everywhere → bias nowhere.

Comparison: Model Fingerprints — 2019 vs 2025

Qwen3-8B (2025)

GQA 4:1 — visible in the K and V shapes: [1024, 4096] instead of [4096, 4096]. Eight KV heads serving 32 query heads. KV cache is 4× smaller than full MHA. SwiGLU — three FFN tensors per layer. The ffn_gate is present. RoPE at a base frequency of 1,000,000 — no learned positional embeddings, no position_embd.weight tensor. No bias anywhere — not on attention projections, not on FFN projections, not even on layer norms. A clean, modern architecture. 151,936-token vocabulary — multilingual, covering Chinese, English, and dozens of other languages in a single embedding matrix.

GPT-2 124M (2019)

Multi-Head Attention — Q, K, and V all the same width, fused into a single attn_qkv.weight [2304, 768]. No GQA. Every head has its own keys and values. GELU — two FFN tensors. No gate. Learned positional embeddings — position_embd.weight [1024, 768] is an explicit tensor in the file, hard-capped at 1,024 positions. No RoPE. Bias on everything — every projection, every norm, every layer. 50,257-token vocabulary — English-centric BPE.

The Fossil Record

The evolution from GPT-2 to Qwen3 is the evolution of the transformer itself, and every step is legible in the file format:

Reading It Yourself

Fifteen lines of Python and you can read any GGUF file:

Pythonfrom gguf import GGUFReader

reader = GGUFReader("model.gguf")

# Architecture and key parameters
arch = reader.fields['general.architecture'].parts[-1].tobytes().decode()
print(f"Architecture: {arch}")
print(f"Layers: {reader.fields[f'{arch}.block_count'].parts[-1][0]}")

# Every tensor: name, shape, quant type
for t in reader.tensors:
    print(f"{t.name}: {t.shape} {t.tensor_type.name}")

After reading the previous four sections, you understand every line of that script. general.architecture tells you the namespace prefix. block_count tells you how many layers. The tensor loop shows you the names, shapes, and quant types you now know how to interpret — the GQA ratio in the K/V shapes, the activation function in the FFN tensor count, the precision policy in the quant types.

The universal language

Every architecture — GPT-2, LLaMA, Qwen, Mistral, Gemma, Phi, Falcon — gets normalized into blk.N.tensor_name. Meta calls their query projection layers.N.attention.wq. HuggingFace calls it model.layers.N.self_attn.q_proj. GGUF calls it blk.N.attn_q. One namespace, every model. Once you learn to read it, you can open any GGUF file and understand the architecture in seconds. The file is the blueprint. Now you know how to read it.
