SYSCALLS & io_uring

The thin membrane between your code and the kernel. Every file you open, every byte you send, every page you map — it all crosses this boundary._

// 01 — the boundary

Your process lives in a cage. It can see its own memory, run its own instructions, touch its own stack. But the moment it wants to talk to the outside world — read a file, send a packet, allocate memory from the OS — it has to ask permission. The CPU enforces this with hardware privilege levels. On x86-64, your code runs in Ring 3 (user mode). The kernel runs in Ring 0 (supervisor mode). There is no way to cheat. The silicon itself checks every instruction.

x86-64 Privilege Rings 

                    ┌─────────────────────────┐
                    │       Ring 0             │
                    │       KERNEL             │
                    │   Full hardware access   │
                    │   All memory visible     │
                    │   All instructions       │
                    │                           │
                    │    ┌─────────────────┐    │
                    │    │    Ring 3        │    │
                    │    │    USERSPACE     │    │
                    │    │  Limited memory  │    │
                    │    │  No I/O ports    │    │
                    │    │  No privileged   │    │
                    │    │  instructions    │    │
                    │    └─────────────────┘    │
                    └─────────────────────────┘

  (Rings 1 & 2 exist but Linux doesn't use them.
   It's Ring 0 or Ring 3. Kernel or peasant.)

When your program wants to do anything involving hardware or shared resources, it executes the syscall instruction. This single instruction is the gateway. Here is exactly what happens in those ~100 nanoseconds:

Anatomy of the syscall instruction 

  USERSPACE (Ring 3)                         KERNEL (Ring 0)
  ─────────────────                          ──────────────
  1. Load syscall number into RAX
  2. Load arguments into RDI, RSI, RDX, R10, R8, R9
  3. Execute SYSCALL instruction
     │
     │  ┌──── CPU hardware transition ────┐
     │  │  Save RIP → RCX                 │
     │  │  Save RFLAGS → R11              │
     │  │  Load RIP from MSR_LSTAR        │
     │  │  Switch to Ring 0               │
     │  │  Load kernel stack pointer      │
     │  └─────────────────────────────────┘
     │
     └──────────────────────────────────→ 4. Kernel entry point
                                          5. Look up handler in sys_call_table[RAX]
                                          6. Validate arguments (pointers, permissions)
                                          7. Execute the actual work
                                          8. Store return value in RAX
                                          9. Execute SYSRET instruction
     ┌──────────────────────────────────←    (restore RIP from RCX, Ring 3)
     │
  10. Check RAX for return value
      (negative = -errno, else success)

The register convention on x86-64 Linux is burned into the ABI. Memorize it:

Syscall Register Convention

Register state during a syscall 
  ┌──────────┬──────────────────────────────────────────┐
  │ Register │ Purpose                                  │
  ├──────────┼──────────────────────────────────────────┤
  │ RAX      │ Syscall number (in) / Return value (out) │
  │ RDI      │ Argument 1                               │
  │ RSI      │ Argument 2                               │
  │ RDX      │ Argument 3                               │
  │ R10      │ Argument 4  (NOT RCX! it gets clobbered) │
  │ R8       │ Argument 5                               │
  │ R9       │ Argument 6                               │
  ├──────────┼──────────────────────────────────────────┤
  │ RCX      │ Clobbered (holds saved RIP after SYSCALL)│
  │ R11      │ Clobbered (holds saved RFLAGS)           │
  └──────────┴──────────────────────────────────────────┘

Notice that R10 replaces RCX for argument 4. In the normal C calling convention, RCX is the fourth argument. But syscall destroys RCX (it saves the return address there). So the kernel ABI uses R10 instead. This is one of those details that will haunt you if you forget it.

Hello World Without libc

Let's write to stdout using nothing but raw syscall. No printf, no write() wrapper, no libc at all. Pure assembly inlined into C.

// hello_raw.c — compile with: gcc -nostdlib -o hello hello_raw.c

void _start() {
    const char msg[] = "hello from ring 3\n";

    // syscall: write(fd=1, buf=msg, count=18)
    // RAX=1 (write), RDI=1 (stdout), RSI=msg, RDX=18
    asm volatile (
        "mov $1, %%rax\n"     // syscall number: write = 1
        "mov $1, %%rdi\n"     // fd: stdout = 1
        "mov %0, %%rsi\n"     // buf: pointer to msg
        "mov $18, %%rdx\n"    // count: 18 bytes
        "syscall\n"
        : : "r"(msg) : "rax", "rdi", "rsi", "rdx", "rcx", "r11"
    );

    // syscall: exit(status=0)
    // RAX=60 (exit_group), RDI=0 (success)
    asm volatile (
        "mov $60, %%rax\n"    // syscall number: exit = 60
        "xor %%rdi, %%rdi\n" // status: 0
        "syscall\n"
        : : : "rax", "rdi"
    );
}

The Insight

This binary is ~1KB. No libc linked, no dynamic loader, no .so files. Just your code and the kernel. Every C program you've ever written eventually does exactly this — the libc write() function is just a wrapper that loads registers and executes syscall. You've been making syscalls your entire career. You just didn't know it.

// 02 — strace: x-ray vision

Before you optimize syscalls, you need to see them. strace intercepts every syscall your program makes and prints it. It is, without exaggeration, the single most powerful debugging tool most programmers never use. Segfault? strace. Program hangs? strace. Slow startup? strace. Permission denied but you don't know which file? strace.

The Basics

# Trace every syscall of a command
$ strace ./my_shard_reader data.shard

# Filter by syscall category
$ strace -e trace=file ./reader    # Only file ops: open, read, write, stat, ...
$ strace -e trace=network ./server # Only network: socket, bind, connect, ...
$ strace -e trace=memory ./alloc   # Only memory: mmap, brk, mprotect, ...
$ strace -e trace=desc ./reader    # Only descriptor ops: read, write, close, ...

# Show timing (microseconds per syscall)
$ strace -T ./reader               # Appends <0.000042> to each line

# Show wall-clock timestamps
$ strace -t ./server               # 14:23:01 write(1, "listening...", 12)

# Statistics mode — the killer feature
$ strace -c ./reader               # Syscall histogram when process exits

# Follow child processes (fork/clone)
$ strace -f ./multi_threaded_app

# Attach to running process
$ strace -p 12345                  # Attach to PID 12345 (need root or ptrace)

// 03 — file I/O: the layers

When you call write(fd, buf, 4096), you probably think the data goes to disk. It doesn't. It goes to the page cache — a region of RAM managed by the kernel. The actual disk write happens later, maybe seconds later, maybe never if you crash first. Understanding these layers is the difference between code that "works" and code that is correct and fast.

The Full I/O Stack — Journey of a write() Call

The Page Cache

Linux keeps recently read and written file data in RAM. This is the page cache, and it typically consumes most of your free RAM (that's why free -h shows low "available" memory — it's being used as cache, and will be released if needed).

// When you read() a file:
// 1. Kernel checks: "Is this page already in the page cache?"
// 2. If YES: memcpy from page cache → your buffer. ~0.5μs. Done.
// 3. If NO:  Read from disk → page cache → your buffer. ~50μs NVMe.
//
// When you write() a file:
// 1. Data goes to page cache. Page marked "dirty".
// 2. write() returns IMMEDIATELY. Your data is NOT on disk.
// 3. Background kernel thread (pdflush/writeback) flushes dirty
//    pages to disk every ~5 seconds or when memory is low.
//
// THIS IS WHY POWER LOSS CAN LOSE DATA.
// write() succeeding does NOT mean data is on disk.
page cache behavior
Danger: The fsync() Tax

If you need data to survive a power failure, you must call fsync(fd) after writing. This forces dirty pages to disk and waits for the hardware to confirm. On NVMe: ~50μs. On spinning rust: ~5-10ms. Databases call fsync on every transaction commit — it is the single most expensive operation in a database. fdatasync() is slightly cheaper: it flushes data but not metadata (timestamps, size). Use it when you can.

O_DIRECT — Bypass the Page Cache

Sometimes the page cache hurts. If you're scanning a 100GB file once, the page cache fills with data you'll never read again, evicting actually useful cached data. O_DIRECT bypasses the page cache entirely — DMA from disk straight to your buffer.

#define _GNU_SOURCE
#include <fcntl.h>
#include <stdlib.h>
#include <unistd.h>

int main() {
    // O_DIRECT: bypass page cache, DMA to our buffer
    int fd = open("huge_file.dat", O_RDONLY | O_DIRECT);

    // O_DIRECT REQUIRES aligned buffers!
    // Buffer address must be aligned to 512 bytes (or 4096 for some FSes)
    // Read size must be a multiple of 512
    void *buf;
    posix_memalign(&buf, 4096, 4096);  // 4K-aligned, 4K buffer
    // ^ This is why shard uses Align64() — alignment matters for DMA!

    read(fd, buf, 4096);  // Goes directly from NVMe → your buffer
                          // No page cache involved. No extra copy.

    free(buf);
    close(fd);
}
C — O_DIRECT bypass

Use Page Cache (default)

Sequential reads of files you'll access again. Small files. Normal workloads. The kernel is surprisingly good at predicting what you'll read next (readahead).

Use O_DIRECT

Databases (they manage their own cache). One-pass scans of huge files. When you don't want to pollute the cache. When you need predictable latency (no cache miss surprises).

Hack: posix_fadvise — Talk to the Cache

Don't want O_DIRECT but want to hint at the kernel? posix_fadvise(fd, 0, 0, POSIX_FADV_SEQUENTIAL) tells the kernel to aggressively read ahead. POSIX_FADV_DONTNEED tells it to drop cached pages after you read them — perfect for one-pass scans. POSIX_FADV_WILLNEED triggers async readahead for data you'll need soon. Free performance, zero code changes.

// 04 — mmap deep dive

Most people think mmap() is "map a file to memory." That's like saying a car is "a thing with wheels." mmap is a virtual memory manipulation primitive. It creates a mapping in your process's page table that says "this range of virtual addresses corresponds to this thing." The "thing" can be a file, anonymous memory, a device, or shared memory with another process. And here's the real kicker: no physical memory is allocated when you call mmap.

The Page Fault Dance

What ACTUALLY happens when you mmap and access a file 

  Step 1: mmap(NULL, 4096, PROT_READ, MAP_PRIVATE, fd, 0)
  ─────────────────────────────────────────────────────
  Kernel adds entry to your page table:
    Virtual addr 0x7f0000 → [not present, backed by fd offset 0]
  Physical memory allocated: ZERO bytes.
  mmap() returns instantly.

  Step 2: Your code reads from ptr[0]    (first access!)
  ─────────────────────────────────────────────────────
  CPU checks page table: "0x7f0000 → NOT PRESENT"
  CPU raises PAGE FAULT exception (#PF, interrupt 14)
       │
       ▼
  Kernel page fault handler runs:
    1. "Oh, this VA is mmap'd to fd at offset 0"
    2. Allocate a physical page (4KB of RAM)
    3. Read 4KB from disk into that physical page
    4. Update page table: 0x7f0000 → physical page (PRESENT, READ-ONLY)
    5. Return from exception
       │
       ▼
  CPU retries the instruction that faulted
  This time: page table says PRESENT → access succeeds
  Your code continues, completely unaware a fault happened.

  Step 3: Your code reads ptr[1], ptr[2], ... ptr[4095]
  ─────────────────────────────────────────────────────
  All within the same 4KB page → no more faults.
  Just normal memory reads at RAM speed.

  Step 4: Your code reads ptr[4096]    (next page!)
  ─────────────────────────────────────────────────────
  Different page → PAGE FAULT again → repeat step 2.
Page fault latency: ~2μs (page in cache) • ~50μs (page on NVMe) • ~5ms (page on HDD)

How Shard's EnableMmap() Works at the Kernel Level

// What shard's EnableMmap() does conceptually:

int fd = open("model.shard", O_RDONLY);

struct stat st;
fstat(fd, &st);  // Get file size

// Map the ENTIRE shard file into virtual address space
void *base = mmap(
    NULL,            // Let kernel choose address
    st.st_size,      // Map entire file (could be 4GB+)
    PROT_READ,       // Read-only — tensors are immutable
    MAP_PRIVATE,     // Private mapping, COW semantics
    fd, 0            // Start at file offset 0
);

// Optional: tell kernel we'll access sequentially
madvise(base, st.st_size, MADV_SEQUENTIAL);

// Now tensor data is accessible as raw pointers:
// float *weights = (float *)(base + tensor_offset);
// Just dereference. Kernel handles page faults transparently.
// No read() calls. No buffers. No copies.
// The pointer IS the data.

// When done:
munmap(base, st.st_size);
close(fd);

The Key Insight

With mmap, a 4GB shard file uses ~0 bytes of RAM initially. The kernel loads pages on demand as you access them. If the system is under memory pressure, it can simply drop clean mmap pages (they can be re-read from disk). This is why mmap scales better than read() for large files — the kernel manages the memory automatically, and you never hold more than you need.

Hack: MAP_POPULATE + mlock = Deterministic Latency

MAP_POPULATE prefaults all pages at mmap time (blocks until all data is read from disk). mlock() pins those pages in RAM so the kernel can never evict them. Combined: zero page faults at runtime, zero disk I/O, fully deterministic latency. This is how real-time trading systems load their data. For shard: if you have the RAM, MAP_POPULATE the entire model file at startup and never pay a page fault during inference.

Zen: madvise Hints

MADV_SEQUENTIAL — aggressive readahead, drop pages after reading. MADV_RANDOM — disable readahead (random access pattern). MADV_WILLNEED — async prefetch specific ranges. Free performance knobs with zero code restructuring.

// 05 — epoll: the reactor pattern

How does nginx handle 10,000 concurrent connections with a single thread? It doesn't use 10,000 threads. It uses one thread, one epoll instance, and the reactor pattern. The idea: register interest in events (data available on socket, connection incoming, timer expired), then sleep. The kernel wakes you up only when something actually happens. Zero CPU wasted on polling.

The Evolution of I/O Multiplexing

APIYearFD LimitComplexityHow It Works
select()19831024O(n) per callBitmap of FDs. Kernel scans ALL bits every time. Copies bitmap in/out.
poll()1986unlimitedO(n) per callArray of pollfd structs. No 1024 limit, but still scans entire array.
epoll2002unlimitedO(1) per eventKernel maintains interest list. Only returns FDs with events. Scales to millions.
select: O(n) scan of 1024 bits • poll: O(n) scan of n FDs • epoll: O(1) per ready FD

The Three epoll Syscalls

// 1. Create an epoll instance (returns a file descriptor!)
int epfd = epoll_create1(0);

// 2. Register interest in a file descriptor
struct epoll_event ev;
ev.events = EPOLLIN;        // Interested in: data available to read
ev.data.fd = client_fd;     // Which FD this event is about
epoll_ctl(epfd, EPOLL_CTL_ADD, client_fd, &ev);
//              ^^^^^^^^^^^^
//              ADD / MOD / DEL — add, modify, or remove interest

// 3. Wait for events (this is where you sleep)
struct epoll_event events[64];
int n = epoll_wait(epfd, events, 64, -1);
//                                     ^^ timeout: -1 = block forever
//                                                  0 = return immediately
//                                               1000 = wait up to 1 second

// n = number of FDs that have events ready
for (int i = 0; i < n; i++) {
    int ready_fd = events[i].data.fd;
    // This FD has data ready — read it NOW
}
C — epoll basics

TCP Echo Server with epoll

#include <sys/epoll.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <unistd.h>
#include <fcntl.h>
#include <stdio.h>

#define MAX_EVENTS 1024
#define PORT 8080

void set_nonblocking(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

int main() {
    // Create listening socket
    int srv = socket(AF_INET, SOCK_STREAM, 0);
    int opt = 1;
    setsockopt(srv, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    struct sockaddr_in addr = {
        .sin_family = AF_INET,
        .sin_port = htons(PORT),
        .sin_addr.s_addr = INADDR_ANY
    };
    bind(srv, (struct sockaddr*)&addr, sizeof(addr));
    listen(srv, SOMAXCONN);
    set_nonblocking(srv);

    // Create epoll instance
    int epfd = epoll_create1(0);
    struct epoll_event ev = { .events = EPOLLIN, .data.fd = srv };
    epoll_ctl(epfd, EPOLL_CTL_ADD, srv, &ev);

    struct epoll_event events[MAX_EVENTS];
    char buf[4096];

    // THE EVENT LOOP — one thread handles thousands of connections
    while (1) {
        int n = epoll_wait(epfd, events, MAX_EVENTS, -1);

        for (int i = 0; i < n; i++) {
            if (events[i].data.fd == srv) {
                // New connection incoming
                int client = accept(srv, NULL, NULL);
                set_nonblocking(client);
                ev.events = EPOLLIN | EPOLLET;  // Edge-triggered!
                ev.data.fd = client;
                epoll_ctl(epfd, EPOLL_CTL_ADD, client, &ev);
            } else {
                // Data available on existing connection
                int fd = events[i].data.fd;
                ssize_t bytes = read(fd, buf, sizeof(buf));
                if (bytes <= 0) {
                    close(fd);  // Connection closed or error
                } else {
                    write(fd, buf, bytes);  // Echo it back
                }
            }
        }
    }
}
C — epoll echo server

Level-Triggered vs Edge-Triggered

Level-Triggered (default)

epoll_wait returns as long as the condition is true. If there's data in the socket buffer, it keeps reporting it. Simpler to use — you can read partial data and epoll will remind you there's more.

// "Hey, FD 5 has data"
// You read 100 bytes
// (200 bytes remain)
// Next epoll_wait:
// "Hey, FD 5 STILL has data"

Edge-Triggered (EPOLLET)

epoll_wait returns only on state change. Data arrives? One notification. You MUST read ALL available data (until EAGAIN) or you'll miss it forever.

// "Hey, FD 5 has data"
// You read 100 bytes
// (200 bytes remain)
// Next epoll_wait:
// *silence* (no state change)
// DATA LOST if you don't
// drain the buffer!
The Connection to Go

Go's runtime uses epoll internally. Every goroutine that blocks on network I/O is parked by the Go scheduler, and the FD is registered with a single epoll instance (the "netpoller"). When data arrives, epoll wakes the runtime, which schedules the goroutine back onto an OS thread. This is why goroutines are cheap — blocked goroutines cost zero CPU. The netpoller is doing the same thing as our echo server above, but managing millions of goroutines instead of raw FDs.

Wild: The C10K Problem is Solved

In 1999, Dan Kegel asked: "Can a single server handle 10,000 simultaneous connections?" With select(), the answer was no (1024 FD limit, O(n) scan). With epoll, nginx routinely handles 100K+ connections per worker thread. The bottleneck moved from "syscall overhead" to "memory per connection" (~10KB/conn) and then to "bandwidth." epoll didn't just solve C10K — it made it boring.

// 06 — io_uring: the revolution

Everything we've discussed so far has a fundamental problem: every I/O operation requires a syscall, and every syscall costs ~200ns of overhead (mode switch, register save/restore, TLB flush, spectre mitigations). If you're doing 100,000 small reads per second, that's 20ms of pure overhead. io_uring, introduced in Linux 5.1 (2019) by Jens Axboe, eliminates this entirely. After initial setup, you can submit I/O operations without any syscalls at all.

The Architecture: Two Ring Buffers

io_uring: shared-memory ring buffers between userspace and kernel 

  USERSPACE                              KERNEL
  =========                              ======

  +----------------------------+
  |  Submission Queue (SQ)     |----------> Kernel reads SQEs
  |                            |            from this ring
  |  +----+----+----+----+     |
  |  |SQE |SQE |SQE |    |    |     SQE = Submission Queue Entry
  |  | 0  | 1  | 2  |    |    |     (describes one I/O operation)
  |  +----+----+----+----+     |
  |   tail-->          <--head |     You write at tail.
  |                            |     Kernel reads from head.
  +----------------------------+

  +----------------------------+
  |  Completion Queue (CQ)     |<---------- Kernel writes CQEs
  |                            |            into this ring
  |  +----+----+----+----+     |
  |  |CQE |CQE |    |    |    |     CQE = Completion Queue Entry
  |  | 0  | 1  |    |    |    |     (result of completed I/O)
  |  +----+----+----+----+     |
  |   head-->          <--tail |     Kernel writes at tail.
  |                            |     You read from head.

  Both rings are in SHARED MEMORY (mmap'd).
  No copies. No syscalls for submission (with SQPOLL).
  Just write to memory, kernel picks it up.

File Read with io_uring

#include <liburing.h>
#include <fcntl.h>
#include <stdio.h>
#include <string.h>

#define QUEUE_DEPTH 64
#define BLOCK_SIZE  4096

int main() {
    struct io_uring ring;

    // Step 1: Initialize the ring (this IS a syscall - io_uring_setup)
    io_uring_queue_init(QUEUE_DEPTH, &ring, 0);

    // Step 2: Open the file
    int fd = open("data.shard", O_RDONLY | O_DIRECT);
    char *buf;
    posix_memalign((void**)&buf, 4096, BLOCK_SIZE);

    // Step 3: Prepare a submission queue entry
    struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
    io_uring_prep_read(sqe, fd, buf, BLOCK_SIZE, 0);
    sqe->user_data = 42;  // Tag this request (returned in CQE)

    // Step 4: Submit (one syscall - or ZERO with SQPOLL)
    io_uring_submit(&ring);

    // Step 5: Wait for completion
    struct io_uring_cqe *cqe;
    io_uring_wait_cqe(&ring, &cqe);

    if (cqe->res < 0) {
        fprintf(stderr, "read failed: %s\n", strerror(-cqe->res));
    } else {
        printf("Read %d bytes, tag=%llu\n", cqe->res, cqe->user_data);
    }

    io_uring_cqe_seen(&ring, cqe);
    io_uring_queue_exit(&ring);
    close(fd);
    free(buf);
}

Batch Read: 100 Tensor Entries in One Submission

This is where io_uring truly shines. Instead of 100 read() syscalls, submit 100 SQEs at once:

#define NUM_TENSORS 100
#define TENSOR_SIZE 4096

// Assume we have tensor offsets from the shard header
off_t offsets[NUM_TENSORS];   // Filled from shard index
char *buffers[NUM_TENSORS];   // Pre-allocated, aligned

for (int i = 0; i < NUM_TENSORS; i++) {
    posix_memalign((void**)&buffers[i], 4096, TENSOR_SIZE);
}

// Submit ALL 100 reads at once
for (int i = 0; i < NUM_TENSORS; i++) {
    struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
    io_uring_prep_read(sqe, fd, buffers[i], TENSOR_SIZE, offsets[i]);
    sqe->user_data = i;  // Tag = tensor index
}

// ONE syscall to submit all 100 (or ZERO with SQPOLL)
io_uring_submit(&ring);

// Reap completions - they may arrive out of order!
int completed = 0;
while (completed < NUM_TENSORS) {
    struct io_uring_cqe *cqe;
    io_uring_wait_cqe(&ring, &cqe);

    int tensor_idx = cqe->user_data;
    if (cqe->res > 0) {
        process_tensor(tensor_idx, buffers[tensor_idx], cqe->res);
    }

    io_uring_cqe_seen(&ring, cqe);
    completed++;
}

Hack: SQPOLL Mode — True Zero-Syscall I/O

Pass IORING_SETUP_SQPOLL to io_uring_setup(). The kernel spawns a dedicated polling thread that continuously checks the SQ ring for new entries. You write SQEs to shared memory, the kernel thread picks them up, results appear in the CQ ring. No syscalls at all after setup. The kernel thread burns a CPU core, but for high-throughput I/O servers, this is the fastest possible path. SPDK-level performance with standard kernel I/O.

Zen: Why io_uring Matters for Shard

A shard file with 500 tensor entries read via individual read() calls: 500 * 200ns = 100μs of pure syscall overhead, before any data moves. Same 500 entries with io_uring batch submission: one io_uring_submit() call, all 500 reads dispatched to the NVMe queue simultaneously, completions harvested in bulk. The NVMe drive can process all 500 in parallel across its internal channels. You've gone from sequential syscalls to fully parallel hardware I/O.

// 07 — sendfile & splice: zero-copy networking

Here's a dirty secret about serving files over the network: the naive approach copies your data four times. Disk to kernel buffer. Kernel buffer to userspace buffer. Userspace buffer back to kernel socket buffer. Socket buffer to NIC. Four copies. Two of them are completely pointless. sendfile() eliminates the userspace copies entirely.

Data Path: Naive vs sendfile vs splice+DMA 

  NAIVE: read() + write()               4 data movements
  ========================
  Disk -> [Kernel Read Buf] -> [User Buffer] -> [Kernel Socket Buf] -> NIC
          copy 1                copy 2           copy 3                DMA 4

  SENDFILE: sendfile(socket, file)       2 copies, 1 syscall
  ====================================
  Disk -> [Kernel Buffer] --in-kernel--> [Socket Buffer] -> NIC
          copy 1           no user         copy 2          DMA 3
                           involvement

  SPLICE + DMA SCATTER-GATHER:           Minimum copies
  ============================
  Disk ---DMA---> [Kernel Buf] ---descriptors---> NIC DMA
          copy 1   (only ptrs     scatter-gather   copy 2
                    to socket)

  For a 1GB file transfer:
    read+write: ~4GB of memory bus traffic
    sendfile:   ~2GB of memory bus traffic
    splice+SG:  ~1GB of memory bus traffic

sendfile() in Practice

#include <sys/sendfile.h>
#include <sys/stat.h>
#include <fcntl.h>

// Serve a shard file over TCP - zero-copy
void serve_shard(int socket_fd, const char *path) {
    int file_fd = open(path, O_RDONLY);

    struct stat st;
    fstat(file_fd, &st);

    // ONE syscall. Data: disk -> kernel -> NIC.
    // Your process never touches the data.
    off_t offset = 0;
    ssize_t sent = sendfile(socket_fd, file_fd, &offset, st.st_size);

    printf("Sent %zd bytes zero-copy\n", sent);
    close(file_fd);
}

// For large files, loop (may not send everything at once):
void serve_shard_full(int sock, const char *path) {
    int fd = open(path, O_RDONLY);
    struct stat st;
    fstat(fd, &st);

    off_t offset = 0;
    size_t remain = st.st_size;
    while (remain > 0) {
        ssize_t n = sendfile(sock, fd, &offset, remain);
        if (n <= 0) break;
        remain -= n;
    }
    close(fd);
}

// 08 — seccomp: syscall firewall

What if you could tell the kernel: "this process is only allowed to use read, write, mmap, and exit. If it tries anything else, kill it immediately." That is seccomp. It is a BPF program that runs before every syscall, inspecting the syscall number and arguments, and deciding whether to allow it, block it, or terminate the process. Docker, Chrome, and Firefox all use seccomp to sandbox untrusted code.

The Two Modes

ModeWhat It AllowsUse Case
SECCOMP_MODE_STRICTOnly read, write, exit, sigreturnExtreme lockdown. Pure computation processes.
SECCOMP_MODE_FILTERCustom BPF filter per syscallDocker containers, browser sandboxes, any real-world use.

Writing a seccomp Filter

#include <linux/seccomp.h>
#include <linux/filter.h>
#include <linux/audit.h>
#include <sys/prctl.h>
#include <stddef.h>
#include <unistd.h>

// BPF filter: allow only read, write, mmap, exit_group
// Kill the process on any other syscall
struct sock_filter filter[] = {
    // Load the syscall number
    BPF_STMT(BPF_LD | BPF_W | BPF_ABS,
             offsetof(struct seccomp_data, nr)),

    // Allow read (0)
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_read, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),

    // Allow write (1)
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_write, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),

    // Allow mmap (9)
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_mmap, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),

    // Allow exit_group (231)
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_exit_group, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),

    // Default: KILL the process
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_KILL),
};

struct sock_fprog prog = {
    .len = (unsigned short)(sizeof(filter) / sizeof(filter[0])),
    .filter = filter,
};

int main() {
    // Required: no new privileges after exec
    prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0);

    // Install the filter
    prctl(PR_SET_SECCOMP, SECCOMP_MODE_FILTER, &prog);

    // This works:
    write(1, "I am sandboxed\n", 15);

    // These would KILL the process:
    // open("/etc/passwd", O_RDONLY);  // SIGKILL!
    // socket(AF_INET, ...);          // SIGKILL!
    // fork();                        // SIGKILL!

    return 0;
}

// 09 — eBPF: code in the kernel

What if you could write a small program, prove it is safe, and then run it inside the kernel at native speed? BPF CO-RE: Compile Once, Run Everywhere

Classic eBPF problem: kernel struct layouts change between versions. A program for kernel 5.15 might crash on 6.1 because a field moved. BPF CO-RE embeds type information at compile time and uses the kernel's BTF (BPF Type Format) to adjust field offsets at load time. Write once, run on any kernel version. This is what made eBPF practical for production tooling.

Hack: eBPF is Replacing Everything

Kernel modules? eBPF is safer. iptables? eBPF (Cilium) is faster. Kprobes scripts? bpftrace is more powerful. strace? eBPF has lower overhead. Performance monitoring? BCC tools give histograms, stack traces, and flamegraphs without sampling bias. If it involves observing or controlling kernel behavior, eBPF is eating it.

Zen: The Verification Guarantee

The eBPF verifier is the key innovation. It proves, mathematically, that your program terminates and does not access invalid memory. It walks every possible code path, tracks register states, validates memory bounds. This guarantee is what makes it safe to run arbitrary user code inside the kernel. No other OS has anything like this. It is the reason Linux is pulling ahead of every other kernel in programmability.

// 10 — vDSO: syscalls without syscalls

Some syscalls are called billions of times per second across all processes on a system. gettimeofday() and clock_gettime() are the biggest offenders. Every log line, every timeout check, every performance measurement calls them. If each one cost 200ns of syscall overhead, the entire system would grind to a halt. The kernel's solution is elegant: don't make them syscalls at all.

The vDSO Trick

vDSO: kernel data, userspace speed 

  Normal syscall:
  ==============
  User code ---> SYSCALL instruction ---> kernel mode ---> read clock
       ^              200ns overhead                         |
       |                                                     |
       +------ SYSRET -------- kernel mode --------- return -+
                   Total: ~200ns

  vDSO call:
  ==========
  User code ---> call vDSO function ---> reads shared page ---> return
                 (normal function           (mapped read-only
                  call, ~2ns)                into your process)
                   Total: ~20ns

  How it works:
  +---------------------------------------------------+
  |  KERNEL                                            |
  |  Updates a shared memory page every tick:          |
  |  - current time (nanoseconds)                     |
  |  - clock source info (TSC frequency, offset)      |
  |  - CPU ID for getcpu()                            |
  +---------------------------+-----------------------+
                              |
                    mmap (read-only, every process)
                              |
  +---------------------------v-----------------------+
  |  EVERY PROCESS                                     |
  |  vDSO: a tiny shared library mapped automatically  |
  |  Contains functions that read the shared page:     |
  |  - __vdso_clock_gettime()                          |
  |  - __vdso_gettimeofday()                           |
  |  - __vdso_getcpu()                                 |
  |  - __vdso_time()                                   |
  |                                                    |
  |  No syscall. No mode switch. No privilege change.  |
  +----------------------------------------------------+
clock_gettime via syscall: ~200ns • via vDSO: ~20ns • 10x speedup, billions of calls/day

How Go Uses vDSO

// Go's time.Now() calls runtime.nanotime1()
// which calls clock_gettime(CLOCK_MONOTONIC)
// which the Go runtime resolves through the vDSO.
//
// Go specifically looks for the vDSO at startup:
// runtime/vdso_linux.go parses the vDSO ELF headers
// to find __vdso_clock_gettime and calls it directly.
//
// This matters because Go programs call time.Now()
// constantly: scheduler timeslice checks, GC pacing,
// timer heaps, deadline contexts. If each were a
// real syscall, Go's goroutine scheduler would be
// 10x slower.
//
// The vDSO is also how getcpu() works on multi-socket
// systems. The kernel updates per-CPU data in the
// shared page, and userspace reads which CPU it's on
// without any syscall. NUMA-aware allocators use this.
Go runtime — vDSO usage
The Design Principle

The vDSO demonstrates a profound systems design insight: if you can't eliminate the cost of crossing a boundary, move the data to the side that needs it. The kernel maintains the authoritative time source. But instead of forcing every process to cross the syscall boundary to read it, the kernel maps a read-only view of that data into every process. The boundary still exists for writes, but reads are free. This same principle applies everywhere: shared memory IPC, memory-mapped configuration, read-only page tables.

Hack: Measuring vDSO vs Syscall Speed

Force a real syscall with syscall(__NR_clock_gettime, CLOCK_MONOTONIC, &ts) instead of the libc wrapper. Compare to clock_gettime(CLOCK_MONOTONIC, &ts) (which uses vDSO). The difference is pure syscall overhead. On modern kernels with Spectre mitigations, the gap is even larger: KPTI page table switching adds ~100ns to every real syscall, but vDSO calls are unaffected because they never enter the kernel.

Benchmark 1: Raw Syscall Latency

#include <time.h>
#include <unistd.h>
#include <sys/syscall.h>
#include <stdio.h>

#define ITERATIONS 1000000

int main() {
    struct timespec start, end;
    clock_gettime(CLOCK_MONOTONIC, &start);

    for (int i = 0; i < ITERATIONS; i++) {
        // getpid() does nothing useful — measures pure syscall overhead
        syscall(__NR_getpid);
    }

    clock_gettime(CLOCK_MONOTONIC, &end);

    double ns = (end.tv_sec - start.tv_sec) * 1e9
              + (end.tv_nsec - start.tv_nsec);
    printf("Syscall latency: %.0f ns/call\n", ns / ITERATIONS);

    // Compare with vDSO call:
    clock_gettime(CLOCK_MONOTONIC, &start);
    struct timespec ts;
    for (int i = 0; i < ITERATIONS; i++) {
        clock_gettime(CLOCK_MONOTONIC, &ts);  // vDSO, no syscall
    }
    clock_gettime(CLOCK_MONOTONIC, &end);

    ns = (end.tv_sec - start.tv_sec) * 1e9
       + (end.tv_nsec - start.tv_nsec);
    printf("vDSO latency:   %.0f ns/call\n", ns / ITERATIONS);
}

Benchmark 2: Context Switch Cost

#include <unistd.h>
#include <time.h>
#include <stdio.h>

#define ROUNDS 100000

int main() {
    int pipe1[2], pipe2[2];
    pipe(pipe1);
    pipe(pipe2);
    char byte = 0;

    if (fork() == 0) {
        // Child: ping-pong via pipes
        for (int i = 0; i < ROUNDS; i++) {
            read(pipe1[0], &byte, 1);
            write(pipe2[1], &byte, 1);
        }
        return 0;
    }

    // Parent: measure round-trip time
    struct timespec start, end;
    clock_gettime(CLOCK_MONOTONIC, &start);

    for (int i = 0; i < ROUNDS; i++) {
        write(pipe1[1], &byte, 1);
        read(pipe2[0], &byte, 1);
    }

    clock_gettime(CLOCK_MONOTONIC, &end);
    double ns = (end.tv_sec - start.tv_sec) * 1e9
              + (end.tv_nsec - start.tv_nsec);
    // Each round = 2 context switches (parent->child, child->parent)
    printf("Context switch: %.0f ns\n", ns / ROUNDS / 2);
}

Benchmark 3: Page Fault Latency

#include <sys/mman.h>
#include <time.h>
#include <stdio.h>

#define PAGES 10000
#define PAGE_SIZE 4096

int main() {
    // Allocate anonymous pages (not backed by file)
    char *region = mmap(NULL, PAGES * PAGE_SIZE,
                       PROT_READ | PROT_WRITE,
                       MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

    struct timespec start, end;
    clock_gettime(CLOCK_MONOTONIC, &start);

    // Touch each page — triggers a page fault per page
    for (int i = 0; i < PAGES; i++) {
        region[i * PAGE_SIZE] = 1;  // First write to page = fault
    }

    clock_gettime(CLOCK_MONOTONIC, &end);
    double ns = (end.tv_sec - start.tv_sec) * 1e9
              + (end.tv_nsec - start.tv_nsec);
    printf("Page fault (anon): %.0f ns/fault\n", ns / PAGES);

    munmap(region, PAGES * PAGE_SIZE);
}

The Decision Table

When to Optimize Syscalls

If your I/O hits disk or network (50μs+), syscall overhead is noise. Focus on reducing I/O count instead. If your I/O hits page cache (0.5μs), syscall overhead is 40% of total — io_uring or mmap gives real wins. If you do millions of tiny operations, syscall overhead dominates — batch with io_uring or eliminate with mmap.

Wild: The Shard Math

Your shard has 500 tensor entries. Each entry is 16KB. Reading them all:

Naive read(): 500 syscalls * 200ns = 100μs overhead + 500 * 50μs NVMe = 25.1ms total.
io_uring batch: 1 syscall + 500 parallel NVMe reads = 50μs (NVMe handles parallel queue) + ~1μs submit = ~51μs total.
mmap + MAP_POPULATE: 1 syscall + sequential prefault = ~25ms at startup, then 0ns per access forever.
mmap lazy: 500 page faults * 50μs = 25ms first pass, then 500 * 2μs = 1ms cached.

Choose based on access pattern: random access = mmap. Sequential scan = io_uring batch. One-shot load = MAP_POPULATE. Serving over network = sendfile.

Zen: Know Your Boundary

Every abstraction has a cost. The syscall boundary costs ~200ns. The page fault boundary costs ~2-50μs. The network boundary costs ~50μs (local) to ~50ms (cross-continent). The disk boundary costs ~50μs (NVMe) to ~10ms (HDD). Once you know these numbers, you know which boundaries to eliminate (mmap removes the syscall boundary for reads), which to batch across (io_uring amortizes syscall overhead), and which to skip entirely (vDSO, sendfile). The kernel is not your enemy. It is a tool. Learn to bend it.
