Era 1 — Just make it run
WEIGHTS → HBM → SRAM → CORES   the real pipeline
You thought the problem was compute. Usually it’s movement.
A 27B model is not a function. It’s a file you keep dragging through memory._

Part I traced a vector through 64 transformer blocks. You counted the FLOPs: 44 billion per token for Qwen 27B. That sounds like a compute problem. It isn’t. Modern GPUs can compute hundreds of teraFLOPs per second. An RTX 3090 does 142 trillion 16-bit operations every second. At that rate, 44 billion FLOPs should take 0.3 milliseconds. One token every 0.3 ms would mean 3,000 tokens per second.

You don’t get 3,000 tokens per second. You get about 30. The model runs 100× slower than the arithmetic alone would suggest. Something else is eating the time. That something is memory bandwidth — the speed at which data travels from GPU memory to the compute cores. The GPU is fast. The pipe feeding it is narrow.

The Memory Wall

A GPU has two relevant resources: compute (how many operations per second) and bandwidth (how many bytes per second it can read from memory). These two numbers define a boundary called the roofline.

GPUFP16 ComputeBandwidthVRAMRidge Point
RTX 3090142 TFLOPS936 GB/s24 GB~152 FLOPs/byte
RTX 4090165 TFLOPS1,008 GB/s24 GB~164 FLOPs/byte
A100 80GB312 TFLOPS2,039 GB/s80 GB~153 FLOPs/byte
H100 SXM989 TFLOPS3,350 GB/s80 GB~295 FLOPs/byte

The ridge point is where the bottleneck shifts. Below it, you’re memory-bound — the GPU finishes its math before the next chunk of data arrives. Above it, you’re compute-bound — the data arrives faster than the GPU can process it. The ridge point is simply peak compute divided by peak bandwidth: how many FLOPs you need per byte of data to keep the GPU fully busy.

Where Decode Lives

During decode, generating one token requires reading the entire model from memory. For a 7B FP16 model, that’s 14 GB per token. The compute is approximately 2 FLOPs per parameter — 14 billion FLOPs. The arithmetic intensity is:

Arithmetic intensity — decode at batch size 1 

  Bytes read per token:  7B params × 2 bytes = 14 GB
  FLOPs per token:       7B params × 2 FLOPs  = 14 GFLOPs

  Arithmetic intensity = 14 GFLOPs / 14 GB = ~1 FLOP per byte

  Ridge point of an A100:  ~153 FLOPs per byte
  Decode intensity:          ~1 FLOP per byte

  Decode is 153× below the ridge point.
  The GPU’s compute units are ~99.3% idle during decode.

  ─────────────────────────────────────────────────

  For any model at FP16, batch size 1:
    Arithmetic intensity ≈ 1 FLOP/byte  ← independent of model size
    The ratio is always ~1 because FLOPs and bytes scale together.

This is the central insight: decode doesn’t get faster with a faster GPU. The H100 has 3.2× the compute of the A100, but only 1.6× the bandwidth. For decode, you get roughly 1.6× the speed — the extra compute sits idle. The H100 is paradoxically less utilized during decode than the A100 because NVIDIA added compute faster than they added bandwidth.

Where Prefill Lives

Prefill is a different animal. Processing a 512-token prompt means batching all 512 tokens through every matrix multiplication simultaneously. The compute scales with sequence length, but the weight reads don’t — you read the weights once and reuse them across all tokens. That changes the ratio:

Arithmetic intensity scales with sequence length during prefill 

  seq_len = 1:     AI ≈ 1 FLOP/byte    memory-bound   (same as decode)
  seq_len = 32:    AI ≈ 32 FLOPs/byte  memory-bound
  seq_len = 128:   AI ≈ 128 FLOPs/byte approaching ridge on A100 (153)
  seq_len = 256:   AI ≈ 256 FLOPs/byte compute-bound on A100
  seq_len = 512:   AI ≈ 512 FLOPs/byte deeply compute-bound
  seq_len = 2048:  AI ≈ 2048 FLOPs/byte GPU fully saturated

  The crossover (memory → compute) happens at the ridge point:
    RTX 3090:  seq_len ≈ 152
    A100:      seq_len ≈ 153
    H100:      seq_len ≈ 295  ← needs longer prompts to saturate

This is why the H100 shows its true advantage during prefill — 3.2× faster than the A100 — but only modest gains during decode (~1.6×). Prefill is a compute race. Decode is a bandwidth race. Same model, same GPU, entirely different physics.

The Utilization Trap

An RTX 3090 generating tokens from a 27B model at batch size 1 uses 0.66% of its compute capability. The other 99.34% is idle, waiting for data. You paid for 142 TFLOPS. You’re using 0.9 TFLOPS. The GPU isn’t slow. The memory bus is narrow.

One Decode Step, Dissected

Here is exactly what happens during a single decode step — one token generated from a 7B FP16 model on an A100:

One token of decode — 7B FP16 on A100 

  read weights            14 GB streamed from HBM
  read KV cache           grows with every token (more later)
  run kernels             matmuls, norms, activations
  write new K, V          append one row per layer
  project logits          multiply by embedding table
  sample one token        temperature, top-p, draw
  repeat

  Time budget:
  Memory read:  14 GB / 2,039 GB/s  =  6.9 ms     ← this dominates
  Compute:      14 GFLOPs / 312 TF  =  0.045 ms   ← 150× faster than read

  Token latency ≈ 6.9 ms → ~145 tokens/sec theoretical max
  (Real: ~100-120 tok/s due to KV cache reads, kernel overhead)
Concrete numbers for popular model sizes on A100 80GB 

  Model       Weight Size   Read Time    Compute Time   Theoretical tok/s
  ────────────────────────────────────────────────────────────────────
  7B  FP16    14 GB         6.9 ms       0.045 ms       ~145
  14B FP16    28 GB         13.7 ms      0.090 ms       ~73
  27B FP16    54 GB         doesn’t fit   —              OOM
  27B Q4      ~16 GB        7.8 ms       0.090 ms       ~128
  72B Q4      ~42 GB        20.6 ms      0.230 ms       ~49

  ────────────────────────────────────────────────────────────────────

  Pattern: tok/s ≈ bandwidth / weight_size.
  Compute time is always <1% of total. Always.
Batching amortizes weight reads across requests 

  Batch size 1:
  Read 14 GB of weights → compute 1 token → 1 FLOP/byte

  Batch size 8:
  Read 14 GB of weights → compute 8 tokens → 8 FLOPs/byte

  Batch size 64:
  Read 14 GB of weights → compute 64 tokens → 64 FLOPs/byte

  Batch size 256:
  Read 14 GB of weights → compute 256 tokens → 256 FLOPs/byte
  ✓ Past the A100 ridge point — now compute-bound!

  This is why serving engines batch aggressively.
  More users = better GPU utilization.
  The single-user experience is memory-bound.
  The fleet experience can be compute-bound.

  But: each batch member needs its own KV cache.
  VRAM fills fast. Batch size is limited by memory, not desire.
  That tension is the central problem of Part III.

The Real Pipeline

Part I showed you the math pipeline: embed → norm → attention → FFN → project → sample. The real pipeline, the one that determines speed, is: weights → HBM → SRAM → cores. Before you optimize the model, you optimize the trip the numbers take.
