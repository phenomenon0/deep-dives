Part III — Serving at Scale

// 13 — One User Becomes Many

PROMPTS → QUEUE → BATCH → SERVICE
The first user is a benchmark. The hundredth user is a policy problem._

The moment multiple people share one model, clean diagrams break. Now prompt lengths vary. Output lengths vary. SLAs appear. Tail latency appears. A single long prefill can freeze every other user’s stream for half a second. Serving begins where toy inference ends.

The Traffic

Little’s Law Meets KV Cache

The queuing math that governs VRAM 

  Little’s Law:  L = λ × W

  L = average number of concurrent requests in the system
  λ = arrival rate (requests per second)
  W = average time a request spends in the system

  Example: λ = 50 req/s, W = 2 seconds
  L = 100 concurrent requests

  For Llama-3.3 70B (GQA, 8 KV heads) at avg 4K context:
  Per-request KV: 320 KiB/token × 4096 = 1.25 GB
  100 concurrent requests: 125 GB of KV cache alone
  + 140 GB model weights = needs multiple A100s just for memory

  ──────────────────────────────────────────────

  The arrival rate sets how many KV caches you hold simultaneously.
  KV cache size determines your maximum concurrency.
  Maximum concurrency determines your maximum throughput.
  Everything connects back to the cache from Chapter 6.

Tail Latency

P50 latency might be 500ms. P99 might be 5 seconds — 10× worse. Tail latency is what users remember. It’s also what SLOs are written against: “99% of requests complete in <2 seconds.”

What causes tail latency in LLM serving 

  1. Long prefills block decode
     A 32K-token prefill takes ~400ms of pure compute.
     Every decode request in the batch is frozen during this.
     → Chunked prefill (Ch 10) is the fix.

  2. Preemption
     When VRAM fills, requests get evicted and recomputed later.
     Preemption accounts for ~70% of P99 latency (vLLM measurements).
     → Better scheduling and memory management (Ch 14).

  3. Heavy-tailed output lengths
     Most responses are short. Some are very long.
     Long responses hold KV cache and batch slots for much longer.
     → Max-tokens limits reduce E[S²] variance by 59%.

  4. Queue buildup at high load
     As utilization approaches 100%, queuing delay explodes.
     M/G/1 queue: wait time ∝ ρ/(1−ρ) × E[S²]
     → Scale out before hitting the knee of the curve.

  A service can feel fast at P50 and be terrible at P99.
  SLOs are written on tails, not averages.
Fast vs Reliable

A demo with one user is always fast. A service with a thousand users needs predictability. The difference between “fast” and “reliable” is that reliability holds under load. Every optimization from Part II — quantization, FlashAttention, continuous batching — becomes a scheduling input in Part III. The question is no longer “how fast can one token be?” It’s “how many tokens can you serve without anyone waiting too long?”

// 14 — The Scheduler

ADMIT · PAUSE · RESUME · EVICT
Kernels make a model fast. Schedulers decide who gets served._

At scale, the scheduler becomes the real product. Who enters the batch? Who gets preempted? Who keeps their KV cache? Who gets evicted when memory runs tight? These decisions, made every decode step, determine whether your service meets its SLO or breaks it.

The Loop

vLLM V1 scheduler — the two-phase loop 

  every decode step:

  Phase 1: Service running requests
  ───────────────────────────────────
  for each running request:
      Can we allocate KV blocks for its next token?
      Yes → continue decoding
      No  → preempt (evict KV, move to waiting queue)

  Phase 2: Admit waiting requests
  ───────────────────────────────────
  remaining_budget = max_num_batched_tokens − decode_tokens
  while waiting_queue not empty and budget > 0:
      req = waiting_queue.pop()
      if enough KV blocks for req’s prompt:
          schedule prefill (or chunked prefill if prompt > budget)
          budget −= prompt_tokens
      else:
          break  ← no room, stop admitting

  Run forward pass with the assembled batch.

  Key controls:
    max_num_seqs = 128      ← max concurrent sequences
    max_num_batched_tokens   ← total token budget per step
    gpu_memory_utilization = 0.9  ← VRAM fraction for KV
Scheduling policies 

  FCFS (First Come First Served)
  Simple. Fair. But head-of-line blocking: one 32K-token prompt
  holds up everything behind it. Most common default.

  Decode-First (vLLM V1 with chunked prefill)
  All pending decode tokens scheduled before any prefill work.
  Prefill chunks fill remaining budget. Protects ITL for active users.
  This is why vLLM V1 has better tail latency than V0.

  Priority-Based
  Premium users get higher priority. Time-sensitive requests first.
  Requires application-level priority signals.

  Sarathi-Serve
  Decode-maximal batching: maximize decode tokens per step.
  2.6–6.3× higher capacity under strict P99 SLOs vs vLLM.
Preemption — when VRAM runs out 

  RECOMPUTE (vLLM V1 default)
  ───────────────────────────────────
  1. Evict most recently arrived request (LIFO under FCFS)
  2. Free its KV cache blocks
  3. Request returns to waiting queue
  4. When space frees: re-enter, recompute KV from scratch

  Cost: one forward pass over (prompt + generated-so-far).
  For a 4K-context request: ~200ms to recompute. Cheaper than swap.

  SWAP (vLLM V0, now deprecated)
  ───────────────────────────────────
  1. Copy KV blocks to CPU RAM via PCIe (32 GB/s)
  2. When space frees: copy back to GPU
  PCIe is 60× slower than HBM. Complex memory management.
  Only needed for beam search where recompute doesn’t work.

  Preemption accounts for ~70% of P99 latency.
  The best preemption strategy is not needing to preempt.
The throughput-latency tradeoff 

  Small batch (e.g., max_seqs=8):
    Latency:     low (few requests competing)
    Throughput:  low (GPU underutilized)
    Preemption:  rare
    Good for: latency-sensitive chat, real-time assistants

  Large batch (e.g., max_seqs=128):
    Latency:     higher (more contention, more preemption)
    Throughput:  high (GPU well-utilized)
    Preemption:  frequent
    Good for: batch processing, high-throughput APIs

  The optimal point depends on your SLO.
  vLLM’s auto_tune.sh script finds it by binary search:
  iterate max_seqs and token budget until P99 meets target.
The Real Product

A serving engine is not just kernels. It is a policy machine. The scheduler decides, every 7 milliseconds, who gets GPU time and who waits. Get it wrong and your P99 explodes. Get it right and the same hardware serves 6× more users. The scheduler is the most underappreciated component in the stack.

// 15 — Prefix Caching

IF THE BEGINNING IS THE SAME, DON’T PAY TWICE
Most chat systems repeat themselves before the user says anything._

System prompts. Policy prompts. Tool instructions. Few-shot scaffolds. RAG preambles. Every request to a chatbot starts with hundreds or thousands of identical tokens. That repetition is money — money you can stop spending. Prefix caching reuses KV blocks that have already been computed, skipping redundant prefill and slashing TTFT.

Shared prefixes in the wild 

  Request A: [system prompt][policy][tools][user A’s message]
  Request B: [system prompt][policy][tools][user B’s message]
  Request C: [system prompt][policy][tools][user C’s message]

  ██████████████████████████████ shared prefix — compute once, reuse forever
  █████████████ unique suffixes — only these need fresh prefill

  10,000 requests × 1,000-token system prompt:
    Without caching: 10,000 × 1,000 = 10M tokens of prefill
    With caching:    1 × 1,000 + 10,000 × suffix only
    TTFT: drops from seconds to milliseconds for cached prefixes
vLLM Automatic Prefix Caching 

  Block-level hashing:
  Each full KV block (16 tokens) gets a SHA-256 hash of:
    parent_block_hash + token_IDs + extra_keys (LoRA, multimodal)

  On new request:
  1. Hash the prompt tokens block-by-block
  2. Look up each hash in BlockHashToBlockMap
  3. Cache hit: reuse physical block, bump ref_cnt
  4. Cache miss: allocate new block, compute KV, register hash

  Eviction:
  LRU on blocks with ref_cnt == 0.
  Leaf blocks (end of sequences) evicted first.
  Near-zero overhead on miss — just a hash lookup.

  Measured improvement (vLLM docs):
  10K-token repeated prompt: TTFT 4.3s → 0.6s  (7× faster)
  vLLM calls this “a free lunch” — near-zero cost, huge win.
SGLang RadixAttention — prefix tree for KV 

  Token-level radix tree (vs vLLM’s block-level hashing):

                         [system prompt]
                        /               \
              [user A history]      [user B history]
              /           \              |
         [turn 3A]    [turn 3A']    [turn 2B]

  Advantages over block hashing:
  • Caches generation tokens too, not just prompts
  • Multi-turn conversations share history branches
  • Fork/merge semantics for parallel sampling
  • Token-level granularity → no block-boundary alignment issues

  Result: 10–20% advantage on multi-turn cached workloads
  Up to 5× throughput over vLLM v0.2.5 on prefix-heavy loads

  RadixAttention is why SGLang wins on chat applications.
  The more your traffic shares structure, the bigger the win.
Prefix caching economics 

  Production cache hit rates:
    Chat applications:     60–90%
    RAG pipelines:         40–70% (document overlap)
    Tool-use agents:       70–85% (repeated function schemas)
    Batch processing:      90%+ (same prompt template)

  Cost impact:
    Cached tokens cost ~10× less than fresh prefill.
    TTFT for cached requests: near-instant (just suffix prefill).
    Memory cost: blocks stay in GPU memory (LRU eviction).

  Cluster-scale (llm-d project):
    Global KV-block index across multiple GPUs.
    Prefix-aware routing: send requests to the GPU that has the prefix.
    Result: 57× faster P90 TTFT vs approximate scheduling.
    365 GB of cached KV tracked in just 339 KB of metadata.

  Prefix caching is the highest-leverage serving optimization
  for any application with repeated prompt structure.
  If you serve a chatbot and don’t cache prefixes, you’re
  paying full price for the same work, thousands of times.
The Free Lunch

Prefix caching is the closest thing to a free lunch in inference. Enable it, pay near-zero overhead on cache misses, and watch TTFT collapse for your most common traffic patterns. The combination of prefix caching + continuous batching + chunked prefill is what makes modern LLM serving feel instant instead of sluggish.

// 16 — Disaggregated Prefill / Decode

ONE CLUSTER READS · ANOTHER SPEAKS
The same model wants different hardware depending on whether it is reading or talking._

Prefill wants compute. Decode wants bandwidth. Forcing one GPU to do both means it excels at neither — compute sits idle during decode, and decode requests freeze during long prefills. The fix is physical separation: dedicated GPU pools for each phase, connected by fast interconnect, each tuned for its workload.

Disaggregated architecture 

  Client request
       ↓
  Smart Router  ← radix tree for prefix/KV location tracking
       ↓
  ┌─────────────────────┐         ┌─────────────────────┐
  │ PREFILL POOL          │  KV    │ DECODE POOL           │
  │                       │ →→→→ │                       │
  │ Compute-bound         │ xfer  │ Bandwidth-bound       │
  │ Dense matmuls         │       │ Read weights for 1 tok│
  │ High GPU utilization  │       │ Needs max bandwidth   │
  │ TP optimized for      │       │ TP optimized for      │
  │ throughput             │       │ latency               │
  └─────────────────────┘         └─────────────────────┘
       ↓                                   ↓
  TTFT optimized independently      ITL optimized independently
Research results 

  DistServe (OSDI 2024)
  ───────────────────────────────────
  Eliminates prefill-decode interference entirely.
  Independent parallelism strategies per phase.
  Placement algorithm accounts for interconnect bandwidth.

  Results: 7.4× more requests served
           12.6× tighter SLO compliance

  Splitwise (ISCA 2024)
  ───────────────────────────────────
  Observed that decode underutilizes compute on high-end GPUs.
  Splits phases across machines via high-bandwidth interconnect.

  Results: 1.4× throughput at 20% lower cost
           2.35× throughput at identical cost
The KV transfer bottleneck 

  Llama-3.3 70B, 4K context, GQA 8 KV heads:
  Per-token KV: 320 KiB × 4096 tokens = 1.34 GB

  Transfer time by interconnect:
  NVLink (900 GB/s):       1.5 ms    ← ideal, same-node
  InfiniBand (400 Gb/s):   27 ms     ← acceptable, cross-node
  100 GbE:                 110 ms    ← painful, adds to TTFT

  Disaggregation only works with fast interconnect.
  NVLink within a node. InfiniBand across nodes.
  Ethernet is too slow for the KV transfer.
Disaggregation in production — March 2026 

  NVIDIA Dynamo
  Open-sourced GTC 2025, production March 2026.
  “Inference operating system” wrapping TRT-LLM, vLLM, SGLang.
  Smart Router with radix tree for KV location tracking.
  Dynamic Planner: switches between disagg and aggregated modes.
  DeepSeek-R1 671B on GB200 NVL72: 30× throughput.

  vLLM
  Experimental disaggregated prefill in production at
  Meta, LinkedIn, Mistral, HuggingFace.

  SGLang
  P/D disaggregation as core feature.
  96 H100s: 52,300 input tokens/sec with 3:9 prefill:decode ratio.

  Every major framework now supports disaggregation.
  This is no longer a research idea. It’s table stakes for
  large-scale deployments in 2026.
The Datacenter Technique

Disaggregation feels like overkill until you run the numbers. A single GPU running both phases wastes compute during decode and blocks decode during prefill. Two specialized pools, connected by fast interconnect, let each phase run at its natural pace. It’s the same insight as separating reads and writes in a database — applied to the two fundamental operations of LLM inference.

// 17 — Parallelism Across GPUs

TP · PP · DP · EP
A bigger model does not just need more memory. It changes how the work is divided._

Once one GPU is not enough, inference becomes distributed systems. The question is how to cut the model. Tensor parallelism splits matrices. Pipeline parallelism splits layers. Data parallelism duplicates replicas. Expert parallelism routes sparse experts. Each strategy has different communication patterns, different latency profiles, and different hardware requirements.

The four parallelism strategies 

  Tensor Parallelism (TP) — split one matmul
  ──────────────────────────────────────────
  Weight matrix split column-wise across GPUs.
  Each GPU computes a slice of every layer.
  Requires all-reduce after each matmul (2× per layer).
  Needs NVLink (600–900 GB/s). PCIe is too slow.
  Best for: 2–8 GPUs on the same node.

  Pipeline Parallelism (PP) — split the layer stack
  ──────────────────────────────────────────
  GPU 0: layers 0–15.  GPU 1: layers 16–31.  Etc.
  Requests flow through the pipeline.
  Bubble fraction: (p−1)/(p+m−1) where p=stages, m=microbatches.
  Lower bandwidth requirement than TP.
  Best for: cross-node, when NVLink isn’t available.

  Data Parallelism (DP) — copy the whole model
  ──────────────────────────────────────────
  Full model on each GPU. Each handles different requests.
  No cross-GPU communication during inference.
  Scales throughput linearly with GPU count.
  Best for: when model fits on one GPU and you want more QPS.

  Expert Parallelism (EP) — route sparse experts
  ──────────────────────────────────────────
  For MoE models (DeepSeek-V3: 671B total, 37B active).
  Different experts live on different GPUs.
  All-to-all communication routes tokens to their experts.
  Best for: MoE inference where only ~10% of params activate.

The Alphabet Soup

TP, PP, DP, EP are not four independent choices. They compose. A 70B model on 8 H100s might use TP=4 within a node and PP=2 across nodes. A MoE model adds EP on top. vLLM, SGLang, and TensorRT-LLM all let you specify these as launch parameters. The trick is matching the parallelism to your hardware topology — NVLink for TP, InfiniBand for PP, replication for DP.

// 18 — LoRA and Multi-Tenancy

ONE BASE · MANY PERSONALITIES
You do not always serve one model. Sometimes you serve one trunk wearing many masks._

Adapter serving changes the economics. Instead of loading many full models, you keep one base resident and swap deltas. A LoRA adapter for a 70B model is typically 100–500 MB — 0.1–0.4% of base model size. That sounds simple until batches mix tenants, prefixes diverge, and memory reuse gets messy.

Multi-LoRA serving — one base, many adapters 

  Base model weights (frozen, resident in VRAM)
  ──────────────────────────────────────────
   + Adapter A (rank 16, ~200 MB)  → Legal assistant
   + Adapter B (rank 8, ~100 MB)   → Support bot
   + Adapter C (rank 32, ~400 MB)  → Finance copilot
   + Adapter D (rank 16, ~200 MB)  → Code reviewer

  In one batch:
  Request 1: base + A     Request 2: base + B
  Request 3: base + A     Request 4: base + C

  Base computation is shared. Only adapter portions differ.
  KV cache layout is the same for all adapters (same architecture).
  Prefix caching works across adapters if prompts match.
S-LoRA — serving thousands of adapters 

  S-LoRA (Sheng et al., 2023)
  ───────────────────────────────────
  Problem: serving thousands of LoRA adapters naively
  means loading/unloading adapter weights per request.

  Solution: Unified Paging
  • Shared memory pool for adapter weights AND KV cache
  • Adapters paged in/out like KV cache blocks
  • Custom CUDA kernels for heterogeneous batches
    (different requests use different adapter ranks)

  Result: 4× throughput over naive adapter serving.
  Can serve thousands of adapters on a single GPU.
  Adapters hot-swapped via LRU eviction.
Multi-LoRA in vLLM and SGLang 

  vLLM configuration:
  --enable-lora
  --max-loras 8             ← max adapters resident simultaneously
  --max-lora-rank 64        ← max adapter rank
  --lora-extra-vocab-size 0

  API usage:
  POST /v1/chat/completions
  {
    "model": "base-model",
    "lora_name": "legal-assistant",
    "messages": [...]
  }

  SGLang: multi-LoRA batching built in.
  Different requests in the same batch can use different adapters.
  Base model forward pass is shared; adapter deltas applied per-request.

  The shift: instead of N separate model deployments,
  one deployment with N adapters. 100× less VRAM.
The Enterprise Pattern

A lot of real products are not “serve one base model to everyone.” They’re “one base model, fine-tuned for each customer.” Multi-LoRA serving turns this from N model deployments into one deployment with N adapters. The infrastructure cost drops 10–100×. The latency stays the same. This is why adapter serving is becoming a first-class feature in every production engine.

// 19 — Metrics That Matter

TTFT · ITL · TPS · CACHE HIT · $/TOKEN
A service can feel fast and still be bad. A service can benchmark well and still be expensive._

If you do not measure the right things, inference lies to you. One number is never enough. Local “tokens per second” is not a service metric. A good TTFT once is not reliability. The metrics that matter depend on whether you’re optimizing for a demo or a fleet.

The metrics taxonomy 

  Latency Metrics
  ──────────────────────────────────────────
  TTFT   Time to First Token         User sees first word
         Good: <500ms  Bad: >2s       Dominated by prefill

  ITL    Inter-Token Latency          Streaming smoothness
         Good: <30ms   Bad: >100ms    Dominated by decode

  E2E    End-to-End Latency           Total time for full response
         = TTFT + (output_tokens × ITL)

  Throughput Metrics
  ──────────────────────────────────────────
  TPS    Tokens Per Second (system)   Total output across all users
  RPS    Requests Per Second          Completed requests per second
  Goodput Requests meeting SLO        RPS that actually matters

  Operational Metrics
  ──────────────────────────────────────────
  Cache Hit Rate     Prefix cache efficiency    Good: >60%
  Preemption Rate    How often requests evicted  Good: <5%
  Queue Depth        Requests waiting            Good: <2× batch
  GPU Utilization    Actual vs theoretical       (misleading alone)

  Economic Metrics
  ──────────────────────────────────────────
  $/M tokens         Cost per million tokens served
  $/request          Amortized cost per user interaction
Seven benchmark traps 

  1. Local tok/s is not a service metric.
     150 tok/s at batch=1 says nothing about 50 concurrent users.

  2. TTFT and ITL are separate concerns.
     A system can have excellent TTFT and terrible ITL (or vice versa).

  3. Average latency hides tails.
     P50=200ms, P99=5000ms is a bad service with a good average.

  4. GPU utilization is misleading.
     90% GPU util during prefill, 1% during decode. “High utilization”
     doesn’t mean the service is fast.

  5. Concurrency changes every graph.
     A system that benchmarks well at 1 user may collapse at 100.

  6. Prompt length distribution matters.
     Benchmarks with uniform 128-token prompts miss the
     heavy-tailed reality of production traffic.

  7. Throughput without SLO is meaningless.
     You can always increase throughput by increasing batch size.
     The question is: at what latency cost?
Measure What Users Feel

Users feel TTFT (how long before the first word appears) and ITL (how smooth the stream is). They do not feel GPU utilization or system throughput. Build your dashboards around what users feel. Optimize for goodput (requests meeting SLO), not raw throughput. And always benchmark at production concurrency levels, not batch=1.

// 20 — The Modern Serving Stack

ROUTER → CACHE → SCHEDULER → ENGINE → METRICS
By the end, the model is only one component in the system built around it._

A modern serving stack is a pipeline of decisions: which request goes where, what work can be reused, what engine runs the model, how memory is managed, how latency is observed, and when to scale. The model — those billions of weights — sits at the center. Everything else exists to feed it efficiently and measure whether it is working.

The full stack — request lifecycle 

  Client
      ↓
  Gateway / Load Balancer
      ↓  rate limiting, auth, routing
  Prefix Cache Lookup
      ↓  KV-aware routing: send to GPU that has the prefix
  Scheduler
      ↓  admit / preempt / chunk prefill / batch assembly
  Engine Runtime
      ↓  PagedAttention, FlashAttention, continuous batching
  GPU(s)
      ↓  TP / PP / EP across devices
  Metrics & Autoscaling
      ↓  TTFT, ITL, cache hit, preemption rate, queue depth
  Policy Loop
      ↑  scale up/down, adjust token budgets, rebalance pools

Choose Your Architecture

Six deployment archetypes 

  1. Local-First
     llama.cpp or ExLlamaV3 on consumer GPU/Mac.
     Single user, no scheduling needed.
     For: developers, hobbyists, privacy-sensitive use.

  2. Single-Node Production
     vLLM or SGLang on one multi-GPU server.
     Continuous batching, prefix caching, OpenAI API.
     For: startups, internal tools, 10–100 concurrent users.

  3. Multi-Node Scale-Out
     Dynamo orchestrating TRT-LLM/vLLM/SGLang across nodes.
     Disaggregated P/D, KV-aware routing, dynamic scaling.
     For: API providers, 1000+ concurrent users.

  4. Adapter-Heavy Enterprise
     Multi-LoRA on SGLang or vLLM.
     One base model, N customer-specific adapters.
     For: SaaS products with per-customer fine-tuning.

  5. Low-Latency Assistant
     Disaggregated P/D with aggressive prefix caching.
     Optimized for TTFT < 200ms, ITL < 20ms.
     For: real-time chat, voice assistants, coding copilots.

  6. High-Throughput Batch
     Maximize batch size, accept higher latency.
     vLLM with max_num_seqs=256, gpu_memory_utilization=0.95.
     For: data pipelines, offline processing, evaluation.
