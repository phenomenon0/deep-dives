Before we optimize anything, we need to understand what we're optimizing. So let's start at the beginning — literally. February 2019. OpenAI releases GPT-2: 124 million parameters, 12 transformer layers, 768-dimensional hidden states, 12 attention heads. The world freaks out. OpenAI calls it "too dangerous to release." Today it's a toy, smaller than the model running your phone's autocomplete. But it contains every architectural idea we'll need. If you understand GPT-2's forward pass, you understand Qwen-3.5 27B's. GPT-2 124M is small enough to hold in your head. Twelve layers. Each layer: attention (12 heads, 64 dims each) then a feed-forward network (768 → 3072 → 768). A vocabulary of 50,257 tokens. Total: 124 million floats. At FP32, that's 497 MB — fits in your laptop's RAM with room to spare. Let's write the entire forward pass.

The Complete Forward Pass

The entire model is a pipeline. Text goes in the bottom, probability distribution comes out the top. Every LLM you've ever used — Claude, GPT-4, DeepSeek, Qwen — does exactly this, with minor variations in the building blocks.


  "The capital of France"
      ↓
  ★ EMBED    Look up each token in a table of 50,257 vectors.
  │           "The" → [0.12, -0.34, 0.56, …]  (768 numbers)
  │           Add a position signal so the model knows word order.
      ↓
  ┌───────────────────────────────────────────────────────┐
  │ ★ NORM      Stabilise the numbers (mean = 0, variance = 1)         │
  │     ↓                                                        │
  │ ★ ATTENTION  Each token asks: "who in this sentence matters  │
  │              to me?"  Produces a relevance-weighted blend    │
  │              of all previous tokens' information.              │
  │     ↓  + skip connection (add original input back)        │
  │                                                               │
  │ ★ NORM      Stabilise again                                   │
  │     ↓                                                        │
  │ ★ FFN       Expand 768 → 3072, apply non-linearity, shrink     │
  │              back to 768.  This is where "thinking"        │
  │              happens — pattern matching, fact recall.       │
  │     ↓  + skip connection                                   │
  └───────────────────────────────────── ×12 identical ──┘
      ↓
  ★ NORM    Final stabilisation
      ↓
  ★ PROJECT  Multiply by the embedding table transposed to get
  │           a score for every word in the vocabulary.
      ↓
  [0.01, 0.02, …, 0.73, …]  ← 50,257 scores (logits)
  Highest score: token 318 = " is"

Seven matrix multiplications per block. Twelve blocks. Plus embedding + final projection. That's 86 big matmuls to produce one probability distribution. For GPT-2 that's ~250 million FLOPs. For Qwen-3.5 27B: 44 billion. Same recipe, 176× the bill.

Python
import numpy as np

def gelu(x):
    return 0.5 * x * (1 + np.tanh(np.sqrt(2/np.pi) * (x + 0.044715 * x**3)))

def softmax(x, axis=-1):
    e = np.exp(x - x.max(axis=axis, keepdims=True))
    return e / e.sum(axis=axis, keepdims=True)

def layer_norm(x, gamma, beta, eps=1e-5):
    mean = x.mean(axis=-1, keepdims=True)
    var  = x.var(axis=-1, keepdims=True)
    return gamma * (x - mean) / np.sqrt(var + eps) + beta

def attention(x, w_qkv, w_proj, b_qkv, b_proj, n_heads):
    # x: [seq_len, 768]
    B, D = x.shape
    qkv = x @ w_qkv + b_qkv          # [B, 3*768]  — project to Q, K, V
    q, k, v = np.split(qkv, 3, axis=-1)  # each [B, 768]

    # Reshape into 12 independent heads of dim 64
    head_dim = D // n_heads            # 768/12 = 64
    q = q.reshape(B, n_heads, head_dim).transpose(1,0,2)
    k = k.reshape(B, n_heads, head_dim).transpose(1,0,2)
    v = v.reshape(B, n_heads, head_dim).transpose(1,0,2)

    # Causal attention: score every pair, mask the future, softmax, blend
    scores = (q @ k.transpose(0,2,1)) / np.sqrt(head_dim)  # [12, B, B]
    mask = np.triu(np.full((B, B), -1e9), k=1)   # -inf above diagonal
    scores += mask
    attn = softmax(scores)             # attention weights
    out = (attn @ v)                   # [12, B, 64]  weighted blend of values

    # Merge heads back to [B, 768] and project
    out = out.transpose(1,0,2).reshape(B, D)
    return out @ w_proj + b_proj

def ffn(x, w_fc, b_fc, w_proj, b_proj):
    h = gelu(x @ w_fc + b_fc)         # [B, 768] → [B, 3072]  expand + activate
    return h @ w_proj + b_proj         # [B, 3072] → [B, 768]  compress back

def transformer_block(x, block):
    # Pre-norm → attention → residual
    h = layer_norm(x, block['ln1_g'], block['ln1_b'])
    x = x + attention(h, block['attn_qkv_w'], block['attn_proj_w'],
                          block['attn_qkv_b'], block['attn_proj_b'], 12)
    # Pre-norm → FFN → residual
    h = layer_norm(x, block['ln2_g'], block['ln2_b'])
    x = x + ffn(h, block['fc_w'], block['fc_b'],
                   block['proj_w'], block['proj_b'])
    return x

def gpt2_forward(token_ids, model):
    x = model['wte'][token_ids]  +  model['wpe'][np.arange(len(token_ids))]
    # wte: [50257, 768] token embeddings
    # wpe: [1024, 768]  position embeddings (max 1024 context)

    for block in model['blocks']:       # 12 transformer blocks
        x = transformer_block(x, block)

    x = layer_norm(x, model['ln_f_g'], model['ln_f_b'])
    logits = x @ model['wte'].T       # tie weights: reuse embedding table
    return logits                       # [seq_len, 50257]

Every LLM does exactly this. RMSNorm instead of LayerNorm. SwiGLU instead of GELU. RoPE instead of learned positions. The skeleton is the same. The variations in Chapter 2.

Hybrid Attention Note

Qwen-3.5 introduces hybrid attention: ~75% of layers use Gated DeltaNet (a linear attention variant for efficient long-range processing), while ~25% use full GQA (standard softmax attention for precise retrieval). We'll focus on the GQA layers throughout this deep dive — they contain all the mechanisms that matter for understanding inference. The DeltaNet layers are a compute optimization that avoids the quadratic attention cost on most layers.

The Autoregressive Loop

A language model generates text one token at a time. Run the forward pass, look at the last position's logits, sample a token, append it, repeat. Hit play to watch it happen.

This is the entire generation loop. The visualization above is running this logic.

Python
def generate(prompt_tokens, model, max_tokens=50, temperature=0.8):
    tokens = list(prompt_tokens)
    for _ in range(max_tokens):
        logits = gpt2_forward(tokens, model)  # run ALL tokens through
        next_logits = logits[-1]               # only last position matters
        probs = softmax(next_logits / temperature)
        next_token = np.random.choice(len(probs), p=probs)
        tokens.append(next_token)
        if next_token == eos_token: break
    return tokens

# "The capital of France" → " is Paris, a city known for..."

Notice the waste: at step 50 we rerun all 50 tokens through all 12 layers — even though tokens 1-49 haven't changed. That's 98% redundant work. The KV cache (Chapter 6) fixes this.

At each step: 12 layers × 7 matmuls = 84 matrix multiplications, plus attention softmax, GELU, and norms. For GPT-2 that's ~250M FLOPs per token. For Qwen-3.5 27B it's 44 billion. Same architecture, 176× the cost.

The Anchor

This 60-line forward pass IS a working inference engine. Everything in the next 17 chapters is "how do we make this faster, bigger, and serve it to thousands of users." Quantization makes the weights smaller. KV caching avoids recomputation. FlashAttention does the attention in fast SRAM. Batching shares the weight reads. But the core loop — embed, attend, project, sample, repeat — never changes.

GPT-2 was 124 million parameters. Today's frontier models are 27 billion (Qwen-3.5 27B), 671 billion (DeepSeek-V3, a Mixture-of-Experts model), or entirely undisclosed (Claude Opus 4.5). The dense architecture is the same. The scale is not. Qwen-3.5 27B: 64 layers, 5120-dimensional hidden state, 24 query heads, 4 KV heads (GQA), 17408-wide FFN. The weight file is 54 GB in FP16. That's too large for a single consumer GPU, but quantize it to 4-bit (~15 GB) and it fits on your RTX 3090 with room to spare. The question is: how do you make those numbers fast?

Step 1: RMSNorm

Modern models use RMSNorm instead of LayerNorm (GPT-2 used LayerNorm). The difference: RMSNorm skips the mean-centering step and has no bias. It's simpler and empirically works just as well.

RMSNorm — why normalize at all? 

  Problem: after each layer, vector magnitudes drift.
  Some dimensions explode, others shrink. Without correction,
  later layers see wildly different scales → training collapses.

  ─────────────────────────────────────────────────────────────

  Before RMSNorm:

    dim 0:  ████████████████████  340.2    ← exploding
    dim 1:  ██                      0.003    ← vanishing
    dim 2:  ████████████            89.7

  After RMSNorm:

    dim 0:  ████                     1.73     ✓ tamed
    dim 1:  █                        0.00002  ✓ ratio preserved
    dim 2:  ██                       0.46     ✓ tamed

  ─────────────────────────────────────────────────────────────

  Divide every dimension by the same number (the RMS).
  Relative proportions stay intact. Absolute scale is reset.
  Then γ (learned per-dimension) re-scales what matters.

  vs LayerNorm: RMSNorm skips mean-centering (no bias shift).
  Simpler, faster, works just as well — that’s why every modern LLM uses it.

Math
# Input: x = [0.5, -1.2, 0.8, 0.3, ...] (768 values)

RMS(x) = sqrt(mean(x²))
       = sqrt( (0.25 + 1.44 + 0.64 + 0.09 + ...) / 768 )
       = sqrt(0.847)    # typical RMS for a normalized hidden state
       = 0.920

x_norm = x / RMS(x)
       = [0.543, -1.304, 0.870, 0.326, ...]

output = x_norm * gamma   # gamma is a learned per-dimension scale [768]
       = [0.543 * 1.02, -1.304 * 0.97, ...]
       = [0.554, -1.265, ...]

# Cost: 3 passes over the vector (square, sum, divide+multiply)
# FLOPs: ~3 × 768 = 2,304. Trivial compared to matmul.

Step 2: QKV Projection

This is where the real cost starts. We multiply our normalized vector by three large weight matrices to get queries, keys, and values:

Math
# For Qwen-3.5 27B with GQA (4 KV heads, 24 query heads, head_dim=256):

Q = x_norm @ W_q     # [5120] × [5120, 6144] → [6144]  (24 heads × 256 dim)
K = x_norm @ W_k     # [5120] × [5120, 1024] → [1024]  (4 heads × 256 dim)
V = x_norm @ W_v     # [5120] × [5120, 1024] → [1024]  (4 heads × 256 dim)

# FLOPs for Q projection alone:
#   5120 × 6144 × 2 = 62,914,560 (63M FLOPs)
# K and V: 5120 × 1024 × 2 × 2 = 20.9M FLOPs
# Total QKV: ~84M FLOPs. Per block. Per token.

Notice the asymmetry: Q is 6× larger than K or V because of GQA. There are 24 query heads but only 4 KV heads — each KV head is shared across 6 query heads. Also note that W_q is not square: the output (6144) is larger than the input (5120) because head_dim (256) × n_heads (24) ≠ d_model (5120).

Q, K, V — what they actually mean 

  Every token produces three vectors from the same input:

  Q = "What am I looking for?"        the question this token asks
  K = "What do I contain?"             the label this token advertises
  V = "What information do I carry?"   the payload delivered if selected

  ─────────────────────────────────────────────────────────────

  Example: token "France" at position 4

  QFrance ≈ "I need context about countries, geography, politics"
  KFrance ≈ "I am a country name, European, noun"
  VFrance ≈ [rich vector encoding France's learned features]

  ─────────────────────────────────────────────────────────────

  The attention mechanism in two steps:

  Attention weights = softmax(Q · KT)
      ↑ match every query against every key
      ↑ high dot product = "this key answers my query"

  Output = Attention · V
      ↑ read the winning payloads, weighted by relevance

  Key insight: K decides WHO gets attention. V decides WHAT flows.
  They're decoupled — a token can be highly relevant (strong K match)
  but carry different information (V) than what K advertises.
Grouped-Query Attention — 64 query heads, 8 KV heads

Step 3: RoPE (Position Encoding)

How does the model know that "Paris" is the fifth word, not the first? Rotary Position Embedding encodes position by rotating query and key vectors in pairs of dimensions. Each dimension pair rotates at a different frequency, creating a unique angular signature for each position.

RoPE — position as rotation 

  Each dimension pair is a clock hand spinning at a different speed.
  Position = how far each hand has rotated from the start.

  ─────────────────────────────────────────────────────────────

  Dimension pair i=0 (slowest clock):

    pos 0:  ↑  0°       pos 1:  ↗  0.06°    pos 5:  ↗  0.3°
            tiny rotation per step → detects long-range gaps

  Dimension pair i=32 (medium clock):

    pos 0:  ↑  0°       pos 1:  →  5.6°     pos 5:  ↓  28°
            moderate rotation → mid-range position sensitivity

  Dimension pair i=63 (fastest clock):

    pos 0:  ↑  0°       pos 1:  ↙  180°    pos 5:  ↙  900°
            rapid rotation → distinguishes adjacent tokens

  ─────────────────────────────────────────────────────────────

  Each position gets a unique angular “fingerprint” across all 64 pairs.
  No two positions produce the same combination of angles.

Python
# RoPE: rotate pairs of dimensions by position-dependent angles
# For dimension pair (2i, 2i+1) at position pos:

freq = 1.0 / (base ** (2*i / d_model))  # base=10000000 for Qwen-3.5
theta = pos * freq

# Low frequency pairs (i=0): rotate slowly → captures long-range position
# High frequency pairs (i=63): rotate fast → captures fine position

q[2*i]   = q_orig[2*i]   * cos(theta) - q_orig[2*i+1] * sin(theta)
q[2*i+1] = q_orig[2*i]   * sin(theta) + q_orig[2*i+1] * cos(theta)

# The key insight: Q·K^T now encodes RELATIVE position.
# "How far apart are these tokens?" — not "what absolute position?"
# This is why context extension (YaRN, NTK) works: adjust frequencies.
Position invariance — only the gap matters 

  Scenario A:  Q at position 5,  K at position 3   → gap = 2
  Scenario B:  Q at position 100, K at position 98  → gap = 2

  ─────────────────────────────────────────────────────────────

  How RoPE makes this work:

  Q·KT after rotation = f(pos_Q − pos_K)

  The rotation of Q at pos 5 minus the rotation of K at pos 3
  = the same angle as Q at pos 100 minus K at pos 98.

  R(5) · R(3)T  =  R(5−3)  =  R(2)
  R(100) · R(98)T =  R(100−98) =  R(2)  ✓ identical

  ─────────────────────────────────────────────────────────────

  Rotation matrices cancel: R(θ) · R(φ)T = R(θ−φ)
  This is why context extension (YaRN, NTK) works:
  rescale the frequencies → stretch the position space.

Soft Lookup Table

Attention is a soft lookup table. Q is the search query, K is the index, V is the data. Unlike a hash table, every entry matches to some degree — softmax just decides how much.

Step 5: SwiGLU FFN

After attention, the FFN processes each token position independently. Modern models use SwiGLU instead of GPT-2's simple GELU:

Math
# SwiGLU: gated activation
gate = x @ W_gate    # [5120] → [17408]
up   = x @ W_up      # [5120] → [17408]
h    = SiLU(gate) ⊙ up   # element-wise: sigmoid(gate) × gate × up
out  = h @ W_down     # [17408] → [5120]

# SiLU(x) = x * sigmoid(x) — smooth version of ReLU
# The gate controls information flow: if gate ≈ 0, output ≈ 0
# This is why the FFN needs 3 matrices (not 2 like GPT-2)

# FLOPs for FFN: 3 × 5120 × 17408 × 2 = 535 MILLION per block per token
# The FFN is ~78% of the compute in each block.

Why Two Projections?

The gate learns which features matter. The up-projection computes candidate values. Multiplying them = selective routing. It's like an if-statement learned from data: if gate[i] > 0: output[i] = up[i], but smooth and differentiable. One projection decides, the other provides — that's why SwiGLU needs three matrices instead of GPT-2's two, and why it works better: the model can learn to shut off irrelevant features instead of blending everything.

We now know what the model is: 64 repetitions of norm → attention → norm → FFN, with residual connections. We know what each weight matrix does and its exact dimensions. Seven matrix multiplications per block. Sixty-four blocks. 44 billion FLOPs per token. For every single token the model generates, all of this runs. Every matrix. Every layer. Every time.

We can write the forward pass. We know the math. Qwen-3.5 27B is 54 GB in FP16 — our GPU has 24 GB. But quantize to Q4 (~15 GB) and it fits. Even so, those 44 billion FLOPs should take about 0.04 ms on an H100, but reading 15 GB of weights takes 4.5 ms. The computation isn't the bottleneck. The memory is. Let's understand why.

// 03 — Choosing Words

The model's forward pass produces a vector of logits — one raw score for every token in the vocabulary (32,000 to 128,000 values). Converting this into a single chosen token is the sampling pipeline's job. Temperature, top-p, top-k, min-p, repetition penalty — each is a filter applied to the logit distribution before sampling, and their order matters.

The Pipeline

The Design Insight

Self-contained + aligned = instant load. GGUF embeds the tokenizer, architecture params, and weights in one file. The tensor data starts at a 16-byte-aligned offset. This means mmap() can map the file directly — the OS's virtual memory system becomes your model loader. A 15 GB model "loads" in under a second because nothing is read until computation touches a page.

Safetensors (Hugging Face) is a multi-file format used by Python-based engines like vLLM. An 8-byte length prefix, then a JSON header describing tensor names/shapes/offsets, then raw tensor data. The tokenizer lives in separate files, and the weights are typically sharded across multiple files (model-00001-of-00004.safetensors) for parallel loading.

The First Problem

54 gigabytes at FP16. Your RTX 3090 has 24 GB of VRAM. It doesn't fit at full precision. But quantize to 4-bit (~15 GB) and it fits with room for a KV cache. What exactly does the forward pass do with these matrices?

// 05 — Two Phases

When a user sends a prompt, the engine processes all prompt tokens at once — a batch matrix multiply, compute-bound, GPU-friendly. This is prefill. When the engine starts generating the response, it produces one token at a time — a vector-matrix multiply, memory-bound, bandwidth-limited. This is decode. Same model, same weights, same code — but the performance profile is so different they might as well be separate workloads.

Chunked Prefill

Modern engines solve this with chunked prefill: split a long prompt into chunks (say, 1,024 tokens each) and interleave them with decode steps for other requests.

The Two-Phase Duality

Optimizing prefill (TTFT) often hurts decode (ITL) and vice versa. The scheduler's job is to balance them. 2025/2026 frontier: disaggregated serving separates prefill and decode onto different GPU pools, each optimized for its workload. Prefill GPUs run at high compute utilization; decode GPUs run at high memory bandwidth utilization.

// 06 — Remembering

Look back at our GPT-2 generate loop from Chapter 1. At step 50, we run the entire sequence of 50 tokens through all 12 layers. But tokens 1-49 haven't changed! Their queries, keys, and values are identical to what we computed at step 49. We're doing 49/50 = 98% redundant work. The KV cache fixes this: save each token's keys and values after computing them, and never recompute.

GQA: Shrinking the Cache by Design

Original transformers used one K head and one V head for every Q head. Qwen-3.5 27B has 24 query heads but only 4 KV heads — each KV head is shared across 6 query heads. This is Grouped-Query Attention (GQA), designed specifically to reduce KV cache memory:

GQA: fewer KV heads, shared across query groups 

  MHA (original):  Q₁ Q₂ Q₃ Q₄ Q₅ Q₆           6 KV heads → big cache
                   K₁ K₂ K₃ K₄ K₅ K₆
                   V₁ V₂ V₃ V₄ V₅ V₆

  GQA (Qwen-3.5 27B): Q₁ Q₂ Q₃ Q₄ Q₅ Q₆           1 KV head → 6× less cache
                   K₁ K₁ K₁ K₁ K₁ K₁
                   V₁ V₁ V₁ V₁ V₁ V₁

Without GQA, Qwen-3.5 27B's cache would be 6× larger: 96 GB for 128K context. With GQA it's 16 GB. Still significant, but manageable.

Ring Buffer: llama.cpp's Cache Strategy

In llama.cpp, the KV cache is a fixed-size ring buffer. When it fills, the oldest entries are overwritten. A head pointer tracks the next write position. RoPE adjustments are applied to the K cache when entries shift, maintaining position correctness. This avoids dynamic allocation entirely — no fragmentation, no malloc overhead, predictable memory usage.

// 08 — Making It Fit

The model is 54 GB in FP16 and the GPU is 24 GB. Something has to give. We could buy more GPUs — and we will, eventually — but first, there's a more elegant solution: store each weight in fewer bits. FP16 uses 16 bits per number. What if we used 4? That's ~15 GB for 27B. It fits on a 3090 with room for KV cache. The question is: do those 4-bit approximations still produce good language?

The answer, astonishingly, is yes. And the history of how we discovered this is a story of increasingly clever tricks, compressed into just three years.

History: The Quantization Revolution

2022, August: Tim Dettmers publishes LLM.int8() — the first paper showing you can quantize LLM weights to 8-bit with no quality loss, using mixed-precision decomposition for outlier features. This is the proof of concept.

2022, October: Elias Frantar publishes GPTQ — 3-4 bit quantization with error redistribution, one column at a time. The first practical method for running 175B models on consumer hardware.

2023, June: Ji Lin publishes AWQ (Activation-Aware Quantization) — protect the 1% of salient weights, quantize the rest. Simpler than GPTQ, better quality.

2023, March-onward: Georgi Gerganov and the llama.cpp community develop the K-quant family (Q4_K_M, Q5_K_M, etc.) — hand-tuned block quantization with superblocks and K-means optimization. The gold standard for local inference.

Q4_0: The Simplest Quantization

Take a block of 32 consecutive weights. Find the largest absolute value — that's your scale factor d. Divide every weight by d, round to the nearest integer in [-8, +7], and store the result in 4 bits. Two weights per byte. One FP16 scale per 32 weights.

C
typedef struct {
    ggml_fp16_t d;       // scale factor (2 bytes)
    uint8_t qs[16];      // 32 weights packed into 16 bytes
} block_q4_0;           // 17 bytes for 32 weights = 4.5 bits/weight

// Dequantize:
for (int i = 0; i < 16; i++) {
    int lo = qs[i] & 0x0F;        // low nibble: 0..15
    int hi = qs[i] >> 4;          // high nibble: 0..15
    out[2*i]     = d * (lo - 8);  // center at zero: -8 to +7
    out[2*i + 1] = d * (hi - 8);
}

GPTQ: Quantizing with Error Redistribution

GPTQ quantizes one column of a weight matrix at a time, using a small calibration dataset (128 random 2048-token segments from C4) to measure how each decision affects the output:

Algorithm (one row)
# W: original weights [d_out, d_in]
# H: Hessian (X^T X) — captures weight importance via activations

for col in range(d_in):
    # 1. Quantize this column
    w_q = quantize(W[:, col])

    # 2. Compute the error
    err = (W[:, col] - w_q) / H[col, col]

    # 3. Redistribute error to ALL remaining columns
    W[:, col+1:] -= err.outer(H[col, col+1:])

# Key insight: error doesn't accumulate — it gets PUSHED FORWARD
# into columns that haven't been quantized yet.
# A 175B model quantizes in ~4 GPU-hours.

It identifies these by analyzing activation magnitudes, scales them up to increase their quantization resolution, and scales the activations down to compensate. Mathematically equivalent, but the important weights now occupy more of the quantization range.

// 09 — Fast Attention

2022. Tri Dao, a PhD student at Stanford, publishes a paper that changes everything. FlashAttention doesn't change the math — the output is bit-for-bit identical to standard attention. It changes where the math happens. By computing attention in small tiles that fit in the GPU's fast on-chip SRAM (20 MB, ~30 TB/s) instead of the slow HBM (80 GB, 3.35 TB/s), it avoids ever materializing the full N×N attention matrix.

Standard Attention: The Problem

Standard attention memory traffic
# For 8K context, one head, FP16:
# S = Q @ K^T  →  write [8192 × 8192] = 128 MB to HBM
# P = softmax(S)  →  read 128 MB, write 128 MB
# O = P @ V  →  read 128 MB
# Total: 512 MB of HBM traffic for ONE head of ONE layer
# × 24 heads × 64 layers = 786 GB of memory traffic (for 27B)
# At 3.35 TB/s: 776 ms just for attention. Absurd.

# For 32K context: matrix is 32K×32K = 2 GB per head.
# With 24 heads: 48 GB just for the score matrix. Doesn't fit.

FlashAttention Tiling: Step by Step

The hard part is the softmax. Standard softmax needs max(x) across the entire row, but if you're processing tiles you don't see the whole row. The online softmax maintains running statistics that get corrected as new tiles arrive:

FlashAttention (tile_size=4, seq_len=8, one head)
# Q tiles: Q[0:4], Q[4:8]  (2 query tiles)
# K,V tiles: K[0:4],V[0:4] and K[4:8],V[4:8]  (2 KV tiles)

═══ Processing Q[0:4] against all KV tiles ═══

Tile 1: Q[0:4] × K[0:4]^T in SRAM
  S₁ = Q[0:4] @ K[0:4]^T / √d_head     # [4×4] score matrix
  m₁ = rowmax(S₁)                        # running max per row
  P₁ = exp(S₁ - m₁)                     # numerator of softmax
  l₁ = rowsum(P₁)                        # running sum
  O₁ = P₁ @ V[0:4]                      # partial output

Tile 2: Q[0:4] × K[4:8]^T in SRAM
  S₂ = Q[0:4] @ K[4:8]^T / √d_head
  m₂ = rowmax(S₂)
  m_new = max(m₁, m₂)                    # update global max

  # Correction: rescale old accumulators to new max
  correction = exp(m₁ - m_new)
  l_new = correction * l₁ + rowsum(exp(S₂ - m_new))
  O_new = correction * O₁ + exp(S₂ - m_new) @ V[4:8]

Final: O = O_new / l_new
# This is EXACT. Not an approximation. Bit-for-bit identical.
# The S matrix never existed in HBM. Saved: 128 MB per head.

FlashDecoding: Parallelizing Over KV Length

During decode, the query is a single vector but the KV cache can be enormous. Standard FlashAttention parallelizes over query tiles — useless when there's only one query. FlashDecoding instead splits the KV cache into P partitions, computes partial attention in parallel, then reduces:

FlashDecoding
# Q: [1, d_head] (single decode token)
# KV cache: [seq_len, d_head] where seq_len could be 100K+

# Split KV into P=16 partitions
parallel for p in 0..P:
    partial_out[p], lse[p] = flash_attn(Q, K[p*chunk:(p+1)*chunk], V[...])

# Reduce: combine using log-sum-exp values for numerically stable merge
# Same online softmax trick — correction factors from max/sum
O = reduce_with_lse(partial_out, lse)
Impact

llama.cpp: 11 → 32 tok/s on M3 Max after FlashAttention. vLLM: 2-3× throughput on long contexts. FA2 on Hopper: 20-50% higher FLOPS/s vs Ampere. The math is identical — it's the same computation. The improvement comes entirely from doing that math in fast SRAM instead of slow HBM. FlashAttention didn't speed up attention. It removed the memory bottleneck that was hiding how fast attention already was.

We now have a working, efficient single-user engine. But "one user" is a toy. A real service handles hundreds of concurrent requests. And there's a wrinkle we've been ignoring: the model processes the prompt and generates tokens through two completely different computational regimes.

// 10 — Many Users

Batching is the escape hatch from the memory wall. Process one request: you read 15 GB of weights to produce one token. Process 32 requests simultaneously: you read 15 GB of weights to produce thirty-two tokens. Same bytes loaded, 32× the useful work. The GPU's wasted compute capacity finally gets used.

This insight has a history. In June 2022, researchers at UC Berkeley published the ORCA paper (OSDI 2022), introducing continuous batching. Before ORCA, inference engines used static batching: collect N requests, pad to same length, wait for the longest to finish. ORCA made scheduling decisions at every decode step. The result: 23× throughput improvement.

Continuous Batching: Five Scheduling Steps

Walkthrough
# 3 slots, requests arrive at different times

═══ Iteration 1 (t=0) ═══

  Action: Admit A and B. Prefill both.
  Slots: [A:prefill, B:prefill, empty]

═══ Iteration 2 (t=1) ═══

═══ Iteration 3 (t=2) ═══

  Slots: [A:decode, B:decode, C:decode]  ← full batch, optimal

═══ Iteration 4 (t=3) ═══
  B hits EOS. Finished! D arrives.
  Action: Free B's slot immediately. Admit D, prefill D.

═══ Iteration 5 (t=4) ═══
  D prefill done.
  Slots: [A:decode, D:decode, C:decode]  ← full batch again

# Slots are reclaimed the instant a request completes.

Batch Scheduler Simulation

But batching 32 requests means 32 separate KV caches. The memory crisis is real. Traditional engines pre-allocate KV cache for the maximum possible sequence length. On average, 60-80% of allocated memory is wasted. We need virtual memory.

// 11 — Virtual Memory for Attention

In 1961, the Atlas computer at the University of Manchester introduced virtual memory. Sixty-two years later, in June 2023, the vLLM team at UC Berkeley applied the same idea to KV cache management, and it was just as revolutionary the second time around.

PagedAttention divides KV cache memory into fixed-size blocks (typically 16 tokens each). Each request gets a block table that maps logical token positions to physical blocks. Blocks are allocated on demand as the sequence grows. When a request finishes, its blocks are freed immediately.

PagedAttention block allocator

  Req A: [P7] [P2] [P11]         3 blocks, 48 tokens used
  Req B: [P1] [P5] [P3] [P9]    4 blocks, 64 tokens used
  Req C: [P4]                      1 block, 10 tokens used

  Physical GPU Memory:
  ┌────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐
  │ P1 │ P2 │ P3 │ P4 │ P5 │FREE│ P7 │FREE│ P9 │FREE│ P11│FREE│
  └────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┴────┘

The Paged Attention Kernel

Pseudocode
def paged_attention_kernel(query, block_table, k_cache, v_cache):
    # query: [1, n_heads, head_dim] — single decode token
    # block_table: [max_blocks] — maps logical→physical block indices
    # k_cache, v_cache: [n_physical_blocks, block_size, n_kv_heads, head_dim]

    output = zeros(n_heads, head_dim)
    m = -inf; l = 0    # online softmax accumulators

    for logical_block in range(num_used_blocks):
        phys_idx = block_table[logical_block]   # indirection!
        k_block = k_cache[phys_idx]             # [block_size, head_dim]
        v_block = v_cache[phys_idx]

        scores = query @ k_block.T / sqrt(head_dim)
        m_new = max(m, max(scores))
        correction = exp(m - m_new)
        l = correction * l + sum(exp(scores - m_new))
        output = correction * output + exp(scores - m_new) @ v_block
        m = m_new

    return output / l

# The block_table indirection is the key: physical memory is scattered,
# but the kernel traverses it through the page table. Just like an OS.

Prefix Sharing: Copy-on-Write

If 100 requests share the same system prompt, they share the same physical blocks for that prefix. Blocks are reference-counted and only copied when a request diverges:

Req B: [S1] [S2] [B1]           ← shares S1, S2
Req C: [S1] [S2] [C1] [C2] [C3]

2-4× throughput improvement, entirely from better memory utilization. No algorithmic change, no approximation, no quality loss. Just virtual memory. The lesson: sometimes the biggest wins come from systems engineering, not ML research.

// 12 — Going Wide

When one GPU isn't enough, split the model across multiple GPUs. Tensor parallelism splits individual matrix multiplications (all GPUs work on every layer together). Pipeline parallelism assigns entire layers to different GPUs (each GPU handles a portion of the depth).

Tensor Parallelism: Cut Horizontally

For attention, the split is natural: GPU 0 gets heads 0-11, GPU 1 gets heads 12-23. Each GPU computes attention independently, then an all-reduce combines results after the output projection.

Per transformer block with TP=2
1. Column-split Q,K,V projections → no communication
2. Each GPU computes attention for its heads
3. Row-split output projection → all-reduce (combine partial sums)
4. Column-split FFN gate+up → no communication
5. Row-split FFN down → all-reduce

Total: 2 all-reduces per block, 128 per forward pass (64 layers)
Each all-reduce sends ~16 KB (the hidden state vector)
On NVLink (450 GB/s): 5.7 μs per all-reduce → negligible
On PCIe 4.0 (32 GB/s): 80 μs per all-reduce → 12.8 ms total → noticeable

For YOUR 3090: NVLink bridge available (2-way only).
Two 3090s with NVLink: 27B Q4 runs faster with split, or 72B Q4 fits!
Two 3090s on PCIe: same model, ~25 tok/s (PCIe overhead).

We can now run models of any size across any number of GPUs. But we've hit a fundamental limit: the autoregressive bottleneck. No matter how many GPUs, each token must wait for the previous one. Can we break this sequential dependency?

// 13 — Cheating Time

Here is a beautiful paradox. The model can process 1,000 tokens in parallel during prefill but must generate them one at a time during decode. Speculative decoding (2023) exploits this asymmetry: use a small, fast draft model to guess the next K tokens, then verify all K guesses in a single forward pass of the target model.

Speculative decoding: draft, then verify 

  Draft Model (1B, fast): generate 5 guesses
  t₁ → t₂ → t₃ → t₄ → t₅

  Target Model (72B): verify ALL 5 in ONE forward pass
  [context, t₁, t₂, t₃, t₄, t₅] → logits for each position

  Compare: t₁ ✓  t₂ ✓  t₃ ✓  t₄ ✗  t₅ (discarded)
  Resample t₄ from adjusted target distribution.

  Result: 4 tokens from 1 target forward pass (3× speedup)

Rejection Sampling: The Math

Algorithm
# For each draft token t with draft probability q(t) and target probability p(t):

acceptance_prob = min(1, p(t) / q(t))

# If accepted: keep the token. Move to next.
# If rejected: resample from the ADJUSTED distribution:
#   p'(x) = max(0, p(x) - q(x)) / Z
# This guarantees the output distribution is EXACTLY p(x).
# Zero quality loss. Pure speed.

# Example with actual numbers:
# Draft says "Paris" with prob 0.7. Target says "Paris" with prob 0.85.
# Accept prob = min(1, 0.85/0.7) = 1.0 → always accepted.

# Draft says "London" with prob 0.3. Target says "London" with prob 0.05.
# Accept prob = min(1, 0.05/0.3) = 0.167 → rejected 83% of the time.

The draft model can be: a smaller model from the same family (Qwen-3.5 0.6B for Qwen-3.5 27B, ~70% acceptance, 2-3× speedup), an N-gram lookup from the prompt (free but lower acceptance), or specialized architectures like EAGLE (extrapolation from hidden states) and Medusa (parallel prediction heads).

When It Helps Most

Speculative decoding shines at batch=1 on your 3090 — the GPU is bandwidth-starved anyway, so verifying K tokens costs barely more than generating 1. At large batch sizes, the target model is already compute-efficient, so the benefit shrinks. Best on predictable tasks: code, translation, structured output.

import numpy as np, json, struct, os

def load_gpt2_weights(model_dir):
    """Load GPT-2 124M from safetensors format."""
    with open(os.path.join(model_dir, "model.safetensors"), "rb") as f:
        header_len = struct.unpack("<Q", f.read(8))[0]
        header = json.loads(f.read(header_len))
        data_start = 8 + header_len
        tensors = {}
        for name, info in header.items():
            if name == "__metadata__": continue
            dtype_map = {"F16": np.float16, "F32": np.float32}
            dtype = dtype_map[info["dtype"]]
            start, end = info["data_offsets"]
            f.seek(data_start + start)
            tensors[name] = np.frombuffer(
                f.read(end - start), dtype=dtype
            ).reshape(info["shape"]).astype(np.float32)
    return tensors

def attention_with_cache(x, w_qkv, w_proj, b_qkv, b_proj, n_heads, cache, layer, pos):
    B, D = x.shape
    qkv = x @ w_qkv + b_qkv
    q, k, v = np.split(qkv, 3, axis=-1)

    # Store K, V in cache (never recompute)
    cache['k'][layer][pos:pos+B] = k  # [1, 768] into slot
    cache['v'][layer][pos:pos+B] = v

    # Attend to ALL cached K, V (including current)
    keys = cache['k'][layer][:pos+B]   # [pos+1, 768]
    vals = cache['v'][layer][:pos+B]

    head_dim = D // n_heads
    q = q.reshape(B, n_heads, head_dim).transpose(1,0,2)
    k = keys.reshape(-1, n_heads, head_dim).transpose(1,0,2)
    v = vals.reshape(-1, n_heads, head_dim).transpose(1,0,2)

    scores = (q @ k.transpose(0,2,1)) / np.sqrt(head_dim)
    mask = np.triu(np.full((B, pos+B), -1e9), k=pos+1)
    scores += mask[..., :scores.shape[-1]]
    attn = softmax(scores)
    out = (attn @ v).transpose(1,0,2).reshape(B, D)
    return out @ w_proj + b_proj

// 18 — The Timeline

The history of inference optimization is surprisingly compressed. The field barely existed before 2022. In three years it went from "nobody cares how fast inference is" to "inference efficiency determines who can compete in AI." Here's the timeline.

The compression is remarkable: from "inference is trivial" (2019) to "a $30B industry built on inference optimization" (2025) in six years.
