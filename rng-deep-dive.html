RANDOMNESS

Nothing is random and everything is a lie._

// 01 — True Randomness Doesn't Exist (or does it?)

For most of human history, randomness was philosophical camouflage for ignorance. Flip a coin and the outcome feels random — but given perfect knowledge of the coin's mass, the force applied, air resistance, and surface friction, Newtonian mechanics predicts the result exactly. This is Laplace's demon: a hypothetical intellect that knows the position and momentum of every particle in the universe could compute every future event with perfect precision. In a classical universe, nothing is random. Everything is just unknown.

Then quantum mechanics showed up and ruined everything — or saved everything, depending on your disposition.

The Quantum Exception

In quantum mechanics, certain events are not merely unknown — they are fundamentally undetermined until observed. Radioactive decay: a single unstable nucleus will decay at some point, but there is no fact of the matter about when, even in principle. Photon polarization: pass a photon through a polarizing filter at 45° and there is a 50% chance it passes through. Not 50% because we don't know enough. 50% because the universe genuinely hasn't decided yet.

The Bell inequality experiments (Aspect, 1982; Hensen et al., 2015) essentially ruled out local hidden variable theories — the universe isn't secretly deterministic with information we're missing. The randomness in quantum events appears to be ontological: baked into reality itself, not just epistemological uncertainty on our part.

Two flavors of randomness 
  Epistemic randomness (classical):
    "I don't know which side the coin landed on."
    Cause: incomplete information
    Theoretically predictable with enough data
    Example: coin flip, dice roll, shuffled deck

  Ontological randomness (quantum):
    "The universe genuinely hasn't decided yet."
    Cause: fundamental indeterminacy
    NOT predictable even in principle
    Example: radioactive decay, photon path, vacuum fluctuations

  Practical implication for computers:
    Both look identical from the outside.
    You cannot tell by examining a bit whether it was
    epistemically or ontologically generated.
The Academic Distinction

Whether randomness is "truly" ontological or merely the limit of epistemic access doesn't matter one whit to your program. What matters is: can an adversary predict the next value? If yes, it's not random enough. If no, you're fine. Philosophy is a luxury; security is not.

// 02 — Pseudorandom — Faking It

Every random number your program has ever generated was a lie. A deterministic function took a starting value — the seed — and produced a sequence of numbers that merely looked random to casual inspection. Feed the same seed back in and you get the exact same sequence, byte for byte, forever. There's nothing random about it. It's arithmetic wearing a disguise.

This is, counterintuitively, often exactly what you want.

Why Determinism Is a Feature

Reproducibility is gold. A game that uses a seeded PRNG generates the same world every time from the same seed — that's how Minecraft's world generation works, how No Man's Sky synthesizes an entire galaxy, how procedural level generators create millions of unique-but-reproducible layouts. Your test suite that shuffles data reproducibly can catch bugs that a truly random shuffle would hide. A simulation seeded to a known value can be replayed exactly to find where things went wrong.

The seed is the entire state of the PRNG. Change the seed, change everything. Fix the seed, fix everything. This is profound leverage.

#include <stdio.h>
#include <stdlib.h>

int main() {
    /* srand seeds the global state; rand() reads from it */
    srand(42);
    for (int i = 0; i < 5; i++) {
        printf("%d\n", rand());
    }

    /* Run again with same seed — identical output */
    srand(42);
    for (int i = 0; i < 5; i++) {
        printf("%d\n", rand());  // same numbers again
    }
    return 0;
}

// TypeScript / JavaScript has NO built-in seeded PRNG.
// Math.random() is a black box — you cannot seed it.
// You must bring your own algorithm.

// Mulberry32 — a fast, seedable 32-bit PRNG
function mulberry32(seed: number) {
  return function(): number {
    seed |= 0;
    seed = seed + 0x6D2B79F5 | 0;
    let t = Math.imul(seed ^ seed >>> 15, 1 | seed);
    t = t + Math.imul(t ^ t >>> 7, 61 | t) ^ t;
    return ((t ^ t >>> 14) >>> 0) / 4294967296;
  };
}

console.log(rng(), rng(), rng()); // reproducible every time

// 03 — Linear Congruential Generators

The oldest trick in the PRNG book, and still lurking everywhere you look. Linear Congruential Generators (LCGs) are comically simple: take the current value, multiply by a constant, add another constant, take the remainder after dividing by a third constant. That's it. One line of arithmetic that has shipped in more software than any other random number algorithm in history — including the rand() function on your system right now.

The formula is almost insultingly minimal:

LCG Formula 
  X(n+1) = (a * X(n) + c) mod m

  Where:
    X(n)   — current state (the "seed" at step n)
    a      — multiplier      (must be chosen carefully)
    c      — increment       (can be 0 for "multiplicative" LCGs)
    m      — modulus         (often a power of 2 for cheap arithmetic)
    X(n+1) — next state / next output

  Example step with glibc params (a=1103515245, c=12345, m=2^31):
    X(0) = 1
    X(1) = (1103515245 * 1 + 12345) mod 2147483648
         = 1103527590 mod 2147483648
         = 1103527590

    X(2) = (1103515245 * 1103527590 + 12345) mod 2147483648
         = ...
         = 2524785265 mod 2147483648     (wraps around)
         = 377301617

Famous LCG Parameters

The choice of a, c, and m matters enormously. A good set produces a full-period sequence covering every value from 0 to m−1 exactly once before repeating. A bad set produces short cycles, obvious patterns, or catastrophically poor distribution.

The Lattice Problem

In 1968, George Marsaglia published a paper with a devastating observation: k-tuples of consecutive LCG outputs, when plotted in k-dimensional space, all fall on a small number of parallel hyperplanes. For 3D output, you get three planes. For higher dimensions, even fewer. The sequence isn't random at all — it has geometric structure visible to anyone who looks.

Marsaglia's Hyperplane Theorem (1968) 
  If (X1, X2, X3, ..., Xk) are consecutive LCG outputs,
  then ALL k-tuples lie on at most m^(1/k) parallel hyperplanes.

  For glibc LCG with m = 2^31:
    k=2:  at most 2^15.5 ≈ 46,341 lines      (looks OK in 2D)
    k=3:  at most 2^10.3 ≈ 1,290 planes      (visible structure in 3D)
    k=6:  at most 2^5.2  ≈ 37 hyperplanes    (catastrophically bad)

  This is why RANDU (IBM's notorious LCG) looked fine in 2D
  but clustered onto exactly 15 planes in 3D.
  Entire physics simulations were corrupted. Quietly.

This doesn't mean LCGs are useless — they're fast, simple, and fine for non-cryptographic work where you just need "looks random enough in 1D." But the moment you generate pairs or triples and use them as coordinates, you've introduced correlated structure. See the lattice structure yourself in the interactive demo below, and also consider how floating-point representation affects the distribution of generated values.

Never Use LCGs for Security

The entire state of an LCG IS the last output. One output, and you know the full state. Two outputs, and you can recover a, c, and m if you don't already know them. LCG output is completely predictable — forward and backward. Using an LCG to generate session tokens, passwords, nonces, or any security-sensitive value is a vulnerability, not a shortcut.

// 04 — Mersenne Twister — The Industry Default

In 1997, Makoto Matsumoto and Takuji Nishimura published a PRNG that made everything before it look embarrassing. The Mersenne Twister had a period of 219937−1 — a number so large it would take longer than the age of the universe to cycle through even if you generated a billion numbers per second. It passed statistical tests that shredded LCGs. It was fast. It was free. Within a decade it had become the default random number generator in Python, Ruby, PHP, R, MATLAB, and countless other systems. It is almost certainly the PRNG generating "random" numbers in your code right now.

The name comes from the Mersenne prime 219937−1. Mersenne primes (of the form 2p−1) have properties that make them useful for constructing PRNGs with guaranteed long periods.

How It Works

The Mersenne Twister maintains a state array of 624 32-bit integers (a total of 19,968 bits of state). On each generation step, it processes this array through two operations:

Mersenne Twister State Machine 
  State: MT[0..623]  — 624 × 32-bit words = 19,968 bits of state

  ┌─────────────────────────────────────────────────────────────┐
  │  MT[0]   MT[1]   MT[2]  ...  MT[396]  ...  MT[622]  MT[623] │
  └─────────────────────────────────────────────────────────────┘
          │                         │
          └─────────┬───────────────┘
                    │  TWIST OPERATION
                    │  (when state exhausted)
                    ▼
    MT[i] = MT[i+397] XOR twist(MT[i], MT[i+1])
    twist involves bitwise operations and matrix multiply over GF(2)

          │
          │  TEMPER OPERATION
          │  (every output)
          ▼
    y = MT[index]
    y ^= (y >> 11)
    y ^= (y << 7)  & 0x9D2C5680
    y ^= (y << 15) & 0xEFC60000
    y ^= (y >> 18)
    output = y

  Twist runs every 624 outputs (regenerates all state).
  Temper runs every output (improves bit distribution).
  Period: 2^19937 - 1.

Why It Won

The Mersenne Twister dominated for good reasons. Its period is astronomically large — no simulation you will ever run will approach its cycle length. It passes the stringent Diehard statistical tests that demolished LCGs. The 624-word state is large enough that simple lattice attacks fail. And the implementation is fast enough to use without a second thought.

Strengths

  • Period 219937−1 (astronomically large)
  • Passes Diehard battery of tests
  • 623-dimensional equidistribution
  • Fast generation (~2ns per number)
  • Widely implemented and tested

Weaknesses

  • NOT cryptographically secure
  • 624 outputs fully reconstruct state
  • Large state (2.5KB) — bad for cache
  • Fails some modern tests (TestU01 BigCrush)
  • Slow initialization from seed

The State Recovery Attack

Here is the devastating fact that makes Mersenne Twister completely unsuitable for security: the temper operation is invertible. Given 624 consecutive outputs, you can exactly reconstruct the internal state. From that point forward, you can predict every future output. Backward too, if you want to.

This isn't theoretical. Libraries exist to do it automatically. You collect outputs, invert the tempering, recover MT[0..623], clone the generator, and step it forward. Game over. Every "random" event from that point is predictable.

624 Outputs — That's All

An attacker who observes 624 consecutive Mersenne Twister outputs can reconstruct your entire PRNG state and predict every subsequent "random" number forever. If your application exposes MT outputs (dice rolls, session tokens, shuffle orders, map generation seeds...) an attacker can reverse-engineer them. This is not hypothetical — online poker sites have been cracked this way.

// 05 — Xorshift & Modern PRNGs

George Marsaglia (again!) introduced xorshift in 2003. The man simply could not stop publishing devastating papers about random number generators. Beautifully simple: three XOR-and-shift operations on an integer. No multiplication, no division, no lookup table — just the three most embarrassingly fast operations a CPU has. And yet the output passes statistical tests that would make a lesser PRNG weep.

The core insight: XOR and bit shifts are reversible. A nonzero state can never reach zero, so the period is guaranteed to be the full 232−1 for a 32-bit state. Three carefully chosen shifts are enough to ensure every bit influences every other bit within a few iterations — that is the entire theory.

xorshift32 — the whole algorithm 
  // State: a single 32-bit nonzero integer
  // Three magic constants chosen by Marsaglia for maximum period

  x ^= x << 13      // left shift creates high-bit dependencies
  x ^= x >> 17      // right shift creates low-bit dependencies
  x ^= x <<  5      // left shift mixes them together
  return x           // done. seriously.

  ──────────────────────────────────────────────────────
  Verification with seed = 0xDEADBEEF:

    State[0]:  0xDEADBEEF  (initial seed)
    State[1]:  0x42A5B2F9
    State[2]:  0xC7D3E1A4
    State[3]:  0x91B6F207
    ...
    After 2^32 - 1 steps:  back to 0xDEADBEEF
    Period: exactly 4,294,967,295  (never visits 0x00000000)

  Cost: 3 instructions. ~0.5ns on modern hardware.

The Modern Successors

Xorshift32 was a revelation, but it has weaknesses in the low bits and fails some of Marsaglia's own tests. The family tree grew rapidly as researchers found better shift constants, longer states, and a clever final mixing step called a "scrambler" that fixes the low-bit issues without sacrificing speed.

xoshiro256** (Blackman & Vigna, 2018): 256-bit state, "xor/shift/rotate" operations plus a double-star scrambler. The current recommendation for most non-cryptographic use. PCG (Melissa O'Neill, 2014): "Permuted Congruential Generator" — an LCG under the hood, but with a clever permutation applied to the output bits. Statisticians love it because it provably has good equidistribution. SplitMix64: used to initialize other generators from a 64-bit seed. Excellent avalanche properties.

Generator Period State Size Speed (ns/call) Quality Used By
MT19937 219937−1 2.5 KB ~4 ns Good Python, Ruby
xorshift128+ 2128−1 16 B ~1 ns Good V8, SpiderMonkey
xoshiro256** 2256−1 32 B ~1 ns Excellent Newer libs
PCG-64 2128 16 B ~1 ns Excellent NumPy, Rust

Note what happened to the state size: MT19937 needs 2.5 kilobytes of state — nearly a full cache line cluster — while xorshift128+ and PCG-64 fit comfortably in two registers. Modern PRNGs are faster and better. This is not normal in computer science.

Implementation: xorshift32 in Five Languages

Because nothing demonstrates the elegance like seeing it fit on a napkin in every language you use:

#include <stdint.h>

// State: any nonzero uint32. Global or per-thread.
static uint32_t xorshift_state = 0xDEADBEEF;

uint32_t xorshift32(void) {
    uint32_t x = xorshift_state;
    x ^= x << 13;
    x ^= x >> 17;
    x ^= x <<  5;
    return xorshift_state = x;
}

// That's it. That's the whole thing.
// Three XORs, three shifts, one assignment.
// Compiles to ~6 instructions. Period: 2^32-1.
C
# Python integers are arbitrary precision,
# so we must mask to 32 bits manually.

class Xorshift32:
    def __init__(self, seed: int = 0xDEADBEEF):
        assert seed != 0, "seed must be nonzero"
        self.state = seed & 0xFFFFFFFF

    def next(self) -> int:
        x = self.state
        x ^= (x << 13) & 0xFFFFFFFF
        x ^= (x >> 17)
        x ^= (x <<  5) & 0xFFFFFFFF
        self.state = x
        return x

rng = Xorshift32(42)
print([rng.next() for _ in range(5)])
Python
// TypeScript: use >>> for unsigned right shift
// JavaScript bitwise ops work on signed 32-bit ints,
// so >>> forces unsigned interpretation.

function makeXorshift32(seed: number = 0xDEADBEEF) {
  if (seed === 0) throw new Error("seed must be nonzero");
  let x = seed >>> 0;   // coerce to uint32

  return function next(): number {
    x ^= x << 13;
    x ^= x >>> 17;
    x ^= x <<  5;
    return x >>> 0;     // always return positive
  };
}

const rng = makeXorshift32(42);

// 06 — The OS Entropy Pool

Where does the seed come from? This is the question that separates "interesting algorithm" from "actually secure." A PRNG seeded with a predictable value is just an elaborate way to generate a deterministic sequence — which is exactly what you do not want for cryptography. The OS gathers entropy from the physical world: interrupt timing, disk seek times, mouse movements, keyboard press timing, network packet arrival jitter. Tiny, unpredictable variations in the physical universe, accumulated over time, fed into a pool of bits that nobody can predict.

Linux: Three Ways to Get Randomness

Linux has three interfaces, each with a different trade-off between security paranoia and pragmatic usability:

Linux entropy interfaces — a brief history of overengineering 
  /dev/random   — historically: blocks when "entropy runs out"
                  Theory: if we haven't gathered enough physical entropy,
                  we refuse to give you more random bytes.
                  Practice: servers with no keyboard/mouse routinely
                  hang at boot waiting for entropy. Classic.

  /dev/urandom  — non-blocking, always returns bytes
                  Uses the CSPRNG, which is computationally secure
                  even after the initial seed. Never blocks.
                  The actually-sane choice for 99% of use cases.

  getrandom()   — Linux 3.17+ syscall (2014)
                  Blocks until initial pool is seeded (good!),
                  then never blocks again (also good!).
                  Not subject to /dev file descriptor limits.
                  The correct modern choice.

  ──────────────────────────────────────────────────────────────
  The entropy debate: old Linux /dev/random blocked when the
  kernel's internal entropy counter hit zero. Modern kernels
  (5.18+, 2022) consider this unnecessary — a CSPRNG seeded
  once is computationally indistinguishable from true entropy.
  The counter is now cosmetic. The blocking is gone.

Other Operating Systems

Windows: BCryptGenRandom from bcrypt.dll. Seeded from CPU timing, hardware RNG (RDRAND if available), and various system state. The old CryptGenRandom is deprecated but still works. macOS/iOS: /dev/urandom backed by the Fortuna CSPRNG, plus the SecRandomCopyBytes API. All modern BSDs: arc4random() — automatic, never-failing, always-secure. No seeding required. The correct API for the userspace programmer who just wants things to work.

How Entropy Gets Into the Pool

entropy flow — from the physical world to your application 
  Physical world
    │
    ├─ CPU interrupt timing (jiffies between IRQs)
    ├─ Disk seek completion times (nanosecond variations)
    ├─ Network packet arrival timestamps
    ├─ Mouse/keyboard event timing
    ├─ Hardware RNG (RDRAND, if present)
    └─ Boot-time measurements (UEFI, TPM)
    │
    ▼
  Interrupt handlers
    │  Each interrupt: mix timing bits into the pool
    │  add_interrupt_randomness() / add_device_randomness()
    ▼

    │  Input: hashed with BLAKE2s
    │  State: continuously updated
    ▼
  CSPRNG (ChaCha20-based, since Linux 5.17)
    │  Generates output stream from pool key
    │  Periodically reseeded from pool
    ▼
  /dev/urandom / getrandom() / /dev/random
    │
    ▼
  Your application

Getting OS Randomness in Your Code

#include <sys/random.h>
#include <stdint.h>
#include <stdio.h>

int main(void) {
    uint8_t buf[32];

    // getrandom() — the modern, correct way (Linux 3.17+)
    // flags=0: block until seeded, then never block again
    ssize_t n = getrandom(buf, sizeof(buf), 0);
    if (n != sizeof(buf)) {
        perror("getrandom");
        return 1;
    }

    printf("32 random bytes: ");
    for (int i = 0; i < 32; i++)
        printf("%02x", buf[i]);
    printf("\n");
    return 0;
    // Note: no file descriptor, no /dev path,
    // not subject to fd limits in containers.
}

// 07 — Hardware RNG

Why not skip software entirely and sample physical randomness? The thermal noise in a transistor, the radioactive decay of an atom, the chaotic path of a photon through a beam splitter — these are genuinely random events, not computable from any prior state. Quantum mechanics guarantees it. If you can measure these events fast enough and feed them into your computer, you get randomness that no algorithm can predict, because no algorithm governs it.

RDRAND and RDSEED: CPU-Level Randomness

Intel introduced the RDRAND instruction with Ivy Bridge (2012), later adding RDSEED. The chip samples thermal noise from on-chip circuits — tiny, unpredictable voltage fluctuations — conditions the signal through a CSPRNG, and returns a random value in a single instruction. AMD added equivalent support shortly after.

RDRAND pipeline — from thermal noise to register 
  Thermal noise source (on-chip analog circuit)
    │  Random voltage fluctuations ≈ 800 MHz sampling rate
    ▼
  Analog-to-digital converter
    │  Digitizes noise into raw random bits
    ▼
  Conditioning (AES-based whitening)
    │  Removes bias, ensures uniform distribution
    ▼
  FIFO buffer
    │  Holds conditioned bits ready for instruction
    ▼
  RDRAND instruction
    │  Returns 16/32/64 bits in ~80 clock cycles
    │  Throughput: ~500 MB/s of random data
    ▼
  General-purpose register (rax, etc.)

  RDSEED: returns raw entropy bits (before CSPRNG conditioning)
          Slower, but directly seeds other generators

The catch, of course, is trust. You are relying on Intel or AMD to have implemented this correctly and honestly. After the Snowden revelations, security researchers became very interested in whether RDRAND might have a backdoor — a subtle bias that the manufacturer could exploit. The instruction is a black box. You cannot audit the silicon. The consensus position among cryptographers: use RDRAND as one entropy source among several, never as the sole source. Linux mixes it with other entropy; do not let it replace everything else.

The Lava Lamp Wall

Cloudflare's famous wall of 100 lava lamps in their San Francisco lobby is not a decoration — it is an entropy source. A camera photographs them continuously, hashes the pixel data, and feeds the result into their entropy pool. The blobs of wax, floating in their unpredictable fluid-dynamics ballet, contribute to the randomness protecting roughly 10% of the internet's encrypted HTTPS traffic. If someone puts a curtain in front of those lamps, the internet gets marginally less random. This is real infrastructure.

Other Hardware Sources

Radioactive decay: Genuinely quantum-random. A radioactive source will emit a particle at a time governed by quantum mechanics — not probability in the statistical sense, but fundamental indeterminacy. Some Hardware Security Modules (HSMs) use a small radioactive source plus a Geiger counter to generate true random numbers. The decay events cannot be predicted by any computation. Physics wins.

Atmospheric noise: random.org captures radio static from the atmosphere using radio receivers and converts it to random numbers. Output rate: approximately 10,000 bits per second. Used by lottery systems, scientific simulations, and anyone who wants to argue that their random numbers came from the universe rather than an algorithm. The latency is not great for high-speed applications.

Photon beam splitters: Quantum random number generators from companies like ID Quantique send single photons at a 50/50 beam splitter. Which path the photon takes is determined by quantum mechanics alone. Gigabit-per-second output. Used in cryptographic key generation for high-security applications. Extremely expensive. Also the coolest way to generate random numbers if you have the budget.

Unconventional Entropy

One security researcher's home entropy source: a webcam pointed at a fish tank. The fish never stop moving, the water constantly refracts light differently, and the resulting pixel stream provides a surprisingly good entropy feed when hashed. The fish are unaware of their cryptographic responsibilities and remain unpredictable regardless.

// 08 — Cryptographic RNG — When Randomness is Life or Death

A CSPRNG adds one critical property beyond statistical quality: forward secrecy and backtrack resistance. Even if an attacker sees all previous outputs — every random number you have ever generated — they cannot predict the next one. And even if they somehow see the current internal state, they cannot reconstruct past outputs. No amount of computation helps, because the security is not based on statistical obscurity but on computational hardness. Breaking it requires solving problems believed to require exponential time.

ChaCha20-Based CSPRNG

Linux's CSPRNG since kernel 5.17 (2022) uses ChaCha20, the same stream cipher used in TLS 1.3. Fast, side-channel resistant, well-analyzed by the cryptographic community. The key is derived from the entropy pool; output is a ChaCha20 keystream. Each CPU core gets its own ChaCha20 state to avoid lock contention. The pool is periodically reseeded from fresh hardware entropy.

AES-CTR-DRBG

The NIST SP 800-90A standard specifies Deterministic Random Bit Generators. AES-CTR-DRBG uses AES in counter mode: encrypt an incrementing counter under a secret key, use the output as random bytes. NIST-certified, used in many HSMs and government-approved implementations. Also the site of the most infamous CSPRNG scandal in history — but we will get to that.

The Horror Stories

Nothing illustrates the importance of correct CSPRNG implementation like the catastrophic failures. These are real events, with real financial and security consequences, caused by implementation mistakes that seem embarrassingly simple in retrospect.

Debian OpenSSL (2008)

A Debian developer was auditing OpenSSL with Valgrind, a memory error detector. Valgrind flagged two lines in the entropy-gathering code as "reading uninitialized memory" — a pattern that looks like a bug. The developer commented out those lines to silence the warning. Unfortunately, those lines were intentional: they fed process memory contents (genuinely unpredictable data) into the entropy pool.

The result: for almost two years, every SSL/TLS key generated on Debian or Ubuntu had an effective entropy of approximately 15 bits. There were only about 32,768 possible keys. Researchers built lookup tables. Every SSH key, every SSL certificate generated on an affected system was trivially breakable. One commented-out line. Two years. Every Debian server on the internet.

The Debian OpenSSL Bug

One line of code, commented out, made every SSL certificate generated on Debian for two years trivially breakable. The developer thought the code was a Valgrind false positive. It was not. The affected window: September 2006 to May 2008. All keys generated in that period had ~32,768 possible values. An attacker could brute-force them on a laptop.

Sony PlayStation 3 (2010)

The PS3 used ECDSA (Elliptic Curve Digital Signature Algorithm) to sign firmware. ECDSA requires a cryptographically random nonce k for every signature. Sony used a constant value. The same k, every time, for every signature.

ECDSA with constant k — the algebra that broke a console 
  ECDSA signature equation:
    s = k_inverse * (hash + private_key * r)  mod n

  Given two signatures (s1, r1) and (s2, r2) from the same k:
    s1 = k_inv * (h1 + priv * r1)
    s2 = k_inv * (h2 + priv * r2)

  Since r1 == r2 (same k → same r), solve for k:
    s1 - s2 = k_inv * (h1 - h2)
    k = (h1 - h2) / (s1 - s2)   ← trivial algebra

  Once k is known, private_key = (s * k - hash) / r

  Result: complete console compromise.
          Every game could be copied, homebrew ran freely.
          The entire security model, destroyed by k = constant.

Android SecureRandom (2013)

Certain Android devices had a bug in the Java SecureRandom implementation: insufficient seeding during PRNG initialization. Some Bitcoin wallet applications running on these devices generated Bitcoin private keys with predictable values. Attackers scanned the blockchain for signatures with repeated nonces (same weakness as the PS3), identified the vulnerable keys, and drained the wallets. Real money, gone, because of PRNG seeding failure.

Dual_EC_DRBG (2006–2014)

The most troubling entry. NIST standardized "Dual Elliptic Curve Deterministic Random Bit Generator" in 2006. The algorithm uses two elliptic curve points, P and Q, whose relationship determines the security. If someone knows the discrete logarithm of Q with respect to P — that is, if they know a value e such that Q = e*P — they can predict all outputs from a short observation window. Nobody knows why those specific P and Q constants were chosen. Nobody has ever explained the selection criteria. The NSA was heavily involved in the standardization process. In 2013, Snowden documents revealed that the NSA had paid RSA Security $10 million to make Dual_EC_DRBG the default in their BSAFE library. NIST withdrew the standard in 2014. The mathematical backdoor, if it exists, has never been confirmed or denied because confirming it would require disclosing classified capability.

Dual_EC_DRBG

A NIST-standardized CSPRNG with constants that may have been chosen by the NSA to allow decryption of traffic secured with it. The algorithm was mysteriously slow, had a known theoretical weakness if someone knew the relationship between its constants, and was promoted aggressively despite two independent cryptographers publicly noting the potential backdoor in 2007. It was withdrawn in 2014. The constants were never explained.

// 09 — Distribution Shaping

PRNGs give you uniform random numbers — every value between 0 and 1 equally likely, a perfectly flat histogram, the statistical equivalent of a beige wall. But reality is nothing like that. Heights follow a bell curve. Time between earthquakes follows an exponential distribution. Particle decay is Poisson. If your simulation produces uniformly distributed everything, your simulation is lying about the universe.

The good news: one distribution is enough to rule them all. Given a uniform source, mathematicians have devised transforms to produce any distribution you need. The machinery is surprisingly elegant — or diabolically clever, depending on how much you like trigonometry.

Box-Muller Transform

George Box and Mervin Muller, 1958. Take two independent uniform randoms u1 and u2 on (0,1]. Apply the following incantation:

z0 = sqrt(-2 * ln(u1)) * cos(2π * u2)
z1 = sqrt(-2 * ln(u1)) * sin(2π * u2)
math — Box-Muller

You get back two standard normal random variables. The proof involves polar coordinates, the Jacobian, and more calculus than is strictly comfortable. The result is miraculous: two worthless flat randoms go in, two proper bell-curved gaussians come out. Box and Muller were pleased with themselves, and they had every right to be.

One practical note: u1 cannot be exactly 0 or you're computing ln(0) = -∞ and your program becomes a philosophy problem. Most implementations sample from (0,1] rather than [0,1]. The difference matters.

Rejection Sampling

When you don't have a neat closed-form transform, there's a blunter instrument: rejection sampling. Draw a point uniformly from a bounding box that contains your target distribution. If the point falls under the distribution's curve, keep it. If not, throw it away and try again.

It works. It's general. It's also wasteful in proportion to how much of the bounding box your distribution doesn't fill. For a standard normal truncated at ±3σ, you reject about 5% of samples. For a distribution with spiky tails in high dimensions, you might reject 99.9% of everything you generate. High-dimensional rejection sampling is how you teach computers to appreciate futility.

Inverse Transform Sampling

The most theoretically elegant approach: if you know the inverse of your distribution's cumulative distribution function (CDF), apply it directly to a uniform random. x = CDF⁻¹(u). The exponential distribution, for instance, has CDF inverse -ln(1-u) / λ. One multiplication, one log, done.

The catch: most distributions don't have a closed-form CDF inverse. The normal distribution's CDF involves the error function, which has no elementary antiderivative. You'd need a numerical approximation — which brings you back to Box-Muller in practice, because Box-Muller is faster than evaluating erfinv().

#include <math.h>
#include <stdlib.h>
#include <stdio.h>

// Box-Muller transform: produces gaussian random from two uniforms
// Returns two values via out_z0 and out_z1
void box_muller(double *out_z0, double *out_z1) {
    double u1, u2;
    // Avoid u1 = 0 to prevent log(0)
    do { u1 = rand() / (double)RAND_MAX; } while (u1 == 0.0);
    u2 = rand() / (double)RAND_MAX;

    double mag = sqrt(-2.0 * log(u1));
    *out_z0 = mag * cos(2.0 * M_PI * u2);
    *out_z1 = mag * sin(2.0 * M_PI * u2);
}

int main(void) {
    double z0, z1;
    box_muller(&z0, &z1);
    printf("z0 = %.6f, z1 = %.6f\n", z0, z1);
    return 0;
}

  let u1 = 0;

// 10 — Gaming the System

If randomness is actually deterministic — a pure mathematical function of an initial seed — then it's not random at all. It's a lock with a very large combination. And locks exist to be picked. Humans are, fundamentally, pattern-matching engines with gambling problems, and the intersection of those two properties has produced some spectacular exploits.

This is the heist movie section. Somewhere out there, right now, someone is counting frames in a video game or logging slot machine outcomes to a server in St. Petersburg, and they are winning.

Speedrun RNG Manipulation

Super Mario 64 has a global RNG state updated on every frame the game runs. Expert speedrunners don't just play the game fast — they manipulate the RNG state to land favorable outcomes. Certain star-collecting sequences require the star to spawn in a specific location, which requires a specific RNG value, which requires entering and exiting rooms in a precise order while waiting exact frame counts. A speedrun that looks like someone playing brilliantly is often someone executing a frame-perfect RNG manipulation chain they rehearsed for hundreds of hours.

Pokémon takes this further. Every encounter, every stat roll, every shiny Pokémon is determined by the RNG state at a precise moment. The Pokémon RNG abuse community reverse-engineered the exact PRNG algorithms used in each game — usually a simple LCG — and built tools to predict the future. To catch a specific legendary with perfect stats, you synchronize your DS clock to a known seed value, then press A at the exact millisecond a timer tells you. The Pokémon is predetermined; you're just recovering the hidden state. It's less like catching a wild animal and more like digging up a buried treasure whose location you already computed.

Poker Software Exploit (1999)

ASF Software's Texas Hold'em Poker for Windows seeded its shuffle algorithm with GetTickCount() — milliseconds since midnight. This seems reasonable until you count: there are only 86,400,000 milliseconds in a day. A standard 52-card deck has 52! ≈ 8×1067 possible shuffles. The software was using a keyhole to represent an entire galaxy of combinatorial space.

Security researchers Russ Sterns and Gib Budelman demonstrated that a fast computer could enumerate every possible deck in the time between the shuffle request and the first card being dealt — roughly 200ms. In practice, you could see two or three of your hole cards, then brute-force the remaining deck from there. The "random" poker game was deterministic, and the house edge was irrelevant once you knew the algorithm.

Slot Machine Reverse Engineering

In 2011, a Russian group led by a man known as "Alex" cracked Aristocrat Mark VI slot machines in casinos across the United States and Europe. The attack was methodological: associates recorded video of slot machine outcomes on phones, uploaded the recordings to servers in St. Petersburg, where analysts reverse-engineered the PRNG state from the sequence of visible results. Once they knew the internal state, they could predict future outcomes. The analysts then sent timing signals back — vibrate the phone, tap the screen — telling the player exactly when to press the spin button to land a jackpot.

The scheme worked because modern slot machines use pseudorandom number generators that cycle through a fixed sequence. Record enough consecutive outcomes, solve for the seed, and the future is a lookup table. Aristocrat patched the vulnerability in subsequent firmware. The Russian group reportedly made millions before being caught. The math was unimpeachable. The jurisdiction was complicated.

Vietnam Draft Lottery (1969)

On December 1, 1969, the United States conducted a draft lottery for men born between 1944 and 1950. Birthdays were written on capsules, placed in a box, and drawn one by one to determine draft order. The first birthday drawn would mean that cohort was drafted first.

The problem: the capsules were loaded into the box by month, January first, December last. They were stirred — but not shuffled. Not really. Statistical analysis of the results showed December birthdates clustering disproportionately among the low numbers (drafted early). January birthdates clustered among the high numbers. The stirring was insufficient to overcome the sequential loading order.

Roughly 850,000 men had their lives materially altered by a clerical decision about which month to pour capsules in first, combined with a failure to understand that "stirring" is not a substitute for a random permutation. Donald Knuth, writing about this event years later, noted that it was a textbook demonstration of how not to shuffle. The Fisher-Yates algorithm, which would have solved the problem in 366 iterations, had been published in 1938. It sat, unused, in a statistics textbook while a jar was stirred by hand.

// 11 — Monte Carlo Methods

Stanislaw Ulam, 1946. He was recovering from encephalitis caused by a bad vaccine — brain surgery, long recovery, playing solitaire to pass the time. He started wondering about the probability of winning a game of Canfield solitaire. Tried to calculate it combinatorially. The branching possibilities were intractable. Then the thought: why not just play the game thousands of times and count? The fraction of wins is the probability. He told John von Neumann. Von Neumann immediately saw the implications for the neutron diffusion problems they were working on at Los Alamos. Monte Carlo simulation was born.

The name came from the casino in Monaco — the connection being "chance." Nicholas Metropolis suggested it. Ulam didn't object. The method is now foundational to physics, finance, medicine, engineering, and computer graphics, which is a remarkable trajectory for a man playing solitaire while his brain healed.

Estimating π

The canonical demonstration: draw a unit square. Inside it, inscribe a circle of radius 0.5, centered at (0.5, 0.5). The circle's area is π × 0.5² = π/4. The square's area is 1. Throw random darts at the square. Count how many land inside the circle. The ratio times 4 converges to π.

It's a terrible way to compute π — you need millions of samples to get five decimal places, and there are polynomial-time algorithms that give you a trillion digits. But as a demonstration of the Monte Carlo principle, it's perfect: replace hard geometry with random sampling, and the answer emerges from the noise.

Ray Tracing: Every Pixel is a Monte Carlo

Every Pixar film, every modern game with path tracing, every photorealistic render you've ever seen: Monte Carlo all the way down. Simulate a photon leaving a light source. At each surface, randomly choose a direction weighted by the bidirectional reflectance distribution function. Bounce it around the scene. When it hits the camera sensor, record the contribution. Repeat for millions of photons per pixel. Average them. The integral of light transport is unsolvable analytically for complex scenes — but tractable by sampling.

The random noise you see in a game running at low ray-count settings? That's Monte Carlo variance. Increase the sample count and the noise resolves into the correct image. Your GPU is spending its energy not computing correct answers, but sampling a probability distribution until the answer stops jiggling. See the GPU deep dive for how this is parallelized — GPUs are, in a real sense, Monte Carlo machines.

Finance and Physics

Wall Street runs on random numbers. Options pricing (Black-Scholes and its successors), Value at Risk calculations, stress testing entire portfolios — all Monte Carlo. You simulate thousands of possible market futures, see what fraction of them result in catastrophic loss, and price accordingly. The 2008 financial crisis was partly a failure of the Monte Carlo models to capture the correct distribution of housing prices — the models assumed gaussians where reality had fat tails. Garbage in, garbage out, but with mortgage-backed securities.

CERN simulates particle collisions via Monte Carlo. The LHC generates real collisions; the Monte Carlo simulation generates synthetic ones to compare against, at a rate of roughly 10 billion simulated events per year. Confirming the Higgs boson required matching observed collision signatures against a distribution of Monte Carlo-generated expected signatures. Random numbers, in other words, helped prove the existence of the particle that gives all other particles their mass. Ulam's solitaire game has had an eventful afterlife.

// 12 — Testing Randomness

How do you prove something is random? You can't. Randomness is not a property you can verify with certainty — only one you can fail to disprove. Every statistical test for randomness is really a test for the presence of a specific non-random pattern. Pass all the tests and you're not proven random; you're merely undetected. The set of possible non-random patterns is infinite, the set of tests finite. There is no certificate of randomness. There is only the absence of known suspicion.

This sounds bleak, but it's workable. If your PRNG passes 160 independent statistical tests at 99.9% confidence, the probability that it's secretly terrible in a cryptographically significant way drops to "please direct your paranoia elsewhere." Most RNG failures are obvious in one or two tests. Nobody is surprised by bad randomness quietly.

Diehard Tests

George Marsaglia, 1995. Marsaglia was perhaps the most prolific PRNG researcher of the 20th century — he invented the Mersenne Twister's predecessor, named his PRNG "KISS" (Keep It Simple Stupid), and distributed his test suite on a CD-ROM titled "A CD ROM Full of Random Numbers." The Diehard battery includes tests with names like Birthday Spacings, Overlapping Permutations, Ranks of Matrices, the Squeeze test, and the Monkey Tests. If your generator fails the Monkey Tests, your generator has failed at something a room of monkeys could have caught.

Marsaglia was also notable for discovering multiple failures in widely-used PRNGs and publishing them bluntly. The generator in early versions of Java's java.util.Random failed several Diehard tests. Marsaglia published the failure. Java fixed it. Marsaglia published another failure. This continued.

TestU01

Pierre L'Ecuyer and Richard Simard, Université de Montréal, 2007. Three test batteries of increasing severity: SmallCrush (10 tests, a few seconds to run), Crush (96 tests, an hour), and BigCrush (160 tests, several hours on a fast CPU). BigCrush is the industry benchmark. If your PRNG fails BigCrush, it fails; if it passes BigCrush, it's considered cryptographically-adjacent in terms of statistical quality — though not cryptographically secure, which is a different property.

Xorshift64, a popular fast non-crypto PRNG, passes SmallCrush but fails BigCrush. Mersenne Twister passes BigCrush but fails specific linear-complexity tests in TestU01 that Marsaglia's suite doesn't cover. xoshiro256** passes BigCrush. The hierarchy is clear; using the right tool depends on whether you need speed, statistical quality, or cryptographic security — and knowing that these are not the same thing.

NIST SP 800-22

The US government's official randomness test suite, maintained by the National Institute of Standards and Technology. Required for cryptographic modules seeking FIPS 140-2 certification — which is required for most software used by US federal agencies, defense contractors, and much of the financial industry. Fifteen tests: frequency, block frequency, runs, longest runs of ones, binary matrix rank, spectral (FFT), non-overlapping template matchings, overlapping template matchings, Maurer's universal statistical test, linear complexity, serial, approximate entropy, cumulative sums, random excursions, random excursions variant.

Passing NIST SP 800-22 is necessary but not sufficient for cryptographic security. It tests statistical properties, not resistance to algorithmic cryptanalysis. /dev/urandom passes it. A well-implemented CSPRNG passes it. A Mersenne Twister also passes it, despite being completely predictable once you observe 624 consecutive outputs. The test suite doesn't know about your threat model.

Chi-Squared Test

The simplest randomness test, and the first one any statistician reaches for. Roll a die N times. Count occurrences of each face. Compare observed vs. expected frequencies using the chi-squared statistic: χ² = Σ (observed - expected)² / expected. If χ² is too large, the die is biased. If χ² is suspiciously small — much smaller than expected — the die is too regular. Either extreme is suspicious. Perfect randomness is not perfectly flat; it's flat with expected variance. A histogram that looks exactly right is as alarming as one that looks wrong.

This is the counterintuitive part that surprises people: if you roll a fair die 6,000 times and every face comes up exactly 1,000 times, you should be deeply concerned. The probability of that happening by chance is roughly one in 107. Something is generating suspiciously uniform outcomes. Either the die is controlled, or someone has manipulated the results to look more random than chance produces. Randomness has texture. The absence of texture is evidence of non-randomness.

In the end, randomness is the one resource computers cannot manufacture. Every shuffle, every simulation, every encrypted connection depends on harvesting genuine unpredictability from the physical world and stretching it with mathematics. The next time you flip a coin, remember: your CPU is doing something far stranger to generate the same outcome.
