CONCURRENCY

Shared state, locks, and the art of not breaking everything_

01 — the dragon

Every program that has ever crashed due to a race condition was haunted by the same dragon: two agents reaching for the same piece of memory at the same time, neither knowing the other exists. The result is not a graceful error. It is silent corruption — a counter that reads 999 instead of 1000, a bank balance that is quietly wrong, a list that has been spliced in two directions at once.

The fundamental problem of concurrency is not about threads, or locks, or channels. It is about shared mutable state. If two threads only read data, there is no problem. If two threads write to completely separate memory, there is no problem. The dragon wakes only when at least one thread writes to memory that another thread can see.

Consider the simplest possible example: a counter. A counter is incremented by two threads, each incrementing it 500,000 times. You expect the result to be 1,000,000. You get 743,218. The reason is that counter++ is not atomic — it is three separate machine instructions: load the value into a register, add 1, store it back. When two threads interleave these steps, updates are lost.

Lost-Update Race Condition — Interleaved Timeline 
  Time  Thread A                       Thread B
  ----  ----------------------------   ----------------------------
   t1   LOAD  reg_A = counter (= 5)
   t2                                  LOAD  reg_B = counter (= 5)
   t3   ADD   reg_A = reg_A + 1 (= 6)
   t4                                  ADD   reg_B = reg_B + 1 (= 6)
   t5   STORE counter = reg_A (= 6)
   t6                                  STORE counter = reg_B (= 6)

  Expected: counter = 7
  Actual:   counter = 6   ← one increment was lost

This is called a lost update or write-write conflict. Thread B read a stale value (5) before Thread A stored its result (6), so Thread B's store overwrote Thread A's work. The increment that Thread A did simply vanished. Run this scenario 500,000 times across two threads and you can lose a substantial fraction of your increments — or all of them, depending on scheduling.

What makes race conditions so treacherous is that they are non-deterministic. The program may run correctly a thousand times in testing, then silently corrupt data in production under load. The window for the bad interleaving might be microseconds wide. The bug does not crash — it quietly poisons state. You may not notice for hours, days, or years. Financial systems have been audited for millions of dollars of unexplained discrepancies that turned out to be races.

C — race condition#include <stdio.h>
#include <pthread.h>

static long counter = 0;

void *increment(void *arg) {
    for (int i = 0; i < 500000; i++)
        counter++;   /* NOT atomic: load + add + store */
    return NULL;
}

int main() {
    pthread_t a, b;
    pthread_create(&a, NULL, increment, NULL);
    pthread_create(&b, NULL, increment, NULL);
    pthread_join(a, NULL);
    pthread_join(b, NULL);
    printf("counter = %ld (expected 1000000)\n", counter);
    /* Typical output: counter = 634,821 */
    return 0;
}
The Dragon

Shared mutable state is the root of all concurrency bugs. Every race condition, deadlock, livelock, and priority inversion traces back to it. The only complete solution is to eliminate sharing (message passing), eliminate mutation (immutability), or control access (synchronization). Everything else is a variation on these three themes.

The antidote to the dragon is discipline: identify every piece of state that can be accessed by more than one thread, and ensure that all such accesses are properly coordinated. Easier said than done — in a large codebase the ownership of data is often implicit and the mutation paths are scattered. This is why languages like Rust encode ownership and aliasing rules in the type system: to make the dragon visible at compile time.

02 — the hardware you ignore

Most programmers write concurrent code without ever thinking about the CPU beneath their feet. This is a mistake. The hardware does not share your model of memory. It has caches, write buffers, out-of-order execution engines, and prefetchers — all optimized for single-threaded throughput, all capable of producing surprises when multiple cores are involved. To write correct concurrent code you need to understand what the hardware is actually doing.

The Cache Hierarchy

Modern CPUs are not connected directly to RAM. Access to RAM takes roughly 100 nanoseconds — an eternity at 3 GHz. To bridge the gap, every core has a private L1 cache (typically 32–64 KB, ~1 ns latency) and a private L2 cache (256 KB–1 MB, ~4 ns). A shared L3 cache (8–64 MB, ~12 ns) sits between the cores and RAM. The hierarchy exists purely for performance: most accesses hit L1 and never touch RAM at all.

Cache Lines and MESI

Memory is not transferred between cache and RAM one byte at a time. It moves in cache lines — fixed-size blocks, universally 64 bytes on x86 and ARM. When you read one byte, you pull 64 bytes into your cache. When you write one byte, the entire 64-byte line becomes "dirty" and must eventually be written back.

When multiple cores have copies of the same cache line, they must agree on who owns it. This coordination is handled by the MESI protocol (or one of its variants like MESIF or MOESI). Each cache line is in one of four states: Modified (dirty, only this core has it), Exclusive (clean, only this core has it), Shared (clean, multiple cores have it), or Invalid (stale, must fetch before use).

When Core 0 wants to write a cache line that Core 1 holds in Shared state, Core 0 must first broadcast a "Request for Ownership" (RFO) on the interconnect. Core 1 transitions its copy to Invalid. Core 0 transitions to Modified and performs the write. This round-trip across the cache interconnect costs roughly 40–80 ns — much worse than L1 but much better than RAM. The key insight is that writing shared data is expensive, not because of software overhead, but because of hardware cache coherence.

False Sharing — Two Variables in the Same Cache Line 
  Cache line (64 bytes)
  ┌────────────────────────────────────────────────────────────────┐
  │  counter_A (8 bytes)  │  counter_B (8 bytes)  │  padding...   │

False Sharing

False sharing occurs when two threads write to different variables that happen to occupy the same cache line. The variables are logically independent — the threads are not sharing data — but the hardware does not know that. Every write by Thread A forces Thread B's copy of the cache line to be invalidated, triggering a coherence round-trip. The result is that two threads which appear to be running in parallel are actually serialized by the cache bus.

False sharing is famously hard to diagnose because the code looks correct and runs correctly — it is just inexplicably slow. The fix is alignment: pad struct fields to cache-line boundaries so hot fields written by different threads live in different lines. The alignas(64) specifier in C11 and the #[repr(align(64))] attribute in Rust both serve this purpose.

False Sharing Can Be Worse Than a Mutex

A benchmark expecting 2x speedup from two threads can show 0.5x — slower than single-threaded — due to false sharing. Tight counter loops can degrade by 5–10× compared to padded versions (Drepper, "What Every Programmer Should Know About Memory," 2007). The coherence traffic saturates the interconnect. Always profile with hardware performance counters (Linux perf stat -e cache-misses) before concluding that a concurrent design is correct.

The Memory Model

Beyond caches, CPUs also reorder loads and stores for performance. A store to address X followed by a store to address Y may be observed by another core in Y-then-X order, because they went into different store-buffer slots. This is the memory model problem, and it is why atomic operations carry ordering annotations (see Chapter 5). The hardware is not being malicious — it is aggressively optimizing for single-threaded throughput. The multi-threaded programmer must explicitly tell the CPU when observation order matters.

For GPU concurrency, the story is even more extreme: thousands of threads running in SIMT (Single Instruction Multiple Threads) lockstep on warps of 32, with a completely different memory hierarchy of registers, shared memory, L1, L2, and global memory. See the GPU deep dive for details.

03 — mutexes from scratch

The oldest and most general solution to shared mutable state is the mutual exclusion lock, or mutex. A mutex is a token: only one thread can hold it at a time. Any thread that wants to access protected state must first acquire the mutex; any thread that cannot acquire it must wait. When the holding thread releases the mutex, one waiting thread is woken and given the token.

Building a Spinlock

The simplest possible mutex is a spinlock. A spinlock uses a single atomic flag. Acquiring means spinning in a loop, attempting a compare-and-swap (CAS) until it succeeds. Releasing means clearing the flag. No OS involvement, no context switch — just tight CPU loops.

C — spinlock via CAS#include <stdatomic.h>

typedef struct { atomic_flag flag; } spinlock_t;

void spin_lock(spinlock_t *s) {
    while (atomic_flag_test_and_set_explicit(
               &s->flag, memory_order_acquire))
        ; /* burn CPU until flag is clear */
}

void spin_unlock(spinlock_t *s) {
    atomic_flag_clear_explicit(&s->flag, memory_order_release);
}

Spinlocks work well when the critical section is extremely short (tens of nanoseconds) and contention is rare. But they waste CPU cycles burning in a loop, which matters when:

  • The lock holder is preempted by the OS (the spinner wastes its entire quantum)
  • The critical section is long (hundreds of microseconds)
  • You have more threads than cores (spinners prevent useful work)

The Futex: Fast Userspace Path

The Linux futex (fast userspace mutex) syscall solves the spinning problem by combining a userspace atomic with an OS wait queue. The fast path — acquiring an uncontended lock — is a single atomic CAS entirely in userspace, no syscall needed. Only on contention does the thread call into the kernel to be suspended, freeing the CPU for other work.

Futex State Machine 
  States: 0 = unlocked, 1 = locked (no waiters), 2 = locked (waiters)

  LOCK:
    CAS(0 → 1)?  ──yes──▶  acquired (fast path, no syscall)
        │no
        ▼
    CAS(1 → 2) or set to 2, then FUTEX_WAIT
    (kernel suspends thread until value != 2)

  UNLOCK:
    atomic_sub → result 0?  ──yes──▶  done (no waiters)
        │no
        ▼
    store 0, then FUTEX_WAKE(1)
    (kernel wakes one waiter)

The key insight of the futex: the common case (uncontended lock) is handled without any kernel involvement. You only pay the syscall overhead when you actually have to wait. This is why pthreads mutexes and Go's sync.Mutex are implemented on top of futexes on Linux: they are cheap when uncontended and correct when contended. See the syscalls deep dive for more on the futex syscall interface.

class AsyncMutex {
  private queue: Array<() => void> = [];
  private locked = false;

  acquire(): Promise<void> {
    return new Promise(resolve => {
      if (!this.locked) {
        this.locked = true;
        resolve();
      } else {
        this.queue.push(resolve);
      }
    });
  }

  release(): void {
    if (this.queue.length > 0) {
      const next = this.queue.shift()!;
      next();           // hand lock to next waiter
    } else {
      this.locked = false;
    }
  }
}

const mu = new AsyncMutex();
let counter = 0;

async function increment() {
  await mu.acquire();
  try { counter++; } finally { mu.release(); }
}

Mutex Poisoning (Rust)

In Rust, if a thread panics while holding a Mutex, the mutex becomes "poisoned." Subsequent lock() calls return Err(PoisonError), forcing callers to decide whether to proceed with potentially inconsistent state. This prevents silent corruption from propagating — a design choice unique to Rust's ownership model.

04 — semaphores and condition variables

A mutex is binary: you either have it or you don't. But many real coordination problems require counting. A connection pool has 10 slots. A rate limiter allows 100 requests per second. A thread pool has a work queue with bounded capacity. For these scenarios, semaphores generalize the mutex concept to an integer counter.

Counting Semaphores

A semaphore has two operations, traditionally called P (proberen, "try") and V (verhogen, "increment"), or equivalently wait/signal, or acquire/release. wait decrements the counter; if it would go negative, the caller blocks. signal increments the counter and wakes one waiter. A binary semaphore (initial value 1) is equivalent to a mutex. A counting semaphore (initial value N) allows N threads to proceed simultaneously.

Producer-Consumer with Bounded Buffer (Semaphores) 
  Semaphores:
    empty  = N   (slots available to fill)
    filled = 0   (slots available to consume)
    mutex  = 1   (guard on buffer itself)

  PRODUCER:                    CONSUMER:
    wait(empty)                  wait(filled)
    wait(mutex)                  wait(mutex)
    buffer[in] = item            item = buffer[out]
    in = (in+1) % N              out = (out+1) % N
    signal(mutex)                signal(mutex)
    signal(filled)               signal(empty)

  ─────────────────────────────────────────────
  Buffer:  [ P ][ P ][ P ][   ][   ]
            ↑out            ↑in
  filled=3, empty=2  →  3 items waiting, 2 slots free

Condition Variables

Semaphores are elegant but limited: they carry a count but no information about why threads are waiting. Condition variables (condvars) solve this. A condvar lets a thread atomically release a mutex and go to sleep until some predicate becomes true. The atomicity of "release and sleep" is critical: without it you get a missed-wakeup race.

The canonical pattern is to always check the condition in a while loop, not an if. This defends against spurious wakeups: the OS is allowed to wake a thread from pthread_cond_wait for no reason. POSIX explicitly permits this as a performance tradeoff. A waiting thread might be woken, re-acquire the mutex, check the condition, find it false, and go back to sleep. The while loop makes this safe; an if would proceed on false data.

#include <pthread.h>
#define BUF_SIZE 8

pthread_mutex_t mu = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  not_full  = PTHREAD_COND_INITIALIZER;
pthread_cond_t  not_empty = PTHREAD_COND_INITIALIZER;
int buf[BUF_SIZE], head = 0, tail = 0, count = 0;

void produce(int item) {
    pthread_mutex_lock(&mu);
    while (count == BUF_SIZE)           /* WHILE, not if */
        pthread_cond_wait(&not_full, &mu);
    buf[tail] = item;
    tail = (tail + 1) % BUF_SIZE;
    count++;
    pthread_cond_signal(&not_empty);
    pthread_mutex_unlock(&mu);
}

int consume() {
    pthread_mutex_lock(&mu);
    while (count == 0)                  /* WHILE, not if */
        pthread_cond_wait(&not_empty, &mu);
    int item = buf[head];
    head = (head + 1) % BUF_SIZE;
    count--;
    pthread_cond_signal(&not_full);
    pthread_mutex_unlock(&mu);
    return item;
}

// TypeScript: async semaphore using Promise queue
class Semaphore {
  private count: number;
  private waiters: Array<() => void> = [];

  constructor(initial: number) { this.count = initial; }

  acquire(): Promise<void> {
    if (this.count > 0) { this.count--; return Promise.resolve(); }
    return new Promise(r => this.waiters.push(r));
  }

  release(): void {
    const next = this.waiters.shift();
    if (next) next();
    else this.count++;
  }
}

  try { await task(); }
  finally { sem.release(); }
}

struct BoundedQueue {
    inner: Mutex<(Vec<i32>, usize)>,
    not_full:  Condvar,
    not_empty: Condvar,
}

impl BoundedQueue {
    fn put(&self, item: i32) {
        let mut guard = self.inner.lock().unwrap();
        let max = guard.1;
        guard = self.not_full.wait_while(guard,
            |state| state.0.len() == max).unwrap();
        guard.0.push(item);
        self.not_empty.notify_one();
    }

    fn get(&self) -> i32 {
        let mut guard = self.inner.lock().unwrap();
        guard = self.not_empty.wait_while(guard,
            |state| state.0.is_empty()).unwrap();
        let item = guard.0.remove(0);
        self.not_full.notify_one();
        item
    }
}
Spurious Wakeup Defence

Always wrap cond_wait in a while loop that re-checks the predicate, never an if. POSIX permits spurious wakeups as an explicit optimization latitude. Linux futex wakeups can also be caused by signal delivery. The while loop costs almost nothing when the predicate is true and prevents silent bugs when it isn't.

05 — atomics: the fine print

Not every shared-state problem requires a mutex. For simple numeric counters, flags, and pointers, atomic operations provide a lighter-weight solution: single machine instructions that are guaranteed to be indivisible. No other thread can observe a partial state. The hardware provides these primitives — compare-and-swap, fetch-and-add, test-and-set — and the language exposes them through a thin wrapper.

Compare-and-Swap

The most powerful atomic primitive is compare-and-swap (CAS). CAS takes three arguments: a memory address, an expected value, and a new value. Atomically, it reads the address, compares to expected, and if equal, writes new. It returns whether the swap happened. Everything that is built on top of atomics — spinlocks, lock-free stacks, queues, skip lists — uses CAS as its building block.

CAS Loop — Lock-Free Increment 
  Atomic increment via CAS:

  do {
      old = load(counter)           ← read current value
      new = old + 1                 ← compute desired value
  } while (!CAS(counter, old, new)) ← retry if someone else changed it

  Thread A: old=5, new=6, CAS succeeds → counter=6
  Thread B: old=5, new=6, CAS fails   → retry: old=6, new=7, CAS succeeds

  No mutex needed. Each thread retries until it "wins" the CAS.

Memory Orderings

An atomic operation being indivisible does not mean it is globally ordered with respect to surrounding non-atomic operations. The CPU and compiler are free to reorder instructions around atomics unless you explicitly constrain them. Memory orderings specify what reorderings are permitted.

Relaxed: The operation is atomic but carries no ordering guarantee with respect to other memory operations. Suitable for counters where you only care about the final total.

Acquire/Release: A Release store pairs with an Acquire load to form a happens-before edge. Everything written before the Release is visible to any thread that subsequently loads with Acquire. This is the primitive for implementing locks and message-passing handoffs.

SeqCst (sequentially consistent): The strongest guarantee. All SeqCst operations across all threads appear in a single total order that every thread agrees on. Safe but requires a full memory fence on x86 (MFENCE) and is significantly slower. Use only when you need cross-thread ordering guarantees that Acquire/Release cannot express.

  RELEASE store (unlock):             ACQUIRE load (lock):

  write A                             lock (acquire load)
  write B                             ──────────────  ← fence
  write C                             read A   ← sees write A
  ──────────────  ← fence             read B   ← sees write B
  store(flag=1, Release)              read C   ← sees write C

  Everything before the Release       Everything after the Acquire
  fence is visible to any thread      sees everything before the

Memory Order Cheat Sheet

Use Relaxed for statistics counters (final value only, no ordering needed). Use Release on the write side and Acquire on the read side of any flag-based handoff. Use SeqCst when you need a globally consistent view of multiple atomics — but benchmark first; it can cost 5–10x over Relaxed on ARM.

06 — lock-free data structures

Lock-free data structures make a stronger promise than mutex-protected ones: the system as a whole makes progress even if some threads are arbitrarily delayed. No thread can hold a lock and be preempted, blocking every other thread indefinitely. This property — called lock-freedom — makes these structures attractive for real-time systems, signal handlers, and high-contention hot paths where even a momentarily-held mutex becomes a bottleneck.

The Treiber Stack

The canonical lock-free data structure is the Treiber stack (1986). It is a singly-linked list whose head pointer is atomic. To push: allocate a new node, set its next pointer to the current head, then CAS the head. To pop: load the head, CAS head to head.next. If either CAS fails because another thread intervened, simply retry. No mutex is ever acquired; forward progress requires only that some thread succeeds each round.

#include <stdatomic.h>
#include <stdlib.h>

typedef struct Node { int val; struct Node *next; } Node;
typedef struct { _Atomic(Node*) head; } Stack;

void push(Stack *s, int val) {
    Node *n = malloc(sizeof(Node));
    n->val = val;
    do {
        n->next = atomic_load(&s->head);
    } while (!atomic_compare_exchange_weak(&s->head, &n->next, n));
}

int pop(Stack *s, int *out) {
    Node *old;
    do {
        old = atomic_load(&s->head);
        if (!old) return 0;
    } while (!atomic_compare_exchange_weak(&s->head, &old, old->next));
    *out = old->val;
    /* NOTE: freeing old requires hazard pointers to avoid UAF */
    return 1;
}

The ABA Problem

The CAS loop has a subtle bug known as the ABA problem. Thread A reads the head as pointer P with value A. Before it can CAS, Thread B pops P, pushes a new node Q, then pushes P back (same pointer address, same bit pattern). Thread A's CAS succeeds — the value matches — but the list has changed beneath it. Q is now a leaked, orphaned node.

ABA Problem Timeline 
  Initial stack: A → B → C

  Thread 1: reads head = A, preempted before CAS
  Thread 2: pops A (head = B→C)
            pops B (head = C)
            pushes A back (head = A→C)
  Thread 1 resumes: CAS(head, A, B) succeeds! (A pointer matches)
  Stack is now: B → C   but B.next still points into freed memory → UAF

  Fix: tagged pointer — encode a version counter in pointer high bits.
  A@tag5 ≠ A@tag6, so Thread 1's CAS correctly fails.

Solutions to ABA include: tagged pointers (version counter in high bits — valid on 64-bit where pointers only use 48 bits), hazard pointers (each thread publishes which pointers it is reading; reclamation is deferred until no thread holds a hazard), and epoch-based reclamation (threads declare their epoch; memory is freed only when all threads have advanced past the deletion epoch). Rust's crossbeam crate uses epoch-based reclamation.

Lock-Free != Bug-Free

Lock-free data structures are significantly harder to write correctly than mutex-protected equivalents. ABA, memory reclamation hazards, and subtle ordering bugs have caused production incidents at major tech companies. Prefer well-tested library implementations over rolling your own: C++ folly, Rust crossbeam, Java java.util.concurrent.

07 — channels and CSP

In 1978, Tony Hoare published Communicating Sequential Processes (CSP) — a mathematical model of concurrent computation based not on shared memory but on message passing. The core idea: processes communicate by sending values through typed channels. No shared state, no locks, no races. The sender blocks until the receiver is ready (or vice versa), providing natural synchronization as a byproduct of communication.

Rob Pike condensed this into one of Go's most quoted maxims: "Don't communicate by sharing memory; share memory by communicating." The practical translation: instead of a mutex protecting a shared map, have one goroutine own the map and receive requests through a channel. The ownership is explicit and exclusive; the coordination is visible in the code's structure.

#include <pthread.h>
#define CAP 4

typedef struct {
    int            buf[CAP], head, tail, count;
    pthread_mutex_t mu;
    pthread_cond_t  not_empty, not_full;
} Chan;

void send(Chan *c, int v) {
    pthread_mutex_lock(&c->mu);
    while (c->count == CAP)
        pthread_cond_wait(&c->not_full, &c->mu);
    c->buf[c->tail] = v;
    c->tail = (c->tail + 1) % CAP;
    c->count++;
    pthread_cond_signal(&c->not_empty);
    pthread_mutex_unlock(&c->mu);
}

int recv(Chan *c) {
    pthread_mutex_lock(&c->mu);
    while (c->count == 0)
        pthread_cond_wait(&c->not_empty, &c->mu);
    int v = c->buf[c->head];
    c->head = (c->head + 1) % CAP;
    c->count--;
    pthread_cond_signal(&c->not_full);
    pthread_mutex_unlock(&c->mu);
    return v;
}

class Channel<T> {
  private buf: T[] = [];
  private cap: number;
  private readers: Array<(v: T) => void> = [];
  private writers: Array<{ v: T; ok: () => void }> = [];

  constructor(cap = 0) { this.cap = cap; }

  async send(v: T): Promise<void> {
    if (this.readers.length > 0) { this.readers.shift()!(v); return; }
    if (this.buf.length < this.cap) { this.buf.push(v); return; }
    await new Promise<void>(ok => this.writers.push({ v, ok }));
  }

  async recv(): Promise<T> {
    if (this.buf.length > 0) return this.buf.shift()!;
    if (this.writers.length > 0) {
      const { v, ok } = this.writers.shift()!;
      ok(); return v;
    }
    return new Promise<T>(r => this.readers.push(r));
  }
}

Ownership Through Channels

When you send a value through a channel, you are transferring ownership. The sender should not touch the value after sending. 08 — async/await

Threads are expensive. Each OS thread carries a stack (typically 2–8 MB), requires a kernel data structure, and context-switching between threads costs hundreds to thousands of nanoseconds. For I/O-bound workloads — web servers handling thousands of simultaneous connections — spawning one thread per connection is impractical. Async/await solves this by multiplexing many logical tasks onto a small number of OS threads using cooperative scheduling.

The Event Loop

The fundamental mechanism is an event loop: a single thread that continuously polls for I/O readiness events (via epoll, kqueue, or io_uring), runs the callbacks or resumes the coroutines associated with those events, then polls again. Since the thread never blocks on I/O, it can handle thousands of concurrent connections with minimal memory overhead — each suspended task holds only its saved registers and stack frame on the heap, not a full OS thread stack. See the syscalls deep dive for io_uring details.

Cooperative vs Preemptive Scheduling 
  PREEMPTIVE (OS threads):
  Thread A  ██████████░░░░░░██████████░░░░
  Thread B  ░░░░░░░░░░██████████░░░░░░████
                      ↑ OS interrupts at arbitrary time

  COOPERATIVE (async tasks on one thread):
  Task A    ██████──────────────██████────
  Task B    ──────████████────────────████
                  ↑ voluntary yield at "await"

  Both models achieve concurrency. Preemptive is simpler for

State Machine Transformation

When a compiler sees an async function, it transforms it into a state machine. Each await point becomes a state transition: the function saves its local variables into a heap-allocated struct, returns control to the executor, and is resumed at the next state when the awaited future completes. The function's frame is on the heap, not the OS stack — which is why thousands of suspended async tasks are memory-cheap while thousands of OS threads are not.

09 — the classic bugs

Every concurrent programmer eventually meets the same four villains: race conditions, deadlocks, livelocks, and starvation. They are distinct in cause and cure. Conflating them leads to wrong fixes. A developer who adds a mutex to cure a livelock has not fixed anything.

Race Condition vs Data Race

A data race is a specific technical term: two threads access the same memory location concurrently, at least one access is a write, and there is no synchronization between them. In C, C++, and Rust, a data race is undefined behavior — the compiler is allowed to assume they cannot happen, and will optimize code in ways that produce nonsensical results when they do. In Java, Go, and Python the semantics are better-defined but still produce incorrect results.

A race condition is a broader term: a bug where the outcome depends on the relative timing of events. You can have a race condition without a data race — for example, a TOCTOU (time-of-check time-of-use) bug where you check a file's existence, then open it, and another process deletes it in between. Perfectly synchronized, no data race, still a race condition.

Data Races Are Undefined Behavior

In C, C++, and Rust, a data race is UB. The compiler may assume the racing variable cannot be observed by another thread and eliminate "redundant" reads, hoist stores out of loops, or constant-fold conditionals. The result is code that appears correct in isolation but silently corrupts state when run concurrently. ThreadSanitizer (TSan) exists specifically to catch these before they reach production.

Deadlock: Dining Philosophers

A deadlock occurs when a set of threads are each waiting for a resource held by another thread in the set, forming a cycle. No thread can make progress; all are stuck forever. Dijkstra's classic formulation: N philosophers sit at a round table, each with a fork to their left and right. To eat, a philosopher needs both forks. If each philosopher simultaneously picks up their left fork, no one can pick up their right fork — deadlock.

Four conditions must all hold simultaneously for deadlock (Coffman conditions): Mutual exclusion (resources cannot be shared), Hold and wait (a thread holds a resource while waiting for another), No preemption (resources cannot be forcibly taken), and Circular wait (a cycle of threads, each waiting for the next). Breaking any one condition prevents deadlock. The most practical: enforce a global lock ordering so all threads acquire locks in the same order, breaking circular wait.

Deadlock Cycle — Dining Philosophers 
        P0
       /  \
    f0      f1
    |        |
  P3          P1
    |        |
    f3      f2
       \  /
        P2

  Each Pn holds fn, waits for f(n+1) mod 4
  → circular dependency → deadlock

  Fix: require all threads to acquire forks in lowest-index-first order.
  P0: take f0, f1   P1: take f1, f2   P2: take f2, f3   P3: take f0, f3
  P3 now takes f0 first (not f3), breaking the cycle.

Livelock and Starvation

A livelock is like a deadlock except the threads are not blocked — they are actively executing and retrying, but making no progress. Two people in a hallway both step aside in the same direction, repeatedly. CPU is burned, nothing gets done. Livelocks often arise from overly polite retry logic: if you always back off the moment someone else tries, you may never succeed.

Starvation is when one thread is perpetually denied access to a resource because other threads are always preferred. A low-priority thread may never acquire a mutex if high-priority threads continuously contend for it. Most OS thread schedulers have aging mechanisms to prevent extreme starvation, but application-level lock implementations may not.

Priority Inversion: Mars Pathfinder

In 1997, the Mars Pathfinder rover began resetting itself periodically. The cause was classic priority inversion: a low-priority meteorological task held a mutex protecting a shared data bus. A high-priority communications task needed that mutex, but was blocked. Meanwhile, a medium-priority task that did not need the mutex was scheduled instead of the low-priority task (because medium > low). The high-priority task waited indefinitely.

Priority Inversion Timeline 
  Priority:  HIGH=3   MED=2   LOW=1

  t0: LOW acquires mutex M (holds it)
  t1: HIGH tries to acquire M → BLOCKS (waiting for LOW)
  t2: MED becomes runnable (no mutex needed)
  t3: Scheduler: MED > LOW → runs MED instead of LOW
  t4: MED runs to completion... HIGH still waiting!
  t5: LOW finally runs, releases M
  t6: HIGH unblocks

  Effective priority: HIGH < MED < LOW — inverted!

  Fix: Priority Inheritance — while LOW holds M, run it at HIGH's
  priority, so MED cannot preempt it.

The fix is priority inheritance: when a high-priority thread blocks on a mutex, the lock holder temporarily inherits the higher priority until it releases. NASA's engineers patched the VxWorks kernel remotely and re-enabled priority inheritance on the shared mutex. The resets stopped. The story is a canonical example of why priority inversion must be addressed in real-time systems.

10 — patterns that work

Beyond the basic mutex, experienced concurrent programmers reach for a toolkit of higher-level patterns. Each pattern trades off simplicity, performance, and correctness in different ways. Knowing which tool to reach for — and equally important, which to avoid — separates effective concurrent programming from fearful serial programming.

Reader-Writer Locks

A plain mutex allows only one thread at a time, even for read-only accesses. If your data is read frequently and written rarely, this serializes all readers unnecessarily. A reader-writer lock (RWLock) allows many readers simultaneously, or exactly one writer, but not both. Reads can proceed in parallel; writes get exclusive access. This pattern is ideal for caches, configuration stores, and any structure with a high read/write ratio.

class RWLock:
    def __init__(self):
        self._read_lock  = threading.Lock()
        self._write_lock = threading.Lock()
        self._readers = 0

    def read_acquire(self):
        with self._read_lock:
            self._readers += 1
            if self._readers == 1:
                self._write_lock.acquire()  # first reader blocks writers

    def read_release(self):
        with self._read_lock:
            self._readers -= 1
            if self._readers == 0:
                self._write_lock.release()  # last reader unblocks writers

    def write_acquire(self): self._write_lock.acquire()
    def write_release(self): self._write_lock.release()

class RWLock {
  private readers = 0;
  private writing = false;
  private queue: Array<() => void> = [];

  async readLock(): Promise<void> {
    while (this.writing)
      await new Promise<void>(r => this.queue.push(r));
    this.readers++;
  }
  readUnlock(): void {
    this.readers--;
    if (this.readers === 0) this._wake();
  }
  async writeLock(): Promise<void> {
    while (this.writing || this.readers > 0)
      await new Promise<void>(r => this.queue.push(r));
    this.writing = true;
  }
  writeUnlock(): void { this.writing = false; this._wake(); }
  private _wake(): void { this.queue.forEach(r => r()); this.queue = []; }
}

    let handles: Vec<_> = (0..4).map(|_| {
        let d = data.clone();
        thread::spawn(move || {
            let v = d.read().unwrap(); // shared
            println!("{:?}", *v);
        })
    }).collect();
    for h in handles { h.join().unwrap(); }
}

Work Stealing

In a naive thread pool, work is dispatched from a single queue. Under load the queue becomes a bottleneck: all threads contend for the same mutex. Work stealing solves this by giving each thread its own double-ended queue (deque). A thread pushes and pops its own work from the local end of its deque. When it runs out of work, it "steals" from the tail of another thread's deque. The algorithm was formalized by Blumofe & Leiserson (1999) in their Cilk system at MIT, which proved optimal expected-time bounds. Java's ForkJoinPool, Go's runtime scheduler, and Tokio's multi-threaded Rust runtime all use work stealing.

Work-Stealing Deques 
  Thread 0 deque:   [T0 T1 T2 T3]  ← push/pop from left (local)
  Thread 1 deque:   [T4 T5]        ← steal from right (foreign)
  Thread 2 deque:   []             ← empty: steal from T0's right end

  Thread 2 steals T3 from Thread 0 — only one CAS needed at the tail.
  Thread 0 works from the head — no contention with the stealer.
  Throughput scales because each thread is mostly self-sufficient.

Double-Checked Locking

The double-checked locking pattern attempts to reduce locking overhead for lazy initialization. Check without the lock first; if the object already exists, return it without acquiring. If null, acquire the lock, check again (in case another thread initialized it while you waited), then initialize. This pattern is broken in C without atomics because the compiler or CPU can reorder the store to the pointer before the stores that initialize the object. In modern C++ (memory_order_acquire/release), Java (volatile), and Go (sync.Once), safe versions exist.

Use sync.Once in Go, std::call_once in C++

Never implement double-checked locking by hand. The standard library implementations are correct, well-tested, and compiled to the right memory barriers for each platform. Hand-rolled versions have subtle bugs that only manifest on ARM or in optimized builds. The performance difference from the standard version is negligible.

11 — thread pools and concurrent maps

Raw threads are expensive to create and destroy. A thread pool pre-allocates a fixed number of worker threads and feeds them work through a queue. The pattern appears everywhere: HTTP servers, database connection pools, image processing pipelines. Understanding the pool pattern — and its failure modes — is essential for any non-trivial concurrent system.

Fixed Thread Pool

A fixed pool has N threads and a work queue. Callers submit tasks to the queue. Workers pull tasks and execute them. The queue provides back-pressure: if the queue is bounded and full, the caller blocks (or gets an error). This prevents runaway resource consumption when producers outpace consumers.

#include <pthread.h>
#include <stdlib.h>
#define NWORKERS 4
#define QCAP     64

typedef void (*task_fn)(void*);
typedef struct { task_fn fn; void *arg; } Task;
typedef struct {
    Task           queue[QCAP];
    int            head, tail, count, shutdown;
    pthread_mutex_t mu;
    pthread_cond_t  not_empty;
    pthread_t       workers[NWORKERS];
} Pool;

static void *worker(void *arg) {
    Pool *p = (Pool*)arg;
    while (1) {
        pthread_mutex_lock(&p->mu);
        while (!p->count && !p->shutdown)
            pthread_cond_wait(&p->not_empty, &p->mu);
        if (p->shutdown && !p->count) {
            pthread_mutex_unlock(&p->mu); return NULL;
        }
        Task t = p->queue[p->head];
        p->head = (p->head + 1) % QCAP;
        p->count--;
        pthread_mutex_unlock(&p->mu);
        t.fn(t.arg);
    }
}

Concurrent Hash Maps

A plain map guarded by a single mutex serializes all reads and writes. For high-throughput systems, striped locking divides the map into N independent shards, each with its own lock. A key is hashed to determine its shard. Threads touching different shards proceed in parallel. Java's ConcurrentHashMap uses 16 stripes by default. Go's sync.Map uses a two-level structure: a fast read-only path using atomics, and a locked dirty map for writes.

Fully lock-free hash maps exist (based on linear probing with CAS operations) but are significantly harder to implement correctly. For most applications, striped locking provides excellent throughput with manageable complexity. See the hash tables deep dive for the full picture of hash map implementations.

Thread Pool Sizing Heuristic

For CPU-bound work: pool size = number of CPU cores (or cores − 1 to leave headroom). For I/O-bound work: pool size = cores × (1 + wait_time/compute_time). If a task waits 9x as long as it computes, a 4-core machine can sustain 40 threads productively. Too few threads and CPUs sit idle. Too many and context switching overhead dominates.

12 — tools

Concurrency bugs are notoriously hard to reproduce. A race that occurs one time in ten million runs is unreproducible in testing but catastrophic in production. Fortunately, a generation of sophisticated tooling has been developed to detect races, deadlocks, and ordering violations — some at compile time, some via runtime instrumentation, some statically. These tools should be part of every concurrent project's CI pipeline.

ThreadSanitizer (TSan)

ThreadSanitizer is a compile-time instrumentation tool originally from Google, now part of LLVM and GCC. Compile your program with -fsanitize=thread (C/C++) and TSan instruments every memory access and synchronization operation. At runtime, it maintains a shadow memory representing happens-before relationships between threads. When it detects a data race — two accesses to the same address with no happens-before ordering — it reports the exact stack traces of both conflicting accesses.

TSan has roughly 5–15x runtime overhead and 5–10x memory overhead, making it impractical for production but excellent for testing. False positives are rare (TSan is sound: it only reports real races). False negatives exist because TSan only detects races that actually happen during the instrumented run, not races that could happen in principle. Run under heavy load with multiple threads to maximize race exposure.

Helgrind (Valgrind)

Helgrind is part of the Valgrind suite. It detects POSIX threading errors: misuse of the pthreads API, data races (via happens-before analysis), and potential deadlocks (lock ordering violations). Unlike TSan, Helgrind does not require recompilation — it instruments binaries at the instruction level. The tradeoff: 20–50x runtime overhead. Use Helgrind when you cannot recompile (third-party libraries) or when you want a second opinion.

Go Race Detector

Go ships a built-in race detector based on the same happens-before algorithm as TSan, activated with the -race flag. Because Go controls the scheduler, the race detector integrates tightly with goroutine scheduling and channel operations. It is significantly easier to use than TSan: just go test -race ./.... Google recommends running all tests with -race on every commit; the overhead is acceptable for test runs.

Rust's Borrow Checker

Rust's borrow checker eliminates entire classes of data races at compile time. The rules are simple: a reference can be either shared (&T, many allowed) or exclusive (&mut T, only one). These two kinds cannot coexist. This is precisely the condition needed to prevent data races: you cannot have a write concurrent with any other access if there is at most one writer and no readers during a write. Safe Rust code literally cannot have data races — the type system forbids the pattern that would cause them.

Simulate detecting a threading issue by tracking lock acquire/release order in Python to find potential deadlocks:

import threading
import collections

# Simple lock-order tracker — records which locks each thread holds
lock_graph = collections.defaultdict(set)
lock_log = []

class TrackedLock:
    def __init__(self, name):
        self.name = name
        self._lock = threading.Lock()
        self._holder = None

    def acquire(self):
        tid = threading.get_ident()
        # Check for potential deadlock: does our target hold any lock we hold?
        lock_log.append(f"Thread {tid} acquiring {self.name}")
        self._lock.acquire()
        self._holder = tid
        lock_graph[tid].add(self.name)

    def release(self):
        tid = threading.get_ident()
        lock_log.append(f"Thread {tid} releasing {self.name}")
        lock_graph[tid].discard(self.name)
        self._holder = None
        self._lock.release()

    def __enter__(self): self.acquire(); return self
    def __exit__(self, *a): self.release()

lockA = TrackedLock("A")
lockB = TrackedLock("B")

with lockA:
    with lockB:
        lock_log.append("Critical section: holding A and B")

print("Lock acquisition log:")
for entry in lock_log:
    print(" ", entry)
print("No deadlock — locks acquired in consistent order A -> B")
The Testing Paradox

Concurrency bugs are Heisenbugs: observing them changes their probability. Adding print statements, running under a debugger, or even compiling with debug symbols can alter thread scheduling enough to hide a race. The correct response is: use TSan/race detector in CI (deterministic instrumentation), test under stress (high core count, synthetic load), and design systems to be obviously correct rather than merely apparently correct.

13 — the kernel underneath

Concurrent programs do not run in isolation. The operating system kernel is the silent choreographer: it decides which thread runs on which core, when it is preempted, how long it sleeps, and when it wakes. Understanding the kernel's role is essential for tuning concurrent systems and for diagnosing performance problems that appear mysterious at the application level.

Futex Deep Dive

The futex (fast userspace mutex) syscall is the foundation of all userspace synchronization on Linux. Its interface is deceptively simple: FUTEX_WAIT(addr, val) sleeps the calling thread if *addr == val; FUTEX_WAKE(addr, n) wakes up to n threads sleeping on addr. The value check in FUTEX_WAIT is atomic with the sleep: this prevents the missed-wakeup race where a signal is sent before the waiter calls FUTEX_WAIT.

C — futex syscall#include <linux/futex.h>
#include <syscall.h>
#include <stdatomic.h>

/* Simplified mutex using futex directly */
static inline int futex_wait(atomic_int *addr, int expected) {
    return syscall(SYS_futex, addr, FUTEX_WAIT_PRIVATE,
                   expected, NULL, NULL, 0);
}
static inline int futex_wake(atomic_int *addr, int n) {
    return syscall(SYS_futex, addr, FUTEX_WAKE_PRIVATE,
                   n, NULL, NULL, 0);
}

atomic_int mutex_state = 0; /* 0=free, 1=locked */

void lock() {
    int expected = 0;
    while (!atomic_compare_exchange_weak(&mutex_state, &expected, 1)) {
        futex_wait(&mutex_state, 1);  /* sleep until state != 1 */
        expected = 0;
    }
}
void unlock() {
    atomic_store(&mutex_state, 0);
    futex_wake(&mutex_state, 1);   /* wake one waiter */
}

CFS: The Completely Fair Scheduler

Linux's Completely Fair Scheduler (CFS) models each runnable task as a point on a virtual timeline. A task's position is its vruntime (virtual runtime): the total CPU time it has consumed, scaled by its priority. CFS always runs the task with the smallest vruntime. When a task blocks (on I/O, mutex, or sleep), its vruntime stops advancing. When it unblocks, it re-enters the run queue with its old (now smaller-than-average) vruntime and gets scheduled promptly — this is why interactive tasks feel responsive.

For concurrent programs, the key implication is that threads that frequently block (waiting on mutexes, channels, I/O) will tend to have lower vruntime and be prioritized when they become runnable. Threads that spin-wait accumulate vruntime and are deprioritized. This is one reason the OS scheduler naturally penalizes spinlocks under contention. Note: as of Linux 6.6 (October 2023), CFS has been replaced by EEVDF (Earliest Eligible Virtual Deadline First), which retains vruntime tracking but adds virtual deadlines for better latency guarantees. The transition is incremental — most documentation still refers to CFS.

SIMT on GPU

GPU concurrency is architecturally different. A CUDA kernel launches thousands to millions of threads simultaneously. These are grouped into warps of 32 threads (NVIDIA) that execute in lockstep — Single Instruction Multiple Thread (SIMT). If threads in a warp diverge (different code paths due to an if branch), both paths execute serially with the non-active threads masked off. This is warp divergence and kills GPU throughput. See the GPU deep dive for warp scheduling, shared memory, and atomic operations on GPU.

io_uring

Since Linux 5.1, io_uring provides a radically more efficient async I/O interface. Two ring buffers — a submission queue (SQ) and a completion queue (CQ) — are shared between userspace and kernel via mmap. Userspace writes I/O requests to the SQ and reads completions from the CQ. For the common case, zero syscalls are needed per operation: the kernel polls the SQ ring in kernel context, and userspace polls the CQ ring. Throughput for small I/O operations can reach millions of IOPS, far exceeding traditional epoll. See the syscalls deep dive for the full io_uring interface.

FUTEX_PRIVATE_FLAG

Always use FUTEX_WAIT_PRIVATE / FUTEX_WAKE_PRIVATE for mutexes that are not shared between processes. The private variants skip the global hash table lookup and go directly to a per-process queue, reducing futex syscall overhead by ~30% under contention. glibc's pthreads uses this optimization automatically for non-shared mutexes.

futex_waitv (Linux 5.16+)

Linux 5.16 (January 2022) added futex_waitv — a multi-futex wait syscall that can block on multiple futex addresses simultaneously. This is critical for Wine and Windows game emulation (which needs WaitForMultipleObjects semantics). It's the first piece of the broader "futex2" redesign, though the full replacement of the original SYS_futex omnibus syscall is still in progress.

14 — the mental model

After fourteen chapters of mechanisms, the most important question is: which tool do you reach for first? The answer depends on the nature of the sharing and the nature of the operation. Layered Correctness

Build concurrent systems in layers. The innermost layer uses primitives: atomics, futexes, spinlocks. Above that, synchronized containers and channels. Above that, higher-level patterns: worker pools, pipelines, actor models. Each layer should be obviously correct given the guarantees of the layer beneath. Never expose raw shared state across layers — encapsulate it behind a safe interface.

The Immutability Escape Hatch

The fastest lock is no lock at all. Immutable data structures can be read by any number of threads without synchronization. Copy-on-write (COW) extends this: when you need to modify, create a new copy, make your changes, then atomically swap the pointer. Readers continue using the old copy; the new copy becomes the current after a single CAS. This pattern appears in Linux's RCU (Read-Copy-Update), persistent data structures, and configuration updates in high-performance servers.

Testing Concurrent Code

Concurrent code requires deliberate testing. Unit tests that run once are insufficient — races have probabilistic windows. Best practices: run all tests with the race detector enabled; use stress tests that spawn many goroutines/threads and run thousands of operations; inject artificial delays at key points to widen race windows; use property-based testing to generate random operation sequences; and test shutdown paths (many races lurk in teardown).

Reach for first

  • Channels for producer-consumer
  • sync.Once / call_once for init
  • Atomic for counters and flags
  • RWLock for read-heavy data
  • Thread pool for CPU-bound tasks
  • async/await for I/O-bound tasks

Avoid until needed

  • Hand-rolled lock-free structures
  • Double-checked locking by hand
  • Spinlocks for long critical sections
  • SeqCst everywhere "to be safe"
  • Unbounded thread creation
  • Shared mutable global state
Final Wisdom

The goal is not to write the cleverest concurrent code. The goal is to write concurrent code that is obviously correct, efficiently enough. Start with the simplest correct solution (usually a mutex or a channel). Profile before optimizing to atomics or lock-free structures. Use the race detector on every CI run. And remember: the most reliable concurrent program is one where concurrent access is isolated to a small, well-defined, well-tested boundary — and everything else is comfortably sequential.
