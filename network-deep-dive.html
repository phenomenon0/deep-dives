THE NETWORK STACK

Every byte you send crosses seven layers of abstraction before it becomes light on a fiber. From socket syscalls to DMA ring buffers — the full journey._

// 01 — the socket

When you call socket(AF_INET, SOCK_STREAM, 0), the kernel builds a small empire of data structures on your behalf. A socket is not a connection. It is not a port. It is a file descriptor that happens to speak TCP. The syscall path is short and surgical: sys_socket() → sock_create() → inet_create(). Three function calls. About 10 microseconds. And when it returns, you have a number — say, 3 — that is your handle to a universe of kernel state.

Two Structs, One Socket

The kernel maintains two separate structures for every socket. The first is struct socket — the VFS (Virtual File System) interface. This is what makes sockets behave like files: you can read(), write(), select(), and close() them. The second is struct sock (often called sk) — the protocol-level state machine. This is where the real action lives.

The sk_prot field is a function pointer table. For TCP sockets it points to tcp_prot, for UDP it points to udp_prot. This is how the kernel implements polymorphism in C — a vtable, hand-rolled. When you call send(), the kernel doesn't check whether you have a TCP or UDP socket with an if-statement. It just calls sk->sk_prot->sendmsg(), and the right function runs.

Key Insight

Sockets are files. That is not a metaphor — it is a literal implementation detail. Every socket gets a struct file, an inode, and a dentry. This is why epoll, select, and read/write all work on sockets. The VFS does not care what is behind the file descriptor.

// 02 — the send path

You call send(fd, buffer, 1500, 0). That single line triggers a cascade that touches five layers of kernel code, copies your data once, adds 54 bytes of headers, and ends with a DMA engine firing photons down a wire. The whole thing takes about 2 microseconds on modern hardware. Let us trace every step.

Stage A: Syscall Entry

The send() libc wrapper executes syscall with __NR_sendto in RAX. The CPU transitions to Ring 0, loads the kernel stack, and dispatches to sys_sendto(). This function finds your socket via the file descriptor table: fdget(fd) → struct file → struct socket. Then it calls sock_sendmsg(), which dispatches through the protocol ops to tcp_sendmsg().

Stage B: TCP Layer

TCP's job is to copy your data into kernel buffers and segment it. First, copy_from_user() moves your 1500 bytes from userspace into kernel memory. Then TCP checks the Maximum Segment Size (MSS) — typically 1460 bytes for Ethernet (1500 MTU minus 20 bytes IP header minus 20 bytes TCP header). Your 1500 bytes become two segments: 1460 + 40. Each gets a TCP header prepended: source port, dest port, sequence number, ACK, window, checksum.

Stage C: IP Layer

Each segment descends to ip_queue_xmit(). The routing table is consulted: which interface? which gateway? The IP header is prepended: version, TTL (typically 64), protocol (6 = TCP), source IP, destination IP. If the resulting packet exceeds the MTU, IP fragmentation kicks in — but TCP already sized segments to avoid this.

Stage D: Link Layer

The Ethernet frame header is prepended: 6-byte destination MAC, 6-byte source MAC, 2-byte EtherType (0x0800 for IPv4). That is 14 bytes. The NIC will also append a 4-byte FCS (Frame Check Sequence) on transmit.

Stage E: DMA & Transmit

The network driver writes a TX descriptor to the NIC's transmit ring buffer. This descriptor contains the DMA address of the packet data and its length. The driver bumps the ring's tail pointer. The NIC notices, DMA-reads the packet data from system memory, and transmits it on the wire.

Cost Analysis

Total cost: 2 context switches (user → kernel → user), 1 data copy (copy_from_user), ~2 route lookups, and 2 DMA writes. The data copy is the expensive part — 1500 bytes at memory bandwidth is ~200ns. The syscall overhead itself is about 100ns. Everything else is pointer arithmetic.

Nagle's Algorithm & TCP_NODELAY

If you call send(fd, "H", 1, 0) followed by send(fd, "i", 1, 0), Nagle's algorithm will buffer the second byte and wait up to 200ms hoping more data arrives. It coalesces small writes into larger segments. This is great for throughput but terrible for latency-sensitive protocols like gaming, SSH, or HTTP/2. Disable it with setsockopt(fd, IPPROTO_TCP, TCP_NODELAY, &one, sizeof(one)).

The Nagle + Delayed ACK Disaster

When Nagle's algorithm (sender) meets delayed ACK (receiver), you get a 200ms latency spike on small writes. The sender buffers the second write, waiting for an ACK. The receiver delays its ACK, waiting for more data or a 200ms timer. Both sides are waiting for the other. This is the single most common networking performance bug in applications. Fix: set TCP_NODELAY on latency-sensitive sockets.

send() in Four Languages

// 03 — the receive path

Receiving is the mirror image of sending, but more complex because it is interrupt-driven and must handle arbitrary arrival order at line rate. A 10Gbps NIC can deliver 14.8 million packets per second. The kernel has about 67 nanoseconds to process each one. Here is how it manages.

Stage A: DMA Write

Before any packet arrives, the driver has pre-allocated a ring of page-sized buffers and told the NIC their DMA addresses. When a packet arrives, the NIC DMA-writes the frame directly into one of these pre-allocated buffers. No CPU involvement yet — the NIC owns the memory.

Stage B: Hardware Interrupt (MSI-X)

The NIC raises a hardware interrupt. On modern hardware this is MSI-X — Message Signaled Interrupts — which targets a specific CPU core. The interrupt handler is tiny: it disables further interrupts for this queue and schedules a NAPI poll. That is all. ~100 nanoseconds.

Stage C: NAPI Polling

The softirq handler calls net_rx_action(), which polls the NIC for completed RX descriptors. It processes up to a budget of 64 packets per poll cycle. This is the key optimization: instead of one interrupt per packet, NAPI batches them. At high load, the system never leaves poll mode — it is essentially busy-waiting, which is faster than interrupt overhead.

Stage D-E: sk_buff & Protocol Processing

For each received frame, the driver allocates an sk_buff and points it at the DMA buffer. Then the stack processes bottom-up: the Ethernet header is parsed (EtherType determines next protocol), the IP header is validated (checksum, TTL, routing), and the TCP header is processed (sequence numbers, ACK generation, window updates).

Stage F-G: Delivery & Wakeup

The completed sk_buff is appended to the socket's sk_receive_queue. If a process is sleeping in recv(), the kernel wakes it up via the socket's wait queue. The process then calls copy_to_user() to move data from kernel buffer to userspace.

GRO: Generic Receive Offload

Before protocol processing, GRO combines multiple small packets belonging to the same flow into one large sk_buff. If 10 back-to-back 1460-byte TCP segments arrive, GRO merges them into one 14600-byte sk_buff. The TCP stack processes it once instead of ten times. This is the receive-side equivalent of TSO.

Interrupt vs Polling: The NAPI Trade-off

At low packet rates, interrupts are efficient: the CPU sleeps between packets. At high rates, the interrupt overhead dominates. NAPI switches between modes dynamically:

< 1,000 ppsInterrupt-drivenNIC interrupts CPU per packet, CPU sleeps between
1,000 - 100,000 ppsHybridInterrupt triggers poll cycle, processes batch, re-enables interrupt
> 100,000 ppsBusy pollingCPU never stops polling, never re-enables interrupts

The 54-Byte Tax

Every TCP packet on Ethernet pays a 54-byte overhead: 14 (Ethernet) + 20 (IP) + 20 (TCP). For a 1500-byte MTU, your maximum payload is 1460 bytes — that is the MSS. For a 64-byte minimum Ethernet frame, you get just 10 bytes of payload. Small packets are catastrophically inefficient.

TIME_WAIT: The Silent Killer

A server handling 10,000 short-lived connections per second accumulates TIME_WAIT sockets at terrifying speed. Each one holds a 4-tuple (src IP, src port, dst IP, dst port) hostage for 60 seconds. That is 600,000 sockets in TIME_WAIT. You will run out of ephemeral ports (default range: 32768-60999 = ~28,000 ports). Fix: net.ipv4.tcp_tw_reuse = 1 lets the kernel reuse TIME_WAIT sockets for outgoing connections. Or use SO_REUSEADDR for servers.

The SYN Queue & Accept Queue

During the handshake, the kernel maintains two queues. The SYN queue (also called the half-open queue) holds connections in SYN_RECV state — the server has received a SYN and sent SYN-ACK but not yet received the final ACK. The accept queue (also called the completed connection queue) holds fully established connections waiting for the application to call accept(). The listen() backlog parameter controls the accept queue size.

SYN Flood Attack

An attacker sends thousands of SYN packets with spoofed source IPs. The server fills its SYN queue with SYN_RECV entries, each waiting for an ACK that will never arrive. Legitimate connections get dropped. Defense: SYN cookies (net.ipv4.tcp_syncookies = 1) encode the connection state in the sequence number itself, eliminating the need for a SYN queue entry entirely. The server only allocates resources after receiving a valid ACK.

// 06 — sk_buff internals

The sk_buff is the most important data structure in the Linux network stack. Every packet, in every direction, at every layer, is represented by one. The genius of its design is this: headers are prepended without copying the payload. Ever. The secret is four pointers and a block of pre-allocated headroom.

The Four Pointers

Every sk_buff has four pointers into a contiguous block of memory: head, data, tail, and end. The head and end pointers are fixed — they mark the allocated region. The data and tail pointers move as headers are added or removed.

sk_buff Memory Layout 

  head ──────► ┌──────────────────────────────┐
               │         headroom             │  ← reserved for headers
               │         (128 bytes)          │
  data ──────► ├──────────────────────────────┤
               │                              │
               │         packet data          │  ← actual content
               │                              │
  tail ──────► ├──────────────────────────────┤
               │         tailroom             │  ← unused
  end  ──────► └──────────────────────────────┘

The Zero-Copy Header Trick

When TCP wants to add its 20-byte header, it does not copy the payload and insert bytes before it. It simply moves the data pointer backward by 20 bytes and writes the TCP header into that space. The payload stays exactly where it is. Same for IP (20 bytes back) and Ethernet (14 bytes back). This is the skb_push() operation.

  tail ─► ├──────────────┤
  end  ─► └──────────────┘
Why This Matters

On a 10Gbps link processing 14.8 million packets per second, each avoided memcpy saves ~200ns. That is 2.96 seconds of CPU time saved per second. The skb_push() trick reduces header prepend from O(n) copy to O(1) pointer arithmetic. This is why the kernel pre-allocates headroom — it knows headers will be added later.

skb_pull(): The Receive Side

On the receive path, the operation is reversed. skb_pull(skb, 14) advances the data pointer forward by 14 bytes, effectively stripping the Ethernet header and exposing the IP header. The Ethernet bytes are still in memory — they are just "behind" the data pointer now. Each layer strips its own header and passes the rest upward.

TSO: Let the NIC Segment

With TCP Segmentation Offload, the kernel can hand the NIC a single sk_buff containing 65,536 bytes of payload. The NIC hardware splits it into 45 separate 1460-byte TCP segments, each with correct sequence numbers and checksums. One sk_buff in, 45 frames out. The CPU does the work of one packet instead of forty-five.

// 07 — raw sockets

Normal sockets are polite. They ask the kernel to handle TCP, IP, and Ethernet on their behalf. Raw sockets are not polite. With socket(AF_PACKET, SOCK_RAW, htons(ETH_P_ALL)), you bypass the entire protocol stack and talk directly to the network interface. Every packet on the wire arrives in your buffer — Ethernet header and all. This is how tcpdump, Wireshark, and every packet sniffer works.

BPF: Kernel-Side Packet Filtering

Receiving all packets is expensive. BPF (Berkeley Packet Filter) lets you attach a bytecode program to the socket. The kernel runs it on every packet before copying to userspace. Packets that don't match are dropped in-kernel. tcpdump compiles your filter expression (tcp port 80) into BPF bytecode automatically.

// BPF program: accept only TCP packets (IP protocol 6)
struct sock_filter code[] = {
    { 0x28, 0, 0, 0x0000000c },  // ldh [12]  — EtherType
    { 0x15, 0, 3, 0x00000800 },  // jeq 0x0800 (IPv4) → next, else skip 3
    { 0x30, 0, 0, 0x00000017 },  // ldb [23]  — IP protocol field
    { 0x15, 0, 1, 0x00000006 },  // jeq 6 (TCP) → accept, else drop
    { 0x06, 0, 0, 0x00040000 },  // ret 262144 — accept
    { 0x06, 0, 0, 0x00000000 },  // ret 0      — drop
};
struct sock_fprog prog = { 6, code };
setsockopt(raw, SOL_SOCKET, SO_ATTACH_FILTER, &prog, sizeof(prog));
Performance Impact

Without BPF filters, a raw socket on a busy 10Gbps link can flood userspace with 14.8M packets/sec, most of which your program doesn't care about. With a BPF filter, the kernel drops irrelevant packets at ~5ns each, never copying them. The difference: 100% CPU vs. 2% CPU.

// 08 — zero-copy networking

In the traditional send path, your data is copied at least once: from your userspace buffer into a kernel sk_buff via copy_from_user(). On a 10Gbps link saturating at 1.25 GB/s, that copy alone consumes an entire CPU core. Zero-copy techniques eliminate this, and each generation of the idea gets more radical.

1. sendfile() — File to Socket

Available since Linux 2.4. Moves data from a file directly to a socket without ever touching userspace. The kernel reads file pages into the page cache, then passes those pages directly to the NIC's DMA engine. The web server never sees the bytes.

Traditional: 4 Copies

disk → page cache     (DMA)
page cache → user buf (CPU)
user buf → sk_buff    (CPU)
sk_buff → NIC         (DMA)

sendfile(): 2 Copies

disk → page cache     (DMA)
page cache → NIC      (DMA)

Zero CPU copies!
2 syscalls → 1 syscall

2. splice() — Pipe-Based

Linux 2.6.17 introduced splice(), which moves data between two file descriptors through an in-kernel pipe buffer. No userspace copy. More flexible than sendfile — works between any pair of fds, not just file-to-socket.

3. io_uring — Batched Async I/O

Since Linux 5.1, io_uring lets you submit dozens of send/recv operations in a single syscall. Two shared ring buffers (submission queue and completion queue) between kernel and userspace, mapped with mmap. The kernel polls the SQ without any syscall at all when IORING_SETUP_SQPOLL is set. Zero syscalls per I/O operation.

4. MSG_ZEROCOPY — Direct DMA

Linux 4.14. Pass MSG_ZEROCOPY flag to send(). The NIC's DMA engine reads directly from your userspace buffer. No copy at all. The catch: the kernel must notify you when the DMA is complete so you don't modify the buffer. This comes via an ancillary message on the error queue.

5. XDP — Before sk_buff

Linux 4.8. XDP (eXpress Data Path) processes packets at the NIC driver level, before sk_buff allocation. An eBPF program runs on each raw packet and can drop, redirect, or pass it. No memory allocation. No protocol processing. Sub-microsecond per packet.

When Zero-Copy Hurts

Zero-copy is not free. MSG_ZEROCOPY has page-pinning overhead (~5μs) and completion notification cost. For sends smaller than ~10KB, the notification overhead exceeds the copy cost. sendfile() requires the data to be in a file. io_uring has a learning curve that makes your eyes bleed. Pick the right tool for your workload.

Why epoll Wins

With 10,000 connections, select() copies a 10,000-bit bitmap to kernel and back every call, then the kernel scans all 10,000 fds linearly. That is O(n) per call. poll() is the same but with a struct array instead of a bitmap. epoll uses a red-black tree to track fds and maintains a ready list. When you call epoll_wait(), it just returns the ready list — O(k) where k is the number of active connections, typically << n.

select/poll: O(n)

// Every call:
// 1. Copy n fds to kernel
// 2. Scan ALL n fds
// 3. Copy results back
// 10,000 fds → 10,000 checks
// Even if only 3 are ready

epoll: O(k)

// Setup once: O(log n) per fd
// Each epoll_wait:
// 1. Return ready list
// 2. Only k items (ready fds)
// 10,000 fds, 3 ready → 3 items
// No scanning, no copying

// 10 — the NIC & DMA

The Network Interface Card is not a dumb pipe. It is a small computer with its own processor, its own memory, and a DMA engine that reads and writes system RAM without CPU involvement. The interface between the driver and the NIC is a pair of ring buffers — one for transmit, one for receive. Understanding these rings is understanding the physical boundary between software and wire.

Ring Buffer Architecture

Interrupt Coalescing

Without coalescing, the NIC fires one interrupt per received packet. At 14.8M pps, that is 14.8M interrupts per second — the CPU spends all its time handling interrupts and never processes any packets. Interrupt coalescing batches them: "fire one interrupt after 100 microseconds or 32 received packets, whichever comes first."

RSS: Receive Side Scaling

Modern NICs have multiple RX queues (typically 8-32), each with its own ring buffer and interrupt targeting a different CPU core. The NIC hashes the packet's 4-tuple (src IP, dst IP, src port, dst port) using a Toeplitz hash and routes the packet to a queue based on the hash. This distributes load across cores without software intervention. All packets from the same flow always go to the same queue, preserving ordering.

Try This
# Check ring buffer sizes
ethtool -g eth0

# Check interrupt coalescing settings
ethtool -c eth0

# Check RSS hash configuration
ethtool -x eth0

# See per-queue statistics
ethtool -S eth0 | grep -E '(tx|rx)_queue'

# Increase ring buffer size (reduces drops under load)
ethtool -G eth0 rx 4096 tx 4096
Ring Size Trade-Off

Larger rings absorb traffic bursts without dropping packets but consume more memory (each RX slot holds a 2KB+ page) and increase latency (more packets buffered before processing). Default is often 256-512 descriptors. High-throughput servers typically set 2048-4096. Monitor ethtool -S eth0 | grep drop to know when you need bigger rings.

// 11 — congestion control

TCP does not blast data as fast as the NIC can transmit. It regulates itself with a congestion window (cwnd) that limits how much unacknowledged data can be in flight. The effective window is min(rwnd, cwnd) — the smaller of what the receiver advertises and what the sender's congestion algorithm allows. Getting this right is the difference between a 10Gbps link running at 10Gbps and running at 100Mbps.

Slow Start & Congestion Avoidance

Every TCP connection begins with slow start. The cwnd starts at 10 MSS (14,600 bytes since Linux 2.6.39, previously 1-4 MSS). Each ACK doubles the cwnd — exponential growth. When cwnd hits ssthresh (slow start threshold), TCP switches to congestion avoidance: linear growth, adding 1 MSS per RTT.

CUBIC vs BBR

CUBIC (Linux Default)

Loss-based. Uses a cubic function to probe for bandwidth. After a loss, it remembers the window size (Wmax) and uses a cubic curve to approach it again:

cwnd(t) = C(t - K)³ + W_max

K = cubic_root(W_max * β / C)
β = 0.7 (multiplicative decrease)
C = 0.4 (scaling constant)

Aggressive near Wmax, cautious elsewhere. Excellent for high-BDP links.

BBR (Google)

Model-based. Measures two things: maximum bandwidth (BtlBw) and minimum RTT (RTprop). Paces packets at BtlBw to keep exactly one BDP in flight:

rate = BtlBw
inflight = BtlBw × RTprop

Probes bandwidth: 1.25× for 1 RTT
Probes RTT: drains queue every 10s

Does not rely on packet loss. Keeps buffers unfilled. Better on lossy links.

Loss Detection & Recovery

Fast Retransmit: 3 duplicate ACKs trigger immediate retransmission without waiting for the RTO timer. SACK (Selective Acknowledgment) tells the sender exactly which segments were received, so it retransmits only the missing ones instead of everything after the gap. RTO (Retransmission Timeout) doubles with each timeout: 200ms → 400ms → 800ms → 1.6s → ... up to 120 seconds.

Window Scaling (RFC 1323)

TCP's window field is 16 bits — max 65,535 bytes. On a 10Gbps link with 50ms RTT, the bandwidth-delay product is 62.5 MB. Window scaling adds a shift count (0-14) negotiated during the handshake. A shift of 7 gives a max window of 65535 × 128 = ~8MB. A shift of 14 gives ~1GB.

Choosing a Congestion Algorithm

Use sysctl net.ipv4.tcp_congestion_control=bbr for internet-facing servers with lossy links. Keep CUBIC for data center traffic with clean links. BBR v2 fixes fairness issues with v1. Always measure before switching — ss -ti shows cwnd, rtt, and retransmits per connection.

// 12 — hardware offloads

Your NIC is not just a transceiver. Modern NICs are programmable packet processors that can compute checksums, segment large payloads, merge small packets, and distribute traffic across CPU cores. Every operation offloaded to hardware is one less thing the CPU has to do. On a 100Gbps NIC, hardware offloads are not optional — they are the only way to keep up.

TSO: TCP Segmentation Offload

Without TSO, the kernel creates one sk_buff per TCP segment (1460 bytes each). To send 64KB, that is 45 sk_buffs, 45 TCP headers, 45 IP headers. With TSO, the kernel hands the NIC one 64KB sk_buff with a single set of headers. The NIC's hardware segments it into 45 frames, computing unique sequence numbers and checksums for each. CPU savings: ~97%.

GRO: Generic Receive Offload

The inverse of TSO. When 45 back-to-back segments arrive from the same flow, GRO merges them into a single large sk_buff before passing it up the stack. The TCP layer processes one 64KB packet instead of 45 small ones. This reduces per-packet processing overhead by a factor proportional to the coalescing ratio.

Checksum Offload

Computing a TCP/UDP checksum over 1460 bytes costs ~200ns of CPU time. At 14.8M pps, that is 3 seconds of CPU per second. With checksum offload, the NIC computes the checksum in hardware on transmit and validates it on receive. The kernel writes a placeholder checksum and sets a flag in the sk_buff.

RSS & RPS

RSS (Receive Side Scaling) is the hardware version: the NIC hashes the 4-tuple and steers packets to different RX queues/CPUs. RPS (Receive Packet Steering) is the software fallback: a hash computed in the driver steers processing to a target CPU via inter-processor interrupt. RSS is faster (no IPI), but RPS works on NICs without multi-queue support.

TSO and GRO create packets larger than the MTU — up to 64KB. If you capture packets with tcpdump on the sending machine, you will see 64KB TCP segments that cannot possibly fit on the wire. // 13 — eBPF & XDP

Imagine injecting your own code into the kernel's packet processing pipeline. Not a kernel module — those can crash the system. A verified, sandboxed, JIT-compiled program that runs at near-native speed inside the kernel. That is eBPF. And when you attach it at the earliest possible point — the NIC driver's receive path, before sk_buff allocation — that is XDP. Together they let you process 10 million packets per second on a single core.

The eBPF Verifier

Before your program runs in the kernel, the verifier proves it is safe. No unbounded loops (all loops must have a provable upper bound). Every memory access is bounds-checked. No reading uninitialized memory. The verifier walks every possible execution path. If it cannot prove safety, your program is rejected. This is how eBPF avoids the kernel module stability problem.

XDP Program Example

#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>
#include <linux/if_ether.h>
#include <linux/ip.h>
#include <arpa/inet.h>

// Drop all TCP packets, pass everything else
SEC("xdp")
int xdp_drop_tcp(struct xdp_md *ctx) {
    void *data     = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;

    // Parse Ethernet header
    struct ethhdr *eth = data;
    if ((void *)(eth + 1) > data_end)
        return XDP_PASS;

    if (eth->h_proto != htons(ETH_P_IP))
        return XDP_PASS;

    // Parse IP header
    struct iphdr *ip = (void *)(eth + 1);
    if ((void *)(ip + 1) > data_end)
        return XDP_PASS;

    // Protocol 6 = TCP
    if (ip->protocol == 6)
        return XDP_DROP;  // Drop TCP at wire speed

    return XDP_PASS;
}

eBPF Maps

Maps are the communication channel between eBPF programs and userspace. Hash maps for lookup tables, ring buffers for event streaming, per-CPU arrays for lock-free counters. An XDP program can increment a per-CPU packet counter, and a userspace daemon can read the aggregated count — all without locks, without syscalls, at millions of operations per second.

eBPF Map Example: Per-IP Packet Counter

// Define a hash map: IP address → packet count
struct {
    __uint(type, BPF_MAP_TYPE_PERCPU_HASH);
    __uint(max_entries, 65536);
    __type(key, __u32);     // source IP
    __type(value, __u64);   // packet count
} ip_count SEC(".maps");

SEC("xdp")
int xdp_count(struct xdp_md *ctx) {
    struct ethhdr *eth = (void *)(long)ctx->data;
    if ((void *)(eth + 1) > (void *)(long)ctx->data_end)
        return XDP_PASS;
    if (eth->h_proto != htons(ETH_P_IP))
        return XDP_PASS;

    struct iphdr *ip = (void *)(eth + 1);
    if ((void *)(ip + 1) > (void *)(long)ctx->data_end)
        return XDP_PASS;

    __u32 src = ip->saddr;
    __u64 *count = bpf_map_lookup_elem(&ip_count, &src);
    if (count) {
        (*count)++;
    } else {
        __u64 init = 1;
        bpf_map_update_elem(&ip_count, &src, &init, BPF_ANY);
    }
    return XDP_PASS;
}

Theory without tools is philosophy. Here is the hacker's toolkit for understanding, debugging, and tuning the network stack. Every tool listed here ships with a standard Linux distribution or is one package install away. Master these and you can diagnose any network problem from application to wire.

Try This

The Debugging Flow

When something is wrong: (1) ss -tni to see connection states, buffer sizes, retransmits. (2) ethtool -S eth0 to check for NIC drops. (3) tcpdump to see actual packets. (4) strace to see what the application is doing. (5) bpftrace to trace kernel functions. This order goes from cheapest to most invasive. Start cheap.
