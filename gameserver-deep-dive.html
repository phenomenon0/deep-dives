01 — The Tick

Every multiplayer game is a lie. You are never seeing the real world — you are seeing the server's last snapshot, smoothed and interpolated and predicted to feel real. The truth lives on the server, and the server speaks in ticks. Twenty times per second, it inhales every player's input, simulates the entire world forward by one step, and exhales a new snapshot to everyone connected. That heartbeat — the tick — is the atom of online game time. Everything we build in this deep dive orbits around it.

The Fixed Timestep Loop

A naive game loop ties simulation to frame rate: update physics, render, repeat. On a fast machine you get 300 updates per second; on a slow one, 30. Physics diverges. A jump that clears a gap at 60fps falls short at 30fps because gravity accumulates differently. The fix is the fixed timestep accumulator:

Python
import time

TICK_RATE = 20            # ticks per second
TICK_DURATION = 1.0 / TICK_RATE  # 50ms per tick

accumulator = 0.0
previous = time.time()

while True:
    current = time.time()
    accumulator += current - previous
    previous = current

    while accumulator >= TICK_DURATION:
        process_inputs()      # read all buffered player inputs
        simulate_world()      # physics, AI, game logic
        broadcast_snapshots() # send new state to all clients
        accumulator -= TICK_DURATION

    sleep_remaining(accumulator)  # don't spin the CPU

The accumulator absorbs variable frame times. If the machine stalls for 150ms, the loop runs three ticks to catch up. If it runs fast, it sleeps. The simulation always advances in exact TICK_DURATION steps, so physics is deterministic regardless of hardware speed.

Server-Authoritative Model

The golden rule: the client proposes, the server disposes. When you press W to walk forward, your client sends "I pressed W at tick 4817." It does not send "I moved to position (340, 0, 782)." The server receives the input, validates it against game rules (can this player move? are they stunned? is there a wall?), applies it, and tells everyone where you actually ended up.

The Authoritative Server

Never trust the client. The client is a liar running in a process you do not control, on hardware you have never seen, modified by software you did not write. The server validates every input against every rule, every tick, no exceptions.

02 — The Socket Layer

Accept connections to our world. Ten thousand players want in, and every one of them needs a pipe to the server that can carry their inputs upstream and world state downstream. The choice of transport protocol shapes everything: latency, bandwidth, reliability, and how much code you write to compensate for what the protocol does not give you.

TCP: The Wrong Choice for Real-Time Games

TCP guarantees ordered, reliable delivery. Sounds perfect — until you understand the cost. TCP's head-of-line blocking means if packet 47 is lost, packets 48, 49, and 50 sit in the kernel buffer waiting for 47's retransmission, even though they contain newer data that the game desperately wants. In a game running at 20Hz, a single lost packet stalls the stream for 100-300ms while TCP retransmits. The player freezes, rubber-bands, and dies.

UDP: Control Your Own Destiny

UDP gives you a raw pipe: datagrams in, datagrams out, no ordering, no retransmission, no connection state. A lost packet is simply gone. For game state, this is ideal — old position data is worthless once newer data arrives. But we still need some reliability. Chat messages must arrive. Inventory changes must not vanish. So we build a thin reliability layer on top of UDP:

Protocol
// Our custom packet header (8 bytes)
struct PacketHeader {
    u16 sequence;      // this packet's sequence number
    u16 ack;           // last sequence we received from peer
    u32 ack_bits;      // bitfield: which of the 32 packets before ack we got
};
// sequence wraps at 65535 → handles ~54 minutes at 20 packets/sec
// ack_bits: bit 0 = ack-1, bit 1 = ack-2, ... bit 31 = ack-32

Every packet carries its own sequence number and acknowledges what it has received from the other side. The ack_bits field is a sliding window — 32 bits covering the last 32 packets. If the sender sees a gap in the ack bitfield, it knows that packet was lost and can selectively retransmit just the reliable messages from that packet. No head-of-line blocking. No waiting for old data.

// Simulate UDP reliable layer: sequence numbers + ack bitfield
class ReliableUDP {
  seq = 0;
  remoteSeq = 0;
  ackBits = 0;
  sent = new Map();
  acked = new Set();

  send(payload) {
    const pkt = { seq: this.seq, ack: this.remoteSeq, ackBits: this.ackBits, payload };
    this.sent.set(this.seq, payload);
    this.seq++;
    return pkt;
  }

  receive(pkt) {
    // Update our ack state
    if (pkt.seq > this.remoteSeq) {
      const shift = pkt.seq - this.remoteSeq;
      this.ackBits = (this.ackBits << shift) | (1 << (shift - 1));
      this.remoteSeq = pkt.seq;
    }
    // Check which of our packets were acked
    this.acked.add(pkt.ack);
    for (let i = 0; i < 32; i++) {
      if (pkt.ackBits & (1 << i)) this.acked.add(pkt.ack - 1 - i);
    }
    return pkt.payload;
  }
}

The UDP Tradeoff

You manage reliability yourself, but you control latency. TCP gives you a black box that sometimes stalls for 300ms. UDP gives you raw materials and says "build what you need." Every serious multiplayer game chooses UDP — or a protocol built on top of it (ENet, KCP, QUIC).

03 — State and Snapshots

How do we represent 10,000 players and a million objects in memory so the server can simulate them all in under 50 milliseconds? The answer is the Entity-Component-System architecture — a design pattern that trades object-oriented elegance for raw speed. In an MMORPG, speed is not optional.

Entity-Component-System (ECS)

In traditional OOP, a Player object inherits from Entity, has a Position field, a Health field, an Inventory field, and methods like move() and attack(). This works fine for 100 players. At 10,000, the CPU cache weeps. Each Player object is scattered across the heap. Iterating over all positions to do collision detection means chasing pointers through megabytes of cold memory.

ECS inverts the model. Entities are just IDs — unsigned 32-bit integers. Components are plain data structs stored in contiguous arrays, one array per component type. Systems are functions that iterate over component arrays. When the movement system runs, it reads the Position array and Velocity array sequentially — cache line after cache line, no pointer chasing, no vtable lookups.

  │ pos.x│pos.y│hp│mana│inv… │  Each object = scattered fields

Spatial Hashing for Interest Management

A player standing in the forest does not need to know about a battle happening three continents away. Interest management filters the world so each client only receives entities within their area of interest. The cheapest way to do this is a spatial hash: divide the world into a grid, hash each entity's position to a cell, and only send entities from nearby cells.

04 — Serialization

Encode the world in as few bytes as possible. Every tick, the server must serialize snapshots for thousands of clients. JSON is out of the question — field names like "position_x" repeated 10,000 times waste bandwidth that should carry actual data. Binary protocols with bit-packing are how real game servers communicate.

Why JSON Kills Game Servers

Consider a single player state in JSON:

JSON
{"id":4817,"x":341.25,"y":0.0,"z":782.50,"hp":85,"flags":5}

That is 65 bytes. The actual data is: 2 bytes for ID, 6 bytes for position (bit-packed), 1 byte for health, 1 byte for flags = 10 bytes. JSON wastes 55 bytes on field names, quotes, colons, and commas. Multiply by 50 nearby entities per client, 10,000 clients, 20 ticks per second: JSON costs 650 MB/s. Bit-packed binary costs 100 MB/s. That is the difference between a $50/month server and a $500/month server.

Bit-Packing

Instead of using a full 32-bit float for health (0.0-100.0), we use 7 bits (0-127). Instead of a 32-bit int for weapon ID when there are only 16 weapons, we use 4 bits. A boolean costs 1 bit, not 1 byte. The key insight: we know the range of every field at compile time, so we can pack perfectly.

Bit-Packing a Player State — Five Languages

#include <stdint.h>
#include <string.h>

typedef struct {
    uint8_t  *buf;
    int       bit_pos;
} BitWriter;

void write_bits(BitWriter *w, uint32_t val, int bits) {
    for (int i = bits - 1; i >= 0; i--) {
        int byte_idx = w->bit_pos / 8;
        int bit_idx  = 7 - (w->bit_pos % 8);
        if (val & (1u << i))
            w->buf[byte_idx] |= (1 << bit_idx);
        w->bit_pos++;
    }
}

// Pack: entity_id(16) + x(12) + y(12) + z(12) + hp(7) + weapon(4) + alive(1) = 64 bits = 8 bytes
void pack_player(BitWriter *w, uint16_t id,
                  float x, float y, float z,
                  int hp, int weapon, int alive) {
    write_bits(w, id, 16);
    write_bits(w, (uint32_t)(x / 4096.0f * 4095), 12); // 0-4096m in 12 bits
    write_bits(w, (uint32_t)(y / 256.0f * 4095), 12);  // 0-256m height
    write_bits(w, (uint32_t)(z / 4096.0f * 4095), 12);
    write_bits(w, hp & 0x7F, 7);      // 0-127 HP
    write_bits(w, weapon & 0xF, 4);   // 16 weapons
    write_bits(w, alive & 1, 1);      // dead or alive
}
class BitWriter:
    def __init__(self):
        self.bits = []

    def write(self, val, n_bits):
        for i in range(n_bits - 1, -1, -1):
            self.bits.append((val >> i) & 1)

    def to_bytes(self):
        while len(self.bits) % 8:
            self.bits.append(0)
        result = bytearray()
        for i in range(0, len(self.bits), 8):
            byte = 0
            for b in range(8):
                byte = (byte << 1) | self.bits[i + b]
            result.append(byte)
        return bytes(result)

def pack_player(w, eid, x, y, z, hp, weapon, alive):
    w.write(eid, 16)
    w.write(int(x / 4096 * 4095), 12)
    w.write(int(y / 256 * 4095), 12)

class BitWriter {
  bits: number[] = [];

  write(val: number, nBits: number) {
    for (let i = nBits - 1; i >= 0; i--)
      this.bits.push((val >>> i) & 1);
  }

  toBytes() {
    while (this.bits.length % 8) this.bits.push(0);
    const out = [];
    for (let i = 0; i < this.bits.length; i += 8) {
      let byte = 0;
      for (let b = 0; b < 8; b++) byte = (byte << 1) | this.bits[i + b];
      out.push(byte);
    }
    return out;
  }
}

    fn write(&mut self, val: u32, n_bits: u32) {

            self.bits.push(((val >> i) & 1) as u8);
        }
    }

        let mut out = vec![];
        for chunk in self.bits.chunks(8) {

    }

Delta Compression

Even with bit-packing, sending full state for 50 entities every tick is wasteful when most of them have not moved. Delta compression sends only what changed since the last acknowledged snapshot. If a player is standing still, their position delta is zero — we send a single "no change" bit. Quake III Arena famously used delta compression to reduce per-player bandwidth from ~100 KB/s to ~14 KB/s.

05 — Client Prediction

Make it feel instant even with 80ms latency. Without prediction, every keypress travels to the server (40ms), gets processed, and the result travels back (40ms). That is 80ms of input lag at minimum — enough to make movement feel sluggish and combat feel unresponsive. The solution: don't wait. Apply the input locally immediately, then reconcile when the server's authoritative answer arrives.

The Prediction Loop

Client-side prediction works in three steps. First, when the player presses a key, apply the movement locally using the same physics code the server runs. Second, buffer every input with its tick number and send it to the server. Third, when the server sends back the authoritative state for a given tick, compare it to our predicted state at that tick. If they match, great. If they differ, snap to the server state and replay all inputs from that tick forward to arrive at a corrected present state.

The key insight from the Source engine (Half-Life 2, CS:GO, TF2): keep an unacknowledged input buffer. Every input you send gets a sequence number. The player feels a gentle "drift" instead of a jarring snap. Overwatch calls this "favor the shooter" on the attack side, but on the movement side it is simply good prediction smoothing.

06 — Entity Interpolation

Make other players look smooth. The server sends updates at 20Hz, but we render at 60fps — three frames between each server update. Without interpolation, other players would teleport between positions 20 times per second, looking like a slideshow. We fix this by rendering other entities in the past, interpolating between two known server states.

The Interpolation Buffer

We maintain a buffer of the last several server snapshots, each timestamped. Instead of rendering at "now," we render other entities at now - 100ms. At any moment, we have two snapshots that bracket our render time, and we linearly interpolate between them.

Why Not Predict Other Players?

You might wonder: why not predict other players the same way we predict ourselves? Because we have our own inputs — we know exactly what we intend to do. We do not have other players' inputs. Predicting their movement means guessing, and guesses are wrong every time they change direction. A mispredicted enemy would jitter and snap constantly. Interpolation with a 100ms delay is far smoother than inaccurate prediction.

Jitter Absorption

Network jitter means snapshots do not arrive at perfectly regular intervals. One might come after 45ms, the next after 62ms. The 100ms interpolation delay acts as a jitter buffer — even if a snapshot is 30ms late, we still have enough buffered data to interpolate smoothly. If the buffer runs dry (extreme lag), we extrapolate briefly and hope the next snapshot arrives before the error becomes visible.

The 100ms Tradeoff

Everything you see about other players happened 100 milliseconds ago. This is the price of smooth rendering. In an MMORPG running at 20Hz, 100ms is exactly two tick intervals — the minimum delay that guarantees we always have two snapshots to interpolate between. Competitive shooters at 128Hz reduce this to ~16ms, but for an MMO, 100ms is imperceptible.

07 — Lag Compensation

Fair combat for players with different pings. You aimed your crosshair at the enemy, you clicked at exactly the right moment, and the shot should hit. But the enemy you see on your screen is 100ms in the past (interpolation delay) plus your own network latency. On the server, that enemy has already moved. Without lag compensation, every shot would miss by the distance an enemy travels in 100-200ms. The game would feel broken.

Rewind and Replay

The server stores a history of world states — one per tick, typically the last 400ms worth. When a "fire" input arrives from a client, it carries a timestamp: "I fired at tick 4820." The server rewinds the world to tick 4820, positions all entities where they were at that moment, performs the hit detection, then fast-forwards back to the present and applies the damage.

The Compensation Window

Most games cap the lag compensation window at 200-250ms. Beyond that, the rewind is so far in the past that hits feel unfair to the victim. If a player's effective latency (ping + interpolation delay) exceeds the window, their shots start missing — the server refuses to rewind that far. This is why high-ping players in competitive games are at a genuine disadvantage.

Write-Ahead Log (WAL)

We never write directly to the database from the game loop — database writes are slow and unpredictable. Instead, every state change appends to a Write-Ahead Log: a sequential, append-only file on fast local storage. A background writer drains the WAL into the database in batches. If the server crashes mid-tick, we replay the WAL on restart to recover everything up to the last completed write.

The Five-Second Rule

Our persistence target: if the server crashes, a player loses at most 5 seconds of progress. Position saves every 5 seconds. Inventory and stats batch-write every 1-2 seconds via WAL. This means the WAL must be on durable storage (not tmpfs), and we must fsync after each batch append. The cost is ~1ms per fsync on NVMe — well within our tick budget since it happens off the game thread.

The 5-Second Rule

If the server crashes, you lose at most 5 seconds of progress. This is the persistence contract. It means: WAL on durable storage with fsync, background batch writes every 1-2 seconds, position snapshots every 5 seconds, and crash recovery that replays the WAL before accepting new connections.

10 — Sharding the World

One server cannot hold 10,000 players. A single core processes the game loop sequentially — every entity every tick. At 10,000 players with 50 nearby entities each, that is 500,000 interest-management checks, 10,000 input validations, thousands of physics updates, all in 50ms. Even modern hardware struggles. The answer: split the world across multiple servers.

Zone-Based Sharding

The simplest sharding model: divide the game world into geographic zones, and assign each zone to its own server process. A forest zone runs on server A, a desert zone on server B, a dungeon on server C. Each server handles only the players and NPCs in its zone. With 20 zones averaging 500 players each, no single server exceeds its tick budget.

The Boundary Problem

What happens when Player A stands at the edge of the Forest zone and Player B stands at the edge of the Plains zone, 20 meters apart? They should see each other. But A is on Server A and B is on Server B. Neither server knows about the other's entities.

The solution is a boundary overlap: each zone server loads entities within 100m of its borders from neighboring zones. These "ghost" entities are read-only copies updated via a fast inter-server channel. When Player A approaches the boundary, Server A starts receiving B's position from Server B. When A crosses the boundary, a handoff occurs:

EVE Online's Time Dilation

What happens when 6,000 players converge on a single zone for a massive battle? The zone server cannot simulate 6,000 players in 50ms. EVE Online's solution is elegant and radical: slow down time. When a zone is overloaded, TiDi (Time Dilation) reduces the tick rate. At 10% TiDi, one second of game time takes 10 real seconds. Everything moves in slow motion, but nothing drops, nothing lags, and every action is processed fairly.

Time Dilation

EVE Online runs battles at 10% speed rather than dropping players. When the zone server cannot maintain its tick rate, it dilates time — the entire zone runs in slow motion. It is a brilliant trade: players accept slow-motion combat because the alternative (disconnections, rubber-banding, lost actions) is worse. Time is not sacred; fairness is.

11 — Matchmaking

Get players into fair games. Whether it is an arena match, a dungeon group, or a PvP battleground, players expect opponents of similar skill. Too hard and they quit in frustration; too easy and they get bored. Matchmaking walks a tightrope between match quality and queue time, using rating systems borrowed from chess and refined by decades of game development.

Elo Rating

The Elo system, invented for chess, is the foundation. Each player has a numeric rating (starting at 1000 or 1500). After a match, the winner gains points and the loser loses points. The amount depends on the expected outcome: beating a higher-rated player earns more points than beating a weaker one.

The expected score for player A against player B:

Math
E_A = 1 / (1 + 10^((R_B - R_A) / 400))

// If R_A = 1500, R_B = 1700:
// E_A = 1 / (1 + 10^(200/400)) = 1 / (1 + 3.16) = 0.24
// A is expected to win 24% of the time
// If A wins: new_R_A = 1500 + K * (1 - 0.24) = 1500 + 32*0.76 = 1524
// If A loses: new_R_A = 1500 + K * (0 - 0.24) = 1500 - 7.7 = 1492

The Queue Dilemma

Tight skill matching means long wait times. Loose matching means stomps. Every matchmaker balances this trade: start with a narrow skill range and widen it over time. 12 — Anti-Cheat

Trust nothing from the client. Every packet that arrives at the server is a potential lie. Speed hacks report movement faster than allowed. Teleport hacks send impossible position deltas. Aimbots produce inhuman aim patterns. Wallhacks exploit the fact that the server sends data about enemies the client should not see. Our job is to detect, prevent, and punish — knowing we can never win completely.

Server-Side Detection

The most reliable anti-cheat runs on the server, where cheaters cannot tamper with it. Every tick, we validate:

Wallhack Mitigation: Visibility Culling

The best defense against wallhacks is to never send hidden players to the client. If an enemy is behind a wall, the server performs a visibility check (raycast or PVS — Potentially Visible Set) and simply omits that player from the snapshot. The client cannot render what it never received. This costs CPU on the server (raycasts are expensive), but it completely eliminates wallhacks for occluded players.

Graceful Shutdown

When a zone server needs to restart, it follows a strict sequence: stop accepting new connections, broadcast "server restarting in 60s" to all players, continue processing ticks normally, at T-10s begin saving all player state to the database (bypassing the WAL for direct writes), at T-0 close all connections and exit. Players reconnect to a fresh instance that loads their state from the database. Total downtime per player: 5-15 seconds.

The Tick Budget

If your tick takes longer than 50ms, the game stutters. Every feature you add to the game loop — AI, physics, interest management, anti-cheat checks — steals from this budget. Profile every tick. Know where every millisecond goes. When you hit 40ms average, stop adding features and start optimizing. The tick budget is the hard ceiling of your server's capability.

Login Flow

A player launches the game client. The client connects to the auth server (Chapter 9) over HTTPS, authenticates with username/password or OAuth, and receives a session token. The auth server queries PostgreSQL for the player's last known zone and position. The client is directed to the appropriate zone server (Chapter 10). The client opens a UDP connection (Chapter 2) to that zone server, presenting the session token. The zone server verifies the token with Redis, spawns the player entity in the ECS (Chapter 3), and begins including them in the tick loop (Chapter 1).

Play Flow

Every tick (Chapter 1), the zone server reads all buffered UDP inputs (Chapter 2), validates them against game rules and anti-cheat checks (Chapter 12), advances the ECS simulation (Chapter 3), performs interest management to determine who sees what, serializes per-client snapshots with delta compression (Chapter 4), and broadcasts them over UDP. The client deserializes the snapshot, interpolates other entities (Chapter 6), reconciles its own predicted position (Chapter 5), and renders the frame.

Combat Flow

A player fires their weapon. The client predicts a hit locally (Chapter 5) and sends the input with a timestamp to the server. The server receives the input, rewinds the world to the shooter's timestamp (Chapter 7), performs server-side hit detection against rewound hitboxes, validates the damage against anti-cheat rules (Chapter 12), applies the damage to the target's health component in the ECS (Chapter 3), and broadcasts the result in the next snapshot. The target's client interpolates the health change (Chapter 6).

Chat Flow

A player types "/g Need healer for raid." The message hits the zone server, which sanitizes input, applies the rate limiter (Chapter 8), and routes it to the chat bus. The chat bus looks up guild membership in PostgreSQL, determines which zone servers host online guild members, and fans the message out. Each zone server delivers it to the relevant clients in their next snapshot.

Save Flow

State changes (inventory updates, quest progress, position) append to the zone server's WAL (Chapter 9). A background writer batches WAL entries and writes to PostgreSQL every 1-2 seconds. Position snapshots write to Redis every 5 seconds. On crash, the replacement zone server replays the WAL to recover all state up to the last batch.

Monitor Flow

Every zone server exports tick duration, entity count, player count, and latency percentiles to Prometheus (Chapter 13). Grafana dashboards visualize all zones. Alerts fire when tick rate drops below 18Hz or P99 latency exceeds 250ms. Deploys roll out zone-by-zone with canary testing. The ops team can time-dilate an overloaded zone (Chapter 10) or spin up additional dungeon instances from the matchmaker (Chapter 11).

What We Built

A server-authoritative MMORPG handling 10,000 concurrent players across sharded zones, with UDP networking, ECS state management, binary serialization with delta compression, client-side prediction and entity interpolation, lag-compensated combat, rate-limited chat, WAL-backed persistence, Elo-based matchmaking, server-side anti-cheat, and zero-downtime deploys. Every piece exists because the one before it was not enough.

What We Didn't Cover

Physics engines (Bullet, PhysX) for collision and ragdoll. Audio spatialization (3D sound positioning over the network). Streaming open worlds (loading terrain and assets dynamically as players move). Player-driven economies (auction houses, crafting markets, inflation control). Modding and scripting (Lua/WASM sandboxes for user-generated content). AI navigation (navmeshes, pathfinding for thousands of NPCs). Each of these is its own deep dive. The server architecture we built is the foundation they all sit on.
