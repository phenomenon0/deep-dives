// 01 — ASCII — The Simple Days

In 1963, a committee of engineers at Bell Labs and the American Standards Association sat down and asked: what is the minimum viable encoding for a computer to speak to a teletype machine? The answer was 7 bits, 128 characters, and a remarkable act of cultural hubris that would haunt us for the next sixty years. They called it ASCII — the American Standard Code for Information Interchange — and for American English, it was perfect.

The 7-bit layout was deliberate engineering. Bit 7 (the high bit) was left free as a parity check bit — a simple error-detection scheme where you flip one bit to make the total count of 1s either even or odd. This meant every ASCII character fit neatly in the low 7 bits of a byte, with the high bit always zero. Elegant, simple, and wrong in exactly one way: it only covered 128 characters, which was not remotely enough for the rest of Earth's writing systems.

Zen: 7 Bits Was Enough for America

7 bits was enough for America, and that was the problem. ASCII made a technical assumption that was also a political statement: that the characters needed to communicate in computing were the characters of English. Every subsequent standard — code pages, ISO-8859, Unicode itself — exists because that assumption was incorrect for roughly six billion people. The simplest designs have the longest shadows.

// 02 — The Tower of Babel

By the 1980s, computers had spread far beyond the English-speaking world, and software vendors needed to handle German umlauts, French accents, Cyrillic alphabets, Greek letters, and Hebrew. Their solution was both ingenious and catastrophic: take that unused high bit of the byte (remember, ASCII only uses 7 bits), and use it to squeeze in 128 more characters. The result was "code pages" — region-specific character tables where the same byte value could mean completely different things depending on which country your computer was configured for.

IBM had Code Page 437 for the US (with box-drawing characters beloved by BBS artists). Western Europe got ISO 8859-1. Russia got ISO 8859-5. Turkey got ISO 8859-9. There were eventually dozens of incompatible encodings, all sharing the same low 128 bytes (ASCII) and all disagreeing about what bytes 128-255 meant. Sending text between systems meant knowing which code page the sender used. Most of the time, nobody did.

Danger: Mojibake

The Japanese have a word for this: mojibake (文字化け) — "character transformation." When text encoded in one encoding is decoded with a different one, you get garbage. Not random garbage: consistent garbage, because the wrong mapping is applied deterministically. Mojibake is still very much alive today wherever systems forget to declare or negotiate encoding, which is more places than you'd hope.

Wild: The ï»¿ Signature

You may have seen files start with the three bytes EF BB BF, displayed as ï»¿ when opened in a Latin-1 editor. That is the UTF-8 encoding of U+FEFF, the Byte Order Mark — a marker some Windows tools prepend to UTF-8 files. It is technically unnecessary (UTF-8 has no byte order), but Windows Notepad wrote it anyway for years, and now it haunts every CSV import pipeline that was not expecting it.

// 03 — Unicode Arrives

In 1991, a group of engineers from Apple, Xerox, and a consortium of software companies published the first version of a radical idea: one unified character set for every writing system on Earth. They called it Unicode. The central promise was simple to state and enormous to deliver — every character in every human language would get a unique number, called a code point, and that number would mean the same thing on every computer everywhere forever. No more code pages. No more mojibake. One encoding to rule them all.

Unicode's address space is organized into 17 planes, each containing 65,536 code points. That gives a total address space of 1,114,112 possible code points, from U+000000 to U+10FFFF. As of Unicode 15.1, about 149,813 of those are assigned. The rest are reserved for future use, private use areas, and the fervent hope that someone eventually digitizes Linear Elamite completely.

Key Insight: Code Points Are Not Characters

Unicode assigns numbers to code points, not characters in the intuitive sense. The letter "é" can be represented as a single code point (U+00E9, "latin small letter e with acute") or as two code points (U+0065 "e" + U+0301 "combining acute accent"). Both are valid. They look identical when rendered. This distinction — between code points and what humans actually see as a "character" — is one of the deepest traps in string programming, and we will excavate it thoroughly in the graphemes section.

// 04 — UTF-8 — The Clever Hack

In September 1992, Ken Thompson and Rob Pike (two of the creators of Unix) sat down at a diner in New Jersey and sketched an encoding on a placemat. The problem they were solving for Plan 9 — their new operating system — was how to store Unicode code points in a way that was backward-compatible with existing ASCII programs, used no null bytes except for U+0000, and was self-synchronizing so you could restart reading from any byte position. They finished the design over dinner, implemented it the same night, and had it passing tests by the next morning. The encoding they invented was UTF-8, and it now carries the majority of all text on the internet.

The key insight: not every code point needs the same number of bytes. ASCII characters are common, so give them one byte. Rare characters can afford four bytes. Use bit patterns in the leading byte to communicate how many bytes follow. The result is a variable-length encoding that is simultaneously backward-compatible with ASCII and capable of representing all 1,114,112 Unicode code points.

The Bit Patterns

UTF-8 Byte Sequence Encoding Rules 
  Code point range      Byte 1      Byte 2      Byte 3      Byte 4
  ─────────────────     ────────    ────────    ────────    ────────
  U+0000  – U+007F      0xxxxxxx

  U+0800  – U+FFFF      1110xxxx    10xxxxxx    10xxxxxx
  U+10000 – U+10FFFF    11110xxx    10xxxxxx    10xxxxxx    10xxxxxx

  Leading byte patterns:
    0xxxxxxx  →  1-byte sequence (ASCII, 0-127)
    110xxxxx  →  start of 2-byte sequence
    1110xxxx  →  start of 3-byte sequence
    11110xxx  →  start of 4-byte sequence
    10xxxxxx  →  continuation byte (NEVER a leading byte)

  The x bits carry the actual code point value, concatenated:
    U+00E9 ('é') = 11101001 in binary
    Fits in 2-byte range (U+0080–U+07FF)
    Split: 110_00011 10_101001  →  0xC3 0xA9

  Self-synchronization trick:
    If you are lost in the middle of a stream, scan forward.
    The FIRST byte that is NOT 10xxxxxx is a leading byte.
    You have found the start of a character. Maximum scan: 3 bytes.

Why Self-Synchronization Matters

This is the property that makes UTF-8 so robust. If you receive a truncated or corrupted byte stream, you can resync by scanning for the next leading byte — any byte that does not match 10xxxxxx. You never get stuck. Compare this to older encodings like Shift-JIS where losing a single byte could misalign the entire rest of the file, turning every subsequent character into garbage.

The continuation byte pattern 10xxxxxx also means that UTF-8 byte sequences for different code points never overlap. A byte that starts a sequence cannot appear as a continuation byte. This property, called byte-level safety, means you can search for ASCII substrings inside UTF-8 text using standard byte-oriented tools like grep or strstr without accidentally matching inside a multi-byte sequence.

C#include <stdio.h>
#include <stdint.h>
#include <string.h>

/* Encode a single Unicode code point to UTF-8.
   Returns number of bytes written (1-4), or 0 on error. */
int encode_utf8(uint32_t cp, uint8_t *out) {
    if (cp < 0x80) {
        out[0] = (uint8_t)cp;
        return 1;
    } else if (cp < 0x800) {
        out[0] = 0xC0 | (cp >> 6);
        out[1] = 0x80 | (cp & 0x3F);
        return 2;
    } else if (cp < 0x10000) {
        out[0] = 0xE0 | (cp >> 12);
        out[1] = 0x80 | ((cp >> 6) & 0x3F);
        out[2] = 0x80 | (cp & 0x3F);
        return 3;
    } else if (cp <= 0x10FFFF) {
        out[0] = 0xF0 | (cp >> 18);
        out[1] = 0x80 | ((cp >> 12) & 0x3F);
        out[2] = 0x80 | ((cp >> 6)  & 0x3F);
        out[3] = 0x80 | (cp & 0x3F);
        return 4;
    }
    return 0;  // invalid code point
}

int main(void) {
    uint8_t buf[4];
    int n;

    n = encode_utf8(0x1F4A9, buf);  // 💩
    printf("U+1F4A9: %d bytes:", n);

        printf(" %02X", buf[i]);
    printf("\n");

// 05 — UTF-16 & UTF-32 — The Other Encodings

UTF-16 is the encoding that Java and JavaScript use internally, the one Windows chose for its entire API surface, and the one that haunts every programmer who has ever dealt with surrogate pairs at 2am wondering why their string is corrupted. It made sense in 1991. Unicode was supposed to fit in 16 bits. Then Unicode expanded, and suddenly every Windows developer was living in a world where a single character might cost 2 code units instead of one.

UTF-16 uses 2 bytes for characters in the Basic Multilingual Plane (U+0000 through U+FFFF) — that is most of the characters you will ever encounter. For everything beyond U+FFFF, the supplementary planes, UTF-16 uses a surrogate pair: two 2-byte units that together encode one code point. Variable width, but only two possible widths.

Surrogate Pairs

The Unicode Consortium carved out a chunk of the BMP — U+D800 through U+DFFF — specifically for surrogate use. These code points are illegal as standalone characters. They exist only to pair up and encode supplementary plane code points.

Surrogate Pair Encoding 
  High surrogates: U+D800 – U+DBFF  (1024 values)
  Low  surrogates: U+DC00 – U+DFFF  (1024 values)
  Together: 1024 * 1024 = 1,048,576 supplementary code points

  To encode code point U (where U > U+FFFF):

    U' = U - 0x10000           (0x00000 – 0xFFFFF, 20 bits)
    High = 0xD800 + (U' >> 10)   (top 10 bits)
    Low  = 0xDC00 + (U' & 0x3FF) (bottom 10 bits)

  Example: U+1F600 GRINNING FACE 😀

    U' = 0x1F600 - 0x10000 = 0xF600
    High = 0xD800 + (0xF600 >> 10)  = 0xD800 + 0x3D = 0xD83D
    Low  = 0xDC00 + (0xF600 & 0x3FF) = 0xDC00 + 0x200 = 0xDE00

    UTF-16: D8 3D DE 00  (4 bytes)

To decode a surrogate pair back to a code point:

Surrogate Pair Decoding 
  Given High surrogate H and Low surrogate L:

    U = 0x10000 + ((H - 0xD800) << 10) + (L - 0xDC00)

  Example: D83D DE00
    H = 0xD83D, L = 0xDE00
    U = 0x10000 + ((0xD83D - 0xD800) << 10) + (0xDE00 - 0xDC00)
    U = 0x10000 + (0x3D << 10) + 0x200
    U = 0x10000 + 0xF400 + 0x200
    U = 0x1F600  ✔

BOM: Byte Order Mark

U+FEFF is the BOM. Placed at the start of a UTF-16 file, it tells you the byte order of everything that follows. If you read FE FF, you are in big-endian territory. If you read FF FE, little-endian. The BOM is invisible in a text editor, but it is there, silently deciding whether your file is readable.

BOM Detection 
  FF FE xx xx ...   UTF-16 Little-Endian  (most common on Windows)
  FE FF xx xx ...   UTF-16 Big-Endian
  EF BB BF xx ...   UTF-8 with BOM (optional, discouraged on Linux/Mac)
  FF FE 00 00 ...   UTF-32 Little-Endian
  00 00 FE FF ...   UTF-32 Big-Endian

  UTF-8 does NOT need a BOM. It is byte-order-independent by design.
  The UTF-8 BOM is a Windows legacy artifact that causes subtle bugs
  in Unix tools: grep, cat, awk, and virtually every shell script.

UTF-32: The Simple Option Nobody Uses

UTF-32 is exactly what it sounds like. Every code point is exactly 4 bytes, always, no exceptions. U+0041 "A" is stored as 00 00 00 41. U+1F600 😀 is stored as 00 01 F6 00. Dead simple to index into. Absolutely brutal on memory.

UTF-32 Memory for "Hi 😀" 
  H  (U+0048): 00 00 00 48
  i  (U+0069): 00 00 00 69
     (U+0020): 00 00 00 20
  😀  (U+1F600): 00 01 F6 00

  4 characters * 4 bytes = 16 bytes
  In UTF-8:  H=1, i=1, space=1, 😀=4 = 7 bytes
  UTF-32 costs 2.3x more for this example
  For pure ASCII, UTF-32 costs 4x more than UTF-8

UTF-32 is occasionally used internally by language runtimes that want O(1) random access into strings by code point index. Python 3's internal string representation uses UTF-32 (or a narrower variant) when a string contains characters above U+00FF.

Wild: The Windows Trap

Windows chose UTF-16 in the 1990s when Unicode was supposed to fit in 16 bits. Then Unicode expanded beyond 16 bits. Now Windows is stuck with the worst of both worlds — variable-width like UTF-8 but with none of the ASCII compatibility. Every Windows API has both an "A" (ANSI/narrow) and "W" (wide/UTF-16) variant. // 06 — Grapheme Clusters — When .length Lies

Every string length function in every programming language lies to you. Not occasionally. Not in edge cases. Constantly, by design. len() does not tell you how many characters are in a string. .length does not tell you how many characters you will see on screen. They tell you something technically true but practically misleading — and understanding the difference is the gap between a programmer who handles text correctly and one who ships broken software to international users.

The root problem is that Unicode separates the concept of a code point (a number assigned to an abstract character) from a grapheme cluster (what a human perceives as a single visual character). These are not the same thing, and the gap between them grows wider with every new emoji and every new combining character sequence.

The é Problem

The letter é — e with an acute accent — exists in two forms in Unicode:

Two Ways to Write é 
  NFC (precomposed):
    U+00E9  LATIN SMALL LETTER E WITH ACUTE
    UTF-8:  C3 A9  (2 bytes)

  NFD (decomposed):
    U+0065  LATIN SMALL LETTER E
    U+0301  COMBINING ACUTE ACCENT
    UTF-8:  65 CC 81  (3 bytes)

  Both render as:  é
  Both are canonically equivalent in Unicode.
  But most languages treat them as different strings.

Same glyph. Same meaning. Different bytes. Different lengths. Most string comparison functions will say these two strings are not equal. This is how you get bugs where a user can log in on Mac (which normalizes to NFC) but not on Linux (which does not normalize).

Measuring String Length Correctly

#include <string.h>
#include <stdio.h>

// C has no grapheme cluster support in stdlib.
// strlen() counts BYTES, not characters.
// For anything Unicode-aware, you need ICU or a similar library.

int main() {
    // "café" in UTF-8: c=1, a=1, f=1, é=2 bytes
    const char *s = "caf\xc3\xa9";

    printf("strlen: %zu\n", strlen(s));   // 5 — counts bytes!

    // To get code point count, you must walk the UTF-8 yourself:
    size_t cp_count = 0;
    for (const unsigned char *p = (const unsigned char *)s; *p; p++) {
        // Count bytes that are NOT continuation bytes (10xxxxxx)
        if ((*p & 0xC0) != 0x80) cp_count++;
    }
    printf("code points: %zu\n", cp_count);  // 4

    // Grapheme clusters: use libicu's UBreakIterator
    // #include <unicode/ubrk.h>
    // UBreakIterator *bi = ubrk_open(UBRK_CHARACTER, "en", ...)
    return 0;
}

// 07 — Emoji — Wingdings to World Domination

Shigetaka Kurita was a designer at NTT DoCoMo in 1999 when he created 176 tiny 12×12 pixel images for the company's i-mode mobile internet platform. He was solving a mundane UX problem: the weather forecast widget needed icons. He drew a sun. He drew a cloud. He drew a face. He did not plan to change human communication forever. He just needed a weather widget.

Those 176 pixelated images — which included faces, weather symbols, hearts, and a pile of poo — became so popular that NTT DoCoMo's competitors launched their own emoji sets almost immediately. By the early 2000s, Japan had three major carriers and three completely incompatible emoji standards. Sending an emoji between networks was a game of telephone played entirely in corrupted data.

The Carrier Wars

DoCoMo, au (KDDI), and SoftBank each had their own mapping tables, their own visual styles, and their own proprietary encoding schemes. A 😄 on DoCoMo might map to a 😀 on SoftBank, or to a garbled question mark, or simply disappear. Cross-carrier messaging was a nightmare, and nobody cared to fix it because proprietary emoji locked users to their carrier.

The Carrier Incompatibility Disaster (circa 2007) 
  DoCoMo sends:    [SUN EMOJI]  -> encoded as F89F
  SoftBank reads:  F89F         -> maps to [ALIEN FACE] or [?]
  au reads:        F89F         -> silently drops it

  3 carriers. 3 encoding tables. 0 compatibility.
  Result: Japanese mobile users could not send emoji between networks.

  Total emoji at this point:
    DoCoMo:   176
    au/KDDI:  ~300
    SoftBank: ~480
    Overlap:  chaos

Google Saves the Day (Kind Of)

In 2007, Google engineers Mark Davis and Kat Momoi submitted a proposal to the Unicode Consortium to standardize emoji. Their motivation was practical: Google was building Gmail for mobile, and they needed to interoperate with Japanese carrier networks. The only way to do that reliably was to get emoji into Unicode.

The Unicode Consortium was not enthusiastic. Unicode was a serious standards body. Emoji were silly pictures. But Google (and later Apple) pushed hard, and in 2010, Unicode 6.0 shipped with 722 emoji. The first standardized emoji set was live.

Apple Goes Global by Accident

Apple had added an emoji keyboard to iPhone OS in 2008, but only for Japanese users — it was hidden behind a Japanese locale setting. In 2011, with iOS 5, Apple made the emoji keyboard available worldwide. The original intent was to support Japanese NTT DoCoMo users. What actually happened was that every iPhone user on earth suddenly had access to 😀❤🎉 and the world was never the same.

The Poop Emoji Situation

U+1F4A9 PILE OF POO (💩) deserves its own paragraph. In Japanese culture, the stylized poo with eyes and a smile was a good luck symbol — a play on the word unko (poop) sounding like un ga ii (good luck). Kurita included it straightforwardly in the original DoCoMo set. When it entered Unicode and became globally accessible, it rapidly became the most culturally significant Unicode character in modern history — spawning an entire film franchise, appearing in political commentary, and serving as the internet's universal expression of displeasure that is somehow still family-friendly.

The pile of poo has appeared in official government communications. It has been painted on buildings. It is, by some metrics, one of the most recognized symbols of the 21st century. Shigetaka Kurita drew it in under an hour as part of a weather widget project.

Wild: The Global Writing System

Emoji are the first truly global writing system that did not require centuries of colonialism to spread. Latin script required the Roman Empire and European colonialism. Arabic script spread through the Islamic Caliphate. Emoji spread through the App Store. 😀 is understood in Lagos, Tokyo, São Paulo, and Oslo without translation.

Rejected proposals include the middle finger (approved eventually), bagels (approved), a frowning pile of poo (rejected — the poo must remain joyful), and lettuce (pending as of this writing).

// 08 — ZWJ Sequences — Frankenstein Emoji

Zero Width Joiner — U+200D — is an invisible character that glues emoji together like Frankenstein's monster with a Unicode sewing kit. You cannot see it. Your text editor will not show it. A naïve string length function will count it. And it is the mechanism by which one family emoji (👨‍👩‍👧‍👦) is actually seven code points that modern rendering engines combine into a single glyph on the fly, or fall back to displaying as four separate emoji on older platforms that have not updated their ZWJ sequence tables yet.

ZWJ sequences are the Unicode Consortium's clever answer to combinatorial explosion. Instead of assigning a unique code point to every possible family combination (two mothers, three children, one dog, varying skin tones — you would need thousands of code points), they defined a composition mechanism. Take existing emoji. Glue them together with ZWJ. If your platform knows about the sequence, you get the combined glyph. If not, you get the individual emoji in a row. Graceful degradation, Unicode style.

// 09 — Normalization — NFC, NFD, NFKC, NFKD

"café" ≠ "café" — they look identical but one uses precomposed é (U+00E9) and the other uses e + combining acute (U+0065 U+0301). Different bytes, different hashes, different database lookups. Your cache misses. Your deduplication fails. Your user complains that they can't log in despite typing the correct password. Normalization is the unglamorous fix that makes string comparison actually work.

Unicode allows the same visual character to be encoded in multiple ways. Canonical equivalence means two sequences display identically and have the same meaning. But a naïve byte comparison will tell you they are different strings. This is the normalization problem.

NFC vs NFD — two ways to write é 
  NFC (Composed):
    é = U+00E9  ───────────────────────────────  1 code point
    UTF-8: 0xC3 0xA9                             2 bytes

  NFD (Decomposed):
    e + ◌́ = U+0065 U+0301 ──────────────────  2 code points
    UTF-8: 0x65 0xCC 0x81                        3 bytes

  Both render as:  é
  .length in JS:   NFC=1, NFD=2
  bytes in UTF-8:  NFC=2, NFD=3
  Equal? byte-wise: NO.  After NFC normalize: YES.

The Four Normalization Forms

Form Full Name What it does Use case
NFC Canonical Composition Decompose then re-compose: e + ◌́ → é Default for web, storage, most APIs
NFD Canonical Decomposition Always decompose: é → e + ◌́ Sorting, character analysis, macOS HFS+
NFKC Compatibility Composition Decompose compatibility chars then compose: ﬁ → fi Search normalization, identifier comparison
NFKD Compatibility Decomposition Decompose everything, including compatibility: Ω → Ω Most aggressive; casefold + NFKD for search

Compatibility decomposition breaks ligatures and other typographic forms that are semantically equivalent but visually distinct. The ligature ﬁ (U+FB01) is a single code point representing "fi" — two letters joined. Under NFKD it becomes U+0066 U+0069, the individual characters f and i.

    NFKD → Ω  (U+03A9 GREEK CAPITAL LETTER OMEGA)

  ½ (U+00BD VULGAR FRACTION ONE HALF)
    NFKD → 1 ⁄ 2  (U+0031 U+2044 U+0032)

  ℠ (U+2120 SERVICE MARK)
    NFKD → S M  (U+0053 U+004D)

  NFC and NFD preserve these — they only handle canonical equivalences.
  NFKC and NFKD go further, collapsing compatibility variants.

// 10 — Security — The Dark Side of Unicode

Unicode is a massive attack surface that most developers never think about. You sanitize your SQL. You escape your HTML. You validate your file paths. And then you let untrusted Unicode waltz through your front door and wreak havoc on your entire security model. The complexity of Unicode — its homoglyphs, its bidirectional text, its combining marks — creates entire classes of vulnerabilities that have compromised real systems.

Homoglyphs — Characters That Look Identical

A homoglyph is a character that is visually indistinguishable (or nearly so) from another. Unicode contains hundreds of these. Cyrillic а (U+0430) looks pixel-for-pixel identical to Latin a (U+0061) in most fonts. This is not a bug in fonts. These are intentionally different characters with different meanings in their respective scripts.

Homoglyph Attack — Spot the Difference 
  paypal.com    (all Latin — the real site)
  pаypаl.com  (Cyrillic а U+0430 — a fake)

  Latin    a  =  U+0061  =  0x61
  Cyrillic а  =  U+0430  =  0xD0 0xB0  (2 UTF-8 bytes)

  Same pixels. Different bytes. Different domain.
  A simple strcmp() or === check will correctly say NOT EQUAL.
  But a human looking at the URL bar cannot tell the difference.

RTL Override — Hiding What You Ship

U+202E RIGHT-TO-LEFT OVERRIDE is a formatting character that reverses the display direction of subsequent text. It was designed for mixing Arabic/Hebrew with left-to-right content. Attackers use it to disguise file extensions.

RTL Override File Name Trick 
  Actual filename bytes:
    h a r m l e s s [U+202E] f d p . e x e

  Rendered in Windows Explorer:
    harmless‮fdp.exe  appears as  harmlessexe.pdf

  U+202E reverses display of everything after it.
  The OS sees: harmless‮fdp.exe  (an executable)
  The user sees: harmlessexe.pdf (a document)

  Fix: strip or reject U+202E, U+202D, and other
  bidirectional override characters in file names.

IDN Homograph Attacks

Internationalized Domain Names (IDN) allow non-ASCII characters in domain names, encoded as Punycode. Every Unicode character that looks like an ASCII letter is a potential impersonation vector.

IDN Homograph — аррle.com 
  apple.com    — U+0061 U+0070 U+0070 U+006C U+0065 (all Latin)
  аррle.com    — U+0430 U+0440 U+0440 U+006C U+0065 (Cyrillic a, r, r + Latin l, e)

  The mixed version encodes as:   xn--le-lbc3bpb.com
  Browsers may or may not warn.   Chrome: shows Punycode xn-- form
  Firefox: shows Unicode if all chars from single script; Punycode for mixed

  The HTTPS lock icon means the connection is encrypted.
  It says NOTHING about whether the domain is who you think it is.

Zalgo Text — Combining Mark Abuse

Unicode allows an unlimited number of combining marks to be stacked on a single base character. Normally this is used for languages like Vietnamese (vowel + tone mark) or precomposed Devanagari. Nothing prevents stacking fifty combining marks above and below a letter until it overflows its text box, corrupts rendering, or crashes naive parsers.

Zalgo — Combining Mark Overflow 
  Normal: H e   c o m e s

  Zalgo:  H̺͙̍̀ e͍̒  c̹̻ͫ̀ͯ͘ o͍̘  mͭ̀͏  è̙̖  sͮ̀͆

  Each character above has 5-10 combining marks (U+0300–U+036F)
  stacked above and below it.

  Attacks:
  - Overflow CSS overflow:hidden boxes
  - Crash regex engines with catastrophic backtracking
  - Inflate database column lengths unexpectedly
  - Visual junk in notifications / chat previews

  Fix: normalize (NFC collapses multiple canonical combiners),
  limit combining mark count, reject sequences > N marks.

Production Unicode bugs are often invisible until they destroy something important. The emoji that silently truncates your user's bio. MySQL's utf8 Disaster

MySQL introduced a character set named utf8 in version 4.1. It was capped at 3 bytes per character — enough for the BMP, not enough for anything outside it. Emoji live in the Supplementary Multilingual Plane. When users started putting emoji in their text, MySQL silently truncated or errored.

MySQL utf8 vs utf8mb4 
  CREATE TABLE posts (
    body TEXT CHARACTER SET utf8    -- WRONG: 3-byte max, no emoji
  );

  INSERT INTO posts (body) VALUES ('I love 💩');
  -- ERROR 1366: Incorrect string value: '\xF0\x9F\x92\xA9' for column 'body'
  -- Or worse: silently truncates at the emoji

  -- The fix:
  ALTER TABLE posts CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

  utf8    = MySQL's broken 3-byte "UTF-8" (actually CESU-8 style)
  utf8mb4 = Real UTF-8 (4-byte, handles all 1.1M code points)

  The migration path: ALTER TABLE for each table, change connection charset,
  update stored procedures, triggers, and application connection strings.
  Fun for the whole team.

Collation — How Strings Sort and Compare

Collation determines how strings are compared and sorted. Choosing the wrong collation causes silent equality failures or incorrect sort orders.

utf8mb4 Collations 
  utf8mb4_bin            — byte-by-byte comparison (case sensitive, accent sensitive)
  utf8mb4_unicode_ci     — Unicode rules, case insensitive, accent insensitive
  utf8mb4_unicode_520_ci — Unicode 5.2.0 rules (more correct)
  utf8mb4_0900_ai_ci     — Unicode 9.0.0, accent insensitive, case insensitive
                           (recommended for MySQL 8+)

  Example with unicode_ci:
    "a" = "A"      (case insensitive)     -- SELECT where name='a' finds 'A'
    "é" = "e"      (accent insensitive)   -- SELECT where name='e' finds 'é'
    "Ω" != "ω"      (different letters)

  Example with bin:
    "a" != "A"     (case sensitive, exact bytes)
    "é" != "e"     (different bytes)

  For passwords: always use binary comparison (or better, a proper hash).
  For names/search: unicode_ci or 0900_ai_ci.
  Mixing collations in a JOIN causes implicit conversion — and table scans.

Unicode-Aware Regex with \p{}

The pattern [a-zA-Z] matches 52 characters. The Unicode database contains over 130,000 letters. If your validation rejects Arabic names, Thai surnames, or Chinese characters, you have a bug, not a security feature.

Unicode Property Escapes in Regex 
  \p{L}          — any Unicode letter (Letter category)
  \p{Lu}         — uppercase letters
  \p{Ll}         — lowercase letters
  \p{N}          — any number
  \p{P}          — any punctuation
  \p{Z}          — any separator (space, line, paragraph)
  \p{Emoji}      — emoji characters
  \p{Script=Han} — Han (CJK) script

  JavaScript (ES2018+):

    /^\p{L}+$/u.test("caf\u00E9") // true
    /\p{Emoji}/u.test("Hello 👋") // true

Terminal Display Width

Not all characters are created equal in terminal width. Your perfectly aligned table becomes garbage the moment someone's name contains an emoji or CJK character.

Character Display Widths 
  Width 0 (zero):  Combining marks (U+0300–U+036F)
                   Zero-width joiner (U+200D)
                   Zero-width non-joiner (U+200C)
                   Variation selectors

  Width 1 (normal): ASCII, Latin, Cyrillic, Arabic, Hebrew...

  Width 2 (wide):   CJK Unified Ideographs (U+4E00–U+9FFF)
                    Emoji with EAW=W property
                    Katakana, Hiragana in fullwidth ranges

  Table alignment disaster:
  | Name    | Score |    ← header: 6 chars "Name  "
  | Alice   | 100   |    ← normal: 5 chars
  | 李小明    | 95    |    ← 3 CJK = 6 display cells but .length=3
  | 👨‍💻      | 88    |    ← ZWJ sequence = 2 display cells, .length=5

  Fix: use wcwidth() in C, the wcwidth npm package in JS,
  or golang.org/x/text/width in Go.

int utf8_display_width(const char *utf8) {
    setlocale(LC_ALL, "en_US.UTF-8");
    wchar_t buf[256];
    mbstowcs(buf, utf8, 256);
    int width = 0;
    for (int i = 0; buf[i]; i++) {
        int w = wcwidth(buf[i]);
        if (w > 0) width += w;
    }
    return width;
}

The Emoji Proposal Process

Any person or organization can submit an emoji proposal. The bar is surprisingly low — a four-page PDF. The timeline is surprisingly long — about two years from submission to a published standard. The rejection rate is high.

             Must include: expected usage frequency evidence,
             distinctiveness argument, sample images

The Politics of Encoding

Unicode encoding is not purely technical. It is political. Decisions about which scripts to encode, which characters are "the same" across traditions, and which flags to include have real consequences for millions of people whose languages and identities are tied to those decisions.

Some of the Hard Decisions 
  Han Unification (CJK):
    Chinese, Japanese, and Korean have overlapping character sets.
    Unicode merged many into a single block (U+4E00–U+9FFF).
    The "same" character may have different glyphs in different countries.
    Japan fought this. Font rendering picks the right glyph using lang tags.

  Flags:
    Only sovereign UN member states get emoji flags (via regional indicators).
    Scotland, Wales, England: got subdivision flag emoji (U+1F3F4 + tags).
    Taiwan: has a flag emoji. Its politics with the mainland are... complex.
    Western Sahara, Kosovo, Palestine: inclusion debates still ongoing.

  Han Character Variants:
    Simplified Chinese (PRC) vs Traditional Chinese (Taiwan/HK) vs
    Japanese Kanji vs Korean Hanja — many are unified; critics call it
    "Han unification imperialism." The counter: 100,000 separate code points
    would make the standard unmanageable.

  Emoji Gender/Skin Tone:
    Skin tone modifiers (U+1F3FB–U+1F3FF) added in Unicode 8.0.
    Gender modifiers via ZWJ sequences. Not every combination is official.
    Which professions get gender variants? Who decides the defaults?
Perspective

Unicode is humanity's most ambitious cataloguing effort. It is an attempt to take every writing system ever used by any human civilization — from Proto-Sinaitic to Mayan hieroglyphs to the scripts being invented right now for languages that were never written — and give each one a number. Every bug you have ever filed about string comparison, every encoding error in your logs, every garbled email subject line: these are the growing pains of a species learning to speak to itself across time and space through a shared symbolic infrastructure. The Consortium's work is unglamorous, politically fraught, perpetually incomplete, and absolutely essential.

Unicode is not finished. As long as humans write, there will be more characters to encode, more scripts to standardize, more edge cases to discover. Every developer who normalizes their strings, uses grapheme-aware length functions, and thinks in code points rather than bytes is doing their small part to make the tower a little less Babel.
