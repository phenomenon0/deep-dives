PROGRAMMABLE SVG OS

A local-first visual runtime for diagrams, graphics, apps, and data-bound scenes._

§1 — The Thesis

Think of it as a runtime for visual objects. Not a drawing app. Not "Inkscape in a browser." Not "open-source Figma." Something else entirely. A system where SVG stops being a file format for illustrations and becomes a scene graph, a document model, a UI substrate, a geometry language, a layout engine, and a programmable canvas operating system.

The key shift is this. A normal design app says: draw rectangle, change fill, export PNG. The SVG OS says: define a card component, bind its color to a score percentile, bind its text to player metadata, let it react to filters, let it render for print, web, animation, and 3D extrusion, let agents modify it, let users fork it like code.

Normal design app

You draw shapes. You move pixels. You export flat files. The document is dumb. The tool does the thinking.

Programmable SVG OS

You create visual entities with behavior, state, rules, data bindings, timelines, automations, and exports. The document is alive.

One Sentence

A browser-native, local-first visual runtime where SVG objects are programmable, data-bound, extensible, and exportable across 2D, interactive web, and lightweight 3D.

Instead of treating vector documents as final artifacts — static, dumb, inert — we treat them as running programs. The canvas is not a viewport. It is a runtime. Every object on it has identity, state, lifecycle, event hooks, and relationships. You don't just look at the document. You query it, bind to it, automate it, and publish it.

§2 — Why SVG Specifically

Because SVG is already secretly a very powerful base layer. It is vector-native, structured as a DOM tree, styleable with CSS, transformable, text-aware, resolution-independent, web-native, composable, and scriptable. It already has the bones of a visual operating system. Most apps just refuse to see it.

Most applications treat SVG as one of three things: an import/export format, a final artifact, or a dumb container. The hacker move — the real unlock — is to treat SVG as:

  • Source code — version it, diff it, branch it, review it
  • Intermediate representation — transform it, compile it, optimize it
  • Live runtime object model — query it, bind to it, react to it, automate it

That changes everything.

SVG is the only vector format that is simultaneously a document model, a rendering format, and a first-class DOM citizen. No other format gives you all three. That is why it is the right substrate for a visual operating system.

§3 — What "Operating System" Means Here

Not kernel, not drivers, not bootloader. Operating system in the softer, truer sense: a system that manages objects, state, processes, events, views, storage, permissions, extensions, transformations, and export targets. The machinery that makes everything else possible.

Visual processes

Things that run on objects, in passes:

  • Layout pass — resolve constraints, compute positions, distribute spacing
  • Constraint pass — enforce pinning, aspect locks, anchor relationships
  • Animation pass — interpolate keyframes, run tweens, update timelines
  • Filter pass — apply SVG filters, effects chains, non-destructive transforms
  • Export pass — flatten for PNG, serialize for PDF, optimize for web
  • Data-binding pass — pull fresh values, update labels, recolor conditionally
  • AI transform pass — let agents rewrite, generate, rebalance, retheme

Visual filesystem

Not just folders of SVGs. A graph of reusable assets:

Visual IPC

Objects talk to other objects. This label updates when this value changes. This badge recolors when the team changes. This chart rerenders when data changes. This layer responds to hover, click, or agent edits. The document is a living network of reactive relationships.

Extension runtime

Users install "apps" inside the SVG OS — not tiny utility scripts, but full products:

  • Icon generator
  • Sports lineup generator
  • Packaging dieline builder
  • Isometric city builder
  • Motion graphics tool
  • Data card compositor
  • Vector-to-3D converter
Why "OS" is the right metaphor

The OS manages documents, windows/views, processes, packages, permissions, events, storage, and interop. The SVG OS manages scenes, canvas views, layout/render passes, plugins, access control, reactive bindings, local-first storage, and multi-format export. The parallel is structural, not metaphorical.

§4 — Layer 1: The Document Kernel

This is the heart of the system. The document is not just SVG text sitting in a file. Internally, it is a structured scene graph — a visual AST — where every node carries identity, semantics, relationships, and behavior. You may still serialize to SVG for interchange, but internally you want something richer. Think SVG++.

A <rect> in raw SVG is just geometry and style. A rectangle in the SVG OS is:

  • Geometry — position, size, corner radius, path data
  • Semantic type — is it a card background? A stat bar? A button?
  • Bindings — what data drives its fill, its text, its visibility?
  • Constraints — what spatial rules does it obey?
  • Component role — is it part of a symbol? A template slot?
  • Export rules — flatten for print? Animate for web? Extrude for 3D?
  • Interaction rules — respond to hover? Click? Agent mutation?
  • Provenance — who created it, when, from what template?

The node tree

The internal model is a tree of typed nodes. Every node has a stable UUID — not a fragile index, not a DOM path. The tree supports groups, symbols, defs, filter references, text runs, and arbitrary metadata. It is the single source of truth. Rendering, export, and collaboration all derive from it.

Internal representation

TYPESCRIPT
interface SvgNode {
  id:          UUID;
  type:        NodeType;          // path | group | text | image | component | effect
  name:        string;
  properties:  Map<string, Property>;
  children:    SvgNode[];
  parent:      SvgNode | null;
  effects:     Effect[];          // non-destructive effect chain
  bindings:    DataBinding[];    // reactive data connections
  constraints: Constraint[];     // spatial rules
  metadata:    Record<string, any>;

  on(event: string, handler: Handler): void;
  query(selector: string): SvgNode[];
  serialize(): string;          // → valid SVG fragment
  clone(): SvgNode;
}
Key Insight

Export to SVG is a serialization target, not your only truth. The internal model is richer than what SVG can express. That delta — the metadata, bindings, constraints, behaviors — is what makes it an OS and not a format.

Failure modes

A runtime that only describes happy paths is a demo, not a product. The document kernel needs defined behavior for every failure state:

FailureBehavior
Node references a missing def (gradient, filter, symbol)Render with fallback (transparent fill, no filter). Flag node as error:missing-ref. Surface in diagnostics panel.
Circular parent-child relationshipReject the mutation. Return error to caller. Tree invariants are non-negotiable.
Duplicate UUIDs after paste/mergeRe-mint UUIDs on the incoming subtree. Always. No "hope it works" path.
Corrupt serialization on loadParse what you can. Wrap unrecoverable subtrees in an error boundary node. Show diagnostic overlay. Never silently drop content.
Plugin mutates tree into invalid stateValidate after plugin transaction. Roll back if invariants broken. Log the violation for plugin debugging.

Every node in the tree can carry an errorState flag. Error nodes render with a visible diagnostic overlay (dashed red border in the editor, invisible in export). This means errors are visible — not silent corruption you discover three hours later.

Non-Negotiable

A runtime that only describes happy paths is a demo, not a product. Define error semantics before you ship. Every data binding, every constraint, every plugin hook, every CRDT merge needs a failure mode. Graceful degradation is an architecture decision, not an afterthought.

§5 — Layer 2: The Object Model

Every thing on the canvas is an object. Not just a shape — an entity with identity, typed properties, methods, event hooks, metadata, and optional behavior scripts. A "player card" is not a group of rectangles and text nodes. It knows it is a player card. It knows the player name, the team, the percentile values, the portrait slot, the stat bars, the theme tokens, the export sizes, and the motion preset.

This is the difference between editing shapes and editing meaning.

Object schema example

OBJECT SCHEMA
const PlayerCard = defineObject({
  type: 'sports/player-card',
  version: '1.0',

  props: {
    playerName:   { type: 'string',  label: 'Player Name' },
    team:         { type: 'string',  label: 'Team' },
    position:     { type: 'string',  label: 'Position' },
    rating:       { type: 'number',  min: 0, max: 100 },
    portrait:     { type: 'image',   aspect: '3:4' },
    teamColors:   { type: 'palette', slots: ['primary', 'secondary', 'accent'] },
  },

  slots: {

    nameLabel:    { type: 'text',     bind: 'props.playerName' },
    ratingBar:    { type: 'meter',    bind: 'props.rating' },
    portraitSlot: { type: 'image',    bind: 'props.portrait' },

    { name: 'social',  format: 'png', size: [1080, 1350] },
    { name: 'print',   format: 'pdf', size: [300, 420], unit: 'mm' },
    { name: 'web',     format: 'svg', interactive: true },
  ],
});

Now editing becomes richer than moving shapes. You change a team property and the colors update. You swap a portrait and the mask recalculates. You bind a rating to a live API and the stat bar animates. The object model makes the canvas semantic.

§6 — Layer 3: Constraint & Layout Engine

This is one of the biggest unlocks. Most vector tools are still too manual — you place things by hand, nudge pixel by pixel, duplicate and re-align when content changes. Instead, objects in the SVG OS can declare spatial rules. The system resolves them. This is what stops it from being "Illustrator clone" and starts making it a design tool, UI builder, parametric CAD-lite, and presentation system all at once.

Constraint types

CONSTRAINTS
// Pin to parent center
pin(card.nameLabel, { to: card, anchor: 'center-top', offset: [0, 24] });

// Distribute children evenly with 12px spacing
distribute(statsGroup, { axis: 'horizontal', gap: 12 });

// Auto-grow text container
autoResize(nameLabel, { axis: 'vertical', minHeight: 20, maxHeight: 60 });

// Preserve aspect ratio on resize
aspectLock(portrait, { ratio: '3:4' });

// Repeat per dataset row
repeatGrid(cardTemplate, {
  data: players,
  columns: 4,
  gap: [16, 16],
  bind: (card, row) => card.bindAll(row)
});

// Snap to isometric grid
isoProject(building, { grid: isoGrid, elevation: 2 });

// Responsive breakpoint
responsive(layout, {
  breakpoints: {
    768:  { columns: 2, gap: 12 },
    1200: { columns: 4, gap: 16 },
  }
});

The solver

The constraint engine runs after data binding but before rendering. It is a multi-pass solver:

Constraint Resolution Pipeline 

  Data Binding Pass              Constraint Solver              Render Pass
  ─────────────────     ──▶     ─────────────────     ──▶     ───────────
  Pull values                   1. Resolve pins               Paint to DOM
  Update props                  2. Compute distributions       Composite
  Evaluate expressions          3. Auto-resize text            Export
                                4. Enforce aspect locks
                                5. Run repeat-grids
                                6. Apply responsive rules
                                7. Snap to grids

Templates work because you define rules, not pixel positions. Change the data, change the viewport, change the theme — the layout engine recomputes everything. Manual nudging becomes the exception, not the default.

Halfway Between Worlds

With a constraint engine, the SVG OS sits at the intersection of design tool, UI builder, parametric CAD, and presentation system. You get the visual quality of a design tool with the structural rigor of a layout engine. That is the territory no current tool occupies well.

§7 — Layer 4: Data Binding

This is the truly modern part. Objects on the canvas bind to external data sources — JSON files, CSV sheets, REST APIs, database rows, user forms, LLM outputs, spreadsheet cells. When the data changes, the canvas updates. The document becomes generative. One template, infinite outputs.

Think about what this unlocks:

  • Sports scouting cards — 200 cards from one template and a player database
  • Social graphics — auto-generated from a CMS with brand tokens
  • Dashboards — designed vector layouts with live data, not chart-library defaults
  • Maps — with live annotations pulled from a spatial API
  • Product catalogs — pages generated from inventory data
  • Certificates — personalized at scale from a registration form
  • Election results — live-updating infographics bound to vote counts
  • Lineup graphics — automated from match-day squad sheets

This is huge because most design tools are catastrophically bad at structured, repeatable graphics generation. They treat every output as a one-off.

Binding syntax

DATA BINDING
// Bind a single object to a data source
const card = doc.query('#player-card');

card.bind({
  source: 'https://api.club.com/players/42',
  refresh: '30s',
  map: (player) => ({
    '.name':       player.display_name,
    '.number':     player.shirt_number,
    '.position':   player.primary_position,
    '.portrait':   player.headshot_url,
    '.rating-bar': { width: `${player.rating}%` },
    '.team-color': player.team.primary_color,
  }),
});

// Bind a repeater to a dataset
const grid = doc.query('#card-grid');

grid.repeat({
  source: './squad.csv',
  template: '#player-card',
  columns: 4,
  gap: 16,
  filter: (row) => row.appearances > 10,
  sort: (a, b) => b.rating - a.rating,
});

// Bind to an LLM output
card.bind({
  source: { type: 'llm', prompt: 'Summarize this player in 2 sentences' },
  map: (result) => ({
    '.bio-text': result.text,
  }),
});
LLM as Data Source

When the data binding layer treats LLM outputs the same as JSON or CSV, agents become first-class content generators. A prompt fills a text slot the same way an API response fills a number slot. The canvas does not care where the data comes from — only that it conforms to the binding contract.

Reactive updates

Bindings are not one-shot. They are reactive. When the source emits new data — a webhook fires, a CSV is re-uploaded, a form field changes — the binding re-evaluates, the constraint solver re-runs, and the affected nodes re-render. The update is surgical: only touched nodes recompute. The rest of the scene is untouched.

Reactive Data Flow 

  Data Source           Binding Layer          Scene Graph         Renderer
  ───────────          ─────────────          ───────────         ────────
  JSON / CSV   ──▶     evaluate map    ──▶    update props  ──▶  repaint
  API poll     ──▶     type coerce     ──▶    mark dirty    ──▶  (only
  WebSocket    ──▶     validate        ──▶    re-constrain  ──▶   dirty
  LLM stream   ──▶     diff prev       ──▶    propagate     ──▶   nodes)
  Form input   ──▶     emit change     ──▶    notify subs

Binding failures

Data sources fail. APIs go down, CSVs have bad rows, LLMs time out, WebSockets disconnect. The binding layer needs defined behavior for every failure — not just the happy path.

FailureBehavior
API returns 404 / 500Hold last-known value. Show stale-data badge in editor. Retry with exponential backoff. After N failures, surface error to user.
Binding evaluates to null / undefinedUse fallback value from schema default. If no default, render placeholder text. Never render "undefined" as visible text.
Type mismatch (string where number expected)Attempt coercion. If coercion fails, flag the binding as error:type and use fallback.
LLM timeout / rate limitShow placeholder or cached result. Queue for retry. Never block the render pipeline on an LLM call.
CSV parse error (bad row)Skip the row. Log it. Continue processing remaining rows. Report skipped count in diagnostics.
WebSocket disconnectFreeze at last state. Reconnect automatically. Replay missed deltas on reconnect if server supports it.
BINDING WITH FALLBACKS
card.bind({
  source: { type: 'api', url: 'https://stats.example.com/player/42' },
  map: (data) => ({
    '.player-name': data.name,
    '.rating': data.rating,
  }),
  fallback: {
    '.player-name': 'Player Name',
    '.rating': '—',

  onError: 'hold-last',       // 'hold-last' | 'use-fallback' | 'hide-node'

    maxRetries: 3,

§8 — Layer 5: Text Layout Engine

Text is the iceberg. Every data-bound template has text. Every card, certificate, social post, dashboard label, and stadium graphic depends on text that wraps correctly, sizes predictably, and exports identically across SVG, Canvas, PDF, and PNG. Defer text entirely and your MVP breaks on its core use cases. The system needs an honest text strategy from day one.

SVG's native <text> element is deceptively simple. It handles single lines fine. But the moment you need paragraph wrapping, auto-sizing text boxes, overflow truncation, mixed fonts, or bidirectional scripts — you are on your own. SVG has no paragraph model. No line-breaking algorithm. No text-box concept. Every design tool that uses SVG has had to build this layer from scratch.

What text actually requires

CapabilityDifficultyWhy it matters
Line breakingMediumEvery multi-line label. Every data-bound text block that might overflow.
Paragraph layoutHardJustified text, widow/orphan control, column flow. Print-quality output.
Font shaping (HarfBuzz)HardLigatures, kerning, Arabic/Devanagari/CJK. Without it, non-Latin text is broken.
Variable fontsMediumWeight, width, slant axes. One font file, infinite tuning. Modern type design depends on this.
Text-on-pathMediumSVG supports it natively via <textPath>, but interaction/editing is complex.
Text wrapping in shapesVery hardWrap text inside a circle, polygon, or arbitrary path. Requires geometry + text cooperation.
Bidirectional textHardHebrew, Arabic mixed with Latin. Unicode Bidi Algorithm (UBA) is non-trivial.
Font fallback chainsMediumMissing glyphs must cascade to backup fonts. CJK coverage requires large fallback stacks.
Emoji sequencesSurprisingly hardFamily emoji = 7 code points, 1 glyph. ZWJ sequences, skin tone modifiers, flag pairs.

The pragmatic approach: two tiers

Do not try to solve all of text on day one. But do not pretend it does not exist either. The strategy is two tiers, shipped in sequence.

Tier 1 — MVP text (ship first)

  • Use the browser's native text layout via foreignObject or positioned HTML overlays for editable text
  • SVG <text> for simple labels and headings
  • Basic auto-size: shrink-to-fit, truncate with ellipsis
  • Font stack declarations per text node
  • Export renders text to positioned <tspan> elements or rasterizes via Canvas

Tier 2 — Full text engine (later)

  • HarfBuzz compiled to WASM for font shaping
  • Custom paragraph layout (Knuth-Plass line breaking)
  • Bidi algorithm for mixed-direction text
  • Text-in-shape wrapping via geometry intersection
  • Variable font axis control in the property inspector
  • Font subsetting for export (only ship used glyphs)

Text as a constraint problem

Text is not separate from the layout engine — it feeds into it. A text node with autoSize: 'height' needs the constraint solver to resize its container. A text node with overflow: 'shrink' needs font-size to become a solver variable. This coupling is why text cannot be a bolt-on afterthought.

TEXT CONSTRAINTS
const titleText = defineTextNode({
  content: '{{ player.name }}',
  fontStack: ['Inter', 'system-ui', 'sans-serif'],
  fontSize: { base: 24, min: 14, max: 36 },
  overflow: 'shrink',          // reduce font size before truncating
  truncation: 'ellipsis',     // last resort
  autoSize: 'width-fixed',    // height grows, width locked to container
  lineHeight: 1.3,
  maxLines: 2,               // clamp to 2 lines max
});

// The constraint solver sees this as:
//   IF text.measuredWidth > container.width
//     THEN reduce fontSize until fit OR fontSize == min
//     THEN truncate with ellipsis
//   IF autoSize == 'width-fixed'
//     THEN container.height = text.measuredHeight
The Iceberg Warning

Text looks simple on the surface. Underneath: Unicode normalization, ligatures, kerning tables, paragraph breaking algorithms (Knuth-Plass), vertical scripts, RTL reordering, emoji ZWJ sequences, and font subsetting for export. Every vector graphics tool that has shipped has learned this the hard way. Tier 1 buys you time. Tier 2 is where the real work lives. Budget for it.

§9 — Layer 6: Automation & Runtime Language

This is the "programmable" in "Programmable SVG OS." You need a scripting layer — a way for users and agents to express operations as code. Not just click-and-drag. Create 100 cards from CSV. Apply a theme pack across all scenes. Normalize spacing in a layout. Convert paths to extrusions. Run batch exports. Generate variants. Attach hover interactions. Animate layers. Publish scenes.

The runtime language is JavaScript/TypeScript with a domain-specific API surface. It runs in the browser, in a sandboxed worker, or server-side for batch operations.

Batch export pipeline

AUTOMATION
import { Doc, CSV, Theme } from '@svgos/runtime';

async function generateSquadCards() {
  const data     = await CSV.load('./squad-2025.csv');
  const template = await Doc.open('./templates/player-card.svg');
  const theme    = await Theme.load('./themes/matchday-dark.json');

  template.applyTheme(theme);

  for (const player of data.rows) {
    const card = template.clone();
    card.bindAll(player);

    // Export social-sized PNG
    await card.export('png', {
      path: `output/social/${player.slug}.png`,
      size: [1080, 1350],
      dpi: 144,
    });

    // Export print-ready PDF
    await card.export('pdf', {
      path: `output/print/${player.slug}.pdf`,
      size: [300, 420],
      unit: 'mm',
      bleed: 3,
    });
  }

  console.log(`Generated ${data.rows.length} cards in 2 formats.`);
}

§11 — Layer 8: Collaboration & Orchestration

Elixir is not the drawing engine. It is the orchestration brain around it. The BEAM VM was built for exactly the kind of work a collaborative graphics platform needs: millions of lightweight concurrent processes, fault-tolerant supervision trees, real-time message passing, and distributed state. One GenServer per open document is not a hack — it is the natural architecture.

  end

  def init(doc_id) do
    state = SvgOS.Store.load(doc_id)
    {:ok, %{doc_id: doc_id, tree: state.tree,
            version: state.version, clients: %{}}}
  end

    # Apply an operation from a client
  def handle_call({:apply_op, op, client_id}, _from, state) do
    {new_tree, delta} = SvgOS.CRDT.apply(state.tree, op)
    broadcast_delta(state.doc_id, delta, client_id)
    {:reply, :ok, %{state | tree: new_tree,
                             version: state.version + 1}}
  end

    # Periodic snapshot for durability
  def handle_info(:snapshot, state) do
    SvgOS.Store.save_snapshot(state.doc_id, state.tree)
    Process.send_after(self(), :snapshot, 30_000)
    {:noreply, state}
  end

  defp broadcast_delta(doc_id, delta, exclude) do
    SvgOS.PubSub.broadcast("doc:#{doc_id}", {:delta, delta}, exclude)
  end
end

One GenServer Per Document

The BEAM was built for this. Each document gets its own lightweight process — isolated memory, supervised lifecycle, garbage-collected when the last client disconnects. A million open documents means a million processes. Erlang/OTP has been doing this for telecom switches since the 1980s. The architecture is free.

§13 — Neumenon Format Synergies

Three formats from the Neumenon ecosystem slot naturally into the SVG OS stack. Not because we need to force integration — but because the problems they solve are exactly the problems a visual runtime has. Binary scene graph serialization. Compact agent communication. Fast asset storage. Each format found its niche organically.

Cowrie — Scene graph wire format

Cowrie is a multi-language binary JSON codec with graph primitives baked in: Node, Edge, NodeBatch, EdgeBatch, GraphShard, Adjlist. That is not a coincidence — it is a direct match for serializing the SVG OS document tree.

Where Cowrie fits:

  • Network sync — Binary-encode scene graph deltas for CRDT transport. Smaller payloads, faster parse than JSON over Phoenix Channels.
  • Snapshot storage — Serialize document state to Cowrie Gen2 with dictionary coding. Repeated component schemas (imagine 200 player cards with identical structure) get ~50% size reduction.
  • Graph queries — The scene graph is a graph. Cowrie's NodeBatch and Adjlist types let you ship subgraph queries (e.g., "all nodes bound to this data source") without deserializing the full tree.
  • Cross-language bridge — Cowrie has multi-language support. The Elixir orchestrator and the TypeScript client can exchange typed graph data without format translation.
COWRIE SCENE SYNC
// Client sends a scene delta as Cowrie-encoded graph patch
const delta = Cowrie.encode({
  type: 'graph_patch',
  nodes: [
    { id: 'n_4f2a', op: 'update', props: { fill: '#e94560' } },
    { id: 'n_7b1c', op: 'insert', parent: 'n_root', type: 'text' },
  ],
  edges: [
    { from: 'n_7b1c', to: 'n_root', rel: 'child_of' },
  ],
});

channel.push('delta', delta);  // binary, compact, typed

Glyph — AI / agent codec

Glyph is a text-based codec for AI and ML systems — compact structured text with deterministic canonicalization, delta-friendly patches, and frame-oriented streaming. When agents talk to the SVG OS, Glyph is the natural language.

Where Glyph fits:

  • Agent instructions — An AI agent describing scene edits in Glyph uses fewer tokens than JSON. That is not a minor optimization — it is a cost multiplier on every LLM call.
  • Delta patches — Glyph's patch encoding describes incremental scene changes cleanly. "Change this label, recolor that bar, move this group" as a structured delta stream.
  • Streaming transforms — Glyph's frame-oriented streaming maps to live AI pipelines. The agent streams partial results, each frame updates the canvas, the user sees real-time generation.
  • Canonical hashing — Deterministic serialization means you can hash scene states for cache keys, version comparison, and change detection.
GLYPH AGENT PATCH
// Glyph-encoded instruction from an AI agent
// Compact, deterministic, fewer tokens than JSON

{action=scene_patch target=doc_8f3a ops=[
  {op=update node=n_label_01 text="Marcus Rashford"}
  {op=update node=n_bar_rating width=78%}
  {op=recolor node=n_card_bg fill=#da291c}
  {op=bind node=n_portrait src=api://players/rashford/headshot}
]}

Shard — Asset container format

Shard is a binary container with O(1) name lookup, per-entry compression, CRC32C checksums, and GPU-aligned memory mapping. It is the natural container for the SVG OS asset layer.

Where Shard fits:

  • Component libraries — Pack hundreds of reusable symbols, templates, and materials into a single Shard file. O(1) lookup by name means loading a specific component is constant-time regardless of library size.
  • Font and image packs — Per-entry compression (zstd/lz4) means each asset uses the optimal codec. A 2KB SVG symbol and a 200KB PNG portrait coexist with independent compression.
  • GPU-aligned textures — When the renderer needs raster data for WebGPU, Shard's 16/32/64-byte aligned data blocks enable zero-copy transfer to the GPU.
  • Role-based profiles — Shard's role byte in the header can signal asset type: ComponentShard, TemplateShard, MaterialShard, IconShard. The system knows what to expect before reading the index.
SHARD ASSET LOADING
// Load a component from a Shard asset pack
const pack = await Shard.open('./assets/sports-kit.shard');

// O(1) lookup — xxHash64 name → offset, no scanning
const card = await pack.get('templates/player-card');   // SVG template
const badge = await pack.get('symbols/team-badge');      // SVG symbol
const font = await pack.get('fonts/athletic-sans');      // WOFF2
const pitch = await pack.get('textures/grass-tile');     // PNG, GPU-aligned

// Integrity verified automatically (CRC32C per entry)
console.log(pack.header.role);  // → 'ComponentShard'
console.log(pack.entries.length); // → 347 assets

§16 — Stadium-Scale Rendering & Multi-Backend Architecture

SVG is the right authoring model. It is not always the right runtime renderer. For stadium jumbotrons, broadcast walls, ribbon displays, dense particle animations, or GPU-driven effects, raw DOM/SVG rendering hits a wall. The architecture needs a render abstraction layer — so you author once in a structured visual model, then render differently depending on the target.

Parallel render libraries

The multi-backend architecture decomposes cleanly into parallel packages:

  • render-svg — Default. Native SVG DOM rendering with dirty-rect tracking. Best for editable, interactive, accessible scenes.
  • render-canvas — Canvas 2D backend. Faster compositing for raster-heavy scenes, image blending, headless export.
  • render-webgpu — GPU rendering pipeline. Mesh generation, shader materials, camera system. For heavy real-time playback.
  • effects-webgpu — GPU compute modules. Particle systems, procedural textures, physics simulations, blur/glow at scale.
  • layout-gpu — Future. GPU-accelerated constraint solving and layout computation for very large scenes (10k+ nodes).

These sit alongside existing packages. They do not replace the core scene graph — they consume it. Any scene authored in the structured model can be pointed at any renderer.

§18 — Inkscape Extraction Plan

Inkscape has three decades of accumulated knowledge about SVG geometry, document semantics, and vector operations. Some of it is invaluable. Some of it is desktop baggage that will sink you. The strategy is surgical: extract the mathematically hard, standards-hard layers. Throw away everything people actually touch. Rebuild the product as a browser-native runtime.

Port / Reuse

  • lib2geom — bezier math, path booleans, affine transforms, intersections, offsets, bounding boxes, curve simplification
  • SVG document semantics — how the SVG spec maps to internal structure: groups, defs, gradients, markers, clipping, masking, styles, ID references
  • Live Path Effect concepts — non-destructive parametric effects, editable source geometry, stackable transforms, effect params
  • Import/export logic — SVG cleanup, style resolution, flattening for PDF/PNG, defs normalization
  • Snapping math — geometric snap, node snap, guide/grid systems, precision transforms

Discard / Rebuild

  • GTK UI — the entire desktop interface, dialogs, panels, menus, toolbar. Rebuild for browser.
  • Desktop interaction code — drag handles, modal flows, desktop window management. Rebuild entirely.
  • Old extension model — Python script extensions, INX descriptors. Replace with sandboxed browser plugin model.
  • Rendering stack — Cairo/desktop redraw. Replace with SVG DOM + Canvas + WebGPU.
  • Action/verb/dialog plumbing — desktop command infrastructure. Replace with web command palette.
  • "Complete parity" ambitions — the path to bloat and death. Do not attempt.
The Trap

Do not port the GTK UI. Do not port the extension system. Do not try for feature parity. These are traps that will consume years of effort and produce a desktop clone nobody wants. The value of Inkscape is in its knowledge, not its application shape.

Only preserve layers that are mathematically hard or standards-hard. Rebuild everything people actually touch. Geometry is mathematically hard — keep it. SVG spec compliance is standards-hard — keep it. Desktop dialogs are neither — discard them.

Programmable SVG OS — A local-first visual runtime for diagrams, graphics, apps, and data-bound scenes. Design, automate, script, and publish vector-native artifacts. SVG as software.

§20 — Execution Roadmap

Do not start with "free draw anything." That is the path to an Illustrator clone you cannot finish. Start with structured mode. The MVP is structured vector scenes plus templates plus data binding plus export pipeline. Not a full illustration suite. Not full pen-tool parity. Not full typography. Not full filter authoring. Win on repeatable output, AI compatibility, structured graphics, and business value. Then expand.

The Clean Strategy

Extract Inkscape's hard-won geometry and document knowledge. Do not inherit its application shape. Rebuild the product as a browser-native runtime for structured graphics. Use Elixir for the orchestration brain. Use Neumenon formats for the wire, the agent codec, and the asset layer. Ship the Visual Automation Studio first. Let the platform grow from there.

SVG is not a file format. It is a runtime substrate. Build on it.
