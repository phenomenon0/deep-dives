Part II — Making It Fast

// 05 — Why Inference Is Slow

Era 1 — Just make it run
WEIGHTS → HBM → SRAM → CORES   the real pipeline
You thought the problem was compute. Usually it’s movement.
A 27B model is not a function. It’s a file you keep dragging through memory._

Part I traced a vector through 64 transformer blocks. You counted the FLOPs: 44 billion per token for Qwen 27B. That sounds like a compute problem. It isn’t. Modern GPUs can compute hundreds of teraFLOPs per second. An RTX 3090 does 142 trillion 16-bit operations every second. At that rate, 44 billion FLOPs should take 0.3 milliseconds. One token every 0.3 ms would mean 3,000 tokens per second.

You don’t get 3,000 tokens per second. You get about 30. The model runs 100× slower than the arithmetic alone would suggest. Something else is eating the time. That something is memory bandwidth — the speed at which data travels from GPU memory to the compute cores. The GPU is fast. The pipe feeding it is narrow.

The Memory Wall

A GPU has two relevant resources: compute (how many operations per second) and bandwidth (how many bytes per second it can read from memory). These two numbers define a boundary called the roofline.

GPUFP16 ComputeBandwidthVRAMRidge Point
RTX 3090142 TFLOPS936 GB/s24 GB~152 FLOPs/byte
RTX 4090165 TFLOPS1,008 GB/s24 GB~164 FLOPs/byte
A100 80GB312 TFLOPS2,039 GB/s80 GB~153 FLOPs/byte
H100 SXM989 TFLOPS3,350 GB/s80 GB~295 FLOPs/byte

The ridge point is where the bottleneck shifts. Below it, you’re memory-bound — the GPU finishes its math before the next chunk of data arrives. Above it, you’re compute-bound. The ridge point is peak compute divided by peak bandwidth.

Where Decode Lives

During decode, generating one token requires reading the entire model from memory. For a 7B FP16 model, that’s 14 GB per token. The compute is approximately 2 FLOPs per parameter — 14 billion FLOPs. The arithmetic intensity is:

Arithmetic intensity — decode at batch size 1 

  Bytes read per token:  7B params × 2 bytes = 14 GB
  FLOPs per token:       7B params × 2 FLOPs  = 14 GFLOPs

  Arithmetic intensity = 14 GFLOPs / 14 GB = ~1 FLOP per byte

  Ridge point of an A100:  ~153 FLOPs per byte
  Decode intensity:          ~1 FLOP per byte

  Decode is 153× below the ridge point.
  The GPU’s compute units are ~99.3% idle during decode.

  ─────────────────────────────────────────────────

  For any model at FP16, batch size 1:
    Arithmetic intensity ≈ 1 FLOP/byte  ← independent of model size

This is the central insight: decode doesn’t get faster with a faster GPU. The H100 has 3.2× the compute of the A100, but only 1.6× the bandwidth. For decode, you get roughly 1.6× the speed — the extra compute sits idle.

Where Prefill Lives

Prefill processes all prompt tokens at once. Compute scales with sequence length; weight reads don’t — you read weights once and reuse them across all tokens.

Arithmetic intensity scales with sequence length during prefill 

  seq_len = 1:     AI ≈ 1 FLOP/byte    memory-bound
  seq_len = 32:    AI ≈ 32 FLOPs/byte  memory-bound
  seq_len = 128:   AI ≈ 128 FLOPs/byte approaching ridge
  seq_len = 256:   AI ≈ 256 FLOPs/byte compute-bound
  seq_len = 2048:  AI ≈ 2048 FLOPs/byte GPU fully saturated

  Crossover happens at the ridge point:
    RTX 3090:  seq_len ≈ 152     A100: ≈ 153     H100: ≈ 295

The Utilization Trap

An RTX 3090 generating tokens from a 27B model at batch size 1 uses 0.66% of its compute capability. The other 99.34% is idle, waiting for data. The GPU isn’t slow. The memory bus is narrow.

One Decode Step, Dissected

One token of decode — 7B FP16 on A100 

  read weights            14 GB streamed from HBM
  read KV cache           grows with every token
  run kernels             matmuls, norms, activations

  project logits          multiply by embedding table
  sample one token        temperature, top-p, draw

  Memory read:  14 GB / 2,039 GB/s  =  6.9 ms     ← dominates

The Real Pipeline

Part I showed the math pipeline: embed → norm → attention → FFN → project → sample. The real pipeline is: weights → HBM → SRAM → cores. Before you optimize the model, you optimize the trip the numbers take.

// 06 — The KV Cache

Era 4 — Fix the KV cache
ATTEND ONCE → STORE FOREVER → PAY IN VRAM
The model remembers by writing down what it already saw.
That memory is the cache. That cache becomes the problem._

Chapter 5 showed you the memory wall: every decode step reads the entire model from HBM. But it reads something else too — the KV cache. Every token the model has ever seen in this conversation has a key vector and a value vector stored per layer. Attention doesn’t recompute them; it rereads them. That cache starts small and grows linearly with every word generated. At long contexts, it becomes the memory problem.

The model’s memory is not its weights. At runtime, its memory is the KV cache.

What Gets Cached

KV cache — what’s stored per layer per token 

  For each layer (64 in Qwen 27B):

  Token 1:   K1 [kv_heads × head_dim]  +  V1 [kv_heads × head_dim]
  Token 2:   K2 [kv_heads × head_dim]  +  V2 [kv_heads × head_dim]
  ⋮
  Token N:   KN [kv_heads × head_dim]  +  VN [kv_heads × head_dim]

  Per token = 2 × layers × kv_heads × head_dim × bytes_per_element

  Qwen 27B (FP16): 2 × 64 × 4 × 256 × 2 = 256 KiB per token
  Llama-2 7B (MHA): 2 × 32 × 32 × 128 × 2 = 512 KiB per token
The GQA Advantage

Llama-2 7B uses full multi-head attention: 32 KV heads. Qwen 27B uses grouped-query attention with only 4 KV heads. Despite being 4× larger, Qwen 27B uses half the KV cache per token. GQA isn’t just a quality tradeoff — it’s a memory architecture decision that determines how many users you can serve.

PagedAttention: Virtual Memory for KV

Traditional KV allocation is contiguous and pre-allocated. The vLLM team measured that existing systems wasted 60–80% of KV cache memory due to fragmentation and over-reservation.

Naive KV allocation — contiguous, wasteful 

  Request A: max_len=2048, actual output=37 tokens
  ██████████████████████████████████████████
  used: 37                 wasted: 2,011 slots (98%)

  Request B: max_len=2048, actual output=412 tokens
  ██████████████████████████████████████████
  used: 412               wasted: 1,636 slots (80%)

  Internal fragmentation: unused slots in live buffers.
  External fragmentation: freed gaps too small to reuse.
  Result: 60–80% of KV memory wasted.
PagedAttention — OS virtual memory for the KV cache 

  OS Concept                   PagedAttention
  ─────────────────────────────────────────────────
  Virtual page               Logical KV block (16 tokens)
  Physical page frame         Physical KV block (GPU memory)
  Page table                  Block table (per sequence)
  Demand paging               Allocate only when tokens arrive

  Mechanism:
  1. KV memory split into fixed-size blocks (16 tokens default)
  2. Each sequence has a block table: logical → physical
  3. Blocks need not be contiguous
  4. New blocks allocated on demand
  5. Attention kernel does a gather via block table

  Request A (37 tokens):
  Block table: [P7] [P42] [P15]  ← 3 blocks, last partial
  Waste: at most 15 slots = <3%

  Average waste = block_size / 2 = 8 tokens per sequence.
  Compare to 60–80% before. Near-zero fragmentation.
Measured improvements (vLLM, SOSP 2023) 

  Before PagedAttention:
    Memory waste:   60–80%

  After PagedAttention:
    Memory waste:   < 4%
    Throughput:     2–4× over HuggingFace Transformers
                    2.2× over Orca
                    up to 24× for beam search

  Copy-on-Write:
    Beam width 4 → branches share all existing blocks.
    Only copied when a branch writes to a shared block.
    Memory: ~1.05× instead of 4×.

  Prefix caching:
    Shared system prompts computed once and reused.
    Blocks identified by SHA-256 hash of token IDs.
    ref_cnt tracks sharing; freed when last user finishes.

  PagedAttention didn’t change the math.
  It changed the memory allocator. That was enough.

KV values have a narrower dynamic range than weights, so quantization works surprisingly well. Many implementations keep the most recent 128–512 tokens in FP16 and quantize older entries. GQA + FP8 KV can reduce memory 8–16× compared to MHA + FP16.

The Central Runtime Object

Weights are static. Activations are transient. The KV cache is the thing that grows, fragments, gets shared, gets evicted, gets quantized, gets paged, gets transferred between machines. It is the central runtime object of inference. Every optimization in Chapters 7–12 touches it. Every scheduling decision in Part III revolves around it.

// 07 — Quantization

Era 2 — Make it fit
16 bits → 8 → 6 → 4 → 3 → 2
Make the weights smaller. Hope the model still recognizes the world._

Chapter 5 showed that decode speed is bandwidth-limited: tokens per second equals bandwidth divided by weight size. The fastest way to double your speed is to halve your weights. A 7B model in FP16 is 14 GB; in 4-bit quantization, it’s 3.5 GB. Same model, 4× less data through the memory bus, roughly 4× faster decode. The question is what you lose.

Quantization is the oldest inference trick and still one of the most important. Smaller weights mean less bandwidth, less memory, bigger models on cheaper hardware. But every lost bit is a wager.

What the Bits Mean

Floating-point formats — what you’re compressing from 

  FP16 (Half Precision)
  ──────────────────────────────────────────
  [S][E E E E E][M M M M M M M M M M]
   1      5 exp          10 mantissa         = 16 bits
  Range: ±65,504    Precision: ~3.3 decimal digits

  FP8 E4M3 (Datacenter, Hopper/Ada native)
  ──────────────────────────────────────────
  [S][E E E E][M M M]
   1     4 exp     3 mantissa               = 8 bits
  Range: ±448     Precision: ~1.9 digits
  Tensor cores consume FP8 directly on H100. No dequant overhead.

  INT8 (Q8_0 in llama.cpp)
  ──────────────────────────────────────────
  [V V V V V V V V]  + 1 shared FP16 scale per 32 values
  8 bits, values −128 to +127             = 8.5 bpw effective

  INT4 (Q4_0 in llama.cpp)
  ──────────────────────────────────────────
  [V V V V]  + 1 shared FP16 scale per 32 values
  4 bits, values −8 to +7                 = 4.5 bpw effective
  Two values packed into one byte.

How It Works: Block Quantization

Q4_0 — the simplest quantization 

  32 FP16 weights → 1 FP16 scale + 32 × 4-bit integers

  Step 1: Find the max absolute value in the group of 32
         max_abs = 1.27

  Step 2: Compute scale = max_abs / 7 = 0.181

  Step 3: Quantize each weight:
         w = 0.53  →  round(0.53 / 0.181) = round(2.93) = 3
         w = -0.87 →  round(-0.87 / 0.181) = round(-4.81) = -5

  Step 4: Dequantize (at inference time):
         3 × 0.181 = 0.543    (was 0.530, error = 0.013)
         -5 × 0.181 = -0.905  (was -0.870, error = 0.035)

  Storage: 2 bytes (scale) + 16 bytes (32×4 bits) = 18 bytes for 32 weights
  Effective: 4.5 bits per weight. Down from 16. 3.6× smaller.
Q4_K_M — the recommended default 

  Two-level hierarchy: super-blocks of 256 values

  Super-block (256 weights):
    fp16 d       ← scale for sub-block scales
    fp16 dmin    ← scale for sub-block minimums
    12 bytes     ← 8 sub-block scales + 8 mins, 6 bits each
    128 bytes    ← 256 weights at 4 bits each

  Dequant for one sub-block of 32 values:
    value = d × sub_scale × nibble − dmin × sub_min

  Total: 144 bytes for 256 weights = 4.50 bpw

  The “M” variant: uses Q6_K for attention.wv and ffn.w2 tensors
  (the most sensitive layers), Q4_K for the rest.
  Effective: ~4.89 bpw across the whole model.
  Only +0.05 perplexity vs FP16. 3.3× smaller.
GPTQ and AWQ — calibration-based approaches 

  GPTQ (Frantar et al., 2022)
  ──────────────────────────────────────────
  Process one column of weights at a time.
  When you round a weight, compute the error.
  Redistribute that error to unprocessed columns
  using the Hessian (second-order sensitivity info).

  The Hessian tells you which directions the model cares about.
  Push rounding errors into directions it cares less about.
  Result: INT4 with far less quality loss than naive rounding.

  Cost: 2–4 hours calibration on 2048+ samples (7B on A100)

  AWQ (Lin et al., 2023, MIT Han Lab)
  ──────────────────────────────────────────
  ~1% of weight channels drive disproportionately large activations.
  Instead of mixed precision, scale those channels UP before
  quantization. They get more of the INT4 grid. They survive better.

  No Hessian, no column-by-column reconstruction.
  Just a per-channel rescaling. Comparable quality to GPTQ.

  Cost: 10–30 min calibration on 128–512 samples

  GPTQ compensates for errors after rounding.
  AWQ prevents the worst errors before rounding.
  Both achieve similar quality at INT4.

Why Quantization Changes Kernels

You can’t just swap the file and get the same speed. INT4 weights must be dequantized to FP16 before the GPU’s tensor cores can use them. That dequant cost is real — but bandwidth savings usually win because decode is memory-bound, not compute-bound.

The dequantization tradeoff 

  7B model, single decode token, RTX 3090 (936 GB/s):

  FP16:  Read 14 GB  →  no dequant  →  15.0 ms
  Q4:    Read 3.5 GB →  + dequant   →  ~5 ms   ✓ 3× faster

  The dequant adds ALU work, but the 4× bandwidth savings dominate.

  ──────────────────────────────────────────

  But kernel quality matters enormously:

  Naive AWQ kernel:   68 tok/s
  Marlin AWQ kernel:  741 tok/s  (10.9× faster, same weights)

  Marlin fuses dequant directly into the tensor core pipeline.
  Both pipelines (ALU + tensor core) stay saturated simultaneously.
  The kernel ships weights pre-shuffled into the exact layout
  tensor cores expect. Zero wasted memory transactions.

  FP8 on H100: tensor cores consume FP8 natively.
  No dequant at all. 2× FP8 FLOPS vs FP16 FLOPS.

Format ↔ Engine ↔ Hardware mapping

                      Runs anywhere: CUDA, Metal, Vulkan, CPU
                      No calibration. Widest hardware reach.
                      Best for: local inference, edge, portability

                      Optimized via ExLlama or Marlin kernels

  Q4_K_S (+0.27 ppl) crushes Q4_0 (+0.47 ppl) at same size.
Fit vs Fast

Quantization serves two masters. “Fit” means: can the model load at all? A 70B at Q4 needs 42 GB — it fits on an A100 but not a 3090. “Fast” means: once loaded, how quickly does it generate? A 7B at Q4 on a 3090 is 3× faster than FP16 because it reads 3.5 GB instead of 14 GB per token. These are different problems with the same solution. Quantization is the first trick you reach for, and often the last one you need.

// 08 — FlashAttention

Era 3 — Reduce attention IO
SAME ATTENTION · FEWER TRIPS
The equations did not change. The path through memory did._

Naive attention spends too much time writing and rereading things it could have kept closer. The score matrix — that N×N grid of how-much-does-this-token-attend-to-that-token — gets computed, written to HBM, read back for softmax, written again, and read one more time for the final multiply. FlashAttention’s insight was simple and brutal: the expensive part was not the arithmetic. It was the traffic.

The Problem: Materializing N×N

Standard attention — what gets written to HBM 

  Sequence length = 2048, head_dim = 128, 32 heads, FP16

  Step 1:  S = Q · KT
           Read Q, K from HBM
           Write S to HBM: 2048 × 2048 × 2 bytes = 8 MB per head
           × 32 heads = 256 MB

  Step 2:  P = softmax(S)
           Read S from HBM (256 MB)
           Write P to HBM (256 MB)

  Step 3:  O = P · V
           Read P from HBM (256 MB), read V
           Write output O

  Total HBM traffic: ~1 GB+ for one layer of attention
  × 64 layers = tens of GB. Per token batch.

  ───────────────────────────────────────────────

  Meanwhile, the A100’s on-chip SRAM:
    192 KB per SM × 108 SMs = ~20 MB total
    The 256 MB score matrix is 12× larger than all SRAM combined.
    It cannot stay on chip. It must be materialized in HBM.
    That materialization is the bottleneck.

The Solution: Tiling and Online Softmax

FlashAttention never builds the full N×N matrix. Instead, it processes attention in small tiles that fit in SRAM, using an online softmax algorithm that computes the exact answer without ever seeing all scores at once.

FlashAttention — the core loop 

  Split Q into Tr row blocks, K/V into Tc column blocks.
  Block sizes chosen so one tile fits in SRAM (~128×128).

  for each Q block Qi:
      Initialize: running_max = −∞, running_sum = 0, output = 0

      for each K/V block Kj, Vj:
          Load Kj, Vj into SRAM                  ← one HBM read

          # Everything below happens on-chip, in SRAM:
          Sij = Qi · KjT                           ← small tile, not N×N
          m_new = max(running_max, rowmax(Sij))
          Pij = exp(Sij − m_new)
          l_new = exp(running_max − m_new) · running_sum + rowsum(Pij)

          output = rescale(output) + Pij · Vj

          running_max = m_new
          running_sum = l_new

      output = output / running_sum      ← final normalization
      Write output to HBM                    ← one HBM write per Q block

  The N×N score matrix never exists. Only Br×Bc tiles in SRAM.
  Online softmax corrects the running statistics as it goes.
  The final answer is exact — no approximation.
FlashAttention-1 (2022) vs FlashAttention-2 (2023) 

  FA1 (Dao, 2022)
  ──────────────────────────────────────────
  Loop order: outer over K/V blocks, inner over Q blocks
  Warp strategy: Split-K — 4 warps split K, must sync to reduce
  Softmax: rescale output at every step
  Forward: 25–40% of A100 peak

  FA2 (Dao, 2023)
  ──────────────────────────────────────────
  Loop order: outer over Q blocks, inner over K/V
    → Q blocks are independent → parallelize across thread blocks
    → better occupancy for long sequences

  Warp strategy: Split-Q — 4 warps split Q, no sync needed
    → each warp writes its output slice directly

  Softmax: defer normalization, store logsumexp = m + log(l)
    → fewer non-matmul FLOPs (16× more expensive on A100)

  Causal mask: skip entire tiles where all columns > all rows
    → 1.7–1.8× speedup for causal attention

  Forward: 50–73% of A100 peak. 2× faster than FA1.
Benchmark results from the papers 

  A100 80GB SXM, forward pass, head_dim=128:

  Seq Len    PyTorch    FA1      FA2       FA2 % peak
  ──────────────────────────────────────────────────────
  2K         57 TF     78 TF    95 TF     30%
  8K         83 TF     95 TF    96 TF     31%
  16K        OOM       95 TF    98 TF     31%

  Forward-only peak: up to 230 TFLOPS = 73% of 312 TF theoretical

  Memory complexity:
    Standard:  O(N²)         16K → 512 MB per head per layer
    Flash:     O(N)          16K → same as 2K  (just Q,K,V,O)
    Savings:   10–20× less memory

  End-to-end training (GPT-2 medium):
    Without Flash:  21.0 days
    With FA2:        6.9 days   (3.0× faster)

  FlashDecoding (decode, long KV cache):
    32K context:  6–7× faster than FA2 for single-token decode
    64K context:  up to 8× faster end-to-end

  FA2 does MORE total FLOPs than naive attention (recomputation).
  But it’s 2–4× faster because it moves far less data through HBM.
  IO-awareness beats raw FLOP counting. Every time.
Why This Matters for Inference

FlashAttention changed what’s possible. Before it, 16K context caused OOM on an A100. After it, 128K context is routine. The memory went from O(N²) to O(N), and speed improved 2–4× — all while computing the exact same attention. No approximation. No quality loss. Just fewer trips through the memory bus. It is the purest example of the principle from Chapter 5: optimize the trip the numbers take.
