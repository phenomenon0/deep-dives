GAME CHARACTERS

Movement physics, animation state machines, pathfinding, behavior trees, combat feel — the systems that make a character feel alive._

01 — The Game Loop

Every game is a loop. Read input, update state, render frame, repeat. The question is how fast, and what happens when the answer is “it depends.” A variable timestep — where dt is whatever the last frame took — seems natural, but it couples your physics to your framerate. Dark Souls II shipped this way. Weapons degraded per frame, not per second. At 60fps on PC, swords broke twice as fast as on the 30fps console version. From Software patched it a year later.

The fix is a fixed timestep with accumulator. Physics runs at a locked rate — say 60Hz — regardless of monitor refresh. The leftover time becomes an interpolation factor for rendering between the two most recent physics states.

Fixed Timestep Accumulator 
frameTime = now - lastFrame          // real elapsed time
accumulator += frameTime

while accumulator >= STEP:            // STEP = 1/60
    previousState = currentState
    update(currentState, STEP)        // deterministic physics
    accumulator -= STEP

alpha = accumulator / STEP            // 0.0 to 1.0
render(lerp(previous, current, alpha))// smooth between states

The alpha value is the key insight. Without it, your rendering jitters because it shows a physics state that may be “in the future” of actual elapsed time. With interpolation, the visual position smoothly blends between the two most recent physics snapshots. The physics stays deterministic. The rendering stays smooth. They never fight.

The Spiral of Death

What if one physics step takes longer than STEP to compute? The accumulator grows faster than it drains. Next frame runs more steps to catch up, which takes even longer. The accumulator explodes. The fix: clamp frameTime to a maximum (e.g., 250ms). The simulation slows down — “bullet time” — instead of spiraling. Games like Factorio make this visible: when the factory gets too complex, UPS (updates per second) drops below 60 and everything moves in slow motion.

const step = time.Second / 60

func RunLoop(update func(float64), render func(float64)) {
    prev := time.Now()
    var accum time.Duration

    for {
        now := time.Now()
        frame := now.Sub(prev)
        if frame > 250*time.Millisecond {
            frame = 250 * time.Millisecond  // clamp: prevent spiral
        }
        prev = now
        accum += frame

        for accum >= step {
            update(1.0 / 60.0)
            accum -= step
        }
        alpha := float64(accum) / float64(step)
        render(alpha)
    }
}

Why This Matters

The demo above shows the difference. In “variable” mode, balls stutter when the frame rate fluctuates. In “fixed” mode, physics is smooth but rendering shows micro-jitter from rounding. In “interpolated” mode, both physics and rendering are silky. Every serious game engine — Unity, Unreal, Godot — runs a fixed timestep under the hood. Now you know why.

Dark Souls II — The 60fps Bug

Weapon durability was decremented each frame when the weapon clipped through an enemy hitbox. At 30fps (PS3/360), this was fine. When the PC version ran at 60fps, double the frames meant double the collision checks, and weapons broke in half the time. The fix in Scholar of the First Sin: multiply durability loss by dt, making it time-based instead of frame-based. A one-line fix for a bug that plagued the game for a year.

02 — Movement Physics

Mario doesn’t move like a car. He doesn’t move like a person. He moves like Mario. The NES game uses 8.4 fixed-point arithmetic — 16 subpixels per pixel — to track position and velocity on hardware with no floating-point unit. Walking caps at 1.5 pixels/frame, running at 2.5. Acceleration doubles when reversing direction, which is why turnarounds feel snappy. Air friction is halved compared to ground friction, so jumps preserve momentum but you can’t accelerate as fast. Every constant was tuned until it felt right.

The core loop is simple: velocity += acceleration * dt, position += velocity * dt. What makes each game feel different is the constants.

This makes aerial movement feel committed — you can adjust, but you can’t reverse on a dime. Ice physics drop friction to near zero, so stopping takes forever and the character slides.

The Movement Update

type Player struct {
    X, Y, VX, VY    float64
    Accel, Friction  float64
    MaxSpeed, AirMul float64
    OnGround         bool
}

func (p *Player) Update(dt float64, inputX float64) {
    mul := 1.0
    if !p.OnGround { mul = p.AirMul }

    if inputX != 0 {
        p.VX += inputX * p.Accel * mul * dt
    } else {
        // Apply friction toward zero
        if p.VX > 0 {
            p.VX = max(0, p.VX - p.Friction*mul*dt)
        } else {
            p.VX = min(0, p.VX + p.Friction*mul*dt)
        }
    }
    // Clamp to max speed
    p.VX = clamp(p.VX, -p.MaxSpeed, p.MaxSpeed)
    p.X += p.VX * dt
}
def update_movement(player, dt, input_x):
    mul = 1.0 if player.on_ground else player.air_mul

    if input_x != 0:
        player.vx += input_x * player.accel * mul * dt
    else:
        # Friction: decelerate toward zero
        if player.vx > 0:
            player.vx = max(0, player.vx - player.friction * mul * dt)
        else:
            player.vx = min(0, player.vx + player.friction * mul * dt)

    player.vx = max(-player.max_speed, min(player.max_speed, player.vx))
    player.x += player.vx * dt
function updateMovement(p: Player, dt: number, inputX: number) {
  const mul = p.onGround ? 1 : p.airMul;

  if (inputX !== 0) {
    p.vx += inputX * p.accel * mul * dt;
  } else {
    // Friction
    if (p.vx > 0) p.vx = Math.max(0, p.vx - p.friction * mul * dt);
    else p.vx = Math.min(0, p.vx + p.friction * mul * dt);
  }
  p.vx = Math.max(-p.maxSpeed, Math.min(p.maxSpeed, p.vx));
  p.x += p.vx * dt;
}

Try the demo above with the three presets. “Mario” has high acceleration and high friction — responsive but stops fast. “Celeste” has moderate acceleration with 65% air control — committed jumps. “Ice” has low friction and low air control — you slide everywhere. Same code, different constants, wildly different feel.

Design Insight

Game feel is not about code. It’s about constants. A designer can spend weeks tuning acceleration from 900 to 1000 to 1050. Celeste shipped with over 40 movement constants in Player.cs, each the result of thousands of playtest iterations. The code is the same for every platformer. The numbers are what make each one unique.

03 — Jump Feel

Jumping is the most scrutinized mechanic in game design. A jump that feels “off” ruins a platformer. Celeste, Hollow Knight, Super Meat Boy — each spent months on jump feel alone. The secret: it’s not one mechanic. It’s at least five, layered on top of each other, and most players never notice any of them.

Click to play — Space to jump, A/D to move

Variable Jump Height

Tap the button: short hop. Hold the button: full jump. Implementation: while the player holds jump during ascent, apply reduced gravity (half or less). When they release, immediately switch to full (or boosted) gravity. Celeste applies half gravity when vertical speed is below HalfGravThreshold (40f), giving a subtle float at the apex.

Coyote Time

Named after Wile E. Coyote, who runs off a cliff and doesn’t fall until he looks down. After walking off a ledge, the player can still jump for a few frames. Celeste: 6 frames (100ms). Without this, players feel cheated — “I pressed jump! I was on the edge!” They were, visually. But their character had already left the platform one frame ago. Coyote time bridges the gap between what the player sees and what the engine knows.

Input Buffering

The player presses jump 4 frames before landing. Without buffering, the input is lost — they weren’t on the ground yet. With a 4-6 frame buffer, the jump input is stored and fires the instant they touch down. Combined with coyote time, this makes jumps feel impossibly responsive. The player doesn’t need frame-perfect timing. The engine quietly helps.

Apex Modifier

At the peak of a jump, when vertical velocity is near zero, reduce gravity further. This gives 2-3 extra frames of hang time at the top, letting the player fine-tune their landing position. Sonic games hold max height for 2-3 frames. Celeste uses HalfGravThreshold. Dead Cells uses a multiplier that smoothly increases gravity as you pass the apex. All achieve the same goal: the jump’s peak feels floaty and controllable.

Squash & Stretch

Purely visual, but critical for weight. On jump takeoff: stretch the sprite vertically (0.8, 1.2). On landing: squash (1.3, 0.7). Lerp back to normal over 3-5 frames. The volume-preservation rule from traditional animation: compress one axis, expand the other proportionally. A character without squash-stretch feels like a rigid box. With it, they feel alive.

The Forgiveness Stack

Coyote time. Input buffering. Corner correction (Celeste nudges you around corners within 4 pixels). Ledge assist. These are all lies the engine tells to make the player feel skilled. Maddy Thorson called this “forgiveness” — the game bends its rules invisibly to make the player’s intent succeed. The best platformers feel precise because they are secretly lenient.

04 — State Machines

A character is always in exactly one state. Idle. Walking. Jumping. Falling. Attacking. Getting hit. Dying. The finite state machine enforces this constraint: one state active, transitions between them guarded by conditions, and each state owns its own update logic. It’s the oldest pattern in game AI and still the most common for player characters.

Each state defines three hooks: enter() (play animation, reset timers), update(dt) (run per-frame logic), and exit() (cleanup). Transitions have priorities — Die overrides everything, Hit overrides Attack, Attack overrides Jump. This prevents impossible states like attacking while dead.

Frame Data

Fighting games formalize state timing with frame data. Every attack has startup frames (vulnerable, no hitbox), active frames (hitbox live), and recovery frames (vulnerable again). A jab might be 4-3-7: 4 frames of windup, 3 frames where it can hit, 7 frames of cooldown. Frame advantage determines who acts first after an exchange — if your recovery ends before the opponent’s hitstun, you’re “plus” and get to act first.

The State Explosion Problem

An FSM with N states potentially needs N×(N-1) transitions. 10 states = up to 90 transition rules. Adding “Swimming” means evaluating transitions to and from every existing state. Code duplication creeps in — RunState and JumpState share camera logic but can’t easily share code in a flat FSM. The solution: hierarchical FSMs. A top-level FSM has states like OnFoot, InVehicle, Swimming, each containing its own sub-FSM. Transitions at the super-state level apply to all children. Halo used this for Elites — a top-level Combat/Search/Patrol FSM, with Combat containing its own Engage/Flank/Retreat sub-states.

When to Move Beyond FSMs

FSMs are perfect for player characters (10-15 states, hand-tuned transitions). For NPCs that need dozens of behaviors that combine in unpredictable ways, behavior trees (Chapter 7) or GOAP scale better. The rule of thumb: if you’re adding transitions faster than states, it’s time to switch.

05 — Pathfinding

Click a destination on a map. An NPC walks there, avoiding walls. Behind this simple interaction is A*, the algorithm that powers pathfinding in nearly every game since 1990. It finds the shortest path on a graph by exploring nodes in order of f(n) = g(n) + h(n) — actual cost from start plus estimated cost to goal.

The heuristic h(n) is what makes A* fast. Dijkstra’s algorithm explores equally in all directions (h=0). A* guides the search toward the goal. On a grid, Manhattan distance (|dx| + |dy|) works for 4-directional movement. Euclidean distance (sqrt(dx²+dy²)) works for any-angle movement. The heuristic must be admissible — never overestimate — or the path may not be optimal.

import heapq

def astar(grid, start, goal):
    """A* on a 2D grid. Returns path as list of (x,y)."""
    open_set = [(0, start)]
    came_from = {}
    g = {start: 0}

    while open_set:
        _, current = heapq.heappop(open_set)
        if current == goal:
            path = []
            while current in came_from:
                path.append(current)
                current = came_from[current]
            return path[::-1]

        for dx, dy in [(0,1),(1,0),(0,-1),(-1,0)]:
            nx, ny = current[0]+dx, current[1]+dy
            if 0 <= nx < len(grid[0]) and 0 <= ny < len(grid) and grid[ny][nx] == 0:
                ng = g[current] + 1
                if ng < g.get((nx,ny), float('inf')):
                    g[(nx,ny)] = ng
                    f = ng + abs(nx-goal[0]) + abs(ny-goal[1])
                    heapq.heappush(open_set, (f, (nx,ny)))
                    came_from[(nx,ny)] = current
    return []  # no path
function astar(grid: number[][], start: [number,number], goal: [number,number]) {
  const open: [number,number,number][] = [[0, start[0], start[1]]];
  const g = new Map<string,number>();
  const from = new Map<string,string>();
  g.set(`${start[0]},${start[1]}`, 0);

  while (open.length) {
    open.sort((a,b) => a[0]-b[0]);
    const [,cx,cy] = open.shift()!;
    if (cx===goal[0] && cy===goal[1]) {
      const path: [number,number][] = [];
      let k = `${cx},${cy}`;
      while (from.has(k)) { const [x,y]=k.split(',').map(Number); path.unshift([x,y]); k=from.get(k)!; }
      return path;
    }
    for (const [dx,dy] of [[0,1],[1,0],[0,-1],[-1,0]]) {
      const nx=cx+dx, ny=cy+dy, nk=`${nx},${ny}`;
      if (nx>=0&&ny>=0&&ny<grid.length&&nx<grid[0].length&&grid[ny][nx]===0) {
        const ng = (g.get(`${cx},${cy}`)||0)+1;
        if (ng < (g.get(nk)||Infinity)) {
          g.set(nk,ng); from.set(nk,`${cx},${cy}`);
          open.push([ng+Math.abs(nx-goal[0])+Math.abs(ny-goal[1]),nx,ny]);
        }
      }
    }
  }
  return [];
}
func AStar(grid [][]int, sx, sy, gx, gy int) [][2]int {
    type node struct{ x, y, f, g int }
    open := []node{{sx, sy, 0, 0}}
    gScore := map[[2]int]int{{sx, sy}: 0}
    from := map[[2]int][2]int{}
    dirs := [][2]int{{0,1},{1,0},{0,-1},{-1,0}}

    for len(open) > 0 {
        // pop lowest f
        best := 0
        for i, n := range open { if n.f < open[best].f { best = i } }
        c := open[best]
        open = append(open[:best], open[best+1:]...)
        if c.x == gx && c.y == gy {
            var path [][2]int
            p := [2]int{gx, gy}
            for p != ([2]int{sx, sy}) { path = append([][2]int{p}, path...); p = from[p] }
            return path
        }
        for _, d := range dirs {
            nx, ny := c.x+d[0], c.y+d[1]
            if nx < 0 || ny < 0 || ny >= len(grid) || nx >= len(grid[0]) || grid[ny][nx] != 0 { continue }
            ng := c.g + 1
            nk := [2]int{nx, ny}
            if prev, ok := gScore[nk]; !ok || ng < prev {
                gScore[nk] = ng
                from[nk] = [2]int{c.x, c.y}
                h := abs(nx-gx) + abs(ny-gy)
                open = append(open, node{nx, ny, ng + h, ng})
            }
        }
    }
    return nil
}

Jump Point Search

A* on a uniform grid explores many symmetric paths — paths that differ in order but have the same cost. Jump Point Search (JPS) eliminates this by “jumping” along straight and diagonal lines until hitting a wall or a “forced neighbor” (a node that requires a turn). This prunes the open set dramatically — an order of magnitude faster on open maps. The catch: JPS only works on uniform-cost grids. Weighted terrain (swamp costs 3, road costs 1) breaks its pruning assumptions.

StarCraft’s Pathfinding Hack

StarCraft 1 used region-based decomposition: the map was split into ~200 regions, with A* finding a path between region waypoints and local steering handling movement within each region. It was famously imperfect — units got stuck on ramps, Dragoons walked in circles. StarCraft 2 switched to navigation meshes with hierarchical A* at the region level and flow fields for local movement within each region, handling 200-unit armies smoothly.

Flow Fields for Armies

When 100 units need to reach the same destination, computing 100 individual A* paths is wasteful. A flow field runs Dijkstra once from the goal outward, giving every cell a direction vector toward the goal. Units just follow the arrow under their feet. Cost is O(cells), not O(units). Supreme Commander 2 and Planetary Annihilation use this for their massive armies.

06 — Flocking & Steering

In 1986, Craig Reynolds simulated bird flocking with three simple rules applied to each “boid”: separate from nearby neighbors, align heading with neighbors, move toward the center of nearby neighbors. No leader. No global coordination. Just three local forces per agent, and emergent flocking behavior appears. The paper, published at SIGGRAPH 1987, remains one of the most cited in game AI.

Click to play — Mouse attracts | Right-click repels

The Three Rules

Separation

Steer away from neighbors within a “protected range.” Force increases as distance decreases (inverse distance). Prevents crowding.

Alignment

Steer toward the average heading of visible neighbors. Makes the flock move in the same general direction.

Cohesion

Steer toward the average position (centroid) of visible neighbors. Keeps the flock together.

Combined

Final steering = w₁·sep + w₂·align + w₃·cohere. Tuning the weights changes behavior: high separation = loose swarm; high cohesion = tight ball; high alignment = school of fish.

The O(n²) Problem

Each boid checks every other boid for neighbor queries. 100 boids = 10,000 distance checks per frame. 1,000 boids = 1,000,000. The solution: spatial hashing. Divide the world into a grid of cells sized to the interaction radius. Hash each boid into a cell. For neighbor queries, only check the boid’s cell and its 8 neighbors (2D). Insertion and lookup are O(1). The grid rebuilds every frame, but that’s just a linear scan. With spatial hashing, 10,000+ boids run at 60fps.

def boid_steer(boid, neighbors, w_sep, w_ali, w_coh):
    sep = [0, 0]; ali = [0, 0]; coh = [0, 0]
    count = 0

    for other in neighbors:
        dx = boid.x - other.x
        dy = boid.y - other.y
        dist = (dx*dx + dy*dy) ** 0.5
        if dist < 1e-6 or dist > boid.visual_range:
            continue
        # Separation: inverse distance
        sep[0] += dx / dist
        sep[1] += dy / dist
        # Alignment: average heading
        ali[0] += other.vx
        ali[1] += other.vy
        # Cohesion: average position
        coh[0] += other.x
        coh[1] += other.y
        count += 1

    if count > 0:
        ali[0] /= count; ali[1] /= count
        coh[0] = coh[0]/count - boid.x
        coh[1] = coh[1]/count - boid.y

    boid.vx += sep[0]*w_sep + ali[0]*w_ali + coh[0]*w_coh
    boid.vy += sep[1]*w_sep + ali[1]*w_ali + coh[1]*w_coh
function steer(boid: Boid, neighbors: Boid[], wS: number, wA: number, wC: number) {
  let sx=0,sy=0, ax=0,ay=0, cx=0,cy=0, n=0;
  for (const o of neighbors) {
    const dx=boid.x-o.x, dy=boid.y-o.y;
    const d=Math.hypot(dx,dy);
    if (d < 1e-6 || d > boid.range) continue;
    sx+=dx/d; sy+=dy/d;   // separation
    ax+=o.vx; ay+=o.vy;   // alignment
    cx+=o.x;  cy+=o.y;    // cohesion
    n++;
  }
  if (n) { ax/=n; ay/=n; cx=cx/n-boid.x; cy=cy/n-boid.y; }
  boid.vx += sx*wS + ax*wA + cx*wC;
  boid.vy += sy*wS + ay*wA + cy*wC;
}

The demo above lets you tune the three weights with the control buttons and attract/repel boids with your mouse. High separation with low cohesion creates a loose swarm that avoids itself. High cohesion with high alignment creates a tight school of fish. Zero alignment makes it look like a crowd of panicked individuals.

Emergence

Flocking is the canonical example of emergent behavior — complex group dynamics arising from simple local rules. No boid knows about the flock. Each follows three rules about its immediate neighbors, and the group-level pattern falls out. The same principle applies to ant colonies, traffic flow, and Conway’s Game of Life. Simple rules, complex results.

07 — Game AI

Halo 2 changed how games think. Before 2004, most NPCs ran on finite state machines — hard-coded transition tables that worked for simple enemies but collapsed under complexity. Damian Isla at Bungie introduced behavior trees: a hierarchical structure where decisions flow from root to leaf, evaluated every tick. The Halo 2 AI used a behavior DAG of about 50 behaviors, and Elites could flank, retreat, throw grenades, and coordinate suppression fire — all without a single scripted sequence.

Behavior Tree Structure 
                    [Selector]              ← try children until one succeeds
                   ╱          ╲
          [Sequence]        [Sequence]       ← all children must succeed
          ╱       ╲          ╱       ╲
    (CanSee?)  (Attack)  (Heard?)  (Investigate)

         │
    (Patrol)                                 ← fallback if nothing else fires

A Selector tries children left to right, stops on the first SUCCESS (logical OR). A Sequence runs children left to right, stops on the first FAILURE (logical AND). Decorators modify a single child — Inverter flips the result, Repeater loops N times, AlwaysSucceed swallows failures. Conditions check world state. Actions do things.

Blackboards

Unreal Engine 4/5 pairs behavior trees with a blackboard — a typed key-value store that serves as the AI’s working memory. Keys like TargetActor, LastKnownPosition, AmmoCount are set by Service nodes (which tick on configurable intervals) and read by Decorator conditions. This decouples perception from decision — a “perception” service updates the blackboard, and the tree reacts to what’s written there.

GOAP: F.E.A.R.’s Innovation

F.E.A.R. (2005) took a radically different approach. Jeff Orkin at Monolith gave each NPC a goal (e.g., “KillEnemy”), a set of actions with preconditions and effects, and an A* planner that searched through action space to find a sequence that achieves the goal from the current world state.

The entire NPC FSM had just three states: Goto (move somewhere), Animate (play animation in place), and UseSmartObject (interact with a prop). All complex behavior — flanking, taking cover, flushing players with grenades — emerged from the planner choosing sequences of these primitives. Adding a new action like “DiveThroughWindow” automatically made it available to any goal whose preconditions it could satisfy, without editing existing behaviors.

// Minimal Behavior Tree runner
type Status = 'success' | 'failure' | 'running';

interface BTNode { tick(bb: Record<string,any>): Status; }

class Selector implements BTNode {
  constructor(private children: BTNode[]) {}
  tick(bb) {
    for (const c of this.children) {
      const s = c.tick(bb);
      if (s !== 'failure') return s;  // success or running
    }
    return 'failure';
  }
}

class Sequence implements BTNode {
  constructor(private children: BTNode[]) {}
  tick(bb) {
    for (const c of this.children) {
      const s = c.tick(bb);
      if (s !== 'success') return s;  // failure or running
    }
    return 'success';
  }
}
type Status int
const ( Success Status = iota; Failure; Running )

type Node interface { Tick(bb map[string]any) Status }

type Selector struct{ Children []Node }
func (s *Selector) Tick(bb map[string]any) Status {
    for _, c := range s.Children {
        if r := c.Tick(bb); r != Failure { return r }
    }
    return Failure
}

type Sequence struct{ Children []Node }
func (s *Sequence) Tick(bb map[string]any) Status {
    for _, c := range s.Children {
        if r := c.Tick(bb); r != Success { return r }
    }
    return Success
}
BT vs FSM vs GOAP

Use FSMs for player characters (few states, hand-tuned). Use Behavior Trees for NPCs with many behaviors that need clear priority ordering (most games). Use GOAP when you want emergent behavior from combining simple actions in novel ways (immersive sims, complex squad AI). Most shipped games use BTs — they’re debuggable, designable, and scale to hundreds of behaviors.

08 — Combat Systems

The first frame of a punch lands. Both characters freeze for 9 frames. A damage number floats upward. The victim flashes white, launches backward, and tumbles. In those 150 milliseconds of hitstop, the player felt the impact — not because of physics simulation, but because the game lied about time. Combat feel is built from temporal manipulation, visual feedback, and precisely timed collision boxes.

Click to play — WASD+J/K

Hitboxes and Hurtboxes

These are separate concepts. A hitbox is attached to an attack — it exists only during the active frames of a move. A hurtbox is attached to a character — it’s where they can be hit. Collision is hitbox-vs-hurtbox only. Two hurtboxes never interact. Two hitboxes never interact. This separation gives precise control: a sword swing’s hitbox can be active for exactly 4 frames, while the character’s hurtbox can shrink during a dodge roll.

AABB Collision

Axis-Aligned Bounding Box: the simplest and fastest collision test. Two rectangles overlap if and only if they overlap on both axes:

// Four comparisons — that's it
function aabbOverlap(a, b) {
  return a.left < b.right && a.right > b.left &&
         a.top < b.bottom && a.bottom > b.top;
}

Damage & Knockback

Common damage formula: finalDamage = max(1, baseDamage * multiplier - defense). The max(1,...) ensures attacks always deal at least 1 damage. Multipliers come from crits (1.5-2×), elemental weakness (2× in Pokémon), or headshots (2-4× in FPS games).

Smash Bros. Melee’s knockback formula factors in the victim’s accumulated damage, the move’s damage, the victim’s weight, and two per-move constants (base knockback and knockback growth). The p * d term makes knockback grow quadratically with damage percentage — this is why kill moves become lethal at high percents.

Hitstop & Freeze Frames

Both attacker and victim freeze in place on hit. Street Fighter: light attacks ~9 frames, heavy ~13 frames. Smash Bros: hitlag = floor(damage/3 + 3), capped at 20 frames. This freeze gives the eye time to register the collision, extends the cancel window for combo inputs, and — most importantly — makes the hit feel heavy. Without hitstop, a sword swing feels like swinging through air. With it, the sword feels like it connected.

function applyHit(attacker: Fighter, victim: Fighter, move: Move) {
  // Damage
  const dmg = Math.max(1, move.baseDamage * move.multiplier - victim.defense);
  victim.hp -= dmg;

  // Knockback vector
  const dir = Math.sign(victim.x - attacker.x) || 1;
  victim.vx = dir * move.knockback * (1 + victim.damagePct * 0.01);
  victim.vy = -move.knockback * 0.5;

  // Hitstop: both fighters freeze
  const hitlag = Math.min(20, Math.floor(dmg / 3 + 3));
  attacker.freezeFrames = hitlag;
  victim.freezeFrames = hitlag;

  // Visual feedback
  victim.flashTimer = 3;           // white flash frames
  victim.iFrames = 20;             // invincibility
  spawnDamageNumber(victim.x, victim.y - 20, dmg);
}

I-Frames Are Not Optional

After being hit, the character must be invulnerable for a window (Dark Souls: ~13 frames during rolls, Mega Man: ~90 frames after damage). Without i-frames, a character can be hit by every frame of a multi-frame attack, draining their health instantly. Multi-hit moves in Smash Bros. work because each hit resets the hitlag and i-frame timer precisely.

09 — Camera Systems

The camera is the most underrated system in any game. A bad camera makes a good game unplayable. A great camera is invisible — the player never thinks about it. The trick is that “just follow the player” feels terrible. Locking the camera to the player’s position causes every movement to translate directly to screen movement, which is nauseating and makes the world feel unstable. Every solution below introduces some form of lag between the player and the camera.

Click to play — WASD

Lerp Follow

camera = lerp(camera, target, t * dt). The camera smoothly chases the player. At t = 0.05, it feels cinematic and floaty. At t = 0.3, it feels responsive. The exponential decay creates natural easing — fast when far, slow when close. This is the baseline for 90% of games.

Dead Zone

A rectangle in screen space where the player can move without the camera reacting. The camera only scrolls when the player exits the dead zone. This gives stability — small movements don’t jostle the view. Classic Mario uses this: inside the zone, nothing happens; at the edge, the camera scrolls to keep Mario inside. A narrow dead zone feels responsive; a wide one feels stable. Most games use a dead zone combined with lerp for the best of both.

Look-Ahead

Pan the camera in the direction the player is moving, showing more of what’s ahead. Implementation: offset the target by velocity.normalized * lookAheadDistance. Smooth the transition with lerp to prevent snapping when the player reverses. Typical distance: 2-5 tiles. This is why running right in a platformer shows more terrain to the right — the camera anticipates where you’re going.

Screen Shake

Impact feedback through camera displacement. The key: a trauma variable (0.0 to 1.0) that decays over time. Actual offset = maxOffset * trauma² * random(-1, 1). The squared term gives a sharp onset that smoothly fades. Add +0.3 trauma for a small hit, +0.7 for an explosion. Perlin noise gives smoother, more organic shake than pure random. Jan Willem Nijman’s “Art of Screenshake” talk is the definitive reference.

Celeste uses room-based camera: the camera is locked to the room, not the player. Within a room, a subtle offset nudges the focal point ahead of Madeline’s movement direction. On room transitions, the camera lerps to the new room center. This avoids disorientation in a precision platformer where stable visual reference points are critical for judging jump distances.

The Invisible System

Try the demo above with each mode. “Locked” feels jittery. “Lerp” feels smooth but laggy. “Deadzone” feels stable but can surprise you at the edges. “Lookahead” feels like the game is reading your mind. Most shipped games combine lerp + deadzone + lookahead. The camera you never notice is the best camera.

10 — Abilities & Combos

World of Warcraft’s Global Cooldown is 1.5 seconds. Every ability triggers it. Rogues get 1.0 seconds. Haste reduces it to a floor of 0.75 seconds. This single timer gates the entire combat system — it prevents button-mashing, creates rhythm, and makes haste feel meaningful. On top of the GCD, individual abilities have their own cooldowns (30 seconds, 2 minutes, etc.). The interaction between global and individual cooldowns creates the “rotation” — the optimal sequence of abilities that maximizes damage per second.

Click to play — WASD+1234

Ability Queuing

WoW introduced an ability queue in Patch 4.0.1. You can press the next ability slightly before the current GCD finishes, and it fires on the first available frame. Without this, network latency creates dead time between abilities — you can’t press the next button until the server confirms the previous one finished. The queue window (configurable as “Lag Tolerance”) bridges this gap, making rotations feel crisp even at 100ms ping.

Combo Detection

Fighting games store recent inputs in a circular buffer with timestamps. Each frame, the buffer is scanned against known patterns. A Hadouken is Down → Down-Forward → Forward + Punch, with each transition within 8-15 frames. The system matches the most specific (longest) pattern first to avoid false positives — if both “D, DF, F + P” (Hadouken) and “F + P” (forward punch) match, the Hadouken wins.

Object Pooling

Abilities spawn projectiles, particles, damage numbers. Creating and destroying these every frame triggers garbage collection spikes. The solution: pre-allocate a pool of objects at load time. Firing a projectile dequeues an inactive one, sets its position and velocity, activates it. When it hits or goes off-screen, it deactivates and re-enters the pool. A bullet-hell might pool 500+ projectiles. An FPS pools 50. Zero per-frame allocation.

Input Buffer — Smash Bros

Smash Bros. Ultimate has a 9-frame input buffer. Any action pressed within 9 frames of becoming actionable will execute immediately when possible. This is why new players sometimes throw out unwanted moves — they pressed a button during endlag that the buffer faithfully queued. Competitive players learn to be precise despite the buffer, not because of it.

11 — Influence Maps

Where should an RTS unit move? Not just “toward the enemy” — toward the enemy while staying near allies, avoiding threats, and preferring high ground. Influence maps answer this by overlaying numeric fields on the game world. Each unit projects influence onto surrounding cells with distance-based decay: influence = strength * e-decay·d². The result is a heatmap that makes spatial reasoning possible for AI.

Click to place units — Right-click enemies

Layered Maps

Multiple independent maps represent different concerns. A threat map shows where enemy units can deal damage. A support map shows where friendly units provide cover. A resource map shows proximity to resource nodes. Combining them with weights creates different AI personalities: aggressive AI weights threat negatively (seeks danger), defensive AI weights support positively (stays near allies).

The combined potential at any cell: potential = w₁·threat + w₂·support + w₃·resource. Units follow the gradient — the steepest ascent toward favorable cells. This is a potential field: attractive potentials pull toward objectives, repulsive ones push away from threats. StarCraft AI bots use this for kiting — ranged units maintain distance from melee threats by following the repulsive gradient while staying within attack range.

When to Use Influence Maps

Influence maps excel at where questions: where to attack, where to build, where to retreat. They complement pathfinding (A* answers how to get there) and behavior trees (which answer what to do). RTS games are the natural fit, but tower defense, stealth games, and even FPS bot AI use influence maps for spatial reasoning.
