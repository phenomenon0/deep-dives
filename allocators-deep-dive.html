ALLOCATORS

From virtual memory to garbage collection. Every object you create, every buffer you fill, every frame you render — an allocator decides where it lives and when it dies._

// 01 — The Illusion

Every process on your machine believes it owns the entire address space. A 64-bit process sees 248 bytes of virtual memory — 256 terabytes — stretching from 0x0000000000000000 to 0x00007FFFFFFFFFFF. None of it is real. The operating system and the CPU's memory management unit (MMU) maintain this illusion through a four-level page table that translates virtual addresses to physical frames on demand. This is the foundation everything else in this deep dive rests on.

When your program dereferences a pointer, the CPU doesn't go directly to RAM. It splits your 48-bit virtual address into five fields and walks a tree of page tables. Each level indexes into a 512-entry table (9 bits), and the final 12 bits give the offset within a 4KB page.

Walking four levels of page tables for every memory access would be catastrophic for performance. The Translation Lookaside Buffer (TLB) caches recent virtual-to-physical translations. A TLB hit resolves in 1–2 cycles. A TLB miss triggers the full page walk — 4 sequential memory reads, potentially 200+ cycles. On a system with 4KB pages, a 64-entry TLB covers only 256KB of memory. This is why huge pages (2MB, 1GB) matter: each TLB entry covers 512× or 262,144× more memory.

The Grand Illusion

Virtual memory is computing's greatest sleight of hand. It gives every process its own universe of addresses, enables demand paging (allocate only what you touch), provides memory protection between processes, and makes the physical layout of RAM irrelevant. Every abstraction in this deep dive — malloc, free lists, arenas, garbage collectors — is built on this illusion.

// 02 — What malloc Actually Does

malloc() is not a system call. It is a userspace library function that manages a pool of memory obtained from the kernel. When you call malloc(32), glibc does not ask the kernel for 32 bytes. It checks its internal data structures — free lists, bins, caches — and returns a pointer to an already-available chunk. The kernel is involved only when the allocator's pool runs dry.

There are two syscalls the allocator uses to get memory from the kernel. brk() moves the "program break" — the top of the heap segment — upward, expanding the contiguous heap region. This is fast but inflexible: you can only grow or shrink the top, never return memory from the middle. mmap() requests anonymous pages from anywhere in the virtual address space. These can be returned independently. glibc's threshold is 128KB (the DEFAULT_MMAP_THRESHOLD) — allocations below this come from the brk heap, allocations above go through mmap. This threshold is dynamically adjusted in glibc 2.10+.

Overcommit: Linux Says Yes to Everything

On Linux, malloc() almost never returns NULL. The kernel promises memory it doesn't have, betting that you won't actually touch every page. When the system runs out of physical memory, the OOM killer picks a process and murders it. This is controlled by /proc/sys/vm/overcommit_memory: 0 = heuristic (default), 1 = always overcommit, 2 = strict accounting. Most production servers set 2 to avoid surprise deaths.

// 03 — Free Lists

At the heart of every general-purpose allocator is a free list — a data structure tracking which chunks of memory are available. When you call malloc(n), the allocator walks its free list looking for a chunk large enough. When you call free(p), the allocator adds that chunk back to the free list. The difference between allocators is how they organize this list and how they search it.

Implicit Free List

The simplest design stores a size header before each block. The allocator walks from block to block by adding the size to the current address. Free blocks have a bit in the header marking them as available. This is called an "implicit" free list because there are no explicit pointers — the structure is implicit in the block sizes.

Implicit Free List with Boundary Tags 
  ┌──────┬────────────────┬──────┬──────┬──────────┬──────┬──────┬────────────────┬──────┐
  │hdr   │   payload      │ftr   │hdr   │  FREE    │ftr   │hdr   │   payload      │ftr   │
  │32|1  │   (32 bytes)   │32|1  │64|0  │ (64 B)   │64|0  │16|1  │  (16 bytes)    │16|1  │
  └──────┴────────────────┴──────┴──────┴──────────┴──────┴──────┴────────────────┴──────┘
   alloc      used          alloc  free    unused     free  alloc     used          alloc
   hdr=header: size|allocated_bit    ftr=footer: duplicate of header (enables backward coalescing)

Explicit Free List

An explicit free list adds forward and backward pointers inside each free block, linking them into a doubly-linked list. Now malloc only visits free blocks instead of walking through allocated ones. This is dramatically faster when memory is mostly allocated — the common case.

Explicit Free List 
  free_head ──▶ ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
                │  hdr: 64|0   │    │  hdr: 128|0  │    │  hdr: 32|0   │
                │  next: ──────┼───▶│  next: ──────┼───▶│  next: NULL  │
                │  prev: NULL  │◀───┼──prev        │◀───┼──prev        │
                │  (free space)│    │  (free space) │    │  (free space)│
                │  ftr: 64|0   │    │  ftr: 128|0  │    │  ftr: 32|0   │
                └──────────────┘    └──────────────┘    └──────────────┘
  Placement policies:
    First fit  — return the first block ≥ requested size (fast, but front-loads fragmentation)
    Best fit   — return the smallest block ≥ requested size (less waste, slower search)
    Next fit   — like first fit but resume search from last position (spreads fragmentation)
#include <stdint.h>
#include <string.h>

// Minimal explicit free-list allocator
typedef struct Block {
    size_t size;       // block size (including header)
    int    free;       // 1 = free, 0 = allocated
    struct Block *next; // next free block
} Block;

static char heap[4096];
static Block *free_list = (Block *)heap;

void heap_init() {
    free_list->size = sizeof(heap);
    free_list->free = 1;
    free_list->next = NULL;
}

void* my_malloc(size_t n) {
    size_t total = n + sizeof(Block);
    Block *b = free_list, *prev = NULL;
    while (b) {
        if (b->free && b->size >= total) {
            // Split if remainder is large enough
            if (b->size >= total + sizeof(Block) + 16) {
                Block *rest = (Block *)((char *)b + total);
                rest->size = b->size - total;
                rest->free = 1;
                rest->next = b->next;
                b->size = total;
                b->next = rest;
            }
            b->free = 0;
            return (char *)b + sizeof(Block);
        }
        prev = b;
        b = b->next;
    }
    return NULL; // out of memory
}

void my_free(void *ptr) {
    Block *b = (Block *)((char *)ptr - sizeof(Block));
    b->free = 1;
    // Coalesce with next block if also free
    if (b->next && b->next->free) {
        b->size += b->next->size;
        b->next = b->next->next;
    }
}

class FreeListAllocator:
    def __init__(self, capacity):
        self.memory = bytearray(capacity)
        self.blocks = [Block(0, capacity)]

    def malloc(self, size):
        # First-fit search
        for i, b in enumerate(self.blocks):
            if b.free and b.size >= size:
                if b.size > size + 16:
                    # Split: remainder becomes new free block
                    rest = Block(b.offset + size, b.size - size)
                    self.blocks.insert(i + 1, rest)
                    b.size = size
                b.free = False
                return b.offset
        return None

    def free(self, offset):
        for i, b in enumerate(self.blocks):
            if b.offset == offset:
                b.free = True
                # Coalesce with neighbors
                if i + 1 < len(self.blocks) and self.blocks[i+1].free:
                    b.size += self.blocks[i+1].size
                    self.blocks.pop(i + 1)
                if i > 0 and self.blocks[i-1].free:
                    self.blocks[i-1].size += b.size
                    self.blocks.pop(i)
                return

class FreeListAllocator {
  private blocks: { offset: number; size: number; free: boolean }[] = [];
  private buf: ArrayBuffer;

  constructor(capacity: number) {
    this.buf = new ArrayBuffer(capacity);
    this.blocks = [{ offset: 0, size: capacity, free: true }];
  }

  malloc(size: number): number | null {
    for (let i = 0; i < this.blocks.length; i++) {
      const b = this.blocks[i];
      if (b.free && b.size >= size) {
        if (b.size > size + 16) {
          this.blocks.splice(i + 1, 0,
            { offset: b.offset + size, size: b.size - size, free: true });
          b.size = size;
        }
        b.free = false;
        return b.offset;
      }
    }
    return null;
  }

  free(offset: number) {
    const i = this.blocks.findIndex(b => b.offset === offset);
    if (i === -1) return;
    this.blocks[i].free = true;
    // Coalesce right
    if (i + 1 < this.blocks.length && this.blocks[i+1].free) {
      this.blocks[i].size += this.blocks[i+1].size;
      this.blocks.splice(i + 1, 1);
    }
  }
}

Coalescing is Everything

Without coalescing, freed memory fragments into tiny unusable slivers. Every free() must check whether the previous and next blocks are also free and merge them. Boundary tags (footers that duplicate the header) enable O(1) backward coalescing — without them you'd need to walk the entire list to find the previous block.

// 04 — The Buddy System

The buddy system (Knowlton, 1965; popularized by Knuth in The Art of Computer Programming) solves coalescing with an elegant constraint: all blocks are power-of-two sizes. When you request n bytes, the allocator rounds up to the next power of two and splits larger blocks in half until it reaches the right size. Each half is the other's "buddy." The critical insight: you can find any block's buddy with a single XOR.

Buddy Splitting: Request 20 bytes from a 128-byte region 
  Step 0:  [            128 bytes (free)              ]
  Step 1:  [    64 (free)     |     64 (free)          ]  split 128
  Step 2:  [ 32 (free) | 32  |     64 (free)          ]  split left 64
  Step 3:  [ 32 (free) | 32 ALLOC |   64 (free)       ]  allocate right 32

  Finding your buddy:  buddy_addr = block_addr XOR block_size
    Block at 0x40, size 32:  buddy = 0x40 ^ 0x20 = 0x60
    Block at 0x60, size 32:  buddy = 0x60 ^ 0x20 = 0x40  ✓ they find each other
Buddy Tree View 
                        [128]
                       /     \
                   [64]       [64] free
                  /    \
              [32]free  [32] ALLOC ← serves the 20-byte request

  On free: check buddy (left 32). If buddy is also free → merge into [64].
  If parent's buddy (right 64) is also free → merge into [128].
  Coalescing cascades up the tree in O(log n) steps.

The Linux kernel uses the buddy system for physical page allocation. Pages come in orders 0 through 10 (1 page = 4KB up to 1024 pages = 4MB). Each memory zone maintains an array of free lists, one per order. The allocator is implemented in mm/page_alloc.c, and it's been stable for decades because the design is just that good.

#include <stdint.h>
#include <string.h>

#define HEAP_SIZE 1024
#define MIN_BLOCK 16
#define MAX_ORDER 6  // log2(1024/16) = 6

static char heap[HEAP_SIZE];

    size_t s = MIN_BLOCK;
    int o = 0;
    while (s < n && o < MAX_ORDER) { s <<= 1; o++; }

}

// buddy_addr = block_addr ^ block_size

class BuddyAllocator:
    def __init__(self, size):
        self.size = size
        self.min_block = 16
        self.max_order = 0
        s = size
        while s > self.min_block:
            s //= 2
            self.max_order += 1
        # Free lists per order
        self.free = {i: [] for i in range(self.max_order + 1)}
        self.free[self.max_order] = [0]  # whole block is free
        self.allocated = set()

    def _order_for(self, n):
        s, o = self.min_block, 0
        while s < n and o < self.max_order:
            s *= 2; o += 1
        return o

    def malloc(self, n):
        order = self._order_for(n)
        # Find smallest available order ≥ requested
        for o in range(order, self.max_order + 1):
            if self.free[o]:
                addr = self.free[o].pop()
                # Split down to requested order
                while o > order:
                    o -= 1
                    buddy = addr ^ (self.min_block << o)
                    self.free[o].append(buddy)
                self.allocated.add((addr, order))
                return addr
        return None

    def free(self, addr, order):
        self.allocated.discard((addr, order))
        while order < self.max_order:
            buddy = addr ^ (self.min_block << order)
            if buddy in self.free[order]:
                self.free[order].remove(buddy)
                addr = min(addr, buddy)
                order += 1
            else:
                break
        self.free[order].append(addr)

class BuddyAllocator {
  private free: Map<number, number[]> = new Map();
  private maxOrder: number;
  private minBlock = 16;

  constructor(size: number) {
    this.maxOrder = Math.log2(size / this.minBlock);
    for (let i = 0; i <= this.maxOrder; i++)
      this.free.set(i, []);
    this.free.get(this.maxOrder)!.push(0);
  }

  malloc(n: number): number | null {
    let order = 0, s = this.minBlock;
    while (s < n) { s *= 2; order++; }
    for (let o = order; o <= this.maxOrder; o++) {
      const list = this.free.get(o)!;
      if (list.length > 0) {
        let addr = list.pop()!;
        while (o > order) {
          o--;
          this.free.get(o)!.push(addr ^ (this.minBlock << o));
        }
        return addr;
      }
    }
    return null;
  }
}

func (b *Buddy) Alloc(n int) int {
    order := 0
    s := b.minBlock
    for s < n { s *= 2; order++ }
    for o := order; o <= b.maxOrder; o++ {
        if len(b.free[o]) > 0 {
            addr := b.free[o][len(b.free[o])-1]
            b.free[o] = b.free[o][:len(b.free[o])-1]

                o--
                b.free[o] = append(b.free[o], addr^(b.minBlock<<o))
            }
            return addr
        }
    }
    return -1
}

The XOR Trick

buddy_addr = block_addr ^ block_size. This works because buddy pairs are always aligned to twice their size. If a block starts at address A and has size S, its buddy is at A+S if A is aligned to 2S, or at A-S otherwise. XOR with S flips exactly the right bit to switch between these cases. One instruction, zero branches.

// 05 — Slab Allocation

In 1994, Jeff Bonwick at Sun Microsystems observed that the Solaris kernel spent enormous time allocating and initializing objects of the same type over and over. Inodes, task structs, socket buffers — the kernel has dozens of fixed-size objects that are created and destroyed millions of times. The buddy system wastes memory rounding these up to power-of-two sizes. Bonwick's solution: slab allocation. Pre-allocate pages, divide them into fixed-size slots for one object type, and cache initialized objects for instant reuse.

  │  (next, count)│  0  │  1  │  2  │  3  │  4   │  5   │  6  │
  └────────────────────────────────────────────────────────────┘
                    ▲                        ▲
                    allocated                free list head

  Three slab states:
    FULL    — all objects allocated (no free slots)
    PARTIAL — some objects allocated, some free (search here first)
    EMPTY   — all objects free (return to page allocator if needed)

The Linux kernel originally used Bonwick's SLAB, then SLUB became the default in kernel 2.6.23 (2007). SLUB simplified the design by storing metadata in page flags rather than per-slab structures. SLOB served embedded systems but was removed in kernel 6.4 (2023), followed by the original SLAB in kernel 6.8 (2024). SLUB is now the sole slab allocator in the Linux kernel.

Per-Thread Caching

Modern allocators like jemalloc add a per-thread cache (tcache) in front of the slab layer. When a thread frees an object, it goes into a thread-local magazine instead of back to the shared slab. When the thread allocates, it checks its magazine first — no lock, no contention. Magazines are drained back to the slab in batches when they fill up. This is Bonwick & Adams' magazine layer (2001), and it's the single biggest performance win in production allocators.

The Fastest Allocation

The fastest allocation is the one that doesn't need a lock. Per-thread caches turn the hot path — alloc and free of common sizes — into a thread-local stack pop or push. No atomics, no cache-line bouncing, no syscalls. This is why jemalloc and tcmalloc can sustain millions of allocations per second per core.

// 06 — Arena and Bump Allocators

Sometimes the fastest allocator is the dumbest one. A bump allocator maintains a pointer into a contiguous region of memory. To allocate n bytes, increment the pointer by n. That's it. One addition. No free lists, no splitting, no coalescing. The catch: you cannot free individual allocations. You free the entire arena at once by resetting the pointer to zero.

This sounds useless, but it's exactly what you want in several common scenarios. Compiler passes allocate thousands of AST nodes, process them, and discard the whole tree. Game engines allocate per-frame scratch data that lives for exactly one frame. HTTP servers allocate request-scoped objects that die when the response is sent. In all these cases, bulk deallocation is not just acceptable — it's ideal.

// 08 — Fragmentation

Fragmentation is the silent killer of long-running processes. Your server has 4GB of free memory, but malloc(64MB) fails because that free memory is scattered across thousands of small gaps. You have the capacity but not the contiguity. Unlike a memory leak — where memory is genuinely lost — fragmented memory is right there, taunting you, but unusable.

Internal Fragmentation

Internal fragmentation is wasted space inside allocated blocks. You ask for 20 bytes, the allocator gives you 32 (the next size class). Those 12 unused bytes are internal fragmentation. Every allocator with size classes or alignment requirements has some. The buddy system is worst: a 33-byte request gets a 64-byte block, wasting 48%. Good allocators space their size classes 12–25% apart to keep internal fragmentation below ~20%.

External Fragmentation

External fragmentation is unusable gaps between allocated blocks. After many alloc/free cycles, the heap becomes a checkerboard of used and free blocks. Each free block is too small for the next large request, even though the total free space is enormous.

  After many alloc/free cycles:
  [USED 32][free 16][USED 64][free 8][USED 16][free 24][USED 128][free 16][USED 32]

  Total free: 16 + 8 + 24 + 16 = 64 bytes
  But malloc(48) FAILS — no single contiguous free region ≥ 48 bytes

  All that free space, completely useless.

Knuth's 50% rule (from TAOCP Vol. 1, Section 2.5): under steady-state random allocation and deallocation with first-fit placement, the number of free holes approaches 50% of the number of allocated blocks. If you have 1000 allocated blocks, you'll have ~500 free gaps. This is a mathematical property of the steady state, not a bug — and it means fragmentation is inevitable in any long-running general-purpose allocator.

Worse Than a Leak

Memory leaks are bad — you lose memory over time. Fragmentation is worse — you have the memory but can't use it. Leaks are at least detectable with tools like Valgrind. Fragmentation is subtle: RSS grows, allocations start failing or triggering expensive compaction, and the only fix is to restart the process. This is why Redis, nginx, and many databases periodically fork or restart workers.

// 09 — mmap and the Kernel

When you call mmap(NULL, 4096, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0), the kernel does surprisingly little. It creates a virtual memory area (VMA) descriptor — a bookkeeping entry that says "this range of virtual addresses is valid." It does not allocate physical memory. No page table entries are created. No physical frames are touched. The actual allocation happens lazily, on first access.

Demand Paging

When your code first reads or writes a freshly mmap'd page, the CPU walks the page table and finds... nothing. No valid mapping. This triggers a page fault — a CPU exception that transfers control to the kernel's fault handler. The handler allocates a physical frame, zeros it (for security), installs the page table entry, and resumes your instruction. Your code never knows it happened, except for a few microseconds of latency on that first access.

Demand Paging Flow 
  Your code: *ptr = 42;
       │
       ▼
  CPU: translate virtual addr → TLB miss → page walk
       │
       ▼
  Page table: no entry! → PAGE FAULT (exception #14 on x86)
       │
       ▼
  Kernel page fault handler:
    1. Check VMA: is this address valid? (yes → continue, no → SIGSEGV)
    2. Allocate physical frame from buddy allocator
    3. Zero the frame (prevent information leaks)
    4. Install PTE (page table entry) mapping virtual → physical
    5. Flush TLB entry
       │

Huge Pages

With 4KB pages and a 64-entry TLB, you can only cover 256KB of working memory without TLB misses. For a database with a 10GB buffer pool, that's catastrophic. Huge pages solve this: 2MB pages (512× coverage per TLB entry) or 1GB pages (262,144×). A 64-entry TLB with 2MB pages covers 128MB; with 1GB pages, 64GB.

Transparent Huge Pages (THP, merged in Linux 2.6.38, 2011) automatically promote 4KB pages to 2MB huge pages when the kernel detects contiguous aligned allocations. The upside: no application changes needed. The downside: khugepaged daemon compaction can cause multi-millisecond latency spikes, and partial huge pages waste memory. Redis and MongoDB both recommend disabling THP in production (echo never > /sys/kernel/mm/transparent_hugepage/enabled).

Lazy by Design

The kernel is lazy about physical memory, and that's a feature. Demand paging means a process can map terabytes of virtual address space while consuming only the physical pages it actually touches. mmap(1GB) is nearly free — the cost comes page by page, exactly when needed. This is why Linux overcommit works: most processes map far more than they use.

// 10 — Garbage Collection

Manual memory management (C, Rust) puts the burden on the programmer. Garbage collection shifts it to the runtime. The GC periodically identifies objects that are no longer reachable from any root (stack variables, global variables, registers) and reclaims their memory. The programmer calls new but never calls free. The tradeoff: predictable latency for development speed.

Reference Counting

The simplest GC: each object has a count of how many references point to it. When the count drops to zero, free immediately. Python uses this as its primary mechanism. It's deterministic — destructors run at predictable times — but it cannot handle cycles. If A points to B and B points to A, both have refcount 1 forever, even if nothing else references them. Python supplements refcounting with a cycle detector that runs periodically.

Mark-and-Sweep

Start from roots, recursively mark every reachable object. Then sweep the heap: anything unmarked is garbage, free it. Simple and handles cycles, but requires a stop-the-world pause — all application threads must halt while the GC traverses the heap. For a 10GB heap, this pause can be seconds.

Generational Collection

The generational hypothesis: most objects die young. A temporary string, a loop variable, a request-scoped object — allocated and discarded within milliseconds. If an object survives a few collections, it's likely to live much longer. Generational GCs exploit this by dividing the heap into generations: the nursery (gen0) is collected frequently and cheaply; survivors are promoted to gen1, then gen2. Only gen0 collections are frequent; full-heap collections are rare.

Tri-Color Marking

Dijkstra et al. (1978) invented tri-color marking to enable concurrent garbage collection — the GC runs alongside the application, not stopping it. Objects are colored: white (not yet seen, presumed garbage), grey (seen but children not scanned), black (scanned, all children are grey or black). The GC repeatedly picks a grey object, scans its children (coloring them grey), and marks it black. When no grey objects remain, all white objects are garbage.

// 11 — Custom Allocators

General-purpose allocators are designed for the worst case. When you know your allocation patterns, you can do dramatically better. Custom allocators aren't premature optimization — they're the right tool when allocation is the bottleneck, and they appear in every high-performance system: game engines, databases, network stacks, and compilers.

Game Frame Allocator

A game engine runs at 60 FPS. Each frame allocates temporary data — visibility lists, particle states, AI decisions, collision results — that is only valid for that frame. A bump allocator with per-frame reset is perfect: O(1) alloc, zero fragmentation, instant bulk-free at frame boundary.

Database Buffer Pool

A database buffer pool is a fixed-size array of page-sized buffers (typically 8KB or 16KB). Pages are read from disk into pool slots and evicted using LRU or clock-sweep. Pin counting prevents eviction of in-use pages. This is a custom allocator in disguise: fixed-size slots, deterministic capacity, no external fragmentation.

Network Slab Allocator

Network stacks allocate millions of fixed-size packet buffers (e.g., 2KB MTU-sized buffers). A slab allocator with a free list of pre-allocated buffers gives O(1) alloc and O(1) free with zero fragmentation. DPDK (Data Plane Development Kit) uses per-core mempools of fixed-size mbufs for exactly this pattern.

When Custom Allocators Make Sense

Build a custom allocator when: (1) profiling shows malloc/free is in the top-5 CPU consumers, (2) you know the size distribution at compile time, (3) you have a clear lifetime pattern (per-frame, per-request, per-connection), or (4) you need guaranteed O(1) allocation for real-time systems. Don't guess — profile first with heaptrack or perf.

// 12 — Pool Allocators

A pool allocator manages a collection of fixed-size blocks. All blocks are the same size, pre-allocated in a contiguous region. The free list is intrusive: the "next free" pointer is stored inside each free block itself, consuming zero extra memory. Allocation is a linked-list pop; deallocation is a push. Both are O(1), branch-free on the hot path.

This is the allocator of choice for uniform objects: network connections, game entities, hash table nodes, ECS components. Any system where you create and destroy millions of same-sized objects benefits from a pool.

// Intrusive free-list pool: next pointer stored in free blocks
typedef struct Pool {
    void   *base;
    void   *free_head;  // head of free list
    size_t block_size;
    size_t count;
} Pool;

Pool pool_create(size_t block_size, size_t count) {
    if (block_size < sizeof(void*)) block_size = sizeof(void*);
    Pool p = { .base = malloc(block_size * count),
               .block_size = block_size, .count = count };
    // Chain all blocks into free list
    p.free_head = p.base;
    for (size_t i = 0; i < count - 1; i++) {
        void *block = (char*)p.base + i * block_size;
        *(void**)block = (char*)block + block_size;
    }
    *(void**)((char*)p.base + (count-1) * block_size) = NULL;
    return p;
}

void* pool_alloc(Pool *p) {
    if (!p->free_head) return NULL;
    void *block = p->free_head;
    p->free_head = *(void**)block;  // pop
    return block;
}

void pool_free(Pool *p, void *block) {
    *(void**)block = p->free_head;  // push
    p->free_head = block;
}

class Pool:
    def __init__(self, block_size, count):
        self.block_size = block_size
        self.count = count
        # Pre-allocate all blocks as a free stack
        self.free = list(range(count))  # indices as "pointers"
        self.data = [bytearray(block_size) for _ in range(count)]

    def alloc(self):
        if not self.free:
            return None
        return self.free.pop()  # O(1)

    def free_block(self, idx):
        self.free.append(idx)   # O(1)

    def available(self):
        return len(self.free)

class Pool<T> {
  private free: number[] = [];
  private items: (T | null)[];

  constructor(private capacity: number, private factory: () => T) {
    this.items = Array(capacity).fill(null);
    for (let i = capacity - 1; i >= 0; i--) this.free.push(i);
  }

  alloc(): { id: number; item: T } | null {
    if (this.free.length === 0) return null;
    const id = this.free.pop()!;
    this.items[id] = this.factory();
    return { id, item: this.items[id]! };
  }

  release(id: number) {
    this.items[id] = null;
    this.free.push(id);
  }
}

const pool = new Pool(4, () => ({ x: 0, y: 0 }));
const a = pool.alloc()!;
console.log("alloc:", a.id, a.item);
pool.release(a.id);
console.log("released, re-alloc:", pool.alloc());

struct Pool<T> {
    items: Vec<Option<T>>,
    free: Vec<usize>,
}

impl<T> Pool<T> {
    fn new(capacity: usize) -> Self {
        let items = (0..capacity).map(|_| None).collect();
        let free = (0..capacity).rev().collect();
        Pool { items, free }
    }

    fn alloc(&mut self, val: T) -> Option<usize> {
        let id = self.free.pop()?;
        self.items[id] = Some(val);
        Some(id)
    }

    fn free(&mut self, id: usize) -> Option<T> {
        let val = self.items[id].take()?;
        self.free.push(id);
        Some(val)
    }
}

    let id = pool.alloc("hello").unwrap();
    println!("alloc at slot {}", id);
    println!("free: {:?}", pool.free(id));
}

The Perfect Allocator for Uniform Objects

A pool allocator gives you O(1) alloc, O(1) free, zero fragmentation, zero metadata overhead, and perfect cache locality (objects are contiguous in memory). The constraint — all objects must be the same size — is not a limitation but a feature. If you can live with it, nothing else comes close.

// 13 — Memory Safety

Memory safety bugs are the #1 source of security vulnerabilities in systems software. Microsoft reported that ~70% of their CVEs are memory safety issues (Matt Miller, BlueHat IL 2019). Google Chrome's security team independently found the same ~70% figure for high-severity browser bugs (2020). These aren't obscure edge cases — they're the dominant class of exploitable vulnerability in the real world.

Use-After-Free (UAF)

Free an object, then access it through a dangling pointer. If the allocator has reused that memory for a different object, you read or write corrupted data. Attackers exploit this by controlling what object fills the freed slot — a technique called heap spraying.

Use-After-Free Exploitation 
  1. Victim allocates object A (64 bytes)
     ┌──────────────────┐
     │  Object A: vtable│ → methodA()
     │  data: "secret"  │
     └──────────────────┘

  2. Victim frees A (but keeps the pointer!)
     ┌──────────────────┐
     │  FREE            │  (allocator marks slot as available)
     └──────────────────┘

  3. Attacker allocates B (same size: 64 bytes) → lands in A's slot
     ┌──────────────────┐
     │  Object B: vtable│ → attacker_code()
     │  data: "pwned"   │
     └──────────────────┘

  4. Victim dereferences old pointer to A → reads B's vtable
     victim→method() → calls attacker_code()  ← arbitrary code execution

Double-Free

Calling free(ptr) twice corrupts the allocator's free list. The freed block is inserted twice, creating a cycle. The next two allocations return the same pointer, allowing one caller to read or overwrite another's data.

Rust's Ownership Model

Rust prevents all of the above at compile time with three rules: (1) every value has exactly one owner, (2) ownership can be transferred (moved) or temporarily borrowed, (3) you cannot have a mutable reference and any other reference simultaneously. The borrow checker enforces these rules with zero runtime cost — no GC, no reference counting, no safety overhead.

Rust Ownership Decision Tree 
  Do you need shared access?
  ├── No → move (transfer ownership)
  │         let b = a;  // a is GONE, only b exists
  │
  └── Yes → borrow
            ├── Read-only? → &T (shared reference, many allowed)
            │     let r1 = &data;
            │     let r2 = &data;  // OK: multiple readers
            │
            └── Need to mutate? → &mut T (exclusive reference, only one)
                  let m = &mut data;
                  // let r = &data;  ← COMPILE ERROR: can't read while mutating

  Lifetimes: compiler tracks how long each reference lives.
  Dangling reference? → compile error. UAF? → impossible.
  Double free? → impossible (only one owner calls drop).
The 70% Problem

~70% of security vulnerabilities in large C/C++ codebases are memory safety bugs: UAF, double-free, buffer overflow, uninitialized reads. This isn't a tooling problem — it's a language problem. Valgrind and ASan catch bugs in testing but can't guarantee their absence. Rust's borrow checker does: if it compiles, these bug classes are structurally impossible.

Zero-Cost Safety

Rust's ownership system adds zero runtime overhead. No GC pauses, no reference count increments, no bounds checks (unless you use [] indexing; .get() is checked). The safety comes entirely from compile-time analysis. The cost is paid in developer time fighting the borrow checker, not in CPU cycles.

// 14 — Tools

Memory bugs hide. They manifest as intermittent crashes, corrupted data hours after the initial bug, or silent data loss that you discover weeks later in production. The tools in this chapter are your weapons: runtime detectors, compile-time sanitizers, and profilers that expose allocation patterns.

Valgrind (Memcheck)

Valgrind instruments every memory access by running your program in a synthetic CPU. It tracks every malloc/free, detects leaks, UAF, reads of uninitialized memory, and invalid frees. The cost: 10–30× slowdown and ~2× memory overhead. Too slow for production, indispensable in CI and testing.

AddressSanitizer (ASan)

Google's AddressSanitizer (Serebryany et al., USENIX ATC 2012) instruments the binary at compile time. It places "red zones" around every allocation — poisoned bytes that trigger an immediate error on access. Overhead: ~2× time, ~3× memory. Ten times faster than Valgrind. Available in GCC and Clang via -fsanitize=address.

heaptrack

heaptrack traces every allocation and deallocation, recording the call stack. It produces flamegraphs showing where memory is allocated, how much, and whether it's freed. Unlike Valgrind, it uses LD_PRELOAD and runs at near-native speed. Essential for finding allocation hotspots before building a custom allocator.

// Compile with ASan: gcc -fsanitize=address -g prog.c
// Or run under Valgrind: valgrind --tool=memcheck --leak-check=full ./prog

#include <stdlib.h>
#include <string.h>

int main() {
    // Bug 1: heap buffer overflow
    char *buf = malloc(10);
    buf[10] = 'x';  // ASan: heap-buffer-overflow
    free(buf);

    // Bug 2: use-after-free
    char *p = malloc(64);

    return 0;
}

use std::alloc::{GlobalAlloc, Layout, System};
use std::sync::atomic::{AtomicUsize, Ordering};

struct CountingAlloc;
static ALLOC_COUNT: AtomicUsize = AtomicUsize::new(0);
static ALLOC_BYTES: AtomicUsize = AtomicUsize::new(0);

unsafe impl GlobalAlloc for CountingAlloc {
    unsafe fn alloc(&self, layout: Layout) -> *mut u8 {
        ALLOC_COUNT.fetch_add(1, Ordering::Relaxed);
        ALLOC_BYTES.fetch_add(layout.size(), Ordering::Relaxed);
        System.alloc(layout)

    unsafe fn dealloc(&self, ptr: *mut u8, layout: Layout) {
        System.dealloc(ptr, layout);
    }
}
