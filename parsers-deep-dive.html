01 — The Chomsky Hierarchy

Noam Chomsky classified formal languages into four levels in 1956, and this hierarchy still governs how we think about parsing 70 years later. Each level requires more powerful computational machinery to recognize. Know which level your problem lives at, and you know which parsing technique to reach for.

Key Insight

Most practical parsing lives at the Type 2/Type 3 boundary. You lex with regular expressions (Type 3), parse with context-free grammars (Type 2), then fix up context-sensitive stuff in a separate semantic pass. This layered approach is how every real compiler works.

Zen

Your glyph format is a beautiful example: @schema, @pool, @tab directives are lexed as regular tokens, the document structure (nesting, references) is context-free, and pool reference resolution is context-sensitive (needs a semantic pass). The three levels map naturally to three phases: lex → parse → resolve.

02 — Regex: The NFA Inside

When you write a(b|c)*d, what actually happens? Most people treat regex as a black box. But inside, there's an elegant state machine doing the work. Understanding it explains why some regexes are instant and others bring servers to their knees.

Thompson's Construction

Ken Thompson (yes, that Ken Thompson) published the algorithm in 1968. It converts any regex to a Nondeterministic Finite Automaton (NFA) with these simple rules:

Thompson's Construction Rules 
  Single character 'a':
    (start) ──a──► (end)

  Concatenation AB:
    (start_A) ──...──► (end_A/start_B) ──...──► (end_B)

  Alternation A|B:
           ε──► (A) ──ε─┐
    (start)─┤            ├──► (end)
           ε──► (B) ──ε─┘

  Kleene star A*:
              ε
    ┌────────────────┐
    │                │
    (start) ──ε──► (A) ──ε──► (end)
    │                          ▲
    └────────ε─────────────────┘

  ε = "epsilon transition" — move without consuming input

NFA for a(b|c)*d

NFA: a(b|c)*d 
              ε──► (3) ─b─► (4) ──ε─┐
  (0) ─a─► (1) ──ε──► (2)─┤         ├──► (5) ──ε──► (1)
                          ε──► (6) ─c─► (7) ──ε─┘
                           │
              (1) ──ε──────┴────────────────────► (8) ─d─► (9✓)

  States: 0-9, Accept: 9
  The NFA can be in MULTIPLE states simultaneously.
  Matching "abbd": {0} →a {1} →ε {2,8} →b {4,∅} →ε {5} →ε {1}
                   →ε {2,8} →b {4,∅} →ε {5} →ε {1} →ε {2,8} →d {9✓}

NFA Simulation

C#define MAX_STATES 256

typedef struct {
    int ch;            // character to match (-1 = epsilon)
    int next1, next2;  // transitions (next2 for splits)
} NFAState;

typedef struct {
    int states[MAX_STATES];
    int count;
} StateSet;

// Add state and follow epsilon transitions
void add_state(StateSet *set, NFAState *nfa, int s, int *visited) {
    if (s < 0 || visited[s]) return;
    visited[s] = 1;
    if (nfa[s].ch == -1) {  // epsilon — follow both transitions
        add_state(set, nfa, nfa[s].next1, visited);
        add_state(set, nfa, nfa[s].next2, visited);
    } else {
        set->states[set->count++] = s;
    }
}

// Simulate NFA: process input one character at a time
int nfa_match(NFAState *nfa, int start, int accept,
              const char *input) {
    StateSet current = {}, next = {};
    int visited[MAX_STATES] = {};

    add_state(&current, nfa, start, visited);

    for (const char *p = input; *p; p++) {
        next.count = 0;
        memset(visited, 0, sizeof(visited));

        for (int i = 0; i < current.count; i++) {
            int s = current.states[i];
            if (nfa[s].ch == *p)
                add_state(&next, nfa, nfa[s].next1, visited);
        }
        current = next;
    }

    for (int i = 0; i < current.count; i++)
        if (current.states[i] == accept) return 1;
    return 0;
}

// This runs in O(n × m) where n=input length, m=NFA states.
// NEVER exponential. NEVER backtracks.
// This is how RE2, Go's regexp, and Rust's regex crate work.
THOMPSON NFA: O(n×m) GUARANTEED — NO CATASTROPHIC BACKTRACKING

The Backtracking Trap

Danger

Python's re, JavaScript's RegExp, Ruby, PHP, Perl — they all use backtracking engines (PCRE-style). These support features NFA simulation can't (backreferences, lookahead), but at a terrifying cost: pathological patterns cause exponential blowup. The regex (a+)+$ on input "aaaaaaaaaaaaaaaaX" takes 2n steps. 30 a's = 1 billion steps. This has caused real production outages (Cloudflare, 2019). Use RE2 or Thompson NFA when processing untrusted patterns.

03 — From NFA to DFA

The NFA simulation tracks multiple states simultaneously. We can precompute all possible state sets and build a Deterministic Finite Automaton where each DFA state represents a set of NFA states. The DFA processes one character, looks up the next state in a table, done. No tracking multiple states at runtime. O(1) per character.

Subset Construction

NFA → DFA for a(b|c)*d 
  NFA state sets → DFA states:
    {0}       → D0 (start)
    {1,2,8}   → D1 (after 'a', with ε-closure)
    {4}       → D2 (in 'b' branch)
    {7}       → D3 (in 'c' branch)
    {9}       → D4 (accept)

  DFA transition table:
           a      b      c      d
    D0  → D1     ∅      ∅      ∅
    D1  → ∅      D1     D1     D4
    D4  → (accept)

  Wait — D2/D3 collapsed into D1 because after ε-transitions,
  they all end up back at {1,2,8}. The DFA is SMALLER than the NFA!

The DFA can have up to 2N states (power set of N NFA states). In practice, most DFAs are small. But adversarial patterns can cause exponential blowup in the DFA construction — which is why some engines lazily build DFA states on demand.

DFA Lookup Table

C// DFA as a simple lookup table
// Each state × each character → next state
#define DFA_DEAD  -1

typedef struct {
    int  transitions[128];  // ASCII → next state
    int  accepting;
} DFAState;

int dfa_match(DFAState *dfa, const char *input) {
    int state = 0;  // start state
    for (const char *p = input; *p; p++) {
        state = dfa[state].transitions[(unsigned char)*p];
        if (state == DFA_DEAD) return 0;
    }
    return dfa[state].accepting;
}

// O(n) time, O(1) per character. Branchless inner loop.
// This is why lexers are fast — they're DFA table lookups.
Key Insight

NFA = easy to construct (linear in regex size), potentially slow to run (multiple states). DFA = potentially expensive to construct (exponential blowup possible), but O(1) per character to run. The choice: if you compile the regex once and match many times, build the DFA. If you match once, simulate the NFA. Go's regex takes the middle path: lazily builds DFA states, caching only the ones actually needed.

04 — Lexing: The First Pass

Before you parse, you lex. Lexing (tokenization) converts a stream of characters into a stream of tokens — meaningful chunks like NUMBER, STRING, LBRACE, IDENTIFIER. This separation is powerful: the lexer handles character-level details (whitespace, escape sequences, number formats), and the parser works with clean, typed tokens.

Token Types

A Hand-Written Lexer

Ctypedef struct {
    const char *start;    // start of current token
    const char *current;  // current scan position
    int         line;
} Lexer;

static char advance(Lexer *L) { return *L->current++; }
static char peek(Lexer *L) { return *L->current; }
static int at_end(Lexer *L) { return *L->current == '\0'; }

static Token make_token(Lexer *L, TokenType type) {
    return (Token){type, L->start, (int)(L->current - L->start), L->line};
}

static void skip_whitespace(Lexer *L) {
    for (;;) {
        switch (peek(L)) {
        case ' ': case '\r': case '\t': advance(L); break;
        case '\n': L->line++; advance(L); break;
        default: return;
        }
    }
}

Token next_token(Lexer *L) {
    skip_whitespace(L);
    L->start = L->current;
    if (at_end(L)) return make_token(L, TOK_EOF);

    char c = advance(L);

    // Identifiers and keywords
    if (isalpha(c) || c == '_') {
        while (isalnum(peek(L)) || peek(L) == '_') advance(L);
        return make_token(L, TOK_IDENT);
    }

    // Numbers
    if (isdigit(c)) {
        while (isdigit(peek(L))) advance(L);
        if (peek(L) == '.') {
            advance(L);
            while (isdigit(peek(L))) advance(L);
            return make_token(L, TOK_FLOAT);
        }
        return make_token(L, TOK_INT);
    }

    // Strings
    if (c == '"') {
        while (peek(L) != '"' && !at_end(L)) {
            if (peek(L) == '\n') L->line++;
            if (peek(L) == '\\') advance(L); // skip escape
            advance(L);
        }
        if (at_end(L)) return make_token(L, TOK_ERROR);
        advance(L); // closing quote
        return make_token(L, TOK_STRING);
    }

    // Single-character tokens
    switch (c) {
    case '(': return make_token(L, TOK_LPAREN);
    case ')': return make_token(L, TOK_RPAREN);
    case '{': return make_token(L, TOK_LBRACE);
    case '}': return make_token(L, TOK_RBRACE);
    case '+': return make_token(L, TOK_PLUS);
    case '-': return make_token(L, TOK_MINUS);
    case '*': return make_token(L, TOK_STAR);
    case '/': return make_token(L, TOK_SLASH);
    case ',': return make_token(L, TOK_COMMA);
    case ':': return make_token(L, TOK_COLON);
    }

    return make_token(L, TOK_ERROR);
}

05 — Recursive Descent: Parse by Hand

Recursive descent is the most practical parsing technique in existence. One function per grammar rule. The call stack is the parse stack. No tables, no generators, no magic — just functions calling functions. Almost every hand-written parser (GCC, Clang, V8, Go, Rust) uses recursive descent.

Grammar → Code

Each grammar rule becomes a function. Terminal symbols consume tokens. Non-terminals call other functions:

Grammar → Recursive Descent 
  Grammar:                      Code:
  ─────────                     ─────
  value  → STRING               Token parse_value() {
         | NUMBER                 if (peek() == STRING) return consume();
         | object                 if (peek() == NUMBER) return consume();
         | array                  if (peek() == LBRACE) return parse_object();
                                  if (peek() == LBRACKET) return parse_array();
                                  error("expected value");
                                }

  object → '{' members '}'      Node* parse_object() {
                                  expect(LBRACE);
                                  Node *obj = parse_members();
                                  expect(RBRACE);
                                  return obj;
                                }

  array  → '[' elements ']'     Node* parse_array() {
                                  expect(LBRACKET);
                                  Node *arr = parse_elements();
                                  expect(RBRACKET);
                                  return arr;
                                }

A Complete JSON Parser

Ctypedef enum { N_NULL, N_BOOL, N_NUM, N_STR, N_ARR, N_OBJ } NodeType;

typedef struct Node {
    NodeType type;
    union {
        double       num;
        char        *str;
        struct { struct Node **items; int count; } arr;
        struct { char **keys; struct Node **vals; int count; } obj;
    };
} Node;

typedef struct { Lexer lex; Token current; } Parser;

static Token advance_p(Parser *P) {
    Token prev = P->current;
    P->current = next_token(&P->lex);
    return prev;
}

static int check(Parser *P, TokenType t) { return P->current.type == t; }

static void expect(Parser *P, TokenType t) {
    if (!check(P, t)) {
        fprintf(stderr, "line %d: expected %d, got %d\n",
                P->current.line, t, P->current.type);
        exit(1);
    }
    advance_p(P);
}

Node* parse_value(Parser *P) {
    if (check(P, TOK_STRING)) {
        Token t = advance_p(P);
        Node *n = calloc(1, sizeof(*n));
        n->type = N_STR;
        n->str = strndup(t.start + 1, t.length - 2);
        return n;
    }
    if (check(P, TOK_INT) || check(P, TOK_FLOAT)) {
        Token t = advance_p(P);
        Node *n = calloc(1, sizeof(*n));
        n->type = N_NUM;
        n->num = strtod(t.start, NULL);
        return n;
    }
    if (check(P, TOK_LBRACE))  return parse_object(P);
    if (check(P, TOK_LBRACKET)) return parse_array(P);
    fprintf(stderr, "unexpected token\n");
    exit(1);
}
Key Insight

Recursive descent is LL(1): it looks at one token ahead to decide which rule to apply. If you can always decide by looking at the current token, you're golden. The main limitation: left recursion (expr → expr '+' term) causes infinite recursion. Fix: rewrite as right recursion or iteration, or use Pratt parsing (next chapter).

Recursive Descent in Production

Almost every major compiler is hand-written recursive descent: GCC (switched from Bison in 2004), Clang, V8 (JavaScript), Go (go/parser), Rust (rustc), TypeScript, Swift, Kotlin. Generated parsers (yacc/bison/ANTLR) dominated in the 1980s-90s but lost ground because hand-written parsers give you full control over error messages, recovery, and incremental parsing. The extra boilerplate is worth it when your error messages are user-facing.

06 — Pratt Parsing: Elegant Expressions

Vaughan Pratt published "Top Down Operator Precedence" in 1973 and solved expression parsing forever. The idea is deceptively simple: assign each operator a binding power (a number representing precedence), and use two kinds of handler functions — nud (null denotation, for prefix position) and led (left denotation, for infix position).

The Core Loop

Ctypedef struct Expr {
    enum { E_NUM, E_UNARY, E_BINARY, E_CALL } kind;
    union {
        double num;
        struct { int op; struct Expr *operand; } unary;
        struct { int op; struct Expr *left, *right; } binary;
    };
} Expr;

// Binding power (precedence) table
static int bp(TokenType t) {
    switch (t) {
    case TOK_PLUS: case TOK_MINUS: return 10;
    case TOK_STAR: case TOK_SLASH: return 20;
    default: return 0;
    }
}

// THE Pratt parsing loop — this is the whole algorithm
Expr* parse_expr(Parser *P, int min_bp) {
    // nud: handle prefix (numbers, unary minus, parens)
    Expr *left;
    if (check(P, TOK_INT) || check(P, TOK_FLOAT)) {
        Token t = advance_p(P);
        left = calloc(1, sizeof(*left));
        left->kind = E_NUM;
        left->num = strtod(t.start, NULL);
    } else if (check(P, TOK_MINUS)) {
        advance_p(P);
        left = calloc(1, sizeof(*left));
        left->kind = E_UNARY;
        left->unary.op = TOK_MINUS;
        left->unary.operand = parse_expr(P, 30); // high bp for unary
    } else if (check(P, TOK_LPAREN)) {
        advance_p(P);
        left = parse_expr(P, 0);
        expect(P, TOK_RPAREN);
    } else {
        fprintf(stderr, "expected expression\n");
        exit(1);
    }

    // led: handle infix operators
    while (bp(P->current.type) > min_bp) {
        Token op = advance_p(P);
        Expr *right = parse_expr(P, bp(op.type)); // left-assoc
        Expr *node = calloc(1, sizeof(*node));
        node->kind = E_BINARY;
        node->binary.op = op.type;
        node->binary.left = left;
        node->binary.right = right;
        left = node;
    }
    return left;
}

07 — PEG Grammars: Ordered Choice

Parsing Expression Grammars (PEGs) look like context-free grammars but work fundamentally differently. The key: the choice operator / is ordered. It tries alternatives left-to-right and commits to the first one that succeeds. No ambiguity. No backtracking dilemma. The grammar is the parser.

PEG vs CFG

CFG (Ambiguous)

expr → expr "+" expr
     | expr "*" expr
     | NUMBER

// "1 + 2 * 3" has two
// valid parse trees!
// Need precedence rules
// to disambiguate.

PEG (Deterministic)

expr ← term ("+" term)*
term ← atom ("*" atom)*
atom ← NUMBER / "(" expr ")"

// Ordered choice means
// exactly ONE parse.
// Precedence is built
// into the grammar.

Packrat Parsing

PEG with memoization = packrat parsing. Every call to a parsing function at a given position is cached. This guarantees O(n) time regardless of the grammar, at the cost of O(n × rules) memory. Think of it as dynamic programming applied to parsing.

C// Memoization table for packrat parsing
typedef struct {
    int computed;   // has this cell been computed?
    int success;    // did the rule match?
    int end_pos;    // position after match
    Node *result;   // parse tree node
} MemoEntry;

// memo[rule_id][position] = MemoEntry
MemoEntry memo[NUM_RULES][MAX_INPUT];

Node* peg_rule(int rule, int pos) {
    MemoEntry *m = &memo[rule][pos];
    if (m->computed) {
        // Cache hit — return memoized result
        global_pos = m->end_pos;
        return m->success ? m->result : NULL;
    }
    m->computed = 1;
    // ... actually parse and fill in memo entry ...
}
Key Insight

PEGs can't handle left recursion naively (it causes infinite recursion, same as LL parsers). Solutions exist (Warth's algorithm for left-recursive PEGs) but add complexity. In practice, you rewrite left-recursive rules as loops, same as with recursive descent.

C#define MAX_STATES  32
#define MAX_EVENTS  16

typedef void (*ActionFn)(void *ctx);
typedef int  (*GuardFn)(void *ctx);

typedef struct {
    int       next_state;
    ActionFn  action;     // run on transition (can be NULL)
    GuardFn   guard;      // only transition if guard returns true
} Transition;

typedef struct {
    Transition table[MAX_STATES][MAX_EVENTS];
    int        current;
    ActionFn   on_enter[MAX_STATES];
    ActionFn   on_exit[MAX_STATES];
} StateMachine;

int sm_send(StateMachine *sm, int event, void *ctx) {
    Transition *t = &sm->table[sm->current][event];
    if (t->next_state < 0) return -1;  // no transition

    if (t->guard && !t->guard(ctx))
        return 0;  // guard rejected

    // Exit current state
    if (sm->on_exit[sm->current])
        sm->on_exit[sm->current](ctx);

    // Run transition action
    if (t->action)
        t->action(ctx);

    // Enter new state
    sm->current = t->next_state;
    if (sm->on_enter[sm->current])
        sm->on_enter[sm->current](ctx);

    return 1;
}

09 — Parser Combinators: Compose Everything

What if parsers were values you could combine with operators? That's the parser combinator idea. A parser is a function that takes input and returns either a result + remaining input, or failure. You build complex parsers by combining simple ones with seq (sequence), alt (choice), and many (repetition).

The Core Combinators

Ctypedef struct {
    int         success;
    const char *rest;      // remaining input
    const char *match;     // matched text start
    int         match_len;
} ParseResult;

typedef ParseResult (*ParserFn)(const char *input);

// Match a single character
ParseResult ch(const char *input, char c) {
    if (*input == c)
        return (ParseResult){1, input + 1, input, 1};
    return (ParseResult){0};
}

// Match a string literal
ParseResult lit(const char *input, const char *s) {
    int len = strlen(s);
    if (strncmp(input, s, len) == 0)
        return (ParseResult){1, input + len, input, len};
    return (ParseResult){0};
}

// Match while predicate holds (like [a-zA-Z]+)
ParseResult take_while(const char *input, int (*pred)(int)) {
    const char *p = input;
    while (*p && pred(*p)) p++;
    if (p == input) return (ParseResult){0};
    return (ParseResult){1, p, input, (int)(p - input)};
}

// Sequence: run A then B
ParseResult seq2(const char *input, ParserFn a, ParserFn b) {
    ParseResult ra = a(input);
    if (!ra.success) return ra;
    ParseResult rb = b(ra.rest);
    if (!rb.success) return rb;
    return (ParseResult){1, rb.rest, input, (int)(rb.rest - input)};
}

// Alternative: try A, if fails try B
ParseResult alt2(const char *input, ParserFn a, ParserFn b) {
    ParseResult ra = a(input);
    if (ra.success) return ra;
    return b(input);
}
Zen

Parser combinators are beautiful because they're compositional. Each combinator is a function, and functions compose. You build a JSON parser by saying "a value is a string OR a number OR an array OR an object, where an array is '[' value (',' value)* ']'." The code reads like the grammar. Haskell's Parsec library made this famous, but the pattern works in any language with first-class functions.

10 — Error Recovery: When Things Go Wrong

A parser that stops at the first error is useless for IDEs, linters, and formatters. Real parsers need to recover from errors and keep going, reporting as many problems as possible in one pass. This is hard. Most of the complexity in production parsers is error handling, not parsing.

Panic Mode Recovery

The simplest strategy: when an error occurs, skip tokens until you find a synchronization point (a token that likely starts a new statement or declaration):

Cvoid synchronize(Parser *P) {
    // Skip tokens until we find a safe restart point
    while (!check(P, TOK_EOF)) {
        // Semicolons end statements — safe to restart after
        if (P->previous.type == TOK_SEMICOLON) return;

        // These tokens start new constructs
        switch (P->current.type) {
        case TOK_FN:
        case TOK_LET:
        case TOK_IF:
        case TOK_WHILE:
        case TOK_RETURN:
        case TOK_LBRACE:
            return;
        default:
            advance_p(P);
        }
    }
}

Embedded vs External DSLs

Embedded DSLs live inside a host language, borrowing its syntax: Ruby blocks (Rake, RSpec, Chef), Kotlin DSL builders (Gradle, Ktor), Elixir macros (Ecto queries), Rust proc macros (serde, clap derive). You get the host language's editor support, type checking, and debugging for free.

External DSLs have their own syntax and parser: SQL, regex, HCL (Terraform), GraphQL, your glyph format. More powerful syntax freedom, but you must build tooling from scratch.

The hybrid: Rust's proc macros are uniquely powerful because syn/quote give you full access to Rust's AST at compile time. You can invent arbitrary syntax inside macro_rules! or derive macros that compiles to valid Rust. It's an external DSL wearing an embedded DSL's clothes.

Rule of thumb: if your DSL needs to express computation (conditions, loops, variables), make it embedded. If it's primarily declarative data (schemas, configs, queries), external is cleaner.

12 — Build Your Own: Expression Parser

Let's tie it all together. A complete, compilable expression parser in C: lexer + Pratt parser + AST + tree-walk evaluator. Handles integers, floats, arithmetic, unary minus, parentheses, and named variables. Under 200 lines.

C#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <ctype.h>
#include <math.h>

// === TOKENS ===
enum { T_NUM, T_ID, T_PLUS, T_MINUS, T_MUL, T_DIV,
       T_MOD, T_POW, T_LP, T_RP, T_COMMA, T_EOF, T_ERR };

typedef struct { int type; double num; char id[64]; } Tok;

typedef struct { const char *p; Tok cur; } Lex;

static void lex_next(Lex *L) {
    while (isspace(*L->p)) L->p++;
    if (!*L->p) { L->cur.type = T_EOF; return; }
    if (isdigit(*L->p) || (*L->p == '.' && isdigit(L->p[1]))) {
        L->cur.type = T_NUM;
        L->cur.num = strtod(L->p, (char**)&L->p);
        return;
    }
    if (isalpha(*L->p) || *L->p == '_') {
        int i = 0;
        while (isalnum(*L->p) || *L->p == '_')
            L->cur.id[i++] = *L->p++;
        L->cur.id[i] = 0;
        L->cur.type = T_ID;
        return;
    }
    L->cur.type = *L->p == '+' ? T_PLUS : *L->p == '-' ? T_MINUS :
                  *L->p == '*' ? T_MUL  : *L->p == '/' ? T_DIV :
                  *L->p == '%' ? T_MOD  : *L->p == '^' ? T_POW :
                  *L->p == '(' ? T_LP   : *L->p == ')' ? T_RP :
                  *L->p == ',' ? T_COMMA : T_ERR;
    L->p++;
}

// === AST ===
enum { A_NUM, A_VAR, A_UN, A_BIN, A_CALL };

typedef struct Ast {
    int kind;
    union {
        double num;
        char var[64];
        struct { int op; struct Ast *x; } un;
        struct { int op; struct Ast *l, *r; } bin;
        struct { char name[64]; struct Ast *args[8]; int nargs; } call;
    };
} Ast;

static Ast* mk(int k) { Ast*a=calloc(1,sizeof(*a)); a->kind=k; return a; }

// === PRATT PARSER ===
static int bp(int t) {
    switch(t) {
    case T_PLUS: case T_MINUS: return 10;
    case T_MUL: case T_DIV: case T_MOD: return 20;
    case T_POW: return 30;
    default: return 0;
    }
}

static Ast* expr(Lex *L, int mbp);

static Ast* nud(Lex *L) {
    if (L->cur.type == T_NUM) {
        Ast *a = mk(A_NUM); a->num = L->cur.num;
        lex_next(L); return a;
    }
    if (L->cur.type == T_ID) {
        char name[64]; strcpy(name, L->cur.id);
        lex_next(L);
        if (L->cur.type == T_LP) {  // function call
            lex_next(L);
            Ast *c = mk(A_CALL); strcpy(c->call.name, name);
            if (L->cur.type != T_RP) {
                c->call.args[c->call.nargs++] = expr(L, 0);
                while (L->cur.type == T_COMMA) {
                    lex_next(L);
                    c->call.args[c->call.nargs++] = expr(L, 0);
                }
            }
            lex_next(L); // skip ')'
            return c;
        }
        Ast *a = mk(A_VAR); strcpy(a->var, name);
        return a;
    }
    if (L->cur.type == T_MINUS) {
        lex_next(L);
        Ast *a = mk(A_UN); a->un.op = T_MINUS;
        a->un.x = expr(L, 25); return a;
    }
    if (L->cur.type == T_LP) {
        lex_next(L);
        Ast *a = expr(L, 0);
        lex_next(L); // skip ')'
        return a;
    }
    fprintf(stderr, "unexpected token %d\n", L->cur.type);
    exit(1);
}

static Ast* expr(Lex *L, int mbp) {
    Ast *left = nud(L);
    while (bp(L->cur.type) > mbp) {
        int op = L->cur.type;
        lex_next(L);
        int rbp = (op == T_POW) ? bp(op) - 1 : bp(op); // right-assoc for ^
        Ast *a = mk(A_BIN);
        a->bin.op = op; a->bin.l = left; a->bin.r = expr(L, rbp);
        left = a;
    }
    return left;
}

// === EVALUATOR ===
typedef struct { char name[64]; double val; } Var;

static double eval(Ast *a, Var *vars, int nv) {
    switch (a->kind) {
    case A_NUM: return a->num;
    case A_VAR:
        for (int i = 0; i < nv; i++)
            if (strcmp(vars[i].name, a->var) == 0)
                return vars[i].val;
        fprintf(stderr, "undefined: %s\n", a->var); exit(1);
    case A_UN: return -eval(a->un.x, vars, nv);
    case A_BIN: {
        double l = eval(a->bin.l, vars, nv);
        double r = eval(a->bin.r, vars, nv);
        switch (a->bin.op) {
        case T_PLUS:  return l + r;
        case T_MINUS: return l - r;
        case T_MUL:   return l * r;
        case T_DIV:   return l / r;
        case T_MOD:   return fmod(l, r);
        case T_POW:   return pow(l, r);
        }
    }
    case A_CALL: {
        double args[8];
        for (int i = 0; i < a->call.nargs; i++)
            args[i] = eval(a->call.args[i], vars, nv);
        if (!strcmp(a->call.name, "sqrt"))  return sqrt(args[0]);
        if (!strcmp(a->call.name, "sin"))   return sin(args[0]);
        if (!strcmp(a->call.name, "cos"))   return cos(args[0]);
        if (!strcmp(a->call.name, "abs"))   return fabs(args[0]);
        if (!strcmp(a->call.name, "max"))   return fmax(args[0], args[1]);
        if (!strcmp(a->call.name, "min"))   return fmin(args[0], args[1]);
        fprintf(stderr, "unknown fn: %s\n", a->call.name); exit(1);
    }
    }
    return 0;
}

13 — Scripting Language Internals

Every scripting language follows the same pipeline we've covered in chapters 4–11 — lexing, parsing, AST construction — but the story doesn't end there. Where compiled languages emit machine code, scripting languages compile to bytecode and run it on a virtual machine. The VM is the execution backend: a loop that fetches an instruction, dispatches it, and repeats. Understanding this pipeline — and the design choices within it — explains why languages feel fast or slow, why closures work, and where JIT compilers fit in.

13.1 — The Pipeline

Scripting Language Pipeline 
  source → lexer → tokens → parser → AST → compiler → bytecode → VM loop → result

Python, Lua, Ruby, JavaScript (V8), and PHP 8 all follow this exact pipeline. The front half — lexer through AST — is identical to what we built in chapters 4–7. The back half — compiler through VM — is where the real divergence happens. The differences come down to VM architecture and optimization strategy.

13.2 — Stack-Based vs Register-Based VMs

There are two fundamental VM architectures. They differ in how operands are addressed — implicitly via a stack, or explicitly via named registers.

Consider the expression a + b * c:

Stack-Based VM (Python/CPython, Ruby MRI, JVM, .NET CLR) 
  LOAD_FAST  a          # push a
  LOAD_FAST  b          # push b
  LOAD_FAST  c          # push c
  BINARY_MULTIPLY       # pop b,c → push b*c
  BINARY_ADD            # pop a,(b*c) → push a+b*c

Five instructions. Operands are implicit — they live on the stack. The compiler never needs to worry about register allocation. Simple to generate, but more instructions means more dispatch overhead.

Register-Based VM (Lua 5.0+, LuaJIT, Android Dalvik/ART) 
  MUL  R2  R1  R0      # R2 = b * c
  ADD  R3  R2  R0      # R3 = a + b*c

Two instructions. Operands are explicit — each instruction names its registers. Fewer dispatches, but the compiler must handle register allocation, and each instruction is wider (more bytes to encode the register operands).

13.3 — Closures and Upvalues

A closure is a function that captures variables from its enclosing scope. The key problem: when the enclosing function returns, the stack frame is gone. Where does the captured variable live?

Lua's elegant solution: upvalues. When a nested function references an outer local, Lua creates an upvalue — an indirection that points to the variable on the stack. When the outer function returns, the variable is "closed over" — moved from the stack to the heap. The upvalue pointer is updated, and the inner function still works. Robert Ierusalimschy's design is the cleanest closure implementation in any language.

Upvalue Lifecycle 
  Before return:   upvalue → stack slot [n=0]     (variable lives on stack)
  After return:    upvalue → heap copy  [n=0]     (stack is gone, value preserved)

13.5 — JIT Compilation

Interpreting bytecode is roughly 10–100x slower than native code. JIT (Just-In-Time) compilers bridge the gap by compiling hot code paths to machine code at runtime. The key insight: most programs spend most of their time in a small fraction of the code. Compile that fraction and leave the rest interpreted.

V8's Tiered Pipeline 
  Source → Parser → AST → Ignition (bytecode interpreter)

LuaJIT's trace compiler: instead of compiling functions, LuaJIT records "traces" — hot loop iterations. It compiles the trace (a linear sequence of operations) to machine code. If a guard fails (type check, unexpected branch), it falls back to the interpreter. Mike Pall's masterpiece — often faster than statically compiled C for numerical code.

PyPy's meta-tracing: write an interpreter in RPython, and PyPy's JIT framework automatically generates a JIT compiler for your language. The interpreter is the specification; the JIT is derived mechanically. A remarkable approach — you get a JIT "for free" by writing an interpreter in the right way.

14 — Parsers in the Wild

Theory is beautiful. Production is war. This chapter covers real-world parsers that serve billions of requests, compile millions of programs, and occasionally crash spectacularly. Each case study reveals a parsing technique from the earlier chapters, battle-tested at scale.

14.1 — CPython's PEG Migration

For 30 years (1990–2020), CPython used an LL(1) parser generated by pgen, a custom parser generator Guido van Rossum wrote for Python's very first release. The grammar lived in Grammar/Grammar, a BNF-like file that pgen compiled into parsing tables. It worked. But LL(1) has a hard constraint: at most one token of lookahead, and no ambiguity in the grammar.

That constraint increasingly chafed. Assignment expressions (:=, the "walrus operator"), match statements, and some starred expressions required awkward grammar workarounds — synthetic intermediate rules that existed only to satisfy LL(1), making the grammar harder to read and maintain. Some valid Python constructs were parsed with an overly permissive grammar rule and then rejected in a post-parse validation pass — a code smell that the grammar and the language had drifted apart.

Python 3.9 (PEP 617) replaced the entire front end with a PEG parser designed by Guido van Rossum and Pablo Galindo Salgado. The new grammar file, Grammar/python.gram, uses PEG syntax with ordered choice (/), unlimited lookahead, and left recursion support. A code generator compiles these rules to C — the generated parser is a recursive descent parser with packrat memoization (chapter 7's technique).

The surprise: the PEG parser is actually faster than the old LL(1) parser. Memoization avoids redundant work that the LL(1) parser paid for through backtracking in its validation passes, and the generated C code has better cache behavior. Python 3.10 removed the old parser entirely.

Hack

Python's grammar file Grammar/python.gram is ~700 lines of PEG rules that compile to ~30,000 lines of C. The grammar is readable; the generated parser is not. This is the parser generator bargain — and why CPython chose PEG over hand-writing a recursive descent parser.

14.2 — V8's Lazy Parsing

V8 doesn't parse all JavaScript upfront. When it encounters a function definition, it makes a bet: will this function be called soon, or never? If V8 guesses "not soon," it pre-parses the function — a fast scan that validates syntax and finds scope boundaries but doesn't build an AST or generate bytecode. Only when the function is actually called does V8 fully parse it (lazy compilation).

This is critical for page load performance. A typical web page loads megabytes of JavaScript but only executes a fraction on first paint. Pre-parsing the rest saves significant time and memory — no AST nodes allocated, no bytecode generated, no constant pool built for code that may never run.

V8 Lazy Parsing 
  function foo() { ... }   ← pre-parsed (fast): syntax ok, scopes found
  foo();                   ← now full-parse: AST → Ignition bytecode
  function bar() { ... }   ← pre-parsed, never called → never fully parsed

The cost: if V8 guesses wrong (pre-parses a function that's called immediately), it pays double — pre-parse then full-parse. IIFEs (function(){...})() are hinted as "eager" to avoid this penalty. V8 also uses heuristics: functions that are arguments to .forEach, .map, etc. are parsed eagerly because they're almost always called immediately.

14.3 — tree-sitter: Incremental Parsing

Most parsers start from scratch every time. Change one character in a 10,000-line file, and the parser re-lexes and re-parses the entire thing. tree-sitter doesn't. It re-parses only the parts of the file that changed — edit-aware incremental parsing.

The key insight: tree-sitter's output is a concrete syntax tree (CST) stored as a persistent data structure. When you edit a character, tree-sitter receives the edit description (position, old text, new text) and re-parses only the affected subtree, reusing all unchanged nodes. This achieves sub-millisecond re-parse on keystroke for any supported language.

tree-sitter is used by Neovim (since 0.5), Helix, Zed, GitHub code navigation, Emacs 29+, and dozens of other tools. It's built on GLR parsing (generalized LR) — a parser that can handle ambiguous grammars by maintaining multiple parse stacks simultaneously. This is necessary for languages like C++ where some constructs are genuinely ambiguous without semantic information (is T(x) a function call or a cast?).

tree-sitter's Meta-DSL

tree-sitter grammars are written in JavaScript — a DSL for defining grammars. The grammar JavaScript file is executed by Node.js to produce a JSON grammar description, which the tree-sitter CLI compiles to a C parser. The generated C code is checked into the repo — no build-time dependency on tree-sitter itself. As of 2025, there are tree-sitter grammars for 200+ languages.

14.4 — Parsing Binary Formats

Binary format parsing is still parsing — the same state machine principles from chapter 8. The critical difference: no lexer needed. In text parsing, the lexer handles character-level details (is this a number or identifier?). In binary parsing, the "tokens" are type tags read directly from bytes. The first byte is the token.

Protobuf uses tag-length-value (TLV) framing. Each field has a wire type (varint, 64-bit, length-delimited, etc.) embedded in the tag byte. Deserialization is schema-driven — the parser needs the .proto definition to interpret field numbers. Without the schema, you can parse the structure but not the semantics.

MessagePack/CBOR use self-describing type tags. The first byte tells you the type AND sometimes the value (fixint, fixstr, fixarray). No schema needed — the format is self-describing. Your cowrie format uses this same pattern: inline types (0x40–0xEF) encode both the type and small values in a single byte.

14.5 — The Billion-Dollar Regex

Cloudflare outage, July 2, 2019. A single WAF regex rule caused catastrophic backtracking, taking down Cloudflare's entire network for 27 minutes. The regex was a standard-looking security pattern, but its (.*)* structure created exponential backtracking in the PCRE engine. One regex, 27 minutes of global downtime.

Stack Overflow, July 2016. A regex in the Markdown renderer caused 100% CPU on certain posts. The pattern worked fine on normal input but exploded on adversarial strings. The fix was to rewrite the regex with possessive quantifiers and atomic groups — PCRE features that prevent backtracking.

ReDoS attacks (Regular Expression Denial of Service) exploit this systematically. Attackers craft input that triggers exponential backtracking in PCRE-style engines. Any regex with nested quantifiers (a+)+, alternation with overlap (a|a)*, or .*(pattern).* is potentially vulnerable. The attack surface is enormous: every regex in every WAF, rate limiter, input validator, and markdown renderer.

The fix is from chapter 2: use Thompson NFA engines (RE2, Go regexp, Rust regex) for any regex that processes untrusted input. They guarantee O(n×m) time — no backtracking, no catastrophic blowup. The Thompson construction we covered isn't just theory; it's a security boundary.

Danger

Every regex in your WAF, rate limiter, input validator, or markdown renderer is a potential DoS vector if you're using a backtracking engine (PCRE, Python re, JavaScript RegExp, Ruby, PHP). Audit them. Switch to RE2/Go/Rust regex for untrusted input. Or set a timeout — .NET's Regex class supports TimeSpan limits, and Python 3.11 added re.TIMEOUT.
