INFERENCE

How LLMs actually run at scale: tokenization, attention, cache management, batching, streaming, and all the unpleasant latency math in between._

We are going to build a chat completion service that feels instant to a human and still stays upright when hundreds of users hit it at once. That means less fascination with model mysticism and more attention to what the machine is really doing: reading bytes, turning them into token IDs, moving weight tensors from HBM, allocating KV pages, scheduling decode steps, and dribbling tokens back over a socket before the caller gets impatient.

Hack: Watch The First Token

If the first token comes out late, the model is not "thinking deeply." Your stack is paying prompt debt. Somewhere upstream, bytes turned into too many tokens, the scheduler admitted a giant prefill at the wrong time, or the cache allocator started rationing survival.

01 - The Request Path

An inference server is not a model with an HTTP wrapper. It is a pipeline. The request arrives as bytes on a TCP socket, is parsed into an API shape, authenticated, templated into a prompt, tokenized, queued, prefixed through the network, decoded token by token, sampled, detokenized, and streamed back out. Miss any one layer and the whole thing feels slow, fragile, or both.

TLS terminate-> parse JSON-> chat template-> tokenize-> prefill-> decode loop-> stream tokens

The front half looks suspiciously like the problems in Parsers and The Network Stack. You are still doing framing, validation, buffering, and backpressure. The back half looks like GPU Compute with a particularly expensive inner loop. That is why good inference engineers are usually good systems engineers first.

Latency budget
The user perceives two different latencies: time-to-first-token and sustained tokens per second. Both matter. Fast first token with sluggish decode still feels bad.
Memory budget
Weights are large, activations spike during prefill, and KV cache grows with every token. Most serving incidents are memory incidents in disguise.
Scheduler budget
Single-user benchmarks lie. Production throughput depends on how you merge short prompts, long prompts, and half-finished generations into one GPU schedule without letting one request poison everyone else.
Concrete target

Assume one 70B-class chat model across 8 GPUs, 8k context, streaming responses, and a product target of roughly 250ms time-to-first-token for short prompts plus 30 to 50 tokens per second once decode is underway. The exact numbers change. The shape of the problem does not.

02 - Tokens Before Math

The model does not see strings. It sees integer token IDs. Everything upstream of that conversion decides cost. A sloppy chat template can add hundreds of tokens. A verbose tool schema can add thousands. The most expensive prompt optimization is often not on the GPU at all. It is deleting useless text.

Prompt assembly is part of inference

Most production stacks take structured input like messages, system instructions, tool definitions, and generation parameters, then serialize them through a model-specific chat template. This step is deterministic and deeply important. Change the delimiters or role headers and you change model behavior. Change the byte layout and you change sequence length, which changes cost.

python# simplified chat templating pass
def render_prompt(messages, tools):
    parts = ["<s>[SYSTEM]\\nYou are concise.\\n[/SYSTEM]\\n"]
    for m in messages:
        role = m["role"].upper()
        parts.append(f"[{role}]\\n{m['content']}\\n[/{role}]\\n")
    if tools:
        parts.append("[TOOLS]\\n" + serialize_tools(tools) + "\\n[/TOOLS]\\n")
    parts.append("[ASSISTANT]\\n")
    return "".join(parts)

The tokenizer then greedily matches byte sequences into subword units. This is why Unicode normalization, whitespace policy, and delimiter choices matter. The byte-stream view is exactly the sort of problem described in Parsers. The model may be glamorous; the front end is glorified lexical analysis.

Prompt source Typical failure Why it hurts
Verbose system prompt Repeats policy and examples on every call Burns prefill time and KV capacity for text the model already "knows"
Tool schemas Massive JSON schemas included even when unused Bloats prompt length and increases first-token latency
Conversation history Blindly appends the entire chat Turns every follow-up into a long-context request
The boring optimization

The cheapest way to serve a model faster is to send fewer tokens into it. This sounds obvious and still gets ignored constantly because people treat prompt construction as product logic rather than infrastructure.

03 - Prefill: Reading the Whole Prompt

Prefill is the first forward pass over the entire prompt. Every token attends to all earlier tokens, layer by layer, and the server materializes the keys and values that future decode steps will reuse. Prefill is parallel across prompt positions, which is the good news. The bad news is that long prompts scale brutally because attention cost grows with sequence length.

One request during prefill 
prompt tokens:  [t0][t1][t2][t3][t4][t5][t6][t7]

layer 1:        full attention over visible prefix for every token
layer 2:        full attention again, new projections, new activations
...
layer N:        final hidden state for token t7 becomes next-token context

output:         logits for the next token
side effect:    KV cache pages written for t0..t7 at every layer

Prefill is where prompt length hurts

For a short prompt, prefill disappears into the noise. For a 6k-token prompt, prefill dominates time-to-first-token. This is why long-context demos look impressive on stage and miserable in production. The arithmetic is not subtle: more tokens means more attention work, more activation memory, and more KV pages to write before the user sees anything.

pseudo# one transformer layer, heavily simplified
X = embed(input_ids)                  # [seq, hidden]
Q = X @ Wq
K = X @ Wk
V = X @ Wv
scores = (Q @ K.T) / sqrt(head_dim)  # [seq, seq]
scores += causal_mask
P = softmax(scores)
O = P @ V
Y = attention_out(O) + X
Z = mlp(layer_norm(Y)) + Y
write_kv_cache(K, V)

The GPU work here looks a lot like the matrix-heavy world in GPU Compute: fused kernels, memory bandwidth, occupancy, and painful sensitivity to data layout. Prefill is usually compute-heavy. Decode, later, becomes memory-heavy. Good serving systems respect that split.

Why prefill and decode behave differently

Prefill processes many tokens in parallel, so the hardware has enough work to keep tensor cores busy. Decode handles one new token per sequence at a time, which means much smaller matmuls and far more time spent reading old KV state. Same model, very different bottleneck.

Why Long-Context Demos Feel Haunted

A stage demo with a 20k-token prompt feels magical because the audience does not see the silence before the first word. Production users do. Prefill is where wonder goes to pick up a latency bill.

04 - The KV Cache

The KV cache is the reason autoregressive decoding is viable at all. Without it, every new token would require recomputing keys and values for the entire prefix. With it, each decode step computes K and V only for the new token, appends them to cache, and reuses the prefix state. This saves absurd amounts of work. It also creates a giant memory management problem.

What is stored

Every transformer layer writes a key vector and a value vector per token. Multiply that by layers, heads, head dimension, batch size, and current sequence length. The result is not theoretical. It is memory pressure that can evict requests, trigger fragmentation, and quietly cap throughput long before you hit nominal FLOP limits.

The allocator story here rhymes with Hash Tables and Compression more than most people expect. You need compact layout, predictable lookup, and good locality. Serving teams spend enormous time on block tables because random scatter over GPU memory is death by bandwidth.

Production truth

If your model weights fit but your KV cache does not, then your model does not fit. Capacity planning that ignores live sequence growth is fiction.

05 - Decode: One Token at a Time

After prefill, the server enters the decode loop. This is the part users watch. The system takes the current token, runs a forward pass using the cached prefix, samples one next token, appends it, and repeats until a stop condition fires. The misery is that this loop is fundamentally serial for each request. You cannot generate token 57 before token 56 exists.

pythonwhile not done:
    hidden = model.forward_one_step(last_token, kv_cache)
    logits = lm_head(hidden)
    next_token = sample(logits, temperature, top_p)
    emit(next_token)
    kv_cache.append(last_token)
    last_token = next_token
    done = stop_condition(next_token, max_tokens)

Decode is a memory problem

During decode, the new work per step is small. What dominates is reading all those old K and V tensors back out for attention. This is why serving can look underutilized on paper while still being slow. The GPU is waiting on memory traffic, not arithmetic. If you want intuition for why this hurts, read GPU Compute with the phrase "global memory loads" echoing in your head.

Good decode regime

Short or medium contexts, many active sequences, continuous batching, and kernels specialized for small-step decode. The GPU stays fed and tokens stream smoothly.

Bad decode regime

One giant context, irregular batch shape, cache fragmentation, and frequent request cancellation. The scheduler thrashes and per-token latency spikes.

Sampling strategy also matters here. Greedy decode is cheap. Top-k and top-p add modest work. Structured decoding, grammar constraints, and tool-call validation add more logic in the loop. None of it is free, and all of it compounds because decode repeats for every token.

06 - Batching and Scheduling

The scheduler is the actual product. Everyone talks about models; production systems live or die by queue discipline. Your GPU wants large batches. Your user wants immediate response. Those desires are not aligned. The scheduler exists to cheat that conflict just enough that both sides remain tolerably happy.

Static batching is not enough

If you wait to form a perfect batch, first-token latency explodes. If you run every request immediately, throughput collapses. Modern servers therefore use continuous batching: admit new prefills while other requests are already decoding, then merge them into subsequent decode steps as cache pages become available.

Continuous batching timeline 
time ---->

req A:  [prefill........][d][d][d][d][d][d]
req B:                  [prefill..][d][d][d]
req C:                        [prefill....][d][d]

GPU step 1: prefill A
GPU step 2: decode A + prefill B
GPU step 3: decode A,B + prefill C
GPU step 4: decode A,B,C

goal: keep the device busy without making later arrivals wait forever
Policy Helps Hurts
FCFS Simple and fair Long prompts block short interactive calls
Shortest prompt first Improves perceived latency Starves long-context jobs
Separate prefill/decode lanes Protects streaming requests from giant prefills More complex capacity balancing

The queueing problem has a family resemblance to lock contention in Concurrency. At low traffic almost any policy works. Under bursty load, tiny scheduler choices decide whether latency degrades gracefully or turns into a cliff.

Tail latency trap

Mean latency is a vanity metric for inference. A single 30k-token request can make P99 first-token latency grotesque for everyone else if it is allowed to monopolize prefill capacity.

07 - Quantization and Weight Layout

A model that only runs in BF16 on a giant cluster is not a serving story. It is a lab story. Production inference aggressively compresses weights because memory is the first hard ceiling you hit. The trick is to remove precision where the model can tolerate it and preserve it where it cannot.

What gets quantized

Weights are the obvious target: FP16 or BF16 becomes INT8, INT4, or specialized packed formats. Activations may stay in higher precision. KV cache may use its own precision policy. The point is not compression for its own sake. The point is fitting more useful work into the same HBM and PCIe budget.

Format Memory win Typical tradeoff
BF16 / FP16 Baseline Best quality, highest footprint
INT8 About 2x smaller than FP16 Usually safe, modest dequant overhead
INT4 / packed 4-bit About 4x smaller than FP16 More quality sensitivity, kernel complexity rises

This is conceptually adjacent to Compression. In both cases you are exploiting redundancy and tolerable loss to reduce footprint. The difference is that an LLM is a decoder living on those weights, so the cost of decompression lands inside your hottest path.

pseudo# blockwise int4 weight load
packed = load_128bit_block()
scales = load_scales()
zero_points = load_zero_points()
w = dequantize_int4(packed, scales, zero_points)
acc += dot(x_fragment, w)
Serving heuristic

Quantization is only a win if the kernel path stays efficient. Saving bytes but destroying memory coalescing or forcing extra unpack work in the inner loop is how teams accidentally make smaller models slower.

08 - Context Windows and Retrieval Boundaries

"Supports 128k context" is not the same thing as "should receive 128k on every request." Long context inflates prefill cost, KV usage, and scheduler unfairness. A competent serving stack decides what belongs in the prompt, what belongs in retrieval, and what should be summarized or dropped entirely.

Three places memory can live

In prompt
Fastest for the model to reason over semantically, most expensive for the server because every token must be prefixed and cached.
In retrieval
Cheap to store outside the model. Expensive only when selected. Requires indexing, ranking, and robust chunking.
In application state
Facts, permissions, and session metadata often do not belong in the prompt at all. They belong in regular software data structures, usually keyed lookups much closer to Hash Tables than to machine learning.

Chunking retrieved text is another parser problem. Delimiters, sentence boundaries, and document structure all matter, which is why Parsers shows up again. Garbage chunks make garbage prompts. Retrieval can save inference cost only if the selected slices are actually relevant.

Healthy context policy

Keep the system prompt short, summarize stale dialogue, retrieve only a few ranked passages, and cap tool schemas to the ones relevant for this turn.

Unhealthy context policy

Shove the whole chat, the whole manual, and every tool definition into every request because "the model can handle it."

Practical rule

If an input token does not materially improve the next answer, it is stealing latency from your user and capacity from your cluster.

09 - The API Surface and the Wire

Inference feels local only because the transport layer is doing its job. In reality the response is a streamed network conversation. The server must authenticate the request, enforce limits, start sending output quickly, and handle cancellation cleanly when the client disconnects. Streaming tokens over the wire is as much a network protocol problem as a model problem.

Why streaming matters

Without streaming, the user waits for the full completion and experiences the entire decode loop as dead air. With streaming, the system can hide much of that cost. Common implementations use Server-Sent Events or chunked HTTP responses because they are simpler than WebSockets for one-way token emission.

httpevent: token
data: {"id":"cmpl_123","index":0,"text":"The"}

event: token
data: {"id":"cmpl_123","index":1,"text":" scheduler"}

event: done
data: {"finish_reason":"stop"}

The transport issues are straight out of The Network Stack: buffering, flow control, packet loss, head-of-line behavior higher up the stack, and connection teardown. The auth and TLS story leans on Cryptography. Most "model" outages in hosted APIs are really admission control failures, bad timeout policy, or backpressure bugs.

Wire concern Why it matters Common mistake
Cancellation Freed requests should return KV pages immediately Letting abandoned generations keep decoding
Rate limiting Protects GPU time from abuse and runaway clients Limiting only requests, not tokens or context length
Authentication Prevents theft of a very expensive resource Treating inference like a cheap stateless endpoint

10 - Ops, Instrumentation, and Failure Modes

You cannot operate an inference system from averages and vibes. You need queue depth, admission latency, prefill latency, decode latency per token, active sequence count, cache occupancy, eviction rate, cancellation rate, and GPU memory headroom. Otherwise you are debugging anecdotes while users watch a spinner.

Metrics that actually help

Metric What it reveals Bad sign
TTFT p95 / p99 Prefill queueing and long-prompt interference Short prompts suddenly get slow during bursts
Tokens/sec by batch size Decode efficiency and scheduler health Throughput falls as traffic rises
KV occupancy Whether context growth is consuming the device High churn or frequent forced evictions
Cancellation lag Whether dead requests stop burning GPU Large gap between client drop and GPU release

Typical production failures

OOM during bursty long-context traffic. Fragmented KV pages causing capacity collapse. Slow tokenizer workers starving GPUs upstream. Admission queues that grow without a hard cutoff. Tool schemas so large that prefill dominates everything. A streaming layer that buffers in memory and defeats the whole point of token streaming.

Classic outage shape

Traffic spikes, queue depth rises, long prompts enter the system, first-token latency stretches, impatient clients retry, retries amplify load, and suddenly the incident channel is full of people debating model quality when the real problem is queue collapse.

None of this is exotic. It is standard capacity engineering. The only unusual part is that the hot resource is GPU memory and the unit of work is a growing token stream rather than a fixed request body.

11 - The Full Serving Stack

Put the pieces together and the architecture looks much less magical. There is an API tier, a prompt construction path, a tokenizer, a scheduler, one or more model workers, a KV memory manager, and a streaming transport back to the caller. Every layer exists because the previous abstraction leaked.

What we built

An inference service that accepts structured chat requests, renders a model-specific prompt, tokenizes it, optionally retrieves supporting context, runs prefill on GPU, retains KV state across decode steps, continuously batches active sequences, streams sampled tokens over HTTP, and exposes the metrics needed to survive real traffic.

What we deliberately did not cover

Training, fine-tuning pipelines, distributed optimizer states, evaluation methodology, multimodal serving, and agent orchestration. Those are neighboring systems, not this one. This dive is about the narrow but brutally consequential question: when a user asks a model for one answer, what exactly does the serving stack do with the machine?

Final mental model

Inference is not "the model thinking." It is a scheduler repeatedly buying one more token by spending memory bandwidth, cache space, and queue fairness. Once you see it that way, most serving decisions become ordinary systems tradeoffs instead of folklore.
