CRYPTOGRAPHY

Building an encrypted messenger, one primitive at a time_

01 — The Padlock Myth

"Just encrypt it" might be the most dangerous phrase in all of software engineering. It sounds responsible. It sounds like you have a plan. In practice it is a hand-wave over an ocean of decisions, any one of which can sink your ship. What algorithm? What mode? What key size? How do you generate the key? How do you exchange it? How do you authenticate the other end? How do you rotate keys after compromise? Every padlock icon in your browser is the visible tip of an iceberg of interlocking protocols, each one designed to cover the failure modes of the one below it.

We are going to build an encrypted messenger in this deep dive — not a toy one, a real one, with the same primitives Signal uses. But we are going to do it one layer at a time, and at every layer we are going to ask: what goes wrong if we skip this?

The Threat Model

Before you write a single line of crypto code, you need to know what you are defending against. Our messenger faces four distinct adversaries:

Eve sits on the Wi-Fi at a coffee shop and captures every packet. Mallory runs a rogue access point and actively rewrites traffic in flight. The server is honest-but-curious: it faithfully delivers messages but might be compelled to hand over everything it stores. And metadata — who talked to whom, when, how often, how much — is the attack surface most messengers ignore entirely.

Kerckhoffs' Principle

In 1883, a Dutch linguist named Auguste Kerckhoffs published six design principles for military ciphers. Only one survived intact into the modern era: the system must be secure even if everything except the key is public knowledge. The algorithm, the protocol, the source code — assume the adversary has all of it. The only secret is the key.

This is why "security through obscurity" is a punchline in cryptography. If your system breaks the moment someone reverse-engineers your binary, it was never secure. AES has been public since 1998. The entire world has been trying to break it for nearly three decades. That is exactly why we trust it.

Don't Roll Your Own

If the whole point is to use well-vetted primitives, why study the internals at all? Because choosing is harder than it looks. AES-CBC or AES-GCM? HMAC-then-encrypt or encrypt-then-MAC? RSA-2048 or X25519? SHA-256 or BLAKE3? Every choice has tradeoffs in performance, security margin, and failure modes. The goal of this deep dive is not to make you implement AES from scratch — it is to make you dangerous enough to choose correctly and recognize when something is wrong.

The Messenger Threat Model

Confidentiality — nobody reads it except the intended recipient. Integrity — nobody alters it in transit. Authenticity — it really came from who it claims. Forward secrecy — compromise today does not decrypt yesterday's messages. These four properties are the contract our messenger must uphold.

02 — Hashing

Every message in our messenger needs a fingerprint — a short, fixed-size digest that uniquely identifies its contents. Change a single character in a gigabyte-long message and the fingerprint changes completely. This is what hash functions give us: a deterministic, one-way compression of arbitrary data into a fixed number of bits.

SHA-256: The Workhorse

SHA-256 belongs to the SHA-2 family, designed by the NSA and published by NIST in 2001. It uses the Merkle-Damgård construction: the input is padded to a multiple of 512 bits, then processed one block at a time. Each block goes through 64 rounds of a compression function that mixes the input with an internal state of eight 32-bit words. The final state is the 256-bit hash.

Each round applies a combination of bitwise rotations, XOR, AND, and modular addition — operations that are cheap on any CPU but collectively create an avalanche of diffusion. After 64 rounds, every bit of the output depends on every bit of the input.

Three Security Properties

Property Meaning Bound for SHA-256
Collision resistance Cannot find any two inputs that produce the same hash ~2128 work (birthday bound)
Preimage resistance Given a hash, cannot find any input that produces it ~2256 work
Second preimage resistance Given input m1, cannot find m2 ≠ m1 with the same hash ~2256 work

The birthday bound deserves explanation. If a hash has n possible outputs, you need roughly √n random inputs before a collision becomes likely — the same math behind the birthday paradox. For 256-bit output, √2256 = 2128. That is still astronomically large, but it is why we don't use 128-bit hashes for collision resistance.

The Avalanche Effect

A good hash function exhibits strict avalanche: flip one input bit and each output bit has a 50% chance of flipping. This is not a nice-to-have — it is the property that makes the hash unpredictable. If changing "a" to "b" only changed a few output bits, an attacker could work backwards from the hash to the input one bit at a time.

03 — HMAC and Integrity

Was this message tampered with? A plain hash cannot answer that question. If Alice sends a message and its SHA-256 hash, an attacker can intercept both, modify the message, recompute the hash, and forward the pair. Bob would verify the hash, find it matches, and trust a message Alice never sent. The hash proves the message wasn't corrupted in transit. It says nothing about whether it was deliberately altered.

To prove integrity, Alice and Bob need a shared secret. If Alice computes the hash using a key that only she and Bob know, the attacker cannot forge a valid hash for a modified message without the key. This is a Message Authentication Code — a MAC.

Length Extension Attacks

The naive approach is to compute SHA-256(key || message). This is broken. Because SHA-256 uses Merkle-Damgård construction, the internal state after processing key || message is exactly the hash output. An attacker who knows the hash (but not the key) can resume hashing from that state and append arbitrary data. The result is a valid hash for key || message || padding || attacker_data — without ever knowing the key.

HMAC Construction

HMAC fixes this with two nested hashes and XOR-padded keys:

HMAC(K, m) = H((K ⊕ opad) || H((K ⊕ ipad) || m))

Where ipad is the byte 0x36 repeated to the block size, and opad is 0x5c repeated. The inner hash produces an intermediate digest. The outer hash processes that digest with a differently-padded key. This double wrapping defeats length extension because the attacker never sees the intermediate state of the outer hash.

HMAC-SHA256 construction 

                    Key K

                  /       \

             |               |
             v               |
      +-------------+        |

Timing Attacks on MAC Verification

Suppose Bob verifies the MAC by comparing byte-by-byte with early exit — the standard == operator in most languages. If the first byte is wrong, the comparison returns false immediately. If the first byte is right but the second is wrong, it takes slightly longer. An attacker can exploit this: try all 256 values for byte 0, measure response times, pick the one that took longest (the correct one), then move to byte 1. After 256 × 32 = 8,192 attempts, the attacker has forged a valid MAC without knowing the key.

The fix is constant-time comparison: iterate over every byte, XOR corresponding bytes together, OR all the XOR results into an accumulator, and check the accumulator at the end. The comparison takes the same amount of time regardless of how many bytes match.

Timing Side Channel

A naive MAC check that returns false on the first wrong byte tells the attacker exactly which byte to fix next. With 32 bytes and 256 possible values per byte, that is 8,192 guesses instead of 2256. Always use constant-time comparison: crypto.subtle.timingSafeEqual in Node, hmac.compare_digest in Python, crypto/subtle.ConstantTimeCompare in Go.

04 — Symmetric Encryption

Now we encrypt the message body. This is the core of what makes our messages unreadable to eavesdroppers: a symmetric cipher that scrambles plaintext into ciphertext using a shared key. The same key encrypts and decrypts — hence "symmetric." The hard part is not the encryption itself but the mode of operation that wraps it.

AES: The Block Cipher

AES (Advanced Encryption Standard) operates on 128-bit blocks. It takes a 128, 192, or 256-bit key and applies a series of transformations in rounds — 10 rounds for AES-128, 14 for AES-256. Each round performs four operations on a 4×4 matrix of bytes called the state:

AES single round 

  State (4×4 bytes)
       |
       v
  SubBytes      S-box substitution: each byte mapped through
       |        a fixed lookup table (Rijndael S-box)
       v
  ShiftRows    Rotate each row by 0,1,2,3 positions
       |
       v
  MixColumns   Multiply each column by a fixed matrix
       |        in GF(2^8) — Galois field arithmetic
       v
  AddRoundKey  XOR the state with the round key
       |
       v
  Next round...

SubBytes provides confusion — a nonlinear substitution that destroys algebraic structure. ShiftRows and MixColumns provide diffusion — spreading each input byte's influence across the entire state. AddRoundKey mixes in the key material. After 10–14 rounds, every output bit depends on every input bit and every key bit.

The ECB Penguin

AES encrypts one 128-bit block at a time. The simplest way to encrypt a long message is ECB (Electronic Codebook): split the message into blocks, encrypt each one independently. This is catastrophically broken for anything with structure. Encrypt the pixels of Tux the Linux penguin with ECB mode and the silhouette is still perfectly visible — because identical plaintext blocks produce identical ciphertext blocks. Patterns in the input are preserved in the output.

Modes of Operation

ECB vs CBC vs CTR 

  ECB — Electronic Codebook (BROKEN for structured data)

  P1 ──> [AES] ──> C1     P2 ──> [AES] ──> C2     P3 ──> [AES] ──> C3
          key               key               key
  Same plaintext blocks → same ciphertext blocks

  ──────────────────────────────────────────────────────────

  CBC — Cipher Block Chaining

  IV ──┐
       v
  P1 ⊕ ──> [AES] ──> C1 ──┐
              key           v
                       P2 ⊕ ──> [AES] ──> C2 ──┐
                                    key           v
                                             P3 ⊕ ──> [AES] ──> C3
                                                          key
  Each block depends on all previous blocks. Sequential only.

  ──────────────────────────────────────────────────────────

  CTR — Counter Mode

  Nonce|0 ──> [AES] ──> ⊕ P1 ──> C1
                key

  Nonce|1 ──> [AES] ──> ⊕ P2 ──> C2
                key

  Nonce|2 ──> [AES] ──> ⊕ P3 ──> C3
                key
  Parallelizable. No padding. Block cipher → stream cipher.

CBC (Cipher Block Chaining) fixes ECB by XORing each plaintext block with the previous ciphertext block before encryption. This hides patterns but makes encryption sequential and requires padding — which opens the door to padding oracle attacks (Vaudenay, 2002).

CTR (Counter Mode) takes a different approach entirely: encrypt a nonce concatenated with a counter, then XOR the result with the plaintext. This turns the block cipher into a stream cipher. It is parallelizable, needs no padding, and is the foundation of GCM.

GCM: Encrypt and Authenticate

AES-GCM (Galois/Counter Mode) combines CTR-mode encryption with a GHASH authentication tag computed over the ciphertext and any associated data. One pass gives you both confidentiality and integrity. This is what our messenger uses.

Why not just use AES-CBC and add HMAC separately? You can — it is called encrypt-then-MAC — but you must get the composition right. Encrypt-and-MAC (SSH's original approach) and MAC-then-encrypt (TLS before 1.3) are both vulnerable to subtle attacks. GCM handles both in a single, well-analyzed construction with no room for composition errors.

C / OpenSSL
#include <openssl/evp.h>
#include <openssl/rand.h>
#include <stdio.h>
#include <string.h>

int main(void) {
    unsigned char key[32], iv[12], tag[16];
    RAND_bytes(key, 32);
    RAND_bytes(iv, 12);

    const char *msg = "attack at dawn";
    int len = strlen(msg);
    unsigned char ct[128], pt[128];
    int outlen;

    /* Encrypt */
    EVP_CIPHER_CTX *ctx = EVP_CIPHER_CTX_new();
    EVP_EncryptInit_ex(ctx, EVP_aes_256_gcm(), NULL, key, iv);
    EVP_EncryptUpdate(ctx, ct, &outlen, (unsigned char *)msg, len);
    EVP_EncryptFinal_ex(ctx, ct + outlen, &outlen);
    EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_GCM_GET_TAG, 16, tag);
    EVP_CIPHER_CTX_free(ctx);

    printf("Encrypted %d bytes\n", len);

    /* Decrypt */
    ctx = EVP_CIPHER_CTX_new();
    EVP_DecryptInit_ex(ctx, EVP_aes_256_gcm(), NULL, key, iv);
    EVP_DecryptUpdate(ctx, pt, &outlen, ct, len);
    EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_GCM_SET_TAG, 16, tag);
    int ok = EVP_DecryptFinal_ex(ctx, pt + outlen, &outlen);
    EVP_CIPHER_CTX_free(ctx);

    pt[len] = 0;
    printf("Decrypted: %s (auth: %s)\n", pt, ok ? "OK" : "FAIL");
    return 0;
}

Why GCM

Authenticated encryption means you cannot flip bits in the ciphertext without detection. AES-CBC without a MAC allows bit-flipping attacks — an attacker who knows the plaintext structure can XOR specific bits in a ciphertext block to alter the corresponding plaintext bits in the next block. GCM catches any tampering via its GHASH authentication tag.

05 — The Key Problem

How do two strangers share a secret? Alice and Bob have never met. Eve watches every byte on the wire. If Alice picks a key and sends it to Bob, Eve intercepts it and the game is over. If Bob picks one and sends it to Alice, same problem. It seems impossible — any key transmitted in the clear is compromised the instant it leaves the sender. And yet in 1976, Whitfield Diffie and Martin Hellman published a protocol that solves this problem completely.

Diffie-Hellman with Concrete Numbers

Let us work through DH with numbers small enough to verify by hand. Alice and Bob publicly agree on a prime p = 23 and a generator g = 5. These values are not secret.

Diffie-Hellman key exchange (p=23, g=5) 

  Alice                                          Bob
  picks secret  a = 6                              picks secret  b = 15
  computes      A = 56 mod 23                     computes      B = 515 mod 23
                A = 15625 mod 23                                B = 30517578125 mod 23
                A = 8                                         B = 19

                    ────── A = 8 ──────>
                    <───── B = 19 ─────

  shared = Ba mod 23                             shared = Ab mod 23
         = 196 mod 23                                    = 815 mod 23
         = 47045881 mod 23                                = 35184372088832 mod 23
         = 2                                            = 2

                        Same secret: 2

Eve saw A=8 and B=19 fly across the wire. She knows p=23 and g=5. To compute the shared secret, she would need to find a such that 5a mod 23 = 8, or b such that 5b mod 23 = 19. With p=23 she could just try all values, but when p is a 2048-bit prime, there is no known algorithm that solves this efficiently.

Why Discrete Log Is Hard

Computing ga mod p is fast — repeated squaring does it in O(log a) multiplications. But given the result ga mod p and the values g and p, finding a is the discrete logarithm problem. For carefully chosen primes, the best known algorithms (General Number Field Sieve) require sub-exponential time — fast enough to make 1024-bit primes nervous, but 2048-bit primes remain out of reach.

This is the core asymmetry that all public-key cryptography exploits: a mathematical operation that is cheap in one direction and prohibitively expensive in reverse.

The Man-in-the-Middle Problem

DH as described above has a fatal flaw: it provides no authentication. Mallory can intercept the exchange and perform DH independently with both sides.

MITM attack on unauthenticated DH 

  Alice              Mallory                Bob
     |                  |                     |
     |── A = ga ──>    |                     |
     |                  |──── Mb = gm ──────>  |
     |                  |                     |
     |                  |  <──── B = gb ──── |
     |  <── Ma = gm' ──|                     |
     |                  |                     |

  Alice thinks          Mallory has           Bob thinks
  shared = Maa         two secrets:          shared = Mbb
                        with Alice: Am'
                        with Bob:   Bm

  Mallory decrypts Alice's msgs, re-encrypts for Bob, and vice versa.
  Neither Alice nor Bob detects anything wrong.

This is why DH alone is not enough. You need to authenticate the public values — via digital signatures, certificates, or a pre-shared trust relationship. In our messenger, this will come from long-term identity keys (chapter 6) and the Signal protocol's triple-DH handshake (chapter 8).

ECDH and X25519

Classical DH uses modular exponentiation with massive primes. Elliptic Curve Diffie-Hellman (ECDH) does the same thing on an elliptic curve, where "exponentiation" becomes scalar multiplication of a curve point. The discrete log problem on well-chosen curves is harder per bit than in modular arithmetic, so 256-bit curve keys offer comparable security to 3072-bit DH primes.

X25519 is the specific ECDH instantiation our messenger will use. Designed by Daniel J. Bernstein, it uses Curve25519 — a Montgomery curve over a 255-bit prime field. The implementation is inherently constant-time (no secret-dependent branches), immune to invalid-curve attacks, and produces 32-byte public keys and 32-byte shared secrets. It is the default key exchange in Signal, WireGuard, TLS 1.3, and SSH.

DH Key Exchange Demo

C
#include <stdio.h>
#include <stdlib.h>

/* Modular exponentiation: base^exp mod m */
long long mod_pow(long long base, long long exp, long long m) {
    long long result = 1;
    base %= m;
    while (exp > 0) {
        if (exp & 1) result = (result * base) % m;
        exp >>= 1;
        base = (base * base) % m;
    }
    return result;
}

int main(void) {
    long long p = 23, g = 5;
    long long a = 6, b = 15;  /* secret exponents */

    long long A = mod_pow(g, a, p);  /* Alice's public value */
    long long B = mod_pow(g, b, p);  /* Bob's public value   */

    long long s_alice = mod_pow(B, a, p);
    long long s_bob   = mod_pow(A, b, p);

    printf("A=%lld  B=%lld\n", A, B);
    printf("Alice's shared: %lld\n", s_alice);
    printf("Bob's shared:   %lld\n", s_bob);
    printf("Match: %s\n", s_alice == s_bob ? "YES" : "NO");
    return 0;
}

06 — Asymmetric Crypto

Symmetric encryption keeps messages secret, but it cannot answer a harder question: who sent this? If Alice and Bob share a key, either of them could have produced any given ciphertext. Our messenger needs a mechanism where Alice can prove she authored a message in a way that anyone can verify but nobody can forge. This is the domain of asymmetric cryptography — public keys and digital signatures.

RSA: The Original

RSA (1977) works on a simple number-theoretic insight. Pick two large primes p and q. Compute n = p × q and φ = (p−1)(q−1). Choose a public exponent e (almost always 65537) and compute the private exponent d = e−1 mod φ. To encrypt: c = me mod n. To decrypt: m = cd mod n. The security rests on the hardness of factoring n back into p and q.

RSA works, but it is slow. A 2048-bit RSA signature requires modular exponentiation on 2048-bit integers — roughly 1ms on modern hardware. Ed25519 achieves the same security level with 256-bit elliptic curve operations in about 0.05ms — a 20× speedup with smaller keys.

RSA key generation and encryption 

  1. Key Generation
  p = large prime          q = large prime
  n = p × q                φ = (p-1)(q-1)
  e = 65537 (public)       d = e-1 mod φ (private)

  2. Encrypt (public key = {e, n})
  ciphertext = messagee mod n

  3. Decrypt (private key = {d, n})
  message = ciphertextd mod n

  Security: factoring n into p × q is computationally infeasible
  RSA-2048 ≈ 112-bit security    RSA-4096 ≈ 128-bit security

Ed25519: The Modern Standard

Ed25519 is an Edwards-curve signature scheme over the finite field GF(2255−19). The key pair is a 32-byte private seed and a 32-byte public key. Signatures are 64 bytes. What makes Ed25519 special is that the signing nonce is deterministic — it is derived from the private key and the message via a hash. There is no random number involved in signing.

This eliminates an entire class of catastrophic bugs. When ECDSA relies on a random nonce k, reusing that nonce even once leaks the private key.

Ed25519 sign and verify 

  Signing (deterministic nonce — no randomness needed)

  seed (32 bytes) → SHA-512 → a (scalar) + prefix
  nonce r = SHA-512(prefix || message)          ← deterministic!
  R = r × B                                      (base point multiplication)
  S = r + SHA-512(R || pubkey || message) × a  (mod group order)
  signature = (R, S)  — 64 bytes

  Verification
  Check: S × B == R + SHA-512(R || pubkey || message) × pubkey

The Sony PS3 Disaster

In 2010, the fail0verflow team demonstrated that Sony used a fixed random number k for every ECDSA signature on PS3 game binaries. In ECDSA, the signature component s = k−1(z + r·d) mod n. With the same k used for two different messages, you get two equations with two unknowns (k and d) — trivially solvable. Sony's private signing key was extracted in minutes. Ed25519 makes this bug structurally impossible because there is no random nonce to reuse.

Authenticating Key Exchange

Recall from Chapter 5 that unauthenticated Diffie-Hellman is vulnerable to man-in-the-middle attacks. The fix: sign your ephemeral DH public key with your long-term identity key. When Alice sends her ephemeral X25519 key to Bob, she also sends an Ed25519 signature over it. Mallory cannot forge this signature because she does not have Alice's private identity key.

07 — The TLS Handshake

Every time you open our messenger, before a single message byte is exchanged, a precise ceremony occurs: the TLS handshake. It negotiates a cipher suite, authenticates the server, and derives encryption keys — all in a single round trip. TLS 1.3 stripped the protocol down to its essential bones, removing decades of accumulated complexity and vulnerability.

TLS 1.3: One Round Trip

TLS 1.3 handshake (1-RTT) 

  Client                                        Server
    │                                              │
    │—— ClientHello ——————————————→  │
    │   cipher suites + key_share (X25519)       │
    │                                              │
    │  ←—————————— ServerHello ——│
    │   chosen suite + key_share                  │
    │                                              │
    │   ══ encrypted with handshake keys ══     │
    │                                              │
    │  ←—— EncryptedExtensions ——————│
    │  ←—— Certificate (X.509 chain) ——│

    │                                              │
    │←———— Application Data —————→│

The key innovation in TLS 1.3: the client sends its X25519 public key in the first message. The server responds with its own key share, and from that point forward the connection is encrypted. In TLS 1.2, two round trips were required before encryption could begin.

What TLS 1.3 Removed

TLS 1.3 deleted RSA key exchange entirely. In TLS 1.2, the client could encrypt the pre-master secret under the server's RSA public key. This provided no forward secrecy — compromising the server's private key would retroactively decrypt every recorded session. TLS 1.3 mandates ephemeral Diffie-Hellman (X25519 or P-256) for every connection.

Also removed: CBC-mode ciphers, RC4, SHA-1, arbitrary cipher suite negotiation, renegotiation, compression (CRIME attack). Every cipher suite in TLS 1.3 uses AEAD — either AES-128-GCM, AES-256-GCM, or ChaCha20-Poly1305.

The Certificate Chain

The server proves its identity with an X.509 certificate chain: Root CA → Intermediate CA → Leaf certificate. Root CAs are trusted by your operating system (roughly 100-150 root certificates). The CertificateVerify message contains a signature over the entire handshake transcript, proving the server actually possesses the private key corresponding to the certificate.

Let's Encrypt (launched 2015) made free, automated certificates ubiquitous. Before it, TLS adoption was limited by certificate cost and complexity. Today, over 80% of web traffic is encrypted.

0-RTT: Speed at a Cost

TLS 1.3 supports 0-RTT early data: if the client has a pre-shared key (PSK) from a previous session, it can send application data in the very first flight — zero round trips of latency. The tradeoff is real: 0-RTT data is vulnerable to replay attacks. An attacker who records the first flight can replay it, causing the server to process the request again. Servers must ensure that 0-RTT requests are idempotent (safe to replay).

Cross-reference: see the Network Deep Dive for TCP handshake details and how TLS interleaves with it.

Certificate Transparency

Certificate Transparency (CT) is a system of public, append-only logs that record every TLS certificate issued by participating CAs. If a rogue or compromised CA issues a certificate for your domain, the CT logs will catch it. Since 2018, Google Chrome requires CT compliance for all newly-issued certificates. This turned certificate mis-issuance from an invisible attack into a publicly auditable event.

08 — The Signal Protocol

This is the crown jewel — the actual end-to-end encryption protocol our messenger uses. TLS protects the connection from your device to the server, but Signal Protocol ensures that even the server cannot read your messages. It solves two problems simultaneously: how to establish a shared secret with someone who is currently offline, and how to ensure that compromising a key today cannot decrypt messages from the past or the future.

The Async Problem

Diffie-Hellman requires both parties to be online at the same time to exchange ephemeral keys. But Bob might be asleep when Alice wants to send her first message. The solution is X3DH — Extended Triple Diffie-Hellman.

X3DH: Key Exchange with Offline Parties

Bob pre-publishes a key bundle to the server:

  • Identity key IKB — long-term Ed25519/X25519 key pair
  • Signed prekey SPKB — medium-term key, rotated weekly, signed by IKB
  • One-time prekeys OPKB — a batch of single-use keys (deleted after use)

When Alice wants to message Bob, she fetches his bundle and computes:

X3DH — four DH computations 

  Alice                                          Bob (offline)
  IK_A (identity)                                IK_B (identity)
  EK_A (ephemeral, fresh)                        SPK_B (signed prekey)
                                                  OPK_B (one-time prekey)

  DH1 = DH(IK_A,  SPK_B)    ← authentication (Alice's identity)
  DH2 = DH(EK_A,  IK_B)     ← forward secrecy
  DH3 = DH(EK_A,  SPK_B)    ← ties ephemeral to signed prekey
  DH4 = DH(EK_A,  OPK_B)    ← one-time prekey (optional, extra security)

  master_secret = KDF( DH1 || DH2 || DH3 || DH4 )

Alice sends her identity key, ephemeral key, and the identifier of which one-time prekey she used. When Bob comes online, he performs the same four DH computations (with the roles reversed) and derives the same master secret. The server never learns the secret — it only relayed public keys.

The Double Ratchet

X3DH establishes the initial shared secret. The Double Ratchet algorithm takes over for all subsequent messages. It interleaves two ratchets:

Symmetric ratchet (KDF chain): Each message derives its encryption key from a chain key. After derivation, the chain key is ratcheted forward and the message key is deleted. This means past message keys are unrecoverable from current state.

DH ratchet: Periodically, when the conversation direction switches (Alice sends, then Bob replies), a new DH key exchange occurs. This produces a fresh root key and resets the chain. An attacker who somehow obtained the current state is locked out after the next DH ratchet step — they do not know the new DH private key.

Double Ratchet chain progression 

  Root Chain         Sending Chain        Messages

  root_key_0
    │
    └— DH ratchet —→ chain_key_0
    │                      │—→ msg_key_0 → encrypt msg #0  (deleted)
    │                      │—→ msg_key_1 → encrypt msg #1  (deleted)
    │                      │—→ msg_key_2 → encrypt msg #2  (deleted)
  root_key_1
    │
    └— DH ratchet —→ chain_key_1       (new DH key pair)
    │                      │—→ msg_key_3 → encrypt msg #3  (deleted)
    │                      │—→ msg_key_4 → encrypt msg #4  (deleted)
  root_key_2
    │  ...
Forward Secrecy

Compromise today cannot decrypt yesterday. Each message key is used exactly once and then deleted. Even if an attacker steals your phone right now, messages from last week are mathematically unrecoverable — the keys that encrypted them no longer exist anywhere in the universe.

Post-Compromise Security

If your key is stolen, the very next time both parties exchange a DH ratchet message, the attacker is locked out. The new DH key pair generates a shared secret the attacker cannot compute. No other widely deployed protocol has this property. It means a compromise is self-healing — the damage is confined to the window between compromise and the next ratchet step.

09 — Password Storage

Our messenger needs a login system. The cryptographic question is deceptively simple: how do you store passwords so that a database breach does not compromise every account? The answer involves deliberately making computation expensive — the exact opposite of what we normally want from software.

Why Fast Hashes Are Catastrophic

An NVIDIA RTX 4090 computes approximately 160 billion MD5 hashes per second and 10 billion SHA-256 hashes per second. A six-character alphanumeric password has roughly 2 billion possibilities. At 160 billion per second, MD5 cracks it in 12 milliseconds. Even an 8-character password falls in under a minute.

Rainbow Tables

Rainbow tables are precomputed hash-to-password lookup tables. A 1TB rainbow table covers virtually all passwords under 8 characters for unsalted MD5. The attack is a single disk read per password. The defense is salting: prepend a random 16–32 byte value to the password before hashing. Each user gets a unique salt, stored alongside the hash. Rainbow tables become useless because they would need a separate precomputed table per salt.

The Three Contenders

bcrypt (1999): Based on Blowfish, with a tunable cost factor. Cost 12 means 212 = 4096 iterations of the key setup — roughly 250ms per hash. The 72-byte password limit is a known quirk. Salt is embedded in the output string ($2b$12$...). Gold standard for two decades.

scrypt (2009): Memory-hard — requires N × r × 128 bytes of RAM. With N=214, r=8, p=1, that is 16MB per hash computation. ASICs cannot be cost-effective because they need expensive memory, not just transistors.

Argon2id (2015): Winner of the Password Hashing Competition. Combines Argon2d (data-dependent memory access — better GPU resistance) with Argon2i (data-independent access — better side-channel resistance). Recommended parameters: memory=64MB, iterations=3, parallelism=4.

10 — Randomness

Every key in our messenger, every nonce, every salt, every ephemeral DH parameter — all of it starts as random bytes. Cryptographic security is only as strong as the randomness that generated the keys. Get this wrong and everything built on top collapses, no matter how strong the algorithms are.

Math.random() Is Not Random

Math.random() in JavaScript (and equivalents in other languages like C's rand()) is a pseudorandom number generator (PRNG). It is deterministic: given the same seed, it produces the same sequence every time. If an attacker can predict or reconstruct the seed — often derived from the system clock at startup — they can predict every “random” number your program will ever produce. Never use it for cryptography.

CSPRNG: The Right Tool

A Cryptographically Secure PRNG (CSPRNG) generates bytes that are computationally indistinguishable from true randomness. Even if an attacker observes a million consecutive output bytes, they cannot predict the next byte with probability better than 1/256. Every operating system provides one:

  • Linux: getrandom(2) syscall, /dev/urandom
  • macOS: arc4random_buf(), /dev/urandom
  • Windows: BCryptGenRandom()

The /dev/urandom Myth

For years, a persistent myth claimed that /dev/random was “more secure” than /dev/urandom because it blocks when the kernel entropy pool is “low.” The reality: /dev/urandom is fine for all cryptographic purposes once the system has booted and gathered initial entropy. The Linux kernel's CSPRNG (ChaCha20-based since 2016) does not degrade when the “entropy estimate” drops. Even the Linux man page was corrected. Linus Torvalds called /dev/random’s blocking behavior “essentially useless.”

Modern CPUs include hardware random number generators. Intel's RDRAND instruction and AMD's equivalent feed entropy into the kernel pool. The Dual EC Backdoor

In 2006, NIST standardized Dual_EC_DRBG as a random number generator. Cryptographers immediately noticed something suspicious: the standard included two elliptic curve points, P and Q, with no explanation of how Q was chosen. If someone knew the discrete logarithm relationship between P and Q, they could predict the generator's output from a small sample.

In 2013, Snowden documents confirmed that the NSA had paid RSA Security $10 million to make Dual_EC_DRBG the default in their BSAFE library. The algorithm was a backdoor. NIST withdrew the standard in 2014. This remains the most significant known case of a deliberately weakened cryptographic standard.

11 — Side Channels

The bugs crypto can’t fix. Every algorithm we have studied so far is mathematically sound — AES has no practical algebraic weakness, Ed25519 relies on a hard discrete-log problem, Argon2id saturates memory bandwidth. None of that matters when the physics of the machine leaks the secret through a side door. A side-channel attack does not break the math. It watches the computer doing the math and extracts the key from timing, power draw, cache behavior, or electromagnetic radiation.

Timing Attacks

The classic example: verifying a MAC byte-by-byte with an early exit.

Timing attack on MAC comparison 

  Naive comparison: if (mac[i] != expected[i]) return false;

  Byte 0 wrong  →  return in  1μs
  Byte 0 right, byte 1 wrong  →  return in  2μs
  Byte 0-1 right, byte 2 wrong  →  return in  3μs
  ...
  All 31 right, byte 31 wrong  →  return in  32μs

  Attacker strategy:
  For each byte position (0..31):
    Try all 256 values, pick the one with longest response time
    → that value is correct for this position

  Total guesses: 256 × 32 = 8,192
  Instead of:    2256 ≈ 1077

A 32-byte MAC should require 2256 brute-force attempts. With a timing side channel, the attacker needs only 8,192 carefully measured requests. The fix is constant-time comparison: XOR every byte pair, OR the results together, and check the final accumulator. Every input takes the same number of cycles regardless of where the first mismatch occurs.

Cache Timing on AES

In 2005, Daniel Bernstein demonstrated that AES implementations using lookup tables (T-tables) leak key bits through CPU cache access patterns. AES computes T[key_byte ⊕ plaintext_byte] — if that table index is already in cache, the lookup is fast (~1ns); if it must be fetched from memory, it is slow (~100ns). By measuring encryption time across many plaintexts, an attacker can determine which table entries were accessed and recover key_byte ⊕ plaintext_byte. Since the attacker controls the plaintext, they recover the key byte directly.

Modern processors include AES-NI — dedicated hardware instructions (aesenc, aesenclast) that perform AES rounds without any table lookups. No tables means no cache-timing signal. This is why every modern crypto library uses AES-NI when available.

Power Analysis

Simple Power Analysis (SPA) targets RSA’s square-and-multiply algorithm. During modular exponentiation, the processor squares for every bit of the private key and additionally multiplies for every ‘1’ bit. Squaring and multiplying draw different amounts of current. An oscilloscope on the power rail reads the private key bit by bit from the power trace.

Differential Power Analysis (DPA) is more sophisticated: it collects thousands of power traces, aligns them statistically, and extracts key bits that individual traces would not reveal. Smart cards and embedded HSMs must use constant-power implementations or random blinding to defend against DPA.

Spectre

Spectre (variant 1) speculative execution flow 

  if (x < array1_size) {          // branch prediction says: TRUE
      y = array2[array1[x] * 256]; // speculatively executed
  }

  CPU Pipeline:
  ┌────────────────┐     ┌────────────────────┐     ┌────────────────┐
  │ Branch predict │ →  │ Speculative read  │ →  │ Cache polluted │
  │ "x is in bounds"│     │ array1[x] fetched   │     │ array2 line now │
  ┘────────────────└     │ even if x is OOB    │     │ in L1 cache     │
                          ┘────────────────────└     ┘────────────────└
                                    ↓                           ↓
                          ┌────────────────────┐     ┌────────────────┐
                          │ Branch rolls back  │     │ Cache persists! │
                          │ architectural state │     │ attacker probes  │
                          │ restored            │     │ timing to read   │
                          ┘────────────────────└     │ the secret byte  │
                                                          ┘────────────────└

Spectre (CVE-2017-5753) exploits CPU speculative execution. The processor guesses a branch outcome, executes instructions speculatively, and rolls back if the guess was wrong. But the cache side-effects of the speculative execution are not rolled back. An attacker arranges for a bounds check to be mispredicted, causing a speculative out-of-bounds read. The secret data touches the cache, and a subsequent timing measurement extracts it. Every major CPU vendor — Intel, AMD, ARM — was affected.

Rowhammer

Repeatedly accessing one DRAM row causes electrical disturbance in adjacent rows, flipping bits in memory the attacker does not own. This has been used to escalate privileges, escape VMs, and forge RSA keys. It is not strictly a crypto attack, but it demonstrates that hardware itself is an attack surface — the DRAM physics violate the isolation guarantees that all software security depends on.

Constant Time is Hard

Compilers will optimize away your constant-time code. A comparison loop with no early exit? The compiler might add one. A conditional move instead of a branch? The compiler might turn it back into a branch for “efficiency.” You need compiler barriers (volatile reads), platform-specific intrinsics, or hand-written assembly to guarantee constant-time execution. Even then, the CPU’s microarchitecture might introduce variable timing through port contention, power throttling, or speculative execution. This is why crypto libraries employ specialists who verify constant-time behavior with tools like ctgrind and dudect.

12 — Real-World Breaks

When theory meets deployment. Four bugs that burned the internet — and none of them broke the math. SHA-256 is fine. AES is fine. RSA is fine. Every single one of these vulnerabilities was an implementation error: a missing bounds check, a duplicate line, a bad random number generator, a padding design flaw.

Heartbleed (CVE-2014-0160)

Disclosed April 7, 2014. OpenSSL’s TLS Heartbeat extension is a keep-alive mechanism: the client sends a payload and the server echoes it back. The protocol message contains a length field and the actual payload. The bug: OpenSSL used the claimed length (attacker-controlled) to decide how many bytes to copy back, without checking whether the actual payload was that long.

Heartbleed — buffer over-read 

  Client sends:
  ┌─────────────────────────────────────┐
  │ type: heartbeat_request       │
  │ payload_length: 65535 (claimed) │
  │ payload: "hi" (actual: 2 bytes) │
  ┘─────────────────────────────────────└

  Server responds with:
  memcpy(bp, pl, payload);  // payload = 65535, not 2!

  ┌──────────┐┌──────────────────────────────────┐
  │ "hi"      ││ 64KB of adjacent server memory  │
  │ (2 bytes) ││ private keys, session cookies,   │
  │           ││ passwords, other users' data ... │
  ┘──────────└┘──────────────────────────────────└

Up to 64KB of server memory per request — containing private keys, session cookies, passwords, and other users’ data. 17% of all TLS-enabled websites were affected. The bug had been in production since March 2012, over two years before discovery. The fix was a single bounds check: verify that the actual payload length matches the claimed length before calling memcpy.

POODLE (CVE-2014-3566)

October 2014. SSLv3’s CBC mode used non-deterministic padding: the padding bytes could be anything (only the last byte, indicating the padding length, was checked). This created a padding oracle. An attacker manipulates the IV to align a target byte at a block boundary, then checks whether the server accepts or rejects the padding. Each server response leaks one bit of information about the plaintext.

256 requests to decrypt one byte. For a 16-byte session cookie: 4,096 requests. Trivial for an attacker on the same network. POODLE killed SSLv3 for good — the protocol was already 15 years old and had no reason to still be in use, but backward compatibility kept it alive.

Goto Fail (CVE-2014-1266)

February 2014. Apple’s SecureTransport SSL/TLS library contained a duplicate goto fail; line:

C — Apple SecureTransport (the bug)
if ((err = SSLHashSHA1.update(&hashCtx, &signedParams)) != 0)
    goto fail;
    goto fail;  // ← DUPLICATE: always executes, skips verification

// This code is NEVER reached:
if ((err = SSLHashSHA1.final(&hashCtx, &hashOut)) != 0)
    goto fail;

err = sslRawVerify(...);  // ← signature verification, never runs

The second goto fail was not inside the if block — it always executed unconditionally, jumping past the signature verification. Any man-in-the-middle could present a fraudulent certificate and it would be accepted. A single copy-paste error made HTTPS meaningless on every Apple device running iOS 7 and OS X 10.9.

ROCA (CVE-2017-15361)

October 2017. Infineon’s RSA key generation on TPM chips and smartcards used a flawed algorithm. Instead of generating random primes, it produced primes of the form p = k × M + (65537a mod M), where M is the product of the first n small primes. This structure makes factoring vastly easier than for truly random primes.

  • 512-bit keys: factorable in seconds
  • 1024-bit keys: factorable in hours
  • 2048-bit keys: factorable in days to weeks

Impact: 750,000 Estonian national ID cards recalled. YubiKey 4 devices affected. Various government and corporate TPM chips compromised. The flaw was in the key generation, not in RSA itself — the algorithm is fine when given properly random primes.

The Common Thread

None of these four bugs broke the underlying mathematics. SHA-256 is still unbroken. AES is still unbroken. RSA with random primes is still unbroken. Every single vulnerability was an engineering failure: a missing bounds check (Heartbleed), a padding design flaw (POODLE), a duplicate line of code (Goto Fail), a weak random number generator (ROCA). The math is the easy part. The implementation is where everything burns.

13 — Crypto in Your Stack

We have studied the internals of every major cryptographic primitive. Now forget all of it — or rather, use it only to evaluate libraries, never to reimplement them. The golden rule of applied cryptography: never implement crypto primitives yourself. Use audited, battle-tested libraries maintained by specialists who understand side channels, constant-time code, and the thousand ways a correct algorithm becomes a broken implementation.

The Recommended Libraries

C: libsodium — by Daniel J. Bernstein and collaborators. High-level API, misuse-resistant design. crypto_secretbox_easy() for authenticated encryption (XSalsa20-Poly1305), crypto_box_easy() for public-key authenticated encryption (X25519 + XSalsa20 + Poly1305), crypto_sign_ed25519() for signatures, crypto_pwhash() for Argon2id password hashing, randombytes_buf() for CSPRNG.

Python: cryptography — by the Python Cryptographic Authority (PyCA). Fernet for simple encrypt/decrypt (AES-CBC + HMAC), AESGCM for low-level AES-GCM, the standard library’s hashlib for SHA-256, secrets module for CSPRNG (in stdlib since Python 3.6), and PyNaCl for libsodium bindings.

TypeScript / Node: Web Crypto API — built into every modern browser and Node.js. crypto.subtle.encrypt({name: "AES-GCM"}, ...), crypto.subtle.digest("SHA-256", ...), crypto.subtle.generateKey(...), crypto.subtle.sign(...), crypto.getRandomValues(). For NaCl-style APIs: the tweetnacl npm package.

Go: standard library — crypto/aes + crypto/cipher (NewGCM) for AES-GCM, crypto/ed25519 for signatures, crypto/sha256 for hashing, golang.org/x/crypto/nacl/box for NaCl box, crypto/rand for CSPRNG.

Rust: ring — by Brian Smith, forked from BoringSSL. ring::aead for AES-GCM and ChaCha20-Poly1305, ring::agreement for X25519 ECDH, ring::signature for Ed25519, ring::rand::SystemRandom for CSPRNG. Alternatives from the RustCrypto project: aes-gcm, ed25519-dalek, argon2 crates.

14 — Build the Messenger

Put it all together. We started with a question: how do you send a message that only one person can read? Thirteen chapters later, we have every primitive we need. Now we wire them into a working end-to-end encrypted messenger and trace a single message from Alice’s keyboard to Bob’s screen.

Step by Step

1. Registration (ch9 + ch6). Alice creates an account. Her password is hashed with Argon2id (ch9) for server-side authentication. She generates an Ed25519 identity keypair (ch6) and an X25519 signed prekey (ch5). The public halves are uploaded to the server. Her private keys never leave her device.

2. First Contact (ch8 + ch5). Bob wants to message Alice. He fetches her prekey bundle from the server — her identity key, signed prekey, and a one-time prekey. He performs X3DH (Extended Triple Diffie-Hellman, ch8): three DH operations that produce a shared secret even though Alice is offline. From this shared secret, he derives an initial root key and a sending chain for the Double Ratchet.

3. Sending a Message (ch10 + ch4 + ch3 + ch2). Bob types “hello.” The Double Ratchet (ch8) advances his sending chain and derives a message key. A fresh nonce is generated from the CSPRNG (ch10). AES-256-GCM (ch4) encrypts the plaintext and produces an authentication tag (which serves as the HMAC from ch3 — GCM folds integrity checking into the cipher). The ciphertext is sent to the server over TLS 1.3 (ch7), which provides transport-level encryption on top of the end-to-end encryption.

4. Receiving (ch8 + ch4). The server stores the ciphertext — it cannot read it. When Alice connects, she fetches the encrypted blob. Her Double Ratchet derives the same message key (the DH ratchet and KDF chain are deterministic given the same inputs). AES-256-GCM decrypts the message and verifies the authentication tag. If the tag is invalid, the message is rejected. Otherwise, Alice sees “hello.”

5. Ongoing Conversation (ch8). Each message advances the KDF chain. When the direction of conversation switches (Alice replies to Bob), a new DH ratchet step occurs: fresh ephemeral keys are exchanged, producing a new shared secret that feeds the KDF chain. This gives forward secrecy (compromising current keys cannot decrypt past messages) and post-compromise security (if an attacker compromises current keys, future DH ratchet steps will lock them out again).

6. Authentication (ch6). Alice and Bob verify each other’s identity keys via safety numbers: a hash of both public identity keys, displayed as a numeric code or QR code. They compare out-of-band (in person, over a phone call). If the numbers match, no man-in-the-middle has substituted keys during the X3DH exchange.

What E2E Doesn’t Hide

End-to-end encryption protects content. It does not protect metadata:

  • Who talks to whom — the server sees sender and recipient
  • When messages are sent — timestamps are visible to the server
  • Message sizes — even encrypted, the length is visible (padding helps but doesn’t fully solve this)
  • IP addresses — the server sees where you connect from
  • Frequency and patterns — traffic analysis reveals communication habits

Signal’s “Sealed Sender” partially addresses the sender-identity problem by encrypting the sender field with the recipient’s identity key. But metadata protection remains fundamentally harder than content protection.

What We Built

From a hash function to a complete end-to-end encrypted messenger. Each chapter added one piece: hashing for fingerprints (ch2), HMAC for integrity (ch3), AES-GCM for confidentiality (ch4), Diffie-Hellman for key agreement (ch5), Ed25519 for identity (ch6), TLS for transport (ch7), the Signal protocol for forward secrecy (ch8), Argon2 for passwords (ch9), CSPRNG for key material (ch10), side-channel awareness (ch11), real-world failure analysis (ch12), and battle-tested libraries (ch13). No single primitive is enough. The stack works only when every piece fits together correctly.

What We Didn’t Cover

Post-quantum cryptography — ML-KEM (Kyber) is already shipping in Chrome’s TLS and Signal’s PQXDH protocol. Lattice-based key exchange will replace X25519 once quantum computers threaten ECDLP. Multi-device sync — how Signal handles your phone, desktop, and iPad sharing one identity. Group messaging — Sender Keys for efficiency vs. pairwise sessions for security. Metadata resistance — onion routing, sealed sender, Private Information Retrieval. Deniability — offline deniability in Signal (neither party can prove authorship to a third party), online deniability is harder. Homomorphic encryption — computing on encrypted data without decrypting. Zero-knowledge proofs — proving you know something without revealing it. The field is vast and moving fast.
