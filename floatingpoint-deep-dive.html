THE NUMBERS THAT LIE

IEEE 754 floating point. The brilliant, cursed encoding that lets 32 bits pretend to represent the entire real number line — and the catastrophic ways it breaks when you believe it._

// 01 — the bit layout

Every floating-point number is a lie agreed upon by your CPU, your compiler, and IEEE standard 754. The lie has three parts: a sign bit that says positive or negative, an exponent that sets the scale, and a mantissa (significand) that holds the actual digits. Together they approximate a real number using the formula:

The Formula 
  value = (-1)^sign  x  2^(exponent - bias)  x  1.mantissa

  For float32:  bias = 127    exponent is 8 bits   mantissa is 23 bits
  For float64:  bias = 1023   exponent is 11 bits  mantissa is 52 bits

That 1.mantissa is doing heavy lifting. The leading 1 is implicit — it's never stored. You get 24 bits of precision from 23 stored bits. Free bit. Beautiful hack. We'll come back to this.

The First Lie

3.14f is not 3.14. It's 3.1400001049041748046875. The mantissa can't represent 0.14 exactly in binary, so it rounds to the nearest representable value. This isn't a bug. It's the fundamental tradeoff of floating point: you get enormous range (10-38 to 1038) but only ~7 decimal digits of precision. Every float is a rounded approximation. Every. Single. One. (Except powers of 2 and integers up to 224.)

FLOAT32: range ±3.4 x 10^38  |  precision: ~7.2 decimal digits  |  smallest normal: 1.18 x 10^-38

FLOAT64: range ±1.8 x 10^308  |  precision: ~15.9 decimal digits  |  smallest normal: 2.22 x 10^-308

// 02 — the exponent trick

Why not just store the exponent as a signed integer? Why this bizarre "bias" encoding where you add 127 before storing? Because it makes comparison trivial. With biased exponents, you can compare two positive floats by comparing their 32-bit representations as unsigned integers. The bits are ordered so that a larger float always has a larger integer representation. This means hardware comparators for integers work on floats for free.

Bias Encoding

The exponent field stores e + bias, where bias = 127 for float32, 1023 for float64. So exponent field value 0 means actual exponent -127, field value 127 means actual exponent 0, and field value 254 means actual exponent +127.

Exponent FieldActual ExponentMeaning
0x00 (0)-126 (special)Subnormal or zero
0x01 (1)-126Smallest normal exponent
0x7F (127)0Numbers in [1.0, 2.0)
0x80 (128)1Numbers in [2.0, 4.0)
0xFE (254)127Largest normal exponent
0xFF (255)specialInfinity or NaN

Two exponent values are reserved: 0x00 and 0xFF. They're the escape hatches for the weird stuff — zeros, subnormals, infinities, and the most cursed value in all of computing: NaN.

The Implicit Leading 1

Every normal float has the form 1.xxxxx x 2^e. Since the leading 1 is always there, why waste a bit storing it? IEEE 754 doesn't. The mantissa field stores only the fractional part after the 1. This gives you 24 bits of precision from 23 stored bits.

The Implicit Bit Trick 
  Stored mantissa:  10010001111010111000011  (23 bits)
  Actual mantissa: 1.10010001111010111000011  (24 bits of precision)
                   ^
                   This 1 is never stored — it's free!

  Exception: when exponent field = 0x00, the implicit bit is 0, not 1.
  These are "subnormal" numbers: 0.xxxxx x 2^(-126)
  This is how IEEE 754 represents numbers smaller than the minimum normal.

Zen: The Logarithmic Insight

Floating point distributes its representable values logarithmically. Half of all positive floats are between 0 and 1. A quarter are between 1 and 2. An eighth between 2 and 4. And so on. There are as many floats between 0.5 and 1.0 as there are between 1,000,000 and 2,000,000. The precision is relative, not absolute. This is both the genius and the trap.

Why Bias Beats Two's Complement

With bias encoding, the exponent field for exponent 0 is 127 (0x7F), and for exponent 1 it's 128 (0x80). The ordering is monotonic. Compare two positive floats: just compare their 32-bit patterns as unsigned integers. No special float comparison hardware needed for ordering. The sign bit, exponent, and mantissa are laid out so that integer comparison gives the right answer for same-sign values. This is why memcmp almost works on floats (it does for positive values).

The GPU Problem: Subnormals Are SLOW

Here's the dirty secret: most CPUs and GPUs can't handle subnormals in their fast path. When a subnormal appears, the hardware traps to microcode — a software fallback inside the CPU. On x86, this is a "denormal assist" that costs 50-150 cycles instead of 4-5. On NVIDIA GPUs, it's even worse: subnormal operations can be 100x slower.

#include <xmmintrin.h>  // SSE intrinsics
#include <pmmintrin.h>  // SSE3 intrinsics

// The nuclear option: tell the CPU to flush subnormals to zero
void enable_flush_to_zero() {
    // FTZ: outputs that would be subnormal become zero
    _mm_setcsr(_mm_getcsr() | 0x8000);  // FTZ bit

    // DAZ: subnormal inputs are treated as zero
    _mm_setcsr(_mm_getcsr() | 0x0040);  // DAZ bit

    // Or both at once:
    _mm_setcsr(_mm_getcsr() | 0x8040);
}

// NVIDIA equivalent in CUDA:
// nvcc --ftz=true  (flush denorms to zero)
// This is the DEFAULT on most NVIDIA GPUs!

// Check your MXCSR register:

    unsigned int mxcsr = _mm_getcsr();

    printf("  FTZ (Flush-to-Zero):    %s\n",

Wild: The Subnormal Performance Cliff

In 2023, a researcher found that a popular ML framework was 10x slower than expected on certain models. The cause? A few neurons were outputting subnormal values, and those subnormals were propagating through the network. Each subnormal multiply triggered a microcode assist on the GPU. The fix was one line: --ftz=true. Ten. Times. Faster. The numbers were so small they didn't affect the model's accuracy at all — but they cratered performance.

Zen: The Subnormal Paradox

Subnormals exist to make floating point more mathematically correct. But in practice, they mostly cause performance bugs. The IEEE 754 committee optimized for correctness. Hardware designers optimized for speed. These goals collide at the bottom of the number range, and in 2024, speed almost always wins.

The V8/LuaJIT Trick

On x86-64, user-space pointers only use 48 bits (bits 0-47). The upper 16 bits are always sign-extended. So a pointer fits in the NaN payload with room for a 3-bit type tag:

NaN-Boxing Encoding (LuaJIT style) 
  If the value is a DOUBLE (actual number):
    Just store it as-is. Any bit pattern that isn't a NaN is a valid double.

  If the value is NOT a number (pointer, boolean, nil, etc.):
    Set exponent to all 1s (NaN), quiet bit = 1, then use remaining bits:

    ┌───┬───────────┬───┬─────┬──────────────────────────────────────────┐
    │ S │ 11111111111│ 1 │ TAG │           48-bit payload                 │
    └───┴───────────┴───┴─────┴──────────────────────────────────────────┘

    TAG = 000  →  pointer to GC object (string, table, userdata, etc.)
    TAG = 001  →  integer (LuaJIT 2.1: 47-bit int fits here)
    TAG = 010  →  boolean (payload = 0 or 1)
    TAG = 011  →  lightuserdata (raw pointer, no GC)
    TAG = 100  →  nil
    TAG = 101  →  undefined (JS)  /  reserved (Lua)
    TAG = 110  →  reserved
    TAG = 111  →  reserved

  Total: ONE 64-bit word holds any value. No boxing. No heap allocation.
  A JS array of mixed types is just a flat array of uint64_t.
#include <stdint.h>
#include <string.h>
#include <math.h>

// NaN-boxed value type — one uint64_t holds everything
typedef uint64_t Value;

// The NaN mask: exponent all 1s + quiet bit = 1
#define QNAN     ((uint64_t)0x7FFC000000000000)
#define TAG_NIL  ((uint64_t)0x0001000000000000)
#define TAG_BOOL ((uint64_t)0x0002000000000000)
#define TAG_PTR  ((uint64_t)0x0000000000000000)

// Encode a double — just bit-cast it
static inline Value val_number(double d) {
    Value v;
    memcpy(&v, &d, 8);
    return v;
}

// Encode nil
static inline Value val_nil() {
    return QNAN | TAG_NIL;
}

// Encode a boolean
static inline Value val_bool(int b) {
    return QNAN | TAG_BOOL | (uint64_t)(b ? 1 : 0);
}

// Encode a pointer (48 bits)
static inline Value val_ptr(void *p) {
    return QNAN | TAG_PTR | ((uint64_t)(uintptr_t)p & 0xFFFFFFFFFFFF);
}

// Decode: is it a number?
static inline int is_number(Value v) {
    return (v & QNAN) != QNAN;  // If NOT a NaN, it's a real double
}

// Decode: extract the double
static inline double as_number(Value v) {
    double d;
    memcpy(&d, &v, 8);
    return d;
}

// Decode: extract the pointer
static inline void* as_ptr(Value v) {
    return (void*)(uintptr_t)(v & 0xFFFFFFFFFFFF);
}

// Decode: is it nil?
static inline int is_nil(Value v) {
    return v == (QNAN | TAG_NIL);
}

C — NaN-boxed value type
Hack: Why NaN Boxing Beats Tagged Unions

The alternative is a tagged union: struct { int type; union { double d; void *p; } } — that's 16 bytes (8 for the union, 4 for the tag, 4 padding). NaN boxing puts everything in 8 bytes. Half the memory. For an array of 1 million values, that's 8 MB vs 16 MB. More values fit in cache. And the type check is a single bitwise AND instead of a memory load. LuaJIT's interpreter is one of the fastest in the world, and NaN boxing is one of the reasons why.

Danger: 5-Level Paging Breaks NaN Boxing

Intel's 5-level paging (LA57) extends virtual addresses to 57 bits. That's 9 bits too many for the 48-bit NaN payload. V8 handles this by compressing pointers to 32-bit offsets within a 4GB "cage." LuaJIT 2.1 limits itself to the low 47 bits and crashes if the OS maps memory above that. This is an active pain point — the NaN-boxing trick has an expiration date as address spaces grow.

// 05 — the magic number: 0x5f3759df

In 1999, id Software shipped Quake III Arena. Buried in the source code of q_math.c was a function that changed how people think about floating point. It computes 1/√x about 4x faster than the standard library, using a bit-level hack so bizarre that even the original author left a comment: "what the fuck?"

// The original Quake III code (cleaned up slightly)
float Q_rsqrt(float number) {
    long i;
    float x2, y;
    const float threehalfs = 1.5F;

    x2 = number * 0.5F;
    y  = number;
    i  = * (long *) &y;                     // evil floating point bit level hacking
    i  = 0x5f3759df - ( i >> 1 );           // what the fuck?
    y  = * (float *) &i;
    y  = y * ( threehalfs - ( x2 * y * y ) ); // 1st iteration of Newton's method
//  y  = y * ( threehalfs - ( x2 * y * y ) ); // 2nd iteration, can be removed

    return y;
}
C — the legendary Q_rsqrt

Let's break this apart. Every step. Because this is the most beautiful abuse of IEEE 754 ever written.

Step 1: Bit-Cast Float to Integer

i = *(long *)&y; reinterprets the float's 32-bit pattern as an integer. No conversion — just a raw reinterpretation. The float y has bit pattern 0x40490FDB (for π), and now the integer i holds that same value: 1078530011.

Here's the insight: the integer representation of a float is an approximation of its log. Why? Because the exponent IS the integer part of log2, and the mantissa IS the fractional part (roughly). The integer value of a float's bit pattern is approximately:

Float bits as approximate logarithm 
  Integer(bits of float x) ≈ 2^23 × (log₂(x) + 127)

  The exponent field contributes:  exponent × 2^23
  The mantissa field contributes:  mantissa (which ≈ log₂(1+frac) × 2^23)

  So: int_bits ≈ 2^23 × (exponent - 127 + mantissa/2^23)
              ≈ 2^23 × (log₂(x) + 127)
              ≈ 2^23 × log₂(x) + 2^23 × 127

  This is a LINEAR function of log₂(x). We're in log space!

Step 2: The Magic Subtraction

i = 0x5f3759df - (i >> 1); — This is where the magic happens.

We want 1/√x. In log space, log(1/√x) = -½ log(x). The right shift i >> 1 divides the log by 2. The subtraction from the magic constant negates it and adds back the bias. The constant 0x5f3759df was chosen (likely by trial and error, possibly by Chris Lomont's later analysis) to minimize the maximum relative error of this initial guess.

The Math Behind the Magic 
  Let I(x) = integer representation of float x
  We established: I(x) ≈ 2^23 × (log₂(x) + 127) = L × log₂(x) + L × B
    where L = 2^23 = 8388608, B = 127

  We want: y = x^(-1/2)
  So: I(y) ≈ L × log₂(y) + L × B
           = L × (-1/2) × log₂(x) + L × B
           = -1/2 × (I(x) - L × B) + L × B
           = 3/2 × L × B - 1/2 × I(x)
           = 3/2 × 8388608 × 127 - I(x)/2
           = 3/2 × 1065353216 - I(x)/2
           = 1598029824 - I(x)/2
           = 0x5F400000 - (I(x) >> 1)

  But 0x5F400000 isn't the constant — it's 0x5F3759DF.
  The difference (0x5F400000 - 0x5F3759DF = 0xCA621) is a correction term
  that accounts for the nonlinearity of the mantissa-to-log approximation.
  It was tuned to minimize worst-case error across all floats.

Step 3: Newton's Method Polish

y = y * (threehalfs - (x2 * y * y)); — One iteration of Newton's method for f(y) = 1/y2 - x = 0. The initial guess from the integer hack is already within 1% of the true answer. One Newton iteration brings it to within 0.175%. A second iteration (commented out) gives full float32 precision.

// Modern version using memcpy (no strict aliasing violation)
float fast_rsqrt(float x) {
    float xhalf = 0.5f * x;
    uint32_t i;
    memcpy(&i, &x, 4);

    // Log-space approximation of x^(-0.5)
    i = 0x5F375A86 - (i >> 1);  // Improved constant (Lomont 2003)

    memcpy(&x, &i, 4);

    // Newton refinement: y = y * (1.5 - 0.5*x*y*y)
    x = x * (1.5f - xhalf * x * x);  // Accurate to ~0.175%
    x = x * (1.5f - xhalf * x * x);  // Accurate to ~0.0003%

    return x;
}

// Why did this matter? In 1999, a hardware sqrt + divide took ~70 cycles.

The Deeper Lesson

The fast inverse square root works because IEEE 754 floats are secretly a logarithmic representation. The integer value of a float's bits is approximately proportional to its log2. This means integer arithmetic on float bits does approximate log-space operations. Shift right = divide the log = take a root. Subtract from a constant = negate the log and add bias = invert. The magic constant is just the right bias for the specific operation 1/√x. You could derive different constants for x2/3, x-1/3, or any power.

Why Google Invented BFloat16

The key insight: during ML training, range matters more than precision. Gradient magnitudes can swing from 10-30 to 1030. FP16 overflows at 65,504 — a single large gradient blows up to infinity and poisons the entire training run. You need "loss scaling" hacks to keep values in FP16's tiny range. BF16 has the same range as float32, so gradients never overflow. The tradeoff is only 2.4 decimal digits of precision, but neural networks are robust to quantization noise — they were trained with dropout and stochastic gradient descent, which add way more noise than rounding errors.

Zen: BF16 Is Just Truncated Float32

BFloat16 has the same exponent width as float32. Converting float32 to BF16 is literally chopping off the bottom 16 bits of the mantissa. No exponent remapping. No range scaling. Just (uint16_t)(float32_bits >> 16). Converting back is (uint32_t)bf16_bits << 16. Google designed it this way specifically so conversion is free in hardware — just wire the top 16 bits.

Conversion Code

#include <stdint.h>
#include <string.h>
#include <math.h>

// ---- FP16 ↔ FP32 ----

float fp16_to_fp32(uint16_t h) {
    uint32_t sign = (uint32_t)(h & 0x8000) << 16;
    uint32_t exp  = (h >> 10) & 0x1F;
    uint32_t mant = h & 0x3FF;

    uint32_t result;
    if (exp == 0) {
        if (mant == 0) {
            result = sign;  // ±0
        } else {
            // Subnormal: normalize it
            exp = 1;
            while (!(mant & 0x400)) { mant <<= 1; exp--; }
            mant &= 0x3FF;
            result = sign | ((uint32_t)(exp + 112) << 23) | (mant << 13);
        }
    } else if (exp == 0x1F) {
        // Inf or NaN
        result = sign | 0x7F800000 | (mant << 13);
    } else {
        // Normal: rebias exponent from 15 to 127
        result = sign | ((uint32_t)(exp + 112) << 23) | (mant << 13);
    }
    float f;
    memcpy(&f, &result, 4);
    return f;
}

uint16_t fp32_to_fp16(float f) {
    uint32_t bits;
    memcpy(&bits, &f, 4);

    uint16_t sign = (bits >> 16) & 0x8000;
    int32_t exp  = ((bits >> 23) & 0xFF) - 127 + 15;  // rebias
    uint32_t mant = bits & 0x7FFFFF;

    if (exp <= 0) return sign;           // too small → ±0
    if (exp >= 0x1F) return sign | 0x7C00; // too big → ±∞

    // Round to nearest even
    uint16_t result = sign | (exp << 10) | (mant >> 13);
    if ((mant >> 12) & 1) {  // round bit set
        if ((mant & 0xFFF) || (result & 1)) result++;  // round up
    }
    return result;
}

// ---- BF16 ↔ FP32 ----
// This is embarrassingly simple.

float bf16_to_fp32(uint16_t h) {
    uint32_t bits = (uint32_t)h << 16;  // Just shift left. That's it.
    float f;
    memcpy(&f, &bits, 4);
    return f;
}

uint16_t fp32_to_bf16(float f) {
    uint32_t bits;
    memcpy(&bits, &f, 4);

    // Round to nearest even (add rounding bias + check sticky bits)
    uint32_t rounding_bias = ((bits >> 16) & 1) + 0x7FFF;
    bits += rounding_bias;

    return (uint16_t)(bits >> 16);  // Just shift right. That's it.
}

// 07 — rounding & catastrophic cancellation

Every floating-point operation lies. Not by much — usually by at most half a ULP (unit in last place) — but the lies accumulate. Add a million numbers and the accumulated rounding error can swamp the result. Subtract two nearly-equal numbers and you lose all your significant digits in a single operation. This chapter is about how floats lie, how to catch them, and how Kahan summation fights back.

The Four Rounding Modes

Round to Nearest, Even (a.k.a. "banker's rounding") is the default. When a result falls exactly between two representable floats, it rounds to the one with a 0 in the last mantissa bit. This prevents statistical bias — half the time it rounds up, half down. The other modes exist for interval arithmetic (bracket the true answer by rounding up and down) and for compatibility with integer truncation.

Catastrophic Cancellation

This is the big one. When you subtract two nearly-equal numbers, the leading significant digits cancel, and you're left with only the noise in the trailing digits.

// Catastrophic cancellation in action
#include <stdio.h>

int main() {
    float a = 1.0000001f;  // 7 significant digits
    float b = 1.0000000f;  // 7 significant digits
    float c = a - b;       // CATASTROPHE

    printf("a = %.10f\n", a);   // 1.0000001192
    printf("b = %.10f\n", b);   // 1.0000000000
    printf("c = %.10e\n", c);   // 1.1920928955e-07

    // The TRUE answer is 0.0000001 = 1.0e-7
    // The computed answer is 1.192e-7
    // That's 19.2% relative error. From ONE subtraction.
    //
    // What happened? Both a and b had 7 correct digits.
    // The subtraction cancelled the first 7 digits (they were the same).
    // What's left is the rounding noise from the 8th digit onward.
    // We went from 7 digits of precision to ~1 digit. In one operation.
}

Danger: The Quadratic Formula Trap

The classic formula x = (-b ± √(b² - 4ac)) / 2a has catastrophic cancellation when b² >> 4ac. The discriminant √(b² - 4ac) is close to |b|, so one root subtracts two nearly-equal values. The fix: compute the root that ADDS (no cancellation), then use Vieta's formula x1 · x2 = c/a to get the other root. Every numerical methods textbook teaches this. Most programmers never learn it.

Kahan Summation: Fighting the Lies

Naive summation accumulates rounding error — O(√n) in the worst case. Kahan summation carries the rounding error in a separate "compensation" variable and adds it back on the next iteration. The result is accurate to within 2 ULP regardless of how many numbers you add.

// Naive summation: error grows with n
float naive_sum(float *arr, int n) {
    float sum = 0.0f;
    for (int i = 0; i < n; i++)
        sum += arr[i];  // Each addition loses bits
    return sum;
}

// Kahan summation: error bounded regardless of n
float kahan_sum(float *arr, int n) {
    float sum = 0.0f;
    float c = 0.0f;    // Compensation for lost low-order bits

    for (int i = 0; i < n; i++) {
        float y = arr[i] - c;    // Add compensation from last round
        float t = sum + y;        // sum is big, y is small — low bits lost
        c = (t - sum) - y;        // (t - sum) recovers the high part of y
                                  // subtracting y gives the lost low part
        sum = t;
    }
    return sum;
}

// True answer: 10,000,000.0

// Kahan sum:   10,000,000.0  (error = 0.0)
//
// For 1 billion values of 0.1f:
// True answer: 100,000,000.0
// Naive sum:   ~83,865,912.0  (16% error!)

The line c = (t - sum) - y is algebraically zero. An "optimizing" compiler with -ffast-math will eliminate it. This destroys Kahan summation entirely. Danger: -O2 Changes Your Math

With x87 math, the compiler decides when to spill registers to memory. Every spill rounds 80 bits to 64 bits. More aggressive optimization means more values stay in registers longer, which means fewer roundings, which means different numerical results. You can get a different answer from the same code by changing -O1 to -O2. This is not a compiler bug. This is a fundamental consequence of excess precision.

The Modern Fix: SSE2

Since AMD64 (2003), all x86-64 CPUs have SSE2, which provides 64-bit double and 32-bit float registers — no excess precision. GCC and Clang default to -mfpmath=sse on x86-64. The x87 problem is essentially solved for 64-bit code. But 32-bit code (still running on embedded, legacy systems, and some game engines) still defaults to x87. And if you compile with -mfpmath=387 on x86-64 for some reason, you get the same problems.

// 09 — float comparison: ULP distance

You've seen the code. You've probably written it yourself. if (fabs(a - b) < 0.0001). It works for numbers near 1.0. It fails catastrophically for numbers near zero (treats 1e-10 and 1e-15 as "not equal" when they're 5 ULPs apart) and for large numbers (treats 1000000.0 and 1000000.1 as "equal" when they're millions of ULPs apart). Epsilon comparison is broken. ULP distance is the right metric.

What Is a ULP?

A ULP — Unit in Last Place — is the gap between a float and its nearest neighbor. For the float 1.0, the next representable float is 1.00000011920929 (in float32), so 1 ULP at 1.0 is approximately 1.19e-7. For the float 1000000.0, 1 ULP is approximately 0.0625. ULP distance scales with the magnitude of the numbers, which is exactly what you want for comparing floats.

ULP Gaps at Different Magnitudes 
  Value          1 ULP gap           Decimal digits of precision
  ─────          ─────────           ───────────────────────────
  1.0            1.19e-7             ~7 digits
  100.0          7.63e-6             ~7 digits
  10,000.0       9.77e-4             ~7 digits
  1,000,000.0    0.0625              ~7 digits
  1e20           1.37e13             ~7 digits

  The ULP gap grows proportionally to the value.
  The number of correct decimal digits stays ~constant (~7 for float32).
  This is EXACTLY why epsilon comparison fails:
    |1e20 - (1e20 + 1e13)| = 1e13 >> 0.0001  (epsilon says "not equal")
    but they're only 1 ULP apart! (nearly identical in float terms)

Computing ULP Distance

The brilliant trick: IEEE 754 floats are ordered such that adjacent floats have adjacent integer representations (for same-sign values). So the ULP distance between two floats is just the absolute difference of their integer representations.

#include <stdint.h>
#include <string.h>
#include <math.h>
#include <stdlib.h>

// ULP distance between two float32 values
int64_t ulp_distance(float a, float b) {
    // Handle NaN
    if (isnan(a) || isnan(b)) return INT64_MAX;

    // Handle sign difference: both ±0 are 0 ULPs apart
    if (a == 0.0f && b == 0.0f) return 0;

    // Same sign? Integer subtraction gives ULP distance directly
    int32_t ia, ib;
    memcpy(&ia, &a, 4);
    memcpy(&ib, &b, 4);

    // Flip negative floats so they sort correctly
    // Negative floats: 0x80000001 (-FLT_MIN) to 0xFF7FFFFF (-FLT_MAX)
    // We want them to map to: -1 (just below +0) to -INT_MAX
    if (ia < 0) ia = (int32_t)0x80000000 - ia;
    if (ib < 0) ib = (int32_t)0x80000000 - ib;

    // Now integer distance = ULP distance, even across zero
    int64_t diff = (int64_t)ia - (int64_t)ib;
    return diff < 0 ? -diff : diff;
}

// Comparison using ULP tolerance
int nearly_equal(float a, float b, int64_t max_ulps) {
    return ulp_distance(a, b) <= max_ulps;
}

The Bit-Cast Insight

The reason ULP distance works via integer subtraction is the same reason biased exponents work for comparison: IEEE 754 was designed so that the integer representation preserves ordering. For positive floats, incrementing the integer by 1 gives the next representable float. The exponent and mantissa fields work together as a single monotonic integer sequence. This wasn't an accident — it was a deliberate design goal of William Kahan and the IEEE 754 committee.

// 10 — fast math tricks

The IEEE 754 bit layout isn't just for correctness — it's a playground for tricks. Because the exponent field IS the integer part of log2, and the mantissa field IS the fractional part, you can do approximate logarithms, exponentials, and powers using nothing but integer arithmetic on float bits. These tricks run in game engines, GLSL shaders, and embedded systems where every cycle counts.

Float-to-Int as Approximate log2

#include <stdint.h>
#include <string.h>

// Approximate log2(x) using bit casting
float fast_log2(float x) {
    uint32_t bits;
    memcpy(&bits, &x, 4);

    // Integer value ≈ 2^23 × (log2(x) + 127)
    // So: log2(x) ≈ bits/2^23 - 127
    return (float)bits / (1 << 23) - 127.0f;
}

// For x = 10.0:  exact log2 = 3.32, fast_log2 ≈ 3.25 (2% error)
// For x = 100.0: exact log2 = 6.64, fast_log2 ≈ 6.56 (1.2% error)
//
// Max error: ~3.5%. Free (no division, no lookup table, no iteration).

// The magic number trick: add 2^23, then subtract 2^23.
// For floats in [0, 2^23), adding 2^23 forces the mantissa to align
// so that the fractional bits fall off the bottom.

float fast_floor(float x) {
    // For positive x in [0, 8388608):
    float magic = 8388608.0f;  // 2^23 = 0x4B000000

    return y - magic;           // Remove the offset — integer remains

The union Power Tool

// Type-punning via union: the classic C hack for float bit manipulation
// (Technically UB in C++ but defined behavior in C11)

typedef union {
    float f;
    uint32_t i;
} float_bits;

// Fast approximate reciprocal: 1/x
float fast_rcp(float x) {
    float_bits fb;
    fb.f = x;
    // In log space: log(1/x) = -log(x)
    // int_bits ≈ 2^23 × (log2(x) + 127)
    // For 1/x: int_bits ≈ 2^23 × (-log2(x) + 127) = 2 × 2^23 × 127 - int_bits
    fb.i = 0x7EF311C3 - fb.i;  // Magic constant for 1/x

    // One Newton iteration: y = y * (2 - x*y)
    fb.f = fb.f * (2.0f - x * fb.f);

    return fb.f;
}

// Fast approximate exp2: 2^x
float fast_exp2(float x) {
    // If we want 2^x, and int_bits ≈ 2^23 × (log2(result) + 127),
    // then int_bits ≈ 2^23 × (x + 127)
    float_bits fb;
    fb.i = (uint32_t)((1 << 23) * (x + 127.0f));
    return fb.f;
    // Max error: ~4%. Fast enough for fog, bloom, particle fading.
}

// Fast approximate pow(x, p):
float fast_pow(float x, float p) {
    // log(x^p) = p × log(x)
    // In integer land: multiply the log-representation by p
    float_bits fb;
    fb.f = x;
    fb.i = (uint32_t)(p * (fb.i - (1 << 23) * 127)) + (1 << 23) * 127;
    return fb.f;
    // fast_pow(2.0, 10.0) ≈ 1024 (exact for powers of 2)
    // fast_pow(3.0, 3.0)  ≈ 27.1 (0.4% error)
}

The union trick also gives you integer comparison of floats. // 11 — float keys in hash maps: cursed

Here's a fun puzzle. Put NaN into a hash map as a key. Try to look it up. You can't. IEEE 754 mandates that NaN ≠ NaN. Every comparison returns false, including equality. So when the hash map finds the bucket and compares your lookup key to the stored key, it gets false. The entry is there, in the right bucket, and you can never retrieve it. Insert NaN twice? Now you have two entries in the same bucket that can never be found. Keep inserting? Congratulations, you've created a memory leak inside a hash map.

#include <math.h>
#include <stdio.h>

int main() {
    double nan1 = nan("");
    double nan2 = nan("");

    printf("nan == nan:   %d\n", nan1 == nan2);   // 0 (false!)
    printf("nan != nan:   %d\n", nan1 != nan2);   // 1 (true!)
    printf("nan < nan:    %d\n", nan1 < nan2);    // 0
    printf("nan > nan:    %d\n", nan1 > nan2);    // 0
    printf("nan <= nan:   %d\n", nan1 <= nan2);   // 0
    printf("nan >= nan:   %d\n", nan1 >= nan2);   // 0

    // EVERY comparison involving NaN returns false (except !=)
    // This is IEEE 754's way of saying "garbage in, garbage out"
    // But it absolutely WRECKS hash maps.

    // The second landmine: +0.0 and -0.0
    double pos_zero = +0.0;
    double neg_zero = -0.0;

    printf("\n+0 == -0:    %d\n", pos_zero == neg_zero);  // 1 (true!)
    printf("+0 bits:     0x%016llX\n", *(uint64_t*)&pos_zero);
    // 0x0000000000000000
    printf("-0 bits:     0x%016llX\n", *(uint64_t*)&neg_zero);
    // 0x8000000000000000

    // They compare EQUAL but have DIFFERENT bit patterns.
    // If your hash function uses the bit pattern (as it should),
    // +0.0 and -0.0 hash to different buckets.
    // But they compare equal, so map[+0.0] and map[-0.0]
    // should be the same entry... right? RIGHT?

Danger: Go's NaN Map Trap

In Go, m[math.NaN()] = "hello" inserts the entry, and then m[math.NaN()] can never find it. It doesn't panic — it silently fails the lookup. The entry accumulates in the map forever, unretrievable but consuming memory. The Go team decided this was acceptable because "don't use NaN as a map key" is obvious advice. But code that processes untrusted float data from a network can end up with NaN values, and if those values become map keys, you have a slow memory leak that's nearly impossible to debug.

The CoGS Policy

The NaN/Inf policy across the cogs codebase reflects the fundamental tension:

Cowrie Binary Format

NaN and Inf are allowed in all 5 language implementations (Go, Python, TypeScript, Rust, C). Binary formats preserve exact bit patterns, including NaN payloads and the sign of zero. If the producer wrote NaN, the consumer gets NaN. No judgment, no filtering.

Glyph Text/JSON

NaN and Inf are rejected across all 5 languages. JSON has no representation for NaN or Infinity — they're not valid JSON tokens. Glyph's canonical float representation uses canon_float(), which returns an error on NaN/Inf. If your data contains NaN, serialize it in cowrie binary, not glyph text.

Zen: Rust Got It Right

Rust's decision to not implement Hash and Eq for f64 is the correct solution. It makes the problem a compile-time error instead of a runtime bug. If you need float keys, you must use the ordered_float crate's OrderedFloat<f64> or NotNan<f64> wrapper, which explicitly handles NaN (either by panicking on construction or by defining a total order). The type system prevents the bug class entirely.

// 12 — quantization math

Quantization is the art of representing high-precision numbers with low-precision integers and a floating-point scale factor. A 7-billion-parameter model in float32 is 28 GB. In int8 with per-row scales, it's 7 GB plus a few megabytes of scale values. Same model, 4x smaller, and if you do the math right, nearly indistinguishable outputs. But "doing the math right" is where IEEE 754 precision bites you in surprising ways.

The Basic Scheme

Quantization and Dequantization 
  QUANTIZE (float32 → int8 + scale):
  ─────────────────────────────────
  1. Find absmax = max(|values|)          e.g., absmax = 3.7
  2. Compute scale = absmax / 127         e.g., scale = 0.02913
  3. Quantize: q[i] = round(v[i] / scale) e.g., 3.7 → 127, -1.85 → -63
  4. Clamp to [-127, 127]                 (symmetric quantization)

  DEQUANTIZE (int8 + scale → float32):
  ──────────────────────────────────
  result[i] = q[i] × scale               e.g., 127 × 0.02913 = 3.699

  Error: original 3.7 → quantized 127 → dequantized 3.699
         0.001 absolute error ≈ 0.027% relative error

  ┌──────────┐   ÷ scale    ┌──────────┐   × scale    ┌──────────┐
  │ float32  │ ──────────→  │   int8   │ ──────────→  │ float32  │
  │ weights  │   round()    │ quantized│              │ restored │
  └──────────┘              └──────────┘              └──────────┘
       28 GB                    7 GB                    ≈28 GB
                            + scales                  (lossy but close)

Per-Tensor vs Per-Row vs Per-Group Scales

// Per-tensor: one scale for the entire weight matrix
// Simple but loses precision if values have different magnitudes per row
void dequant_per_tensor(int8_t *q, float scale, float *out, int n) {
    for (int i = 0; i < n; i++)
        out[i] = q[i] * scale;
}

// Per-row: one scale per row of the weight matrix
// Each row uses its own absmax → better precision
void dequant_per_row(int8_t *q, float *scales,
                     float *out, int rows, int cols) {
    for (int r = 0; r < rows; r++)
        for (int c = 0; c < cols; c++)
            out[r * cols + c] = q[r * cols + c] * scales[r];
}

// Per-group (e.g., GPTQ, AWQ): one scale per group of 32-128 values
// Best precision, but more scale overhead
void dequant_per_group(int8_t *q, float *scales,
                      float *out, int n, int group_size) {
    for (int i = 0; i < n; i++)
        out[i] = q[i] * scales[i / group_size];
}

The Accumulation Problem

Matrix multiplication with quantized values multiplies int8 × int8 and accumulates in int32. Thousands of products are summed per output element. Then the int32 accumulator is converted to float32 and multiplied by the output scale. Where does IEEE 754 precision matter here?

Quantized Matrix Multiply Accumulate 
  For one output element:
  acc = Σ (q_weight[i] × q_activation[i])    for i in [0, K)
        ↑ int8 × int8 = int16, accumulated in int32

  K = hidden dimension = 4096 (typical for 7B model)
  Each product: max |int8 × int8| = 127 × 127 = 16,129
  Sum of 4096 products: max = 4096 × 16,129 = 66,064,384

  int32 max = 2,147,483,647   ← plenty of headroom (32x)

  Final step: float_out = (float)acc × scale_w × scale_a
  The (float) cast is exact for |acc| < 2^24 = 16,777,216
  Above 2^24, we lose precision in the cast — but we have 32x headroom,
  so in practice the accumulator rarely exceeds 2^24 for well-scaled models.

  The REAL precision constraint: the SCALES.
  scale = absmax / 127. For absmax = 3.7: scale = 0.029133858...
  In float32, this is stored as 0.029133858531713485...
  In float16, this would be 0.02911376953125 — already 0.07% error.
  In bfloat16, this would be 0.029296875 — 0.56% error.

  For the quantized weights (only 8 bits of information), even float16
  scales have more precision than the data they're scaling. float32 scales
  are overkill. This is why GPTQ stores scales in float16.

Why FP16 Scales Are Enough

// The information content argument:
//
// INT8 quantized values have at most 8 bits of information (256 levels).
// The scale factor's job is to map those 256 levels to the right range.
//
// FP16 has 10 mantissa bits (11 with implicit 1) = 2048 levels per octave.
// That's 8x more precision than the data it's scaling.
//
// BF16 has 7 mantissa bits (8 with implicit 1) = 256 levels per octave.
// That's EXACTLY the same precision as int8 — barely enough.
// For 4-bit quantization (Q4_K, GGUF), BF16 scales are fine.
// For 8-bit quantization, FP16 scales are the sweet spot.
//
// FP32 scales for int8 data = 23 bits of precision for 8 bits of data.
// That's 15 wasted mantissa bits. Pure overhead in memory and bandwidth.
//
// The hierarchy:
//   Data precision    Best scale format    Wasted precision
//   ──────────────    ────────────────    ────────────────
//   INT4 (4 bits)     BF16 (7+1 bits)     4 bits wasted
//   INT8 (8 bits)     FP16 (10+1 bits)    3 bits wasted
//   INT16 (16 bits)   FP32 (23+1 bits)    8 bits wasted

The Absmax Calibration

#include <math.h>
#include <stdint.h>
#include <stdlib.h>

// Absmax calibration: find the scale that maps the range to [-127, 127]
float compute_absmax_scale(float *values, int n) {
    float absmax = 0.0f;
    for (int i = 0; i < n; i++) {
        float a = fabsf(values[i]);
        if (a > absmax) absmax = a;
    }
    return absmax / 127.0f;
}

// Quantize a row of float32 values to int8 with absmax scaling
void quantize_row(float *in, int8_t *out, float *scale, int n) {
    *scale = compute_absmax_scale(in, n);

    if (*scale == 0.0f) {
        // All values are zero — avoid division by zero
        for (int i = 0; i < n; i++) out[i] = 0;
        return;
    }

    float inv_scale = 1.0f / *scale;
    for (int i = 0; i < n; i++) {
        float v = in[i] * inv_scale;

        // Round to nearest, clamp to [-127, 127]
        int q = (int)roundf(v);
        if (q > 127) q = 127;
        if (q < -127) q = -127;

        out[i] = (int8_t)q;
    }
}

// The scale is the ONLY floating-point value in the quantized tensor.
// Everything else is int8. The entire precision of the dequantized
// output depends on two things:
//   1. The quantization step size (absmax / 127)
//   2. The precision of the scale value itself
//
// For a 4096-column weight matrix with per-row scales:
//   Weight data:  4096 × 1 byte  = 4,096 bytes
//   Scale:        1 × 4 bytes    = 4 bytes (float32)
//   Scale:        1 × 2 bytes    = 2 bytes (float16) — saves 2 bytes/row
//
// For a model with 100,000 rows, float16 scales save 200 KB total.
// Not much in absolute terms, but the bandwidth savings on GPU
// matter when you're moving data through tensor cores at 3 TB/s.

The Quantization Insight

Quantization works because neural networks are over-parameterized. A 7B parameter model does not have 7 billion independent degrees of freedom — the weights are highly correlated and live on a low-dimensional manifold. Reducing each weight from 32 bits to 8 bits (or even 4 bits) barely moves the output because most of the 32 bits were noise anyway. The 23-bit float32 mantissa was carrying at most 4-8 bits of useful information per weight. Quantization strips the noise and keeps the signal.

Hack: GGUF's Block Quantization

GGUF (the format behind llama.cpp) uses block quantization: groups of 32 values share one FP16 scale and one FP16 minimum. Within each block, values are stored as 4-bit integers (Q4_0, Q4_1) or even 2-bit with more complex scale structures (Q2_K). The K-quants (Q4_K_M, Q6_K) use a two-level scale hierarchy: per-superblock FP16 scale, then per-subblock 6-bit relative scales. This is quantization of the quantization parameters — turtles all the way down.

Wild: The 1-Bit Frontier

BitNet (2024) pushes quantization to the extreme: ternary weights ({-1, 0, +1}). No multiplication needed — matrix multiply becomes addition and subtraction. The "scale" is a single per-row float that just scales the accumulator at the end. At 1.58 bits per weight (log2(3)), a 7B model fits in 1.4 GB. IEEE 754 precision in the scale factor is irrelevant when each weight carries 1.58 bits of information. The scale could be stored in int8 and nobody would notice.
